3、观察Storm UI界面参数，来分析Storm并行度

http://192.168.199.147:8080

我们总结下：Storm的默认并行度

  • 一个supervisor节点最多启动4个worker进程
  • 每一个topo默认占用一个worker进程
  • 每个worker进程会启动一个executor（acker）

注释：有很多对这个不太理解，其实，当我们提交一个topology作业的时候，如果没有任何spout/bolt，就会只有一个executor，这个executor其实就是acker线程了。我们每new一个spout/bolt就会多一个线程。

  • 每个executor(包括Acker线程)启动一个task

4、Storm并行度调整

1）修改我们的Storm程序

  • 修改worker数量
Config config = new Config();config.setNumWorkers(2);config.setNumAckers(0);

通过ui可以看到2个worker

通过jps也可以看到2个worker

  • 修改executor数量
topologyBuilder.setSpout("number_spout_id", new NumberSourceSpout(), 2);topologyBuilder.setBolt("sum_bolt_id", new NumberSumbolt(), 2).shuffleGrouping("number_spout_id");
  • 修改task数量
topologyBuilder.setBolt("sum_bolt_id", new NumberSumbolt(), 2).setNumTasks(4).shuffleGrouping("number_spout_id");
  • acker作用

一个worker会有一个acker，一个acker会跟踪一个topology，保证tuple的可靠传输，我们这里仅做简单介绍，后面我们讲storm可靠性的时候，会详细讲解acker的，欢迎持续关注接下来的视频

运行中的topology我们能改变其并行度吗（并行度动态调整）

答案是肯定的。

Storm 的一个很好的特性是可以增加或减少 worker 进程 和/或 executor（执行器）的数量, 而无需重新启动集群或 topology（拓扑）. 这样做的行为称之为 rebalancing（重新平衡）.

您有两个选项来 rebalance（重新平衡）一个 topology（拓扑）:

  1. 使用 Storm web UI 来 rebalance（重新平衡）指定的 topology.
  2. 使用 CLI 工具 storm rebalance, 如下所示.

以下是一个使用 CLI 工具的示例:

## 重新配置 topology "mytopology" 以使用 5 个 worker 进程,## 该 spout "blue-spout" 要使用 3 个 executors（执行器）并且## 该 bolt "yellow-bolt" 要使用 10 executors（执行器）.​$ storm rebalance mytopology -n 5 -e blue-spout=3 -e yellow-bolt=10
