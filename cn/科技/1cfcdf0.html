干货｜STM32 SDIO折腾记

2020-10-29
| 科技

月初打样回来的PCB，断断续续搞到现在，离完成度还有距离。这块被我老婆称为“背心”的板子，其实是我的STM32F4实验平台（Nucleo我有，但是光插针用起来也不方便，还得扩展卡）。这块板子的布局是为一个数字音乐播放器设计的，但现在软件工作还缺很多。

这个月里实验过了STM32F411的时钟配置、DMA、I2C、I2S、SDIO. 其中 SDIO 折腾得最久，拿出来和大家分享下。

SDIO这个模块主要是为了读写SD存储卡（也可以操作SDIO wifi卡之类的），ST从STM32F10x开始就已经支持SDIO/SDMMC了，因为是硬件专用模块，比用SPI方式读写SD卡效率要高很多，软件也大大简化。我曾经（五年前的事了）在FPGA上为了记录数据，用SPI方式写过SD卡，因为状态机太复杂，用起来很不好用。在8位MCU的时代，用SPI方式操作SD卡已比较流行，可以找到不少的参考代码。SD卡数据线有4条，在SPI模式下就利用不了全部的带宽，STM32的SDIO则是全部数据线都能用上的。

上图是常见标准尺寸SD卡和MicroSD卡(和TF是同样东西不同叫法)的引脚，下图是STM32 SDIO模块的硬件连接，对应Card bus的引脚连到SD卡(卡座)上。不过，SD卡只有D0..D3四条数据线，STM32的SDIO_D7..4这四条多出来的线以及8位宽度模式是给什么用的？我也不知道。反正多的线也没地方连。

焊好SD卡座之后：对应SD卡pin都接到MCU上了。卡座上另有CD引脚是检测卡插入的，卡插入后CD与GND连通。还有WP引脚是识别写保护开关用的，我没连它。

个人的DIY就是折腾，ST官方库已经有SDIO的函数了为何不用？重新发明轮子？我就喜欢这么干，不喜欢烦琐的一层层打包的库函数啊，直接写寄存器才好玩

先得知道这些信号都干吗的才是，ST手册这个图表示得差不多了

除了 SDIO_CK 是时钟，只由MCU产生之外，SDIO_CMD和SDIO_Dn都是双向的I/O。SDIO_CMD 是命令通道，由MCU写命令，SD卡写应答；两边交替驱动，都不驱动时为高阻态。SDIO_Dn是实际传输存储数据才用的，在读卡的时候被SD卡驱动，在写卡的时候由MCU驱动写数据和校验，SD卡响应状态。具体时序已经被MCU硬件搞定，CRC计算都已经做好，软件只需要操作SDIO设备APB总线上的寄存器就可以了，要说简单也的确不难。

上电的操作：检测是否卡片插入了卡座，再使能SDIO模块，给SD卡供电(我加了一个PMOS管做电源开关)，配置SDIO时钟，然后等一小会儿就可以发送SD命令了。

SD卡/MMC卡在初始化阶段，时钟频率不能超过400kHz，所以一开始时钟要设得慢，待初始化完成后再调整。CLKCR这个寄存器还是容易看懂的。在STM32F411上，SDIO部分和USB OTG公用一个48MHz的时钟，所以分频以48MHz计算，而不是从CPU运行频率或APB的频率来算。

第一条SD命令应当是 GO_IDLE_STATE, 也就是 CMD0, 将SD卡恢复到IDLE状态。不过看SD规范的文档，这条命令是没有应答的，所以执行成功与否也无从知道，比如电路本身就有问题的话……

发送命令还不难，写SDIO_ARG和SDIO_CMD寄存器就是了，命令号在SDIO_CMD里。MCU里面有一个状态机来实现命令过程(Command Path State Machine)：

写命令寄存器，将CPMSEN位置1，才会开始命令过程。然后，状态会反映在 SDIO_STA 这个寄存器中。怎么知道 CMD0 已经完成了呢？SDIO_STA 里面有个 CMDSENT 位. 于是我是这么写的

接下来该继续初始化的其它命令了。按文档中的介绍，需要告诉SD卡它操作的电压，不满足的不能继续（呃，难道不是已经给电了才能初始化么），好吧，需要 SEND_OP_COND (CMD1)

我发送了 CMD1 却没有任何响应，搜了下网上资料，这条命令是MMC卡支持的，SD卡不支持…… (汗，ST也写该注明一下吧) 根据我下载的 "SD Physical Layer Spec 2.0" 文档，应该使用 CMD8:

于是我发送CMD8, 设置电压为2.7~3.6V档，SD卡依然没有反应。

哪里出问题了？命令返回的状态是 timeout, 我用简易示波器看了的确一直没有应答。CMD0 没有起作用？上电再多延时，还是不行。CMD0 和 CMD8 之间延时，也不行。PCB做错了？ 又核对一次的确CMD, CLK都没连错啊。从命令表里找了条别的命令来试，也没有响应。换个卡吧，这个实验用的是32MB的，换2GB的，以及4GB的SDHC卡，都还是没有应答……

搜网上的初始化代码参考吧（然后我发现——天下代码一大抄啊）。发现一个问题是，CMD8 是 SD规范2.0版本才支持的，我的卡可能根本不支持呢。根据CMD8是否响应，判断是否是2.0的卡。什么是2.0？又补了补知识，原来SDHC是2.0以后的，SDXC是3.0以后的了。我只知道SD卡非HC的最大只支持到2GB.

好吧，我找到一个SD 1.10规范的文档了。心急吃不了热豆腐。

哦，1.0是说发送ACMD41, 也就是要先发一个 CMD55. 终于，我测试CMD55时，32M的SD卡发出了应答。硬件是没问题的

接下来发送 ACMD41, 却发现返回了CRC错误！ 哦，原来这个命令特殊，应答类型没有CRC. 补充说一下，SD命令的应答不尽相同，主要有长应答和短应答两类，预期是长还是短应答需要在SDIO_CMD寄存器中设好，在RESPx寄存器中读取返回的应答数据。

ACMD41 命令后，应该是CMD2 ALL_SEND_CID命令了。这时又出现了问题，又不响应了。

该仔细读文档的。ACMD41 命令会返回 busy 状态，需要等待 busy 结束后才能继续。

注意红字，所以我套一个循环，要判断下若busy则继续循环。结果呢，没有发生busy啊，一次就Pass了。

还会有哪个步骤出了问题呢？还得读文档，不能着急。我逐渐意识到，SD卡的初始化是一个命令接一个命令相扣的，内部状态不同，能接收的命令也不同，必须一道一道门进入，如这个图：

然后，里我发现了一个乌龙：居然和上面的话矛盾，BUSY位是该1还是该0表示忙呢？

若我把busy位的判断改成0继续循环，就好了。重试若干次ACMD41后终于Pass. 在关于OCR寄存器的部分，文档是这么说的

这和我的实践结果一致。哎，这个文档里面自相矛盾太害人了。我找的这个文档是山寨的么？？？

接下来还比较顺利，几个命令后进入到了Transfer state, 可以读卡里面的块数据了，并可以设置4-bit模式。

读一个Block数据，直接用SD命令17, 给一个32位地址就行。需要注意的是，如果是SDHC卡，这个地址就不是字节地址而是块地址了。涉及到的SDIO寄存器有 DTIMER(控制超时), DLEN (传输数量), DCTRL (数据传输使能等), FIFO (获取读来的数据)。STM32 SDIO负责数据的部分也有一个状态机(Data Path State Machine)

不过，两个状态机都用同一个状态寄存器STA（不同的位）. 我的读程序目前是这么写的

这个子程序设好传输超时和最大长度，在发送完读命令之后，才把DPSM打开（过早打开了会出错）。然后等待FIFO中有数时，读FIFO寄存器(一次4字节)写到自己的buffer里，直到数据块传输成功的状态，如果出现异常就终止传输。这么查询式读数据效率不高，因为现在程序也不干别的就罢了。将来做播放器我肯定会用DMA来传数据的。

折腾中出现的插曲还有：32M的SD卡设置时钟到12MHz也正常读，但我另一张2GB的卡最多只能设到1MHz，再快就timeout了。后来用示波器看，发现读命令应答后到出现数据这段等待的时间很长，导致了超时。DTIMER寄存器是按clock个数来定的，SD时钟越快，超时就越短了。我这个2GB的卡，第一次读等待的时间长，后面连续读等待就缩短了。
