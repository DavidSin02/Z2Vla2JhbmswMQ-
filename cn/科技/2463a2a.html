上面的解决方法可以理解为一个常见的模式，装饰模式。

装饰模式（Decorator Pattern）：动态的给一个对象额外的职责。就增加功能来说，装饰模式相比生成子类更为灵活。装饰模式是对继承的补充。

先看看装饰模式的通用类图：

装饰模式的通用类图

其中的角色说明：

  • Component抽象构件，通常为一个接口或者抽象类，是核心对象，也就是最原始的对象。
  • ConcreteComponent具体的构建，是核心接口或者抽象类的具体实现。
  • Decorator抽象装饰角色，一般为一个抽象类，里面有个变量指向Component。
  • ConcreteDecorator具体的装饰角色，要把Component装饰成其他的。

来看下代码，仔细看代码，会发现很多有意思的东西。

package decorator;public class Client {    public static void main(String[] args) {        Component component = new ConcreteComponent();        component = new ConcreteDecorator1(component);        component = new ConcreteDecorator2(component);        component.operate();    }}/** * 抽象核心构件Component */interface Component {    void operate();}/** * 具体的核心构件 */class ConcreteComponent implements Component {    public void operate() {        System.out.println("我是核心类");    }}/** * 抽象装饰着 */abstract class Decorator implements Component {    private Component component;    public Decorator(Component component) {        this.component = component;    }    //委托给被装饰者执行    public void operate() {        component.operate();    }}/** * 具体的装饰角色 */class ConcreteDecorator1 extends Decorator {    public ConcreteDecorator1(Component component) {        super(component);    }    @Override    public void operate() {        //调用装饰方法        doSomething();        super.operate();    }    /**     * 装饰方法     */    private void doSomething() {        System.out.println("ConcreteDecorator1 我来美化你");    }}/** * 具体的装饰角色 */class ConcreteDecorator2 extends Decorator {    public ConcreteDecorator2(Component component) {        super(component);    }    @Override    public void operate() {        //调用装饰方法        doSomething();        super.operate();    }    /**     * 装饰方法     */    private void doSomething() {        System.out.println("ConcreteDecorator2 我来美化你");    }}

输出结果如下，可见在核心逻辑执行前，动态的添加了一下额外的功能。

ConcreteDecorator2 我来美化你ConcreteDecorator1 我来美化你我是核心类

装饰模式的优缺点：

1、优点：松耦合，装饰和被装饰者独立发展，是继承关系的补充，可以动态的添加功能。

2、缺点：多层的装饰增加复杂性。
