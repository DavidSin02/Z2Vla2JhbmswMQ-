基于PG实验理解3种事务隔离级别--读已提交、可重复读、串行化

2020-10-29
| 科技

概述

昨天分享过PG的3个事务隔离级别，有些朋友可能理解的不是很透彻，所以简单弄了一个实验来帮助大家理解下。

实验--测试隔离级别

postgres=# \h beginCommand: BEGINDescription: start a transaction blockSyntax:BEGIN [ WORK | TRANSACTION ] [ transaction_mode [, ...] ]where transaction_mode is one of: ISOLATION LEVEL { SERIALIZABLE | REPEATABLE READ | READ COMMITTED | READ UNCOMMITTED } READ WRITE | READ ONLY [ NOT ] DEFERRABLE

1、读未提交

表示可以读到其他会话未提交的数据（postgresql不支持）。

2、读已提交

表示可以读到其他会话已提交的数据。

2.1、创建一张表为test，插入一条记录

postgres=# create table test(a int, b varchar(128));CREATE TABLEpostgres=# insert into test values(1, 'hello');INSERT 0 1

2.2、在会话1中打开事务进行查询

postgres=# begin;BEGINpostgres=# select * from test; a | b ---+------- 1 | hello(1 row)

2.3、在会话2中打开事务进行更新

postgres=# begin;BEGINpostgres=# update test set b='hwb';UPDATE 1

2.4、查询会话1

此时在会话2中还没有关闭事务，在会话1中进行查询

postgres=# select * from test;

2.5、会话2提交

发现会话1中的记录并没有进行改变。当提交会话2中的事务，在会话1中进行查询值已经改变

--会话2postgres=# commit;--会话1postgres=# select * from test; a | b ---+----- 1 | hwb(1 row)

3、可重复读

表示在一个事务中，执行同一条SQL，读到的是同样的数据（即使被读的数据可能已经被其他会话修改并提交）。

3.1、设置事务级别

在会话1中打开可重复读事务，进行查询

postgres=# begin transaction isolation level repeatable read;postgres=# select * from test;

3.2、在会话2中进行更新操作

先在会话2更新不提交，然后在会话1中进行查询，发现会话1中的记录没有因为会话2的提交而变化

--会话2postgres=# update test set b='hwb2';--会话1postgres=# select * from test; a | b ---+----- 1 | hwb(1 row)

3.3、会话1提交

在会话1中进行提交，再查询，发现会话1中的记录变化了

postgres=# commit;postgres=# select * from test;

4、串行化

表示并行事务模拟串行执行，违反串行执行规则的事务，将回滚。

4.1、设置事务级别

--在会话 1中打开事务串行postgres=# begin transaction isolation level serializable;--在会话2中打开事务串行postgres=# begin transaction isolation level serializable;

4.2、在会话1中进行插入操作

postgres=# insert into test select * from test; postgres=# select * from test;

4.3、在会话2中进行插入操作

postgres=# insert into test select * from test; postgres=# select * from test;

4.4、提交会话1中的事务，能够正常提交，会话1进行查询能够查到插入的记录，会话2查询不到

postgres=# end;postgres=# select * from test;

4.5、当提交会话2的事务时会报错

postgres=# end;
