mysql explain 详解

2020-11-14
| 科技

mysql explain 详解：

使用：explain select * from table_name；可以用 desc 代替 explain；

  • id 列

若没有子查询和联合查询，id则都是1。

Mysql会按照id从大到小的顺序执行query，在id相同的情况下，则从上到下执行。

  • select_type 列

（1）SIMPLE：

简单的SELECT语句（不包括UNION操作或子查询操作）

explain select * from auct_item where item_id=9920140904244;

（2）PRIMARY/UNION：

PRIMARY：查询中最外层的SELECT（如两表做UNION或者存在子查询的外层的表操作为PRIMARY，内层的操作为UNION）

UNION：UNION操作中，查询中处于内层的SELECT（内层的SELECT语句与外层的SELECT语句没有依赖关系）

explain select * from auct_item where item_id=(select MAX(item_id) from auct_item);

（3）DEPENDENT UNION/UNIOIN RESULT：

DEPENDENT UNION：UNION操作中，查询中处于内层的SELECT（内层的SELECT语句与外层的SELECT语句有依赖关系）

UNION RESULT：UNION操作的结果，id值通常为NULL

explain select * from auct_item a where item_id=9920140904244 UNION select * from auct_item b where item_id=9920140904242;

（4）SUBQUERY/DEPENDENT SUBQUERY：

SUBQUERY：子查询中首个SELECT（如果有多个子查询存在），示例参考（2）

DEPENDENT SUBQUERY：子查询中首个SELECT，但依赖于外层的表（如果有多个子查询存在）

explain select * from auct_item a where item_id=(select MAX(item_id) from auct_item b where a.item_id=b.item_id);

（5）DERIVED/MATERIALIZED：

DERIVED：被驱动的SELECT子查询（子查询位于FROM子句）

explain select * from (select * from auct_item where item_id>0) a where item_id=(select MAX(item_id) from auct_item b where a.item_id=b.item_id);

MATERIALIZED：被物化的子查询

（6）UNCACHEABLE SUBQUERY/UNCACHEABLE UNION：

UNCACHEABLE SUBQUERY：对于外层的主表，子查询不可被物化，每次都需要计算（耗时操作）

UNCACHEABLE UNION：UNION操作中，内层的不可被物化的子查询（类似于UNCACHEABLE SUBQUERY）

  • table 列

输出的行所引用的表。

有时看到的是<derivedN>，其中N对应的是id列的值。

  • type 列

这列很重要，显示了连接使用了哪种类别，有无使用索引。

包括（由左至右，由最差到最好）：

| All | index | range | ref | eq_ref | const,system | null |

（1）ALL

全表扫描，MySQL 从头到尾扫描整张表查找行。

explain select item_id,item_name from auct_item ;

（2）index

该联接类型与ALL相同，除了只有索引树被扫描。这通常比ALL快，因为索引文件通常比数据文件小。　当查询只使用作为单索引一部分的列时，MySQL可以使用该联接类型。

explain select item_id from auct_item ;（item_id为主键）

（3）range

只检索给定范围的行，使用一个索引来选择行。key列显示使用了哪个索引。key_len包含所使用索引的最长关键元素。在该类型中ref列为NULL。　当使用=、<>、>、>=、<、<=、IS NULL、<=>、BETWEEN或者IN操作符，用常量比较关键字列时，可以使用range。

explain select item_id from auct_item where item_id in (123,1234);

（4）ref

非唯一性索引访问(换句话说，就是无法根据该值只取得一条记录)

explain select * from auct_item_record where item_id=9920140904242;

（5）eq_ref

对于每个来自于前面的表的行组合，从该表中读取一行。这可能是最好的联接类型，除了const类型。它用在一个索引的所有部分被联接使用并且索引是UNIQUE或PRIMARY KEY。　eq_ref可以用于使用= 操作符比较的带索引的列。比较值可以为常量或一个使用在该表前面所读取的表的列的表达式。

explain select a.AUCT_ID,a.ITEM_ID,b.ITEM_NAME,a.USER_ID from auct_item_record a,auct_item b where a.USER_ID=6000063536 and a.ITEM_ID=b.ITEM_ID;

（6）const

表最多有一个匹配行，它将在查询开始时被读取。因为仅有一行，在这行的列值可被优化器剩余部分认为是常数。const表很快，因为它们只读取一次！　const用于用常数值比较PRIMARY KEY或UNIQUE索引的所有部分时。

explain select * from auct_item_record where record_id=4448;

（7）system

这是const联接类型的一个特例。表仅有一行满足条件.如下(id是primary key)

EXPLAIN SELECT * FROM (SELECT * FROM auct_item_record WHERE record_id=4448) a;

（8）null

在优化过程中就已得到结果，不用再访问表或索引。

explain select min(record_id) from auct_item_record;

  • possible_keys 列

possible_keys列指出MySQL能使用哪个索引在该表中找到行。

注意，该列完全独立于EXPLAIN输出所示的表的次序。这意味着在possible_keys中的某些键实际上不能按生成的表次序使用。　如果该列是NULL，则没有相关的索引。

  • key 列

key列显示MySQL实际决定使用的键（索引）。

如果没有选择索引，键是NULL。要想强制MySQL使用或忽视possible_keys列中的索引，在查询中使用FORCE INDEX、USE INDEX或者IGNORE INDEX。

  • key_len 列

key_len列显示MySQL决定使用的键长度。

如果键是NULL，则长度为NULL。　使用的索引的长度。在不损失精确性的情况下，长度越短越好。

  • ref 列

ref列显示使用哪个列或常数与key一起从表中选择行。

常见的值有 const, func, NULL, 具体字段名。当 key 列为 NULL ，即不使用索引时，此值也相应的为 NULL 。

ref 为 func 的情况出现在子查询中:

explain select * from auct_item_record where record_id in (select record_id from auct_item_record);

  • rows 列

rows列显示MySQL认为它执行查询时必须检查的行数。

  • Extra 列

显示上述信息之外的其它信息，但却很重要。

常见的值有Using index、Using where、Using temporary、Using filesort。

（1）Using index

此查询使用了覆盖索引(Covering Index)，即通过索引就能返回结果，无需访问表。

explain select record_id from auct_item_record;

（2）Using where

表示 MySQL 服务器从存储引擎收到行后再进行“后过滤”（Post-filter）。所谓“后过滤”，就是先读取整行数据，再检查此行是否符合 where 句的条件，符合就留下，不符合便丢弃。因为检查是在读取行后才进行的，所以称为“后过滤”。并非带 WHERE 子句就会显示"Using where"的。

explain select * from auct_item_record WHERE USER_ID=6000063536;

（3）Using temporary

看到这个的时候，查询需要优化了。这里，MYSQL需要创建一个临时表来存储结果，这通常发生在对不同的列集进行ORDER BY上，而不是GROUP BY上。

explain select distinct NICKNAME from auct_item_record;

MySQL 使用临时表来实现 distinct 操作。

（4）Using filesort

看到这个的时候，查询就需要优化了。

MYSQL需要进行额外的步骤来发现如何对返回的行排序。它根据连接类型以及存储排序键值和匹配条件的全部行的行指针来排序全部行。

  1. 对于没有索引的表，只要 order by 必会出现 Using filesort 。
  2. 当 type 为 rang、 ref 或者 index 的时候才有可能利用索引排序，其它，如 ALL ，都无法通过索引排序，此时若有 order by ，如上例，便会出现 Using filesort 。

explain select * from auct_item_record order by AUCT_ID DESC;
