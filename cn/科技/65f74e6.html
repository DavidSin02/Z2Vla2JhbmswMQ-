虚拟内存系列(一)硬件基本结构

2020-10-29
| 科技

0

最近花了很多时间在虚拟内存的研究，发现自己对虚拟内存的理解只是停留在页，页表，页框等等这些基本概念上。举例来说，我一直认为虚拟地址的分段数决定了查表次数。其实仔细想想，哪怕是一级页表映射，也必然有两个地址分段。本章节，我将对ARM的虚拟内存的映射部分做一个系统性的总结。

PAGE的故事

ARM提供了一级和两级地址映射。总的来说，可以分为两类：

  • Section 方式映射
  • PAGE 方式映射

页大小可以为1MB(sections), 64KB, 4KB, 1KB。在linux中常用的是4KB的页大小，它是由宏PAGE_SHIFT来决定的。有一点需要注意，如果PAGE大小为1M时，使用的是section方式映射，此时只有一级页表(PGD)。 那么ARM具体是知道使用哪种方式呢，这是由PGD descriptor中的最后2bit来决定。

  • 00 : 未定义的
  • 01 : Coarse page table, 可使用PAGE大小: 64K, 16K
  • 10 : Section, 可使用PAGE大小: 1M
  • 11 : Fine page table: 可使用PAGE大小: 64K, 16K, 4K

ARM 地址映射过程

下面这张图展示了ARM使用不同方式映射的地址查找过程。

程序首先会通过TTBR寄存器找到一级映射表的入口，取虚拟地址的[31:20] bit作为这张表的Index，找到对应的PGD descriptor。取PGD中的二级页表物理地址，将虚拟地址的[19:12/10]作为二级页表的索引，找到PTE descriptor。 PTE记录了页的物理地址(4K/1K对齐)。

虚拟地址中的[31:20]bit是作为TTBR的Index，也就是Translation table base 后的[13:0]bit 输入，拼成一个32bit的地址，最后定位到PGD descriptor的位置。可以看到一级页表是16KB对齐的。这是由于ARM为一级页表提供了4096个entry(使用虚拟地址的高12bit用来做Index)，每个entry为4字节，总共16K字节。

二级页表基址本结构如下，对应了三种映射方式。记录了下一级页/页表的基址以及Domain和页权限(section)，二级页表基址是一级页表的context。

Section base address记录了Section方式的页物理地址。对于其他两种映射方式，PGD所记录的是二级页表的base地址。可以看到，二级页表基址(我们以Coarse page table base为例)它记录的地址是[31:10]bit，而[9:0]是没有的。那是不是意味着这种方式的二级页表是1KB对齐的呢？确实是这样的。因为ARM中为Coarese page table提供的了256个entry，按每个entry4字节计算，每张表占用1KB内存。所以，对于Coarese page table base而言，它一定是1KB对齐的。对于Fine page table，ARM提供了1024个entry，所以它是4K对齐。

我在看这部分内容时，特别容易混淆地址、地址上存的内容这两个概念。习惯性地用页表的base address 加上虚拟内存中的PAGE_OFFSET来计算一个地址。例如Coarse page table baseaddress是[31:10]共22bit，然后取虚拟地址中的[9:0]这10bit组成一个地址。当然，这是一种错误的计算方法。

小结

1，对于映射部分的理解，不要局限于4K page。

2，页权限控制bit的具体位置。

3，理解一级、二级页表的基址(base address的来源)，以及页表基址是上一级页表内容换算。

4，虚拟内存分段后的各bit段用作页表/页的index(这比理解为物理地址更确切一点)。
