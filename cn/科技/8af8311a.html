5G 网元如何瞬时处理海量数据报文（2）

2020-11-14
| 科技

之前的文章《5G 网络设备中的软件如何做到瞬时处理海量数据报文（1）》提到，要想在最短的时间内处理掉尽可能多的数据报文，必须借助多线程技术。于是，说到了线程（Thread）、队列（Queue）和任务（Task）的部署关系。这里我们继续这一话题，并初步探究下5G 网元的工作原理。

并行多任务下的线程、队列部署关系

在 5G 网元中，如果只有一个任务，那么这个任务必然非常庞大，且不好管理。于是，我们需要将一个总的任务划分成很多小的子任务，于是就出现了上图的部署方式。

顺序多任务下的线程、队列部署关系

在 5G 网元中，一个总任务的执行往往需要分为多个阶段（Phase）。于是，我们在阶段1（Phase 1）的时候执行 Task 1，在阶段二（Phase 2）的时候执行 Task 2，那么 Task 1 和 Task 2 就是顺序的关系。也就是说，报文经过 Task 1处理完，需要扔到 PacketInQueue 2 里，然后 Task 2 从队列 PacketInQueue 2 里取报文，执行 Task 2，最后再从出口发送出去。

报文处理状态机

在 5G 网元中，对报文的处理一般都采用状态机模式，每个状态对应一个处理动作，状态之间的跳转需要满足特定的条件。上图中，状态 1 可以跳转到状态 2 ，触发条件是C12. 状态 1 也可以跳转到状态 3，触发条件是 C13.

while (1) {

switch (条件)

{

case 条件 1： 动作 1

case 条件 2： 动作 2

┇

case 条件 n： 动作 n

default: 动作 n+1

}

}

比如，

报文1进入 5G 网元后的处理路线为：初始状态 0 -> 状态 1 -> 状态 2 -> 状态 3；

报文 2 进入 5G 网元后的处理路线为：初始状态 0 -> 状态 2 -> 状态 3.

这怎么理解呢？举个例子，如果你访问某个黄色小网站，5G 网元识别出来之后，肯定将你的报文阻塞掉。如果你访问腾讯系 APP，而且你还买了腾讯大王卡，那么你的报文将畅通无阻，电信运营商也不会对你的流量计费。

虽然说 5G 网元内部基本处理逻辑是状态机模式，但是，可以想象的是：5G 网元系统内部很复杂，涉及的状态非常多，而且涉及的条件也非常复杂，如果只是采用简单的状态机模式，显然无法管理，很容易失控。那么怎么管理这些状态和跳转条件呢？借助 Go 语言中事务（Transaction）和 事件（Event）的概念 会是一个不错的解决方案。详细内容且听下回分解。
