CrashScope：一种实用的安卓应用自动化测试工具

2020-11-14
| 科技

摘要

由于移动应用程序具有普遍的事件驱动特性和复杂的上下文功能（例如传感器，通知）的原因，在测试移动应用程序时会遇到特别的挑战。由于所需的工具或平台依赖性，当前用于安卓应用程序的自动输入生成方法通常不适合开发人员使用，并且通常无法有效地行使上下文功能。为了在开发人员的移动测试任务中更好地支持开发人员，在本项目中，我们介绍了一种新颖的自动化工具 CRASHSCOPE。该工具根据静态和动态分析提供的几种策略，使用系统化的输入生成功能来探索给定的安卓应用，其内在目标是触发崩溃。当检测到崩溃时，CRASHSCOPE 会生成一个增强的崩溃报告，其中包含屏幕截图，详细的崩溃复现步骤，捕获的异常堆栈跟踪信息以及可完全重播的脚本，该脚本可在目标设备上自动复现崩溃。初步研究的结果表明，CRASHSCOPE 能够发现一样多的其他最新技术工具发现的崩溃，同时为开发人员提供详细的有用崩溃报告和测试脚本。

I 简介

用户越来越喜欢在智能手机或平板电脑的便捷环境中执行计算任务，这样的用户群推动了移动硬件和应用程序市场的快速发展。但是，移动应用程序的手势驱动特性已经引发了程序员在开发和维护期间，特别是在测试和调试方面的新挑战。错误报告的创建和解决是最困难、最重要的维护任务之一。关于应用程序崩溃的报告对于开发人员而言尤其重要，因为崩溃表示严重的错误，用户将直面崩溃错误且该错误会立即影响应用程序的可用性。如果某个应用因崩溃，功能缺失或其他错误而无法正常运行，则近一半的用户可能会在当下市场竞争的环境下在 Google Play 等市场中放弃该应用。因此，为了确保其应用程序的正常运行，开发人员需要自动化的支持措施来发现，复现并修复崩溃。

尽管在移动应用测试和自动生成输入方面已经取得了显著进展，可用的工具却通常存在一些值得注意的局限性：（i）在自动输入生成过程中，先前的方法缺乏为发现的故障提供详细，易于理解的测试结果的能力，开发人员需要整理和理解堆栈跟踪信息、日志文件以及非表达性事件序列；（ii）大多数自动生成输入的方法对于开发人员都不实用，通常是由于仪器或安装程序困难。这也说明了为什么开发人员通常更喜欢手动而不是自动测试方法。（iii）没有一种方法可以在单一整体方法中结合使用不同的 GUI 开发策略，上下文功能的目标测试以及用于用户文本输入的多种策略。这些缺点导致自动测试方法的采用率低，同时让基于手动和脚本的测试技术得到偏爱。

通过了解开发人员在工作流程中采用自动化测试工具时面临的上述问题，为了解决这些难题，我们设计并实现了 CRASHSCOPE。CRASHSCOPE 是一种实用的系统，它可以自动发现，报告和复现安卓应用程序的崩溃过程。 CRASHSCOPE 根据一组输入生成策略来探索给定的应用程序，并生成具有明确步骤的表达性崩溃报告，并以易于阅读的自然语言格式进行报告生成。这种方法仅需要一个.apk 文件（以及可选的源代码）和一个 Android 仿真器或设备即可运行，而不依赖主题应用程序或 Android OS 的检测。 CRASHSCOPE 的整个工作流程都是自动化的，除了阅读生成的报告外，无需开发人员干预。我们工具支持的不同探索策略旨在引起 Android 应用程序崩溃，幷包括基于文本输入字段允许字符的上下文自动生成文本的功能，以及针对上下文特征进行测试（例如设备的方向，网络的接口和传感器）。我们专门定制了这些功能，以测试先前研究，所确定的应用程序崩溃的常见原因。在执行期间，CRASHSCOPE 捕获有关主题应用程序的详细信息，例如输入，屏幕截图，GUI 信息，异常和崩溃。然后，此信息将转换为详细的崩溃报告和可重播的脚本，从而大大减轻了开发人员修复故障的负担。

II CRASHSCOPE 自动化测试工具

CRASHSCOPE 通过几种方式解决了现有工具的一般局限性，这些方式说明了该方法的新颖性：（i）CRASHSCOPE 能够自动生成表达性的错误报告（和可重播的脚本），用于发现异常或崩溃；（ii）这种方法很实用，只需要一个.apk 文件即可运行；（iii）探索策略能够对上下文特征和自适应文本输入生成进行有针对性的测试；以及（iv）该工具具有抗崩溃的应用程序功能；它可以检测到崩溃，并在处理崩溃后继续测试 GUI 中未访问的组件和状态。 CRASHSCOPE 的目标用户群是希望将实用的自动化测试工具纳入其工作流程的移动应用程序开发人员和测试人员。为了使该工具易于使用并有效地集成到典型的开发人员的工作流程中，我们开发了一种基于云的 Java Web 应用程序，其中封装了 CRASHSCOPE 工具。该 Web 应用程序包括一个执行引擎后端，该后端会执行测试要求的静态和动态分析，以及易于提交应用程序进行测试的用户友好前端。这种体系结构可确保 CRASHSCOPE 作为基于 Web 的工具的灵活性，可伸缩性和可扩展性。

A． CrashScope Design

CRASHSCOPE 的总体体系结构如图 1 所示。该工具工作流程的第一步是使用 Web 应用程序的 New Task 表单上传应用程序.apk 文件（以及可选的 src 代码），提交用于测试的应用程序。并选择 CRASHSCOPE 用于探索该应用程序的策略配置。一旦开发人员启动了任务，就可以在 Testing-Dashboard 中查看进度（图 2）。在测试仪表板页面上，开发人员可以查看提交的任务和当前正在运行的任务的执行进度（例如，图 2 的底部）。另外，可以查看统计信息，包括运行时间，应用信息和发现的崩溃数量，以了解已完成的任务（例如，图 2 的顶部）。启动任务后，Web 应用程序将通过 json 文档将包括.apk 文件在内的任务信息发送到 CouchDB 的实例中（充当消息队列）。后端执行引擎会定期轮询队列中的新任务，并在收到消息时启动测试过程。执行引擎在测试过程中执行的第一步是对提交的应用程序进行静态分析，以识别上下文功能（图 1-1）。为此，它要么利用用户上传的应用程序源代码（如果已提供），要么尝试使用多种工具来反编译该应用程序。然后，它将检测与上下文功能相关的活动（例如屏幕），以针对此类功能进行测试。换句话说，CRASHSCOPE 仅在识别应用程序代码中的功能实例时才会测试上下文应用程序功能（例如，网络开/关）（第 II-B 节）。

图 2 CRASHSCOPE web 应用实验仪表板

接下来，GUI 翻录引擎（图 1-2）使用用户指定的组合策略（第 II-E 节）系统地执行应用程序，包括在由静态分析识别的相关屏幕上启用和禁用上下文功能（如果在模拟器上运行）。 如果在执行过程中引发了未捕获的异常，或者应用程序崩溃，则在系统的探索过程中系统将动态执行信息保存到 CRASHSCOPE 的数据库（图 1-3），其中包括有关每个事件的详细信息（例如，触摸事件，文本输入，传感器值更改）。可以在大量并发运行的物理设备或使用 android-x86 映像和 VirtualBox 实例化的 Android 虚拟设备（AVD）上执行此过程。

将执行数据保存到 CRASHSCOPE 数据库之后，自然语言报告生成器（图 1-4，第 II-F 节）分析数据库并处理每个事件的信息，这些信息是以崩溃结束的执行序列。然后自然语言报告生成器将该信息以 json 形式发送给 CouchDB。 然后，Java Web 应用程序将为用户生成基于 HTML 的自然语言崩溃报告，并带有表达性再现步骤（图 1-6）。 此外，崩溃脚本生成器（图 1-5，第 II-G 节）分析数据库并提取崩溃执行中每个步骤的相关信息，以创建包含 adb 输入命令和上下文状态更改标记的可重播脚本。脚本重放器（图 1-7，第 II-G 节）是一个小型 Java 程序，能够重放这些脚本。

B． 提取活动和应用级别的上下文特征

CRASHSCOPE 使用基于抽象语法树（AST）的分析来提取与上下文功能的调用有关的 API 调用链。尤其是，它可以检测与网络连接和传感器（例如，加速度计，磁力计，温度传感器和 GPS）相关的 Android API 调用。由于 API 调用可能不会直接由与实现 Activity 的类相关联的代码直接执行（在 Android 中包括屏幕），因此 CRASHSCOPE 执行调用图分析以提取路径，该路径以调用上下文 API 的方法作结。由于某些 API 调用可能无法通过向后传播的调用链进行跟踪（例如，作为服务实现的传感器或网络连接），因此 CRASHSCOPE 使用两种粒度来测试上下文功能：活动（屏幕）级别和应用程序级别。如果 API 分析表明某个应用程序实现了特定的上下文功能，但是无法通过调用链分析将该功能链接回特定屏幕，则将在“应用程序级别”进行测试或在每个屏幕上测试该功能。通过分析应用程序的 AndroidManifest.xml 文件可以找到可旋转的屏幕。

C． 应用 & 崩溃检测的探索

为了探索应用程序，CRASHSCOPE 动态提取在探索过程中访问的每个应用程序屏幕的 GUI 层次结构（使用 uiautomator），并标识要执行的可单击和可长时间单击的组件，以及用于文本输入的可用组件（例如 EditText 框）。在每个屏幕上观察时，可（长时间）单击的组件被添加到动态组件堆栈中，并根据堆栈的顺序执行。当前，翻录引擎支持轻敲，长按和键入事件。如果在特定的应用程序屏幕中可以使用文本输入字段，则在执行屏幕上的每个可（长时间）单击组件之前，都要填写每个文本框。 此外，从一个独特活动到另一个独特活动的每个转换以及导致转换的事件都记录在转换图中。 组件和转换图使得 CRASHSCOPE 可以在执行期间构建应用程序的准确实时模型。

来自用户的文本输入是许多安卓应用程序功能的主要部分，因此，CRASHSCOPE 的 GUI 翻录引擎采用了独特的文本输入生成机制。CRASHSCOPE 通过使用 adb shell dumpsys input_method 命令查询与文本字段相关联的键盘类型，来检测文本字段期望的文本类型（例如，电话号码，电子邮件地址）。 一旦检测到预期输入的类型，CRASHSCOPE 将使用两种方法生成文本输入：预期输入和意外输入。预期的启发式会在键盘参数内生成一个字符串，而没有任何标点符号或特殊字符；意外的启发式会生成具有给定键盘类型的所有允许的特殊字符的随机字符串。

除了文本输入生成策略之外，CRASHSCOPE 还可以根据几种 GUI 探索策略之一来生成触摸输入。当前，该工具从层次结构的底部向上或从层次结构的顶部向下（通过将订单组件放置在组件堆栈中进行控制）以类似于深度的方式遍历 GUI 层次结构。具有两种这样策略的基本原理是模仿用户的操作（即在没有预定顺序的情况下执行 GUI 事件）。但是，CRASHSCOPE 可扩展到其他类型的探索方法，包括基于随机的方法或有偏随机的方法。为了检测和捕获异常，CRASHSCOPE 过滤了 logcat 以获取仅与被测试的应用程序相关的未捕获异常。为了检测崩溃，CRASHSCOPE 检查标准安卓崩溃对话框的外观。如果遇到崩溃，则将执行信息记录到数据库中。但是，由于 GUI 翻录引擎维护转换图和未访问组件堆栈的原因，执行可以继续操作剩余的程序路径。

D． 在不同的上下文环境中测试移动应用

当 GUI 翻录开始时，CRASHSCOPE 首先检查应根据探索策略进行测试的应用程序级别的上下文功能。每次访问活动时，GUI 翻录引擎都会检查是否应启用/禁用上下文功能，并根据当前策略设置功能值。上下文功能的测试仅在使用 telnet 命令的仿真器或 AVD 上有效。虽然 telnet 命令确实支持为仿真器打开/关闭网络，但它们不支持启用/禁用传感器（加速度计，磁力计，GPS，温度传感器），但是可以模拟这些传感器的值。 因此，为了在不利条件下测试与传感器相关的功能，请禁用网络连接，并为其他传感器（如 GPS 和陀螺仪）设置意外的（例如，高度不可行的）值。

E． 多种执行策略

CRASHSCOPE 最强大的功能之一就是能够通过结合各种可支持的测试功能，根据几种不同的策略来探索应用程序。这些策略来自三个主要的特征启发法：1）GUI 遍历策略（例如 dfs-top-down，dfs-bottomup 或基于随机的），2）生成用于用户文本输入字段的输入的方法（无文本，期望的文本，意外的文本），最后，3）启用或禁用对不利的上下文状态的测试（例如，如果发现某个活动已经利用了网络，则应将其打开还是关闭？）。这些策略的不同组合有可能发现不同类型的应用崩溃。在不同策略中通过这三个功能启发式的所有组合来运行应用程序，CRASHSCOPE 可以有效地测试常见诱导崩溃的不同类型。还可以通过在一组或一组仿真器实例云上同时运行一个应用程序的几种组合来并行化这些策略（例如使用执行引擎），从而进一步减少测试开销。

F． 生成表达性的自然语言崩溃报告

CRASHSCOPE 生成一个崩溃报告（图 1-6），其中包含四种主要信息：1）常规信息，包括应用程序名称和版本，Android OS 的版本，图标——指示应用在复现步骤中的当前上下文状态，设备以及崩溃发生时的屏幕方向和分辨率； 2）自然语言句子，使用有关每个步骤的 GUI 事件和上下文状态的详细信息来描述复现崩溃的步骤（图 3）； 3）应用程序的屏幕窗口，其中突出显示了针对特定崩溃情况在执行场景中与每个屏幕交互的组件； （4）删除堆栈跟踪，仅包含执行期间发生的应用程序异常。一旦执行引擎报告已完成提交任务的所有策略，它就会通过 json 向 CouchDB 发送一条消息。该消息由 Java Web App 检索，并且在崩溃报告页面中可以查看当前任务的结果和崩溃报告。

G． 生成&重放复制脚本

崩溃脚本生成器（图 1-5）从 CRASHSCOPE 数据库中解析保存执行信息，并生成可重播的脚本，其中包含用于触摸和文本输入的 adb 输入命令以及用于更改上下文状态的标记。 通过解析数据库以获取与特定执行中每个步骤相关的所有 GUI 事件的脚本。然后，系统会分析该应用的系统探索过程中记录的每个组件的座标，并根据每个组件的大小推断中心座标。这些座标用于生成 adb 输入命令以重现 GUI 事件。这种方法依赖于我们之前在安卓应用中重播测试序列事件的工作。脚本可通过脚本重播器（图 1-7）进行重播，脚本重播器是一个小型 Java 程序，它将记录的脚本作为输入，并重播 adb 命令和状态更改标记（例如“ 关闭 WiFi”）。

III 评估

为了评估 CrashScope，我们进行了两项实证研究，旨在调查该工具的崩溃发现能力以及所生成报告的可重复性和可读性。首先，在“崩溃发现研究”中，我们利用 61 种开源主题应用程序中的一部分主题应用程序和工具，将 CRASHSCOPE 与 4 种最先进的 Android 自动输入生成工具（以及标准的 Monkey 工具）进行了比较，该工具可用于 61 个开源主题应用程序 Androtest 测试套件。其次，在“崩溃报告研究”中，我们评估了 CRASHSCOPE 生成的自然语言报告与在线问题跟踪器中发现的人工书面报告相比的可重复性和可读性。在这项研究中，我们从与 F-Droid 开源市场上的应用程序链接的在线问题跟踪器中提取了 8 个真实的崩溃报告，CRASHSCOPE 能够为这些崩溃报告生成新报告。然后，有 16 名研究生参与者试图重现 8 个错误报告（其中 4 个来自 CRASHSCOPE，4 个来自在线问题跟踪器），随后回答了有关两种匿名错误报告的调查问题。这些研究的结果表明，（i）CRASHSCOPE 在发现崩溃方面的有效性与其他先进的 Android 自动输入生成工具相同，同时减少了由于使用其他工具而需要的工具导致的误报的数量，以及（ii）生成了报告 CRASHSCOPE 的报告与人类书面报告一样具有可重复性，并且从开发人员的角度来看更具可读性和实用性。

IV 项目备注与未来工作

在这个项目中，我们介绍了 CRASHSCOPE，这是针对 Android 应用程序的实用的自动化测试方法的新颖实现。我们的工具通过（i）利用静态分析以针对性的方式定位和测试上下文特征来克服以前方法的一些缺点；（ii）使用几种不同的策略组合来探索应用程序；（iii）生成非常有用的崩溃报告和可重播的测试方案脚本。将来，我们的目标是研究通过在基于模型的 GUI 测试和静态分析中采用有前途的新兴方法，将错误报告整理为仅有必要步骤的模式。同时我们将改善我们的系统探索策略技术。此外，我们旨在将 CRASHSCOPE 分发到更广泛的安卓开发社区，并评估其实用性。
