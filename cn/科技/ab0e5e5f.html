java concurrent包下，同步器你了解多少

2020-11-14
| 科技

同步器，这个概念陌生吗？可能很多人发没反应过来这是什么，这个玩意到底有什么作用呢，今天主要聊一聊它。

当我们遇到有相互作用的线程集满足了以下将要提到的模式时，那么我们可以直接使用合适的 库类，而不需要也不建议我们手工去实现一些锁与判别条件。

1、信号量（Semaphore）

之所以第一个提它，可能是因为这个在我们日常中相对最常用的一个了吧。通俗说，信号量就是管理许多的“通行证”，如果要通过信号量，则线程需要调用acquire请求才可得到许可。信号量仅仅只是维护一个计数，而“通行证”的数量是固定的，由此来限制通过的线程数量。而其他线程想要通过，则需要调用release释放许可（实际中，任何线程都可以释放任意数目的许可，带来的后果是可能会增加通行证的数目以至于超过原始设置的阈值，需要谨慎使用）。

总结，它能做什么：允许线程集等待直到被允许继续允许为止，达到限制访问资源的线程总数的目的。假设允许数目设为1，那么会阻塞线程直到另一个线程去释放许可为止。

2、倒计时门栓（CountDownLatch）

这个也比较常用，它让一个线程集等待直到计数变成0为止。值得一提的是，它是一次性的，一旦计数为0，就不能再重用了。

举个例子，假设有两个线程A、B、C，A和B要完成最终的任务，但是需要一些前置数据，而C能完成A和B所需要的数据，这时，A和B就会在启动后一直等待（就像是被栓在门外了），当C的数据准备完成时，调用了countDown，此时，A和B就可以拿着C的数据继续工作了。

总结，它能做什么：允许线程集等待直到计数器减至0。

3、障栅（CyclicBarrier）

这个相对前两个相对陌生一些了吧，解释一下，障栅就是CyclicBarrier类实现了一个集结点。假设这样一个场景，有大量线程运行在一次计算的不同部分时，当所有部分都准备好了，需要把结果组合在一起。在这过程中，一个线程完成了他的那部分任务后，我们让它运行到“障栅”处，一旦所有线程都到达了这个障栅，障栅就撤销了，线程就可以继续运行了。

用伪码简单描述下：

//先构造一个障栅CyclicBarrier cb = new CyclicBarrier(threads);//每个线程可以做一些自己的事，完成后再障栅上调用awaitpublic void run(){ doSomething(); //await方法可以设置超时参数，如cb.await(1000,TimeUnit.MILLISECOND); cb.await();}

如果任何一个在障栅上等待的线程离开了障栅，则障栅就被破坏了。比如线程可能执行时中断，或者在调用await方法超时了。此时那些已经在等待的线程将会立即终止await的调用。当我们在构造障栅时，还可以提供一个障栅动作，当所有线程到达障栅时就会执行这个动作。不如：

//动作action可以收集那些thread的运行结果Runnable action = ....;CyclicBarrier cb = new CyclicBarrier(threads,action);

值得注意的CyclicBarrier是循环的，即在所有等待线程被释放后它是可以被重用的，这一点与CountDownLatch不同，大家可以对比掌握。

总结，它能做什么：允许线程集等待直至其中预定数目的线程到达一个公共障栅，然后可以选择执行一个处理障栅的动作。

4、交换器（Exchanger）

这个简单说明下，当两个线程在同一个数据缓冲区的两个实例上工作时，就可以使用交换器了。比如：一个线程A向缓冲区填充数据，另一个线程B消耗这些数据，当A、B都完成后，互相交换缓冲区。

总结，它能做什么：允许两个线程在要交换的对象准备好时交换对象。

5、同步队列（SynchronousQueue）

同步队列就是一种将生产者与消费者线程配对的机制。当一个线程调用SynchronousQueue的put方法时，它会阻塞直到另一个线程调用take方法为止，反之亦然。

注意SynchronousQueue类实现了BlockingQueue接口，它没有包含任何元素，因此调用它的size方法时始终会返回0。

总结，它能做什么：允许一个线程把对象交给另一个线程，与Exchanger不同，数据仅沿着一个方向传递，不像Exchanger是双向的。
