什么是事务（transaction）？

事务是指不可分割的逻辑工作单元，完全执行，或者完全不执行。该逻辑工作单元可能由多条SQL处理语句组合而成。

事务的特点：

  1. 同生共死。
  2. 有可能对表数据进行添加或删除操作，但是对查询无用。
  3. 实际上是一个处理机制，用以维护数据库的完整性。

事务的4大属性（ACID特性）：

1.原子性（Atomic）（Atomicity)：

事务必须是一个自动工作的单元，要么全部执行，要么全部不执行。原子性消除了系统处理操作子集的可能性。

2.一致性（Consistent）(Consistency)：

事务结束的时候，所有的内部数据都是正确的。

3.隔离性（Insulation）(Isolation)：

并发多个事务时，各个事务不干涉内部数据，处理的都是另外一个事务处理之前或之后的数据。由并发事务所作的修改必须与任何其它并发事务所作的修改隔离。

由于高度隔离会限制可并行执行的事务数，所以一些应用程序降低隔离级别以换取更大的吞吐量。

4.持久性（Duration）(Durability）：

事务提交之后，数据是永久性的，不可再回滚。该修改即使出现致命的系统故障也将一直保持。

事务的语法：

1.事务起点-begin transaction

begin tran[saction] [<transaction name>|<@transaction variable>][with mark[<'description'>]]

注：

表示一个事务单元的开始，也是回滚的起点。如果没有提交，那么回滚就从这里开始。

2.事务提交- commit transaction

commit tran[saction][<transaction name>|<@transaction variable>]

注：

表示事务的终点，事务单元语句顺利执行，提交表示这个事务正常进行。

3.事务回滚-rollback transaction

rollback tran[saction][<transaction name>|<save point name>|<@transaction variable>|<@savepoint variable>]

注：

表示事务回滚，不想或者不能提交，回滚从begin开始的所有语句，类似撤销。

4.事务保存-save transaction

save tran[saction][<save point name>|<@savepoint variable>]

注：

事务保存即创建一个标记或书签（bookMark）。在回滚中引用这个标记，即可回滚到代码中的特定点上。类似电脑系统恢复时，设置的定点恢复位置。

事务的3大处理模型：

1.隐式事务

--***************模式1*******************set implicit_transactions oncommit transaction--****************模式2******************set implicit_transactions onrollback transaction

隐式事务是指每一条数据操作语句都自动地成为一个事务，事务的开始是隐式的，事务的结束有明确的标记。

2.显式事务

显式事务是指有显式的开始和结束标记的事务，每个事务都有显式的开始和结束标记。

--***********显式模式1*****************begin transaction commit transaction--**********显示模式2*******************begin transaction rollback transaction

3.自动事务

自动事务是系统自动默认的，开始和结束不用标记。即每条Sql语句都被看成一个事务进行处理。

显式事务3大模式

1.事务开始-事务提交

--begin transaction --.... --commit transaction

2.事务开始-事务回滚

--begin transaction --rollback transaction

3.事务开始--事务标记--定点回滚

--begin transaction --save transaction trac--rollback transaction trac

并发控制简述

1． 数据库系统一个明显的特点是多个用户共享数据库资源，尤其是多个用户可以同时存取相同数据。

串行控制：如果事务是顺序执行的，即一个事务完成之后，再开始另一个事务

并行控制：如果DBMS可以同时接受多个事务，并且这些事务在时间上可以重叠执行。

2.事务是并发控制的基本单位，保证事务ACID的特性是事务处理的重要任务，而并发操作有可能会破坏其ACID特性。

DBMS并发控制机制的责任：

  1. 对并发操作进行正确调度，保证事务的隔离性更一般，确保数据库的一致性。
  2. 如果没有锁定且多个用户同时访问一个数据库，则当他们的事务同时使用相同的数据时可能会发生问题。

由于并发操作带来的数据不一致性包括：

  1. 丢失数据修改
  2. 读”脏”数据（脏读）
  3. 不可重复读
  4. 产生幽灵数据

注：

一个逻辑工作单元要成为事务，必须满足所谓的ACID（原子性、一致性、隔离性和持久性）属性。事务是数据库运行中的逻辑工作单位，由DBMS中的事务管理子系统负责事务的处理。
