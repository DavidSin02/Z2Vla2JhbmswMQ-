HashMap容量为什么要是2的幂次方？看完你就明白了！

2020-10-29
| 科技

相信看过我前面几篇文章的小伙伴们一定发现了，那就是HashMap的容量都为2的幂次方，每次扩容也是扩大一倍，那么为啥要保持容量为2的幂次方呢？又是如何保证的呢？让我们来一起揭秘吧！

首先，我们来看下第一个问题，为什么容量总是为2的n次幂？这到底是作者深思熟虑之后的考量，还是作者随手的设定呢？

看过源码的小伙伴们一定知道HashMap计算节点存储位置下标是通过 (n-1)&hash 来计算的，这里的n就是HashMap的容量。因为n始终为2的幂次方，那么n-1始终能保证后几位都是1(例如：n=16,那么n-1:0000 ... 1111)。那么当 (n-1)&hash 做运算时，就会保留hash值后几位中的1。废话不多说，让我们动手来实践下吧。

我们先假设 n = 16, hash值从0开始递增，(n-1)&hash结果分布很均衡。

那我们再来看看 假设 n = 15, (n-1)&hash的情况又是怎样呢？

不对比不知道，一对比吓一跳，如果容量不为2的幂次方的时候，极大的提升了hash碰撞的可能，并且浪费了奇数位的存储位置。

那么我们通过上述的对比就可以得知

HashMap 容量始终设定为2的幂次方是为了 (n-1)&hash 索引位置均匀分布，减少hash碰撞。

那么我想大家接下来就会很疑惑：

如果我通过构造函数传入一个初始容量不为2的幂次方的数据，那岂不是就会出现上面所看到容量不为2的幂次方所出现的问题？

答案是不会的

首先，我们来看下HashMap的有参构造器。

public HashMap(int initialCapacity, float loadFactor) {		if (initialCapacity < 0)				throw new IllegalArgumentException("Illegal initial capacity: " +					initialCapacity);		if (initialCapacity > MAXIMUM_CAPACITY)				initialCapacity = MAXIMUM_CAPACITY;		if (loadFactor <= 0 || Float.isNaN(loadFactor))				throw new IllegalArgumentException("Illegal load factor: " +					loadFactor);		this.loadFactor = loadFactor;		this.threshold = tableSizeFor(initialCapacity);}

我们可以看到构造器里面调用了一个tableSizeFor(initialCapacity) 方法，那这个方法里面有隐藏着什么呢？我们继续往下看：

/*** Returns a power of two size for the given target capacity.* 对于给定的目标容量，返回两倍大小的幂。*/static final int tableSizeFor(int cap) {	int n = cap - 1;	n |= n >>> 1;	n |= n >>> 2;	n |= n >>> 4;	n |= n >>> 8;	n |= n >>> 16;	return (n < 0) ? 1 : (n >= MAXIMUM_CAPACITY) ? MAXIMUM_CAPACITY : n + 1;}

看到这段代码，是不是有点云里雾里呢？

我们来试着分析下，我们先假设我们输入的初始容量 initialCapacity = 10

int n = cap -1; //运算完之后 n = 9;n |= n >>> 1; //将9进行无符号右移一位并9做或运算并将结果赋予n// 0000...1001 | 0000 ... 0100 = 0000 ... 1101 结果 11n |= n >>> 2; //再次将 n 和 无符号右移两位的n做或运算// 0000 ... 1101 | 0000 ... 0011 = 0000 ... 1111 结果 15n |= n >>> 4;// 0000 ... 1111 | 0000 ... 0000 = 0000 ... 1111 结果 15n |= n >>> 8;// 0000 ... 1111 | 0000 ... 0000 = 0000 ... 1111 结果 15n |= n >>> 16;// 0000 ... 1111 | 0000 ... 0000 = 0000 ... 1111 结果 15n = n+1; //结果 16

由此我们可以看到，当在实例化HashMap实例时，如果给定了initialCapacity，由于HashMap的capacity要保证容量为2的幂次方，因此这个方法就是用来找到大于等于initialCapacity的最小的2的幂。

那么我想我们还会有一个疑问，那就是为要执行 int n = cap -1 ？来，举个例子，相信我们很快就能明白，我们继续假设我们输入的初始容量 initialCapacity = 8 ，我们不去做 int n = cap -1 操作，那么到底会发生什么呢？

n |= n >>> 1; //将8进行无符号右移一位并8做或运算并将结果赋予n// 0000...1000 | 0000 ... 0100 = 0000 ... 1100 结果 10n |= n >>> 2; //再次将 n 和 无符号右移两位的n做或运算// 0000 ... 1100 | 0000 ... 0011 = 0000 ... 1111 结果 15n |= n >>> 4;// 0000 ... 1111 | 0000 ... 0000 = 0000 ... 1111 结果 15n |= n >>> 8;// 0000 ... 1111 | 0000 ... 0000 = 0000 ... 1111 结果 15n |= n >>> 16;// 0000 ... 1111 | 0000 ... 0000 = 0000 ... 1111 结果 15n = n+1; //结果 16

发现问题没有？我们给定的初始容量 initialCapacity = 8 ，就是 2的幂次方，但是上面最终返回给我们的是16 ？我们继续来看看，我们执行 int n = cap -1 会有什么样的结果呢？

int n = cap - 1; //n = 7n |= n >>> 1; //将7进行无符号右移一位并7做或运算并将结果赋予n// 0000...0111 | 0000 ... 0011 = 0000 ... 0111 结果 7n |= n >>> 2; //再次将 n 和 无符号右移两位的n做或运算// 0000 ... 0111 | 0000 ... 0001 = 0000 ... 0111 结果 7n |= n >>> 4;// 0000 ... 0111 | 0000 ... 0000 = 0000 ... 0111 结果 7n |= n >>> 8;// 0000 ... 1111 | 0000 ... 0000 = 0000 ... 1111 结果 7n |= n >>> 16;// 0000 ... 1111 | 0000 ... 0000 = 0000 ... 1111 结果 7n = n+1; //结果 8

如果不减去1 得到的结果为16,不符合我们的预想，输入8的时候返回不小于输入结果的最小2的幂次方的结果应该是8，所以这里进行 int n = cap -1 就是为了避免这种情况。

最后不得不说，写这段源码的作者的这个算法实在是太犇了。
