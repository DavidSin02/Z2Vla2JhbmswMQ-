白话大数据之 白话进程

2020-10-29
| 科技

这一次我们来聊一下进程。之所以聊进程，是因为进程是计算的灵魂。假如我们买了一台电脑，我们很少会将它打扮的飘飘亮亮的放在客厅，从来不开机，每次下班回家看到这台漂亮的电脑就会深吸一口气然后感叹真特么美，生活真幸福。大部分情况下，我们要将它开机，然后让他干活，一旦开机，计算机就进入到了进程的世界，一个进程一个进程的处理，直到你要求它关机休息。所以，只要涉及到计算机，无论你是做大数据开发还是web开发，都逃不开操作系统，也就逃不开进程。其实分布式系统就是网络+单机操作系统+针对多机处理的算法，那么，我们有必要看一下进程是什么。

进程是什么

他山之石，我们首先看一下百度百科上进程的定义

进程-百科定义

定义中提到，进程是程序的执行实体，进程是线程的容器。所以实际上，在现代操作系统中，程序真正执行的实体是线程，进程则是在线程管理上建立起的一个抽象，如果非要给线程一个实体表示，那么就是在程序运行过程中涉及到的地址空间（内存空间），地址空间中包括程序运行的正文（编译后），打开的文件，子进程，即将发生的定时器，信号处理程序，账号信息等。也就是说，进程是一个程序能够运行所依赖的各种资源总和的抽象。清晰了这个概念，我们便容易理解，每个进程因为已经包含了它所依赖的所有资源，那么其他的进程便跟它没有关系，所以对于每个进程来说，它都认为自己是操作系统中唯一的进程，操作系统实现了一种抽象，这种抽象将进程彼此隔离开。

进程的生成

进程是运行的程序，如么进程的产生过程就是程序提交到机器上执行的过程。一般而言，程序是用特定的语言编写的（C，python，R，java等），计算机并不认识这些语言，之所以能够运行这些程序，是因为程序被解释成了机器码形式的指令--计算机能够识别的语言，计算机通过对这些指令不停的出栈与入栈进行程序的运行。解释的过程如下图（图片来自于网络）

程序编译过程

进程在创建之后，操作系统就需要持有一个能够到达该进程的句柄，也就是说，操作系统要能够找到这个进程。比如说，我们一般将 WEB 应用发布到 Tomcat 中，Tomcat 作为一个常驻进程，承担起 网站服务器 的功能，当我们提交 Web请求的时候，操作系统需要能够将我们发送的请求内容传递给 Tomcat， 传递的前提是操作系统必须能够准确找到 Tomcat 这个进程。所以进程创建成功后会返回一个进程号（PID），这个进程号便唯一表示了这个进程，我们可以通过进程号找到进程。但问题是，因为进程号是随机分配的，每次分配的都不一样，也就是说 Tomcat 一旦重启端口号就会改变，这样找起来就太麻烦了。所以操作系统又在端口号和进程号之间建立了映射关系，无论进程号怎么边， Tomcat 对应的端口号是不变的（可配置）， 当进行信息交互的时候，只需要找到对应的端口就可以了。

进程间通信

上文中我们提到，由于操作系统的宠溺，每个进程都会认为自己是唯一存在的进程，所以它不可能自己找到其他的进程进行交互。进程和进程之间的交互需要依赖其他的工具或者组件，在

1、本地进程通信

Unix操作系统中，本地进程之间的交互可以通过管道来传递，比如说下面这个命令

ps -ef | grep redis

这一段代码就是在查询 redis 对应的进程号，ps -ef 这个进程的输出传递给 grep redis 进程作为输入。当然我们也可以将前一个进程的输入写到一个文件中，后一个进程读取这个文件。其实上面的管道本质上也是写文件，所有的通信本质上都是写文件，因为网络是一种分块文件，管道是一种内存文件，在Unix操作系统中，万物皆文件。

另外我们也可以借助消息中间件，如 Kafka，Rabbit MQ，Zero MQ 等来实现进程的通信，这些中间件可以实现超大吞吐量等特殊场景下的通信，但是其本质与上述并没有差异。

2、远程进程通信

我们可以将两个进程之间的通信理解为一个过程调用另一个过程，当主调过程和被调过程在不同的进程空间时，就需要RPC（Remote Procedure Call）——远程过程调用，RPC是一个概念，各种语言或者框架有自己的实现方法，在面向对象的语言中，有一中实现方法便是RMI（ Remote Method Invocation)——远程方法调用，其实现原理便是在服务端实现其和客户端约定好的接口。

那么通信过程到底是怎么实现的呢，就本质上来说（panda真美，这句是被强制写上的），进程通信就是将要传递的信息数据从一个进程的缓冲区转移到另一个进程的缓冲区。

通信模式与进程状态

上文中提到，进程通信的本质就是通信数据在进程缓冲区之间的转移，当数据到达接收方的缓存区，接收方就及时将信息取走。这里就引入一个问题，接收方在等待数据到达的时候干啥？

现在，接收方可以选择一直傻等，但是计算器受不了啊，你自己等数据不能占着CPU不放啊，人家其他的进程还要运行呢。所以在等数据的时候，接收方进程会被置入阻塞状态（睡着了），数据到达后，合作进程回去叫醒他，说，别睡了，你的数据来了。

那么什么是阻塞状态呢？要回答这个问题，我们先看一下进程共有几种状态

在解释这些状态前，我们先要了解CPU是怎么工作的，如果CPU是单核心，那么同一时间只能运行一个进程，如果有多个可运行的进程同时到达，操作系统会将这些进程放在一个就绪队列里面，CPU从队列中取出这个进程运行一会儿，一看到点了就放回去，然后取出另一个进程运行一段时间。但是如果运行一个进程时发现它要读等数据（I/O）——无论时等其他进程的数据还是读取硬盘文件，这个过程很慢，CPU不愿意等，就把它放到阻塞队列中，说，你慢慢等吧，直到解除条件出发，这个进程被重新放到阻塞队列中。

上文中我们提到，进程是资源总和的抽象，在进程创建的时候，就会申请一部分内存资源（物理内存）。但是内存是有限的，当内存被占用满之后，就无法提交新的进程了，为了解决这个问题，操作系统对内存做了虚拟化抽象，也就是将一部分硬盘空间（可配置）和物理内存都映射成虚拟内存，新添加的进程可以先放在硬盘映射出的虚拟内存空间中，这时进程处于三种状态以外的另一种状态——挂起状态。现在，进程便拥有了以下五种状态

进程五态模型

等待态→挂起等待态：如果当前不存在就绪进程，那么至少有一个等待态进程将被对换出去成为挂起等待态；操作系统根据当前资源状况和性能要求，可以决定把等待态进程对换出去成为挂起等待态。挂起等待态→挂起就绪态：引起进程等待的事件发生之后，相应的挂起等待态进程将转换为挂起就绪态。挂起就绪态→就绪态：当内存中没有就绪态进程，或者挂起就绪态进程具有比就绪态进程更高的优先级，系统将把挂起就绪态进程转换成就绪态。就绪态→挂起就绪态：操作系统根据当前资源状况和性能要求，也可以决定把就绪态进程对换出去成为挂起就绪态。挂起等待态→等待态：当一个进程等待一个事件时，原则上不需要把它调入内存。但是在下面一种情况下，这一状态变化是可能的。当一个进程退出后，主存已经有了一大块自由空间,而某个挂起等待态进程具有较高的优先级并且操作系统已经得知导致它阻塞的事件即将结束，此时便发生了这一状态变化。运行态→挂起就绪态：当一个具有较高优先级的挂起等待态进程的等待事件结束后，它需要抢占 CPU，，而此时主存空间不够，从而可能导致正在运行的进程转化为挂起就绪态。另外处于运行态的进程也可以自己挂起自己。新建态→挂起就绪态：考虑到系统当前资源状况和性能要求，可以决定新建的进程将被对换出去成为挂起就绪态。参考 https://blog.csdn.net/qwe6112071/article/details/70473905

回到模式的讨论，上面一直傻等的情况就是阻塞模式。这种模式是比较傻的，就因为等个数据，进程就直接被打入冷宫了。聪明点的做法是一会儿过来看一下，数据有没有到达缓冲区，如果还没到就去干其他的事情。也就是非阻塞模式。这里又出现另外一个问题，一会儿来看看数据有没有到达的是进程，缓冲区是跟着进程走的，那么如果进程还没过来看的时候，数据先到达了，它存储在什么地方呢？实际上，这种情况下，操作系统会暂时把数据存储到自己的缓冲区，等读取进程到达了，再转移到进程的缓冲区。接收方进程自己一会儿来看一下的模式叫做同步模式，这个模式还是有些麻烦，可以不自己过来看，专门委托一个人一会儿看一下，如果到了就通知自己，这时候自己再去取数据，这种模式叫做异步模式。

非阻塞的实现方式是在提交请求之后直接返回错误码，而不再等待响应。值得注意的是，不止接收方有这些模式，发送方是同样的道理，根据发送方要不要等到数据被接收后再返回可以进行模式切分。

进程调度

上文中提到，CPU是从就绪队列中取进程分时执行的，那么先取谁，后取谁，就涉及到进程调度的问题。常见的调度算法如下

1、先来先服务（FIFO），按照进程到达的时间进行调度

2、最短作业优先（SJF），优先调度运行时间最短的作业

3、时间片轮转，分时运行

4、最短剩余作业优先 SRJN (Shortest Remaining Job Next) ,优先调度剩余时间最短的作业

针对多核CPU，还有其他的优化调度方法，如多级队列，多级反馈队列等。

值得注意的是，进程到达的时候，CPU根本就不知道它会运行多长时间，所以最短作业优先，最短剩余作业优先调度方法会对时间进行预测，这些调度方法有时间再详做阐述。
