基于故障注入的以太坊性能测试系统

2020-10-29
| 科技

图 1 系统流程示意图


1. 项目意义

近年来区块链技术广受关注。区块链的去中心化、不可篡改的特性有效降低了交易成本，使人们得以在无人监管的模式下构建可信交易平台。因此区块链技术有很高的应用价值。智能合约是一种信息化的计算机协议，无需人来监督执行。区块链去中心化、不可篡改的特性为实施智能合约提供了必要的可信环境。以太坊是首个支持图灵完备的智能合约编程语言的区块链平台。以太坊支持私有链，私有链是一种可以根据个人需求进行部署的区块链，具有可配置、规模小、需要许可等特点。这使得以太坊私有链可以更好的满足企业和个人的使用需求。

区块链去中心化的特性、复杂的共识流程导致其吞吐量低、交易延迟高、性能较差。目前，支付宝等支付方式的吞吐量早已突破 10000 笔/秒，以太坊公有链平均 15 笔/秒，很难满足当今的数字支付场景。私有链是一种良好的解决方案，可以限制用户规模，降低通信开销，进而提升性能。由于用户使用以太坊私有链的场景对性能可能有要求，需要进行性能评估。同时，由于分布式系统的真实部署环境不稳定，可能会面临各种类型的故障。开发人员需要验证以太坊私有链能否在故障条件下满足用户的需求。因此，在实际应用中，应仔细考虑以太坊私有链的性能，并模拟真实部署场景进行测试。

2. 项目内容

为了尽可能暴露性能问题，帮助以太坊私有链维护人员评估系统的性能与可靠性，本文实现了基于故障注入的以太坊性能测试系统。本文中依次实现了：（1）通过对以太坊私有链的性能分析，提出两个以太坊性能影响因素，Difficulty、Gas Limit；（2）通过对分布式系统故障与区块链的特性进行分析，提出四种类型的故障，即应用故障、共识故障、智能合约故障、网络故障。本系统可以获取以太坊私有链的秒级性能指标信息，从而对以太坊的性能进行准确评估。本系统引入故障注入技术，可以在测试环境中注入故障，探究在故障情况下以太坊私有链的稳定性。最终，本系统可以对整个以太坊私有链进行一个完整的系统的评估。

本系统的整体架构设计如图 1 所示。系统整体分成三部分：第一部分为前端展示，用户通过浏览器进行访问，输入配置并输出测试结果；第二部分为测试管理模块，主体为 Web 服务器，用于解析、存储用户输入的测试配置，根据功能可划分为测试配置子模块与结果管理子模块；第三部分为测试执行模块，主体为执行测试任务的命令行工具，根据功能可划分为测试准备子模块、性能测试子模块与故障注入子模块。

测试配置子模块。测试配置子模块负责读取、验证并生成测试配置。测试配置信息包括三种配置，即区块链配置、性能测试配置与故障注入配置。区块链配置为以太坊私有链启动配置与创世区块的配置。性能测试配置包括测试的请求速率、持续时间以及工作负载智能合约。故障配置包括故障类型、参数、开始、持续时间以及每种故障各自的配置。Difficulty 与 Gas Limit 的设置包含在区块链配置中。图 2 为这两个参数影响以太坊吞吐量和交易延迟的原理，Difficulty 影响生成区块的间隔、Gas Limit 影响区块所能容纳的交易数量。

图 2 Difficulty 与 Gas Limit 对以太坊性能的影响


结果管理子模块。结果管理子模块主要负责处理测试任务执行完成后产生的测试结果数据，并对处理结果进行展示。在测试执行模块完成测试任务后，会生成包含秒级性能指标的测试结果 JSON 文件。结果管理子模块会读取这些文件中的数据，存储在 MongoDB 数据库，以便后续查看。测试结果使用 Python 进行数据清洗，并计算吞吐量、延迟等指标的平均值、中位数等。

测试准备子模块。测试准备子模块主要处理开始执行测试任务之前所需的准备工作。模块首先通过 Docker 启动以太坊节点，并搭建 Docker Subnet 使节点之间可以互相识别。模块通过文件夹挂载的形式传递节点的启动命令、配置文件等信息。测试链搭建完成后，系统还会进行智能合约部署以及故障初始化等步骤。在本模块所有步骤完成后，即可开始性能测试。测试结束后，容器自动销毁，不保留区块数据，减少存储空间占用。

性能测试子模块。性能测试子模块在测试准备子模块执行结束后开始运行，主要负责按照设定的负载发送交易请求，收集并处理返回数据。本模块使用 NodeJS 中的 Promise 机制实现大量性能测试交易请求并行处理。模块以线程控制器为控制中心，通过向预先启动好的线程发送开始测试的消息，所有线程将开始向每个以太坊节点发送交易请求，每个节点对应一个线程。线程使用以太坊团队官方提供的 web3.js 作为与以太坊客户端交互的 API。最终，每个线程会记录每个节点的返回结果的数据，统一传给主线程进行数据处理。主线程会计算出每秒的性能指标，得出最终的测试结果。

故障注入子模块。故障注入子模块负责在性能测试执行过程中注入故障，主要由故障控制器与四类故障注入的实现组成。故障注入子模块首先会通过测试准备模块对待注入的故障进行初始化。随着性能测试的开始，故障计时器开始计时，当到达设定的时间时触发故障注入事件。本模块一共实现了四种类型的故障，分别是应用、共识、智能合约与网络故障。除共识故障外，每种故障有可以配置参数以表示故障的严重程度。应用故障为流量波动故障，该故障由控制每个线程的请求速率实现。共识故障为节点的退出与加入，通过对 Docker 容器进行暂停和重启实现。智能合约故障则实现了一个低性能的智能合约，通过将一半的正常请求转化为低性能智能合约请求实现。网络故障则通过 Docker 的混沌工程工具 Pumba 实现，可以对处于局域网中 Docker 容器注入网络丢包和延迟。

3. 项目效益

为验证性能影响因素与故障对以太坊私有链的性能与稳定性的影响，本文设计了两组实验：性能影响因素验证实验与故障注入验证实验。性能影响因素实验分别设计了两组 Difficulty 与 Gas Limit 值，探究以太坊私有链在不同请求速率下的性能表现。故障注入实验针对每种故障设计了 3 组参数（除共识故障），在以太坊私有链稳定运行一段时间后注入故障，持续一段时间后恢复，探究其在故障注入后与恢复后的性能指标变化。

图 3-1 为性能影响因素验证实验的实验结果。其中左右两侧分别为 Difficulty、Gas Limit 验证实验的吞吐量与延迟指标图。每个图 x 轴为性能影响因素，y 轴为性能测试请求速率，z 轴为性能指标。图中可以看出这两个影响因素对性能指标的影响呈相反趋势。随着 Difficulty 升高或 Gas Limit 降低，吞吐量越低，交易延迟越高，性能也越差。图 3-2 为四类故障注入验证实验的实验结果。其中从左到由分别为应用、共识、智能合约、网络故障的吞吐量指标。每个图分为四个阶段，即启动、稳定、注入、恢复。图中可以看出这四类故障的注入均对以太坊私有链的吞吐量产生了显著的影响，且不同参数的故障对性能影响程度有一定差异。在故障恢复后吞吐量均不能迅速恢复到稳定运行时的水平。

根据实验结果，我们可以认为，本系统可以对以太坊私有链进行充分的性能测试，并可以通过故障注入等手段在测试环境下模拟真实的部署场景。本文识别的两个性能影响因素与四类故障对以太坊私有链均有明显的性能影响。
