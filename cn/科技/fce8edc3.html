gzthermal使用方式如下：

gzthermal: displays compression efficiency heatmap of gzipped files.Version 0.4b (23 May 2014) by Frederic KayserUsage: gzthermal [-e|-s|-m|-l] [-n|-w] [-b] [-z] [-g] file.gzOutput file is a PNG image called "gzthermal-result.png"Options: e, s, m, l  extra small, small, medium or large text output         n, w        narrow or wide text lines (default is in between)         b           binary/hex mode (small text size equivalent only)         z           bicolor (distinguish LZ matches from literals)         g           grayscale mode (may help colorblind people)

除了以上，还包括未记录的数字选项（压缩级别），只需添加数字即可将压缩级别设置为0到9，例如，“ gzthermal -ew9”应生成比“ gzthermal -ew”小的输出PNG文件（默认压缩级别为4）。

HTML文档（http://encode.su/登陆页面在我的第一个示例中，HTTP / 1.1 Web服务器通常使用基于Gzip / Deflate编码动态压缩文本文档（html，css，js，svg ...），例如Apache使用mod_deflate（调用Zlib）发送Gzip数据。
第一个想到就是HTML5文档类型声明应使用小写而不是大写，因为大写=相当罕见=不良压缩：
<！DOCTYPE html> 压缩率差
<！doctype html> 压缩率好（压缩后保存2个字节）

如果熟悉香农熵，这是显而易见的，红色区域现在标记了不佳的压缩位置。
<charset =“ UTF-8”->最好写成“ utf-8”也是一样，（基于IANA Charset：这可以“忽略大小写”）。

压缩原理

Gzip / Deflate压缩基本上使用LZSS + Huffman算法，由Huffman算法提供的可变长度编码（VLE，Variable Length Encoding ）应用于三种不同类型的元素：文字（不属于LZ匹配的独立符号），LZ长度和LZ匹配对的LZ距离。
用于表示符号的位数是从其出现频率得出的，基本上，较多出现的符号通常以较短的代码结尾（例如3、4、5位，这也取决于源字母表的大小），而较少的经常以较大的代码（例如8、9、10，...位）结尾。这种VLE（可变长度编码）实际上早于Huffman编码，Morse（摩斯码）以这种方式工作，Lineotype排字机键盘布局也是根据字符频率设计。
在文件末尾（显示更蓝，更深的蓝色）或至少在处理了一定数量的数据（几千个字节/符号）之后，压缩会会更好地工作。预热期之后的主要原因是，在数据流的早期，字典（或更确切地说是此处的滑动窗口）几乎是空的，并且第一次出现了许多符号/词，之后出现了一组符号/词。一个单词/一组单词再次出现，它将由对先前出现的引用的替换（这是LZ匹配），这样可以节省大量空间，显示为蓝色区域。相比之下，首次出现似乎显得很庞大，这也是首次出现的代价。
这也是支持CSS和JavaScript文件结合的一个很好的论据。。
