為什麼 0.1 + 0.2 = 0.3，原來你不知道

2020-10-29
| 科學

為什麼這麼設計（Why’s THE Design）是一系列關於計算機領域中程序設計決策的文章，我們在這個系列的每一篇文章中都會提出一個具體的問題並從不同的角度討論這種設計的優缺點、對具體實現造成的影響。

0.1 + 0.2 = 0.3 這個等式的成立看起來是理所當然的，然而前面的文章 為什麼 0.1 + 0.2 = 0.300000004 分析了為什麼這個等式在絕大多數的編程語言中都不成立，標準的浮點數可以通過 32 位單精度浮點數或者 64 位的雙精度浮點數保證有限的精度，所有正確實現浮點數的編程語言都會遇到如下所示的『錯誤』：

> 0.1 + 0.20.30000000000000004

浮點數作為編程語言中必不可少的概念，需要在性能和精度方面做出的權衡，過高的精度需要更多的位數以及更多的計算，過低的精度也無法滿足常見的計算需求，這種重要的決策會影響上層千千萬萬的應用和服務，然而這個決策需要面對的問題與軟件工程中需要解決的問題也沒有太多區別 — 如何儘可能地利用有限地資源實現特定目的。

圖 1 - 性能和精度的權衡

雖然浮點數提供了相對優異的性能，但是在金融系統中使用精度低的浮點數會有非常嚴重的後果。假設我們在交易所或者銀行使用 64 位的雙精度浮點數存儲賬戶的餘額，這時就存在被用戶攻擊的可能，用戶可以利用雙精度浮點數的精度限製造出更多餘額：

圖 2 - 金融系統與浮點數

當用戶分別先向賬戶中充值 0.1 單位和 0.2 單位的資產後，使用雙精度浮點數在計算時會得到 0.30000000000000004，用戶將這些資產全部提現可以得到 0.00000000000000004 的意外之財[^1]，如果用戶重複的次數足夠多，就可以把銀行提破產，大家加油，下面是一段使用浮點數處理充值和提現的代碼：

var balance float64 = 0func main() {	deposit(.1)	deposit(.2)	if balance, ok := withdraw(0.30000000000000004); ok {		fmt.Println(balance)	}}func deposit(v float64) {	balance += v}func withdraw(v float64) (float64, bool) {	if v <= balance {		balance -= v		return v, true	}	return  0, false}

上面的代碼也只是理想的情況，今天的成熟金融系統不可能~~（其實不一定）~~犯這種低級的錯誤，但是一些新興的交易所中仍然存在這種可能，不過想要真正實施上述操作還是非常困難。如果我們可以控制的資源是無限的，自然就可以實現無限精度的小數，然而資源永遠都是有限的，一些編程語言或者庫會通過下面的兩種方法提供精度更高的小數保證 0.1 + 0.2 = 0.3 這個等式的成立：

  • 使用具有 128 位的高精度定點數或者無限精度的定點數；
  • 使用有理數類型和分數系統保證計算的精度；

上述這兩種方法都可以實現精度更高的小數系統，但是兩者的原理卻略有不同，接下來我們將分析它們的設計原理。

十進制小數

在很多時候浮點數的精度損失都是因為不同進制的數據相關轉換造成的，正如我們在 為什麼 0.1 + 0.2 = 0.300000004 一文中提到的，我們無法使用有限的二進制位數準確地表示十進制中的 0.1 和 0.2，這就造成了精度的損失，這些精度損失不斷累加在最後就可能累積成較大的錯誤：

圖 3 - 二進制與十進制精度的損失

如下圖所示，因為 0.25 和 0.5 兩個十進制的小數都可以用二進制的浮點數準確表示，所以使用浮點數計算 0.25 + 0.5 的結果也一定是準確的[^2]：

圖 4 - 0.25 和 0.5 的浮點數表示

為了解決浮點數的精度問題，一些編程語言引入了十進制的小數 Decimal。Decimal在不同社區中都十分常見，如果編程語言沒有原生支持 Decimal，我們在開源社區也一定能夠找到使用特定語言實現的 Decimal 庫。Java 通過 BigDecimal 提供了無限精度的小數，該類中包含三個關鍵的成員變量 intVal、scale 和precision[^3]：

public class BigDecimal extends Number implements Comparable<BigDecimal> {    private BigInteger intVal;    private int scale;    private int precision = 0;    ...}

當我們使用 BigDecimal 表示 1234.56 時，BigDecimal 中的三個字段會分別以下的內容：

  • intVal 中存儲的是去掉小數點後的全部數字，即 123456；
  • scale 中存儲的是小數的位數，即 2；
  • prevision 中存儲的是全部的有效位數，小數點前 4 位，小數點後 2 位，即6；

圖 5 - BigDecimal 實現

BigDecimal 這種使用多個整數的方法避開了二進制無法準確表示部分十進制小數的問題，因為 BigInteger 可以使用數組表示任意長度的整數，所以如果機器的內存資源是無限的，BigDecimal 在理論上也可以表示無限精度的小數。

雖然部分編程語言實現了理論上無限精度的 BigDecimal，但是在實際應用中我們大多不需要無限的精度保證，C# 等編程語言通過 16 字節的 Decimal 提供的 28 ~ 29 位的精度，而在金融系統中使用 16 字節的 Decimal 一般就可以保證數據計算的準確性了[^4]。

有理數

使用 Decimal 和 BigDecimal 雖然可以在很大程度上解決浮點數的精度問題，但是它們在遇到無限小數時仍然無能為力，使用十進制的小數永遠無法準確地表示 1/3，無論使用多少位小數都無法避免精度的損失：

圖 6 - 無限小數的精度問題

當我們遇到這種情況時，使用有理數（Rational）是解決類似問題的最好方法，部分編程語言因為科學計算的需求會將有理數作為標準庫的一部分，例如：Julia[^5] 和 Haskell[^6]。分數是有理數的重要組成部分，使用分數可以準確的表示 1/10、1/5和 1/3，Julia 作為科學計算中的常用編程語言，我們可以使用如下所示的方式表示分數：

julia> 1//31//3julia> numerator(1//3)1julia> denominator(1//3)3

這種解決精度問題的方法更接近原始的數學公式，分數的分子和分母是有理數結構體中的兩個變量，多個分數的加減乘除操作與數學中對分數的計算沒有任何區別，自然也就不會造成精度的損失，我們可以簡單瞭解一下 Java 中有理數的實現[^7]：

public class Rational implements Comparable<Rational> {    private int num;   // the numerator    private int den;   // the denominator    public double toDouble() {        return (double) num / den;    }    ...}

上述類中的 num 和 den 分別表示分數的分子和分母，它提供的 toDouble 方法可以將當前有理數轉換成浮點數，因為浮點數在軟件工程中雖然更加常用，當我們需要嚴密的科學計算時，可以使用有理數完成絕大多數的計算，並在最後轉換回浮點數以減少可能出現的誤差。

然而需要注意的是，這種使用有理數計算的方式不僅在使用上相對比較麻煩，它在性能上也無法與浮點數進行比較，一次常見的加減法就需要使用幾倍於浮點數操作的彙編指令，所以在非必要的場景中一定要儘量避免。

總結

想要保證 0.1 + 0.2 = 0.3 這個公式的成立並不是一件複雜的事情，作者相信除了文中介紹的這些方案之外，我們還會有其他的實現方式，但是文中介紹的方案是最為常見的兩種，我們再來回顧一下如何使 0.1 + 0.2 = 0.3 這個公式成立：

  • 使用十進制的兩個整數 — 整數值和指數表示有限精度或者無限精度的小數，一些編程語言使用 128 位的 Decimal 表示具有 28 ~ 29 位精度的數字，而一些編程語言使用 BigDecimal 表示無限精度的數字；
  • 使用十進制的兩個整數 — 分子和分母表示準確的分數，可以減少浮點數計算帶來的精度損失；

有理數和小數是數學中的概念，數學是一門非常嚴謹和精確的學科，通過引入大量的概念和符號，數學中的計算可以實現絕對的準確；但是軟件工程作為一門工程，它需要在複雜的物理世界，利用有限的資源解決有限的問題，所以我們需要在多個方案之間做出權衡和選擇，數學中的有理數和無理數其實都可以在軟件中實現，但是在使用時一定要想清楚 — 為了得到這些我們犧牲了什麼？到最後，我們還是來看一些比較開放的相關問題，有興趣的讀者可以仔細思考一下下面的問題：
