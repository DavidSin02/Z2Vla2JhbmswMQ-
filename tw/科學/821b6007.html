從公有云到混合雲-青雲，靈雀雲提供雲原生解決方案解析

2020-11-14
| 科學

在前面專門分析了阿里，騰訊和華為的雲原生解決方案。可以看到在當前公有云的雲原生解決方案提供上，三家公司從DevOps持續集成和交付，容器雲，微服務，服務網格，PaaS技術服務等各個方面都提供了一個完整的融合解決方案。對於其它的雲平臺服務商要想在公有云服務領域競爭實際上機會已經很小。

在談雲原生解決方案的時候談到一個重點就是協助企業上雲，那麼一定就存在一個企業私有云建設，企業私有云到公有云遷移，或者說私有云和公有云共存的這麼一個階段。

其次，可以看到雲原生整體的解決方案和架構設計實際上還是具有很大的技術複雜性，如果是企業私有云內部要自己搭建這麼一套基礎平臺難度也挺大，而不像公有云一樣只需要企業使用最終的產品或服務，因此一定存在比較重的基礎架構諮詢，平臺建設和實施工作。而這些反而會成為其它的一些原有公有云服務廠商的重要機會。

青雲-KubeSphere管理平臺


談到容器雲的時候一定要談到基於Kurbernetes實現的容器資源編排和動態調度能力。雖然Kurbernetes提供了大量標準的開放API接口服務能力。但是為了更好的管理k8s集群，仍然有不少的開源集群管理平臺軟件，比如Rancher，KubeOperator等。

青雲的KubeSphere也是屬於同類的基於Kurbernetes集群管控治理平臺，同時又進一步基礎了中間件和產品倉庫，基於Jekins的DevOps持續集成能力。KubeSphere當前最新版本3.0，該項目本身完全開源，並且也是入駐CNCF基金會的一個重要開源項目。

其官網介紹如下：

QKE（QingCloud KubeSphere Engine），KubeSphere 是一款基於 Kubernetes 構建的企業級容器平臺，QKE 在 QingCloud 公有云上交付 KubeSphere 容器平臺全能力，並可統一管理跨雲、跨基礎設施的 Kubernetes 集群，通過極簡的人機交互實現 CI / CD、微服務、以及集群運維管理，幫助用戶更敏捷地構建雲原生應用與統一管理應用的全生命週期。

在今年新推出的3.0版本實現了一次大的升級，新增“多集群管理”功能，可幫助企業實現集群的靈活增減、跨集群的應用部署以及全生命週期管理，僅需 6 分鐘即可完成一個生產級別的集群部署，大幅提升企業 IT 資源利用率和應用管理效率，加快雲原生生產落地和數字化轉型，創造更多商業價值。

根據其網上介紹的資料可以看到：

首先，在多集群功能上，支持 Solo 和 Federation 兩種管理模式，Solo 模式支持將多個互相獨立的集群在同一平臺統一運維管理，解決了企業 IT 運維人員管理多個 Kubernetes 集群的困擾；Federation 模式下，通過聯邦能力，可將多個 Kubernetes 集群池化，讓應用在多集群間的“漂移”。

其次，在管理層面，QKE 3.0 可以實現單一租戶聚合管理多集群，也可將某一租戶和某一集群進行一對一管理，既適應大大型企業的統一化管理訴求，也可以針對企業用戶多分支機構這種場景做靈活管理配置。

在最近中國信通院舉辦的雲原生產業大會上，青雲的陳少文分享了加速 DevOps 落地的方案——以 KubeSphere 快速構建跨集群應用，建立企業級 DevOps 平臺，實現跨集群的流水線管理、跨集群應用部署。

青雲的混合雲解決方案

青雲基於KubeSphere引擎實現的混合雲解決方案一定是當前青雲的一個差異化競爭的亮點。前面已經談到在QKE 3.0增加了多集群管理功能後，已經彌補了原來和Rancher對比的不足，同時增加了對DevOps和微服務架構開發的支撐能力後進一步方便應用開發和集成。

為何需要混合雲？

前面已經談到過企業上雲是一個循序漸進的過程，不可能一上雲就把自己內部的數據中心和私有云全部替換掉，因此從企業角度需要一套雲管理平臺，實現對自己私有云和公有云環境的同時集中化管理能力。同時在集中化管理基礎上，還能夠更好的實現私有云和公有云之間資源的分配，應用的遷移能力。

我們可以舉個簡單場景，當企業自有的私有云資源服務滿足業務場景和併發需求的時候，我們可以通過混合雲管理實現應用向公有云集群的遷移和交付，實現資源的動態擴展。而當需求下降後，我們又可以搜索公有云資源使用，降低成本。

其次，企業自有私有云環境最好不要和某一個公有云服務商完全綁定，即能夠通過混合雲管理平臺實現對多家公有云服務平臺的適配和對接能力。這實際和我在講SOA架構思想談到的多個能力源適配和聚合一樣。

可以看到KubeSphere本身可以在阿里，華為，騰訊等各大公有云平臺安裝和部署，管理底層的虛擬化資源能力。那麼混合雲管理就能夠同時實現向多個不同的公有云平臺交付的能力，或者說也可以間接實現一個在多個公有云平臺之間進行環境遷移的能力。而這個能力本身將成為混合雲解決方案所提供的關鍵能力。

對於微服務和DevOps，對於很多功能實際是面向開發人員的，而不是簡單的容器雲部署後的資源監控和運維。在網上有文章專門比較了，實際KubeSphere作為集群管理平臺，在對開發人員的支撐和優化性上是明顯優於Rancher，KubeOperator等的。

微服務治理和灰度發佈

KubeSphere 深度整合 Istio，提供無代碼侵入的微服務治理平臺，提供金絲雀發佈、藍綠部署、熔斷、流量管控等完善的治理功能。同時，為企業提供微服務改造諮詢以及技術支持服務，助力企業將核心應用微服務化。

簡單來說，青雲KubeSphere提供基於Istio的微服務治理和灰度發佈能力。從網上資料可以看到沒有太多的關於對微服務開發框架的定製，微服務開發方面的支撐。這塊相當其它幾大公有云廠商整體方案來說較弱，但是也足夠開放，給了企業應用開發更多的選擇靈活性。

DevOps持續集成和交付

青雲KubeSphere看介紹是基於Jekins和Ansible開源工具集成來實現完整的流水線開發和設計能力，實現完整的持續交付和集成。

KubeSphere支持代碼直達鏡像倉庫（Source to Image）、二進制文件直達鏡像倉庫（Binary to Image）的方式構建 Docker 鏡像，開發人員無需關注 Dockerfile 或 Docker 鏡像的配置，即點即生成 Docker 鏡像。無需配置 Jenkins，圖形化拖拽編輯 CI / CD 流水線，並內置 10+ 中 CI / CD 流水線模版。輕鬆實現 CI / CD，有效推動企業 DevOps 落地。

雖然青雲沒有重點推DevOps部分的能力，但是可以看到當前QKE引擎和管理平臺對於基礎的DevOps過程支撐能力已經完全具備。

簡單總結：

對於青雲本身是一家偏技術驅動的企業，在2012年成立，在2013年公有云平臺正式上線，包含完整的 IaaS 組件：計算、存儲、網絡、安全；資源秒級響應，按秒計費；100% 二層隔離。當時推出的資源秒級響應，按秒計費也產生了一定的轟動。在2015年推出超融合系統和桌面雲服務，同時進一步向PaaS雲和大數據平臺服務發展。

從2019年開始可以說是其發展關鍵的一年，多個項目加入CNCF雲原生基金會項目拓展雲原生共建生態，同時推出QKE容器雲管理平臺，拓展混合雲解決方案。

從當前情況來看，青雲KubeSphere 已在本來生活、四川航空、國控生物、中國人保壽險、紫金保險、新浪網、Aqara 等行業客戶的生產環境中完成了跨集群流水線管理的可行性驗證。在如何實現和阿里，華為雲等的差異化競爭上找到一個關鍵切入點。

靈雀雲-從容器到雲原生整體方案

對於靈雀雲，比青雲成立晚兩年左右，在剛成立不久就瞭解過這家企業，知道是做容器雲服務的廠家。而當前的基於容器技術的公有云PaaS平臺只能說是在起步階段，還沒有得到大家的重視，就我自己而言當時也絕對沒有想到容器雲平臺和雲原生在最近5年時間能夠發展如此迅猛。

由原微軟Azure雲平臺的核心創始團隊創立，CEO左玥和CTO陳愷都有在微軟做公有云的經驗。但是實際上大家也可以看到，從14年到16年，靈雀雲平臺本身基本是屬於消失在大家視野下，我原來也談到過，面對阿里，華為，騰訊等各大公有云服務廠商的激烈競爭下，你做公有云服務想要快速的突出重圍是相當困難的事情。

而且當時靈雀雲只做基於容器技術的自動化應用託管和部署，而這本身又對企業本身的應用開發架構等各方面都有要求，包括類似微服務化改造，技術標準化等，這些企業如果沒有專門的線下諮詢規劃和實施輔導，很難完全遷移到雲環境上面來。其次，類似一個完整PaaS需求的數據庫，中間件，消息緩存，對象存儲等諸多PaaS技術服務能力當時靈雀雲也不具備或做的很弱。

所有這些都導致你如果簡單做一個公有的容器雲平臺服務不現實。

在2016年開始，雲原生和Kubernetes容器資源編排越來越受到重視，也在這個時候靈雀雲轉向為大型企業或集團構建基於容器技術的私有云平臺，而這個轉型可以說取得了效果。我們可以看下網上對左玥的一段訪談材料：

之前做公有云大多面向的是中小企業客戶，主要精力都花在產品上，產品標準化程度較高，形成了產品層面的競爭力。轉做私有云以後，公司可以基於以往公有云的核心產品去做定製，實現快速迭代，這是很多“後來者”不具備的核心競爭力。在左玥看來，做私有云容器產品，需要時間積累。如果是某個團隊只用一、兩年時間突攢出來的一款產品，在交付的時候會出很多問題，因為他們沒有辦法理解DevOps流程、微服務改造、容器化運維、智能化運營以及跨雲管理等等，這些概念的真正內涵。

從單純的容器雲到雲原生解決方案提供

在前面我已經談到，在轉向到企業私有云PaaS平臺的時候，並不是單純的提供一套基於Kurbernetes和Docker的容器雲平臺產品，而是需要提供一套協助企業傳統IT架構轉型的雲原生解決方案。這套解決方案裡面包括了微服務架構改造，敏捷研發和過程管理，DevOps持續集成和交付，容器雲平臺，後期的IT運維監控和技術運營等，而這些往往才是企業最缺失的內容。

即我們對雲原生解決方案的理解不能是簡單的產品，而應該是：

產品+諮詢規劃+實施落地

在你協助企業數字化轉型的過程中，你會發現這三點都必不可少，當你從公有云服務轉型到私有云市場的時候，一定就存在傳統軟件實施類無法標準化的項目人員和工作量投入。

而在這點上可以看到，靈雀雲當前思路是對整個產品線和團隊進行分工。

一部分是核心技術團隊負責最底層的核心產品和通用性產品研發，更關注穩定、硬科技等內容，並且這個團隊跟騰訊合作比較多，跟騰訊公有云內部容器團隊完全打通，採用同一個版本、同一個技術來為用戶提供服務。

一部分是交付團隊。同時交付團隊又可以分為兩層：一層負責標準化產品開發，包括應用的改造，DevOps流程和容器化等。一層負責定製化開發，包括平臺功能的增加或者改動，以及周邊核心功能等。 在行業定製化解決方案方面，靈雀雲精準地引入生態合作伙伴。

從雲原生解決方案到生態整合

我在很早談私有云PaaS平臺的時候就談到，整個PaaS平臺是一個大生態，裡面涉及到IT基礎設施，數據庫，應用中間件，消息，緩存，應用開發商，技術服務提供商等諸多合作伙伴。對於類似阿里，華為可以自己為主提供完整解決方案，但是對於其它雲服務解決方案商來說更多的是圍繞自己的核心平臺能力，整合外圍生態合作伙伴。

技術驅動的思路往往是我只賣一個容器雲平臺產品，你拿過去如何用，如何安裝消息，緩存等各類技術服務產品我不關心。但是雲原生解決方案下的思路應該是提供一個完整的覆蓋軟件開發交付全生命週期，覆蓋各種業務需要的技術組件和能力提供的完整解決方案。

容器網絡組件Kube-OVN

靈雀雲開源的基於OVN的Kubernetes網絡組件Kube-OVN列入CNCF最新版本的CloudNative Landscape全景圖譜。作為國內首個被收錄的開源網絡系統，Kube-OVN通過將OpenStack領域成熟的網絡功能平移到Kubernetes，幫助用戶應對更加複雜的基礎環境和應用合規性要求，實現所有數據平面的統一，從而降低問題排查難度和工作量。

Kube-OVN是提供了大量目前Kubernetes不具備的網絡功能，並在原有基礎上進行增強。通過將OpenStack領域成熟的網絡功能平移到Kubernetes，來應對更加複雜的基礎環境和應用合規性要求。

Kube-OVN主要具備五大主要功能：Namespace 和子網的綁定，以及子網間訪問控制，靜態IP分配，動態QoS，分佈式和集中式網關，內嵌 LoadBalancer。將OpenStack社區的網絡功能向Kubernetes平移，從而彌補Kubernetes網絡的不足，打造更加完整的網絡體系。

Alauda Kubernetes管理平臺

對於Alauda Kubernetes是靈雀雲推出的Kubernetes集群管理平臺，可以理解為和青雲的KubeSphere類似的一個產品，但是本身是商用產品沒有開源。

該產品主要由Alauda Kubernetes Engine (AKE) 、系統運維管理（管理平臺、監控、日誌）、權限管理、賬號體系管理等多個業務模塊組成，可以根據用戶使用場景提供最佳網絡、存儲技術及解決方案。產品本身更好的發揮了 Kubernetes 產品特性，讓開發者更關注業務本身，讓整個平臺可以像管理產品一樣管理應用，更好的提高資源利用率。

而且當前我們也看到對於監控使用標準Prometheus + Grafana 解決方案，對於日誌採集和分析採用標準的ELK（Elasticsearch、Logstash、Kibana）或EFK（Elasticsearch , Fluentd, Kibana）方案已經成為一種通用的選擇。

即微服務治理應該是和DevOps同樣平級的一個內容，貫穿整個軟件開發和交付全生命週期管理。原來我思考是將微服務治理放在微服務開發框架或API網關裡面，但是後續主流微服務治理應該是採用ServiceMesh服務網格下類似Istio方案來實現，因此將其單獨拿出。

雲原生加速企業數字化轉型

架構和理念與數字化轉型趨勢一脈相承，為開發高效、可擴展且可靠的軟件，形成高效 IT 研發能力開闢了道路，助力企業更加順暢地數字化轉型。

容器、持續交付&DevOps、微服務構成了雲原生技術黃金三角，這是所有希望數字化轉型的客戶都逃不開的“黃金三角”，三大核心技術的不斷成熟促成了雲原生理念的興盛。從靈雀雲服務客戶的實踐來看，靈雀雲當前的雲原生解決方案在類似招商銀行，中石油等大型企業都得到了進一步的應用和實踐。

融資情況

在17年獲得騰訊雲戰略領投，高榕資本，寬帶資本跟投上億人民幣。在19年又獲得Intel領投，明照資本跟投的新一輪投資。可以說傍上騰訊和Intel兩個大款後，整體靈雀雲發展更加迅速，而實際對於靈雀並沒有參加騰訊公有云建設和運營，也就是說從騰訊思路也是將靈雀定位到延伸到企業私有云服務領域的一個重要抓手。

簡單來說靈雀雲當前並沒有運營公有云服務平臺，如果在企業實施私有云後，如何企業後續要進一步實施混合雲或上到公有云服務，那麼自然會平滑銜接到騰訊雲服務上。

騰訊本身確實是一家擅長收購和資源整合的公司，騰訊雲收購了Coding擴展資源研發平臺和DevOps能力，同時又投資靈雀雲拓展面向企業私有云環境的雲原生解決方案能力，從這幾點來看騰訊雲後續的雲原生髮力仍然不可小視。

簡單總結：

在前面談青雲的雲原生解決方案和容器雲的時候，談到一個核心優勢就是混合雲解決方案，其次就是容器雲集群管理平臺對微服務，對DevOps過程支撐的融合。

而轉到靈雀雲可以看到，實際上私有云PaaS平臺還不止上面這些內容，還包括了類似微服務架構規劃諮詢，改造，敏捷研發過程改進，各類技術服務選型和應用實踐，遺留系統遷移等諸多內容，而這些內容往往不同的垂直行業需求還不同。靈雀雲本身的優勢則是體現在面向企業內部的諮詢規劃和實施能力，這塊本身是靠人來堆積的重資產，但是隻要切入的足夠久，積累的足夠多，那麼差異化競爭優勢越大。

對於阿里雲本身不會做企業內部實施服務這種堆人的事情，因此也看到阿里雲技術團隊很多人出來創業，為企業實施中臺戰略，先不管中臺是否是忽悠，都明確的可以看到toB市場不僅是簡單的賣產品，更加重要的是提供諮詢+產品+實施落地的完整解決方案。
