java腳本引擎的設計原理淺析

2020-10-29
| 科學

QLExpress是一個腳本引擎工具，類似Groovy，JRuby等，是為了解決當時電商規則動態編譯、表達式高精度計算、複雜布爾運算、自定義函數和操作符號、語法樹生成等需求而設計的。QLExpress項目開源自2012年，截至目前已經迭代了60多個版本，（在阿里的專有開源社區 index - Taocode ）後在社區和集團內部卻意外獲得非常好的反響。

然而，我要很不客氣的講，在眾多優秀的腳本引擎工具list中，QLExpress還很不完美。本篇結合自己的工作和技術研究，列舉其中一些比較有意思的。

一、分類標準

1、編譯型 vs 解析性

如果能夠產生一個獨立的class文件則屬於前者，例如：fel，simpleEl，groovy

否則通過編譯成自定義的內存指令就屬於後者，例如：QLExpress，aviator，JEXL

2、java語法 vs 表達式語言(EL expression language) vs 腳本(script)

如果語法和java保持一致，不做任何擴展，就是屬於第一種：

如果語法大量簡化（比如去掉顯示類、方法、變量聲明，異常處理，邏輯跳轉循環等等），只支持簡單的數學公式、對象方法成員變量調用， 就屬於第二種：fel，simpleEl，aviator

介於兩者之間，即提供很好的語法糖，又支持大部分java語法：for循環，if判斷，函數定義，就屬於第三種：groovy，QLExpress

3、靜態類型 vs 動態類型 vs 強類型 vs 弱類型

區別看這裡： 弱類型、強類型、動態類型、靜態類型語言的區別是什麼？

假設兩個都很成熟的腳本引擎，強類型可以比弱類型提高一個數量級的運算效率，但是帶來業務的靈活性也就差很多。比如fel、simpleEl比Groovy、QLExpress效率肯定要高一個量級，因此，Groovy也提供了靜態類型的模式犧牲部分靈活性來提高性能。

二、設計思路

思路一：通過文本替換成java源碼文件。

編譯時，使用 jdk工具（ javax.tools.JavaCompiler ）或者 eclipse工具 動態編譯成class文件

運行時，取出class文件放入jvm直接運行

思路二：通過 antlr 等 語法解析器 把文本流解析成抽象語法樹（AST）。

編譯時，把AST的節點重新編排成class文件或者指令集。

運行時，把其中的操作符映射成自定義的執行器。

三、看一些案例

1、simpleEL

作者：溫紹錦(高鐵@alibaba) https://github.com/alibaba/simpleel

demo代碼：

DefaultExpressEvalService service = new DefaultExpressEvalService();service.regsiterVariant(Integer.class,”a”,"b");//註冊數據類型service.setAllowMultiStatement(true); //設置支持多行語句Map<String, Object> ctx = new HashMap();ctx.put("a", 2);ctx.put("b", 1);String dsl = "if (@a > @b) { return @a - @b; } else {return @b - @a; }";Object result = service.eval(ctx, dsl);//return 1

內部原理剖析：

if (@a > @b) { return @a - @b; } else { return @b - @a;}

通過@標識變量，直接轉化為一下的java源碼，然後再轉化為class文件

再比如：

DefaultExpressEvalService service = new DefaultExpressEvalService();service.regsiterVariant(MyBean.class,"bean");Map<String, Object> ctx = new HashMap();ctx.put("bean", new com.taobao.el.MyBean());String dsl = "@bean.getId() + ':' + @bean.getName()"Object result = service.eval(ctx, dsl);

特點:

簡單如其名，最最simple的EL，源碼實現也極其簡單。

依賴變量標識和強類型註冊，不借助任何語法解析工具，就實現了非常高效簡單的EL計算。

因為太簡單，所以後續功能開發停滯了，2014-9-17後停止更新。

2、fel

作者：未知 https://code.google.com/archive/p/fast-el/

demo代碼

FelEngine fel = new FelEngineImpl();FelContext ctx = fel.getContext();ctx.set("首重價格", 10.5); ctx.set("續重價格", 2.5);ctx.set(“首重",5); ctx.set("總重量", 15);String dsl = "首重價格*首重+續重價格*(總重量-首重)";Expression exp = fel.compile(dsl, ctx);//必須變量全部注入後，才能編譯通過Object result = exp.eval(ctx);System.out.println(result);

內部原理剖析：

首重價格*數量+續費價格*(首重-數量)

使用antlr 對文本流進行語法分析，轉化為AST之後生成簡單的java源碼

特點：

和simpleEL相比，這是一個非常追求極致效率的作品，它能識別重複變量，開提供function註冊、SourceBuilder等編譯後期優化，基本可以保持到和java源碼靜態編譯後只低一個數量級的效率。

靜態類型，不支持多行表達式，new對象等，2013-3-26後停止更新

3、Groovy

大名鼎鼎的開源項目： The Apache Groovy programming language

demo:

GroovyClassLoader groovyLoader = new GroovyClassLoader();Class<Script> groovyClass = (Class<Script>) groovyLoader.parseClass("a+b*c");Script script = groovyClass.newInstance();Binding bind = new Binding();bind.setVariable("a", 1);bind.setVariable("b", 2);bind.setVariable("c", 3);script.setBinding(bind);Object result = script.run();//return 7

特點：

由於大規模的使用和發展歷史悠久，groovy功能強大到無與倫比，本身和java源碼文件無縫兼容，又提供一大堆的語法糖和功能庫。

4、QLExpress

作者:強輝、包行傑 alibaba/QLExpress

ExpressRunner runner = new ExpressRunner();DefaultContext<String, Object> context = new DefaultContext<String, Object>();context.put("a",1);context.put("b",2);context.put("c",3);Object result = runner.execute("a+b*c",context,null,true,false);

特點：

從阿里巴巴的電商業務土壤中開始發展，融合了眾多奇葩卻又非常普遍的業務特性。

從2012年2.0版本開始開源，到現在一直在不斷的迭代功能和新特性。

弱類型動態類型，高精度數值計算，複雜邏輯運算定製，函數、操作符的重載、覆蓋，對象方法的攔截重定義，語法樹、變量、函數定義輸出等等。

腳本本身具備線程安全，高性能高併發的特性。

和同類型的groovy相比：

1、groovy比qlExpress更兼容java語法

相對功能複雜處理過程的大段java腳本，groovy可以直接拷貝過來運行，

而qlexpress的語法很輕量，對原始的java代碼有一定的兼容性問題，一般需要把數據的類型聲明全部去掉，同時不支持異常處理、for循環的集合寫法 等

2、qlExpress比groovy更強調功能擴展

因為qlExpress就是誕生於阿里的電商系統，定製了很多特別的常用功能需求（宏定義，語法解析，公式計算，布爾邏輯處理，操作符函數的內置替換），可讀性和功能更貼合業務需要，詳細看qlExpress的擴展能力部分

3、qlExpress比groovy實現更簡潔，操作系統以及應用領域更廣

目前我們的qlExpress因為所有的語法解析、運算執行都自己編碼實現，依賴實現極其簡單，在阿里的新零售領域，在大量的低版本android系統的pos機上都有大規模使用，穩定性非常好。

4、qlExpress和groovy性能相當

qlExpress和groovy同屬弱類型語法，比如a+b，可以在運行時支持字符串，數字等多種計算模式，相比fel，simpleExpress 等強類型語言性能會差一個數量級。

他們都支持編譯期做了緩存功能，qlExpress轉化為InstructSet，groovy轉化為一個特殊的groovyclass子類

文章總結

限於篇幅，本文沒有再枚舉更多的腳本引擎，對groovy和QLExpress也沒有充分展開原理介紹，最後只表述本人的對這個行業的一些看法。

1、如何自己造輪子：基於jvm的java語言的腳本引擎本身門檻並不高。

我見過好幾個case：開發能力比較強的團隊在瞭解 jdk字節碼生成工具，antlr語法生成工具，深入閱讀一些編譯原理文章之後，都可以在1-2月內開發出一個腳本引擎，滿足業務述求。

但是，往往會因為與業務系統耦合、功能太少、通用性太差、不具備開放性等原因逐漸消亡，無法大範圍推廣。

2、如何選輪子：性能、穩定性、語法靈活性綜合考慮

某種角度來說，腳本引擎因為可以動態解析執行，往往成為業務配置化的一部分。

業務控制邏輯越複雜，越需要流程引擎，規則引擎，腳本引擎一類的東西來編排業務。

作為業務架構師，底層一定要選擇一個靠譜好用的腳本引擎工具。

性能上:

 大眾化工具>小眾化工具(內部工具實現上有很多可以優化的點) 強類型 > 弱類型（在兩者都很成熟穩定的情況下）

穩定性上:

 簡單的設計原理>強大的開源社區>小眾化工具引擎

語法的靈活性:

 代碼越多、越複雜的實現方式功能會涵蓋越多，語法糖、容錯性、兼容性
