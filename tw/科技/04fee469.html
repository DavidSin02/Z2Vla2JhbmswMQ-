老生常談，來深入聊聊JS作用域

2020-11-14
| 科技

前言

最近重拾了一下JavaScript的一些基礎，作用域就是其中一個比較重要的概念。理解它有助我們更好理解JavaScript代碼的執行。我們知道，儲存變量值幾乎是所有編程語言最基本的功能之一，可以對變量的值進行訪問和修改，正是這種儲存和訪問變量的能力使得程序有了狀態。那麼變量是如何儲存，並且是如何去訪問它們的？這些問題也說明了我們需要一套設計良好的規則來存儲變量，並且之後可以方便的訪問它們。沒錯，這套規則就是我們所說的作用域了。下面我們探討一下，JavaScript是如何設置這些作用域規則的。

先講講編譯

JavaScript是一門動態編譯的語言，在講作用域之前，簡單的講講傳統編譯語言一般會經歷的三個步驟：

  • 詞法分析
    此過程會將由字符組成的字符串分解成有意義的代碼塊，即所謂的詞法單元（token）。例如，var a= 1；這句簡單的賦值語句，會被分解成：var、a、=、2、；等這些詞法單元。結構大致如下，會生成tokens數組，其中每個token是詞法分析的最小單元，不能再分解
var a = 1;tokens:[    { "type": "Keyword","value": "var" },    { "type": "Identifier","value": "a" },    { "type": "Punctuator","value": "=" },    { "type": "Numeric","value": "1" },    { "type": "Punctuator","value": ";" }]
  • 語法分析（Parse）
    此過程是將詞法單元流即上面的tokens數組，轉換成一個由元素逐級嵌套所組成的程序語法結構的樹，即抽象語法樹（AST）。在這個過程中會校驗語法，有錯誤會拋出語法錯誤。

大致結構如下：

// AST的結構大致如下：{    type,    sourceType,    start,    end,    ...    // body是一個數組，包含多個內容塊對象，即statement，類型大致有：變量聲明，函數定義，if語句，while循環等。    body: [{        type,        start,        end,        kind,        // 變量的內容塊，也是個數組        declarations:[            {                type,                start,                end                ...            }        ]    }]}var a = 1;// AST結構如下{    "type": "Program",  "start": 0,  "end": 12,  "body": [      "type": "VariableDeclaration", //變量聲明      "start": 0,      "end": 10,      "declarations": [        {           "type": "VariableDeclarator", // 變量聲明          "start": 4,          "end": 9,          "id": {            "type": "Identifier", //標識符            "start": 4,            "end": 5,            "name": "a"          },          "init": {            "type": "Literal",            "start": 8,            "end": 9,            "value": 1,            "raw": "1"          }        }      ],      "kind": "var"    }  ],  "sourceType": "module"}
  • 代碼生成
    將AST轉換為可執行的過程被稱為代碼生成，當然這個過程比較複雜，這裡我們只是有個大概的意識，簡單的講就是可以將 var a = 1; 的AST轉化為一組機器指令，用來創建一個叫做 a 的變量，分配內存並將一個值儲存在a中。

這裡講編譯只是大概的說明了解下代碼大致編譯過程，給我們有一個整體的代碼運行的流程，簡單瞭解引擎可根據需要創建存儲變量即可，當然JavaScript編譯過程要複雜得多，筆者也只是簡單瞭解皮毛而已，沒關係，我們這裡討論的重點還是作用域哈哈。

對於JavaScript來說，大部分情況下編譯發生在代碼執行前的幾微秒，甚至更短的時間內，比如對簡單的一句賦值語句 var a= 1，編譯器會先進行編譯，做好執行它的準備，然後會馬上執行它。


理解作用域

1、三部曲

在JavaScript中對程序處理執行，一般是由引擎，編譯器，以及作用域這三個配合完成的。

  • 引擎
    負責JavaScript程序的編譯以及執行過程。
  • 編譯器
    負責語法分析以及代碼生成等。
  • 作用域
    負責收集維護所有聲明的標識符（變量）組成的一系列查詢，確定代碼執行時對這些標識符的訪問權限。

它們三者之間會協同工作，一起來完成JavaScript代碼的執行。

比如處理var a = 1;這段簡單的程序，大致是這麼處理的：1、引擎編譯執行的時候，遇到var a，編譯器去會詢問作用域是否已有該變量存在其中，若有編譯器會忽略該聲明繼續編譯，否則會在作用域中聲明一個新的變量，並命名為a。2、接下來編譯器會為引擎生成運行時所需的代碼，這些代碼被用來處理 a = 1 這個賦值操作，引擎運行時會首先詢問作用域，在當前的作用域集合中是否存在一個叫做 a 的變量，若有引擎就會使用這個變量，否則，引擎就會繼續查找該變量，最終找到就會賦值給它，若沒有找到則會拋出異常。

總結：變量的賦值操作會有兩個步驟：1、編譯器在當前作用域聲明一個變量，已聲明則忽略；2、運行時引擎會在作用域查找該變量，若找到就會進行賦值，否則拋出異常。


2、變量查詢

在上述瞭解代碼執行的大致過程，我們知道引擎在執行代碼時會通過查找變量a來判斷它是否已聲明過，而查找的過程中由作用域進行協助的，但引擎進行怎樣的查找，會影響最終的查找結果。

比如在上述例子中，引擎會為變量a進行LHS查詢，還有另外一個查找類型叫做RHS。顧名思義，“L”和“R”的含義，分別是左側和右側，即當變量出現在賦值操作的左側時進行LHS查詢，出現在右側時進行RHS查詢。只是從字面這麼講，但不一定意味著就是賦值操作符的左側或者右側。

下面代碼來講講LHS與RHS的區別：

var a = 1; // 明顯是LHS查詢console.log(a); // 這裡對變量a的查詢就是一個RHS查詢

我們可以看到console.log(a)中的變量a，並沒有賦予任何值，只是需要查找並取得a的值，才能將值傳遞給console.log()；而對於a=1;來說，對a的查詢則是LHS查詢，這裡我們可以理解成只是要為=1這個賦值操作找到一個目標。

因此，我們可以進一步理解：去查找賦值操作的目標是誰就是LHS查詢，而去查找已知目標（變量）是在哪裡進行賦值操作的則是RHS查詢

我們再看下面代碼：

function foo(a) {    console.log(a);}foo(2); // 2

這段代碼首先聲明瞭一個函數foo，接著調用它，這裡的函數調用就是對foo進行RHS查詢，意思是去查找foo的值，並把它給我，並且它最好是一個函數類型的值。

其實這段代碼也有LHS查詢，那就是代碼中隱式的a = 2賦值操作，因為在foo調用時，2被當作參數傳遞給foo函數，此時分配給了參數a，這裡需要一次LHS查詢，接著console.log(a)就是我們上面說的RHS查詢。

接下來我們來捋一捋上面那段代碼在執行時會發生什麼：1、引擎會在當前作用域中對foo進行RHS查詢；而編譯器編譯過程中已經聲明瞭foo函數，於是引擎找到了foo；2、接著繼續在當前作用域對a進行LRS查詢，編譯器將a聲明成了函數foo的一個形參了，於是也找到了a;3、以此類推，最終引擎完成了這段代碼的解析執行。


3、作用域嵌套

接著我們在上面那段代碼基礎上再加一個變量：

function foo(a) {    console.log(a + b);}var b = 2;foo(2); // 4

上面代碼中多了一個變量b，這裡對b進行的是RHS查詢，但此時無法在函數foo內部完成，不過可以在其上一級作用域即全局作用域中完成。我們可以得出一個規則：引擎從當前執行的作用域中開始查找變量，若找不到就會逐級往上查找，當抵達最外層作用域時，無論找沒找到，查找過程都會停止。其實，這種嵌套作用域一級一級的查找就是我們常說的作用域鏈了。這種作用域鏈就好比一幢高樓，引擎執行所在的作用域就是當前樓層，LHS和RHS查詢都會在當前樓層進行查找，若沒找到，就會上一層樓層（作用域）去查找，直到頂層（全局作用域）。



異常

上面我們瞭解的LHS和RHS查詢變量的一些簡單規則，也瞭解到兩種查詢規則的行為是不一樣的，尤其是變量還沒有進行聲明的情況下，如以下代碼：

function foo(a) {    console.log(a + b);  b = a;}foo(2);

上面代碼在對b進行RHS查詢時時在所有作用域中是無法找到該未聲明的變量的，此時引擎就會拋出ReferenceError異常。

相比較之下，當引擎執行LHS查詢時，如果直到在全局作用域都沒有找到該目標變量，那麼在非嚴格模式下就會創建一個具有該名稱的變量，並將其給到引擎，注意是在非嚴格模式下，如果是在嚴格模式下，則會跟RHS查詢那樣拋出一樣的異常。

接下來，如果RHS查詢找到一個變量，但是你執行了不合理操作，比如對一個非函數類型的值進行函數調用，或者引用null或undefined類型的值中的屬性，引擎就會拋出TypeError異常。


總結

作用域其實就是一套規則，用於確定在哪裡以及如何查找變量，如果查找的目的是對變量賦值，那麼就會使用LHS查詢，如果目的是獲取變量的值，就會使用RHS查詢。並且這兩種查詢都會在當前作用域中開始，找不到時會向上一層作用域繼續查找目標變量，直到全局作用域為止。這樣一層一層的查詢作用域，構成了我們所說的作用域鏈。不成功的RHS查詢會導致拋出ReferenceError異常，而不成功的LHS查詢會導致自動隱式創建一個全局變量（非嚴格模式下），在嚴格模式下會同RHS查詢一樣拋出異常。
