計算機網絡: 鏈路層(入門)

2020-11-14
| 科技

原文：https://www.jianshu.com/p/0e2d800b1520

術語

  • Nodes：主機和路由器
  • Links: 負責連接相鄰兩個節點的連接
  • 有線網
  • 無線網
  • 局域網
  • Packets: frame（鏈路層對包的表述），它主要封裝了上一層 datagram

每層的 Packet 形式

Packet 就是網絡中傳遞的信息，但在不同層它會被封裝成不同形式。下面舉例了 3 層。

鏈路層是幹什麼用的

鏈路層是網絡結構中的倒數第二層，最後一層是物理層，一般屬於硬件部分，所以就從鏈路層開始說起。鏈路層提供了很多的服務（幹了很多事）。

Framing

很重要的一個功能就是將上一層的 datagram (packet的一種形式) 封裝成 frame。並在這個信息前後添加了 Header 和 Tailer。這裡的 Header 一般包含了兩臺正在溝通設置的 Source 和 Destination 地址。

Link Access

既然是"鏈"路層了，肯定也要提供鏈接服務嘛。這一層主要提供兩個：

  1. Point to Point，兩臺機器直接溝通
  2. Shared Medium，也就是我們所說的廣播，一個人在微信群裡喊

Flow Control

想像一下家裡的水管，如果突然供水太多，那就爆水管了，網絡也是這樣。信息傳遞的時候需要用一些機制去控制傳輸速度。這個機制就叫 Flow Control，具體方法可以使用 Buffer 來做一個緩衝器。

Reliable Delivery

除了節流，這一層還負責保持信息的穩定傳輸。傳個信怎麼會出錯呢？因為現在是在底層，所以很多因素都會影響傳輸質量，如信號干擾，網絡不穩定。想一想平時有沒有出現看視頻掉幀的情況，還有耳機出現的噪音，這都是對信息傳遞干擾。所以鏈路層提供一些服務來保證傳遞質量，這些機制主要也是要使用於不同網絡的如有線或者無線。

鏈路層在哪

說了一堆鏈路層的作用，好像太過抽象，不如說說具體位置。其實鏈路層在每臺主機都有，它主要是在 Network Interface Card(NIC)，也就是我們的網卡。

鏈路層的信息溝通

發送方

  1. 將 datagram 封裝成 frame
  2. 添加 error bits, flow control, error recovery 等元信息

接收方

  1. 先看元信息，一然後做準備工作，如：如果發現錯誤就把這個包扔了，讓發送方再發一次（retransmission）
  2. 然後將 frame 提取成 datagram，再向上傳遞

錯誤機制

就像剛剛說到的，傳遞信息過程中一定會出現錯誤的，鏈路層對此有兩種機制：

  1. Error Detection
  2. Error Recovery

Error Detection

接收方會去檢測錯誤率有多少，根據錯誤率去決定是否恢復數據還是讓發送方重傳。雖然說會檢測錯誤，但是不是所有錯誤都能檢測到的，有可能 miss 掉。其中EDC（Error Detection and Correction）越大，越容易檢測到。

Error Recovery

錯誤恢復機制一般是兩個

Correction

接收方會自己去改正對應錯誤的 Bits，但是需要大量的 EDC bits，才能做到 Correction，畢竟更多的信息才能正確去改正嘛。一般用於那些不想等待 retransmit 的數據

Retransmission

接收方會丟掉一些包，發送方就會重傳。具體操作方法是直接超時，發送方得不到 ACK（就是一個接收完返回來的信號），發送方就得重傳。或者，接收方直接發一個 Netgative ACK，發送方也得重傳。

多路傳輸協議

不知道這讓翻譯對不對，英文是叫 Multiple access links protocol。在說這個協議有什麼用之前，先說說以前的傳輸方式。

傳輸方式

  1. Point to Point，就是
發送方 <-> 接收方

，這和微信找好友聊天是一樣的。一般是有線網絡或者無線網絡。 2. Broadcast，就是

發送方 <-> 多個接收方

，這和群聊是一樣的。一般是WIFI或者老的以太網使用。

這個協議有什麼用

上面說的兩個傳輸方法都是在一個總線去完成消息傳遞的。所以總會出現同一個 Channel 中你在傳數據的同時，別人也在傳數據。不要想著什麼多線程，這裡只能一個一個來，所以就有了多路傳輸協議，主要是用於多人傳的時候決定誰先傳誰後傳的問題，最後做出一個任務表出來。

有人一聽到協議就不想看了，又要寫 C，C++搞協議，好難啊。嘿嘿，其實這個協議主要理念特別智障，細節很難（也就是寫C，C++那塊，好在我們瞭解一下就行了）。常用的有：

  1. Channel Partitioning
  2. 將 Channel 分成更小塊 (time slots, frequency, code)
  3. 分配某一塊給那個節點使用
  4. Random Access
  5. 不分配，允許衝突
  6. 但是會在衝突以後進行恢復操作
  7. Taking Turns
  8. 節點會輪流發送信息，但是如果某個要發很多信息，就會佔很長時間

CSMA

這裡主要說說 Random Access 裡的 CSMA 做法，為什麼只拿這個來說呢，因為我們老師就講了這個啊。

主要理念很簡單：如果這個 Channel 空閒，我就傳輸，否則等到空閒再傳。是不是很智障？聽起來是的，細節反正不管了。

但是這個方法也有問題，如果兩個節點特別遠，自己想傳數據的時候別人已經開始傳了，但是因為太遠了我沒聽見別人說"我開始傳了，別人都不許動"，還以為這個 Channel 還是空閒的，所以我傳了數據，這就造成衝突。當衝突發生後，又要重傳。可以我都把一個包的信息都發過去了呀，這整塊時間就白白浪費掉了。

上面的交集部分就是衝突後浪費掉的時間。

CSMA/CD (Collision Detection)

這個算是增強版的 CSMA。以前的 CSMA 是兩個人發完了再去重傳，現在這個不一樣了。在發送數據過程上，會去檢測是否有衝突，如果有就放棄傳輸，等待下一次。畢竟還是要等一段時間才能檢測出是否有衝突，所以依然有時間被浪費，但是會少一些。

關於怎麼檢測是否有衝突，可以檢測信號的長度，對比傳輸和接收的信號（但是侷限於距離和更小的 frame），可以在無線局域網比較難，接收信號距離太長。

NIC 接收從網絡層 datagram，並創建 framelabel: step2if (channel Idle): 開始傳輸 frameelse if (channel busy): 等待直到 Idleif (檢測是否有別的 transmission): abort wait K * 512 bit 的時間(K 是一個隨機數) goto step2else transmit

以太網

局域網裡最出名也就是以太網了。電腦之間的聯繫基本為下面兩種：

使用一條或幾條總線。

或者一個主機，路由，現在一般是這種方式。

這裡說一下 Switch 和 Hub 的區別。

  1. Switch: 節點傳輸不會有衝突，使用 Buffer 去做傳輸隊列
  2. Hub: 每個節點會 Broadcast 信息，不使用 Buffer，有可能出現衝突

以太網的信息結構

我們知道在 HTTP 請求的時候總會有請求頭，以太網的傳輸也是一樣，也有頭部信息。

preamble: 用來同步接收方和發送方的 Clock Rates addresses: Source 和 Destination 地址 type: 表示這個 frame 裡最高層次是哪一層 CRC: 用於檢測錯誤

以太網的特性

兩個詞：unreliable, connectionless

  • connectionless: 發送方和接收方沒有握手流程
  • unreliable: 接收方的 NIC 不會發送 ACK 和 NACK
  • 如果數據被丟了就不會做恢復
  • 自己偷懶，更依賴於高層的數據傳輸

設備接口地址

啊，這是什麼鬼東西，Device Interface Address 就是 IP 地址。

MAC 地址

我們可以會在家裡的路由器看到這個詞，MAC 地址就是該設備唯一的編號，IP 只是網絡裡的唯一編號。

這麼說吧，IP 就像你的住址，你可以今天在北京買套房，你的IP就是北京了，明天去上海買套房搬去那邊住，IP 就變成了上海。雖然說能變，但是你沒有美國綠卡，再怎麼變也不能逃不去中國，所以中國就是你所在的網絡。說不定日本也有叫北京的地方，但是你們處於兩個國家（兩個網絡），不會造成影響。

但是 MAC 就不一樣了，就你的身份證一樣，一直跟隨著你。

ARP

那這個 MAC 有什麼用呢？可以用來標記你在哪呀，別人發數據給你的時候是一定要不僅知道你的 IP，也要知道你的 MAC 的。你可能會問了，大哥我的 MAC 是寫在路由器上的，難道你來我家看嗎？電腦可不這麼傻。

他會先廣播一個叫 ARP 的請求，大概意思是："那個XXX，給我出來！"。XXX 聽到後就說："我在YYY這呢，你有種過來幹我。"。

再給個具體的鴿子：

比如 A 想發信息給 B，下面是前提條件

  1. B 一定要在同一個局域網裡，而且 A 知道 B 的 IP
  2. A 不知道 B 的 MAC 地址

具體操作：

  1. 首先 A 在本地網絡裡去 Broadcast 一下，這個請求就是 ARP 請求包，其中包含了 B 的 IP
  2. 本地的全部節點都會接收到這個 ARP 請求，並處理
  3. 如果 B 此時也接收到這個請求，就會發 ARP replay 給 A，同時會帶上 B' 的 MAC 地址。
  4. 這個過程都是在 Link Layer 完成的

還有一些問題

問：如果給自己發 ARP 會怎麼樣？ 答：不會怎麼樣，一般是沒有響應罷了。不過這樣可以檢測同一個網絡中是否存在別的主機的IP是否和自己一樣。

問：如果給不存在的主機發 ARP 會怎麼樣？ 答：也不會怎麼樣，最後就失敗唄
