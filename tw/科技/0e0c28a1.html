1-Spring IoC簡介

（1）IoC是一個容器，在Spring 中，它會認為 Java 資源都是Java Bean，容器的目標就是管理這些Bean和它們之間的關係。

（2）所以在Spring IoC裡面裝載的各種Bean，也可以理解為Java的各種資源，包括 Java Bean 的創建、事件、行為等，它們由IoC容器管理。

（3）除此之外 ，各個Java Bean之間會存在一定的依賴關係，假設老師、學生都是 Java Bean ，那麼顯然二者之間形成了依賴關係，老師和學生有教育和被教育的關係。

（4）這些 Spring IoC 容器都能夠對其進行管理。只是Spring IoC管理對象和其依賴關係，採用的不是人為的主動創建，而是由 Spring IoC自己通過描述創建的，也就是說Spring是依靠描述來完成對象的創建及其依賴關係的。

（5）比如插座，它依賴國家標準（這個標準可以定義為一個接口， Socket ）去定義，現有兩種插座（ Socket1和 Socket2 ），如圖1-1 所示。有兩種插座可供選擇，具體使用哪種呢？我們可以通過代碼來實現使用插座1（Socket1）：

Socket socket= new Socketl() ;

user.setSocket(socket) ;

user.useSocket() ;

使用 Socket socket= new Socket1()；後，國家標準插座接口(Socket)就和插座1(Socket1) 捆綁在一起了 。這樣就會有一個弊端：如果要使用其他的插座，就需要修改代碼了 。這種情況 socket接口和其實現類 Socket1耦合了，如果有一天不再使用 Socket1，而是要使用Socket2 ，那麼就要把代碼修改為如下的樣子:

Socket socket= new Socket2();

user .setSocket(socket) ;

user . useSocket() ;

如果有其他更好的插座，豈不是還要修改源碼？一個大型互聯網的對象成千上萬，如果要不斷修改，那麼對系統的可靠性將是極大的挑戰， Spring IoC 可以解決這個問題。

首先，我們不用new方式創建對象，而是使用配置的方式，然後讓Spring IoC容器自己通過配置去找到插座。先用一段XML描述插座和用戶的引用插座1:

<bean id="socket" class="Socket1" />

<bean id="user ” class="xxx.User" >

<property name="socket" ref="socket" />

</bean>

請注意這些不是 Java 代碼，而是 XML 配置文件，換句話說只要把配置切換為：

<bean id="scocket " class= "Socket2 ">

就可以往用戶信息中注入插座 2，切換插座的實現類十分方便 。 這個時候Socket接口就可以不依賴任何插座，而通過配置進行切換，如下圖所示：

（6）這是一種被動的行為，而需要的資源（Bean）通過描述信息就可以得到，其中的控制權在 Spring loC 容器中，它會根據描述找到使用者需要的資源，這就是控制反轉的含義 。

（7）這樣的好處是 Socket 接口不再依賴於某個實現類，需要使用某個實現類時我們通過配置信息就可以完成了。這樣想修改或者加入其他資源就可以通過配置完成，不需要再用 new關鍵字進行創建對象，依賴關係也可以通過配置完成，從而完全可以即插即拔地管理它們之間的關係。

（8）你不需要去找資源，只要向 Spring loC 容器描述所需資源， Spring loC 自己會找到你所需要的資源，這就是 Spring loC 的理念。這樣就把 Bean 之間的依賴關係解耦了，更容易寫出結構清晰的程序。除此之外， Spring loC 還提供對Java Bean生命週期的管理，可以延遲加載，可以在其生命週期內定義一些行為等 ， 更加方便有效地使用和管理 Java 資源，這就是Spring loC 的魅力 。
