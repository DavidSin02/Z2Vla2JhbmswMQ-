1前言

最近和很多人（阿里的，騰訊的......）聊PostgreSQL，發現大家對於PG最大的認同點是他的詞法分析和語法分析做的非常好，認為是替代Oracle的最佳方案。本人最開始接觸PostgreSQL開發，便是做的兼容。經過多年的學習，比較認同新手接觸PostgreSQL可以先通過系統函數、語法等方式入手。討論了許多，正好將自己對於這方面的瞭解整理一下，也算對自己當年有個交代。產品的兼容代碼由於是商業祕密，在這裡無法公開，這裡會以PG已有內容或開源代碼進行介紹實現方式。

2為什麼要做兼容

PostgreSQL本身就有很完善的SQL體系，為什麼要做兼容？尤其是為什麼要做Oracle的兼容？最主要的原因我認為是Oracle經過多年的市場耕耘，造就了一大批基於Oracle的開發人員。目前市場上很多應用都是基於Oracle開發的。這就對數據庫遷移產生了非常大的阻礙。國內這幾年一直在喊去IOE，但是效果了了。從亞馬遜完成Oracle遷移後，進行慶功就很能說明其難度之大。遷移的不僅僅是數據庫，而是整個數據庫生態。從應用端開發，到數據庫管理，都需要做出改變。同時，在這裡不得不說一下Oracle確確實實在數據庫生態方面做了非常多的工作，其他數據庫難以望其項背。為什麼要做兼容？尤其是Oracle兼容？因為目前PostgreSQL在國內的推廣程度較小，酒香其實也怕巷子深。可以通過兼容Oracle擴大市場，推廣PG。其次，Oracle成功有他的道理，他的生態是非常的完善，學習也是值得去做的。如果完成了Oracle兼容，那麼我們可以做到：

  1. 更快、更好地應用遷移；
  2. 完成部分機構的數據庫要求；
  3. 減少新語法的學習成本；
  4. 完成數據庫生態。

3兼容級別

那麼如果我們要做兼容，那麼應該怎麼做？兼容也不一定是一味的完全一模一樣，重複造輪子是沒有價值的。我將兼容級別劃分為三類：

  1. 完全相同，使用方法和功能完全一致，最終達到用戶不需要改動任何應用程序，即可使用；
  2. 部分相同，使用方法一致，但實現的功能有部分不同，可以針對用戶需求完成功能，減少時間消耗；
  3. 完全不同，實際解決的問題相同，但使用方法不相同，Oracle有很多功能是基於其整體架構的，所以如果想這部分內容，成本非常大，但不如找到根本問題，通過其他方法解決。

這三種級別也不是完全獨立的，我們可以有選擇性地對兼容內容進行不同級別的兼容開發。比如系統函數我們就可以做的一模一樣，而部分類型可以選擇部分兼容。

4兼容開發方式

我們有以下2種方式開發兼容功能：一是完全開發，組織團隊，根據需求或者Oracle官方文檔（這裡吐槽一下，實際使用可能存在比文檔展示的方法更多）一一對比，完全實現。二是部分開發，基於開源組件二次開發（比如Orafce），或者購買EDB動態庫或者兼容代碼（資金雄厚的儘可能選擇此類）。

6共享組件

共享組件提供一系列的工具共數據庫管理使用，旨在提高數據庫管理便宜性。比如AWR（Automatic Workload Repository），能夠提供更多的數據庫監控信息，供DBA使用，快速定位、解決數據庫問題。遷移工具：完成數據結構、數據遷移以及存儲過程、函數轉換。此類內容對於數據庫遷移或者應用開發影響甚小，其作用主要在於便於數據庫遷移、管理。內容非常多，這裡就不一一列舉了。

7語法

語法，也就是我們常說的SQL，主要涉及到的內容是編譯原理。PostgreSQL使用Flex和Bison兩個工具完成的語法實現。Flex和Bison是Linux下兩個用來生成程序的工具，可以處理結構化輸入，它們生成的程序分別叫做詞法分析器和語法分析器，他們一般結合使用來處理複雜的文件解析工作，在PostgreSQL中主要是用來生成SQL語句的詞法和語法分析器。

Flex & Bison

Flex（快速詞法分析器生成器）是lex的免費開源軟件替代品。它是生成詞法分析器（也稱為“掃描器”或“詞法分析器”）的計算機程序。它可以利用正則表達式來生成匹配相應字符串的C語言代碼，其語法格式基本同Lex相同。經常連同Yacc或GNU Bison一起使用。Postgresql 中的Flex 文件為scan.l，通過Flex編譯為scan.c。

Bison是一種通用解析器生成器，它將帶註釋的上下文無關文法轉換為使用LALR（1）解析器表的確定性LR或廣義LR（GLR）解析器 。作為一項實驗性功能，Bison還可以生成IELR（1）或規範的LR（1）解析器表。一旦您精通Bison，就可以使用它來開發各種語言解析器，從用於簡單臺式計算器的語言解析器到複雜的編程語言。Bison與Yacc向上兼容：所有正確編寫的Yacc語法都應與Bison一起使用，而無需進行任何更改。熟悉Yacc的任何人都應該可以輕鬆使用Bison。您需要精通C或C ++編程才能使用Bison。還支持將Java作為實驗功能。Postgresql 中的Bison 文件為gram.y，通過Bison編譯生成gram.h，gram.c。

PostgreSQL中語法工作各個文件之間的調用關係：

  • scan.l：詞法分析器在文件，定義詞法結構，編譯生成scan.c
  • gram.y：語法分析器在文件，定義語法結構，編譯生成gram.c和gram.h；
  • parser.c：raw_parser函數，主要通過調用生成的base_yyparse函數來實現詞法分析和語法分析的工作。
  • kwlist.h：SQL關鍵字定義，注意：關鍵字名要小寫，按照字符串值順序定義
  • kwlookup.h：定義結構體ScanKeyword；
  • kwlookup.c：使用kwlist.h初始化關鍵字數組ScanKeywords，提供ScanKeywordLookup函數
  • scanup.c：提供幾個詞法分析時常用的函數。

我們將SQL轉化為語法分析樹：

  1. scanner發現起始位置為‘select’，並確認為關鍵字SELECT，返回token值與數值；
  2. parser接收到scanner的返回值，將‘select’移進，發現屬於stmtblock->stmtmulti->stmt->SelectStmt->select_no_parens->simple_select；
  3. scanner發現oid,relname
    1. 符合移進條件opt_target_list；
    2. 符合移進條件target_list；
    3. 其中oid在simple_select符合移進條件target_list->target_el->a_expr->c_expr->columnref->Colld->IDENT，即移進，這時發現符合歸約條件target_el，即歸約，返回target_el=oid；
    4. 其中relname在simple_select符合移進條件target_list->target_el->a_expr->c_expr->columnref->Colld->IDENT，即移進，這時發現符合歸約條件target_el，即歸約，返回target_el=relname；
    5. target_el=oid，target_el=relname歸約符合target_list，即歸約；
    6. target_list符合歸約條件opt_target_list，也進行歸約。
  4. scanner發現from，移進from未符合歸約條件，繼續移進pg_class，符合from_list->table_ref->relation_expr->qualified_name->ColId->IDENT，符合歸約條件，則返回from_list；
  5. scanner發現where
    1. 符合移進條件where_clause，移進where未符合歸約條件，繼續移進，發現符合移進條件a_expr；
    2. 發現符合移進條件a_expr '=' a_expr；
    3. 繼續移進relname，符合a_expr->c_expr->columnref->Colld->IDENT，符合歸約條件，則返回a_expr；
    4. 繼續移進‘pg_class’，符合a_expr->c_expr->AexprConst->Sconst->SCONST，符合歸約條件，則返回a_expr；
    5. 符合歸約條件，返回where_clause；
  6. 發現符合歸約條件，返回simple_select；
  7. 發現符合歸約條件，返回select_no_parens，SelectStmt，stmt，stmtmulti，stmtblock。

JIT

利用LLVM對SQL查詢計劃進行優化。隨著現在存儲技術的發展，尤其是內存、ssd硬件的發展，以及存儲結構，如列存，如分佈式，I/O瓶頸逐漸的在降低，而帶來的影響就是cpu的瓶頸來了。很多人可能會問cpu有什麼瓶頸？現代數據庫為了完成更多的用戶操作，有很多的冗餘操作。比如我們即便向數據庫輸入一個常量，數據庫還是首先以字符串的形式拿過來，進行處理，經過很多步驟後，確認是常量，然後放入相應位置。這中間就出現了很多人工就能減少的步驟，而數據庫或者計算機是不得而知的。Llvm是一款編譯器後端。他能將程序語言轉化為自己的中間語言，然後進行優化。它主要的作用就是減少cpu的指令數量，以及提高cpu cache利用率。如何理解llvm，如果大家瞭解java的話，可能就對JIT不陌生了。我們的Java之所以能夠一次編程，處處運行，就是因為我們的JDK就是一個解釋器。所有的java程序，都經過編譯，編譯為class文件。這個可以理解為java語言與機器語言的中間語言。而jdk可以將這中間語言重新解釋，並優化為機器碼，從而進行執行。pg現有的JIT技術也是相同的方式進行的。不同點在於，這裡使用的是llvm作為中間語言的編譯和解釋、優化再執行。
