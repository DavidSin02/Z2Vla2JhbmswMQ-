穩定性專題 | 通過鏈路追蹤快速發現和定位業務問題的實踐

2020-11-14
| 科技

引言

在可觀察性領域，Metrics，Tracing 和 Logging 的介紹由來已久。三種之間的邊界越來越模糊。OpenTracing 中已經支持 LogEvent，OpenTelemetry 已經把 OpenMetric 和 OpenTracing 整合到一塊。今天我們要介紹的鏈路追蹤的業務分析功能，通過對鏈路數據進行聚合統計，可以查看各種業務報表。

為什麼需要業務分析關聯鏈路數據？

精細化分析，全鏈路數據，快速發現和定位問題。

在傳統的 APM 分析中，主要是關注各種接口的響應時間，對業務不夠貼切，這樣會帶來一些問題：

發現問題

同一個接口被很多業務方調用，某一個重要業務調用出了問題，由於這個業務佔比不高，無法通過平均數或者 p90 分析出來。例如創建訂單中重要客戶，這些客戶訂單佔比不高，但是重要性比較高，需要重點分析和監控。

排查定位問題

某些用戶反饋下單或者付款出現問題，但是這種情況只存在少量用戶，重現比較困難。如果把用戶的異常和鏈路關聯起來，分析異常用戶的鏈路數據，比如出入參數，用戶屬性等，可以發現一些線索。例如查看用戶 ID 對應的失敗訂單，分析訂單上鍊路，發現失敗訂單的來源都是某個舊版的客戶端調用。

業務分析概述

在介紹業務分析功能之前，先簡單提兩個問題，大家發散思考下怎麼解決這兩個問題。1、某個應用的對外接口流量突增，是不是由於某個用戶或者地域流量暴漲？2、應用出現比較多空指針異常，這些異常對業務有多大的影響？

通用的監控系統主要是對操作系統，分佈式調用，數據庫組件進行監控。如果對業務監控，那就需要開發同學對調用鏈埋點添加業務屬性（用 tag 來記錄各種業務屬性）。通過對 tag 標籤進行過濾和統計聚合，達到業務分析和監控的效果。

業務分析的實現方式

業務分析是在鏈路上標註一些業務數據，以下我們用 Jaeger 和 Skywalking 為例子講解業務埋點。

Jaeger 中添加 Tag 信息

以 Java 語言為例，

// 獲取業務中的用戶IdString userId = ****;Tracer tracer = GlobalTracer.get();if (tracer != null && tracer.activeSpan() != null) {  // 將 UserId以 Tag 的形式存放到鏈路中  tracer.activeSpan().setTag("userId", userId);}

Skywalking 中添加 Tag 信息,不插入代碼的方式:https://github.com/apache/skywalking/blob/master/docs/en/setup/service-agent/java-agent/Customize-enhance-trace.md

<class class_name="test.apache.skywalking.testcase.customize.service.TestService2">     <method method="staticMethod(java.lang.String,int.class)" operation_name="/is_2_static_method" static="true">          <tag key="tag_2_1">arg[0]</tag>          <log key="log_1_1">arg[1]</log>     </method></class>

用 OpenTracing 插入代碼方式

// global tracer objectprivate static Tracer tracer = new SkywalkingTracer();... if (tracer != null && tracer.activeSpan() != null) {      tracer.activeSpan().setTag("userId", yourUserId);  }

Java 代碼方式:

ActiveSpan.tag("userId", yourUserId);

Java 註解方式

@Tag(key = "userId", value = "arg[0]")@Tag(key = "requestId", value = "arg[1]")public someMethod(final String userId, final String requestId) {    // ...}

對鏈路數據進行業務分析

Tag 信息就好比給圖書打上編號分類信息，讓我們可以在圖書館海量的圖書中找到我們需要圖書。通過 Tag 來進行分類、查找、統計，可以快速找到我們需要的數據。

業務標籤的過濾

可以通過篩選 Tag 進行過濾，尋找所需要的鏈路。如下圖，查看手機端下單的請求的變化趨勢，詳情等。

業務標籤的聚合

指定 Tag 進行聚合，分析 Tag 聚合的統計。如下圖查看每個入口（PC 和 Mobile）的請求數，響應時間，異常查詢。可以識別是不同業務的對比情況。

業務分析的一些使用場景案例

Tag 功能主要是業務開發同學用來進行業務監控的， 通過在鏈路上寫入 Tag，可以把業務和鏈路打通，從而很好的發現，定位，跟蹤問題。

我們接下來做一些業務場景舉例。

業務異常分析

生成訂單時，會調用優惠，庫存，物流，商品等系統接口。這些接口都有可能會返回異常，有些異常有些比較重要，有些只是用於展示。我們怎麼快速識別當前應用的異常嚴重程度呢？可以通過加 Tag 的方式來實現。我們對異常進行分 1～5 級，1 級表示非常嚴重。

那我們每次調用遠程接口完成後，對返回的異常添加 Tag，例如調用優惠時返回系統異常加 Tag： tracer.activeSpan().setTag("bizErr", “1-PromotionSysExcep”); ，調用庫存返回庫存不足，可以添加tracer.activeSpan().setTag("bizErr", “5-InventoryNotEmpty”);

那麼通過 Tag bizErr 進行聚合，對數量進行排序下，如果 1-** 開頭的比較多，那表示當前系統比較嚴重，需要緊急處理。查看這些異常鏈路的請求參數或者返回值，我們可以快速定位問題。

用戶分析

將 UserId 用 Tag 方式寫入到 Span 中，可以統計到沒有用戶的數據，例如訪問量，響應時間。我們之前遇到一個問題，Web 的流量突然變大，響應比較慢。從監控來看流量比較大，但為什麼流量會比較大呢，通過 UserId 分組統計後，發現某個用戶的流量暴漲，導致整個後臺響應比較慢，對這個用戶進行限流後，web流量和相應時間恢復正常。

業務大盤

中臺系統中各個業務的創建訂單大盤。

在中臺應用中，各個業務使用同一個交易系統，通過訂單中的一個業務類型字段來區分不同的業務。通過在鏈路上的標籤，可以查看到不同業務線的下單成功和失敗曲線。比如大盤上可以展示天貓，聚划算，盒馬生鮮等業務的訂單情況。一旦有異常就可以分析鏈路的明細信息來查找原因。

和傳統的統計不同點

傳統的 ELK 方式，用戶通過插入代碼，將業務的屬性都打到日誌裡面。通過數據聚合，可以生成各種 Metrics，添加告警等。這樣做監控是合適的，但是需要更深層次的挖掘問題的話，還需要和鏈路打通。將監控的內容上下游的信息都串起來，可以更方便的定位問題。

比如：更新庫存失敗，發現異常都是上游某個業務的某臺機器上，而這條機器是灰度機器，使用了非穩定版本的。

總結

業務排查的問題方式有很多種，調用鏈和業務信息打通已經是一種行之有效的方法。通過在調用鏈寫入業務信息，將會給調用鏈配上顯微鏡，快速發現隱藏在鏈路數據中的各種問題。而業務數據關聯調用鏈，也給業務排查插上翅膀，從鏈路的上下文信息中快速找到線索。
