一、什麼是WebRTC

WebRTC技術是激烈的開放的Web戰爭中一大突破-Brendan Eich, inventor of JavaScript。

簡單來說，WebRTC 是一個音視頻處理+及時通訊的開源庫。在實時通信中，音視頻的採集和處理是一個很複雜的過程。比如音視頻流的編解碼、降噪和回聲消除等。由Google發起開源，其中包含視頻音頻採集，編解碼，數據傳輸，音視頻展示等功能，我們可以通過技術快速地構建出一個音視頻通訊應用。雖然其名為WebRTC，但是實際上它不只是支持Web之間的音視頻通訊，還支持Android以及IOS端，此外由於該項目是開源的，我們也可以通過編譯C++代碼，從而達到全平臺的互通。

二、基礎概念

流和軌

  • Track 軌道，可以理解每一路音頻或視頻，為一個軌，互不相交，類比火車軌道。
  • MediaStream 媒體流，每個媒體流中包含若干軌道，可以將音頻軌，視頻軌打包在一起。

三、幾個關鍵類

  • MediaStream 媒體流類，MeidiaStream用於將多個MediaStreamTrack對象打包到一起。一個MediaStream可包含audio track 與video track,並且可以添加或者刪除。
  • RTCPeerConnection 連接類，包含非常多重要功能，屏蔽複雜技術細節，便於應用層使用，包括但不限於連接管理，P2P類型檢測，NAT穿透，中轉等。
  • RTCDataChannel 非音視頻數據傳輸類，這個類在我們的例子中沒有涉及到。可以簡單理解為將媒體流信息或者數據信息塞到連接中，進行傳輸。

四、端對端連接流程

兩個不同網絡環境瀏覽器，要實現點對點的實時音視頻對話，需要處理哪些問題?

媒體協商

雙方需要知道對方支持的媒體格式，SDP（Session Description Protocol）是一種會話描述協議，視頻通訊的雙方必須先交換SDP信息，才能進一步互相通信。

網絡協商

雙方要了解對方的網絡情況，嘗試尋求一個可以互相通訊的鏈路，其中有尋路選擇，如果確實沒辦法建立點對點鏈路，會使用中繼服務器來進行轉發。如果是內網，或者大部分NAT網絡環境下，是可以建立端到端連接。在解決網絡打通問題時候，有幾個概念。

  • STUN（Session Traversal Utilities for NAT，NAT會話穿越應用程序）是一種網絡協議，它允許位於NAT後的客戶端找出自己的公網地址，查出自己位於哪種類型的NAT之後以及NAT在公網的端口映射信息。這些信息被用來在兩端創建UDP連接通信。
  • TURN (Traversal Using Relays around NAT)，如果客戶端在NAT之後， 那麼在一些網絡情景下，有可能建立點對點的通訊連接，這時就需要公網的服務器作為一箇中繼， 對數據進行轉發。

學習過程中，STUN和TURN服務器我們可使用coturn開源項目來搭建。

數據交換服務-信令服務器

WebRTC實現並沒有規定信令服務器的實現方式和相關協議，這給了業務方技術選型極大的靈活。我們今天就是使用PHP+Swoole協程實現一個簡單信令服務器。下面是一個端到端連接的流程圖，整個核心流程邏輯都在圖裡面。

信令服務端

我們介紹了建立連接的過程，針對服務端代碼，可以看到信令服務器端的代碼很少，加上http的服務總計100行代碼左右，怎樣達到通過同步編程的方式實現異步非阻塞IO，並且可以很輕鬆的實現併發百萬呢？

  • 首先通過構造函數$server = new Swoole\Coroutine\Http\Server('0.0.0.0', 9509, true);會創建server對象。
  • 當調用$server->start();方法後，會循環進行accept，accept連接後，會創建一個協程，這個協程內所有的消息收發，都會引起協程調度。
  • 可以低成本創建成千上萬協程，併發百萬沒問題，底層會為每個協程開闢獨立的棧空間，並基於多路複用技術（Linux下為EPOLL）來進行調度。

信令服務器利用Swoole協程技術，單進程支持異步非阻塞IO高併發，但編程完全是同步阻塞的模式。如果想進一步要利用多核，可以採用Process Pool，加reuse port(Linux kernel 3.9)技術，開啟多個進程同時處理,代碼倉庫中有一份server_co_pool.php的相關實現

服務端處理核心邏輯為將當前連接加入內存map中，以供新的連接到來查找廣播，連接關閉時，清理對應的主題和fd。
