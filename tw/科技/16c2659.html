Istio 是如何在 Kubernetes 幕後工作的？

2020-10-29
| 科技

作者：Gaurav Agarwal

翻譯：Bach（才雲）

校對：星空下的文仔（才雲）、bot（才雲）

如果在 Kubernetes 上使用微服務，那麼像 Istio 這樣的服務網格將發揮出巨大作用。這裡我們先討論一下 Istio 的體系結構。

Istio 通過兩個主要組件幫助管理微服務：

  • 數據平面（Data Plane） ：這是 Istio 在微服務中的 Envoy sidecar，它們在服務之間進行實際路由，並收集遙測數據。
  • 控制平面（Control Plane） ：這是告訴數據平面如何路由流量的組件，它存儲、管理配置，幫助管理員與 sidecar 代理進行交互並控制 Istio 服務網格。控制平面就像是 Istio 的大腦。

經過 Istio 的流量有兩種類型： 數據平面流量（與業務相關的流量）和控制平面流量（由消息和 Istio 組件之間的交互組成） 。

控制平面組件

在當前 Istio 版本（Istio v1.5 及更高版本）中， 控制平面以單個二進制 Istiod 的形式提供，由三部分組成：Pilot、Citadel 和 Galley。

Istio 架構


Pilot

Pilot 是服務網格的中央控制器，負責使用 Envoy API 與 Envoy sidecar 進行通信。它會解析 Istio manifest 中定義的高級規則，並將其轉換為 Envoy 配置。它有助於服務發現、流量管理和智能路由，使我們可以進行 A/B 測試、藍綠部署、金絲雀發佈等。它能通過配置 sidecar 來提供超時、重試和斷路功能，以在網格中提供足夠的彈性。

另外，Pilot 還負責在 Istio 配置和 Istio 運行的基礎基礎結構之間提供鬆散的耦合。因此，我們可以在Kubernetes、Nomad 或 Consul 上運行 Istio，這些與 Istio 交互的方式是相似的。Pilot 知道自己在哪個平臺上運行，可以確保流量管理是相同的，和平臺無關。它可以託管流量管理 API，該 API 可幫助定義配置並提供更精細的服務網格中流量管理。

Citadel

服務之間的身份和訪問管理是 Istio 的核心功能，它允許 Kubernetes Pod 之間進行安全通信。這意味著，當開發人員設計了具有不安全 TCP 的組件時，Envoy proxy 要確保 Pod 之間的通信是加密的。

我們可以通過為每個 Pod 配置一個證書來實現相互 TLS ，但這非常麻煩，如果是大型微服務應用程序，那麼最終將要管理數百個證書。對此， Istio 提供了一個名為 Citadel 的身份和訪問管理器以及證書存儲庫。 Citadel 可幫助管理與服務進行的通信以及它們是如何相互識別和認證的。Citadel 也提供了用戶身份驗證、憑證管理、證書管理和流量加密。如果有任何 Pod 需要驗證憑證，Citadel 就可以提供。

Galley

Galley 負責為服務網格提供配置驗證、接收、處理和分發。它是 Istio 控制平面與之交互的基礎 API 接口。例如，如果我們應用了新策略，Galley 將對其進行驗證、處理並推送到服務網格的正確組件。

數據平面組件

數據平面組件包含 Envoy proxy。這些是第 7 層代理，可根據收到的消息的內容進行關鍵決策。與業務流量交互的唯一組件是 Envoy proxy，能幫助提供：

  • 動態服務發現。
  • 負載均衡。
  • TLS 終止。
  • 分階段推出基於百分比的指標。
  • 故障注入。
  • 健康檢查。
  • 斷路。
  • 收集遙測數據。

當所有流量流經 Envoy proxy 時，它們會收集有關業務流量的基本數據，這可以幫助我們收集信息，做好規劃。

Istio 提供了開箱即用的 Prometheus 和 Grafana，用於監視和可視化此數據，還可以將數據作為 Elastic Logstash Kibana（ELK）堆棧發送到日誌分析工具，以瞭解趨勢並根據這些指標進行機器學習。

由於 Istio 使用 sidecar，因此無需重新設計 Kubernetes 上運行的微服務應用程序，它是開發、安全和運營團隊之間的絕佳分離器。開發人員可以像以前一樣開發代碼，而不必擔心操作和安全性方面。安全和運營團隊可以使用 Istio 在應用程序中使用 Sidecar，這有助於他們貫徹管理和安全策略。

Envoy proxy 可幫助提供 Istio 的大多數功能，例如：

  • 流量控制 ：這有助於控制流量通過服務網格，例如提供 HTTP、TCP、Websocket 和 gRPC 流量的路由規則。
  • 安全性和身份驗證 ：它對 Pod 實施身份和訪問管理，以便只有正確的 Pod 才能與另一個 Pod 交互。此外，還實現了雙向 TLS 和流量加密，以防止中間人攻擊。它們提供速率限制，從而防止成本失控和拒絕服務攻擊。
  • 網絡彈性 ：它有助於提供網絡彈性功能，例如重試、故障轉移、斷路和故障注入。

它還提供了一種使用 Web 程序集擴展模型插入自定義策略實施和遙測數據收集的方法。

組件如何交互？

Istio 所做的一切都是通過 Envoy proxy 進行的。讓我們看一下通過 Istio 的數據包採取的典型流程。

圖源 Istio.io


我們查看上面的圖片，可以發現有 一個 Ingress，兩個微服務和一個 Egress 。服務 A 和服務 B 是在容器上運行的微服務，它們與 Envoy proxy 共享同一 Pod。部署它們後，Istio 會將 Envoy proxy 自動注入到 Pod 中。正如圖片中那樣，服務 A 是前端服務，服務 B 是後端服務。

流量數據包按照以下路線在服務網格中移動：

Ingress

流量通過 Ingress 資源到達服務網格。Ingress 是由一個或多個 Envoy proxy 組成的庫，這些代理在部署後由 Pilot 配置。Pilot 使用 Kubernetes 服務端點配置 Ingress 代理，並且 Ingress 代理知道其後端，因此，Ingress 知道接收到的數據包將發送到何處。它會進行健康檢查和負載平衡，並根據已定義的度量標準（例如負載、數據包、配額和流量平衡）做出決定來發送消息。

服務 A

一旦 Ingress 路由到第一個 Pod，它將命中 Service A Pod 的代理，而不是容器。由於 sidecar 與微服務容器形成同一 Pod 的一部分，因此它們共享相同的網絡命名空間，並位於同一節點中。 兩個容器具有相同的 IP 地址，並共享相同的 IP 表規則。這使得代理可以完全控制 Pod，並處理通過 Pod 的所有流量。

Envoy proxy 與 Citadel 進行交互，以瞭解是否需要執行任何策略。它還檢查是否需要加密通過鏈的流量並與後端容器建立 TLS。

服務 B

服務 A 將加密的數據包發送到服務 B， 服務 B 遵循與服務 A 相同的步驟 。它會標記數據包的發送者，並與源代理進行 TLS 握手。在建立信任後，它將接受該數據包並將其轉發到 Service B 容器，然後這些將移動到該 Egress 層。

Egress

所有從網狀網絡流出的流量都會使用 Egress 資源。 該 Egress 層定義了哪些流量可以從網格中傳出並使用 Pilot ，其配置與該 Ingress 層相似。使用 Egress 資源，我們可以編寫策略以限制出站流量，並且僅允許所需的服務將數據包發送出網格。

遙測數據收集

在上述所有步驟中，當流量通過代理時，它可以收集遙測數據並將其發送到 Prometheus。數據可以在 Grafana 中可視化，或發送到外部工具（如 ELK）以進行進一步的分析和指標的機器學習。
