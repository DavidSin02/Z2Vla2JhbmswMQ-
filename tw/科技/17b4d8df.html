HashMap的數據結構

在編程領域，當涉及到數據存儲，都離不開數據結構，HashMap 作為K,V形式存儲的數據結構，通過看它的源碼，我們不難發現，它是通過數組和鏈表來實現，在JDK1.8 之後假如加入紅黑樹組合，所以現在HashMap 的數據結構是數據+鏈表+紅黑樹

HashMap實際是一種“數組+鏈表”數據結構。在put操作中，通過內部定義算法尋止找到數組下標，將數據直接放入此數組元素中，若通過算法得到的該數組元素已經有了元素（俗稱hash衝突，鏈表結構出現的實際意義也就是為了解決hash衝突的問題）。將會把這個數組元素上的鏈表進行遍歷，將新的數據放到鏈表末尾

為了驗證數據結構的正確性，接下來我們將通過源碼分析來驗證

我們可以看到有Node元素的數據，同時node是一個鏈式結構

HashMap 的初始化方式有三種

HashMap();//構造一個具有默認初始容量 (16) 和默認加載因子 (0.75) 的空 HashMap。HashMap(int initialCapacity);//構造一個帶指定初始容量和默認加載因子 (0.75) 的空 HashMapHashMap(int initialCapacity, float loadFactor);//構造一個帶指定初始容量和加載因子的空 HashMapHashMap(Map<? extendsK,? extendsV> m); //構造一個映射關係與指定 Map 相同的 HashMap

HashMap根據 (n - 1) & hash 求出了元素在node數組的下標，主要分三個階段：計算hashcode、高位運算和取模運算。

key.hashCode()計算出key的哈希值，然後將哈希值h右移16位，再與原來的h做異或^運算。正好是int類型32位對半。將高16位和低16位的信息"融合"到一起，也稱為"擾動函數"。這樣才能保證hash值所有位的數值特徵都保存下來而沒有遺漏，從而使映射結果儘可能的鬆散。

取模運算，HashMap數組下標的計算方式 tab[(n - 1) & hash]，n 是數組的長度。其實該算法的結果和模運算的結果是相同的。對CPU來說，位運算是最快的動作，除法和求餘數（模運算）是最慢的動作。當n是2的N次冪的時候。Hash%n == （n-1）& hash。這就是為什麼HashMap 的容量建議是 2的冪次方。hash 算法的目的是為了讓hash值均勻的分佈在數組中，當容量只不是2的冪次方時，會發現，有時候會形成長鏈，影響讀寫的性能

如果Map中已有數據的容量達到了初始容量的 75%，那麼散列表就會擴容，而擴容將會重新將所有的數據重新散列，性能損失嚴重，所以，我們可以必須要大於我們預計數據量的 1.34 倍。

get 方法分析

講完put方法，get方法其實很容易理解

HashedMap的重新擴容機制

首先我們明白，resize重新擴容就是建立一個比原來更大的數組來存儲。

直接上代碼

final Node<K,V>[] resize() {    Node<K,V>[] oldTab = table;    int oldCap = (oldTab == null) ? 0 : oldTab.length;    int oldThr = threshold;    int newCap, newThr = 0;    if (oldCap > 0) {        if (oldCap >= MAXIMUM_CAPACITY) {            threshold = Integer.MAX_VALUE;            return oldTab;        }        else if ((newCap = oldCap << 1) < MAXIMUM_CAPACITY &&                 oldCap >= DEFAULT_INITIAL_CAPACITY)            newThr = oldThr << 1; // double threshold    }    else if (oldThr > 0) // initial capacity was placed in threshold        newCap = oldThr;    else {               // zero initial threshold signifies using defaults        newCap = DEFAULT_INITIAL_CAPACITY;        newThr = (int)(DEFAULT_LOAD_FACTOR * DEFAULT_INITIAL_CAPACITY);    }    if (newThr == 0) {        float ft = (float)newCap * loadFactor;        newThr = (newCap < MAXIMUM_CAPACITY && ft < (float)MAXIMUM_CAPACITY ?                  (int)ft : Integer.MAX_VALUE);    }    threshold = newThr;// 上面的代碼很簡單，總結來說，1、當HashMap初始化時指定數據大小為16，閾值為16 * loadFactor（負載因子）   2、將數組的大小擴大為原來2倍，閾值設置為數組大小乘以 負載因子。    @SuppressWarnings({"rawtypes","unchecked"})     Node<K,V>[] newTab = (Node<K,V>[])new Node[newCap];    table = newTab;    if (oldTab != null) {        for (int j = 0; j < oldCap; ++j) {// 遍歷原來的數組            Node<K,V> e;            if ((e = oldTab[j]) != null) {                oldTab[j] = null; // 將oldTab[j]=null只是清除舊錶的引用                if (e.next == null)		// 當前存儲桶只有一個元素的時候，直接hash賦值到新的數組上即可                    newTab[e.hash & (newCap - 1)] = e;                else if (e instanceof TreeNode)		// 如果該存儲桶裡面存的是紅黑樹, 則拆分樹                    ((TreeNode<K,V>)e).split(this, newTab, j, oldCap);                else { // preserve order		// 以下代碼主要將一個鏈表拆為兩個鏈表lo,hi                    Node<K,V> loHead = null, loTail = null;	// lo鏈表的頭節點和尾節點                    Node<K,V> hiHead = null, hiTail = null;	//  hi鏈表的頭節點和尾節點                    Node<K,V> next;                    do {// 遍歷當前當前存儲桶的鏈表                        next = e.next;                        if ((e.hash & oldCap) == 0) {	// (e.hash & oldCap) == 0 作為鏈表拆為的條件，為什麼以該條件作為鏈表拆分條件，下文再詳解                            if (loTail == null)                                loHead = e;                            else                                loTail.next = e;                            loTail = e;                        }                        else {                            if (hiTail == null)                                hiHead = e;                        newTab[j] = loHead;                    }                    if (hiTail != null) {                        hiTail.next = null;                        newTab[j + oldCap] = hiHead;                    }                }            }        }    }    return newTab;}

對於鏈表拆分，我們以下面這個圖可能更好理解點

(e.hash & oldCap) == 0 j 以及 j+oldCap

  1. oldCap一定是2的整數次冪, 這裡假設是2^m
  2. newCap是oldCap的兩倍, 則會是2^(m+1)
  3. hash對數組大小取模(n - 1) & hash 其實就是取hash的低m位

(16-1) & hash 自然就是取hash值的低4位,我們假設它為 abcd.以此類推, 當我們將oldCap擴大兩倍後, 新的index的位置就變成了 (32-1) & hash, 其實就是取 hash值的低5位. 那麼對於同一個Node, 低5位的值無外乎下面兩種情況。

0abcd1abcd

其中, 0abcd與原來的index值一致, 而1abcd = 0abcd + 10000 = 0abcd + oldCap故雖然數組大小擴大了一倍，但是同一個key在新舊table中對應的index卻存在一定聯繫： 要麼一致，要麼相差一個 oldCap。

而新舊index是否一致就體現在hash值的第4位(我們把最低為稱作第0位), 怎麼拿到這一位的值呢, 只要:

hash & 0000 0000 0000 0000 0000 0000 0001 0000

上式就等效於

hash & oldCap

故得出結論:

如果 (e.hash & oldCap) == 0 則該節點在新表的下標位置與舊錶一致都為 j
如果 (e.hash & oldCap) == 1 則該節點在新表的下標位置 j + oldCap

  1. resize發生在table初始化, 或者table中的節點數超過threshold值的時候, threshold的值一般為負載因子乘以容量大小.
  2. 每次擴容都會新建一個table, 新建的table的大小為原大小的2倍.
  3. 擴容時,會將原table中的節點re-hash到新的table中, 但節點在新舊table中的位置存在一定聯繫: 要麼下標相同, 要麼相差一個oldCap(原table的大小).

JDK1.8 之後為什麼要調整HashMap的數據結構

紅黑樹調整Hashap的優點

  1. · 添加時，當桶中鏈表個數超過 8 時會轉換成紅黑樹；
  2. 刪除、擴容時，如果桶中結構為紅黑樹，並且樹中元素個數太少的話，會進行修剪或者直接還原成鏈表結構；
  3. 查找時即使哈希函數不優，大量元素集中在一個桶中，由於有紅黑樹結構，性能也不會差
