阿里巴巴高級技術專家周洋，親述混沌工程技術背後那些事兒

2020-10-29
| 科技


阿里巴巴技術團隊對混沌工程的研發迭代持續了數年，對於這一技術，阿里為什麼如此堅持？

2010 年底，Netflix 向全世界推出 Chaos Monkey （混亂猴子），該技術以在生產環境中隨機關閉服務節點而“惡名遠揚”，這就是目前熟知的混沌工程的早期雛形。《Chaos Engineering》書中是這樣描述的：

混亂猴子的美妙之處就在於此，它能儘可能地將服務節點失效的痛苦提到最前，同時讓所有工程師在構建一個具有足夠彈性應對失敗的系統上，達成一個一致的目標。

混亂猴子的靈感來自於 Netflix 幾年前搬遷上雲的過程，主要是為了解決該階段暴露的問題。然而，目前不少企業的上雲進程依舊緩慢，這種狀態下是否還需要混沌工程？隨著軟件的測試流程越來越成熟且完善，是否有必要花費精力搞混沌工程？如果需要，這一領域有哪些開源工具可供快速部署實踐？

針對上述問題，InfoQ 記者在QCon 2019 全球軟件開發大會的現場採訪了阿里巴巴高級技術專家周洋（花名：中亭），並與業內多名技術專家進行了不同程度的交流，本文將試圖通俗講解混沌工程的含義、作用及阿里巴巴的實踐經驗。

混沌工程的含義

混沌工程是在分佈式系統上進行實驗的學科 , 目的是建立對系統抵禦生產環境中失控條件的能力以及信心 。

這句話同樣出自《Chaos Engineering》，粗看起來，這一含義有些晦澀難懂。如果簡單概括下，周洋對混沌工程的理解是如下四點：

  • 一種擁抱失敗的技術文化
  • 一套抽象嚴謹的實踐原則
  • 一種主動防禦的穩定性手段
  • 一個高速發展的技術領域

從原理性角度來講，周洋認為，實施混沌工程是分佈式系統構建業務的必然選擇。一方面架構的複雜性讓系統負責人員很難自信承諾實際情況一定符合預期設計。另一方面組織對 ROI（投資回報率）的追求，又要求穩定性團隊不能只是一個成本中心，所有措施要富有成效。業內不少工程師傾向於將混沌工程比作疫苗，通過 " 接種疫苗 " 的方式，讓系統具備抵擋 " 重大疾病 " 的能力。混沌工程的原則又可以概括為：建立一個圍繞穩定狀態行為的假說；多樣化真實世界的事件；在生產環境中運行實驗；持續自動化運行實驗；最小化爆炸半徑。


想必看到混沌工程的那一刻，不少人腦海中會閃過一個詞“測試”。如今，故障注入和故障測試的手段已經非常成熟，為什麼還需要混沌工程。周洋認為，這之間存在一定重疊性，但混沌工程是發現新信息的實踐。

故障注入和故障測試是通過引入故障，讓程序走入一些不常經過的路徑，以此提高程序覆蓋率，是對特定條件、變量的驗證。混沌工程雖然也使用短期度量結果代表系統狀態，但該度量結果一般是監控指標，而非具體的接口返回值。混沌工程是圍繞穩態進行驗證和探索的過程，比如混亂猴子隨機殺死部分節點的行為，每次實驗都可能產生新的數據。

舉例來說，故障注入的典型場景是對系統服務注入通信故障，比如超時、錯誤等，但無法探究流量激增、資源競爭條件等其他異常行為，這個實踐過程本質上並不能發掘系統內未知或尚不明確的認知。而混沌工程的輸入條件可能是模擬整個雲服務區域或數據中心故障；挑選一個時間段，和針對一部分流量，對其涉及的服務間調用注入一些特定的延時；方法級別的混亂（運行時注入），讓方法隨機拋出各種異常等。

此外，實施混沌工程的人員也有所不同。周洋表示，以阿里內部的全鏈路壓測為例，實施主要還是研發人員為主。傳統意義上，測試人員更關注上線前的狀況，之後交由 SRE 團隊負責運維。混沌工程可以理解為測試的思想，但並不一定只有測試人員操刀，可以是 SRE、研發、技術支持甚至運營。

不為，就不需要混沌工程？

所謂不為，此處可以理解為較低頻次的迭代和平緩的業務變更。如上文所言，Netflix 對混沌工程的探索起源於數據中心遷移上雲的過程，因此不乏有企業認為只要不進行大規模的雲遷移，就不需要實踐混沌工程。

其實，這種想法是非常片面的。無論是激進的架構變遷，還是平緩的系統迭代，很多問題並不是因為遷移上雲才出現，只是因為遷移上雲才被暴露出來。

只要有過在生產環境中實際運行分佈式系統的經歷，一定會清楚分佈式系統天生包含大量交互、依賴點，可以出錯的地方數不勝數，比如硬盤故障、網絡不通、流量激增壓垮系統等，人力並不能完全阻止故障發生，而應該在異常行為被觸發之前，儘可能多地識別出系統中脆弱且易出故障的環節。


根據 Netflix 的實踐歷程，嘗試破壞系統和服務很簡單，但並不是全都可以有建設性、高效地發現問題。周洋認為，嚴格來講，混沌工程與雲的關係不大，但目前大部分企業都處於雲遷移、雲就緒或者雲原生的不同階段，上雲已經成為無法阻擋的趨勢，該階段正是故障爆發的高發期，這些問題可能潛藏在系統中良久，只是一直未被集中發現。雖然上雲集中變革的是基礎設施層面，但需要某種手段驗證上層技術平臺是否可以在這個過程中不受影響並保持一致，混沌工程的引入可以讓問題提前暴露、提前解決，讓企業更全面地理解這些系統性固有現象，從而在分佈式系統中實現更好的工程設計，不斷提高系統彈性。

此外，周洋特別強調，混沌工程並不是大公司的專屬，除非企業完全不在乎系統運行情況。相比較而言，金融、遊戲、電商、航空航天等業務發展較快且對可用性具備高要求的領域更加需要混沌工程。如果本身的研發團隊或者業務團隊規模較小，同樣需要混沌工程提高效率和產出比。

阿里巴巴混沌工程實踐

早在 2011 年，阿里電商就開始嘗試通過故障注入技術去解決微服務的依賴問題，從注入實現、實驗效率、業務影響等多個方面進行演進；2012 年，阿里推出同城容災，進行斷網演練；2015 年，異地多活上線；2016 年，MonkeyKing 被用於故障演練；2019 年，阿里開源混沌工程工具ChaosBlade（混沌之刃）。


周洋介紹，阿里之所以下定決定做這件事情，源自大促備戰的反思，雖然內部有完善的體系和流程，但真實情況下的系統穩定性是未知的。因此，阿里決定手段與技術並行，通過混沌工程解決技術和組織層面的問題。

實踐早期，團隊希望通過混沌工程解決微服務依賴治理的問題，因為當時的微服務拆分還沒有特別徹底。舉例來說，一個核心服務與某個非核心服務之間可能存在強依賴關係，非核心服務出現問題時，可能拖垮核心服務，這肯定是不合理的，也是業務無法接受的。因此，團隊通過引入混沌工程進行強依賴治理，優化整個流程。

一些基礎技術團隊也嘗試通過混沌工程解決容器相關的問題。周洋介紹，作為雲服務，容器平臺本身屬於 PaaS 層，通過實施混沌工程可以保證系統本身沒有問題；其次，傳統企業上雲時，面臨著應用向容器轉換的過程，設計模式等都需要進行改變，如果對業務穩態具有要求，混沌工程的引入可以解決這一階段出現的問題。

目前，集團安全生產團隊開始牽頭運作 " 突襲演練 " 這種跨公司的穩定性戰役。在一個可控的環境下，通過有計劃的打擂和無計劃的突襲，讓安全生產藍軍和業務紅軍進行對抗，通過實戰演練的方式驗收穩定性措施的全面性和有效性。

回顧整個發展過程，周洋認為，比較難的事情，一是讓團隊意識到可以通過混沌工程技術趨同整個穩定性鏈條，更自動化、智能化得提升平臺穩定性。因為，平臺能力越差，支持各個團隊的人力成本就越高，平臺能力越高，防禦相應就越差；二是，讓更多人接受並實施該理念，無論是內部還是雲上的客戶，都可以瞭解平臺的問題，阿里團隊也希望將內部穩定性的經驗通過平臺化的方式對外輸出，ChaosBlade 便由此開源。

ChaosBlade 經歷了 6 年改進和實踐，累計在線上執行演練場景達數萬次，是一款遵循混沌實驗模型，提供豐富故障場景實現，幫助分佈式系統提升容錯性和可恢復性的混沌工程工具，其特點是操作簡潔、無侵入、擴展性強。基於 Apache License v2.0 開源協議，目前有 chaosblade 和 chaosblade-exe-jvm 兩個倉庫，後續還會添加 C++、Node.js 等其他語言的混沌實驗執行器。

目前，混沌工程領域已經出現不少優秀的開源工具，比如 kube-monkey、ChaosIQ 等，但很多開源工具的功能存在高度重疊，或者難以滿足實際訴求。阿里希望可以同用一套工具體系，解決所有故障場景問題，未來通過開源社區的力量完善實驗場景，共同推進混沌工程領域的發展。

周洋強調，AHAS（阿里雲應用高可用服務）是面向業務穩定性的雲服務，這對企業而言非常重要。企業上雲，並不僅僅是為了享受穩定的基礎設施，也需要保證業務穩定性，而目前很少有廠商注意到這一點，AHAS 的出現不是為了拉動營收，而是幫助雲上用戶更好得搭建雲原生系統。


目前，ChaosBlade 已經可用並處於快速演進階段，團隊大概每週都會發版，按照微服務、容器和雲原生的路線迭代。近期，該項目會增強 JVM 演練場景；支持更多 Java 主流產品，比如 Redis，gRPC；增強 Kubernetes 演練場景以及對 C++、Node.js 等應用的支持。周洋透露，目前團隊也希望更多志同道合的朋友加入，感興趣的朋友可以通過 Github 平臺瞭解該項目。
