JavaWeb中文編碼問題全面解析

2020-10-29
| 科技

需要編碼的原因

  • 計算機中存儲的最小單元是一個字節，即8bit，所以能表示的字符範圍是0~255個。
  • 人類要表示的符號太多，無法用一個字節來完全表示。
  • 要解決這個矛盾必須要有一個新的數據結構char，從char到byte必須編碼。

編碼格式

  1. ASCII
  • ASCII碼共有128個，用一個字節的低7位表示，0~31是控制字符，如換行、回車、刪除等；32~126是打印字符，可以通過鍵盤輸入並能夠顯示出來。
  1. ISO-8859-1
  • 128個字符顯然是不夠用的，ISO組織在ASCII碼的基礎上又制定了一系列標準用來擴展ASCII編碼，他們是ISO-8859-1~ISO-8859-15，其中ISO-8859-1涵蓋了大多數西歐語言字符，所以應用最廣泛。ISO-8859-1任然是單字節編碼，它總共能表示256個字符。
  1. GB2312
  • 它的全稱是《信息交換用漢字編碼字符集基本集》，它是雙字節編碼，總的編碼範圍是A1~F7，其中從A1~A9是符號區，總共包含682個字符。從B0~F7是漢字區，包含6763個漢字。
  1. GBK
  • 全稱《漢字內碼擴展規範》，為了擴展GB2312加入了更多的漢字，它的編碼是和GB2312是兼容的，也就是說GB2312編碼的漢字可以用GBK來解碼，並且不會有亂碼。
  1. GB18030
  • 是我國強制標準，它可能是單字節、雙字節、或者四字節編碼，與GB2312兼容，應用並不廣泛。
  1. UTF-16
  • 用兩個字節來表示Unicode轉化格式，它是定長的表示方法，不論什麼字符都可以用兩個字節表示，兩個字節是16bit，所以叫UTF-16。每兩個字節表示一個字符，這就大大簡化了字符串的操作，這也是java以UTF-16作為內存字符存儲格式的一個很重要的原因。
  1. UTF-8
  • UTF-16同意採用兩個字符表示一個字節，但是很大一部分字符用一個字節就可以表示現在卻要用兩個字符表示，存儲空間放大了一倍，而現在網絡帶寬還非常有限，這樣會增大網絡傳輸的流量，而且也沒必要。而UTF-8採用了一種變長的技術，每個編碼區域有不同的字碼長度。不同類型的字符可以由1~6個字節組成。

java中需要編碼的場景

I/O操作中存在的編碼

  • 涉及編碼的地方一般在字符到字節或者字節帶字符的轉換上，二需要這種轉換的場景主要是I/O。
  • Reader類是java的i/o中讀字符的父類，而inputstream類是讀字節的父類，inputstreamreader類就是關聯字節到字符的橋樑，它負責在I/O過程中處理讀取字節到字符的轉換，而具體字節到字符的解碼又委託streamdecoder去做，在streamdecoder解碼過程中必須有用戶指定charset編碼格式，如果沒有指定charset，將使用本地環境中的默認字符集。


  • 寫的情況也類似，字符的父類是writer，字節的父類是outputstream，通過outputstreamwriter轉換字符到字節。StreamEncoder類負責將字符編碼成字節，編碼格式和默認編碼規則與解碼是一致的。
  • 強烈建議不要使用操作系統的默認編碼，因為這樣你的應用程序的編碼格式就和運行環境綁定起來了，在跨環境是很可能出現亂碼。

內存操作中的亂碼

  • 內存中進行字符到字節的轉換也很常見。
String s = "這是一段中文字符";byte[] b = s.getBytes("utf-8");String n = new String(b,"utf-8");Charset charset = Charset.forName("utf-8");ByteBuffer byteBuffer = charset.encode(string);CharBuffer charBuffer = charset.decode(byteBuffer);

Java中如何編解碼

  • 以字符串“I am 君山”為例。
  • 按照ISO-8859-1編碼


  • ISO-8859-1是單字節編碼，中文“君山”被轉化成值是3f的byte，3f也就是“?”字符。所以經常會出現中文變成“?”，很可能就是錯誤地使用了ISO-8859-1編碼導致的。
  • 按照GB2312編碼


  • GB2312字符集有一個char到byte的碼錶，不同的字符編碼就查這個碼錶找到與每個字符的對應字節，然後拼裝成byte數組。
  • 按照GBK編碼
  • GBK與GB2312編碼結果是一樣的；
  • 由此可以看出來GBK編碼是兼容GB2312編碼的，他們的編碼算法是一樣的。
  • 不同的是它們的碼錶長度不一樣，GBK包含的漢字字符更多，所以只要是經過GB2312編碼的漢字都可以用GBK進行解碼，反之則不然。
  • 按照utf-16編碼


  • 用utf-16編碼將char數組放大了一倍，單字節範圍內的字符在高位補0變成兩個字節，中文字符也變兩個字節。
  • 編碼效率非常高，規則很簡單。
  • 按照utf-8編碼


  • UTF-16採用順序編碼，不能對單個字符的編碼值進行校驗，如果中間的一個字符碼值損壞，後面所有的碼值都將受到影響。
  • 而UTF-8不存在這些問題，UTF-8對單字節範圍內字符任然用一個字節表示，對漢字採用三個字節表示。
  • UTF-8編碼與GBK和GB2312不同，不用查碼錶，所以在編碼效率上UTF-8的效率會更好，所以在存儲中文字符時UTF-8編碼比較理想。
  • 幾種編碼格式的比較
  • GB2312與GBK的編碼規則類似，但是GBK範圍更大，所以GB2312與GBK比較，應該選擇GBK；
  • utf-16與utf-8都是處理Unicode編碼，編碼規則不太相同，相對來說utf-16編碼效率最高，字符到字節相互轉換更簡單，進行字符操作也更好，它適合本地磁盤和內存之間使用，可以進行字符和字節中間的快速切換，java內存編碼就採用utf-16編碼；
  • 但是UTF-16不適合網絡之間的傳輸，因為網絡傳輸容易損壞字節流，一旦字節流損壞就很難恢復，相比較而言，utf-8更適合網絡傳輸，單個字符的損壞不會影響後面其他字符，編碼效率介於GBK和UTF-16之間；
  • UTF-8在編碼效率上和安全性上做了平衡，是理想的中文編碼方式。

Java Web中涉及的編解碼

  • URL的編解碼
  • 瀏覽器編碼URL將非ASCII字符按照某種編碼格式編碼成16進制數字後在每個16進製表示的字節前加上“%”，所以就出現瞭如下情況：
http://tanqingbo.com/2016/05/11/%E5%A4%8F%E4%BB%A4%E8%90%A5%E6%B1%87%E6%80%BB/
  • http Header的編碼
  • header中傳遞參數，如：Cookie、redirectPath等，這些用戶設置的值可能存在編碼問題。
  • 對header進行解碼實在調用request.getHeader時進行的，這個方法將byte到char的轉化使用的是ISO-8859-1，不能手動設置Header的其他解碼格式，如果有非ASCII字符肯定會有亂碼；
  • 不要在header中傳遞非ASCII字符，如果一定要出傳遞，可以先將這些字符用org.apache.catalina.util.URLEncoder編碼，然後再添加到header中。
  • 訪問數據庫都是通過客戶端JDBC驅動來完成的，用JDBC來存取數據要和數據的內置編碼保持一致，可以通過設置JDBC URL來指定，如：MySQL：
jdbcUrl="jdbc:mysql://localhost:3306/boke?characterEncoding=utf-8"

JS中的編碼問題

  • 外部引入JS文件


  • 如果script沒有設置charset，瀏覽器就會以當前這個頁面的默認字符集解析這個JS文件，如果外部的JS文件的編碼格式與當前頁面的編碼格式一致，那麼可以不設置這個charset，但是如果script.js文件的編碼格式與當前頁面不一致，上面的那段中文輸入就會變成亂碼。
  • JS的URL編碼
  • 實際上JS中處理URL編碼有三個函數，只要掌握了這三個函數，基本上就能正確處理JS的URL亂碼問題了；
  1. escape()
  • 這個函數是將非ascii字符轉化成Unicode編碼值，並且在編碼值前加上“%u”；
  1. 解碼通過unescape()函數；
  2. 通過將特殊字符換成Unicode編碼值可以避免因為編碼的字符集的不兼容而出現的信息丟失問題，在服務端通過解碼參數就可以避免亂碼的問題。
  • encodeURL()
  • 與escape()相比，encodeURL()是真正的JS用來對URL編碼的函數，它可以將整個URL中的字符(除了一些特殊字符，如：符號、數字、字母)進行UTF-8編碼，在每個值之前加上“%”；
  • 解碼通過encodeURL函數。
  • encodeURLComponent()
  • encodeURLComponent()這個函數比encodeURL()編碼還要徹底；
  • 通常用於將一個URL當做參數放在另一個URL中；
  • 其他需要編碼的地方
  • XML文件可以通過設置投來制定編碼格式：
<?xml version"1.0" encoding="UTF-8">
  • Velocity(基於Java的模板引擎)設置編碼格式：
services.VelocityService.input.encoding=uft-8
  • Jsp設置編碼格式：
<%@page contentType="text/html; charset=utf-8">

  • 我們通過request.getParameter獲取參數值時，直接調用：
String value = request.getParameter(name);
  • 會出現亂碼，但是用如下方式：
String value = new String(request.getParameter(name).getBytes("ISO-8859-1"),"GBK");
  • 解析時取得的value會是正確的漢字字符。
