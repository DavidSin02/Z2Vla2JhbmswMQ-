五分鐘學習Linux系列之 -- ACLs 訪問控制列表

2020-10-29
| 科技

今天是五分鐘學習Linux系列的第六次更新，今天的主題是ACLs訪問控制列表，為什麼要說這麼一個這個點呢？之前的系列裡面不是已經講過文件權限控制了嗎，還要一個訪問控制列表幹什麼？ 是的，我們一直在強調我們不說沒用的東西，不說太多理論的東西，只說我們十幾年來在IT工作過程中最常使用的東西，對的，這個東西它就是。

ACLs為什麼重要

ACLs為什麼重要呢？我們舉一個日常工作中最常碰到的場景作為例子，你所在的項目有一個項目文件夾叫做"openstack", 它的下面有兩個子目錄分別叫做“codes”和“docs”, 殿下你就是這個項目的owner, 你為你的組建立了一個名叫"clouddevelopers"的group, 好了，按照傳統的文件權限管理一定是這樣的，你本人對這個openstack目錄及子目錄一定有rwx的權限，你的組clouddevelopers同樣也有rwx的權限，others一定是沒有訪問權限的(others=全世界)，大家一直相安無事，有一天運營組的老大Tony來到你的辦公室，說我想要訪問你們的openstack下的docs這個文件夾，因為我們經常要查看裡面的文件，可以嗎？當然可以啊，這個要求一點都不過分，因為docs這個文件夾本來就是要共享給其他組成員看的，但問題來了，按照傳統的文件權限設定方法，你如何單獨開放這個docs給tony呢？你要是把tony加進clouddevelopers這個組，他就不止僅僅有docs文件夾的訪問權限了，codes目錄他也能訪問了，可是你們code寫的那麼爛，你好意思讓他看了去嗎？如果你們code寫的很好，他拷走了怎麼辦？他一不小心刪除了又怎麼辦？ 當然你可以說codes由git來管理，但git不也是在一個目錄下嗎？ 你可能又想我把others的權限設定成只讀，他就有權限查看了，啊完美，但是同時全世界也都有權限看到你們的docs了，這樣真的好嗎？你能確定一旦老闆看到你們的docs裡面哪些排版如狗啃一樣的東西他不會生氣嗎？

那麼如何做到單獨給Tony設定查看docs這個子目錄的權限又讓他沒有辦法查看其它子目錄的內容呢？這就需要用到ACLs， ACLs是為了更加精細的設定文件/夾的訪問權限而存在的。

哪些文件系統支持ACLs？

ext3及以後的文件系統都可以支持ACLs，但Centos6及以前的版本默認安裝沒有ACLs的管理包，需要手工安裝並啟用，Centos7默認就安裝並啟用了。

如果你是CentOS6或以前版本，安裝一下：

yum install libacl acl

然後用下面兩個命令之一查看一下

tune2fs -l /dev/mapper/vg_basesystem-lv_rootdumpe2fs -h /dev/mapper/vg_basesystem-lv_root

如果出現圖上紅線這一行就表示已經支持，沒有就用下面命令啟用

mount -o remount,acl /

這是臨時起效，要永久起效就修改/etc/fstab把選項加進去

vim /etc/fstab

在要啟用acl的那一個掛載項的defaults後面加入逗號和acl,例如

/dev/mapper/vg_basesystem-lv_root / ext4 defaults,acl 1 1

這就啟用了。 另外提一下，CentOS7默認強制開啟ACLs所以就不用做上面這些了。

使用ACLs控制文件訪問

回到上面那裡例子，我們如何給Tony添加對docs這個文件夾的只讀訪問權限而不讓他訪問其它地方？做法如下：

setfacl -m u:tony:r /openstack/docs/

當然你也可以對某個特定組做同樣的操作，把只讀權限賦予給一整個組

setfacl -m g:operationteam:r /openstack/docs/

做完後用getfacl docs/ 查看權限變化，以後你想把對另外一個運營組的同事做相同的設定就把他加到這個operationteam組裡面就可以了，如何加一個用戶到組裡面，看我們的上一篇文章。

如果過了一段時間Tony離職了，你想把他的權限刪除，命令如下：

setfacl -x u:tony /openstack/docs/

就刪除了Tony所有的權限，刪除組的權限類似。
