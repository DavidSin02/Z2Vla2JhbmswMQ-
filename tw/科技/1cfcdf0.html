乾貨｜STM32 SDIO折騰記

2020-10-29
| 科技

月初打樣回來的PCB，斷斷續續搞到現在，離完成度還有距離。這塊被我老婆稱為“背心”的板子，其實是我的STM32F4實驗平臺（Nucleo我有，但是光插針用起來也不方便，還得擴展卡）。這塊板子的佈局是為一個數字音樂播放器設計的，但現在軟件工作還缺很多。

這個月裡實驗過了STM32F411的時鐘配置、DMA、I2C、I2S、SDIO. 其中 SDIO 折騰得最久，拿出來和大家分享下。

SDIO這個模塊主要是為了讀寫SD存儲卡（也可以操作SDIO wifi卡之類的），ST從STM32F10x開始就已經支持SDIO/SDMMC了，因為是硬件專用模塊，比用SPI方式讀寫SD卡效率要高很多，軟件也大大簡化。我曾經（五年前的事了）在FPGA上為了記錄數據，用SPI方式寫過SD卡，因為狀態機太複雜，用起來很不好用。在8位MCU的時代，用SPI方式操作SD卡已比較流行，可以找到不少的參考代碼。SD卡數據線有4條，在SPI模式下就利用不了全部的帶寬，STM32的SDIO則是全部數據線都能用上的。

上圖是常見標準尺寸SD卡和MicroSD卡(和TF是同樣東西不同叫法)的引腳，下圖是STM32 SDIO模塊的硬件連接，對應Card bus的引腳連到SD卡(卡座)上。不過，SD卡只有D0..D3四條數據線，STM32的SDIO_D7..4這四條多出來的線以及8位寬度模式是給什麼用的？我也不知道。反正多的線也沒地方連。

焊好SD卡座之後：對應SD卡pin都接到MCU上了。卡座上另有CD引腳是檢測卡插入的，卡插入後CD與GND連通。還有WP引腳是識別寫保護開關用的，我沒連它。

個人的DIY就是折騰，ST官方庫已經有SDIO的函數了為何不用？重新發明輪子？我就喜歡這麼幹，不喜歡煩瑣的一層層打包的庫函數啊，直接寫寄存器才好玩

先得知道這些信號都幹嗎的才是，ST手冊這個圖表示得差不多了

除了 SDIO_CK 是時鐘，只由MCU產生之外，SDIO_CMD和SDIO_Dn都是雙向的I/O。SDIO_CMD 是命令通道，由MCU寫命令，SD卡寫應答；兩邊交替驅動，都不驅動時為高阻態。SDIO_Dn是實際傳輸存儲數據才用的，在讀卡的時候被SD卡驅動，在寫卡的時候由MCU驅動寫數據和校驗，SD卡響應狀態。具體時序已經被MCU硬件搞定，CRC計算都已經做好，軟件只需要操作SDIO設備APB總線上的寄存器就可以了，要說簡單也的確不難。

上電的操作：檢測是否卡片插入了卡座，再使能SDIO模塊，給SD卡供電(我加了一個PMOS管做電源開關)，配置SDIO時鐘，然後等一小會兒就可以發送SD命令了。

SD卡/MMC卡在初始化階段，時鐘頻率不能超過400kHz，所以一開始時鐘要設得慢，待初始化完成後再調整。CLKCR這個寄存器還是容易看懂的。在STM32F411上，SDIO部分和USB OTG公用一個48MHz的時鐘，所以分頻以48MHz計算，而不是從CPU運行頻率或APB的頻率來算。

第一條SD命令應當是 GO_IDLE_STATE, 也就是 CMD0, 將SD卡恢復到IDLE狀態。不過看SD規範的文檔，這條命令是沒有應答的，所以執行成功與否也無從知道，比如電路本身就有問題的話……

發送命令還不難，寫SDIO_ARG和SDIO_CMD寄存器就是了，命令號在SDIO_CMD裡。MCU裡面有一個狀態機來實現命令過程(Command Path State Machine)：

寫命令寄存器，將CPMSEN位置1，才會開始命令過程。然後，狀態會反映在 SDIO_STA 這個寄存器中。怎麼知道 CMD0 已經完成了呢？SDIO_STA 裡面有個 CMDSENT 位. 於是我是這麼寫的

接下來該繼續初始化的其它命令了。按文檔中的介紹，需要告訴SD卡它操作的電壓，不滿足的不能繼續（呃，難道不是已經給電了才能初始化麼），好吧，需要 SEND_OP_COND (CMD1)

我發送了 CMD1 卻沒有任何響應，搜了下網上資料，這條命令是MMC卡支持的，SD卡不支持…… (汗，ST也寫該註明一下吧) 根據我下載的 "SD Physical Layer Spec 2.0" 文檔，應該使用 CMD8:

於是我發送CMD8, 設置電壓為2.7~3.6V檔，SD卡依然沒有反應。

哪裡出問題了？命令返回的狀態是 timeout, 我用簡易示波器看了的確一直沒有應答。CMD0 沒有起作用？上電再多延時，還是不行。CMD0 和 CMD8 之間延時，也不行。PCB做錯了？ 又核對一次的確CMD, CLK都沒連錯啊。從命令表裡找了條別的命令來試，也沒有響應。換個卡吧，這個實驗用的是32MB的，換2GB的，以及4GB的SDHC卡，都還是沒有應答……

搜網上的初始化代碼參考吧（然後我發現——天下代碼一大抄啊）。發現一個問題是，CMD8 是 SD規範2.0版本才支持的，我的卡可能根本不支持呢。根據CMD8是否響應，判斷是否是2.0的卡。什麼是2.0？又補了補知識，原來SDHC是2.0以後的，SDXC是3.0以後的了。我只知道SD卡非HC的最大隻支持到2GB.

好吧，我找到一個SD 1.10規範的文檔了。心急吃不了熱豆腐。

哦，1.0是說發送ACMD41, 也就是要先發一個 CMD55. 終於，我測試CMD55時，32M的SD卡發出了應答。硬件是沒問題的

接下來發送 ACMD41, 卻發現返回了CRC錯誤！ 哦，原來這個命令特殊，應答類型沒有CRC. 補充說一下，SD命令的應答不盡相同，主要有長應答和短應答兩類，預期是長還是短應答需要在SDIO_CMD寄存器中設好，在RESPx寄存器中讀取返回的應答數據。

ACMD41 命令後，應該是CMD2 ALL_SEND_CID命令了。這時又出現了問題，又不響應了。

該仔細讀文檔的。ACMD41 命令會返回 busy 狀態，需要等待 busy 結束後才能繼續。

注意紅字，所以我套一個循環，要判斷下若busy則繼續循環。結果呢，沒有發生busy啊，一次就Pass了。

還會有哪個步驟出了問題呢？還得讀文檔，不能著急。我逐漸意識到，SD卡的初始化是一個命令接一個命令相扣的，內部狀態不同，能接收的命令也不同，必須一道一道門進入，如這個圖：

然後，裡我發現了一個烏龍：居然和上面的話矛盾，BUSY位是該1還是該0表示忙呢？

若我把busy位的判斷改成0繼續循環，就好了。重試若干次ACMD41後終於Pass. 在關於OCR寄存器的部分，文檔是這麼說的

這和我的實踐結果一致。哎，這個文檔裡面自相矛盾太害人了。我找的這個文檔是山寨的麼？？？

接下來還比較順利，幾個命令後進入到了Transfer state, 可以讀卡里面的塊數據了，並可以設置4-bit模式。

讀一個Block數據，直接用SD命令17, 給一個32位地址就行。需要注意的是，如果是SDHC卡，這個地址就不是字節地址而是塊地址了。涉及到的SDIO寄存器有 DTIMER(控制超時), DLEN (傳輸數量), DCTRL (數據傳輸使能等), FIFO (獲取讀來的數據)。STM32 SDIO負責數據的部分也有一個狀態機(Data Path State Machine)

不過，兩個狀態機都用同一個狀態寄存器STA（不同的位）. 我的讀程序目前是這麼寫的

這個子程序設好傳輸超時和最大長度，在發送完讀命令之後，才把DPSM打開（過早打開了會出錯）。然後等待FIFO中有數時，讀FIFO寄存器(一次4字節)寫到自己的buffer裡，直到數據塊傳輸成功的狀態，如果出現異常就終止傳輸。這麼查詢式讀數據效率不高，因為現在程序也不幹別的就罷了。將來做播放器我肯定會用DMA來傳數據的。

折騰中出現的插曲還有：32M的SD卡設置時鐘到12MHz也正常讀，但我另一張2GB的卡最多隻能設到1MHz，再快就timeout了。後來用示波器看，發現讀命令應答後到出現數據這段等待的時間很長，導致了超時。DTIMER寄存器是按clock個數來定的，SD時鐘越快，超時就越短了。我這個2GB的卡，第一次讀等待的時間長，後面連續讀等待就縮短了。
