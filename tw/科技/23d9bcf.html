概念辨析：同步/異步，阻塞/非阻塞

2020-10-29
| 科技

同步與異步

同步與異步要從服務調用方的視角區分.

以下發 Flink 計算任務為例:

我們調用 Flink 的 API 接口提交計算任務,

  • 如果需要等到 Flink 計算完成後, API 再響應, 那麼對於調用方來說, 這個 API 是同步的;
  • 如果不需要等到 Flink 計算完成, 調用 API 後立即響應一個類似於 JobId 之類的中間結果, 調用方需要通過中間結果去查詢計算任務的進度, 那麼對於調用方來說, 這個 API 就是異步的.

阻塞與非阻塞

阻塞與非阻塞是針對 CPU 而言的.

以上面的例子繼續談, 當調用方調用一個同步 API 時, 在等待服務方返回結果期間,

  • 如果調用方的 CPU 也陷入等待, 那麼這就是阻塞的;
  • 如果調用的 CPU 掛起, 趁機去執行其他的操作, 那麼這就是非阻塞的.

舉例

人們常說, JavaScript 是一門異步編程語言, 就是因為它的很多接口都是異步的, 調用一個接口時返回的是一箇中間結果, 而不是最終結果. 為了解決異步編程的困擾, JS 先後有 callback, Promise, Async/Await 等編程方案.

為什麼 JavaScript 會設計成異步的呢? 同步明明更省事啊. 因為 JavaScript 在瀏覽器上最初設計時是單線程的, 如果設計成同步的, 調用一些耗時接口時, 該進程就會阻塞, 後續的操作就無法執行了.
