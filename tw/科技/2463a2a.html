面向對象的可複用設計模式-裝飾模式（java代碼實現）

又快到到週末了，需要向領導彙報工作，這周確實進展不太順利，心裡總是提心吊膽的。正在一籌莫展時，突然來了個“美女”，正要好好欣賞一下，仔細一看這不是公司的小黑嗎？一打扮也是個美女！突來靈感，我也把我的彙報內容給裝飾一下，說幹就幹，三下五除二就弄好了。我先向領導彙報最近處理生產問題佔了一部分時間，指導新員工又花了不少時間，導致自己真正的工作沒有太大進展，z再彙報自己負責的項目進度......

上面的解決方法可以理解為一個常見的模式，裝飾模式。

裝飾模式（Decorator Pattern）：動態的給一個對象額外的職責。就增加功能來說，裝飾模式相比生成子類更為靈活。裝飾模式是對繼承的補充。

先看看裝飾模式的通用類圖：

裝飾模式的通用類圖

其中的角色說明：

  • Component抽象構件，通常為一個接口或者抽象類，是核心對象，也就是最原始的對象。
  • ConcreteComponent具體的構建，是核心接口或者抽象類的具體實現。
  • Decorator抽象裝飾角色，一般為一個抽象類，裡面有個變量指向Component。
  • ConcreteDecorator具體的裝飾角色，要把Component裝飾成其他的。

來看下代碼，仔細看代碼，會發現很多有意思的東西。

package decorator;public class Client {    public static void main(String[] args) {        Component component = new ConcreteComponent();        component = new ConcreteDecorator1(component);        component = new ConcreteDecorator2(component);        component.operate();    }}/** * 抽象核心構件Component */interface Component {    void operate();}/** * 具體的核心構件 */class ConcreteComponent implements Component {    public void operate() {        System.out.println("我是核心類");    }}/** * 抽象裝飾著 */abstract class Decorator implements Component {    private Component component;    public Decorator(Component component) {        this.component = component;    }    //委託給被裝飾者執行    public void operate() {        component.operate();    }}/** * 具體的裝飾角色 */class ConcreteDecorator1 extends Decorator {    public ConcreteDecorator1(Component component) {        super(component);    }    @Override    public void operate() {        //調用裝飾方法        doSomething();        super.operate();    }    /**     * 裝飾方法     */    private void doSomething() {        System.out.println("ConcreteDecorator1 我來美化你");    }}/** * 具體的裝飾角色 */class ConcreteDecorator2 extends Decorator {    public ConcreteDecorator2(Component component) {        super(component);    }    @Override    public void operate() {        //調用裝飾方法        doSomething();        super.operate();    }    /**     * 裝飾方法     */    private void doSomething() {        System.out.println("ConcreteDecorator2 我來美化你");    }}

輸出結果如下，可見在核心邏輯執行前，動態的添加了一下額外的功能。

ConcreteDecorator2 我來美化你ConcreteDecorator1 我來美化你我是核心類

裝飾模式的優缺點：

1、優點：鬆耦合，裝飾和被裝飾者獨立發展，是繼承關係的補充，可以動態的添加功能。

2、缺點：多層的裝飾增加複雜性。
