(原創)Java的ReentrantLock(可重入鎖)詳解上篇

2020-11-14
| 科技

Java併發包下的ReentrantLock(可重入鎖)，實現了高併發下的加鎖機制，原理是通過AQS(AbstractQueuedSynchronizer，抽象的隊列同步器)中循環調用CAS(compareAndSet,原子性操作)操作來實現加鎖，它的性能比較好是因為盡最大努力的避免了使線程進入內核態的阻塞狀態，Java併發包下的很多鎖功能，本質都是使用了AQS(AbstractQueuedSynchronizer，抽象的隊列同步器)，該實現的作者是Doug Lea，ThreadPoolExecutor線程池也是這位外國大佬的傑作。

先看一下ReentrantLock(可重入鎖)的類結構，如下圖：

ReentrantLock(可重入鎖)的類結構

ReentrantLock類包含三個內部類，分別是NonfairSync(非公平鎖)，FairSync(公平鎖)，Sync(同步器)，其中Sync(同步器)又繼承了AbstractQueuedSynchronizer(AQS),本質也是由AQS實現了絕大部分的鎖功能。

實例化創建公平鎖和非公平鎖，默認創建非公平鎖。

實例化創建公平鎖和非公平鎖

公平鎖和非公平鎖的差別在於：

公平鎖：先請求鎖的線程會優先獲取到鎖，原理是所有需要獲得鎖的線程都會進入到隊列中，隊列的特點是先進先出，先進入的請求線程會在頭部，後進入的請求線程都會在隊列的尾部。

非公平鎖：則不會按照線程請求獲得鎖的先後順序，會立馬進行一次獲取鎖的請求操作。

在源碼中體現在，公平鎖會判斷一下隊列中是否已經存在等待獲取鎖的線程，如果存在，則把自己加入到隊列的尾部。

公平鎖獲取鎖

非公平鎖，不會判斷隊列中是否已經存在等待獲取鎖的線程，而是直接進行一次獲取鎖的操作。

非公平鎖獲得鎖

我們以公平鎖的加鎖操作，來看一下加鎖的過程，進入lock.lock()方法，最終會調用如下這個方法。

獲取鎖操作

這個方法中包含三個方法，分別是tryAcquire(),addWaiter(),acquireQueued(),每個方法的用途如下。

tryAcquire():嘗試獲取一次鎖，如果獲取鎖失敗，則會調用addWaiter()方法。在tryAcquire()方法中，先獲取volatile變量修飾的state，該status就是多線程爭搶的抽象鎖，status等於0，代表鎖沒有被使用，status等於1或其他，則代表鎖已經被一個線程佔用。

hasQueuedPredecessors()判斷線程中是否存在等待獲取鎖的線程。compareAndSetState(0, acquires)將status的值更新為1，代表鎖已經被佔用。setExclusiveOwnerThread()將當前線程設置為獨佔此鎖的線程。在else if的邏輯中，則是ReentrantLock(可重入鎖)的邏輯體現，意是如果還是獨佔這個鎖的線程進行了加鎖操作，則status的值在加1，這樣的話解鎖也需要多次解鎖。

tryAcquire()方法

addWaiter():將自己包裝成Node，然後添加到等待隊列的尾部，加入隊列成功後，則會調用acquireQueued()方法。

addWaiter()方法

acquireQueued():在加入到隊列中後仍進行嘗試獲取鎖的操作，然後判斷是否需要將該線程中斷掛起。

final Node p = node.predecessor()獲取當前線程所在Node的的前置Node節點。p == head && tryAcquire(arg)如果前置節點是head頭節點，則進行一次獲取鎖的操作，頭節點是一個空節點，head頭節點後的第一個節點，代表同步隊列中的第一個等待獲取鎖的線程，如果成功獲取鎖則返回，不需要中斷此線程。shouldParkAfterFailedAcquire()如果上面獲取鎖失敗後，判斷是否需要中斷此線程。parkAndCheckInterrupt()進行線程中斷的實際操作。

acquireQueued()方法

下面來重點看下一shouldParkAfterFailedAcquire()方法，參數為當前線程所屬的Node，以及它的前置節點Node，Node中定義瞭如下幾個waitStatus，

static final int CANCELLED = 1;//含義是該節點包含的線程已取消

static final int SIGNAL = -1;//含義是該節點包含的線程等待喚醒

static final int CONDITION = -2;//含義是該節點包含的線程等待條件喚醒

static final int PROPAGATE = -3;//這個是共享模式用到的狀態，前三個是獨佔模式用到的，我們的ReentrantLock是獨佔鎖，會用到上面的三個狀態

shouldParkAfterFailedAcquire()方法

第一個if的含義是：如果前置節點的的waitStatus狀態是SIGNAL，那麼前置節點的線程在等待喚醒，所以當前節點的線程可以中斷掛起，因為公平鎖是順序獲得的，當前線程的前面已經有一個線程在等待喚醒了，所以可以把當前線程掛起。

第二個if的含義是：如果前置節點的的waitStatus狀態是CANCELLED，那麼需要在隊列中移除該節點，因為該線程已經取消。

第三個else的含義是，把前置節點的waitStatus更新為SIGNAL，為什麼會這樣，因為我們的acquireQueued()方法是一個自旋的方法for (;;)，第二次循環時，就會判斷出把當前線程中斷，下面舉例說明。

比如我們有兩個線程A，B，第一個線程A已經獲取到鎖了，B在獲取鎖的時候肯定獲取不到，則會把B包裝成Node，然後加入到同步的等待隊列中，此時隊列中只有一個head頭節點和線程B所在的Node節點，head頭節點在初始化時，它的waitStatus等於0，在第一次循環時，把頭節點的waitStatus更新為SIGNAL，也就是-1，第二次循環時，因為head節點額waitStatus等於-1，所以在第一個if時，直接返回了true，告訴我們可以把當前線程中斷掉。

在parkAndCheckInterrupt()方法中，使用了LockSupport類的方法，將當前線程進行了中斷操作，

線程中斷的實際方法

非公平鎖的加鎖邏輯，只是會優先進行一次獲取鎖的邏輯，忽略掉隊列中是否存在等待線程的判斷，在獲取不到鎖時，進入隊列以及後續在隊列中獲取鎖的邏輯和公平鎖一樣。

以上就是加鎖過程的實際調用邏輯，簡單總結來說就是A,B,C三個併發線程，共同爭奪抽象鎖status，A優先獲取到鎖後，B，C按照調用順序進入同步的等待隊列，中斷自己的線程，然後在隊列中按照順序等待獲取鎖。
