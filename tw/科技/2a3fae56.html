100行代碼實現一個簡單有趣的記錄鼠標走過距離的小程序

2020-10-29
| 科技

100行代碼實現一個簡單有趣的記錄鼠標走過距離的小程序，稍加修改可以記錄鼠標走過的位置和所作的操作

記著那時候的我們還在搗鼓Windows下的軟件開發，模仿使用MFC寫程序，奈何MFC的消息驅動、宏、虛函數，層層繼承，整個框架定義的實在是太繞了，搞不清楚到底是如何運作的，消息函數怎麼關聯，何時派發等等，實在是摸不清楚門道。找了一段時間後發現Windows下也可以通過原生Win32編程，而MFC則是Win32編程的抽象和封裝。

一個窗口程序，分為2部分，窗口UI和對應的消息處理邏輯。

1、操作系統會提供一些UI組件，比如按鈕，文本框，進度條等等，程序只要調用系統API就可以在顯示器上畫出UI的圖形。

2、整個Windows程序是基於消息驅動的。操作系統負責管理外設（鍵盤、鼠標），統一管理各種請求，並負責將這些輸入/請求，轉化為一個通知，告訴應用程序某個事情發生了。比如單擊鼠標、改變窗口尺寸、按下鍵盤上的一個鍵。應用程序收到消息後就要處理。

Windows下的消息型如下圖。系統會給每一個程序創建一個隊列用來存放消息結構體，每產生1個消息就會增加到隊尾。應用程序則有一個死循環不斷的從隊頭讀消息，並調用用戶定義的函數處理。隊列中沒有消息時就等待，整個過程就這麼簡單。


Windows消息處理示意圖

WinMain是入口函數，通過DialogBox API，按照IDD_DIALOG1模板創建了一個對話框UI，並將WndDlg作為消息處理函數

Wint WINAPI WinMain(HINSTANCE hInstance,HINSTANCE hPreInstance,LPSTR lpCmdLine, int nShowCmd){	hInst=hInstance;	if (__argc == 1)						DialogBox(hInst,"IDD_DIALOG1",NULL,WndDlg);     //DialogBox封裝了窗口UI，WndDlg是消息處理函數	else																																							//也就是示意圖中的WndProc	{		HANDLE hProcessOrig = (HANDLE) _ttoi(__targv[1]);		WaitForSingleObject(hProcessOrig, INFINITE);		CloseHandle(hProcessOrig);		DeleteFile(__targv[2]);	} 	return 1;}

穿透模式，當鼠標向桌面外側滑動時，會自動從另一側進入，也是在定時器中檢測鼠標位置，實時修改。

刪除自身，其實就是把自己拷貝一份到系統臨時目錄下，拷貝時使用關閉後刪除，很多卸載程序應該也是這樣刪除自身的。


正常情況下要通過調用代碼逐步創建UI，但這樣不但繁瑣，而且易錯，基本上沒法搞。因此通過資源文件，可以預先設計好程序的UI和每個組件的ID和屬性，通過系統提供的API函數一次性加載創建。

Linux下的窗口處理程序也是基於消息驅動的，這種方式的好處是，實現了操作系統和程序之間的解耦，過程是異步的，沒有消息處理時就阻塞等待。系統增加新消息時應用程序可以不用例會，因為後邊還有系統提供過的默認函數處理，應用程序只處理自身關心的部分，擴展性很強。

現在的軟件開發模型，抽象層次越來越高，個人認為不管學什麼，如果時間有富餘，還是要抽點時間瞭解些底層的實現方式，這樣能更好的掌握所學知識。
