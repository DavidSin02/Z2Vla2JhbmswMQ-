關於使用機器學習識別 API 參考文檔中的知識

2020-10-29
| 科技


引用：Fucci D, Mollaalizadehbahnemiri A, Maalej W, et al. On using machine learning to identify knowledge in API reference documentation[J]. foundations of software engineering, 2019: 109-119.

摘要：使用 API 參考文檔（如 JavaDoc）是軟件開發的不可或缺的組成部分。先前的研究引入了分類法，該分類法劃分了 12 種類型的 API 文檔知識，包括有關 API 的功能，結構和質量的知識。我們研究了現代文本分類方法如何很好地自動識別包含特定知識類型的文檔。 我們將傳統的機器學習方法（k-NN 和 SVM）與在手動註釋的 Java 文檔、.NET API 文檔上訓練的深度學習方法進行了比較。當分別對知識類型進行分類（例如，多重二分類器）時，最佳 AUPRC 達到 87％。深度學習和 SVM 分類器看起來是互補的。對於四個知識類型（概念，控制，模式和非信息），SVM 明顯優於深度學習，而對於其餘的知識類型則是深度學習更準確。當一次考慮多種知識類型（即多標籤分類）時，深度學習的表現優於簡單基線方法和傳統機器學習，MacroAUC 可達 79％。我們還使用在通用文本語料庫和 StackOverflow 上預訓練的 embeddings 對分類器進行了比較，但未觀察到明顯的改進。最後，為了評估分類器的普遍性，我們在另一個外部不可見的 Python 文檔數據集中對它們進行了重新測試。功能，概念，目的，模式和指令的分類器似乎能夠從 Java 文檔和.NET 文檔適用到 Python 文檔。 我們討論了結果以及它們如何為支持開發人員共享和訪問 API 知識的工具開發提供信息。

1. 引言

軟件開發人員通過應用程序編程接口（APIs）來重用庫和框架。他們通常依靠參考文檔來確定哪些 API 元素與手頭任務相關，或者如何實例化，配置和組合 API。與其他知識來源（例如教程和 Q＆A 門戶）相比，像 JavaDoc 和 PyDoc 等參考文檔被視為官方 API 技術文檔。 它們提供有關 API 元素，組件，操作和結構的詳細和基本信息。

由於 API 文檔可能長達數千頁，訪問其中的特定知識可能既繁瑣又耗時。完成任務所需的信息可能散佈在多個元素（例如類，方法和屬性）的文檔頁面上。因此，開發人員嘗試使用其他資源來滿足他們的信息需求。例如，儘管 Java Development Kit（JDK）API 文檔包含 7000 多個頁面，但是截至 2019 年初，StackOverflow 上已經有超過 300 萬個 Java 相關的帖子。

在過去的十年中，軟件工程研究人員研究了信息開發人員在諮詢 API 文檔時所需的信息。 一項研究重點是使信息需求與文檔中可用的知識類型相匹配。Maalej 和 Robillard 通過開發 API 參考文檔中找到的 12 種知識類型的經驗驗證分類法，朝著這個方向邁出了第一步。一個文檔頁面可以包含多種知識類型（圖 1）。功能和指令是完成開發任務所需的特定知識類型，而信息類型僅包含無信息的樣板文本。Maalej 和 Robillard 認為，他們的知識分類方式能夠做到理解和改善文檔質量和滿足開發人員的信息需求。

研究團隊研究了 API 參考文檔中的特定的知識類型。 例如，Montperrus 等和 Seid 等研究了使用指令類型的知識以防止違反 API 使用限制。Robillard 和 Chhetri 在向開發人員推薦 API 時過濾掉了非信息類型的知識。目前所推崇的自動化方法都是基於語言特徵工程或句法模式。

這項工作調查了現代文本分類方法如何很好地自動識別 Maalej 和 Robillard 在 API 文檔中建議的知識類型。基於 5574 個 Java 和.NET 的文檔的數據集，我們訓練、測試和比較了傳統的機器學習方法（如 k-NN 和支持向量機 SVM）以及深度學習方法（如 LSTM 和 RNN）。 RNN 從通用文本的語義表示中學習特徵。因此，我們研究了使用來自 StackOverflow 的軟件開發專用語料庫訓練的神經網絡對結果造成的影響。 最後，我們研究了從 Python 標準庫獲得的未知數據集的分類器的普遍使用性。

本文做出了三點貢獻。第一，我們為 API 文檔提供了詳細的分類基準。第二，當我們共享本研究的代碼和數據時，幾個表現最好的分類器（例如 AUPRC≥80％）已經具有關聯性。第三，我們的發現和相關工作的討論為研究人員，工具供應商和從業人員提供了有關機器學習如何幫助更好地組織，訪問和共享 API 知識的見解。

2. 研究設置

2.1 研究問題和方法

Maalej 和 Robillard 提出了一種經過實證驗證的 12 種知識類型的分類法。表 1 列出了這些知識類型，它們代表了這項工作的基礎。我們的主要目標是研究在沒有附加特徵工程或 NLP 技術的情況下，用於文本分類的簡單機器學習如何識別這些知識類型。也就是說，我們的分類器使用一種或多種知識類型來標記文檔。

本文研究了兩種主要的文本分類方法。傳統方法通常從訓練集中某些關鍵詞或短語的出現中學習。計算密集型方法（通常指深度學習）使用關鍵字的語義-即關鍵字的上下文。

對於傳統方法，我們研究了兩種經常用於文本分類的算法：k-NN 和 SVM。對於深度學習，我們將 RNN 與 LSTM 層一起使用，這對於文本分類問題特別有效。

本研究解決的任務是給 API 文檔標記知識類型。由於文檔可以包含多個知識類型，因此該任務被建模為一個多重二分類問題，包括針對每種知識類型獨立訓練一個二進制分類器。另一種方法是訓練多標籤分類器，例如一個會輸出一組知識類型而不是單個知識類型的分類器。當回答以下研究問題時，我們將分析並展示兩種方法的結果。

RQ1：基於文本的分類器如何很好地識別 API 參考文檔中的知識類型？ 特別是，深度學習能否優於傳統方法？

對於文本分類任務，RNN 的輸入層通常由在大型的無標記文本語料庫上訓練的 embeddings 組成，以捕獲豐富的語義特徵。經過預訓練的 embeddings 可以輕鬆地嵌入神經網絡中。 但是，儘管這些嵌入節省了計算時間並很好地代表了常見的語言任務，但它們可能會丟失軟件工程或特定於 API 的語義。這引出了我們的第二個研究問題。

RQ2：與通用軟件相比，特定於軟件開發的文本嵌入是否可以改善分類結果？

機器學習評估的一個常見問題是在某個數據集上訓練的模型是否可以推廣到其他數據集。原始數據集包括標準 Java 和.NET 庫的文檔。由於我們旨在評估以不同風格編寫的 API 參考文檔的方法的通用性，因此我們手動註釋了從 PythonStandardLibrary 文檔中採樣的新數據集。我們用此數據集作為附加測試集來展示分類器的性能。

RQ3：是否可以跨 API 歸納出基於知識類型的文檔分類？

我們使用 10％的數據集作為測試集，基於 10 倍交叉驗證方式來評估模型。在比較各個知識類型分類器時，我們展示了精確召回曲線下的面積（AUPRC）。 精確召回曲線是一種用於評估二元分類的通用方法，可以通過在不同概率閾值處繪製精確度和召回值來獲得。特別是，它們用於評估在不平衡數據集上訓練的機器學習模型。 因此，無論特定閾值如何，AUPRC 都是性能的彙總度量。在比較多種知識類型的分類器時，0 們根據兩種類型的指標（基於項目和基於標籤）報告績效。基於項目的度量標準是：a）Hamming Loss，即錯誤分類的標籤與標籤總數之比（最佳值為零）和 b）Subset Accuracy，即預測標籤集與實際標籤集之間完全匹配的百分比

基於標籤的指標是精度，召回率，F1 量度（公式 1）和 ROC 曲線下面積（AUC）。

ROC 曲線是通過在不同的概率閾值下針對誤報率（FPR，公式 2）繪製召回圖來創建的。 因此，AUC 不取決於特定閾值。 為了計算陽性，假陽性和假陰性的值，我們使用 0.5 作為概率閾值。

基於標籤的指標是宏平均的。宏平均將度量應用於每個預測標籤的二分類，然後對結果取平均值——即標籤對最終結果的貢獻相等。相反，微平均首先彙總每個標籤的各個指標成分（即陽性，假陽性，陰性和假陰性），然後將它們平均。 因此，微平均偏向多數類別，在評估不平衡數據集時應避免。 我們將分類器的結果與原始基線 MF1，MF2 和 RAND 進行比較。 前兩個總是為每個文檔分配第一個（分別對前兩個而言）最頻繁的標籤，而後者則分配一個隨機標籤。

2.2 研究數據

我們使用 Maalej 和 Robillard 創建的 CaDO 數據集作為對 JDK 6 和.NET 4.0 API 參考文檔的內容分析的結果。CaDO 包含 5,574 個觀測值。數據集中包括 API 元素的名稱（例如，類，方法或屬性），其文檔文本以及 12 個二進制數值，這些值指示相應知識類型的存在（或不存在）。表 2 總結了數據集的文本屬性。

大多數文檔（90.5％）包含 12 種知識類型中的 1-5 種。我們使用 SCUMBLE 分數（∈[0，1]）報告不平衡程度。對於給定的標籤，較高的 SCUMBLE 得分表示所有其他同時出現的標籤的頻率之間存在較大差異。通常，高分的數據集對於分類任務是有問題的。但是，對於較低的 SCUMBLE 得分的數據集，重採樣可以減少不平衡性。 CaDO 數據集的平均 SCUMBLE 得分為 0.11。

我們對數據集的 90％（即訓練集）應用了隨機欠採樣和過採樣。 我們沒有對測試集（數據集的 10％）進行重新採樣以避免採樣偏差。

為了進行重採樣，我們刪除了標籤集中 30％的包含功能和非信息的文檔，並複製了 50％的包含環境和質量的文檔。閾值是根據 SCUMBLE 得分憑經驗獲得的。重新採樣後，訓練集和測試集分別包含 3876 和 430 個觀測值。圖 2 展示了重新採樣後用於訓練模型的數據集中的標籤頻率。

我們從 Python 2.7 標準庫中準備了一個由 100 個 API 文檔頁面（即模塊，類型，屬性和方法）組成的新 Python 數據集。我們選擇 Python 標準庫是因為其代碼的組織方式與 Java 和.NET 不同。它廣泛使用定義了功能，類和變量的模塊。Python 編程範例比 Java 和.NET 更具功能性，而 Java 和.NET 則遵循面向對象的範例。Python 是動態類型的，其參考文檔通常集中在功能上，而類型文檔可嵌入到源代碼中（例如，通過 Docstrings）。 最後，它的開發和文檔由一個開源的非營利社區（Python 軟件基金會）驅動，而 Java 和.NET 則歸公司所有。我們遵循了 Maalej 和 Robillard 建議的抽樣策略，即分層隨機抽樣。我們首先為每個基本模塊創建了層次，然後從每個層次按其頻率成比例地對 API 文檔隨機採樣。

兩位習慣使用 Python 的軟件工程專業博士生在每個文檔中手動標記了知識類型。對於此任務，我們為他們提供了 Maalej 和 Robillard 的相同準則，並進行了少量修改，例如提供使用 Python 編程語言的示例。在標籤集上的一致率為 14％，即在 100 個示例中，有 14 個被完全相同的知識類型集標記。總體協議率為 75％——即，在 1200 個標籤（100 個示例 ×12 個標籤）中，有 300 個存在衝突。其中兩位作者解決了衝突並創建了最終數據集。圖 3 顯示了 Python 數據集中知識類型的分佈。圖 3 顯示了 Python 數據集中知識類型的分佈。功能性是最多數的標籤。Python 代表了一個額外的測試集（即沒有使用該數據集中的示例來訓練分類器），我們沒有對其進行重新採樣以避免產生偏差的結果。

對於這兩個數據集，我們執行了幾個簡單的操作來清理和準備文本數據。我們對 API 文檔文本進行了字母小寫化，標記化和刪除停用詞。然後，我們將術語轉換為 one-hot 向量並保留其順序。對於我們基準測試中的深度學習分類器，我們基於四個大型語料庫訓練了 GloVe embeddings，結果總結於表 3。Common Crawl（CC）是預訓練的 embeddings，於 2018 年 3 月下載。它包含 840B tokens 和一個 220 萬詞的詞彙表。語料庫包含從 Internet 爬取下來的高質量通用文本。但是，CC 語料庫在 CaDO 數據集中缺少特定領域術語。因此，在 CCOTF 的 embeddings 中，對 CC 語料庫中的缺失詞進行了即時訓練。最後，我們通過訓練兩個附加語料庫（StackOverflow（SO）和 StackOverflow API（SOapi））上的 embeddings 來獲得特定專業領域的表示形式的輸入。前者包括 2000 萬個帖子，而後者包括 400 萬個標記為 java 或.net 的帖子。

3. 分類器配置

3.1 傳統機器學習

我們為分類任務選擇的機器學習方法是 SVM 和 k-NN，以及它們對多標籤問題的適應版本，即 One-vs-Rest SVM（OvRSVM）和 Multi Label k-NN（ML-kNN）。我們使用從 CaDO 數據集中提取的 unigram 和 bigrams 作為輸入特徵，因為 n-gram 語言模型易於計算和使用。此外，它們還用於將機器學習和自然語言處理應用於軟件工程環境的研究中。

3.2 含 LSTM 層的 RNN

圖 4 顯示了我們在這項工作中使用的架構。該網絡由一個 LSTM 層，兩個稠密層和一個輸出層組成。LSTM 層中的單元數量與用於表示每個單詞的向量的維度成比例。稠密層分別包含 128 和 64 個單元。輸出層中的單元數是知識類型的數量（即 12 個單元）。LSTM 層的核心組件是一個存儲單元，用於存儲與網絡中以前的分析步驟有關的信息。在訓練的每個步驟中，網絡都基於 a）新輸入，b）RNN 其他隱藏層的前置狀態以及 c）存儲單元的當前狀態來預測輸出。門學會了如何修改存儲單元以提高預測精度（參見圖 5）。

遺忘單元在我們的網絡中的作用是在新輸入中更改與先前知識類型相關的信息時，以最佳方式丟棄這些信息。例如，當新輸入中的知識類型為“指令”時，“忘記門”將刪除與其他知識類型相關的信息。因此，當學習個別類型時，忘門減少了存儲單元的歧義性。當前輸入文檔中與知識類型相關聯的特徵將移動到存儲單元中。在存儲單元中，輸入門決定應存儲哪些信息。例如，當前輸入包含指令時，將使用輸入門提取其特徵並將其存儲在存儲單元中。 最後，輸出門選擇與指令類型相關的最重要的特徵。

由於具有相同長度的輸入向量加快了構建 RNN 嵌入層的過程，因此我們將 300 作為最大向量長度，並用零填充較短的向量。因此，我們的輸入層是一個 2D 矩陣，其中每一行都是一個 300 維單元（人工神經元）。輸入層中的單元數取決於用於訓練嵌入的語料庫的詞彙量。當將文檔送入 RNN 時，與文檔術語關聯的單位將被激活。第一個隱藏層是 LSTM 單元，用於學習輸入文檔的文本特徵。它適合於學習長期依賴性，例如在使用大型 API 參考文檔文本的情況下。為了防止過擬合，我們對權重矩陣和偏差向量應用了一種丟棄技術。

LSTM 層（即可以與知識類型相關聯的一組特徵）的輸出經過兩個完全連接的稠密層。稠密層提供了由 LSTM 層提取的特徵的深入表示，並使網絡能夠了解其層次結構和組成特徵。為了減輕由於要素從輸出層的高維空間投影到低維空間而導致的特徵損失，該模型將稠密層中的單元數從 128 個平滑地減少到 64 個。我們對稠密層使用 ReLU 激活功能，以防止過擬合。

輸出層使用 sigmoid 激活函數來預測知識類型。因此，輸出層中的單位數就是模型學習的標籤數。由於 sigmoid 函數的輸出是一個介於 0 和 1 之間的概率值，因此該層中的每個神經元都會估算觀察到某一個標記的概率。為了將預測概率二值化，我們根據不同的指標使用了不同的閾值。

單個知識類型。我們使用通用語料庫訓練了兩個基於 RNN 的分類器，以獲得輸入層——RNNCC 和 RNNCCotf 的 embeddings。表 4 展示了我們對分類器的評估。

對於所有單個知識類型分類，RNN 和傳統的機器學習方法比簡單基線可提高 74％（平均 41％）。SVM 比 k-NN 表現更好。

對功能，示例和環境類型進行分類時，深度學習在不同的概率閾值（AUPRC≥80％）上表現出了高精度和高召回率，優於傳統機器學習方法。RNN 在指令，目的，參考，概念和控制類型方面產生了低於標準的結果（AUPRC<50％）。 但是，僅對於後兩種類型，最佳 SVM 優於最佳 RNN。對於質量，結構，樣式和非信息類型的最佳分類器，其 AUPRC 介於 59％和 78％之間。同樣在這種情況下，僅針對後兩種類型，最佳機器學習方法（即 SVM）優於最佳 RNN 分類器。與機器學習相比，RNN 對八種知識類型進行了更好的分類。

多種知識類型。第二步是將我們的任務視為多標籤分類問題，而不是針對每種知識類型構建單獨的分類器。我們將 RNN 分類器與相同的兩個機器學習模型和兩個簡單的基線的多標籤適應進行了比較（參見表 5）。ML-kNN 和 OvRSVM 的性能比基於項目的指標的基線更差，而 RNNs 的性能最佳。在最嚴格的指標上（即子集準確性），最佳的 RNN 比機器學習分類器提高了 11％。對於 MacroPrecision，MacroRecall 和 MacroF1，RNN 有 25％到 28％的改進。 從 MacroAUC 方面看，MF1 的性能要優於傳統的機器學習，RNNs 的性能提高了 17％。

回答 RQ1：可以自動識別三分之一的知識類型，並且效果良好（即 AUPRC≥80％）。與傳統的機器學習方法相比，RNN 可以更準確地（> 10％）識別 12 種知識類型中的 8 種。在考慮多標籤分類時，RNN 在基於項目和標籤的指標方面優於傳統的機器學習方法。

4.2 特定於軟件開發的語料庫

單個知識類型。OneRNN 使用基於通用文本語料庫（即 RNNCC）的免費可用的預訓練 embeddings，而 RNNCCotf 使用相同的語料庫且可以從 CaDO 數據集中即時學習缺失的單詞。諸如 GloVe 之類的文本統計表示形式背後的假設是，文檔的含義取決於其中出現的單詞的含義。因此，RNNSO 和 RNNSOapi 在更接近 API 文檔之一的域中使用語料庫。如表 4 所示，在這些 RNN 中，最好的 RNN 的性能與普通 RNN 相似（∆AUPRC = 4％）。在功能，用途，控制和結構類型方面，差異最小（1%-2％）。但是，對於質量和環境，使用特定於軟件開發的 embeddings 時性能會大幅下降（分別為 10％和 14％）。總體而言，鑑於獲得語料庫和計算 embeddings 的開銷，這種改進相當有限。

多種知識類型。表 5 顯示，在考慮基於標籤的指標（MacroAUC 除外）時，RNNCC 和 RNNCCotf 優於 RNNSO 和 RNNSOapi，而在考慮基於項目的指標時，其表現也類似。使用 StackOverflow 上的 Java 和.NET 相關帖子訓練得到的 RNN 達到了最高的 MacroAUC（79％）。

回答 RQ ２：使用特定於軟件開發的 embeddings 的 RNN 與使用通用 embeddings 對單個知識類型進行分類的 RNN 相比，改善效果甚微。在考慮進行多標籤學習時，除了 MacroAUC 之外，使用通用 embeddings 在基於項目和標籤的指標上會產生更好的結果。

4.3 分類器普遍適用性

單個知識類型。表 6 報告了 Python 測試集上各個基於 RNN 的分類器的性能。同樣在這種情況下，沒有任何簡單的基準可以比傳統的或深度學習方法更好。RNN 是 7 種知識類型的最佳分類器，而其餘 5 種 SVM 則顯示出了最佳結果。與 CaDO 設置一致，SVM 是概念，模式和非信息類型的最佳分類器。與 CaDO 設置相比，功能，概念和用途類型的分類器有些許性能的改進（∆ AUPRC = 8.3％）。

在考慮指令，質量，控制，結構，示例和環境類型時，兩個設置之間存在較大的絕對差異（∆ AUPRC = 33％），這表明這些知識類型取決於設置。平均而言，在 12 種知識類型上，Python 數據集的性能下降了約 16％。

多種知識類型。表 7 列出了多標籤分類任務的結果。對於基於項目的指標，我們的分類器相對於簡單的基準而言表現較差或相當。分類器有較低的精度（最佳分類器為 SVM，40％）和召回率（最佳分類器為 RNNCCotf 和 RNNSOapi，26％）。 SVM 還獲得了最佳的 F1（30％）。 RNNSOapi 有最高的 MacroAUC（64％）。

回答 RQ ３：功能，概念，目的，模式和指令類型的分類器，似乎能夠從 Java 和.NET 文檔適用到 Python 文檔。但是多種知識類型分類器的適用性有限。
