HTTP/HTTPS協議簡介：一文看懂超文本傳輸協議

2020-11-14
| 科技

本文將向你介紹最基本的萬維網（WWW）的工作原理。

它的核心技術其實就是HTTP-超文本傳輸​​協議。這是你平常瀏覽網絡時使用的通信協議。從根本上來說，當你訪問網站時，瀏覽器會向服務器發出HTTP請求。然後，該服務器以資源（圖像，視頻或網頁的HTML）進行響應，然後您的瀏覽器會為你顯示這些來自服務器的資源。

這是HTTP的基於消息的基本模型。每個HTTP交互都包括一個請求和一個響應。而HTTP本質上是無狀態的。

無狀態意味著所有請求都是相互獨立的。因此，來自瀏覽器的每個請求必須單獨包含足夠的信息，服務器才能滿足該請求。這也意味著基於消息的HTTP模型的每件事務都必須與其他事務分開處理。

正在使用HTTP協議網上衝浪的老奶奶

網址

URL（統一資源定位符）可能是Web上最知名的概念了，它也是最重要和實用的概念之一。URL可以用於標識Web資源的具體地址。

Web的最初思想就是要圍繞資源構建，從一開始創立的時候，Web就是用於共享文本/ HTML文件，文檔，圖像等的平臺，因此可以將其視為資源的一個大集合。

URL示例

協議頭  -大多數情況下，它們是HTTP（或HTTPS（用於HTTP的安全版本））。

其他值得注意的協議是：

  • 文件傳輸協議（FTP）-是一種標準協議，用於通過網絡在客戶端和服務器之間傳輸文件。
  • 簡單郵件傳輸協議（SMTP）是電子郵件傳輸的標準。

域名  -名稱，用於標識資源所在的一個或多個IP地址。

路徑-指定服務器上的資源位置。它使用與你在閱讀本文的設備上所使用的資源位置採用相同的邏輯（即/search/Documents/C++從入門到入土.pdf 與 C:/我的文檔/C++從入門到入土.pdf）。

參數  -用於標識或過濾服務器上資源的其他數據。

注意：在搜索有關HTTP的文章和更多信息時，您可能會遇到術語URI（Uniform Resource Identifier）。有時使用URI代替URL，這個用法完全沒錯，這就是非常規範的術語！

HTTP請求

在HTTP中，每個請求都必須具有URL地址。此外，請求需要一個方法。四種主要的HTTP方法是：

  • GET
  • PUT
  • POST
  • DELETE

    • 本文會在“HTTP Method”的部分中解釋這些方法（就在下面！），以及包含更多內容。

    這些HTTP請求方法直接對應於動作：

    • 讀（read）
    • 更新（update）
    • 新建（create）
    • 刪除（delete）

    所有HTTP消息都有一個或多個標頭，後跟可選的消息正文。主體會包含與請求一起發送的數據或與響應一起接收的數據。

    HTTP請求方法

    HTTP請求中還有一些其他有趣的東西：

    Referer標頭  -告知請求發起的URL。

    User-Agent標頭  -有關用於生成請求的瀏覽器的其他信息。

    主機頭  -唯一標識主機名；同一服務器上託管多個網頁時，這就十分有必要了。

    Cookie標頭  -向客戶端提交其他參數。

    HTTP響應

    就像HTTP請求一樣，HTTP響應也包含三項：

    1. 第一部分是所使用的HTTP版本。
    2. 第二部分是請求結果的數字代碼。
    3. 第三部分是第二部分的文字描述。

    同樣的，HTTP響應中也有一些其他有趣的信息：

    服務器標頭  -有關正在使用哪種Web服務器的信息，比如CentOS，Nginx。

    Set-Cookie標頭  -將cookie發送給瀏覽器。

    消息正文  -HTTP響應通常會抓取消息正文。

    Content-Length標頭  -以字節為單位告知消息正文的大小。

    HTTP Method

    最常見的請求方法就是GET和POST。但是，還有一些其他的請求方法。

    GET  —使用此方法從指定的資源請求數據，而不會以任何方式修改數據。GET請求不會更改資源原本的狀態。

    POST-  使用此方法將數據發送到服務器以創建資源。

    PUT —使用此方法通過使用請求正文中的內容來更新服務器上的現有資源。這可以被視為“編輯”某些內容的一種方式。

    HEAD  —使用此方法的方式與使用GET的方式相同，但區別在於HEAD方法的返回結果不應在響應中包含主體。但是返回將包含與使用GET相同的標頭。可以使用HEAD請求方法在發出GET請求之前檢查資源是否存在。

    TRACE—使用此 方法用於診斷目的，響應將在其主體中包含請求消息的確切內容。

    OPTIONS  —使用此方法來描述可用於目標資源的通信選項（HTTP方法）。

    PATCH-  使用此方法可以對資源進行部分修改。

    DELETE —使用此方法刪除指定的資源。

    HTTP頭

    請求/響應結構由三個主要部分組成。這些包括：

  • First line
  • Headers
  • Body/Content

    • 我們已經討論過HTTP請求和響應的第一行（First line），並且還提到了body函數。現在，我們將討論HTTP標頭。

    HTTP標頭將添加在第一行之後，並定義為以冒號分隔的name：value。HTTP標頭用於與請求或響應一起發送其他參數。

    正如本文前面說過的那樣，消息的主體包括要與請求一起發送的數據或隨響應一起接收的數據。

    根據標頭的用法，可以將標頭分為4大類：

    • 常規標頭  -既可以在請求消息中也可以在響應消息中使用的標頭，它們與正在交換的數據無關。
    • 請求標頭  -這些標頭定義了所請求數據的參數或提供有關發出請求的客戶端的重要信息的參數。
    • 響應標頭  -這些標頭包含有關傳入響應的信息。
    • 實體標頭  -實體標頭描述了組成正文的內容。

    HTTP狀態碼

    瀏覽各種網站時，您可能會遇到“ 404 Error：未找到資源”頁面或“ 500 Error：服務器未響應”頁面。

    這些都是HTTP狀態碼。

    每個HTTP響應消息的第一行必須包含HTTP狀態代碼，用來告訴我們請求的結果。

    HTTP狀態碼

    有五組狀態碼，按數字的第一位分組：

    • 1xx -信息性
    • 2xx —請求成功。
    • 3xx —客戶端被重定向到其他資源。
    • 4xx —請求包含某種錯誤。
    • 5xx —服務器在滿足請求時遇到錯誤。

    HTTPS（安全超文本傳輸​​協議）

    HTTP協議的安全版本是安全的超文本傳輸​​協議（HTTPS）。HTTPS在瀏覽器（客戶端）和網站（服務器）之間提供加密的通信。

    在HTTPS中，使用傳輸層安全性（TLS）或安全套接字層（SSL）對通信協議進行加密。

    因此，該協議通常也稱為基於TLS的HTTP或基於SSL的HTTP。

    TLS和SSL協議均使用非對稱加密系統。非對稱加密系統使用公共密鑰（加密密鑰）和私有密鑰（解密密鑰）對消息進行加密。

    任何人都可以使用公共密鑰來加密消息。但是，私鑰是完全私密的，這意味著只有預期的接收者才能解密消息。

    SSL / TLS握手

    當你請求與網站的HTTPS連接時，該網站會將其SSL證書發送到您的瀏覽器。你的瀏覽器和網站開始通信的過程稱為“ SSL / TLS握手”。

    SSL / TLS握手會涉及一系列步驟，在這些步驟中，瀏覽器和網站會相互驗證並開始通過SSL / TLS隧道進行通信。

    你可能已經注意到了，在HTTPS連接期間使用受信任的安全隧道時，綠色的掛鎖圖標會顯示在瀏覽器的地址欄中。

    HTTPS的好處

    HTTPS的主要好處是：

    • 客戶信息（例如信用卡號和其他敏感信息）已加密，不能被截獲。
    • 訪問者可以驗證你是註冊企業，並且能夠證明你是該域名的擁有者。
    • 客戶知道他們不應該訪問沒有HTTPS的站點，他們可能更信任和完成從使用HTTPS的站點進行的購買。

    其實在我們的生活中，大多數瀏覽器已經對非https的網站進行了很多警告，當然比http更危險的是使用未經認證的https證書（比如自簽名證書），瀏覽器會盡全力禁止用戶去訪問這類網站，因為這些網站往往是最不安全的，所以如果要想搭建一個自己的網站，第一件事就是去可信的簽發機構去獲得一個HTTPS證書。
