面向對象設計之依賴倒置原則

2020-11-14
| 科技

概述

依賴倒置原則（Dependency Inversion Principle，DIP）的表述是指抽象不應到依賴細節；細節應當依賴於抽象。即要面向接口編程，不要針對具體的實現編程，這樣可以最大程度降低模塊之間的耦合。

深入理解

具體在Java程序代碼中，如何理解面向接口編程?其實，簡單的講，就是使用Java接口或抽象類進行變量的類型聲明、方法參數的類型聲明、方法返回值的類型聲明，以及數據類型的轉換等，而不應到或儘量避免在這些地方使用具體的Java類型進行聲明或類型轉換。舉個例子，比如訂單服務中，會調用（依賴）支付服務以完成訂單的支付。此時的訂單服務實現類代碼中，聲明或持有的應該是一個支付服務PayService接口，而不是具體實現類比如Alipay或WeChatPay，所有需要用到支付服務的地方，調用的都是PayService接口變量，即代碼中都是接口，等到在運行時才實例化或注入具體的實現類，編寫代碼時不需要關心具體實現類，也就是面向接口/抽象編程。這樣，在應用程序擴展或集成新的支付服務時，原來的訂單服務代碼可以不需要進行修改任何代碼，這樣也就符合了開閉原則。

要保證做到遵循依賴倒置原則，一個具體的Java類應當只實現Java接口或抽象類中聲明的方法，而不應當給出多餘的方法。

代碼示例

下面以訂單服務調用支付服務為例，看看在具體的代碼中如何遵循應用依賴倒置原則。

  • 支付服務接口代碼
// 1.支付服務接口public interface PayService {    // 提交支付    void pay();}
  • 現有的支付服務實現：微信支付服務
// 2.微信支付實現類public class WeChatPay implements PayService {    @Override    public void pay() {        // TODO 調用微信支付API    }}
  • 現有的支付服務實現：支付寶支付服務
// 3.支付寶支付實現類public class AliPay implements PayService {    @Override    public void pay() {        // TODO 調用支付寶API    }}
  • 依賴支付服務的訂單服務代碼
// 4.1業務方代碼：訂單服務接口public interface OrderService {    void pay();}// 4.2訂單服務實現類public class OrderServiceImpl implements OrderService {    // 依賴支付服務，聲明為PayService接口類型    // 如果這裡聲明為Alipay，將無法使用WeChatPay，限縮了擴展性    // 要滿足使用WeChatPay，還要修改代碼    private PayService payService;    // 構造器注入依賴的支付服務，也可以選擇使用setter方法    public OrderServiceImpl(PayService payService){        this.payService = payService;    }    @Override    public void pay() {        payService.pay();    }    // 5.模擬業務方調用    public static void main(String[] args) {         // 假設用戶選擇了微信支付服務        PayService payService = new WeChatPay();        // 通過構造器注入        OrderService orderService = new OrderServiceImpl(payService);        orderService.pay();    }}
