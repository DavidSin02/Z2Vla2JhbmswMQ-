Go 之禪：Dave告訴你編寫簡單、可讀、可維護代碼的十個工程經驗

2020-11-14
| 科技



編寫簡單、可讀、可維護 Go 代碼的十個工程經驗，大神 Dave Cheney 於 2020/02/03 在 GopherCon Israel 2020[1] 的演講。

每個 package 實現單一的目的

精心設計的 Go 軟件包提供了一個單一的想法，以及一系列相關的行為。一個好的 Go 軟件包首先選擇一個好名字。您可以將 package 的名稱想像成一個升降梯，用一個詞來描述 package 的內容。

顯式處理錯誤

健壯的程序由處理錯誤的案例片段組成。if err！= nil {return err} 可能冗長，但在發生故障時刻意處理每個故障條件的價值遠超其冗長。panic 和 recover 也不例外，它們並非打算以這種方式使用。

儘早返回，而不是使用深嵌套

每次縮進時，都會在程序員的堆棧中添加另一個先決條件，這會佔用他們短期內存中的 7±2 個插槽之一。避免需要深縮進的控制流。與其深入嵌套，不如使用保護子句將成功路徑保持在左側。

讓調用者選擇併發

讓調用者選擇是否要異步運行您的庫或函數，不要強加於他們。如果您的庫使用併發，則應透明地進行。

在啟動一個 goroutine 時，需要知道何時它會停止

Goroutines 會佔用資源：鎖，變量，內存等。釋放這些資源的可靠方法是停止相應的 goroutine。

避免 package 級別的狀態

通過提供類型需要的依賴項作為該類型上的字段，而不是使用包變量，來尋求明確的，減少耦合和詭異的問題。

簡單很重要

簡單性不是老練的代名詞。簡單並不意味著粗糙，而是可讀性和可維護性。如果可以選擇，請遵循較簡單的解決方案。

編寫測試以鎖定 package API 的行為

不論您的 package API 是否是您與用戶的合同，如果您要進行 100％ 的測試覆蓋率或希望減少測試範圍，請先進行測試，再進行後續測試。測試是寫入這些合同的保證。請確保測試了用戶可以觀察和依賴的行為。

如果覺得慢，首先編寫 benchmark 來證明

以表現為名犯下了許多危害可維護性的罪行。優化會破壞抽象，暴露內部和緊密耦合。如果您選擇承擔這筆費用，請確保有充分理由這樣做。

節制時一種美德

適度使用 goroutine，channel，鎖，接口，嵌入。

可維護性

清晰，易讀，簡單是可維護性的所有方面。離開後，您可以努力維護的東西可以保留嗎？您今天該如何做，才能使以後的人們變得更輕鬆？
