如何構建更好，更快和更強大的軟件

2020-11-14
| 科技


本文介紹了軟件開發的執行階段，其主要重點是如何使工程團隊能夠構建易於開發，測試，部署和維護的軟件。 本文主要針對工程團隊負責人，但對項目經理和產品經理也很有用，因為它們是使工程團隊能夠更好，更快，更強大地發揮作用的關鍵。

讓我們先來看一下軟件開發生命週期的各個階段：

SDLC的所有階段都具有相同的重要性，如果規劃不當，就會導致執行不力，而執行不當，則規劃就毫無用處。 SDLC可以分為兩個階段，即計劃（階段1至3）和執行階段（階段4至6）。 本文的主要目標是瞭解如何改善執行階段的每個階段：

· 實現

· 測試與集成

· 維護

實現

實現是編寫代碼以解決軟件實際需求的過程，這些需求應該已經定義，並在SDLC的第一階段中進行了詳細記錄。 由於可以以多種方式實施需求，因此取決於團隊來尋找最佳需求。

環境設定

> https://unsplash.com/photos/MFLNpz5FZRk

大多數團隊的共同之處之一是設置環境以開始編寫代碼的過程，如果您閱讀許多項目的說明，就會發現有大量的終端命令可以下載，安裝，更新和 配置事物以使計算機準備開始開發。

這樣做的最大問題是，大多數情況下，這些命令是基於特定計算機的，並且在另一臺計算機上嘗試使用這些命令時，會出現許多問題，例如與計算機上安裝的其他組件（甚至是操作系統）不兼容。 本身），缺少的內容，不同的版本等。

為什麼這是個問題？ 因為當有人加入團隊時，除了學習代碼的體系結構，所使用的工具，如何使用它們的方法等之外，他們還需要花費大量時間來嘗試建立自己的環境，並且在大多數情況下， 安裝程序的功能不夠強大，以至於在計算機中進行某些修改後，所有程序都會停止工作，因此必須重新設置所有程序。

如何解決呢？ 通過使環境設置最直接和最不可知。 實現此目的的一種方法是通過將項目封裝在虛擬環境，容器甚至虛擬機中來確保所有環境都相同。 可以用於此目的的一些工具是：

· 虛擬環境，例如Pipenv（如果您使用的是python）

· Docker或Kubernetes等容器

· 像Vagrant這樣的虛擬化

通過使用這種方法，環境設置可能需要幾分鐘而不是幾天。 當然，配置這些工具中的任何一個在開始時都將花費一些時間，但是在那之後，可能要運行兩個命令來準備開始開發，就像下面的示例一樣。

docker build --tag bulletinboard:1.0 .docker run --publish 8000:8080 --detach --name bb bulletinboard:1.0

遵循此方法的最好之處在於，它不僅對開發環境有用，而且在部署到其他環境時也將有很大幫助。

合作

> https://unsplash.com/photos/842ofHC6MaI


團隊合作的一個關鍵方面是協作，而對於協作，則需要溝通。 在談論編寫代碼時，有兩個有用的東西可以改善溝通和協作：版本控制系統和定義的工作流。

版本控制系統。

沒有版本控制，開發人員最終將覆蓋其他人可能沒有意識到的代碼更改。 即使團隊成員之間的溝通幾乎完美，這種情況最終還是會發生。

通過使用版本控制系統，團隊中的任何成員都可以在提交任何代碼之前根據需要比較文件，識別差異並解決衝突。

版本控制的另一個重要用途是在解決問題時，因為它提供了對代碼所做的所有更改的歷史記錄，因此比較上一個工作版本與有問題的版本比較容易，從而減少了查明原因的時間。

今天，最受歡迎的VCS是Git，儘管還有其他選擇，例如SVN。

工作流程。

僅通過安裝VCS並不能使所有工作都完美無缺，實際上需要定義的工作流程才能使其根據項目的需要而表現良好。 兩個常用的工作流是Git流和GitHub流。

Git流非常適合顯式版本化的項目，或者需要同時支持多個版本的項目，它基於兩個主要分支，即master以及開發和使用標籤來識別版本。

> Git-flow model

GitHub flow更適合於持續交付的項目，它僅基於master分支，並且包括六個主要階段。

> GitHub flow model

這兩個工作流程只是指導，可以按原樣使用它們，也可以對其進行調整以滿足團隊和項目的需求。

代碼質量

> https://unsplash.com/photos/pgSkeh0yl8o

更好的軟件與它所基於的代碼的質量成正比，更好的質量意味著更好的可維護性，減少了修復錯誤或實現新功能的時間。 編寫高質量代碼時需要考慮的一些方面是：

· 可維護性。 修復，更新和升級軟件應儘可能簡單，而不是固有地複雜。

· 可重用性和模塊化。 該代碼應該是安全，可靠和可靠的，以便可以重複使用。

· 可讀性。 閱讀和理解代碼部分應該很容易，這是通過清晰，簡單和文檔化的方式實現的。

· 效率。 代碼效率與算法效率和軟件的運行時執行速度直接相關。 這是確保高性能的關鍵因素。

· 錯誤傾向。 軟件行為應該是可預測的，並且不易出現隱藏錯誤。

如何提高代碼質量？

規則。 定義一組規則是提高代碼質量的最佳方法，因為它們使所有團隊的代碼風格同質，這些規則可以包括文件結構，命名約定，縮進，文檔，註釋等。

工具。 諸如linters，代碼格式化程序或git hook之類的工具對於執行以前定義的規則有很大的幫助，它們也是節省時間的絕佳工具。

代碼審查。 工具目前無法執行的操作是根據上下文強制執行規則，也許一段代碼根據定義的規則編寫得很好，但是它不能滿足項目的需求，可能是因為它具有 糟糕的性能，有更好的方法來實現，有人已經實現了相同的功能（可重用性），它的功能超出了預期，或者僅僅是開發人員在編寫代碼時沒有看到的錯誤。

代碼審查是維護代碼質量，將其視為質量保證過程的好方法，其他團隊成員可以分析代碼並檢查代碼可能存在的任何缺陷，以便在將其包含在代碼庫中之前對其進行修復。

代碼審查已經存在了很長時間，並且是一個非常廣泛的主題。看看世界上最大的軟件公司之一如何處理此過程。

測試與集成

繼續執行代碼主題，在將某些內容包含到代碼庫中之前，必須確保它可以按預期工作，為此需要對其進行測試。 測試通常由質量檢查工程師完成，但開發人員自行進行測試並不少見。 有兩種類型的軟件測試，功能測試和非功能測試。

功能測試

顧名思義，這種類型的測試僅針對軟件的功能部分，這意味著它只有在所有可能的情況下才能通過這種類型的測試。 功能測試的一些示例是：

· 單元測試。 這種類型的測試是為了確保代碼（單元）能夠按預期隔離地工作，通常是由開發人員編寫一個額外的代碼片段來測試該單元來完成的，有兩種方法可以做到這一點，首先編寫單元，然後 測試，或者反之，將需求轉換為測試用例，然後開發人員編寫代碼以解決這些測試用例，最後一個過程稱為"測試驅動開發"。

> A basic example of Unit testing

· 集成測試。 確保將代碼隔離工作後，將其與其餘代碼集成並進行測試以確保其正常工作，儘管已與所有代碼集成，但僅測試了新代碼。

· GUI測試。 此測試是為了確保圖形界面（如果有）符合設計團隊的規格，顏色，大小，間距，版式等。

· 迴歸測試。 在驗證了引入的代碼工作是否與其他代碼隔離並集成之後，有必要驗證整個代碼庫是否像引入新代碼之前一樣繼續工作，這種測試是詳盡的，並且通常由自動化工具完成。

非功能測試

非功能測試是在功能測試通過後進行的，僅代碼是行不通的，它需要以安全，快速和可靠的方式工作。 這些測試的一些示例是：

· 性能測試。 新代碼應儘可能快，並且僅使用必要的資源來交付預期的結果，例如讀/寫操作（數據庫，內存或磁盤），進程之間傳遞的數據量，算法的複雜性（ s）使用等

· 安全測試。 根據軟件的性質，此測試可能會有所不同，在談論Web應用程序時，根據OWASP，此測試可確保代碼不易受注入，身份驗證損壞，敏感數據公開，XML外部實體（XXE），訪問中斷的影響 控制，安全性錯誤配置，跨站點腳本XSS，不安全的反序列化，使用具有已知漏洞以及日誌和監控不足的組件。

集成

集成是將新創建的代碼包含到代碼庫中，因此可以對其進行構建和部署。 為了改進此過程，應建立一種一致，自動化的方式來構建，打包和測試軟件。

有了集成過程中的一致性，團隊就更有可能更頻繁地提交代碼更改，從而帶來更好的協作和軟件質量。

集成策略應該是前面定義的工作流的一部分，因為集成是與VCS並行進行的，為實現代碼的自動化集成而創建的一些工具是：

· Jenkins

· CircleCI

· Travis CI

部署方式

使軟件對用戶可用的最後一步是部署，該過程還取決於應用程序的性質，對於Web應用程序來說，與移動應用程序或混合應用程序不同。

對於Web應用程序，此過程的自動化也是集成過程中提到的工具的一部分，通常同時執行這兩個過程非常普遍。

對於移動應用程序，此過程可能有所不同，因為它取決於外部實體來批准構建，例如iOS的App Store或Android的Google Play商店。 進行連續部署會更加困難，因為要在商店中看到該應用會花費更長的時間。 對於某些Web應用程序（如Wordpress VIP）也是如此，它們遵循與移動應用程序相同的方法。

還記得本文開頭提到的使環境設置簡單易行的重要性嗎？ 現在，它具有更大的相關性，因為自動化部署過程的工作將取決於完成安裝的方式。

維護

如果軟件是基於高質量的代碼構建的，則其工作團隊之間將保持良好的溝通和協作，建立了良好的工作流程，測試套件可靠，並且集成和部署過程實現了良好的自動化，因此不應對軟件進行維護 代表任何問題。 添加，升級或修復功能將比以往更加輕鬆。

維護階段有助於確定先前的所有階段是否運行良好，或者是否需要進行某些更改。
