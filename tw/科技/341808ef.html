借鑑租約機制的DB和緩存雙淘汰一致性方案

2020-11-14
| 科技

借鑑租約機制的DB和緩存一致性雙淘汰方案：

1.寫請求時，基於雙淘汰方案，先淘汰緩存，再寫數據庫，最後再次淘汰緩存

2.緩存第一次淘汰時為了加快其老化，同時又不造成緩存雪崩。如果沒有該步驟，假設租約為N小時，如果第二次緩存淘汰失敗的話，那就可能會存在N小時的數據不一致性；如果第一次淘汰失敗的話，則可直接返回了

3.緩存第二次淘汰，參考租約機制，加上隨機超時時間，同時也可以防止緩存雪崩

具體偽代碼如下：

1)寫請求：

public void write(string key, object value){

if getDistributedLock(){ //獲取分佈式鎖

writeToCache(key, value, shortTime) // 緩存第一次淘汰，加快其老化，如果key存在，則讓緩存失效

writeToDB(key, value) // 更新到數據庫中

writeToCache(key, value,getRandomExpiredTime()) //緩存第二次淘汰，後更新緩存

}

}

備註：如果第二次緩存淘汰失敗，那麼會存在shortTime時間範圍內的不一致性

2)讀請求：

public object read(string key){

object value=getFromCache(key)

if value==null{

int expiredTime=getRandomExpiredTime() // 加上隨機時間，防止緩存雪崩

value=getFromDB(key)

if value==null{

expiredTime=shortTime // 把null也緩存起來，設置短時間的存活，防止緩存穿透

value="null"

}

writeToCache(key, value, expiredTime)

}

return value

}
