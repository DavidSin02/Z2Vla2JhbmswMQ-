來談談JAVA面向對象 - 繼續說多態~

2020-10-29
| 科技

為什麼要多態？

一般來說，伴隨著接口和類的繼承，多態就會產生。在多態的過程中，我們需要關注其中的一些要點。

首先我們來演示一下類的繼承。

父類和子類：

public class Zi extends Fu{   public void play(){ System.out.println("釣魚"); }}public class Zi extends Fu{   public void play(){ System.out.println("LOL"); }}

接下來，我們看類的繼承中，多態是如何體現的。

假如有一天，兒子穿著父親的衣服到他老子的單位裡去玩耍。如果這個時候，他老子的同事請他去釣魚，這個是不行的，因為他到底的愛好是LOL，如果強行調用了play方法，就會露餡。

現在，假設他老子有一個吃飯的方法，喜歡吃水煮魚，這一點，他兒子也是如此，所以，就不需要重寫這個方法。

public class Fu {  public void eat(){ System.out.println("我正在吃飯，吃水煮魚！"); }  public void play(){ System.out.println("釣魚"); }}

那麼，如果他兒子跟他老子的同事一起去吃飯，並且吃了水煮魚，就不會露餡。

假如下班後，他兒子去網咖玩“吃雞”，但是衣服還沒有來得及換。現在，兒子有吃雞的方法：

public class Zi extends Fu{   public void play(){ System.out.println("LOL"); }  public void eatChicken(){ System.out.println("大吉大利，今晚吃雞！"); }}

當他去了網吧，強行調用自己的吃雞方法。

竟然報錯了！！！

因為現在他對外的身份還是父親，而父親是沒有吃雞的方法的，所以無法調用了。

於是啊，他一氣之下，在網咖裡脫衣服，迴歸自己的原來身份！

public class Test04 {

public static void main(String[] args) {

Fu f = new Zi();

((Zi) f).eatChicken(); //強制轉換

}

}

總結：

多態就是，子類繼承父類後，用父類型 new出自己的實例對象。就好比剛才例子中，兒子穿著他老子的衣服去他老子的公司做一些不可描述的事情。那些他老子沒有，但是他沒有的方法，不能使用，除非強制轉換為子類的身份。

接著我們來演示一下類實現接口。

/** * 游泳的能力 接口 * @author Administrator * */public interface Swimming { public void goSwimming();}/** * 飛行能力的接口 * @author Administrator * */public interface Fly {  public void goFlying();}

讓子類去實現游泳接口，表示其已經擁有了游泳的能力，和類的繼承不同，如果一個類實現了某一個接口，就必須重寫接口內所有的方法。

public class Zi extends Fu implements Swimming{   public void play(){ System.out.println("LOL"); }  public void eatChicken(){ System.out.println("大吉大利，今晚吃雞！"); } @Override public void goSwimming() { System.out.println("我已使出洪荒之力來游泳！"); }}

編寫一個飛鳥類，實現這個飛行的接口：

public class Bird implements Fly{ @Override public void goFlying() { System.out.println("I can Fly , can you ?"); }}

那麼，多態的體現就是，我們可以用接口類型去new出所有實現該接口的子類對象。

我們發現，new出來的子類對象除了Object類中方法之外，只能調用接口裡的方法，不能調用自己的方法。就連他繼承自父類的方法也不能調用！！

看到這裡，大家可能會困惑，說這個多態有什麼好，多態了之後，自己的方法好多都訪問不了了！

在實際的項目開發中，多態更多的用處就是方便傳參，就像上一篇文章中的例子一樣，大喬在設計的時候，大招方法根本不知道你要傳進來什麼英雄，所以權宜之下，就設置參數為所有英雄的父類。

回到最初的問題，為什麼要多態？

所以我個人認為多態更多的就是一種開發上的權宜之計，因為我們在編寫一個統一的接口時，考慮不到那麼多，只能大致的預留一個參數，這個參數當時可能是一個單獨的類，也可能已經是一些子類的父類。可是在當時的情況下，我還並不知道這個項目後期會出現哪些重大的變化，會增加哪些新的功能？也許，隨著項目的壯大，會出現越來越多的子類，我設計的方法也可能在某一天無法滿足項目的需求。

如果那時我已經離職不在之前的公司，後續的程序員看到我的代碼也不會去修改，最多就是繼承一下我的類，然後重寫或者重載我的方法。

因為誰也不敢保證自己改動原先的代碼會不會引起項目的不穩定，所以一般都是繼承後重新寫。

後期有機會的話我會考慮專門寫一個例子來說明這個事情。
