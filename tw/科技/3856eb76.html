窗口應用程序的事件驅動、消息響應、映射、自定義

2020-10-29
| 科技

控制檯程序的流程是由代碼編寫的順序來控制，是過程驅動，而窗口程序卻是事件驅動。

在Win32的窗口程序中，採用switch_case結構進行消息處理，在MFC中通過消息映射的方法，將消息映射到各個處理函數。

窗口是位於屏幕中的一個矩形區域，它用於接收用戶的輸入，然後以文本或圖形的形式顯示輸出。 窗體（控件）類就要是對輸入輸出界面的封裝。

1 Windows的編程模型與MS-DOS編程模型之比較

第一，用C語言編寫基於MS-DOS的應用程序時，唯一絕對需要的是一個名為main的函數。而當Windows操作系統啟動一個程序時，調用的是WinMain函數。

第二，許多MS-DOS程序直接寫顯存和打印機接口。這種技術的不利之處是對每一種設備需要其支持的驅動程序軟件。Windows引入了一個名為圖形設備接口（GDI）的抽象化外層，所以用戶不必知道有關係統設備的類型。Windows程序不是尋址硬件，而是調用GDI函數。

第三，要在MS-DOS環境下進行數據驅動編程，必須或者為把數據編碼成為初始化常量或者提供獨立的數據文件讓程序來讀。進行Windows編程時，使用大量已經確立的格式在資源文件中存儲數據。

第四，在MS-DOS環境下一個程序的所有對象模塊在建立過程中是靜態連接的。Windows允許動態鏈接，這意味著特別創建的庫可以在運行時加載和鏈接。多個應用程序可以共享動態鏈接庫（DLLs），它節省內存和磁盤空間。動態鏈接增加了程序的模塊性。

2 事件驅動、消息響應

所謂消息就是用於描述某個事件發生的信息，而事件是對於Windows的某種操作，消息相當於事件的數據化，一般通過一個結構體來描述。

事件和消息密切相關，事件是因，消息是果，事件產生消息，消息對應事件。所謂消息的響應，其實質就是事件的響應。

消息驅動是Windows應用程序的核心，所有的外部響應（如鍵盤、鼠標和計時器等）都被Windows先攔截，轉換成消息後再發送到應用程序中的目標對象，應用程序根據消息的具體內容進行處理。

消息是操作系統將事件傳遞給用戶程序的數據格式，由消息類型和相關數據組成，如當鼠標單擊了一個按鈕時，應用程序就會接到一個消息，消息的類型是WM_COMMAND，而數據是一個按鈕的ID，應用程序根據ID可以判斷用戶單擊的是哪個按鈕。

消息不僅可由Windows發出，它也可由應用程序本身或其它程序產生。Windows為每一個應用程序都維護一個或多個消息隊列，發送到每個程序窗口的消息都排成一個隊列。


Windows本身維護一個系統消息隊列，對於每一個正在運行的Windows應用程序，系統都會為其建立一個消息隊列，用於存放該應用程序可能創建的各種消息。消息隊列是一個先進先出的緩衝區，通常是一個某種變量類型的數組。消息隊列裡的每一個元素都是一條消息。操作系統將生成的每個消息按先後順序放進隊列裡。

一般來說，應用程序都包含一段消息循環代碼，用於從消息隊列中檢索這些消息。應用程序總是先取走第一條消息。消息取走後，第二條消息成為第一條，剩餘的消息依次往前推。應用程序取得消息後，便能夠知道用戶的操作和程序狀態的變化，並把它們分發到相應的函數中進行處理。

消息響應是用戶通過編碼實現的，這也是Windows程序的主要代碼。

在消息響應代碼中，很可能又要調用操作系統提供的API函數，以便完成特定的功能。如果用戶收到窗口的WM_CLOSE消息，可以調用API函數DestroyWindow來關閉該窗口，或是用MessageBox函數來提示用戶是否真的要關閉窗口。

Windows程序設計中，在應用程序中要完成某個功能，都是以函數調用的形式實現的。同樣，應用程序也是以函數調用的方式來通知操作系統執行相應的功能。Windows操作系統所能夠完成的每一個特殊功能通常都有一個函數與其對應。也就是說，操作系統把它所能夠完成的功能以函數的形式提供給應用程序使用。應用程序對這些函數的調用就叫做系統調用。這些函數的集合就是Windows操作系統提供給應用程序編程的接口（Application Programming Interface），簡稱Windows API。

Windows編程的主要代碼都集中在對消息的處理上，或者說編寫消息響應（處理）函數是進行Windows編程的主要工作。

3 消息類別

當用戶需要完成某種功能時會調用操作系統的某種支持，然後操作系統將用戶的需要包裝成消息，並投遞到消息隊列中，最後應用程序從消息隊列中取得消息並進行響應。

在應用程序中，用戶所有的操作都是通過消息機制（Message）來傳遞給操作系統的。操作系統將每個事件都包裝成一個稱為消息的結構體MSG來傳遞給應用程序。例如用戶在某個程序活動時按了一下鍵盤，操作系統馬上能感知到這一事件，並且能夠知道用戶按下的是哪一個鍵。操作系統並不決定對這一事件如何作出反應，而是將這一事件轉交給應用程序。由應用程序決定如何對這一事件作出反應，對事件作出反應的過程就是消息響應。

消息用一個結構體來描述，區別消息一般是對結構體中的主消息message、附加參數wParam和lParam3個字段進行判斷。

 typedef struct tagMSG {  HWND hWnd; // 消息來源的窗口句柄 UINT message; // 消息ID WPARAM wParam; // 消息附加參數1（附加信息，16位） LPARAM lParam; // 消息附加參數2（附加信息，32位） DWORD time; // 消息發送時間 POINT pt; // 消息發送時鼠標的屏幕座標} MSG;

在MFC中，消息分為窗口消息，命令消息和控件消息三種類型。

許多情況下，Windows編程也就是編寫消息處理函數。Windows程序設計中最常用的一些消息有：

1) 鍵盤消息

2) 鼠標消息

3) 窗口消息

4) 焦點消息

5) 定時器消息

6) 命令

4 基於資源的程序設計

Windows應用程序常常包含眾多圖形元素，例如光標、菜單、工具欄、位圖、對話框等，在Windows環境下，每一個這樣的元素都作為一種可以裝入應用程序的資源來存放。這些資源就像C++程序中的常量一樣，可以被編輯、修改，也可以被其他應用程序所共享。VisualC++中就提供這樣的編輯器，可“所見即所得”地對這些不同類型的資源進行設計、編輯等。

Windows程序中產生的任何資源（要佔用某一塊或大或小的內存），如圖標、光標、窗口和應用程序的實例（已加載到內存運行中的程序）等，都要將它們放入相應的內存，併為這些內存指定一個唯一的標識號。這個標識號即是該資源的句柄。

操作系統要管理和操作這些資源，都是通過句柄來找到對應的資源的。一般來說，按資源的類型，可以將句柄細分成圖標句柄（HICON）、光標句柄（HCURSOR）、窗口句柄（HWND）、應用程序實例句柄（HINSTANCE）等各種類型的句柄。例如，操作系統給每一個窗口指定的一個唯一的標識號即窗口句柄。

在Visual C++中所有諸如對話框、菜單等都稱為資源。資源由資源編輯器進行管理。資源編輯器存在於Workspace工作區中。它提供了一個所見即所得的菜單編輯器和一個強大的對話框圖形編輯器。另外，它還包含了編輯圖標（Icon）、位圖（BMP）和字符串（String）等資源的工具。

每一個工程通常有一個文本格式的資源腳本（RC）文件來描述工程的菜單、對話框、字符串等資源。RC文件也可以通過#include語句（包含庫文件語句）從其他子目錄中引進資源。這些資源包括位圖、圖標以及所有Visual C++程序共用的資源。用戶可以通過直接編輯文本形式的RC文件來編輯各種資源。但是一般來說，推薦使用通過資源編輯器來編輯資源。

Visual C++的資源編譯器從資源編輯器中讀取ASCII資源腳本文件，並且向鏈接程序提供一個二進制RES文件。

鏈接器從C/C++編譯器和資源編譯器產生的目標文件OBJ 文件和資源文件RES 文件中讀取信息，連同Windows 運行庫和MFC 庫，訪問LIB 文件，最後生成工程的EXE 文件。鏈接時間的長短取決於對源文件改動的大小。

5 在類中實現事件

在許多語言中都有對事件的定義，事件的作用是可以在類外實現一個事件，然後在類中調用這個事件。這樣在設計類時就可以不必實現某些功能，而這些功能可以交給外部函數來處理，這樣更增加了程序的靈活性。

在C++語言中實現事件可以使用函數回調的方法，而函數回調就是使用函數指針來實現的。注意這裡針對的是普通的函數，不包括完全依賴於不同語法和語義規則的類成員函數。聲明函數指針回調函數是一個程序員不能顯式調用的函數，通過將回調函數的地址傳給調用者從而實現調用。要實現回調，必須首先定義函數指針。儘管定義的語法有點不可思議，但如果你熟悉函數聲明的一般方法，便會發現函數指針的聲明與函數聲明非常類似。

6 MFC編程與消息映射

無論是API還是MFC編程，程序員最關心的內容有三個：

1) 程序入口；

2) 窗口、資源等的創建和使用；

3) 鍵盤、鼠標等所產生的事件或消息的接收和處理；

從MFC的角度看一個對話框是由對話框資源和對話框類共同生成的。

1) 創建對話框模板資源，並向對話框模板資源添加控件；

2) 生成對話框類，並添加與控件關聯的成員變量和消息處理函數(類的成員函數包括普通成員函數和消息響應函數)；

3) 在程序中顯示對話框並訪問與控件關聯的成員變量。

Win32工程的消息處理機制是每個窗口都關聯一個全局的消息回調函數，在消息回調函數中，通過對消息類型的判別來響應不同的消息。MFC的消息處理不是使用一個全局的回調函數，而是創建一個與窗口關聯的派生類，用於消息關聯的成員函數來接收和處理窗口的消息。

MFC採用消息映射（Message Map）機制取代C/C++語言中的switch-case結構來處理消息。

MFC消息映射機制包括一組消息映射宏。一條消息映射宏把一個Windows消息和其消息處理函數聯結起來。

當窗口接收到消息時，會到消息映射表中查找該消息對應的消息處理函數，然後由消息處理函數進行相應的處理。SDK編程時需要在窗口過程中一一判斷消息值進行相應的處理，相比之下MFC的消息映射機制要方便好用的多。

在Win32的消息處理機制中，每一個消息類型發送到窗口時，可能在wParam和lParam中攜帶一些相關的數據。例如，在WM_COMMAND的消息中，在wParam中記錄了單擊的按鈕或者菜單項等ID號碼。而在WM_LBUTTONDOWN消息中，在lParam中記錄了單擊界面的(x,y)座標等。Win32解析這些附帶的數據比較繁瑣，而MFC的消息映射機制，一般不需要開發者解析這些複雜的數據，因為消息映射函數會自動在參數中將解析好的數據傳遞過來。如OnMouseMove(UINT nFlags, CPoint point)函數。

除了一些沒有基類的類或CObject的直接派生類外，其他的類都可以自動生成消息映射表。

在類（如Records）聲明中用DECLARE_MESSAGE_MAP()宏來聲明使用消息映射:：

//{{AFX_MSG(Records)	afx_msg void OnButton18();	……	//}}AFX_MSG	DECLARE_MESSAGE_MAP()

用BEGIN_MESSAGE_MAP()和END_MESSAGE_MAP()宏來定義消息映射：

BEGIN_MESSAGE_MAP(Records, CDialog)	//{{AFX_MSG_MAP(Records)	ON_BN_CLICKED(IDC_BUTTON18, OnButton18)	……	//}}AFX_MSG_MAPEND_MESSAGE_MAP()

然後實現OnButton18()函數：

void Records::OnButton18(){	……}

7 添加自定義消息

MFC允許用戶自己定義消息，自定義消息必須在WM_USER的基礎上定義，且自定消息的消息映射使用ON_MESSAGE。

7.1. 定義自定義消息

#define WM_MYMESSAGE (WM_USER + 1)

7.2. 聲明自定義消息處理函數

void CMymsgDlg::OnMyFunction();//可以有參數WPARAM wParam, LPARAM lParam

7.3. 實現自定義消息處理函數

void CMymsgDlg::OnMyFunction(){	ShellExecute(m_hWnd,"open", "https://www.baidu.com/", NULL, NULL, SW_SHOW);}

這是一個普通成員函數，非消息響應函數，什麼時候有機會執行呢？被消息響應函數調用，或者在消息響應函數中發送自定義消息，通過消息映射得以執行。

7.4. 用宏實現消息映射

BEGIN_MESSAGE_MAP(CMymsg2Dlg, CDialog) ……	ON_MESSAGE(WM_MYMESSAGE, OnMyMessage)END_MESSAGE_MAP()

7.5. 發送自定義消息到消息隊列

可以在一切普通成員函數、消息響應函數，重寫的事件響應函數中發送自定義消息，如OnInitDialog()、OnPaint()等。

void CMymsgDlg::OnButton1() {	PostMessage(WM_MYMESSAGE); //將自定義消息插入消息隊列並返回 SendMessage(WM_MYMESSAGE); //將自定義消息插入消息隊列並返回}

或者：

//可以重寫OnKeyUp()等鍵盤、鼠標事件中發出消息void CMymsgDlg::OnKeyUp(UINT nChar, UINT nRepCnt, UINT nFlags) { HWND hWnd = GetSafeHwnd(); //接收消息的窗口句柄 if(nChar == VK_UP) //光標上移鍵 { ::PostMessage(hWnd, WM_MYMESSAGE, 0, 0); //將自定義消息插入消息隊列並返回 return ; }		CDialog::OnKeyUp(nChar, nRepCnt, nFlags);}
