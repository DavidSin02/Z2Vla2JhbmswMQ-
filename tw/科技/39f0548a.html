端智能系列文章｜端側複雜事件實時處理框架

2020-11-14
| 科技

背景

現在移動網絡越來越發達，移動生活越來越豐富，在用戶手機上可能同時存在數百種APP，這注定了用戶使用某一款APP的時間也將逐漸縮短。如果用戶在APP內僅瀏覽了幾分鐘甚至幾十秒，那我們將很難為用戶提供更有價值的服務與信息，大部分應用的做法是將最最熱銷的產品或最最火爆的活動放在應用的閃屏或是首焦上，對於閒魚這樣具有豐富業務生態的應用來說，顯然還不夠。

那如何在用戶駐留的短暫時間內為用戶提供更有價值的信息呢，閒魚基於用戶的訪問路徑，為用戶提供了更加豐富的優質信息與服務。例如用戶發佈了商品，我們會推薦當前正在進行的包郵活動，用戶沒買到想要的商品，我們會及時給用戶推薦同款。

為了實現這樣的信息提供模式，閒魚打造了一套流量調控系統，本篇將介紹該系統端側部分的實時處理框架。

實時處理框架是什麼

閒魚實時流量調控系統的端側版本是一個合作項目，我們稱之為BehaviR，BehaviR框架在縱向流程上，涵蓋了用戶行為的實時處理，高質量的數據供給，強實時的數據傳輸，實時計算，決策與用戶觸達，在廣度上能支撐用戶在應用內的全場景，同時對典型場景會有定製的支持。而本篇介紹的實時處理框架即是其中的一部分，負責實時計算的閉環，讓我們看下BehaviR全貌：

為了完成實時計算的閉環，實時處理框架需要具備的能力：

在實時處理框架中，負責規則計算的部分即是端側計算引擎，同雲側計算引擎一樣，端側計算引擎的目的是也將獲取到的無終結狀態的數據進行有狀態的實時流式計算。但由於計算髮生在端側，在一個輸入源相對有限、運算資源相對有限、有版本限制的運算環境下，要完成上述目的，它需要解決以下幾個問題：

  • 如何在端側保證無終結狀態的數據持續供給

  • 如何在端側進行有狀態的實時計算

  • 如何將實時計算結果進行有效輸出

數據

在端側，數據的供給流程如圖：

數據的供給能力由BehaviR的數據模塊完成，業務方在APP內集成BehaviR，然後將頁面操作數據、網絡請求數據、存儲操作數據等有序灌給BehaviR，BehaviR會根據計算配置將待計算數據按需進行持久化和供給。例如端上指定如下配置：

  1. {
  2. "actionTypeIn":[
  3. "leave"
  4. ],
  5. "sceneIn":[
  6. "https://market.m.taobao.com/app/idleFish-F2e/idlefish-renting/home",
  7. "https://market.wapa.taobao.com/app/idleFish-F2e/idlefish-renting/home"
  8. ],
  9. "taskArray":[
  10. {
  11. "taskType":"py_backtrace",
  12. "pythonName":"test_cep_rent_rule2",
  13. "filter":[
  14. {
  15. "alias":"e1",
  16. "actionType":"leave"
  17. },
  18. {
  19. "alias":"e2",
  20. "actionType":"pv"
  21. }
  22. ]
  23. }
  24. ]
  25. }

我們會在"actionType"內配置觸發器，即BehaviR將會在"actionType"為"leave"的事件發生時，將擁有的數據提供給計算引擎。在計算髮生前，計算所需要的數據類型是可預期的，因此BehaviR會根據"filter"內的配置，將數據過濾後傳輸給計算引擎。

在APP中運行計算任務前，每一個計算任務都將擁有一份這樣屬於自己的配置，來指定需要的計算數據類型。在BehaviR中根據端側所有計算任務的數據需求，可以擇優進行持久化來儘可能少的佔用磁盤空間和內存空間。

計算

雲側實時計算能力已經成為了業界先驅，Flink、Siddhi、Spark等如數家珍，而端側並沒有計算框架先例，那我們在端側需要做的，就是深入瞭解雲側引擎，然後針對端側環境進行輕量化和定製化。結合我們已承接業務的經驗，將計算框架從功能層面劃分如下，並與雲側進行比較：

實時計算的窗口，可以對數據進行一定的預處理，例如按時間間隔每N秒處理一次，按數量每N個事件處理一次等。而在端上由於每個行為都有典型的特徵，而大多數場景都對這些特徵有著明確的預期，因此端上以Trigger的方式進行事件的處理，在Trigger上會描述我們需要開始計算的事件具有的屬性，當事件屬性與Trigger的屬性匹配時，即開始計算。

（不確定有限）狀態機和共享緩衝區是計算處理的核心，其中共享緩衝區是對狀態計算中狀態與事件對應關係的存儲優化，我們在初版上也實現了其必要的組成部分。

在併發計劃和事件時序控制上，端上暫時通過單端事件的有序錄入，來保障絕大多數事件的時序，而併發的優化並不是初版的瓶頸所在，於是我們直接放棄了併發控制，在單線程上進行計算。

簡而言之，我們去掉了多流處理的設計，在容錯機制上弱化處理，但保留了核心的計算能力，同時將實時流處理上的概念進行融合，更加貼近我們的應用場景。關於計算部分的具體實現，後續將提供更詳細的介紹。

實現方案選型

當框架的概念與功能確定後，我們需要考慮使用什麼樣的方式去實現。於是將預備的實現方案從動態性、執行效率、開發效率等方面進行了比較：

由於在整個端側處理框架研發上，我們是首次嘗試落地，期望能快速驗證可行性，因此項目上線及運行過程會具備很高的不確定性及穩定性風險。綜上幾個因素，恰逢集團能擁有高效、穩定的Python運行時框架Walle，因而我們選擇在Python運行時框架下Walle，使用Python進行研發。

描述及編譯

在開始運算前，我們會有一套基於Python的應用編程接口來描述計算邏輯，當使用該接口編寫完成後，會在運行時編譯並構建計算圖實例，然後進行計算。該編程接口的表達以“訪問詳情然後離開”為例，我們可以描述為：

  1. Pattern('e1') \
  2. .where(KVCondition('actionType', 'pv')) \
  3. .and(KVCondition('scene', 'item_detail')) \
  4. .followby('e2') \
  5. .where(KVCondition('actionType', 'leave')) \
  6. .and(KVCondition('scene', 'item_detail'))

為了雲與端的體驗一致性，我們也將支持從調控系統的標準DSL（該DSL已成為Blink支持的標準）到Python描述的轉換，上述代碼用DSL來描述，即：

  1. EVENT: e1->e2
  2. WHERE e1.extra_info.actionType = 'pv' 
  3. AND e1.extra_info.scene = 'item_detail' 
  4. AND e2.extra_info.actionType = 'leave'
  5. AND e2.extra_info.scene = 'item_detail'

輸出

當計算完成後，我們會將計算結果進行輸出，在端側的輸出主要分為三個方面，一是業務權益觸達，二是算法模型輸入，三是另一次計算輸入。

由於在業務權益觸達形式上，不同的應用有著自己的展現形式和實現方式，因此我們僅保留公用的通訊協議來將計算結果進行格式化輸出。在閒魚端內，我們會介入協議的響應方，並接入閒魚的決策分發模塊進行統一管控。決策分發模塊將對單次策略的觸達效果進行管控，同時對同一用戶的所有策略進行調控。在閒魚應用內已經注入了豐富的觸達形式供決策模塊選擇。

端側的算法模型運行時，經常需要特定的行為作為模型運行的觸發器，那麼當計算結果作為算法模型輸入時，由於在集團內端側算法模型有統一的運行平臺，因而我們需要將計算輸出與統一的模型運行框架對接，支持特定的計算結果自動喚醒對應模型的運行，然後將計算結果經過有選擇性的篩選後輸出給算法模型。

如果需要通過多個計算策略串聯計算後產出結果，那我們會提前約定好計算串聯前的數據規範，當另一次計算處理需要以當前計算結果作為輸入時，我們會在本次計算結果模擬為一次約定的特殊類型的行為數據並流入數據模塊，然後在另一次計算配置上添加該類型數據的監聽，即可順利執行。

流程回顧

當我們瞭解了端側複雜事件實時處理框架的各個部分後，可以用下圖再回顧下整個運行過程：

我們會在應用啟動的時候去同步本次啟動需要進行的計算任務配置，同時數據模塊會開始實時處理數據，當處理的數據與任務配置相匹配時，即開始給計算框架供給數據，然後進行實時計算。計算完成後，會流轉決策分發模塊，由決策模塊決定計算結果的輸出方向，如果決策為觸達用戶，則會採用相關的觸達配置與形式進行觸達。

效果及展望

我們目前在閒魚APP線上穩定運行了該實時處理框架的首個版本，雲側需要處理5秒左右，而現在全部流程可在毫秒級內完成，對服務器資源零消耗。

在計算細節上，我們仍然有很多需要打磨的地方，例如端上進程退出帶來的計算終斷問題， 少量事件亂序問題，需要能承接更多複雜場景的聚合計算等。由此，我們將為了更高效的計算和更穩定的服務而努力。更加詳情的計算細節介紹，盡請期待！
