利用Sboxr自動發現和利用DOM型XSS-Part 1

2020-10-29
| 科技

大家好，我將發佈一系列關於javascript的安全文章，向你展示如何使用Sboxr來自動化識別DOM型XSS。為了更好的展示，我在https://domgo.at上搭建了DOM型XSS漏洞的10個演示，並且放出了簡單的PoC。

這篇文章主要包含前兩個演示的具體設置和利用方法。剩餘的演示我將在接下來的文章中一一說明。此外，我們還有一個gitbook，包含海量的安全演示說明，網址為https://appsecco.com/books/。

什麼是DOM XSS / Client XSS？

縱觀XSS漏洞的歷史，DOM型XSS在安全人員的心中都佔有特殊的地位。因為一般的XSS檢測技術很難檢測到這種類型的XSS漏洞，而且這種類型的漏洞也很容易出現使用了眾多JS腳本的網站。

DOM型XSS並不會影響服務器本身返回的HTTP響應。只是單純的改變客戶端瀏覽器的行為。而且，DOM型XSS的攻擊角度千變萬化，沒有具體的定式，完全取決於JS的複雜程度和所實現的功能。

手動或通過代碼審查來檢測DOM型XSS可能會花費大量時間。因此，最好能有一種自動化監控，測試DOM型XSS的工具。

Sboxr

Sboxr是一種用於調試Web應用的工具，特別是使用了大量javascript代碼的Web應用。Sboxr主要通過往瀏覽器和服務器之間注入自己的JS代碼（被稱為DOM傳感器）來監控JS的使用，源，接收器，變量分配，函數調用等。然後，它通過Web控制檯向用戶展示所監控數據的變化視圖。

設置Sboxr和Chrome

我使用了 Ubuntu 18.04 來安裝我們的工具以及Chrome 72。具體步驟如下：

  1. 獲取Sboxr——https://sboxr.com/
  2. Sboxr還需要.NET Core SDK的支持。如果是Linux系統，可按照https://dotnet.microsoft.com/download/linux-package-manager/ubuntu18-04/sdk-current進行安裝；而對於Windows系統，可按照https://dotnet.microsoft.com/download上進行操作。
  3. 一旦安裝完成，運行啟動命令dotnet Sboxr.dll
  4. 然後你將在本地3333端口看到Sboxr的Web界面（用於管理和分析），3331端口是代理端口。
  5. 如果你希望使用Burp或其他代理，請到http://localhost:3333/console頁面單擊HTTP Sensor去設置代理的IP地址和端口（例如Burp或OWASP ZAP）。

設置完成後，我們還需配置瀏覽器向Sboxr發送流量（然後再轉發到Burp或OWASP ZAP）。

Sboxr尚不支持SOCKS代理，因此你需要使用Burp或OWASP ZAP在某些特定的時刻攔截流量。

如果碰到HTTPS站點，Sboxr可能無法正常工作，因此我需要設置Chrome忽略證書錯誤--ignore-certificate-errors。

在Linux上，命令如下：

mkdir -p ~/.chrome;/opt/google/chrome/chrome -incognito --ignore-certificate-errors --proxy-server=http=http://localhost:3331\;https=http://localhost:3331 --user-data-dir=~/.chrome

在Windows上，操作如下（假設是默認安裝路徑）：

"C:\Program Files (x86)\Google\Chrome\Application\chrome.exe" -incognito --ignore-certificate-errors --proxy-server=http=http://localhost:3331;https=http://localhost:3331 --user-data-dir="C:\Users\%Username%\AppData\Local\Temp\TestChromeProxy"

Start

現在，我們要開始挖掘DOM型XSS了，首先從https://domgo.at開始。

以下我將展示我搭建的DOM型XSS演示的前兩個，包括髮現流程和利用方法，方便你創建自己的PoC。

演示1

1.進入https://domgo.at/，然後單擊左側的演示1，你可以看到很詳細的漏洞原理以及簡單的利用方式。

2.切換到Sboxr控制檯，點擊Sboxr側欄的Code Execution。

3.從HTML上下文中可以看出，存在某種輸入數據源，涉及location.hash，並且輸入的數據最終被接收器HTMLElement.innerHTML所利用。

4.單擊View Code Execution，你會看到前端代碼的詳細執行流程。

5.通過單擊View event location details圖標，我們可以清楚地看到我們輸入的數據在何處被接收器HTMLElement.innerHTML所利用。

6.為了展示location.hash到底有沒有缺陷，我輸入一個JS代碼看最終是否執行。

7.我插入了<svg onload=alert(document.domain)>，最終成功彈框。PoC為:https://domgo.at/cxss/example/1?payload=abcd&sp=x#<svg%20onload=alert(document.domain)>

演示2

1.單擊側欄上的演示1以加載練習1

2.單擊側欄上的演示2以加載演示2。記住，不能直接通過URL訪問，因為這涉及document.referrer屬性。

3.我們按照與上一個演示相同的步驟開始，然後點擊Sboxr側欄中的Code Execution


4.從演示代碼中可以看到，如果referrer中的URL中有一個payload參數，則直接將其提取並傳遞給接收器。

5.我可以使用簡單的HTML頁面構建我們的exploit。創建一個exercise2.html文件，其功能是導航到https://domgo.at/cxss/example/2。文件具體代碼如下：

<html> <body> <h2>PoC for Exercise 2 of https://domgo.at</h2> <script> window.location="https://domgo.at/cxss/example/2" </script> </body></html>

利用方法是點擊：http://127.0.0.1/exercise2.html?payload=<svg%20onload=alert(document.domain)>

6.在上一步點擊後，頁面將立即重定向到演示2，此時彈框出現。
