Redis的5種數據類型與編碼結構分析

2020-11-14
| 科技

一、概述

  • Redis作為一個分佈式緩存實現，相對於Memecache，除了支持持久化之外，一個重要的特性是Redis支持豐富的數據類型，即Memecache只支持字符串類型，所有鍵值對都是字符串類型，而Redis的值支持字符串，列表，字典，集合，有序集合五種類型，故可以提供更加豐富的操作。
  • Redis的每種數據類型都支持多種底層數據結構實現，即每種數據類型並不是綁定為一種數據結構的，而是可以多種。這種設計的原因是：Redis是一種內存數據庫，所有數據都保存在內存中，故在設計時，需要在保證性能的前提下，儘可能的減少內存消耗，在內存和性能之間進行一個平衡，從而可以存儲更多的數據。所以在介紹數據類型之前，先介紹一下Redis底層的數據結構。

二、數據結構

簡單動態字符串

  • Redis是使用C語言編寫的，Redis沒有直接使用C語言的字符串實現，而是基於C語言的字符串實現了簡單動態字符串。其主要原因是C的字符串是底層的API，存在以下問題：（1）沒有記錄字符串長度，故需要O(n)複雜度獲取字符串長度；（2）由於沒有記錄字符串長度，故容易出現緩衝區溢出問題；（3）每次對字符串拓容都需要使用系統調用，沒有預留空間（4）C的字符串只能保存字符。
  • 基於以上問題，Redis的簡單動態字符串提供了：（1）通過len記錄字符串長度，實現O(1)複雜度獲取；（2）內部字符串數組預留了空間，減少字符串的內存重分配次數，同時實現了自動拓容避免緩衝區溢出問題；（3）內部字符數組基於字節來保存數據，故可以保存字符和二進制數據。具體數據結構設計如下：
struct __attribute__ ((__packed__)) sdshdr64 { // 已使用字符串長度 uint64_t len; /* used */ // 一共分配了多少字節，alloc - len就是預留的 uint64_t alloc; /* excluding the header and null terminator */ unsigned char flags; /* 3 lsb of type, 5 unused bits */ // 字節數組 char buf[];};

雙向鏈表

  • C語言沒有提供雙向鏈表的實現，故Redis自身實現了一個雙向鏈表，主要用於存放列表數據。具體數據結構如下：
  • 鏈表節點定義：
typedef struct listNode { // 前置節點 struct listNode *prev; // 後置節點 struct listNode *next; // 數據，使用void指針實現多態，即可以存放多種數據類型的數據 void *value;} listNode;
  • 鏈表定義：
typedef struct list { // 鏈表頭指針 listNode *head; // 鏈表尾指針 listNode *tail; void *(*dup)(void *ptr); void (*free)(void *ptr); int (*match)(void *ptr, void *key); // 鏈表長度 unsigned long len;} list;

鏈式哈希字典

  • Redis的哈希表實現為鏈式哈希，即衝突節點使用鏈表來維護，具體數據結構如下：
  • 哈希鏈表節點：
// 哈希鏈表節點typedef struct dictEntry { // 鍵key void *key; union { // 值value void *val; uint64_t u64; int64_t s64; double d; } v; // 下一個鏈表節點 struct dictEntry *next;} dictEntry;
  • 鏈式哈希表實現：
// 鏈式哈希表實現typedef struct dictht { // 鏈式哈希實現 dictEntry **table; // 哈希表元素個數 unsigned long size; // 哈希掩碼，用於基於&與運算來計算哈希索引，大小總是為size -1 unsigned long sizemask; unsigned long used;} dictht;
  • Redis哈希結構：
typedef struct dict { dictType *type; void *privdata; // 兩個哈希表，其中一個空著在rehash的時候使用 dictht ht[2]; long rehashidx; /* rehashing not in progress if rehashidx == -1 */ unsigned long iterators; /* number of iterators currently running */} dict;

壓縮列表ziplist

  • 壓縮列表主要是基於節省內存的目標設計的，是由一些特殊編碼的連續內存塊組成的順序型數據結構。
  • 當列表元素較少且元素類型都是小整數或者比較短的字符串時，使用壓縮列表來存儲。因為壓縮列表的數據存取一般是需要遍歷，即線性O(n)時間複雜度，但是元素較少時，不會造成性能問題，類似於常量O(1)時間複雜度了。
  • 壓縮列表的整體結構如下：主要包括了列表總字節數zlbytes，列表尾節點距離列表起始地址多少個字節數zltail，列表總元素個數zllen，列表元素數據節點entry（圖片均引自黃建宏的《Redis設計與實現》）

  • 元素列表的元素節點：每個節點都包含了前一個節點的大小，從而基於當前元素的位置計算前一個節點的位置實現遍歷；encoding表示content所保存的值的數據類型和長度，content為具體的值。

數集合intset

  • 整數集合是集合的一種數據結構，當集合只存在整數元素且集合元素個數不多時，使用這種數據結構。
  • 具體數據結構定義如下：
typedef struct intset { uint32_t encoding; uint32_t length; // 整數元素集合，類型為int8整數 int8_t contents[];} intset;

跳躍表skiplist

  • 跳躍表是一種特殊的鏈表結構，在每個鏈表節點包含了指向其他鏈表節點的指針，故可以快速訪問其他鏈表節點，而不需要順序遍歷，實現了通過空間換時間的方法來優化性能。跳躍表的讀寫平均複雜度為O(logN)，故樹結構差不多，不過結構比樹結構簡單。
  • 在Redis中，跳躍表主要用於實現有序集合zset，每個鏈表節點包含了成員對象和分數。從頭結點到尾節點，以分值從小到大的升序排序。
  • 跳躍表定義：
typedef struct zskiplist { // 跳躍表的頭結點和尾節點 struct zskiplistNode *header, *tail; // 鏈表節點總個數 unsigned long length; // 層的個數 int level;} zskiplist;
  • 跳躍表節點定義：
typedef struct zskiplistNode { // 成員對象 sds ele; // 分數 double score; // 後置節點 struct zskiplistNode *backward; // 層級 struct zskiplistLevel { // forward指針，執行下一層的某個節點 struct zskiplistNode *forward; // forward指針對應的節點相對當前節點的層的跨度 unsigned int span; } level[];} zskiplistNode;

三、數據類型

  • 以下數據類型為提供給應用程序使用的，在底層實現中使用以上的一種或者多種數據結構進行編碼。
  • 對應每個鍵key可以使用TYPE命令查看其值的數據類型，使用OBJECT ENCODING來查查底層的編碼類型，如下：
127.0.0.1:6379> set test 123OK127.0.0.1:6379> type teststring127.0.0.1:6379> object encoding test"int"127.0.0.1:6379>

字符串string

  • 在Redis中，對應整數類型，如果不超過C語言long類型的範圍，則使用long類型來存儲，如果超過則使用embstr或者raw字符串來存儲；對應字符串和浮點數統一使用embstr或raw字符串來存儲。
  • int：可以使用long類型來保存的整數。
  • embstr：字符串的字節數小於等於32時，使用embstr來編碼，注意無法使用long類型來保存的整數，浮點數在Redis中都是使用字符串來存儲的，如果遇到計算命令，如incrby遞增，則Redis在讀出該字符串後，先轉換為相應的整數或浮點數進行計算。
  • raw：embstr無法存儲的字符串，即字節數大於32個字節的，則使用raw來編碼。
  • embstr和raw編碼一樣，都是用於存儲字符串，embstr是對raw在存儲短字符時的一種優化。不同的是embstr主要用於存放短字符串，在內存分配方面使用一次內存分配來創建數據類型string在Redis內部對應的對象redisObject和用於存儲數據的簡單動態字符串sds，而raw由於需要分配較大的sds，故使用兩次內存分配分別創建以上兩個對象。

列表list

  • 列表主要用於存放多個相同或不同的元素，在底層編碼方面也存在壓縮列表和雙向鏈表兩種數據結構。
  • 選擇規則：列表中的所保存的字符串元素的長度都小於64個字節，且列表中元素個數小於512個時，使用壓縮列表，否則使用雙向鏈表。
  • 以上為默認規則，其中切換臨界點：字符串長度和列表元素個數可以在配置文件redis.conf中通過list-max-ziplist-value，list-max-ziplist-entries來修改。

字典hash

  • 字典主要用於存放鍵值對數據，在底層編碼也存在壓縮列表和鏈式哈希表兩種數據結構。
  • 選擇規則：哈希對象的鍵和值的字符串長度都小於64字節且所有哈希對象個數小於512個，則使用壓縮列表，否則使用鏈式哈希表。
  • 以上為默認規則，也可以通過redis.conf中的hash-max-ziplist-value和hash-max-ziplist-entries參數來修改。

集合set

  • 集合set是一個存放無重複元素的集合，在底層編碼方面存在整數集合和鏈式哈希表兩種數據結構。
  • 選擇規則：集合中所有元素均為整數值（具體為int8）且元素個數不超過512個，則使用整數集合intset，否則使用鏈式哈希表。
  • 以上為默認規則，其中512的可以在redis.conf中通過set-max-intset-entries參數來修改。

有序集合zset

  • 有序集合主要實現了通過分數score來排序的功能，其中score可以重複，成員對象member不能重複，即有序是面向分數score的，集合是面向成員對象member的。
  • 在底層編碼也存在壓縮列表和跳躍表兩種數據結構。
  • 選擇規則：有序集合元素個數少於128個且每個元素的成員對象member的長度都小於64個字節，則使用壓縮列表，否則使用跳躍表。
  • 以上規則可以在redis.conf中通過zset-max-ziplist-entires和zset-max-ziplist-value來修改。

四、編碼降級

  • 除了字符串的值可以在int，embstr，raw之間切換之外（字符串類型對於SET命令其實是一個新的對象了），其他類型，即壓縮列表，整數集合intset和其他數據結構之間是不能降級的，如對於集合set，值從整數集合inset升級為鏈式哈希表之後，在刪除元素時，不能再降級為整數集合，如下：
127.0.0.1:6379> sadd set2 1(integer) 1127.0.0.1:6379> object encoding set2"intset"// 添加hello字符串從整數集合intset升級為hashtable127.0.0.1:6379> sadd set2 hello(integer) 1127.0.0.1:6379> object encoding set2"hashtable"// 移除hello之後，集合只存在1這一個整數，集合不會降級為整數集合intset127.0.0.1:6379> srem set2 hello(integer) 1127.0.0.1:6379> smembers set21) "1"127.0.0.1:6379> object encoding set2"hashtable"
