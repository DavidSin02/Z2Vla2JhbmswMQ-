VXLAN幀格式以及數據和控制層面

2020-10-29
| 科技

RFC7348規定了VXLAN報文的格式：



• Outer MAC Header：封裝外層以太頭，14字節，如果有VLAN TAG則為18字節。其中，源MAC地址（Outer Source MAC Address）為源VM所屬VTEP的MAC地址，目的MAC地址（Outer Destination MAC Address）為到達目的VTEP的路徑上下一跳設備的MAC地址（通常是網關Mac 地址）。類型字段為0x0800，指示內層封裝的是IP報文。

• Outer IP Header：封裝外層IP頭，20字節。其中，源IP地址（Outer Source IP Address）為源VM所屬VTEP的IP地址，目的IP地址（Outer Destination IP Address）為目的VM所屬VTEP的IP地址。協議字段為0x11，指示內層封裝的是UDP報文。

• UDP Header：UDP報文頭，8字節。其中，UDP目的端口號（UDP Destination Port）固定為4789，指示內層封裝報文為VXLAN報文。UDP源端口號（UDP Source Port）為隨機任意值，VTEP設備動態分配的。通過內層頭部的L2/L3/L4信息做Hash計算得出。可以用於VTEP之間多路徑負載分擔的計算。UDP的封裝可以穿越三層網絡，比VLAN有更好的擴展性。

• VXLAN Header：VXLAN協議新定義的VXLAN頭，8字節。

• Flags：8 bit，RRRRIRRR。“I”位為1時，表示VXLAN頭中的VXLAN ID有效；為0，表示VXLAN ID無效。“R”位保留未用，設置為0。

• VXLAN ID（VNI）：24 bit，用於標識一個單獨的VXLAN網絡。

• Reserved：分別為24 bit和8 bit。保留位。

• Original L2 Frame：原始以太網報文。

從報文的封裝可以看出，VXLAN頭和原始二層報文是作為UDP報文的載荷存在的。在VTEP之間的網絡設備，只需要根據Outer MAC Header和Outer IP Header進行轉發，利用UDP Source Port進行負載分擔，這一過程，與轉發普通的IP報文完全相同。這樣，除了VTEP設備，現網的大量設備無需更換或升級即可支持VXLAN網絡。

不過，新增加的VXLAN報文封裝也引入了一個問題，即MTU值的設置。

一般來說，虛擬機的默認MTU為1500 Bytes，也就是說原始以太網報文最大為1500字節。這個報文在經過VTEP時，會封裝上50字節的新報文頭（VXLAN頭8字節+UDP頭8字節+外部IP頭20字節+外部MAC頭14字節），這樣一來，整個報文長度達到了1550字節。而現有的VTEP設備，一般在解封裝VXLAN報文時，要求VXLAN報文不能被分片，否則無法正確解封裝。這就要求VTEP之間的所有網絡設備的MTU最小為 1550字節。

如果中間設備的MTU值不方便進行更改，那麼設置虛擬機的MTU值為1450，也可以暫時解決這個問題。

2. Vxlan的數據和控制平面

(1) 數據平面---隧道機制

VTEP為虛擬機的數據包加上了層報頭，這些新的報頭在數據到達目的VTEP後才會被去掉。

中間路徑的網絡設備只會根據外層包頭內的目的地址進行數據轉發，對於轉發路徑上的網絡來說，一個Vxlan數據包跟一個普通IP包相比，除了個頭大一點外沒有區別。

由於VXLAN的數據包在整個轉發過程中保持了內部數據的完整，因此VXLAN的數據平面是一個基於隧道的數據平面。

(2) 控制平面----改進的二層協議

VXLAN不會在虛擬機之間維持一個長連接，所以VXLAN需要一個控制平面來記錄對端地址可達情況。控制平面的表為(VNI，內層MAC，外層vtep_ip)。Vxlan學習地址的時候仍然保存著二層協議的特徵，節點之間不會週期性的交換各自的路由表，對於不認識的MAC地址，VXLAN依靠組播來獲取路徑信息(如果有SDN Controller，可以向SDN單播獲取)。

另一方面，VXLAN還有自學習的功能，當VTEP收到一個UDP數據報後，會檢查自己是否收到過這個虛擬機的數據，如果沒有，VTEP就會記錄源vni/源外層ip/源內層mac對應關係，避免組播學習。
