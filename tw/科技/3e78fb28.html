python窗口編程-7：收尾與打包

2020-10-29
| 科技

一個看似簡單的計算器項目，算上這次，我們一共用了六節課才把它講完整。這次我們解決最後剩下的幾個問題，然後將做好的代碼製作成EXE文件。並對整個項目做一個總結。


改變風格

前面幾節課一直是深棕色風格，不知道大家看得有沒有審美疲勞，但一聲多少有一點了。好在python開發的窗口程序更改風格比較容易，這次讓我們換一個色調的風格。改變風格後，所有默認的顏色都會自動改變，但我們主動設置的顏色卻不會變，可以手工調整一下。

這裡看到的效果是已經打包成EXE文件後的效果，具體的打包方法後面介紹。


sg.theme('GreenMono')funColor= '#239221'textColor= 'green'

代碼中更換了主題風格（theme），把功能鍵的顏色（funColor）和文本框的顏色（textColor）都重新定義了。

我們注意到，顏色的定義，都是一個字符串。但內容既可以用顏色的英文名（'green'），也可以用更加精確的RGB方式來定義（'#239221'）。具體地說：開頭必須為#，後面跟6個數字，2位一段，分為三段，分別代表紅色R，綠色G，藍色B的成分。數字是十六進制的，最小是00，最大是FF。

顏色設置改為變量

以前代碼中，設置顏色背景按鈕的地方，我們用的是常量，現在的修改，我們把它改為變量。因為它既然能夠改變一次，就有可能改變第二次。以後再改變按鈕，我們就只須改一個地方就可以了。朋友們如果想調整按鈕文字的顏色，可以做類似的修改。

涉及小數的計算

從我們代碼的內部機制來看，涉及小數的計算沒有什麼大問題，雖然浮點數不夠精確，對於我們這個只有12位數碼的顯示屏的計算器來說，也夠用了。唯一的問題是顯示問題。

由於我們的顯示屏寬度有限，所以如果結果是一個小數，那麼我們該顯示小數點後面多少位呢？

比較合理的是這樣一個邏輯：首先保證整數部分完全顯示，剩下的寬度，有多少位就給小數部分多少位，最後一位四捨五入即可。

對於無限小數，這個邏輯沒有問題。但對於有限小數來說，就不合理了。9/6=1.5，不應當顯示為1.5000000（如果是按鍵輸入當然可以）。所以邏輯補充：小數部分按分配的位數顯示時，捨去末尾的0。

上面這個邏輯，是自然語言描述的，我們看看它翻譯為python語言是什麼樣。

話說，作為一個編程者，我們的工作在某種意義上，很像一個翻譯，不是麼？

if isinstance(result, float):    # 第一次顯示小數，保留20位小數，實際我們只關心整數    str1= '%.20f'%result    # 計算有多少位整數    v1= len(str1.split(".")[0])    if v1<11:        # 四捨五入到保留的小數位11-v1        result= round(result, 11-v1)        # 再次顯示小數，按最大可分配的位數，這次關注小數尾有多少個0        str1= ('%%.%df'%(11-v1))%result        # 捨去結尾的0，得到正確小數位        decNum= len(str1.strip('0').split(".")[1])+1    else:        # 如果整數位超範圍，簡單粗暴置0        result= 0        decNum= 0else:    # 如果是整數，並且超範圍，簡單粗暴置0    str1= '%d'%result    v1= len(str1)    if v1>11:        result=0

詳細的講解，請看代碼中的註釋。文章裡重點講三個新引入的字符串用法。

  1. 字符串切割（split）這個功能將一個字符串以某個字符（參數）為標記，切分成一個列表。對於小數來說，以小數點為標記的切割，第一個（編號[0]）是整數部分，第二個（編號[1]）就是小數了部分。
  2. 字符串削皮（strip）這個功能，是去掉字符串兩邊的指定字符（參數），嚴格來說，這裡為了去掉小數尾部的0，可以用rstrip（strip, lstrip, rstrip）但這裡因為並不關心左邊，所以strip也得到正確的結果。
  3. 連百分號（%%），前面課程學到%d，%f都具有特殊含義，如果是兩個連續的百分號%%，就代表%本身了。這樣可以把動態的字符串寫在一行代碼中。這種技巧多了，代碼會更簡短，但人類閱讀起來就更加難懂了。

關於超範圍的處理，當然還有其它的方法，有興趣可以自己創意。比如：再增加一個狀態error，在error狀態下，屏幕給出相應提示。


現在的效果是，如果運算的結果足夠大，會直接變成0；而當小數運算時，如果數字足夠小，也會直接變成0。

被0除的問題

數位超長，並不是一個真正的錯誤。在我們所提供的計算功能中，還真有一個出錯的可能，就是被0除。如果我們點擊9/0=這組按鈕，程序會報錯。如果計算功能更加複雜，比如我們開發的是一個科學計算器，那就有更多的出錯可能了（比如試圖計算負數的對數等）。

讓程序報出內部錯誤，不像是一個正規軟件的作為。我們應當在代碼中捕獲這種錯誤，並進行相應的提示和處理。

try:    result= opnd1/ opnd2except BaseException:    result=0

這裡我們就看到了python編程中的錯誤捕獲代碼try...except。在解決問題的邏輯中，可以完全沒有try...except的存在，但在稍微複雜一點的實際應用中，我們到處會發現 try...except 的身影。這說明解決問題（在大多數情況下）其實只是很簡單是一件事。全面地考慮各種可能出現的問題，才是複雜的地方。

這裡的BasicException代表任何錯誤，說明我們不關心實際出錯的細節，只是急於把結果簡單置0。這當然這樣處理並不嚴謹，只是因為異常捕獲的問題不是本節課的重點。我們現在只需記住有這樣一種表達方法即可。

這與我們開車有點類似：開車本身並不難，直接地解決問題就如同沿著筆直的大路駕駛，即使是新手也能做得很好；真正的麻煩是處理各種可能的錯誤，就如同駕駛過程中出現的各種突發事件（前車忽然急剎、有人強行併線等等），那才是老司機展示能力的領域。

引申一下：如果你當前編程學習的狀態是不斷提升解決問題的能力，這是一個很有趣的世界，希望你能感受到編程學習的樂趣；如果你發現幾乎已經沒有問題不能解決，恭喜你來到真實世界（Welcome to the real world.），可以開始用編程語言和這個世界對話了。

編譯成EXE發佈

做出一個完整的窗口界面程序，我們就會希望製作成EXE並與他人分享，這個時候我們的程序就可以叫做一個“軟件產品”了。如果你做的這個軟件真的能解決一些實際的問題，說不定真的會流行起來（那就是另一個領域的話題了）。

“打包”這個詞語，我們理解它的本意是：打成一個完整的包裹，便於發佈。所以，把python源代碼做成EXE的過程，可以算是“打包”。如果你的程序除了可運行的EXE文件，還有其它資源文件，打包一般就指製作安裝文件。因為我們的計算器軟件只包含一個EXE可運行文件，所以把它進一步壓縮成一個ZIP或RAR文件，也就可以算是一個綠色安裝包了。所以壓縮文件，有時也稱打包。下文為了簡便，說打包一概指製作EXE文件。

1、安裝pyInstaller

python默認沒有打包功能，需安裝pyInstaller庫，執行安裝代碼如下：

pip install pyinstaller -i https://pypi.doubanio.com/simple/

如果代碼執行過程出錯，出現這樣的提示

error: invalid command 'bdist_wheel'

則須先安裝一下 wheel工具，代碼如下：

pip3 install wheel

然後再次用前面的代碼安裝pyInstaller即可

2、生成EXE

生成窗口EXE的一般指令是：

pyinstaller -wF calc.py

在Python代碼的目錄下，打開命令行窗口，執行即可。

如果想使用自帶的圖標也是可以的。首先找到一個圖標文件（擴展名是.ico），有些資源網站會提供這類資源。將這個文件與python代碼放在同一個目錄下，執行指令：

pyinstaller -i calc.ico -wF calc.py

這樣，生成的EXE文件，就帶有你選擇的圖標了。
