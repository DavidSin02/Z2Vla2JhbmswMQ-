Canal 如何實現數據庫庫事務的一致性

2020-10-29
| 科技

本文將主要介紹在 EventParser binlog 日誌同步流程中關於環形緩存區的使用技巧。

1、環形緩存區

關係型數據庫講究的是ACID 4個特性，故引入了數據庫事務的概念，一個數據庫事務中的多條SQL引發的多條數據變更要麼全部成功，要麼全部失敗，即數據的一致性，那同樣在數據同步的場景，在解析一個事務的 binlog 日誌時，一次數據同步應該至少以事務為單位，一個事務內的所有 Event 應該作為一個批次提交到數據消費端，讓消費端有能力一次同步一個事務中的數據，而不是一條一條變更日誌的處理，這樣容易造成數據不一致。

環形緩存區的引用就是為了解決將一個事務的完整數據一次提交到消費端，既然是多條消息，故一定需要用到緩存，環形緩存區就在這樣的背景下被引入。

在 Canal 中關於事務 Event 的環形緩存區實現類為 EventTransactionBuffer。

1.1 類圖

EventTransactionBuffer 的類圖如下：

根據類圖我們可以到其存儲結構還是比較簡單的。

  • int bufferSize
    環形緩存區的長度，默認為 1024，該長度必須為 2 的冪次方，因為對位運算非常友好。
  • int indexMask
    環形緩存區下標掩碼，其值為 bufferSize - 1 ，sequence * indexMask 能快速定位序號 sequence 所在環形緩存區中的具體下標。
  • CannalEntry.Entry[] entries
    環形緩存區數據數組，即緩存區實際存儲數據的內存區域，為數組結構，長度為 bufferSize。
  • AtomicLong putSequence
    當前寫入的序號，每調用 add 方法添加一條數據，該值增加一，可超過緩存區的實際長度。
  • AtomicLong flushSequence
    當前已處理的數據序號，flushSequence <= putSequence，（putSequence - flushSequence）表示未處理的數據，即緩存區累積的有效數據。
  • TransactionFlushCallback flushCallback
    flush 回調函數，這個和環形緩存區本身關係不大，這個與 Canal 特定業務的，環形緩存區中收集到一個完整的事務變更日誌列表後，將這部分內容傳入業務回調方法，並重新利用這些緩存空間。

環形緩存區的重大要義就是循環利用。

首先根據 binlog 事件類型來決定是否調用 flush 方法，這個就是實現將一個事務的事務一起提交到消費端，回到環形緩存區的具體實現，我們重點關注 put 方法 與 flush 方法的實現。

EventTransactionBuffer#put

其實現的核心步驟：

  1. 檢測當前環形緩存區是否已滿，如果未滿，則向緩存區中添加一條數據。添加數據的具體邏輯：
  • 獲取下一個寫入的序號 next，等於當前已寫入的序號 + 1，即 putSequence + 1。
    • 通過 next & indexMask 取得放入 CannalEntry.Entry[] entries 中的下標，與 next % bufferSize 效果等同。
  1. 如果已滿，則首先將緩存區中的數據刷新，即將未處理的數據全部抽取，提交到數據消費方，然後釋放緩存區，繼續添加數據。

此時 putSequence 為 4，flushSequence 為 -1 ，我們應該能發現，在第一輪時，由於 sequeue 小於 bufferSize ，如果不執行 flush 操作，連續寫入 8條數據，sequence = 7 時，sequence - bufferSize > flushSequence 這個表達式都不會滿足，即代表緩存區未滿，但在寫入第9條消息時，sequence = 8 ，此時 sequence - bufferSize > flushSequence 已滿足，即緩存區已滿，需要先刷新數據，然後才能再填充。

再回到本示例中，一個事務只包含5條日誌，在寫滿 5條日誌後會即調用 flush 方法，將環形緩存區中下標為 0~4 的消息傳入數據消費方，在 Canal 中會將這批消息一次傳入 EventSink 組件。執行完 flush 方法後，flushSequence 等於4，其環形緩存區如下圖所示：

此時 putSequence = flushSequence = 4，那這個時候環形緩存區的容量是多少呢？其實就是又恢復到 bufferSize 了，那我們怎麼計算環形緩存區當前已寫入的消息呢？其實很簡單，putSequence - flushSequence 表示已寫入的元素數量。那當前剩餘容量就等於 bufferSize - (putSequence - flushSequence )，即只需要 bufferSize - (putSequence - flushSequence ) > 0 就表示有剩餘空閒。有了這一層思路，就能明白 checkFreeSlotAt 的算法，這也是環形緩存區的核心所在。

思考，Canal 基於環形緩存區的實現，一定能保證一個事務的所有變更日誌都一次提交到 EventSink 組件嗎，大家可以簡單思考一下，在文末的總結部分有筆者的思考。

答案是否定的，如果一個事務包含的日誌條目超過了環形緩存區的長度，為了保證數據不丟失，會首先將環形緩存區的數據全部提交，然後接收新的數據，這樣一個事務中的消息會被分成多次提交到 EventSink。
