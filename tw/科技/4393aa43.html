遊戲開發系列之正確的使用斷言與返回

2020-11-14
| 科技

有一天的清晨，老A早早的來到了辦公室，決定進行一次code review，當然，對象必然是入職不久的小C。小C打開svn，展示了一段最近寫的代碼.

function ProcessItem()

assert(gItem) //斷言道具表存在

assert(gEquip) //斷言裝備表存在

--其他代碼

end

function ProcessItemEquip()

if not gItem then

return

end

if not gEquip then

return

end

//其他代碼

end

這2個函數的意圖很明顯都是在函數入口處對配置不存在的時候進行的處理。

那麼我們首先應該清楚在邏輯上，這2個配置表可能是為nil的，也就是空的。那麼我們究竟應該選擇哪一種寫法呢？

先討論一個問題，什麼時候應該用assert進行斷言，什麼時候應該用return。

a.在邏輯上面允許發生的情況下，我們應該用return。

b.如果是邏輯上認為不應該發生的，我們應該用斷言。

在上面的問題上，如果在設計上在調用ProcessItemEquip可能存在gItem為空的情況，那麼我們在這邊應該用return，如果在設計上，ProcessItem調用的時候gItem應該存在了，那麼應該用assert。當然，從這個問題延伸的是更深的一個設計問題，我們應該把邏輯設計成在一個函數調用某個配置的時候，這個配置就是存在的，這樣會大大減輕後續代碼的各種判斷。從這個角度也可以說明，我們應該將設計設計為更適合使用assert這樣的環境。

討論下面一個場景,寫一個引用計數的2個接口

self.nRef = 0;

function incRef()

Self.nRef = Self.nRef + 1

End

下面是2種寫引用計數的寫法，哪種更正確？

function decRef()

self.nRef = self.nRef – 1

assert(self.nRef >= 0)

end

function decRef()

self.nRef = self.nRef – 1

if self.nRef <= 0 then

end

end

第一種是斷言，第二種是容錯。

小C覺得使用第二種，就沒有必要關注外界用錯導致引用計數被多減的情況，方便了外界邏輯層的調用。那麼為什麼我們還是推崇第一種寫法，因為第一種寫法是在問題發生的時候第一時間把錯誤斷住了，它要求邏輯調用者必須找到引發斷言的根本性問題。

那麼我們經常在什麼時候會寫return這種兼容的代碼？比如說一個網絡消息過來，檢測道具數量夠不夠，第一次協議過來是夠的，第二次協議過來就不夠了。那麼我們在檢測道具不足的時候用return而不是assert。那你可能會說客戶端也加檢測就可以assert了。考慮到第一條協議可能還沒返回新的數據給客戶端的時候第二條協議也過來(數據層我們統一以服務器為準)，這時候客戶端是攔不住的，所以這種情況我們使用return。
