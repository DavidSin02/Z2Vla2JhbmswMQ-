面試必問：嵌套函數、閉包與裝飾器

2020-10-29
| 科技

Python 函數可以嵌套定義，我們先考察一個典型的例子：

def adder(n):    def handler(x):        return n+x    return handler

adder 函數負責創建處理函數 handler ，處理函數計算參數 x 與固定值 n 的和。因此，如果我們需要一個為參數加上 1 的函數，調用 adder(1) 即可輕鬆得到：

>>> add1 = adder(1)>>> add1(10)11>>> add1(15)16

同樣，如果我們需要一個將參數加上 5 的函數，調用 adder(5) 即可：

>>> add5 = adder(5)>>> add5(10)15>>> add5(15)20

很顯然，對於 add1 來說，n 的值是 1 ；對於 add5 來說，n 的值是 5 ；兩者保存獨立，互不干擾。

理論上，當函數 adder 返回，局部變量 n 應該就被回收了，為什麼 handler 函數還能訪問到它呢？

另外，像 adder 函數和 handler 函數這種嵌套寫法，到底有什麼作用？適用於什麼開發場景？有什麼需要特別注意的地方嗎？

為了解答這諸多疑惑，我們需要深入學習 嵌套函數 與 閉包 變量的來龍去脈。

嵌套函數

像 adder 函數和 handler 這樣，在一個函數的函數體內定義另一個函數，就構成了 嵌套函數 。我們在 虛擬機 部分講解 作用域 時，已對嵌套函數有所瞭解。你還記得嵌套函數與作用域之間密切的聯繫嗎？

根據我們在虛擬機部分學到的知識，adder-handler 這段簡單的代碼卻包含著 3 個不同的作用域：

作用域是一個靜態概念，由 Python 代碼語法決定，與編譯後產生的 代碼對象 一一對應。作用域規定了能夠被某個代碼塊訪問的變量有哪些，但對變量具體的值則一概不關心。

一旦 Python 程序開始運行，虛擬機需要為作用域中的變量分配一定的存儲空間，這就是 名字空間 。名字空間依照作用域規則實現，它決定了某個變量在運行時的取值，可以看做是作用域在運行時的動態表現方式。

全局與局部這兩個名字空間我們已經非常熟悉了，那麼 閉包 名字空間又該如何理解呢？

閉包

閉包 ( closure ) 是 詞法閉包 ( Lexical Closure ) 的簡稱，是指引用了自由變量的函數。這些被引用的自由變量將和這個函數一同存在，即使已經離開了創造它的環境也不例外。

以 adder(10) 為例，它是一個 handler 函數對象，閉包變量 n 值總是 10 。那麼，內層函數是如何訪問閉包作用域的呢？我們對函數代碼對象進行反彙編，從中可以看出端倪：

>>> add10 = adder(10)>>> add10<function adder.<locals>.handler at 0x10dc2b6a8>>>> add10.__code__<code object handler at 0x10dbe5150, file "<stdin>", line 2>>>> dis.dis(add10.__code__)  3           0 LOAD_DEREF               0 (n)              2 LOAD_FAST                0 (x)              4 BINARY_ADD              6 RETURN_VALUE

我們發現一條全新的字節碼 LOAD_DEREF ，正是它執行了閉包變量查找工作！LOAD_FAST 指令則負責局部變量查找，而局部名字空間我們已經有所瞭解，它藏身於棧幀對象中。

那麼，閉包名字空間又藏在哪呢？

順著虛擬機處理 LOAD_DEREF 字節碼的代碼，不難找到答案。與該字節碼處理相關的源碼位於 Python/ceval.c 文件，有興趣的同學可以深入看一下。

因篇幅所限，這裡直接給出答案：當閉包函數 handler 執行時，閉包變量藏身於 PyFrameObject 中。

還記得嗎？每次函數調用虛擬機都會創建一個 PyFrameObject 對象，用於保存函數調用上下文。全局名字空間與局部名字空間都藏身其中，閉包名字空間也不例外。

前面章節提過，PyFrameObject 結構體最後部分是不固定的，依次存放著靜態局部名字空間、閉包名字空間以及臨時棧。以 add10(1) 為例，函數運行時 PyFrameObject 狀態如下如下：

我們重點關注 PyFrameObject 末尾處，局部變量、閉包變量以及臨時棧依次排列。

由於函數局部變量、閉包變量個數在編譯階段就能確定，運行時並不會增減，因此無須用 dict 對象來保存。相反，將這些變量依次排列保存在數組中，然後通過數組下標來訪問即可。這就是所謂的靜態名字空間。

對於局部變量 n ，數組對應的槽位保存著整數對象 1 的地址，表示 n 與 1 綁定。而閉包變量 x 則略有差別，槽位不直接保存整數對象 10 ，而是通過一個 PyCellObject 間接與整數對象 10 綁定。

那麼，閉包變量是如何完成初始化的呢？

我們知道全局名字空間 f_globals 來源於函數對象，那麼閉包名字空間是否也是如此呢？再次研究 CALL_FUNCTION 字節碼處理邏輯，印證了我們的猜測：

函數對象 PyFunctionObject 中有一個字段 func_closure ，保存著函數所有閉包變量。我們可以通過名字 __closure__ 可以訪問到這個底層結構體字段：

>>> add10.__closure__(<cell at 0x10dc09e28: int object at 0x10da161a0>,)

這是一個由 PyCellObject 組成的元組，PyCellObject 則保存著閉包變量的值。當函數調用發生時，Python 虛擬機創建 PyFrameObject 對象，並從函數對象取出該元組，依次填充相關靜態槽位。

接著我們來考察 PyCellObject 對象的行為，通過 cell_contents 屬性可以訪問閉包變量值：

>>> add10.__closure__[0]<cell at 0x10dc09e28: int object at 0x10da161a0>>>> add10.__closure__[0].cell_contents10

我們嘗試將函數閉包變量進行修改，發現函數的行為將受到影響：

>>> add10.__closure__[0].cell_contents = 20>>> add10(1)21

這種行為雖合乎常理，但千萬不要在實際項目中應用，不然可能會踩坑！

至此，一切都明瞭了！從 adder 函數字節碼告訴我們，它將 handler 函數所有閉包變量存為 tuple 對象，再執行 MAKE_FUNCTION 字節碼完成函數對象創建。該元組最終將作為 closure 參數傳給函數對象，並保存在 func_closure 字段中。

>>> dis.dis(adder)  2           0 LOAD_CLOSURE             0 (n)              2 BUILD_TUPLE              1              4 LOAD_CONST               1 (<code object handler at 0x102864150, file "<stdin>", line 2>)              6 LOAD_CONST               2 ('adder.<locals>.handler')              8 MAKE_FUNCTION            8             10 STORE_FAST               1 (handler)  4          12 LOAD_FAST                1 (handler)             14 RETURN_VALUE

那麼，為什麼閉包變量要通過 PyCellObject 間接引用呢？我們將函數對象與運行時的棧幀對象放在一起來討論：

最新的 Python 提供了 nonlocal 關鍵字，支持修改閉包變量。如果沒有 PyCellObject ，函數在運行時直接修改 PyFrameObject ，函數返回就被回收了。藉助 PyCellObject ，函數在運行時修改的是 ob_ref 。這樣一來，就算函數返回，修改還是隨函數而存在。

裝飾器

藉助閉包，我們可以讓函數具備搭積木的魔法，例如：函數調用前後做一些統一的處理操作。

事不宜遲，我們來實踐一下，實現 log_call 函數，在指定函數調用前後各輸出一句話：

def log_call(func):    def handler(*args, **kwargs):        print('start to call')        result = func(*args, **kwargs)        print('end of call')        return result    return handler

log_call 接受一個函數對象 func 作為參數，這就是我們想為其注入魔法，以便在調用前後輸出提示的函數，姑且稱為 原函數 。它返回一個閉包函數 handler ，姑且稱為 代理函數 ，閉包變量 func 與 handler 緊緊綁定在一起。

當 handler 函數被調用時，它先輸出提示，再調用原函數 func ，函數調用完畢再次輸出提示。注意到，由於原函數參數是未知的，因此 handler 通過可變參數 *args 以及 **kwargs 進行傳遞。

現在，我們將 log_call 用起來！對於任意的函數 foo ，log_call 輕鬆即可為它注入魔法：

>>> def foo():...     print('foo processing')...>>> log_call(foo)()start to callfoo processingend of call

如果我想為其他函數，例如 bar 注入 log_call 魔法，可以這樣實現：

def bar():    print('bar processing')bar = log_call(bar)bar()

這代碼未免太醜陋了！為此，Python 引入了語法糖 @xxxx ：

@log_calldef bar():    print('bar processing')bar()

這段代碼與上面那段完全等價，卻更加優雅、清晰！

像 log_call 這樣，為其他函數注入新功能的函數，就是所謂的 裝飾器 ( decorator )
