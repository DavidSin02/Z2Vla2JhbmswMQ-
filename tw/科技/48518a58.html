入侵JSON Web令牌（JWT）

2020-11-14
| 科技

攻擊者如何偽造令牌並以其他人身份登錄

JSON Web令牌是一種在商業應用程序中廣泛使用的訪問令牌。它們基於JSON格式，幷包含令牌簽名以確保令牌的完整性。

今天，我們將討論使用JSON Web令牌（通常是基於簽名的令牌）的安全隱患，以及攻擊者如何利用它們繞過訪問控制。

JSON Web令牌如何工作？

JSON Web令牌包含三個組件：標頭，payload和簽名。

標頭

JSON Web令牌的標頭部分標識用於生成簽名的算法。它是JSON Blob的base64編碼的字符串，如下所示：

{ "alg" : "HS256", "typ" : "JWT"}base64 encoded string: eyBhbGcgOiBIUzI1NiwgdHlwIDogSldUIH0K

最常用的算法是HMAC和RSA算法。

payload

payload部分包含實際用於訪問控制的信息。此部分在用於令牌之前也已進行base64編碼。

{ "user_name" : "admin",}base64 encoded string: eyB1c2VyX25hbWUgOiBhZG1pbiB9Cg

簽名

簽名是用於驗證令牌未被篡改的部分。它是通過將標頭與payload串聯在一起，然後使用標頭中指定的算法進行簽名來計算的。

signature = HMAC-SHA256(base64Encode(header) + '.' + base64Encode(payload), secret_key)// Let's just say the value of secret_key is "key".-> signature function returns 4Hb/6ibbViPOzq9SJflsNGPWSk6B8F6EqVrkNjpXh7M

對於此特定令牌，字符串“ eyBhbGcgOiBIUzI1NiwgdHlwIDogSldUIH0K.eyB1c2VyX25hbWUgOiBhZG1pbiB9Cg”已用密鑰“ key”的HS256算法簽名。生成的字符串是4Hb / 6ibbViPOzq9SJflsNGPWSk6B8F6EqVrkNjpXh7M。

完整的令牌

通過將每個部分（標頭，有效負載和簽名）與每個部分之間的“。”串聯起來，可以獲得完整的令牌。

eyBhbGcgOiBIUzI1NiwgdHlwIDogSldUIH0K.eyB1c2VyX25hbWUgOiBhZG1pbiB9Cg.4Hb / 6ibbViPOzq9SJflsNGPWSk6B8F6EqVrkNjpXh7M

繞過JSON Web令牌控件的方法

如果正確實現，JSON Web令牌將提供一種安全的方式來標識用戶，因為payload部分中包含的數據無法被篡改。（由於用戶無權訪問密鑰，因此她不能自己對令牌進行簽名。）

但是，如果實施不正確，攻擊者可以通過多種方式繞過安全機制並偽造任意令牌。

更改算法類型

攻擊者可以偽造自己的令牌，其中一種方法是篡改標頭的alg字段。如果應用程序不限制JWT中使用的算法類型，則攻擊者可以指定要使用的算法，這可能會損害令牌的安全性。

  1. 無算法

JWT支持“無”算法。如果將alg字段設置為“ None”，則如果其標記部分設置為空，則任何令牌都將被視為有效。例如，以下令牌將被視為有效：

eyAiYWxnIiA6ICJOb25lIiwgInR5cCIgOiAiSldUIiB9Cg.eyB1c2VyX25hbWUgOiBhZG1pbiB9Cg。

它只是這兩個Blob的base64編碼版本，沒有簽名。

{ "alg" : "None", "typ" : "JWT"}{ "user" : "admin"}

此功能最初用於調試目的。但是，如果沒有在生產環境中將其關閉，則攻擊者可以通過將alg字段設置為“ None” 來偽造他們想要的任何令牌。然後，他們可以使用偽造的令牌模擬網站上的任何人。

2. HMAC算法

用於JWT的兩種最常見的算法類型是HMAC和RSA。使用HMAC，將使用密鑰對令牌進行簽名，然後使用相同的密鑰進行驗證。對於RSA，將首先使用私鑰創建令牌，然後使用相應的公鑰進行驗證。

HMAC -> signed with a key, verified with the same keyRSA -> signed with a private key, verified with the corresponding public key

至關重要的是，將HMAC令牌的私鑰和RSA令牌的私鑰保密，因為它們用於簽名令牌。

現在，我們假設有一個最初設計為使用RSA令牌的應用程序。令牌用私鑰A簽名，私鑰A對公眾保密。然後，使用任何人都可以使用的公鑰B驗證令牌。只要令牌始終被視為RSA令牌，就可以。

Token signed with key A -> Token verified with key B (RSA scenario)

現在，如果攻擊者改變的ALG到HMAC，她可能通過與RSA公鑰B.簽署偽造的令牌能夠創建有效令牌

這是因為最初使用RSA對令牌進行簽名時，程序會使用RSA公鑰B對其進行驗證。當將簽名算法切換為HMAC時，仍使用RSA公鑰B來驗證令牌，但是這次是使用令牌可以使用相同的公鑰B簽名（因為它使用的是HMAC）。

Token signed with key B -> Token verified with key B (HMAC scenario)

提供無效的簽名

令牌的簽名在到達應用程序後也可能永遠不會被驗證。這樣，攻擊者可以通過提供無效簽名來簡單地繞過安全機制。

暴力破解密鑰

也有可能暴力破解用於簽署JWT的密鑰。

攻擊者從一開始就有很多信息：她知道用於對令牌進行簽名的算法，已簽名的有效負載以及生成的簽名。如果用於對令牌進行簽名的密鑰不夠複雜，則她可能可以輕鬆地對其進行暴力破解。

洩漏密鑰

如果攻擊者無法蠻力破解密鑰，則可以嘗試洩漏祕密密鑰。如果存在另一個允許攻擊者讀取存儲密鑰值的文件的漏洞（如目錄遍歷，XXE，SSRF），則攻擊者可以竊取密鑰並簽署任意令牌。

KID操作

KID代表“Key ID”。它是JWT中的可選標頭字段，它使開發人員可以指定用於驗證令牌的密鑰。KID參數的正確用法如下所示：

{ "alg" : "HS256", "typ" : "JWT", "kid" : "1" // use key number 1 to verify the token }

由於此字段是由用戶控制的，因此攻擊者可能會操縱它並導致危險的後果。

  1. 目錄遍歷

由於KID通常用於從文件系統中檢索密鑰文件，因此，如果在使用前未對其進行清理，則可能導致目錄遍歷攻擊。在這種情況下，攻擊者將能夠在文件系統中指定任何文件作為用於驗證令牌的密鑰。

“kid”: “../../public/css/main.css” // use the publicly available file main.css to verify the token

例如，攻擊者可以迫使應用程序使用公開可用的文件作為密鑰，並使用該文件對HMAC令牌進行簽名。

2. SQL注入

KID還可以用於從數據庫檢索密鑰。在這種情況下，可能可以利用SQL注入來繞過JWT簽名。

如果可以在KID參數上進行SQL注入，則攻擊者可以使用該注入返回她想要的任何值。

“kid”: "aaaaaaa' UNION SELECT 'key';--"// use the string "key" to verify the token

例如，上面的注入將使應用程序返回字符串“ key”（因為數據庫中不存在名為“ aaaaaaa”的鍵）。然後將使用字符串“ key”作為密鑰來驗證令牌。

標頭參數處理

除密鑰ID外，JSON Web令牌標準還使開發人員能夠通過URL指定密鑰。

  1. JKU標頭參數

JKU代表“ JWK設置URL”。它是一個可選的標頭字段，用於指定指向一組用於驗證令牌的密鑰的URL。如果允許該字段，並且沒有適當地限制該字段，則攻擊者可以託管自己的密鑰文件，並指定應用程序使用它來驗證令牌。

jku URL -> file containing JWK set -> JWK used to verify the token

2. JWK標頭參數

可選的JWK（JSON Web密鑰）標頭參數允許攻擊者將用於驗證令牌的密鑰直接嵌入令牌中。

3. X5U，X5C URL操作

類似於jku和jwk標頭，x5u和x5c標頭參數允許攻擊者指定用於驗證令牌的公鑰證書或證書鏈。x5u以URI形式指定信息，而x5c允許將證書值嵌入令牌中。

其他JWT安全問題

如果未正確實施，還會產生其他JWT問題。這些不是很常見，但是絕對值得關注：

信息洩漏

由於JSON Web令牌用於訪問控制，因此它們通常包含有關用戶的信息。

如果令牌未加密，則任何人都可以通過base64解碼令牌並讀取令牌的有效負載。因此，如果令牌包含敏感信息，則它可能成為信息洩漏的來源。JSON Web令牌的正確實現的簽名部分可提供數據完整性，而不是機密性。

命令注入

有時，當KID參數直接傳遞到不安全的文件讀取操作中時，可能會將命令注入代碼流中。

可能允許這種類型的攻擊的函數之一是Ruby open（）函數。此功能使攻擊者只需在KID文件名之後將命令添加到輸入即可，即可執行系統命令：

“key_file” | whoami;

這只是一個例子。從理論上講，每當應用程序將未經消毒的任何頭文件參數傳遞給任何類似於system（），exec（）等的函數時，就會發生此類漏洞。

最終，JSON Web令牌只是用戶輸入的另一種形式。他們應始終持懷疑態度並嚴格審查。

免責聲明：這篇文章旨在引起人們對JSON Web令牌漏洞的關注，並幫助開發人員識別常見的陷阱。請不要使用此信息來攻擊網站或冒充他人。

在沒有測試權限的系統上嘗試此操作是非法的。如果您發現了漏洞，請以負責任的方式向供應商披露。幫助使我們的互聯網更安全。
