停止用CSV格式保存Pandas數據幀

2020-10-29
| 科技

CSV是一種很好的數據交換格式。所有人都能理解它，並且可以在普通記事本中進行編輯。但是，這並不意味著它適合保存所有的數據幀。用過的人應該都知道，CSV不僅讀寫速度慢，而且有時候需要下載數據，佔用更多的磁盤空間，最重要的是CSV不存儲數據類型的信息。

CSV的優勢：

  • 通俗易懂；
  • 大多數編程語言都支持解析和創建。

CSV的缺點：

  • 更大的磁盤佔用率；
  • 讀寫速度較慢；
  • 不存儲有關數據類型的信息。

在開始之前，我們首先來了解一下這些文件格式如何處理各種數據類型。

Pandas數據類型

Pandas 支持非常多數據類型，其中一些還具有多個子類型，用於提高處理大數據幀的效率。基本數據類型包括：

  • 對象 -- 字符串或混合類型
  • 字符串 -- 從熊貓1.0.0開始
  • int -- 整數
  • float -- 浮點數
  • bool -- 布爾值True和False值
  • datetime -- 日期和時間值
  • timedelta -- 兩個日期時間之間的時差
  • category -- 有限的值列表存儲了內存效率高的查找

由於 pandas 使用 numpy 數組作為其後端結構，因此 ints 和 float 可以區分為內存效率更高的類型，例如 int8，int16，int32，int64，unit8，uint16，uint32 和 uint64 以及 float32 和 float64。

CSV 不存儲有關數據類型的信息，所以我們必須在每個 read_csv（）中對它進行指定。在不告知 CSV 閱讀器的情況下，它將推斷所有整數列為效率最低的 int64，將 float 轉換為 float64，將類別作為字符串以及日期時間加載。

#對於每個加載的數據集，必須指定讓數據幀更高效的格式df = pd.read_csv(new_file,                 dtype={"colA": "int8",                        "colB": "int16",                        "colC": "uint8",                        "colcat": "category"},                 parse_dates=["colD","colDt"])

TimeDeltas以字符串的形式存儲在 CSVs-5days+18:59:39.000000 中，我們必須編寫一個特殊的解析器來將這些字符串轉換回 pandas 的 timedelta 格式。

時區看起來像 2020-08-06 15:35:06-06:00，並且需要在 read_csv（）中進行特殊處理。

使用沒有參數的read_csv（）比較原始dtypes和自動推斷的類型

CSV替代方案

幸運的是，csv不是保存數據幀的唯一選擇。閱讀 Pandas 的 IO 工具，你會發現一個數據幀可以寫入多種格式、數據庫，甚至是剪貼板。

接下來我們來詳細描述數據是如何創建的，並將教大家使用真實的數據幀完成性能測試和健全性檢查。

Pickle 和 to_pickle（）

Pickle 是用於對象序列化的 python 本地格式。 它允許 python 代碼實現任何形式的增強，例如PEP574 酸洗帶外數據緩衝區中描述的最新協議5。

（PEP574協議鏈接：https://www.python.org/dev/peps/pep-0574/）

這也意味著很難在 Python 生態系統之外進行酸洗。 但是，如果你想存儲一些預處理的數據以備後用，或者不想在不立即使用數據的情況下浪費幾個小時的分析工作，那麼你只需對它們進行 pickle 處理即可。

# Pandas的to_pickle方法df.to_pickle(path)

與 .to_csv（）相反，.to_pickle（）方法僅接受3個參數:

  • path（路徑）-- 數據將存儲到的位置；
  • compression（壓縮）-- 允許選擇各種壓縮方法；
  • protocol（協議）-- 更高的協議可以更有效地處理更大範圍的數據。

pickle 的優點：

  • 比 CSV 更快（取決於壓縮方法，寫入CSV的5–300％和讀取CSV的15–200％）；
  • 生成的文件更小（約為csv的50％）；
  • 保留有關數據類型的信息（100％）；
  • 無需指定過多的參數。

pickle 的缺點：

  • 原生於 python，因此缺少其他編程語言的支持；
  • 即使在不同的 python 版本中也不可靠。

Pickle能夠100％序列化的padnas數據類型

Parquet 與 to_parquet()

apache parquet 是 Hadoop 生態系統中使用的壓縮二進制列存儲格式。它允許序列化複雜的嵌套結構，支持按列壓縮和按列編碼，並提供快速讀取，甚至不需要讀取整個列，因為我們只需要部分數據。

# Pandas的to_parquet方法df.to_parquet(path, engine, compression, index, partition_cols)

.to_parquet（）方法僅接受以下幾個參數：

  • path（路徑） -- 數據將存儲到的位置；
  • engine（引擎） -- pyarrow 或 fastparquet 引擎。 pyarrow 通常更快，但與 timedelta 格式比較困難。 fastparquet 可能會明顯變慢；
  • compression（壓縮） -- 允許選擇各種壓縮方法；
  • index（索引） -- 是否存儲數據框的索引；
  • partition_cols -- 指定列分區的順序。

parquet 的優勢：

  • 比 CSV 更快（從10行開始，pyarrow 大約快5倍）；
  • 生成的文件較小（約為 CSV 的50％）；
  • 保留有關數據類型的信息（Pyarrow 無法處理較慢的 fastparquet 可以處理的 timedelta）；
  • 在 Hadoop 生態系統中的廣泛支持允許對多個分區進行快速過濾。

parquet 的缺點：

  • 不支持重複的列名；
  • pyarrow 引擎不支持某些數據類型。

Excel和to_excel（）

將數據導出到 excel 中可以更加方便操作。它以最慢的讀寫速度為代價，就是為了更加易於操作。它還忽略了許多數據類型，而且時區根本不能寫入 excel 。

# 將數據框導出到exceldf.to_excel(excel_writer, sheet_name, many_other_parameters)

to_excel 中非常有用的幾個參數：

  • excel_writer -- Pandas 的 excel writer 對象或文件路徑；
  • sheet_name--數據將輸出到的圖紙的名稱；
  • float_format -- Excel 的本地數字格式；
  • 列 -- 別名數據幀列的選項；
  • startrow -- 向下移動起始單元格的選項；
  • engine（引擎） -- openpyxl 或 xlsxwriter；
  • Frozen_panes -- 凍結行和列的選項。

excel的優點：

  • 允許自定義格式和單元格凍結；
  • 可讀和可編輯的格式。

excel的缺點：

  • 讀/寫非常慢（慢20倍/ 40倍）；
  • 限制為 1048576 行；
  • 帶有時區的 datetimes 的序列化失敗。

只有54％的列保留原始數據類型，它佔用了CSV大小的90％，但是寫入時間增加了20倍，讀取時間增加了42倍

HDF5和to_hdf（）

使用適合於大量異構數據的內部文件狀結構的壓縮格式。如果我們需要隨機訪問數據集的各個部分，這也是理想的選擇。如果數據存儲為表（PyTable），則可以使用 store.select（key，where =“ A> 0或B <5”）直接查詢 hdf 存儲。

# 將數據幀導出到hdfdf.to_hdf(path_or_buf, key, mode, complevel, complib, append ...)

to_hdf（）中比較有用的幾個參數：

  • path_or_buf -- 文件路徑或HDFStore對象；
  • key（鍵）-- 已識別或存儲區中的組；
  • mode（模式）-- 寫入，追加或讀取追加；
  • format（格式） -- fixed 用於快速讀寫，而 table 僅允許選擇數據的子集。

HDF5的優點：

  • 對於某些數據結構，其大小和訪問速度都非常驚人。

HDF5的缺點：

  • 數據幀的大小可能非常大（甚至比csv大300倍）；
  • HDFStore 不是線程安全的寫入；
  • 固定格式無法處理分類值。

SQL和to_sql（）

通常，將數據長保存到數據庫中，是非常有必要的。像 sqlalchemy 這樣的庫就是專門用於此任務。

# 設置sqlalchemy引擎engine = create_engine(    'mssql+pyodbc://user:pass@localhost/DB?driver=ODBC+Driver+13+for+SQL+server',    isolation_level="REPEATABLE READ")# 連接到 DBconnection = engine.connect()# 將數據幀導出到SQLdf.to_sql(name="test", con=connection)

to_sql（）中非常有用的幾個參數：

  • name -- SQL表的名稱；
  • con -- 通常由 sqlalchemy.engine 連接的引擎；
  • chunksize -- 可選地以 chunksize 的批量加載數據。

SQL的優點：

  • 比保留在磁盤上慢（讀取10次/寫入5次，但是可以對其進行優化）；
  • 每個程序員都可以理解數據庫。

SQL的缺點：

  • 某些數據格式不保留：類別、整數、浮點數和時間增量；
  • 取決於數據庫性能可能很慢；
  • 在某些情況下，你可能很難建立數據庫連接。

其他方法

Pandas 提供了非常多的方法，在此，我省略了 json 和 fix-width 文件，因為它們與 csv 具有類似的特性。另外，你可以用 .to_gbq（）或 stata 格式直接寫入 Google Big Query。滿足各種雲提供商通信需要的新格式就會出現。

性能測試

很多方法保存的數據格式都比 CSV 好用，但是當 CSV 更容易被理解時，這些不同尋常的方法值得使用嗎？讓我們測試一下性能。

在進行性能測試時，需要重點關注4個關鍵指標：

  • 數據類型保留 -- 讀取後保留原始類型的列百分比；
  • 壓縮/大小 -- 文件佔csv的百分比；
  • 寫入時間 -- 以csv寫入時間的百分比表示，寫入這種格式需要多長時間；
  • 讀取時間 -- 以csv讀取時間的百分比形式讀取此格式需要多長時間。

為此，我準備了一個50K大小，包含隨機數、字符串、類別、日期時間和布爾值的數據集。

data = []for i in range(1000000):    data.append(        [random.randint(-127,127),  # int8         random.randint(-32768,32767),  # int16

完整性檢查

對隨機樣本進行測試，在符合實際的情況下，有助於建立我們的應用程序或工具有多好的第一印象。為了避免意外，你一定要在實際數據的基礎上測試代碼。在這裡，我選擇了我最喜歡的數據集——美國證券交易委員會季度數據轉儲，並對其進行了性能測試。取得了非常相似的結果，證明我的假設並非完全錯誤。

總結

pickle 在性能上處於領先，但是我們還是會根據不同的數據集，選擇不同的數據格式，在實際數據測試中，性能也可能會有所不同。

對我個人而言，.to_pickle（）在存儲預處理的數據集時是非常有用的，因為不需要擔心數據格式，只需要 read_pickle（）即可。
