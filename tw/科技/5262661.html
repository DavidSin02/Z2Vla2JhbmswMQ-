任務管理和時間管理簡介

任務管理

uCOSII系統的主要工作就是對任務進行管理和調度，那麼什麼是任務呢？

生活中我們處理一個大問題的時候通常都是將這個問題“分而治之”，把大問題分成多個

小問題，小問題被逐步的解決掉，大問題也就隨之解決了。那麼這些小問題就可以看成是很多

個小任務。

在我們設計複雜、大型程序的時候也是一樣的，將這些負責的程序分割成許多個簡單的小

程序，這些小程序就是單個的任務，所有的小任務和諧的工作，最終完成複雜的功能。在操作

系統中這些小任務可以併發執行，從而提高CPU的使用效率。

uCOSII就是一個可剝奪的多任務系統，我們使用uCOSII的一個重要的原因就是它的多任務

處理能力。在uCOSII中任務就是程序實體，uCOSII能夠管理和調度這些小任務(程序)。

uCOSII中的任務由三部分組成：任務堆棧、任務控制塊和任務程序代碼（任務函數），如

圖 17.1.1所示：


圖 17.1.1 任務的組成

任務控制塊：用來記錄任務的堆棧指針、當前狀態、優先級別等一些與任務管理有關的屬

性的表就叫做任務控制塊。任務控制塊相當於一個任務的身份證，系統就是通過任務控制塊來

感知和管理任務的，沒有任務控制塊的任務不能被系統承認和管理。

任務堆棧：在任務切換和響應中斷時，保存CPU寄存器中的內容，以及任務調用其他函數

時需要保存的局部變量，每一個任務都有一個獨立的任務堆棧。

任務程序代碼：指任務的執行程序。

任務如何才能切換回上一個任務並且還能接著從上次被中斷的地方開始運行？恢復現場

即可，現場就是CPU的內部各個寄存器。因此在創建一個新任務時，必須把系統啟動這個任務

時所需的CPU各個寄存器初始值事先存放在任務堆棧中。這樣當任務獲得CPU使用權時，就把任

務堆棧的內容複製到CPU的各個寄存器，從而可以任務順利地啟動並運行。

把任務初始數據存放到任務堆棧的工作就叫做任務堆棧的初始化，用戶一般不會直接操作

堆棧初始化函數，任務堆棧初始化函數由任務創建函數OSTaskCreateExt調用。

接下來我們介紹下uCOSII系統的任務狀態。uCOSII的每個任務都是一個死循環，每個任務

都處在以下5種狀態之一的狀態下，這5種狀態是：睡眠狀態、就緒狀態、運行狀態、等待狀態

(等待某一事件發生)和中斷服務狀態。

睡眠狀態（DORMANT）：任務在沒有被配備任務控制塊或被剝奪了任務控制塊時的狀態。

可以通過調用OSTaskDel()函數讓一個任務進入睡眠狀態。也可以通過使用OSTaskCreate()或

OSTaskCreateExt()函數來建立任務，使得任務進入就緒狀態，準備運行。

就緒狀態（READY）：系統為任務配備了任務控制塊且在任務就緒表中進行了就緒登記，

任務已經準備好了，但由於該任務的優先級比正在運行的任務的優先級低，還暫時不能運行，

這時任務的狀態叫做就緒狀態。

運行狀態（RUNNING）：該任務獲得CPU使用權，並正在運行中，此時的任務狀態叫做運行

狀態。

等待狀態（WAITING）：正在運行的任務，需要等待一段時間或需要等待一個事件發生再

運行時，該任務就會把CPU的使用權讓給別的任務而使任務進入等待狀態。

中斷服務狀態（ISR）：一個正在運行的任務一旦響應中斷申請就會中止運行而去執行中

斷服務程序，這時任務的狀態叫做中斷服務狀態。

在多任務系統中，令CPU中止當前正在運行的任務轉而去運行另一個任務的工作叫做任務切換，而按某種規則進行任務切換的工作就叫做任務調度。uCOSII的任務調度策略非常簡單，

它只需要檢查等待列表中各任務的優先級，誰的優先級高就運行誰。

這種簡單的任務調度策略，雖然可以節省嵌入式系統的資源，但同時也會帶來一些問題，

比如說：如果一個高優先級的任務一直佔用CPU，那麼低優先級的任務將一直等待，因此我們

應該合理地分配任務的優先級。

當有多個任務處於就緒狀態時，系統需要在這些任務中選擇一個來運行。就緒任務有多個，

CPU卻有一個，所以CPU需要一個規則來進行選擇。uCOSII採用優先級搶佔規則。系統中的每一

個任務根據其重要性都配有一個唯一的優先級(uCOSII中，每一個優先級只能有一個任務)，優

先級高的任務先得到執行，優先級低的任務後執行。(數值越小優先級越高)。

uCOSII中的任務分為兩種：系統任務和用戶任務；系統任務是指由系統提供併為系統管理

服務的任務；我們一般可以不用管它。用戶任務是指為了解決應用問題由用戶編寫的任務。

uCOSII預定義了兩個系統任務：空閒任務和統計任務。

1、空閒任務：uCOSII必須創建的任務，此任務由uCOSII自動創建，不需要用戶手動創建。

2、統計任務：可選任務，用來統計CPU使用率和各個任務的堆棧使用量。此任務是可選任

務，由宏OS_TASK_STAT_EN控制是否使用此任務。

接下來，我們看看在uCOSII中，與任務相關的幾個函數：

1）建立任務函數（OSTaskCreateExt）

如果想讓uCOSII管理用戶的任務，必須先建立任務。uCOSII給我們提供了2個建立任務的

函數：OSTaskCreate和OSTaskCreateExt。建立任務函數已經在前一章節進行了講解，詳細講

解請查看前一章節的內容。

2）刪除任務函數（OSTaskDel）

所謂的任務刪除，其實就是把任務置於睡眠狀態，並不是把任務代碼給刪除了。uCOSII提

供的任務刪除函數原型為：INT8U OSTaskDel(INT8U prio)，其中參數prio就是我們要刪除的

任務的優先級，可見該函數是通過任務優先級來實現任務刪除的。

特別注意：任務不能隨便刪除，必須在確保被刪除任務的資源被釋放的前提下才能刪除！

3）請求任務刪除函數（OSTaskDelReq）

前面提到，必須確保被刪除任務的資源被釋放的前提下才能將其刪除，所以我們通過向被

刪除任務發送刪除請求，來實現任務釋放自身佔用資源後再刪除。uCOSII提供的請求刪除任務函數原型為：INT8U OSTaskDelReq(INT8U prio)，同樣還是通過優先級來確定被請求刪除任務。

4）改變任務的優先級函數（OSTaskChangePrio）

uCOSII在建立任務時，會分配給任務一個優先級，但是這個優先級並不是一成不變的，而

是可以通過調用uCOSII提供的函數修改。uCOSII提供的任務優先級修改函數原型為：INT8U

OSTaskChangePrio(INT8U oldprio,INT8U newprio)。

5）任務掛起函數

任務掛起和任務刪除有點類似，但是又有區別，任務掛起只是將被掛起任務的就緒標誌刪

除，並做任務掛起記錄，並沒有將任務控制塊任務控制塊鏈表裡面刪除，也不需要釋放其資源，

而任務刪除則必須先釋放被刪除任務的資源，並將被刪除任務的任務控制塊也給刪了。被掛起

的任務，在恢復（解掛）後可以繼續運行。uCOSII提供的任務掛起函數原型為：INT8U

OSTaskSuspend(INT8U prio)。

6）任務恢複函數

有任務掛起函數，就有任務恢複函數，通過該函數將被掛起的任務恢復，讓調度器能夠重

新調度該函數。uCOSII提供的任務恢複函數原型為：INT8U OSTaskResume(INT8U prio)。

更加詳細的任務函數講解，請參考邵貝貝老師的《嵌入式實時操作系統uCOS-II》一書的

第四章。

時間管理

uCOSII中的任務是一個無限循環並且還是一個搶佔式內核，為了使高優先級的任務不至於

獨佔CPU，可以給其他優先級較低任務獲取CPU使用權的機會，uCOSII中除空閒任務外的所有任

務必須在合適的位置調用系統提供的延時函數，讓當前的任務暫停運行一段時間並進行一個任

務切換。

uCOSII規定：除了空閒任務之外的所有任務，必須在任務中合適的位置調用系統提供的任

務延時函數。

延時函數使當前任務的運行延時一段時間並進行一次任務調度，以讓出 CPU的使用權。該

函數是以系統時鐘節拍為基準的，如果系統時鐘節拍延長或縮短了，任務的實際延時時間也跟

著延長或縮短。

在這裡我們主要介紹五個與時鐘節拍相關的系統服務：

1）任務延時函數OSTimeDly()

使用該函數後，任務可以延時一段時間（時間的長短由時鐘節拍數決），也就是該任務會

被掛起，並且開始執行處於就緒態中優先級最高的任務。當延時期滿或者有別的任務使用

OSTimeDlyResume()取消了延時，該任務就會立刻進入就緒狀態。

調用該延時函數時，使用的節拍數處於1到65535之間。當設置的節拍數為0時，表示不延

時；每次時鐘節拍發生的時候，OSTCBDly（任務延時掛起的時間）的值都會被減去一，當該值

為0的時候，內核就會把任務放入就緒隊列。

2）按時分秒延時函數OSTimeDlyHMSM()

使用OSTimeDlyHMSM()這個函數後，就可以以小時(H)、分(M)、秒(S)和毫秒(m)的形式來

延時了，使用起來非常簡單，其工作原理與OSTimeDly()一致，推薦使用OSTimeDlyHMSM()函數。

3）讓延時期的任務結束延時的函數OSTimeDlyResume()

用戶可以不用等待延時結束，通過使用OSTimeDlyResume()和要恢復的任務的優先級，來

取消延時，進入就緒狀態。

4）得到系統時間函數OSTimeGet()

用戶調用OSStart()初始化多任務的時候，會有一個32位的計數器開始累加。隨著時鐘節

拍的發生，計數器累計數到最大值的時候，會從零開始重新計數。使用OSTimeGet()能夠獲得

該計數器的當前值。

5）修改系統時間函數OSTimeSet()

使用OSTimeSet()可以改變上面提到的計數器中的值。

實驗任務

本節通過一個實例，讓大家學習和掌握uCOSII系統的任務管理和時間管理功能。

硬件設計

Qsys系統搭建的步驟以及使用的IP核和“創建第一個uC/OSII系統”章節中完全一致，頂

層模塊代碼除了模塊名不一致外，其餘也完全一致。如果大家有不明白的地方，可以參考“創

建第一個uC/OSII系統”章節實驗，這裡不再贅述。

軟件設計

uCOSII系統的創建可以參考“創建第一個uC/OSII系統”章節中的軟件設計部分。創建完

uCOSII系統之後，只需修改生成的代碼，代碼修改如下：

1 #include <stdio.h>

2 #include "includes.h"

3

4 void time_task(void* pdata);

5 void CPUInfo_task(void* pdata);

6

7 /* Definition of Task Stacks */

8 #define TASK_STACKSIZE 2048

9 OS_STK start_stk[TASK_STACKSIZE];

10 OS_STK time_stk[TASK_STACKSIZE];

11 OS_STK CPUInfo_stk[TASK_STACKSIZE];

12

13 /* Definition of Task Priorities */

14

15 #define START_PRIORITY 100

16 #define TIME_PRIORITY 101

17 #define CPUINFO_PRIORITY 102

18

19 //開始任務

20 void start_task(void* pdata)

21 {

22 OSTaskCreateExt(time_task,

23 NULL,

24 (void *)&time_stk[TASK_STACKSIZE-1],

25 TIME_PRIORITY,

26 TIME_PRIORITY,

27 time_stk,

28 TASK_STACKSIZE,

29 NULL,

30 0);

31

32

33 OSTaskCreateExt(CPUInfo_task,

34 NULL,

35 (void *)&CPUInfo_stk[TASK_STACKSIZE-1],

36 CPUINFO_PRIORITY,

37 CPUINFO_PRIORITY,

38 CPUInfo_stk,

39 TASK_STACKSIZE,

40 NULL,

41 0);

42

43 OSTaskDel(START_PRIORITY);

44 }

45

46 /* 獲取系統時間 */

47 void time_task(void* pdata)

48 {

49 while (1)

50 {

51 printf("The System Time is %d\n",OSTimeGet());

52 OSTimeDlyHMSM(0, 0, 0, 500);

53 }

54 }

55 /* 獲取CPU使用率 */

56 void CPUInfo_task(void* pdata)

57 {

58 while (1)

59 {

60 printf("The CPU Usage is %d\n",OSCPUUsage);

61 OSTimeDlyHMSM(0, 0, 1, 0);

62 }

63 }

64 /* 創建開始任務 */

65 int main(void)

66 {

67 OSTaskCreateExt(start_task,

68 NULL,

69 (void *)&start_stk[TASK_STACKSIZE-1],

70 START_PRIORITY,

71 START_PRIORITY,

72 start_stk,

73 TASK_STACKSIZE,

74 NULL,

75 0);

76

77 OSStart();

78 return 0;

79 }

首先main函數創建了一個開始任務（start_task），然後啟動uCOSII系統。我們在前面說

過，任務是一個無限的循環，由於開始任務只需要運行一次，因此開始任務不需要寫成while(1)

的形式。

除了在main函數創建任務外，在任務中也可以創建任務，如代碼的第19行至第44行所示。

開始任務（start_task）實現的功能是創建了獲取系統時間任務（time_task）和獲取CPU使用

率任務（CPUInfo_task）。獲取系統時間的函數為OSTimeGet，獲取CPU使用率的函數為

OSCPUUsage，並通過printf函數打印出來；OSTimeDlyHMSM函數用於延時，並交出CPU的使用權，

以便於其它準備就緒的任務獲取CPU的使用權。
