mysql explain 詳解：

使用：explain select * from table_name；可以用 desc 代替 explain；

  • id 列

若沒有子查詢和聯合查詢，id則都是1。

Mysql會按照id從大到小的順序執行query，在id相同的情況下，則從上到下執行。

  • select_type 列

（1）SIMPLE：

簡單的SELECT語句（不包括UNION操作或子查詢操作）

explain select * from auct_item where item_id=9920140904244;

（2）PRIMARY/UNION：

PRIMARY：查詢中最外層的SELECT（如兩表做UNION或者存在子查詢的外層的表操作為PRIMARY，內層的操作為UNION）

UNION：UNION操作中，查詢中處於內層的SELECT（內層的SELECT語句與外層的SELECT語句沒有依賴關係）

explain select * from auct_item where item_id=(select MAX(item_id) from auct_item);

（3）DEPENDENT UNION/UNIOIN RESULT：

DEPENDENT UNION：UNION操作中，查詢中處於內層的SELECT（內層的SELECT語句與外層的SELECT語句有依賴關係）

UNION RESULT：UNION操作的結果，id值通常為NULL

explain select * from auct_item a where item_id=9920140904244 UNION select * from auct_item b where item_id=9920140904242;

（4）SUBQUERY/DEPENDENT SUBQUERY：

SUBQUERY：子查詢中首個SELECT（如果有多個子查詢存在），示例參考（2）

DEPENDENT SUBQUERY：子查詢中首個SELECT，但依賴於外層的表（如果有多個子查詢存在）

explain select * from auct_item a where item_id=(select MAX(item_id) from auct_item b where a.item_id=b.item_id);

（5）DERIVED/MATERIALIZED：

DERIVED：被驅動的SELECT子查詢（子查詢位於FROM子句）

explain select * from (select * from auct_item where item_id>0) a where item_id=(select MAX(item_id) from auct_item b where a.item_id=b.item_id);

MATERIALIZED：被物化的子查詢

（6）UNCACHEABLE SUBQUERY/UNCACHEABLE UNION：

UNCACHEABLE SUBQUERY：對於外層的主表，子查詢不可被物化，每次都需要計算（耗時操作）

UNCACHEABLE UNION：UNION操作中，內層的不可被物化的子查詢（類似於UNCACHEABLE SUBQUERY）

  • table 列

輸出的行所引用的表。

有時看到的是<derivedN>，其中N對應的是id列的值。

  • type 列

這列很重要，顯示了連接使用了哪種類別，有無使用索引。

包括（由左至右，由最差到最好）：

| All | index | range | ref | eq_ref | const,system | null |

（1）ALL

全表掃描，MySQL 從頭到尾掃描整張表查找行。

explain select item_id,item_name from auct_item ;

（2）index

該聯接類型與ALL相同，除了只有索引樹被掃描。這通常比ALL快，因為索引文件通常比數據文件小。　當查詢只使用作為單索引一部分的列時，MySQL可以使用該聯接類型。

explain select item_id from auct_item ;（item_id為主鍵）

（3）range

只檢索給定範圍的行，使用一個索引來選擇行。key列顯示使用了哪個索引。key_len包含所使用索引的最長關鍵元素。在該類型中ref列為NULL。　當使用=、<>、>、>=、<、<=、IS NULL、<=>、BETWEEN或者IN操作符，用常量比較關鍵字列時，可以使用range。

explain select item_id from auct_item where item_id in (123,1234);

（4）ref

非唯一性索引訪問(換句話說，就是無法根據該值只取得一條記錄)

explain select * from auct_item_record where item_id=9920140904242;

（5）eq_ref

對於每個來自於前面的表的行組合，從該表中讀取一行。這可能是最好的聯接類型，除了const類型。它用在一個索引的所有部分被聯接使用並且索引是UNIQUE或PRIMARY KEY。　eq_ref可以用於使用= 操作符比較的帶索引的列。比較值可以為常量或一個使用在該表前面所讀取的表的列的表達式。

explain select a.AUCT_ID,a.ITEM_ID,b.ITEM_NAME,a.USER_ID from auct_item_record a,auct_item b where a.USER_ID=6000063536 and a.ITEM_ID=b.ITEM_ID;

（6）const

表最多有一個匹配行，它將在查詢開始時被讀取。因為僅有一行，在這行的列值可被優化器剩餘部分認為是常數。const表很快，因為它們只讀取一次！　const用於用常數值比較PRIMARY KEY或UNIQUE索引的所有部分時。

explain select * from auct_item_record where record_id=4448;

（7）system

這是const聯接類型的一個特例。表僅有一行滿足條件.如下(id是primary key)

EXPLAIN SELECT * FROM (SELECT * FROM auct_item_record WHERE record_id=4448) a;

（8）null

在優化過程中就已得到結果，不用再訪問表或索引。

explain select min(record_id) from auct_item_record;

  • possible_keys 列

possible_keys列指出MySQL能使用哪個索引在該表中找到行。

注意，該列完全獨立於EXPLAIN輸出所示的表的次序。這意味著在possible_keys中的某些鍵實際上不能按生成的表次序使用。　如果該列是NULL，則沒有相關的索引。

  • key 列

key列顯示MySQL實際決定使用的鍵（索引）。

如果沒有選擇索引，鍵是NULL。要想強制MySQL使用或忽視possible_keys列中的索引，在查詢中使用FORCE INDEX、USE INDEX或者IGNORE INDEX。

  • key_len 列

key_len列顯示MySQL決定使用的鍵長度。

如果鍵是NULL，則長度為NULL。　使用的索引的長度。在不損失精確性的情況下，長度越短越好。

  • ref 列

ref列顯示使用哪個列或常數與key一起從表中選擇行。

常見的值有 const, func, NULL, 具體字段名。當 key 列為 NULL ，即不使用索引時，此值也相應的為 NULL 。

ref 為 func 的情況出現在子查詢中:

explain select * from auct_item_record where record_id in (select record_id from auct_item_record);

  • rows 列

rows列顯示MySQL認為它執行查詢時必須檢查的行數。

  • Extra 列

顯示上述信息之外的其它信息，但卻很重要。

常見的值有Using index、Using where、Using temporary、Using filesort。

（1）Using index

此查詢使用了覆蓋索引(Covering Index)，即通過索引就能返回結果，無需訪問表。

explain select record_id from auct_item_record;

（2）Using where

表示 MySQL 服務器從存儲引擎收到行後再進行“後過濾”（Post-filter）。所謂“後過濾”，就是先讀取整行數據，再檢查此行是否符合 where 句的條件，符合就留下，不符合便丟棄。因為檢查是在讀取行後才進行的，所以稱為“後過濾”。並非帶 WHERE 子句就會顯示"Using where"的。

explain select * from auct_item_record WHERE USER_ID=6000063536;

（3）Using temporary

看到這個的時候，查詢需要優化了。這裡，MYSQL需要創建一個臨時表來存儲結果，這通常發生在對不同的列集進行ORDER BY上，而不是GROUP BY上。

explain select distinct NICKNAME from auct_item_record;

MySQL 使用臨時表來實現 distinct 操作。

（4）Using filesort

看到這個的時候，查詢就需要優化了。

MYSQL需要進行額外的步驟來發現如何對返回的行排序。它根據連接類型以及存儲排序鍵值和匹配條件的全部行的行指針來排序全部行。

  1. 對於沒有索引的表，只要 order by 必會出現 Using filesort 。
  2. 當 type 為 rang、 ref 或者 index 的時候才有可能利用索引排序，其它，如 ALL ，都無法通過索引排序，此時若有 order by ，如上例，便會出現 Using filesort 。

explain select * from auct_item_record order by AUCT_ID DESC;
