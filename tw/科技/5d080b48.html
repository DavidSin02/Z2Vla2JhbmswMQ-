Freertos printf可重入的問題

2020-11-14
| 科技

調試程序時候常用printf來作為調試日誌輸出，方便定位問題所在位置。但最近出現一個現象程序運行過程中會出現莫名的異常死機。

但是在死之前有個徵兆，觀看打印的日誌，有兩條信息穿插打印的現象。所以才想起來去考慮是不是printf引起的。

在整個項目中我運行的有8個任務，而在每一個任務裡面幾乎都會有使用printf進行調試信息打印。所以從開始就埋下了一顆定時炸彈。只是要碰下運行，運氣好的時候一切風平浪靜。運氣不好的時候就炸的異常到處橫飛。

在看下自己代碼printf重映射的fput接口用的還是HAL_UART_Transmit 阻塞發送函數接口。這樣在波特率為115200的情況下，假如打印一條200bytes的日誌信息就要約等於19ms的時間。這豈不嚴重影響其他任務的實時性。

所以先把fput改為非阻塞的形式，這個可以使用freertos的stream_buffer,這個可以把要發送的數據壓入buffer，然後真正的發送就使用串口發送為空中斷從stream_buffer一個個byte進行讀取發送。這樣可以保證調用printf——》到fput 就只是寫stream_buffer。而不再是阻塞式調用一次printf就要等待串口完全把數據發送完。fput的代碼如下：

int fputc(int ch,FILE *file){ xStreamBufferSend(streamDebug,&ch,1,1000); __HAL_UART_ENABLE_IT(&DEBUG_UART,UART_FLAG_TXE);  return ch;}

本以為把阻塞改為非阻塞，這個bug解決完的美妙心情就要到來了的時候，按下了復位按鍵，啥？運行了還沒10秒鐘就又死了，甚至比之前更慘烈。

而現在調用printf的最關鍵的一條代碼就是xStreamBufferSend() 只能默默的打開freertos的參考手冊，看下stream的詳細描述。而在注意裡面有下面一段話：

有多個不同的writers或者多個不同的readers是不安全的。也就是說在多個不同的任務中調用發送就會有可能有問題。這不正符合了我現在的場景，多個任務可能同時調用printf，相當於多個任務同時調用stream的write接口。

stream有這個限制，那註定printf就是不可重入的了。而針對不可重入的資源最簡單的辦法就是用互斥量進行保護。所以我就把printf進行包裝了一層取名為debug_printf:

void debug_printf(const char *fmt, ...){  va_list args;  if(xSemaphoreTake(semDebug,portMAX_DELAY); va_start(args,fmt);  vprintf(fmt,args);  va_end(args);  xSemaphoreGive(semDebug); }

當調用debug_printf的時候先去獲取信號量semDebug，如果獲取不到就一直等待直到獲取。該函數執行完畢以後再釋放信號量。這樣就充分保證了資源的獨佔，同時只可能有一個任務再進行debug_printf。需要說明的是vprintf底層最終還是會調用我們重映射的fput函數。

再次上電，打印終於正常了，但是，還有但是……一波未平一波又起。開鎖的時候失敗了，開鎖任務是一個需要實時的任務，要檢測傳感器轉動位置。而這個異常明顯是實時性得不到滿足。

仔細分析下，問題就出在printf互斥量的獲取。有可能造成高優先級任務產生延時，甚至造成優先級翻轉。

假如有三個任務：任務A、任務B、任務C，優先級從低到高。現在任務A調用了debug_printf 獲取了信號量，而高優先級的任務C也需要調用debug_printf,但因為獲取不到信號量就會造成暫時掛起，等到任務A打印完成把信號量釋放了任務C才能得以運行。這樣高優先級的任務無法搶佔低優先級任務而降低了實時性。如下圖所示：

上圖的是一種情況，還不算最壞的。還有可能出現優先級翻轉的問題，造成延時更大。同樣最低優先級任務A先運行並且調用了debug_printf獲取到了信號量，而這時候任務c也需要運行並調用了debug_printf。但是因為A已經把信號量佔用而無法獲取到，所以暫時被掛起。任務A開始打印，在打印到一半的時候，被中等優先級的任務B給打斷了（任務B不調用debug_printf所以可以打斷A）。這時候A又掛起了，任務B開始運行，等到B運行完以後繼續運行A。等A運行完信號量釋放了，高優先任務C終於可以得到運行了。如下圖所示，這時候看看任務C本來可以實時的運行的，現在卻中間又穿插先運行了中優先級任務B。造成一箇中優先級比高優先級任務先得到運行的優先級翻轉現象。而實時性相比上圖就更差了。

並且這個延時是不可估量的，想想任務B還可能再被更高優先級的任務打斷。高優先級任務C反而被一再延時執行。所以出現高優先級任務出現問題就不難理解了。

發現了問題還是要解決問題。根據分析肯定要改造一下debug_printf，同時為了實時性得到滿足。我就把任務C高優級的打印LOG暫時關閉。

而針對debug_printf中獲取信號量修改為有時間限制，不再無限期等待。這樣頂多出現獲取不到信號量5ms超時返回少打印一條log信息，總比把任務卡出異常要好的多。改造後的debug_printf如下：

void debug_printf(const char *fmt, ...){  va_list args;  if(xSemaphoreTake(semDebug,5)==pdFALSE)return; va_start(args,fmt);  vprintf(fmt,args);  va_end(args);  xSemaphoreGive(semDebug); }
