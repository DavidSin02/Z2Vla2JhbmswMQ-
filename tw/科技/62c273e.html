SpringBoot配置文件詳解

2020-10-29
| 科技

前言

約定 > 配置 > 編碼

Spring Boot 對於開發人員最大的好處在於可以對 Spring 應用進行自動配置。在SpringBoot項目中，很多時候需要用到一些配置的信息，這些信息可能在測試環境和生產環境下會有不同的配置，針對這種情況，項目所需屬性就需要靈活設置，儘可能地不要將這些配置在代碼中寫死，最好就是寫到配置文件中，以便後面根據實際業務需求靈活修正。

1、自動化配置

Spring Boot會根據應用中聲明的第三方依賴來自動配置 Spring 框架，而不需要進行顯式的聲明。比如當聲明瞭對 HSQLDB 的依賴時，Spring Boot 會自動配置成使用 HSQLDB 進行數據庫操作。

Spring Boot 推薦採用基於Java 註解的配置方式，而不是傳統的 XML。只需在主配置Java類引@EnableAutoConfiguration”註解就可以啟用自動配置。當然，根據目前SpringBoot版本的迭代更新，在多數情況下，我們只需注入@SpringBootApplication這個註解即可替代。

而且，Spring Boot 的自動配置功能是沒有侵入性的，只是作為一種基本的默認實現。開發人員可以通過定義其他bean來替代自動配置所提供的功能。比如當應用中定義了自己的數據源bean時，自動配置所提供的 HSQLDB 就不會生效。這給予了開發人員很大的靈活性。既可以快速的創建一個可以立即運行的原型應用，又可以不斷的修改和調整以適應應用開發在不同階段的需要。可能在應用最開始的時候，嵌入式的內存數據庫（如 HSQLDB）就足夠了，在後期則需要換成MySQL數據庫。Spring Boot 使得這樣的切換變得很簡單。

2、外部化配置

在應用中管理配置並不是一個容易的任務，尤其是在應用需要部署到多個環境中時。通常會需要為每個環境提供一個對應的屬性文件，用來配置各自的數據庫連接信息、服務器信息和第三方服務賬號等。

通常的應用部署會包含開發、測試和生產等若干個環境，不同的環境之間的配置存在覆蓋關係。測試環境中的配置會覆蓋開發環境，而生產環境中的配置會覆蓋測試環境。Spring 框架本身提供了多種的方式來管理配置屬性文件，Spring 3.1 之前可以使用 PropertyPlaceholderConfigurer。Spring 3.1 引入了新的環境（Environment）和概要信息（Profile）API，是一種更加靈活的處理不同環境和配置文件的方式。不過 Spring 這些配置管理方式的問題在於選擇太多，讓開發人員無所適從。Spring Boot 提供了一種統一的方式來管理應用的配置，允許開發人員使用屬性文件、YAML 文件、環境變量和命令行參數來定義優先級不同的配置值。
Spring Boot 所提供的配置優先級順序比較複雜。按照優先級從高到低的順序，具體的列表如下所示。

命令行參數。

  • 通過 System.getProperties() 獲取的 Java 系統參數。
  • 操作系統環境變量。
  • 從 java:comp/env 得到的 JNDI 屬性。
  • 通過 RandomValuePropertySource 生成的“random.*”屬性。
  • 應用 Jar 文件之外的屬性文件。
  • 應用 Jar 文件內部的屬性文件。
  • 在應用配置 Java 類（包含“@Configuration”註解的 Java 類）中通過“@PropertySource”註解聲明的屬性文件。
  • 通過“SpringApplication.setDefaultProperties”聲明的默認屬性。

Spring Boot 的這個配置優先級看似複雜，其實是很合理的。比如命令行參數的優先級被設置為最高。這樣的好處是可以在測試或生產環境中快速地修改配置參數值，而不需要重新打包和部署應用。

2.1 配置文件

舉個栗子：

搭建一個最基礎的SpringBoot工程。

如上邊所示，Spring Boot工程的基礎結構包括四個部分(具體路徑根據用戶生成項目時填寫的GroupId和ArtifactId有所差異)。

  • src/main/java：存放主要編碼的區域。如上，SpringBootMavenApplication.java這個類就是工程的主程序入口類，可以通過直接運行該類來啟動Spring Boot應用。
  • src/main/reources：存放默認配置的目錄。 該目錄存放應用級別配置文件----application.yml文件(看個人喜好，這個文件作用等同於application.peoperties文件，唯一區別在於語法格式的不同)，用來存放應用的一些配置信息， 如應用名、服務端口、數據庫鏈接等。如果在搭建工程初期就引入Web模塊，maven工程會自動產生static目錄與templates目錄，前者用於存放靜態資源， 如圖片、css、JavaScript等；後者存放Web頁面的模板文件， 可根據實際需要靈活產生或刪除這兩個目錄。
  • src/test：單元測試目錄。 生成的測試類通過JUnit 4實現， 可以直接用運行Spring Boot應用的測試。
  • pom.xml：SpringBoot項目級別的核心配置文件。這個pom.xml主要描述了項目的maven座標，依賴關係，屬性配置，開發者需要遵循的規則，缺陷管理系統，組織和licenses，maven插件配置以及其他所有項目相關因素。

Spring Boot的配置文件除了可以使用傳統的 properties文件之外，還支持現在被廣泛推薦使用的YAML文件。Spring Boot 的默認配置文件位置為：src/main/reources/application.yml。SpringBoot應用的配置內容都可以集中在該文件中， 根據pom.xml文件中引入的不同Starter模塊，可以在application.yml文件內定義容器端口號、 數據庫連接信息、 日誌級別、MyBatis、redis等等各種配置信息。

比如， 我們需要自定義Web模塊的服務端口號，可以在application.yml中添加如下內容來指定SpringBoot項目的應用配置信息。#application-level config

#profiles activeserver:         port: 8080#log configlogging:         config: /HL/IdeaProjects/SpringBoot-Item/springboot-slf4j/src/main/resources/logback.xml  level:           com.hl.magic: error

關於YML文件定義，這裡簡單總結一下。

YAML (英語發音為l'j知回／，， 尾音類似camel駱駝）是 一個可讀性高， 用來表達資籵序列的格式。YAML參考了其他多種語言， 包括C語言、Python、Perl,並從XML、 電子郵件的數據格式(RFC 2822)中獲得靈感。 Clark Evans在2001年首次發表了這種語言，Ingy dot Net與OrenBen寸q如也是這種語言的共同設計者。 目前已經有數種編程語言或腳本語言支持（或者說解析）這種語言。

YAML是 YAML 扣n't a Markup Lan伊1age (YAML 不是一種標記語言）的縮寫。在開發這種語言時， YAML的意思其實是： Yet Another Markup Language (仍是一種標記語言），但為了強調這種語言以數據作為中心，而不是以標記語言為重點，而用反向縮略語重新命名。

YAML的語法和其他高階語言類似， 並且可以簡單地表達清單、 散列表、 標量等形態。它使用空白符號縮排和大量依賴外觀的特色， 特別適合用來表達或編輯數據結構、各種設定文檔、 文件大綱（例如， 許多電子郵件標題格式和 YAML 非常接近）。儘管它比較適合表達階層式 (hierarchical model)的數據結構， 不過也有精緻的語法可以表示關聯性 (relational model)的資籵。 由於YAML 使用空白符號和分行未分隔資籵， 使得它特別適合用 grep / Python / Perl /Ruby操作。其讓人最容易上手的特色是巧妙避開各種封閉符號，如引號、各種括號等， 這些符號在巢狀結構時會變得複雜而難以辨認。——————維基百科

YAML 採用的配置格式不像 properties 的配置那樣以單純的鍵值對形式來表示，而是以類似大綱的縮進形式來表示。 下面是一段 YAML 配置信息：spring:

  profiles:           active: dev#MVC  mvc:           servlet:                    load-on-startup: 1

上邊的配置信息在application.properties文件中等同配置如下：

spring.profiles.active=dev

spring.mvc.servlet.load-on-startup=1

通過 YAML 的配置方式我們可以看到，配置信息利用階梯化縮進的方式， 其結構更為清晰易讀， 同時配置內容的字符量也得到顯著減少。

除此之外，YAML 還可以在一個單個文件中通過使用 spring.profiles 屬性來定義多個不同的環境配置。 例如下面的內容，在指定為test環境時，server.port 將使用8081端口；在 prod 環境中，server.port將使用8082端口；而在 dev環境中，server.port將使用8083端口；如果沒有指定環境，server.port將使用默認的8080端口。

server:         port: 8080---spring:         profilea: testserver:         port: 8081---spring:         profilea: prodserver:         port: 8082---spring:         profilea: devserver:         port: 80832.2

2.2 自定義參數

除了可以在 Spring Boot 的配置文件中設置各個Starter 模塊中預定義的配置屬性， 也可以在配置文件中定義一些我們需要的自定義屬性。 針對application.yml文件和 application.properties文件中自定義參數，栗子。

【1】在 application.yml文件中定義如下：

book:         name: Spring Boot In Action         author: xiaoming

【2】在application.properties文件中定義如下：

book.name=Spring Boot In Actionbook.author=xiaoming

上邊兩種文件內的自定義參數，實際上是等效的。

接下來，我們在應用中可以通過@Value註解來加載這些自定義的參數，如：

@Componentpublic class Book {    @Value("${book.name}")    private String name;    @Value("${book.author}")    private String author;    /**省略getter和setter方法*/}

@Value 註解加載屬性值的時候可以支持兩種表達式來進行配置， 如下所示。

• 一種是上面介紹的 PlaceHolder 方式，格式為＄{．．．}，大括號內為 PlaceHolder。

• 另一種是使用SpEL 表達式 (Spring Expression Language), 格式為#{．．．}， 大括號內為 SpEL 表達式。

2.3 參數引用

對於參數引用的方式， 其主要區別在於application.yml文件和 application.properties文件中對於參數定義及引用方式。

【1】 application.yml文件

在 application.yml中的各個參數之間可以直接通過使用 PlaceHolder 的方式來進行引用 ， 就像下面的設置：

book:         name: Spring Boot In Action         author: xiaoming         desc: ${book.name} is writing 《${book.name}》

book.desc 參數引用了上邊中定義的book.name和book.author 屬性， 最後該屬性的值就是xiaoming is writing《Spring Boot In Action》。

【2】 application.properties文件

在 application.yml中的各個參數之間可以直接通過使用 PlaceHolder 的方式來進行引用 ， 就像下面的設置：

book.name=Spring Boot In Actionbook.author=xiaoming

book.des=${book.name} is writing 《${book.name}》 book.desc 參數引用了上邊中定義的book.name和book.author 屬性， 最後該屬性的值就是xiaoming is writing《Spring Boot In Action》。

2.4 使用隨機數

在一些特殊情況下， 我們希望有些參數每次被加載的時候不 一個固定的值， 比如密鑰、 服務端口等。 在 SpringBoot的屬性配置文件中， 可以 通過 使用${random}配置來產生隨機的int值、long值或者string字符串，這樣我們就可以容易地通過配置隨機生成屬性，而不是在程序中通過編碼來實現這些邏輯。

【1】 application.yml文件

${random}的配置方式主要有以下幾種， 讀者可作為參考使用。

＃隨機字符串com：         didispace：                  blog：                           value: ${random.value} ＃隨機int                           number: ${random.int} ＃隨機long                           bignumber: ${random.long} # 10以內的隨機數                           test1: ${random.int(l0)}# 10-20的隨機數                           test2: ${random.int[l0,20]}

【2】 application.properties文件${random}的配置方式主要有以下幾種， 讀者可作為參考使用。

＃隨機字符串com.didispace.blog.value=${random.value} ＃隨機intcom.didispace.blog.number=${random.int} ＃隨機longcom.didispace.blog.bignumber=${random.long} # 10以內的隨機數com.didispace.blog.tes七 1=${random.int(l0)}# 10-20的隨機數com.didispace.blog.test2=${random.int[l0,20]}

2.5 命令行參數

啟動SpringBoot應用， 可以使用命令java -jar 來啟動的方式。 當然，該命令除了啟動應用之外， 還可以在命令行中指定應用的參數， 比如java -jar xxx.jar --server.port = 8080，直接以命令行的方式來設置server.port屬性， 並將啟動應用的端口設為8080。

在用命令行方式啟 動 Spring Boot 應用時， 連續的兩個減號“- -”就是對application.properties中的屬性值進行賦值的標識。 所以 ， java -jar xxx.jar--server.port = 8080命令， 等價於在 application.properies中添加屬性server.port = 8080。

通過命令行來修改屬性值是 SpringBoot非常重要的一個特性。 通過此特性， 理論上已經使得應用的屬性在啟動前是可變的， 所以其中的端口號也好、 數據庫連接也好， 都是可以在應用啟動時發生改變的， 而不同於以往的Spring應用通過Maven的Profile在編譯器中進行不同環境的構建。

SpringBoot的這種方式， 可以讓應用程序的打包內容貫穿開發、測試以及線上部署， 而Maven不同Profile的方案為每個環境所構建的包，其內容本質上是不同的。 但是， 如果 每個參數都需要通過命令行來指定， 這顯然也不是 一個好的方案， 所以下面我們看看如何在SpringBoot中實現多環境的配置。
