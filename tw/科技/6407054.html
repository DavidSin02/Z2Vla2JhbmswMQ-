適配器模式的應用

已知某公司現有一員工管理系統，我們現在從改系統的員工個人信息獲取和薪金獲取兩個模塊進行分析，從而解釋適配器的兩種模式。先來看看系統的員工信息獲取的類圖：

員工個人信息分為個人基本信息、員工家庭信息、員工公司信息三部分，其源碼如下：

而另一個系統需要當前系統中的員工信息，而另一個系統中針對員工信息並沒有這麼細緻的分類，而是統稱為員工信息，那麼該如何很好的實現兩個系統的員工信息展示的對接呢，下面我們將利用適配器的對象適配器模式成功實現兩個系統的協調工作。首先我們看一下新系統的員工信息類圖:

先不說系統設計的好壞，反正系統已經投入運營了，總得實現對接吧，接下來看看新系統的獲取員工信息是如何調用原有系統的獲取員工信息方法的。

具體的的代碼如下：

public interface IUserInfo { public String getUserName(int userID); public String getUserHomeAddress(int userID); public String getMobileNumber(int userID); public String getOfficeTelNumber(int userID); public String getJobPosition(int userID); public String getHomeNumber(int userID);}public class UserInfo implements IUserInfo{ private UserBaseInfo userBaseInfo = new UserBaseInfo(); private UserHomeInfo userHomeInfo = new UserHomeInfo(); private UserOfficeInfo userOfficeInfo = new UserOfficeInfo(); @Override public String getUserName(int userID) { // TODO Auto-generated method stub //0存儲員工姓名，1存儲員工手機號碼 ArrayList<String> userInfo = userBaseInfo.getUserBaseInfo(userID); if(userInfo != null) return userInfo.get(0); return null; } @Override public String getUserHomeAddress(int userID) { // TODO Auto-generated method stub //0存儲用戶家庭住址,1存儲用戶家庭電話 ArrayList<String> userInfo = userHomeInfo.getUserHomeInfo(userID); if(userInfo != null) return userInfo.get(0); return null; } @Override public String getMobileNumber(int userID) { // TODO Auto-generated method stub //0存儲員工姓名，1存儲員工手機號碼 ArrayList<String> userInfo = userBaseInfo.getUserBaseInfo(userID); if(userInfo != null) return userInfo.get(1); return null; } @Override public String getOfficeTelNumber(int userID) { // TODO Auto-generated method stub //0存儲公司職位，1存儲公司電話 ArrayList<String> userInfo = userOfficeInfo.getUserOfficeInfo(userID); if(userInfo != null){ return userInfo.get(1); } return null; } @Override public String getJobPosition(int userID) { // TODO Auto-generated method stub //0存儲公司職位，1存儲公司電話 ArrayList<String> userInfo = userOfficeInfo.getUserOfficeInfo(userID); if(userInfo != null){ return userInfo.get(0); } return null; } @Override public String getHomeNumber(int userID) { // TODO Auto-generated method stub //0存儲用戶家庭住址,1存儲用戶家庭電話 ArrayList<String> userInfo = userHomeInfo.getUserHomeInfo(userID); if(userInfo != null) return userInfo.get(1); return null; }}

僅僅知道個人信息還不滿足，又有一個需求是知道每一個人的月薪，這樣新系統好發工資啊。那好吧，還是先看看原有的系統是如何實現獲取工資的，發現原有的工資的展示是以數組的形式返回的，包括基本工資、獎金、出差費，而新系統我才不管你有什麼費用的，我只需要知道一個總的工資，我好發錢就行。那麼此時可以用類適配器，其類圖如下：

public interface IUserSalary { public int getSalary(int userID);}public class AdapterUserSalary extends UserSalary implements IUserSalary{ @Override public int getSalary(int userID) { // TODO Auto-generated method stub HashMap<String, Integer> salary = (HashMap<String, Integer>) super.getUserSalary(userID); int userSalary = 0; Iterator iter = salary.entrySet().iterator(); while(iter.hasNext()){ Map.Entry entry = (Entry) iter.next(); userSalary += (int)entry.getValue(); } return userSalary; }}

適配器模式的優點

  • 適配器模式可以讓兩個沒有任何關係的類運行在一起。
  • 增加了類的透明性，即高層模塊只需知道最後的薪金，並不需要知道薪金具體由哪幾部分組成，以及如何獲取都不關心。
  • 提高了類的複用性，源角色在原有系統中還是可以正常使用，而且可以再新的系統中充當新的角色。
  • 靈活性好，突然之間不需要做適配了直接去掉適配器就可以了。
