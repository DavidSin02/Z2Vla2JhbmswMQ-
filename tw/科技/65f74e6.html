虛擬內存系列(一)硬件基本結構

2020-10-29
| 科技

0

最近花了很多時間在虛擬內存的研究，發現自己對虛擬內存的理解只是停留在頁，頁表，頁框等等這些基本概念上。舉例來說，我一直認為虛擬地址的分段數決定了查表次數。其實仔細想想，哪怕是一級頁表映射，也必然有兩個地址分段。本章節，我將對ARM的虛擬內存的映射部分做一個系統性的總結。

PAGE的故事

ARM提供了一級和兩級地址映射。總的來說，可以分為兩類：

  • Section 方式映射
  • PAGE 方式映射

頁大小可以為1MB(sections), 64KB, 4KB, 1KB。在linux中常用的是4KB的頁大小，它是由宏PAGE_SHIFT來決定的。有一點需要注意，如果PAGE大小為1M時，使用的是section方式映射，此時只有一級頁表(PGD)。 那麼ARM具體是知道使用哪種方式呢，這是由PGD descriptor中的最後2bit來決定。

  • 00 : 未定義的
  • 01 : Coarse page table, 可使用PAGE大小: 64K, 16K
  • 10 : Section, 可使用PAGE大小: 1M
  • 11 : Fine page table: 可使用PAGE大小: 64K, 16K, 4K

ARM 地址映射過程

下面這張圖展示了ARM使用不同方式映射的地址查找過程。

程序首先會通過TTBR寄存器找到一級映射表的入口，取虛擬地址的[31:20] bit作為這張表的Index，找到對應的PGD descriptor。取PGD中的二級頁表物理地址，將虛擬地址的[19:12/10]作為二級頁表的索引，找到PTE descriptor。 PTE記錄了頁的物理地址(4K/1K對齊)。

虛擬地址中的[31:20]bit是作為TTBR的Index，也就是Translation table base 後的[13:0]bit 輸入，拼成一個32bit的地址，最後定位到PGD descriptor的位置。可以看到一級頁表是16KB對齊的。這是由於ARM為一級頁表提供了4096個entry(使用虛擬地址的高12bit用來做Index)，每個entry為4字節，總共16K字節。

二級頁表基址本結構如下，對應了三種映射方式。記錄了下一級頁/頁表的基址以及Domain和頁權限(section)，二級頁表基址是一級頁表的context。

Section base address記錄了Section方式的頁物理地址。對於其他兩種映射方式，PGD所記錄的是二級頁表的base地址。可以看到，二級頁表基址(我們以Coarse page table base為例)它記錄的地址是[31:10]bit，而[9:0]是沒有的。那是不是意味著這種方式的二級頁表是1KB對齊的呢？確實是這樣的。因為ARM中為Coarese page table提供的了256個entry，按每個entry4字節計算，每張表佔用1KB內存。所以，對於Coarese page table base而言，它一定是1KB對齊的。對於Fine page table，ARM提供了1024個entry，所以它是4K對齊。

我在看這部分內容時，特別容易混淆地址、地址上存的內容這兩個概念。習慣性地用頁表的base address 加上虛擬內存中的PAGE_OFFSET來計算一個地址。例如Coarse page table baseaddress是[31:10]共22bit，然後取虛擬地址中的[9:0]這10bit組成一個地址。當然，這是一種錯誤的計算方法。

小結

1，對於映射部分的理解，不要侷限於4K page。

2，頁權限控制bit的具體位置。

3，理解一級、二級頁表的基址(base address的來源)，以及頁表基址是上一級頁表內容換算。

4，虛擬內存分段後的各bit段用作頁表/頁的index(這比理解為物理地址更確切一點)。
