11.5.1媒體管理系統架構

IEEE 1244標準描述了客戶端/服務器架構（圖11.9）。諸如網絡備份系統之類的應用程序承擔使用可移動媒體管理系統的服務的客戶端的角色。該標準定義了所需的服務器組件，數據模型以及這些組件與應用程序之間的接口。

詳細地，IEEE 1244引入了以下組件。媒體管理器充當所有元數據的中央存儲庫，並提供控制和協調媒體，庫和驅動器使用的機制。庫管理器代表媒體管理器控制庫硬件，並將庫的屬性和內容傳輸到媒體管理器。驅動器管理器代表媒體管理器管理驅動器硬件，並將驅動器的屬性傳輸到媒體管理器。

IEEE 1244數據模型指定了為可移動介質建模管理系統所需的所有對象及其各自的屬性（表11.1）。 IEEE 1244規定通過在運行時添加特定於應用程序的屬性來擴展對象。這使得對象模型能夠動態且靈活地適應相應的任務，而無需諸如媒體管理器或協議的組件被改變。

此外，該標準定義了與這些組件通信的接口。應用程序通過媒體管理協議（MMP）與媒體管理器聯繫，以使用符合IEEE 1244標準的管理系統的服務和資源來實現可移動媒體。庫管理協議（LMP）定義媒體管理器和庫管理器之間的接口，驅動器管理協議（DMP）定義媒體管理器和驅動器管理器之間的接口。

圖11.9用於可移動介質管理的IEEE 1244標準的體系結構。

這些協議使用TCP / IP作為傳輸層。與流行的Internet協議（如HTTP，FTP或SMTP）一樣，命令通過TCP / IP以文本消息的形式發送。這些協議可以像在不同平臺上一樣簡單地實現和使用。

IEEE 1244體系結構的優點是媒體管理器可以將組件實現為通用應用程序，即獨立於所使用的特定庫和驅動器硬件。這些差異，特別是與控件的差異，被封裝在庫管理器或驅動管理器中，用於所討論的硬件。因此，對於新的磁帶庫，只需要實現一個新的庫管理器組件，該組件將庫的特定接口轉換為LMP，以便可以將其鏈接到可移動介質的IEEE 1244兼容管理系統的現有安裝中。

11.5.2媒體管理員和MMP

從應用程序的角度來看，媒體管理器充當等待來自應用程序的請求的服務器。它區分特權和非特權應用程序。非特權應用程序（如備份系統）只能與已授予適當授權的對象一起使用。特權應用程序（通常是管理應用程序）可以執行所有操作並操縱所有對象。它們主要用於將非特權應用程序定義到系統中並建立適當的訪問控制。

表11.1 IEEE 1244數據模型的最重要對象。

應用程序使用媒體管理協議（MMP）來利用IEEE 1244兼容系統的媒體管理服務。 MMP是一種基於文本的協議，它通過TCP / IP交換消息。各種協議消息的語法和語義在MMP規範IEEE 1244.3中規定。 MMP允許應用程序分配和裝載卷，讀取和寫入元數據，以及獨立管理和共享庫和驅動器平臺。由於額外的抽象級別，應用程序與硬件的直接控制分離。因此，可以獨立於連接的硬件的能力開發應用程序，並且可以使應用程序可用於大量不同類型的可移動介質。

MMP是一種異步協議。應用程序將命令作為MMP消息發送到媒體管理器。媒體管理器首先確認接受命令，將其傳遞以進行處理，然後發送帶有命令結果的第二條消息。一旦應用程序收到確認接受上一個命令的消息，它就可以發送更多命令。因此，應用程序必須為命令提供任務標識符，以便它們可以相應地關聯來自媒體管理器的響應。

示例1應用程序要使用名稱backup-199912-31安裝卷。為此，它將以下命令發送給媒體管理器：

裝載任務[“1”]

volname [“backup-1999-12-31”]

報告[MOUNTLOGICAL。“MountLogicalHandle”];

媒體管理器已識別該命令並接受它進行處理，因此發送以下響應：

響應任務[“1”]被接受;

現在，媒體管理器將包含卷的盒式磁帶傳輸到應用程序可以訪問的驅動器中。成功插入盒式磁帶後，介質管理器會嚮應用程序發送響應，如下所示：

響應任務[“1”]成功文本[“/ dev / rmt0”];

這就是應用程序如何獲知用於讀取和描述具有所請求卷的驅動器的路徑（設備句柄）。

介質管理器將所有命令存儲在任務隊列中，直到執行所需的所有資源都可用。一旦所有資源都可用，媒體管理器就會從任務隊列中刪除該命令並執行它。如果存在需要相同資源的若干命令，則媒體管理器基於優先級或先來先服務的基礎選擇要執行的下一個命令。所有其他命令保留在任務隊列中，直到相關資源再次釋放為止。通過這種方式，可以共享庫，驅動器和盒式磁帶。可以使用“取消”命令再次刪除任務隊列中的命令。

示例2應用程序想要安裝卷SU1602。因此，它將以下命令發送到媒體管理器：

裝載任務[“1”]

volname [volname“SU1602”]

報告[MOUNTLOGICAL“MountLogicalHandle”。];

媒體管理員已經識別該命令並接受它進行處理。因此，它發送以下響應：

響應任務[“1”]被接受;

現在應用程序可以發送另一個命令。例如，可以請求所有有缺陷的驅動器的名稱列表。

show task [“2”] report [DRIVE。“DriveName”]

匹配[streq（DRIVE。“DriveStateHard”“broken”）];

媒體管理員已經識別該命令並接受它進行處理。因此，它發送以下響應：

響應任務[“2”]被接受;

由於當前正在使用所有驅動器，因此show命令在mount命令之前執行。媒體經理的迴應如下：

響應任務[“2”]成功案文[“LTO-DRIVE-3”“LTO-DRIVE-7”

“LTO-DRIVE-8”];

一段時間後，mount命令所需的所有資源都可用。介質管理器指示負責的庫管理器將卷所在的盒式磁帶發送到適當的驅動器。然後，它指示負責的驅動程序管理器加載盒式磁帶並生成驅動器的任務標識符（訪問句柄）。然後，應用程序從媒體管理器接收此MountLogical Handle（任務標識符）並使用它來訪問卷：

響應任務[“1”]成功文本[“/ dev / tape / LTO123”];

然後以這種方式處理一個接一個的命令，直到任務隊列為空。

在示例2中，show命令與match子句一起使用以顯示所有有缺陷的驅動器。 “顯示”和“匹配”構成了一種強大的查詢語言，可用於生成複雜的報告。圖形用戶界面可以提供預定義的報告，這些報告被轉換為相應的MMP命令。

Example 3 In the spring of 2003 an administrator suspected that drive LT0123 was indicating a high error rate. Now it wants to display all the cartridges mounted in the mentioned drive since 2003 where an error occurred during the mount. The list should be sorted in descending order according to the number of write errors. The corresponding show command would look like the following:

show task[“1”]

report[DRIVEACARTRIDGEACCESS.“CartridgePCL”] match[

and(

streq(DRIVECARTRIDGEACCESS.“DriveName”“LTO123”)

strgt(DRIVECARTRIDGEACCESS.“TimeUnmount”

“2003 01 01 00 00 00 000”)

or(

numgt(DRIVECARTRIDGEACCESS.“HardReadErrorCount” “0”) numgt(DRIVECARTRIDGEACCESS.“SoftReadErrorCount” “0”) numgt(DRIVECARTRIDGEACCESS.“HardWriteErrorCount” “0”) numgt(DRIVECARTRIDGEACCESS.“SoftWriteErrorCount” “0”)

)

]

order[numhilo(DRIVECARTRIDGEACCESS.“HardWriteErrorCount”)];

A characteristic of IEEE 1244 is the extension of the data model during runtime. Applications can add optional attributes to any object in order to file vendor-specific and applications-specific characteristics of objects in the media manager. For example, a backup system could create a list of all files on a cartridge as an attribute or a video rental shop could assign the number of the last customer who rented a particular video cassette (cartridge). MMP provides the appropriate attribute command.

Example 4 An application wants to set the applications-specific attribute ‘color’ to the value ‘red’ for the drive with the name LTO123. It sends the following command to the media manager:

attribute task[“3”] set[DRIVE.“Color” “red”];

match[streq(DRIVE.“DriveName” “LTO123”)];

The media manager has also recognised this command, accepted it for processing and then sends the following response:

response task[“3”] accepted;

The media manager applies this attribute dynamically if it has never been applied to this drive before. Otherwise only the value of the attribute is changed. Lastly, the media manager sends the following message back to the application in order to signal that processing has been successful:

response task[“3”] success;

11.5.3庫管理器和驅動器管理器

IEEE 1244通過庫管理協議（LMP）指定媒體管理器和庫管理器之間的接口。庫管理器接收媒體管理器的LMP命令，並將它們轉換為相應庫模型的特定命令。因此，從媒體管理器的角度來看，所有庫都存在隱藏相應硬件特徵的統一抽象接口。這使得新硬件可以通過合適的庫管理器集成到管理系統中，而無需更改整個系統。

以類似的方式，驅動器管理協議（DMP）指定媒體管理器和驅動器管理器之間的接口，從而使得能夠以統一的方式激活不同的驅動器硬件。但是，驅動器管理器必須考慮運行使用介質管理系統的應用程序的不同客戶機平臺的特定特徵。如果應用程序在兼容Unix的平臺上運行，則驅動器管理器必須提供設備專用文件的名稱，例如/ dev / rmt0，以便訪問該驅動器。在Windows下，這種驅動器管理器必須提供特定於Windows的文件名，例如\\。\ TAPE0 \。

與MMP一樣，LMP和DMP協議基於通過TCP / IP交換的文本消息。這兩種協議都用於控制硬件，傳輸磁帶（LMP）和將磁帶插入驅動器（DMP）。各個LMP消息的語法和語義在規範IEEE 1244.5中描述。 DMP消息在IEEE 1244.4中指定。

11.6摘要

數據量的大量增長和磁帶盒數量的增加需要改進的解決方案，以便有效管理存儲網絡中的可移動介質。這種系統在存儲網絡中的要求在很大程度上與磁盤子系統的要求相當。但是，由於可移動介質的特殊屬性，例如介質和驅動器的單獨存儲，還有其他問題需要掌握。

我們詳細描述了可移動介質管理的以下要求：

集中管理和控制所有資源（如媒體，驅動器和庫），使高效管理變得更加容易。

智能共享庫，驅動器和臨時池，可以更輕鬆地有效利用可用資源，並可用於實現高可用性解決方案（跨庫驅動器和臨時池）。

用戶/應用程序的身份驗證和授權有助於遵守數據保護準則和資源分配。

Mount請求管理（包括請求排隊）可提高整個系統的穩健性，並允許對驅動器進行優先處理和分配掛載請求。

不間斷地跟蹤介質可確保在每個時間點識別介質的位置，從而防止數據丟失。

可移動介質的自動化生命週期管理極大地使管理員免於重複發生的任務。

在線和離線位置的完全集成允許安全概念，其中可移動介質必須存儲在受到良好保護的位置。

Monitoring提供系統中錯誤的自動記錄和處理。基於規則的機制可以執行適當的錯誤糾正操作，或者可以發出手動干預信號。

不幸的是，目前（2009年）仍然沒有可移動媒體的跨製造商和跨應用程序管理層和接口。因此，每個製造商都會在市場上推出自己的專有系統，這通常是備份解決方案的一部分。這些系統中的一些系統不能協同使用連接硬件的多個實例，因此不能充分利用存儲網絡中資源共享的潛力。其他解決方案的可擴展性不是很高，這就是為什麼它們只能合理地用於一定數量的驅動器或盒式磁帶。對於來自多個製造商的不同應用，目前（2009）幾乎不可能合作使用所有可移動媒體資源。

IEEE 1244媒體管理系統標準為有效管理存儲網絡中的可移動媒體和智能共享資源提供了基礎，因此描述了可移動媒體的非對稱虛擬化的架構。 由於在TCP / IP上使用基於文本的協議，許多平臺可以很容易地利用基於該標準的系統的服務，其方式與廣泛的因特網協議（SMTP，HTTP）類似。 該標準解決了處理存儲網絡中可移動介質時最重要的問題，並提供瞭解決方案的工具。 特別重要的是不僅為應用程序（MMP）和硬件（LMP和DMP）定義了接口，而且還描述了與這些接口最佳匹配的體系結構。
