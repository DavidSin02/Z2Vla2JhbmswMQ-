SpringCloud配置中心動態加載(刷新)配置文件

2020-10-29
| 科技

前言

現在的分佈式系統中，通常會使用一箇中間件作為配置中心，目的是實現在多服務器集群的場景下，只需要統一修改配置中心的配置文件，然後所有的服務器全部獲取配置中心最新的配置文件信息，服務器不用重新啟動，即可以在所有的服務器上面生效。要想實現這個功能，它的原理大概如下：

場景1:服務器首次啟動-->獲取配置中心最新的配置-->然後實例化bean，並根據配置文件信息初始化bean的所有的屬性。這個場景很好理解，那些原本配置在本地的配置文件，也就是這樣使用的，只不過配置中心的配置，需要遠程獲取一下。

場景2:配置中心配置信息修改->通知服務器獲取最新的配置-->獲取到最新的配置後，銷燬所有的bean-->重新實例化所有的bean，然後根據最新的配置文件信息初始化所有的bean屬性。這個過程就是SpringCloud框架，給我們實現的配置中心動態加載(刷新)配置文件。

注：下面所說的Spring框架包含:Spring FrameWork，Spring Boot，Spring Cloud。

Spring框架實現

在Spring框架中是這樣實現的。

1.在類上加@RefreshScope註解。

2.在屬性中引入@Value註解

3.手動調用刷新http://localhost:8080/refresh，我猜想這個接口會觸發銷燬所有註解了@RefreshScope註解的bean，然後再後續使用bean的時候在實例化，初始化。因為沒有看源碼，我猜想是這個原理。

新版本中需要引入actuator，並添加如下的配置，來暴露refresh接口。

源碼分析

來看一下調用過程。

@Endpoint(id = "refresh")public class RefreshEndpoint {   private ContextRefresher contextRefresher;      public RefreshEndpoint(ContextRefresher contextRefresher) {      this.contextRefresher = contextRefresher;     }   @WriteOperation       public Collection<String> refresh() {      //refresh方法最終會調用此邏輯     Set<String> keys = this.contextRefresher.refresh(); //實際調用ContextRefresher去做的刷新邏輯          return keys;      }}//this.contextRefresher.refresh();方法public synchronized Set<String> refresh() {		Set<String> keys = refreshEnvironment();		this.scope.refreshAll();//實際調用這個方法去刷新		return keys;	}//this.scope.refreshAll();方法@ManagedOperation(description = "Dispose of the current instance of all beans "			+ "in this scope and force a refresh on next method execution.")	public void refreshAll() {		super.destroy();//銷燬所有的bean		this.context.publishEvent(new RefreshScopeRefreshedEvent());//發佈刷新事件	}

我自己的疑惑是：

我用zookeeper作為配置中心，我發現我在zookeeper上面修改了配置信息，在應用程序中自動就生效了，因為沒有找到源碼，我猜想是Spring框架最終底層是調用了refreshAll()方法，但是我發現對象是沒有被重新實例化的，因為通過IDE調試時，對象的編碼還是和以前一樣，如下圖。


帶著這個疑惑，繼續看源碼中...

後續：

1.當修改zookeeper配置中心的配置時，RefreshEventListener監聽器會被觸發，部分 源碼如下：

private ContextRefresher refresh;public void handle(RefreshEvent event) {   if (this.ready.get()) { // don't handle events before app is ready          log.debug("Event received " + event.getEventDesc());          Set<String> keys = this.refresh.refresh();  //實際調用ContextRefresher去做的刷新邏輯，     //與手動調用   最終觸發的邏輯是一樣的。      log.info("Refresh keys changed: " + keys);   }}

在ContextRefresher方法中會觸發配置信息刷新的邏輯，部分源碼如下：

public synchronized Set<String> refreshEnvironment() {   Map<String, Object> before = extract(         this.context.getEnvironment().getPropertySources());           addConfigFilesToEnvironment();  //這個邏輯會觸發重新去zookeeper獲取最新的配置信息       Set<String> keys = changes(before,  extract(this.context.getEnvironment().getPropertySources())).keySet();     this.context.publishEvent(new EnvironmentChangeEvent(this.context, keys));     return keys;//返回值是value值改變了的key集合}
