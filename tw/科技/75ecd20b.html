0777-5.16.2-Hive中使用Date函數用於條件查詢結果異常分析

2020-11-14
| 科技

作者：輝少

異常描述

  • 測試環境

1.RedHat7.2

2.CM和CDH版本為5.16.2


在CDH5.16.2 中使用Hive時 ,當Hive 的查詢where條件中使用Date函數後，函數中的列的值會在返回結果中被改變，導致結果不正確。具體表現為使用Date 函數當查詢條件後導致string 格式的日期丟失了時間部分，只有日期。展示如下:

select * from testdate;SELECT * from testdate where Date(str2) == '2020-05-24';


異常重現

重新創建表復現該問題

create table testdate2 (str1 string , str2 string ,str3 string, str4 string);insert into table testdate2 values ("日期測試1",'2020-05-24 19:50:32','2020-05-24','testdate');insert into table testdate2 values ("日期測試2",'2020-05-24 20:20:32','2020-05-25','testdate2');


分別使用帶Date 函數條件、不帶條件 、以及其他條件查看，確認的確存在該問題

SELECT * from testdate2 where Date(str2) == '2020-05-24';select * from testdate2;SELECT * from testdate2 where str3 == '2020-05-25';


異常分析

關於以上問題，與Hive的一個已知BUG HIVE-22513[1]有關，主要是由於Hive對列條件過濾操作中的持續傳播從而導致錯誤的結果。Hive 自0.14.0開始，加入了一項CBO (Cost based Optimizer)，來對HQL執行計劃進行優化，這個功能通過”hive.cbo.enable”來開啟。Hive 1.1.0 之後，該功能默認開啟，而CDH5.16.2 中Hive 版本1.1.0，因此受影響。


異常解決和總結

對於Date函數持續傳播從而導致錯誤的結果基於上述分析，提供以下2種解決辦法


1.單個查詢中臨時關閉CBO，在Hive 1.1.0 可以通過set hive.optimize.constant.propagation = false; 來臨時關閉單個查詢，缺點是可能會影響query的性能。從參考文檔[2][3]中可以看出，使用constant.propagation會在一定條件下在query進入執行階段前預先計算部分值，所以對query影響具體是因query不同而不同的。

set hive.optimize.constant.propagation = false; SELECT * from testdate2 where Date(str2) == '2020-05-24';


2.使用其他函數代替Date 函數，比如substr(str2,1,10) = '2020-05-24'

SELECT * from testdate2 where substr(str2,1,10) = '2020-05-24';
