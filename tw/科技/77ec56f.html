超詳細sharding-jdbc分庫分表實現，及原理分析

2020-10-29
| 科技

背景

傳統應用通常將數據集中存儲在單一的數據節點中，已經越來越不能滿足現代互聯網的海量數據場景。隨著用於用戶的增長，對併發性能要求越來越高，系統後臺服務可以很容易的進行擴容（負載均衡等），這樣最終的瓶頸就是數據庫了，單一的數據節點或者簡單的主從結構是難以承受的。  儘管目前已經有 nosql/newsql 能夠支撐海量的數據，但是其對傳統 sql 是不兼容的，而且生態圈頁不太完善，關係型數據庫的地位還是無法撼動的。  由此產生了數據分片的概念。按照某個分片維度將存放在單一數據庫中的數據分散地存放至多個數據庫或表中以提升性能。數據分片的拆分方式分為：垂直分片和水平分片兩種。

垂直分片

按照業務邏輯拆分的方式稱為垂直分片，又稱為縱向拆分。核心理念就是專庫專用。將一個大的數據庫按照業務邏輯分類，拆分為多個小的數據庫，每個業務邏輯使用各自的數據庫，從而將壓力分散到不同的數據庫中。垂直分片往往需要對架構和設計進行調整，類似微服務的概念。但是垂直拆分只能屬於一個治標不治本的辦法，隨著業務量進一步加大，超過了單個表能承受的閾值，還是會出現性能問題。然後就需要水平分片來進一步處理了。

水平分片

水平分片又稱橫向分片。相對於垂直分片，不根據業務邏輯分，而是通過某個（幾個）字段，根據某種規則將數據分散到多個庫或表中，每個分片僅包含數據的一部分。例如根據用戶主鍵分片，對 2 取餘為 0 的放入 0 庫（或表），為 1 的放入 1 庫（或表）。如下所示：

從理論上來說水平分片是可以無限拓展的，屬於分庫分表的標準解決辦法。但是有利就會有弊，雖然分片解決了性能問題，但是分佈式的架構頁引入了新的問題。首先面對的就是數據庫的運維管理更加複雜，之前只是單庫有限的幾張表，現在數據被分散到了多庫多表中，難以運維。其次就是很多之前能夠正常運行的 sql 語句，在水平分片後無法正常運行，比如分頁、排序、聚合分組等操作。 最後就是分佈式事務。跨庫事務目前任然是一個較為棘手的事情。主要有兩種解決辦法：基於 XA 的分佈式事務和最終一致性的柔性事務。

實戰

瞭解了分庫分表的概念下面就是實戰了，最初是考慮使用 Mycat 的，但最終沒有采用的原因主要是以下兩點：

  • Mycat 較為混亂，沒有明確的開發計劃和完善的文檔（文檔是一本書）。而且首頁看著像是 90 年代的頁面。
  • 最關鍵的一點，Mycat 已經很久沒有更新了，自從 17 年發佈了最後一個版本就再沒有動靜了（官網說的是籌備 2.0 中）   在查找資料的過程中瞭解到噹噹的一款開源分庫分表產品–sharding-jdbc.目前已經遷移到 apache 孵化中心，首頁也相應的換了，現在的官網首頁,同時也改名了：ShardingSphere。目前最新的發佈版本是 3.1，有以下三個產品：
  • Sharding-JDBC。ShardingSphere 就是在此基礎上發展來的。僅支持 java，屬於輕量級 java 框架，在 java 的 JDBC 層提高額外服務，相當於加強版 JDBC 驅動，因此可以與任何上層 ORM 框架配合使用，支持任意的數據庫連接池，支持任意實現 JDBC 規範的數據庫。本篇代碼就是基於 Sharding—JDBC。
  • Sharding-Proxy:跟 MyCat 一樣屬於數據庫代理，對代碼透明，目前僅支持 MySQL 數據庫。
  • Sharding-Sidecar:規劃中的產品，定位為雲原生數據庫代理。  下面開始基於 Spring-boot 的實戰。總體如下：
  • 分庫分表使用
  • 廣播表使用
  • 默認庫使用 （PS：官方也有使用示例：https://github.com/apache/incubator-shardingsphere-example）

    事務處理

      默認使用的是本地事務，但是如果業務邏輯拋出錯誤，還是會對所有的庫進行回退操作的，只是如果出現斷電斷網的情況會導致數據不一致。詳見：官方文檔. 可通過@ShardingTransactionType(TransactionType.XA)註解，切換為 XA 事務或者柔性事務（示例中未配置，切換為柔性事務會報錯）。
