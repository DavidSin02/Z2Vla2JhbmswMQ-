後傳

小D：“這就是狀態圖，描述這個問題的模型就是狀態機，是不是我們生氣這麼複雜的流程被狀態圖畫出來就很明瞭了。每一個狀態都可以跳轉到其他狀態，而藍色字的描述就是觸發條件，達到了這個條件就會進行狀態轉移，這樣我們最終才能又回到了開心的狀態。”

小D：“你工作裡面那麼多理不順的事情，你就用狀態機把事情的狀態和轉移條件畫出來，這樣所有狀態和轉移條件描述完了，整個流程你就理順了，處理起來也不會暈暈乎乎！”

小Q：“我懂了！麼麼噠！”

寫給程序員：

我一直覺得其實很多代碼中解決問題的模型都可以從生活中來獲取，反過來有些在代碼中解決問題的思想也可以用在生活中。

我們繼續聊生氣這個狀態模型，這樣的圖畫出來，你寫代碼的思路是不是就清晰，在出現異常問題（生氣）的時候該如何處理，最後才能恢復到正常的狀態一目瞭然。現在讓我們用一些不嚴謹的代碼來把模型描述出來，當描述完的時候你的代碼就寫好了，可以進行測試了。

void let_wife_happy(){ uint8_t 狀態; while(1) { switch (狀態) { case 開開心心: if(小D不理小Q)狀態=小Q生氣; else if(小Q吼小D)狀態=小D生氣; else if(一直哄)狀態=開開心心; else 狀態=小Q生氣; break; case 小Q生氣: if (哄小Q)狀態=開開心心; else if(吼小D)狀態=小D生氣; case 小D生氣: if (發酵)狀態=倆人生氣; break; case 倆人生氣: if(等10分鐘)狀態=倆人平靜; else if(吵起來)狀態=冷戰; break; case 兩人平靜: if (哄小Q)狀態=開開心心; break; case 冷戰: if (踢小D下床)狀態=兩人談心; break; case 倆人生氣: if (哄小Q)狀態=開開心心; break; default: break; } }}

關於狀態機更深層討論

醒醒，別隻沉浸在開開心心的狀態中。咱再進行一些技術層面更深層次的探討。再往更抽象的一層看，如上所示的狀態機主要解決了什麼問題，這種生氣的場景和在開發中的哪些場景有點類似？

狀態機主要解決了什麼問題

最重要的就是把錯綜複雜的問題先進行建模進行梳理。在開發過程中，但不僅僅限於嵌入式軟件開發，針對某一個問題處理可能出現很多種可能，如果想到一種可能就寫一點代碼，不把自己寫暈進去才怪。最後寫過去的自己都不知道寫的啥。比如一個wifi聯網設備，從上電初始化——》連接路由器——》連接服務器——》給服務器發送數據——》收到服務器返回數據——》進入低功耗模式。

從整體來思考流程不就是一個個狀態的轉移，藉助狀態機建模把整個流程切分成一個個狀態。當然在大框架下，可能還能細分出來一個個小的狀態機模型，比如初始化裡面可能還有很多個步驟，可以把這些步驟再用一個狀態機來描述。 當你把所有流程理清了。寫代碼只是最後最容易的一步。

生氣的場景和開發中哪些場景類似？

在嵌入式開發中，一個功能開發好的產品有時候為什麼會出bug。歸根結底就是對異常情況的處理。如果保證一直都是正常的情況，一直運行當然沒問題。但是總會出現一些異常（就像例子中的“生氣了”）。就比如wifi聯網設備，連接服務器過程中和路由器斷電了，是不是異常？給服務器發送數據沒收到服務器返回數據是不是異常？

而我們用狀態機建模就可以把能想到的所有異常的狀態都描述出來，最終通一定的條件轉移讓系統修復迴歸到正常的狀態。
