多態&動態綁定&方法匹配&方法綁定

2020-11-14
| 科技

1）多態：父類型的變量，可以引用子類型的對象


  2. class A{
  3. }
  5. class B extends A{
  6. }

那麼可以 A a = new B(); 其中這裡的A叫做聲明類型，B叫做實際類型那麼這裡的a就可以調用B中的方法（如果有的話），那麼是不是什麼方法都能調用呢？往下看

2）動態綁定

定義：如果一個方法在父類中定義，在子類中重寫，那麼這個方法的調用是由實際類型決定的。這就是動態綁定

例子：


  2. class A{
  3. public void f(){
  4. System.out.println("AAA");
  5. }
  6. }
  8. class B extends A{
  9. public void f(){
  10. System.out.println("BBB");
  11. }
  13. public void m(){
  14. System.out.println("MMM");
  15. }
  16. }

那麼 A a = new B(); a.f(); 結果是輸出BBB。。因為實際類型是B

注意：看清楚動態綁定的前提，一個方法在父類中定義，在子類中重寫。

a.m()會出錯，原因是父類裡面根本沒有這個方法，只是在子類裡有。

那麼多態調用方法具體原理是什麼呢？往下看

3）方法匹配

定義：調用方法時，引用變量的聲明類型，決定了編譯時匹配哪個類的哪個方法

例子：上面的a.m()，由於a的聲明類型是A，所以在類A裡面找m()這個方法，發現沒有，自然就出錯了。

那要是有這個方法呢，為什麼又調用了子類的方法？這就要說到方法綁定了

4）方法綁定

定義：也就是上面的動態綁定。是在運行時實現的。

也就是說編譯時，方法匹配決定匹配聲明類型中的哪個方法；運行時，去看看這個方法在實際類型中是不是重寫了，重寫了就調用實際類型的這個方法，不然就聲明類型的那個方法

例子：解釋上面的a.f()的過程。

首先a的聲明類型是A，那麼編譯時去A這個類裡面找f()方法，找到了，那麼編譯通過

然後運行時，發現a的實際類型是B，那麼去B這個類裡面看看有沒有重寫f()這個方法。有。好，就用B的f()

所以輸出BBB

5.2）解釋

5.2.1）String s = "123"; String s = "456"; 是為了驗證變量是由什麼類型決定的。。。可以看到變量是由聲明類型決定

5.2.2）前三個輸出（A and A，A and A，A and D），很正常，最多涉及一個函數參數的自動類型轉換（子類自動轉父類），沒什麼說的。

5.2.3）中間三個輸出（B and A，B and A，A and D），先是在類A裡面進行方法匹配，運行時再到類B裡面看看有沒有重寫，發現只有一個public void show(A obj)重寫了，那麼就只有這個方法是調用B的方法，其他都調用A的方法

5.2.4）後面四個輸出（B and B，B and B，A and D，B and E），我們只解釋A and D為什麼。由於繼承的原因，其實A中的非private方法都被B繼承了，所以B自己找不到方法的，會去父類A裡找。

所以才會有A and D這個輸出

5.3）再來看看那句，被註釋的，編譯不過的

a2.show(e);//這句編譯不過

5.3.1）原因呢？編譯階段，方法匹配不成功。因為A裡面沒有 show(E obj)的，而且也沒有show(O obj)的。（其中O是E的父類型）

5.3.2）那麼怎樣才能編譯通過呢？

沒辦法，只好轉回實際類型：B a2 = (B)a2; a2.show(e)就沒問題了

那我不想轉成實際類型怎麼辦呢？ ((B)a2).show(e)也行

6）來點書上的判斷對錯題

前提Number是Double、Integer等類型的父類

6.1）


  2. Number numberRef = new Integer(0);
  3. Integer IntegerRef = (Integer)numberRef;

可行，原因是numberRef的實際類型本來就是Integer，轉換回去當然可以6.2）


  2. Number numberRef = new Integer(0);
  3. Double DoubleRef = (Double)numberRef;

不行，編譯能通過，但是運行會出錯。原因是Integer類型和Double類型不能相互轉換。要想轉換，那麼先轉成數值，再轉換。也就是Double DoubleRef = new Double(numberRef.doubleValue());

6.3）


  2. Number x = new Integer(3);
  3. x.compareTo(new Integer(4));

不行，編譯不能通過。原因是Number裡面沒有compareTo方法，雖然Integer裡面有。。但是上面講的很清楚了，方法匹配是從Number類開始的
