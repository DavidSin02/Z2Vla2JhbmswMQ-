面向對象設計原則總結

先說軟件開發過程中的兩個重要概念：

（1）可維護性(Maintainability)：指軟件能夠被理解、改正、適應及擴展的難易程度

（2）可複用性(Reusability)：指軟件能夠被重複使用的難易程度

面向對象設計的目標之一在於支持可維護性複用，一方面需要實現設計方案或者源代碼的複用，另一方面要確保系統能夠易於擴展和修改，具有良好的可維護性。

面向對象的設計原則：

（一）、單一職責原則(Single Responsibility Principle, SRP)

一個對象應該只包含單一的職責，並且該職責被完整地封裝在一個類中。單一職責原則是最簡單的面向對象設計原則，用於控制類的粒度大小，因為一個類（大到模塊，小到方法）承擔的職責越多，它被複用的可能性就越小，當一個職責變化時，可能會影響其他職責的運作，應該將這些職責進行分離，將不同的職責封裝在不同的類中，將不同的變化原因封裝在不同的類中，故單一職責原則是實現高內聚、低耦合的指導方針。

單一職責原則：一個對象應該只包含單一的職責，並且該職責被完整地封裝在一個類中。Single Responsibility Principle (SRP): Every object should have a single responsibility, and that responsibility should be entirely encapsulated by the class.

（二）、開閉原則(Open-Closed Principle, OCP)

軟件實體應當對擴展開放，對修改關閉。Open-Closed Principle (OCP): Software entities should be open for extension, but closed for modification.

在開閉原則的定義中，軟件實體可以是一個軟件模塊、一個由多個類組成的局部結構或一個獨立的類，開閉原則是指軟件實體應儘量在不修改原有代碼的情況下進行擴展。

（三）、里氏代換原則(Liskov Substitution Principle, LSP)

所有引用基類的地方必須能透明地使用其子類的對象。Liskov Substitution Principle (LSP): Functions that use pointers or references to base classes must be able to use objects of derived classes without knowing it.

在軟件中將一個基類對象替換成它的子類對象，程序將不會產生任何錯誤和異常，反過來則不成立。如果一個軟件實體使用的是一個子類對象的話，那麼它不一定能夠使用基類對象。在程序中儘量使用基類類型來對對象進行定義，而在運行時再確定其子類類型。

（四）依賴倒轉原則(Dependence Inversion Principle, DIP)

依賴倒轉原則：高層模塊不應該依賴低層模塊，它們都應該依賴抽象。抽象不應該依賴於細節，細節應該依賴於抽象。Dependency Inversion Principle (DIP): High level modules should not depend upon low level modules, both should depend upon abstractions. Abstractions should not depend upon details, details should depend upon abstractions.

含義：要針對接口編程，不要針對實現編程：在程序代碼中傳遞參數時或在關聯關係中，儘量引用層次高的抽象層類，即使用接口和抽象類進行變量類型聲明、參數類型聲明、方法返回類型聲明，以及數據類型的轉換等。

在程序中儘量使用抽象層進行編程，而將具體類寫在配置文件中。

（五）接口隔離原則(Interface Segregation Principle, ISP)

客戶端不應該依賴那些它不需要的接口。Interface Segregation Principle (ISP): Clients should not be forced to depend upon interfaces that they do not use.

當一個接口太大時，需要將它分割成一些更細小的接口，使用該接口的客戶端僅需知 道與之相關的方法即可。

每一個接口應該承擔一種相對獨立的角色，不幹不該乾的事，該乾的事都要幹。

（六）合成複用原則(Composite Reuse Principle, CRP)

合成複用原則：優先使用對象組合，而不是繼承來達到複用的目的。Composite Reuse Principle (CRP)：Favor composition of objects over inheritance as a reuse mechanism.

合成複用原則就是在一個新的對象裡通過關聯關係（包括組合關係和聚合關係）來使用一些已有的對象，使之成為新對象的一部分，新對象通過委派調用已有對象的方法達到複用功能的目的，複用時要儘量使用組合/聚合關係（關聯關係），少用繼承。

1、繼承複用：實現簡單，易於擴展。破壞系統的封裝性；從基類繼承而來的實現是靜態的，不可能在運行時發生改變，沒有足夠的靈活性；只能在有限的環境中使用。（“白箱”複用 ）

2、組合/聚合複用：耦合度相對較低，有選擇性地調用成員對象的操作；可以在運行時動態進行，新對象可以動態地引用與成員對象類型相同的其他對象。（“黑箱”複用 ）

（七）迪米特法則(Law of Demeter, LoD)

迪米特法則：每一個軟件單位對其他的單位都只有最少的知識，而且侷限於那些與本單位密切相關的軟件單位。Law of Demeter (LoD): Each unit should have only limited knowledge about other units: only units "closely" related to the current unit.

迪米特法則要求一個軟件實體應當儘可能少地與其他實體發生相互作用，應用迪米特法則可降低系統的耦合度，使類與類之間保持鬆散的耦合關係。

迪米特法則要求在設計系統時，應該儘量減少對象之間的交互，如果兩個對象之間不必彼此直接通信，那麼這兩個對象就不應該發生任何直接的相互作用。如果其中一個對象需要調用另一個對象的方法，可以通過“第三者”轉發這個調用，通過引入一個合理的“第三者”來降低現有對象之間的耦合度。
