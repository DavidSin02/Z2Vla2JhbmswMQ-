代碼風格不統一？GitHub 高星代碼格式化工具拯救程序員強迫症

2020-11-14
| 科技

對程序員來說，熬得禿頂也無法解決的四大難題是：

  • 1. 世界上最好的語言是什麼？
  • 2. 世界上最好的編輯器是什麼？
  • 3. 用 Tab 好還是 Space 好？
  • 4. 到底要不要寫分號？

而世界上有兩種程序員：一種用 Tab 鍵，另一種用 Space 鍵。

“Tab vs Space”是業界曠日持久的聖戰，事關程序員的信仰！

每一名程序員都有權利將自己的代碼賦予標籤與個性，毫不誇張的說，如果團隊沒有統一代碼風格，那麼這個團隊有多少名成員就有多少種代碼風格。

分號時隱時現撲朔迷離，Tab Space 交相呼應琴瑟和鳴，兩個四個空格處處空格相映成趣，好一派“百家爭鳴”的自由和諧！

光是縮進這一件事就能夠產生如此大的分歧， 那項目開發代碼無數行，不同的代碼風格，就為之後審核代碼，帶來了巨大的工作量。

在團隊整體項目開發過程中，維護代碼的時間遠遠大於新功能開發，成員變更更是導致很多程序深陷“填坑”旋渦。因此編寫符合團隊編碼規範的代碼是至關重要的，這樣做不僅可以很大程度地避免基本語法錯誤，也保證了代碼的可讀性。

不僅如此，對於代碼版本管理系統（svn 和 git 或者其他），代碼格式不一致帶來的問題更是十分嚴重，代碼一致卻格式不同觸發版本管理系統標記為 diff 會導致無法檢查代碼和校驗。

那麼重點來了，怎麼統一代碼風格？怎麼搞定那群 Tab/Space 黨讓他們的代碼和諧共存？

此時小編挖出了 GitHub 29.6k 星代碼格式化神器---Prettier

就像它的名字一樣，有了它，你的代碼肯定“更美”！就是完全統一完全一致的那種美！完美拯救“強迫症”！

Prettier 主要是為了格式化代碼，作為一個有態度的代碼格式化工具，支持多種語言，集成大多數編輯器，使用 Prettier 只需要保存文件代碼就會被整理格式，再也不用在代碼複查的時候爭論風格這一“百家爭鳴”的話題了，節約的時間和精力。

當你意識到再也不用自己手動去調整代碼風格時，有沒覺得很幸福？

而在沒有 Prettier 之前，是用 ESlint-fix 和編輯器自帶代碼格式來進行代碼格式化的。但是每種編輯器會有不一樣的代碼格式，而且配置會比較麻煩。Prettier 已經逐漸成為業界主流的代碼風格格式化工具，使用 Prettier 可以減輕 ESlint 等工具的校驗規則，因為將代碼樣式校驗交給了 Prettier，所以可以將代碼校驗的規則更準確地應用到代碼真正的規範上面。

舉個例子，輸入代碼是這樣的：

if (food === 'pizza'){ alert('Pizza ;-)'); }else{  alert('Not pizza ;-(');}function foo(items) { return items .filter(item => item.checked) .map(item => item.value) ;}const food = [ 'pizza', 'burger', 'pasta',]

Prettier 處理之後的代碼是這樣的：

if (food === "pizza") { alert("Pizza ;-)");} else { alert("Not pizza ;-(");}function foo(items) { return items.filter(item => item.checked).map(item => item.value);}const food = ["pizza", "burger", "pasta"];

此處率先聲明，ESlint 是主要還是負責代碼規則校驗，Prettier 只調整代碼風格代碼樣式，ESlint 才是真正檢查代碼是否符合規範的工具。

所以，如果讓 Prettier 結合 ESlint 或 EditorConfig 等會有意想不到的奇效哦！

GitHub地址：https://github.com/prettier/prettier
