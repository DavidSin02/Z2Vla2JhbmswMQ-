詳解｜天貓搜索前端技術歷代記

2020-10-29
| 科技

導讀：作者從 17 年雙十一前開始接手天貓搜索前端，開發第一個需求—— H5 湊單頁，到今天已經將近兩年了。在這兩年裡，天貓搜索的前端體系發生了比較大的變化。今天分享一篇階段性的總結文章，記錄天貓搜索前端技術的過去、現在，以及自己作為業務目前的唯一前端對未來的思考。

大體劃分

首先基於前端技術的演進，大體上可以將天貓搜索前端的發展歷程和未來趨勢總結成幾個時代：

  • PC 時代
  • H5 時代
  • MV* 時代
  • Weex 時代
  • 搭建時代
  • 深度搭建時代
  • 智能時代

之所以這麼劃分，主要是基於天貓搜索的前端技術方向以及天貓乃至淘系前端技術體系發生的較大變化。而在這其中可以再提煉時代的關鍵詞：

  • 封閉：PC 時代、H5 時代、MV* 時代
  • 開放：Weex 時代、搭建時代、深度搭建時代
  • 智能：智能時代

下面就來介紹各個時代天貓前端的技術狀態和一些思考。

1、PC時代

PC 時代可以說是天貓搜索前端的上古時代，那還是手機流量非常貴的 3g/2g 時代。因此大多還是簡單的 WAP 頁面，大多數人都還是習慣用 PC 進行購物。

▐ 技術方案

模塊化

PC 時代的前端技術方案採用的是 KISSY + MUI 3 ，MUI 3 就是那套 KISSY 的模塊規範 KMD 。頁面上還有一些非常老舊的 YUI 依賴。在那個 jQuery 王霸天下的時代，基本上天貓的 PC 頁面都是採用集團內部根據 YUI 自研的大而全的 KISSY 框架。KISSY框架包含了前端所需要的幾乎所有基礎功能：模塊加載器、 DOM 操作、事件處理、異步請求等等。

頁面渲染

PC 天貓搜索採用了同步渲染的方式，頁面的主要內容通過 VM 模板渲染輸出到前端，因此前端需要維護大量的 Velocity 模板代碼以保證 HTML 中的內容和自己 JS 代碼能夠配合得當。需求修改一旦涉及到 HTML ，就需要改造 Velocity 模板從而陷入 aone 發佈的同步模板和部署煉獄之中。

模塊管理

而 PC 時代的天貓搜索前端頁面的模塊管理方式十分粗暴，基於業務邏輯劃分的大量 KISSY 模塊，他們共同操作一個統一的頁面 DOM ，交叉修改時有發生。模塊間的通信也是直接調用模塊實例方法來實現，模塊耦合非常嚴重。

▐ 小結

由此可以看到那個時代最主要的問題有如下幾個：

  • 大而全的框架使得頁面笨重、依賴繁雜
  • 前端花費大量成本去維護不熟悉的 Velocity 代碼
  • 基於應用環境，經常性需要同步模板或部署，應用不穩定時甚至無法調試
  • 頁面內模塊化粗暴，模塊耦合嚴重
  • 重 DOM 操作，DOM 管理混亂

2、H5 時代

隨著智能手機的普及，4g 技術的發展使得流量費用的大幅下降，無線端需求不斷增多，流量不斷增大，天貓前端也將方向定為 mobile first 。顯然 PC 搜索這一套技術方案無法滿足 H5 搜索需求，因此 H5 搜索採用了全新的技術方案。

▐ 技術方案

模塊化

H5 搜索採用 Zepto + MUI 4 作為模塊化方案，MUI 4 相對 MUI 3 主要變化是將 KMD 規範修改成了業內較為通用的 AMD 規範。同時引入了前端模板來實現 DOM 的更新，首先 Zepto 要比大而全的 KISSY 輕量許多，同時前端模板的引入使得模塊自我管理了 DOM ，有效的降低了模塊之間 DOM 操作交叉引發的混亂。

頁面渲染

H5 搜索的頁面渲染基於應用控制，但僅僅將框架、篩選等一些少量 DOM 同步渲染，大多數 DOM 如商品列表都是採用異步渲染前端模板的方式實現。因此同步模板中的內容量相對較小，維護成本大幅度降低。同時引入了前後端分離的 wormhole app ，前端不再需要維護不熟悉的 VM 模板，而是維護和異步模板語法相同的 xtpl 模板，進一步降低了模板維護的成本。

模塊管理

與 PC 連翻頁都是頁面跳轉的方式不同，如篩選、翻頁等行為的渲染需要前端異步處理，模塊通信的需求也相對複雜。H5 搜索採用了 MDV 框架——通過將模塊封裝成一個個模型，再由 MDV 建立的模型之間的訂閱機制進行模塊的實例管理，以此來實現模塊通信。

▐ 小結

H5 搜索採用的技術解決了幾個 PC 搜索時代的問題：

  • 減小依賴，滿足了H5的性能要求
  • 基於wormhole app的前後端分離開發，降低了模板維護成本
  • 引入前端模板，降低了DOM交叉管理的混亂
  • 引入MDV進一步使得模塊解耦，規範了模塊通信

但隨著業務的發展，依然發現有一些問題：

  • 雖然引入了前端模板，但所有更新和狀態同步依然是手動的，依舊會頻繁地 DOM 操作
  • 模塊劃分還是基於邏輯劃分，而非基於 DOM ，因此當一個邏輯模塊需要操作多個 DOM 時，依然會引發交叉
  • 雖然模塊對外是個黑盒，但模塊內部狀態經常因為交互邏輯和業務邏輯繁瑣導致混亂，經常出現更新了狀態沒更新 DOM 的情況
  • 依賴應用環境，需要經常性同步模板和部署，畢竟 wormhole app 也是應用

3、MV*時代

在 all in 無線的大環境下，由於搜索結構較為固定、無線端開發資源較為充裕等等各種原因，天貓搜索業務在端內完全被 Native 承接，而 H5 更加註重端外和站外。這也導致了 H5 搜索的業務能力和 Native 脫節，僅保留核心篩選功能以及喚端功能。因此 H5 搜索很長一段時間幾乎沒有前端資源投入，而在 2017 年 8 月左右由於組織架構調整，我手上的天貓商品業務交接後，我開始逐步接手天貓搜索前端業務。而我要做的第一個需求，就是一個包含 H5 搜索 80% 功能的 H5 搜索湊單頁。

▐ 技術方案

模塊化

我和團隊同學採用 Preact + MUI 5 方案來實現，MUI 5 其實就是 MUI 4 的升級，採用 CommonJS 的寫法來編寫模塊，再經過構建工具編譯封裝生成 AMD 模塊。其實當時會場已經大規模採用 Vue Weex 了，那我們為什麼要採用 Preact 呢，主要是基於如下幾點考量：

  • 團隊內對於 React 技術棧有相當程度的瞭解
  • Rax 的發展和普及以及 Weex 團隊的一些調整，讓我們認為 Rax 是未來 Weex 的主要 DSL
  • Preact 相對於其他框架更加輕量，而搜索的頁面雖然有一定交互，但比較簡單，剛好能滿足需求
  • H5 是未來大的趨勢，當時兄弟團隊在天貓超市的極致 H5 體驗和沉澱給了我們充足的信心

因此在這個階段，我們使用 Preact 做了很多的搜索周邊的功能頁面，如上面提到的 H5 湊單、還有貓搜的領券彈層、湊單領券頁面、搜索分類頁面等等。

頁面渲染

由於 Preact 的異步渲染特性，頁面渲染我們直接拋棄了應用，而是採用斑馬源碼頁面的方式。這樣的好處是，前端不再需要和 aone 頻繁的打交道，服務端只需要封裝一個 mtop 接口給前端，前端在斑馬上創建基礎 HTML 引入資源並調用接口獲取數據渲染即可，前端自己完全掌控了調試、開發節奏。由於對於應用的依賴僅限於一個 mtop 接口，因此前端可以提前定好數據規範，在服務端開發接口的同時基於本地調試工具進行 mock ，大大降低了聯調成本。

模塊管理

React 生態有非常多的模塊管理方案，我們調研過的 Flux、Redux、Reflux 等等都可以滿足我們的需求。但經過仔細的思考我們選擇直接擼，不使用任何框架來管理模塊通信。如此考量的原因主要遵循如下理念：

  • 首先我們需要規劃頁面的模塊結構和通信結構
  • 若通信結構與模塊結構有較大的出入，且通信結構並非扁平，那麼我們就需要引入模塊通信框架來實現通信結構的扁平化
  • 若通信結構與模塊結構出入不大，且通信結構本身扁平，那麼我們就沒有必要引入通信框架
  • 保持模塊結構清晰、分離，面向未來，若未來業務發展逐漸繁瑣，可以再引入框架來處理模塊通信

▐ 小結

引入 MV* 框架後，我們的技術方案又得到了一些提升：

  • 由於引入 MV* 框架，模塊的劃分不再基於邏輯而是基於 DOM，DOM 的操作不再需要手動處理， DOM 交叉操作被徹底杜絕；
  • 使用斑馬源碼頁面，不再依賴應用環境，更方便的數據模擬和調試。不再需要維護模板；
  • 模塊內部狀態基於 MV* 管理，狀態不再混亂，保證了 DOM 和模塊狀態的一致性；

這裡還是有一些問題：

  • H5 總是面對性能的挑戰
  • 依然是源碼開發，不夠靈活，也無法共建
  • 每來一個需求，就需要頁面整體修改和發佈，有類似行業定製，只能重新開發一個

借鑑天貓超市極致 H5 經驗，搜索在分類頁做了非常多性能優化嘗試，比如：

1、代碼優化

  • 列表拆分 cell 進行懶加載
  • 接入 crossimage 優化圖片加載的 cdn 後綴
  • 自建 solution ，去除通用 solution 中各種無用依賴
  • 升級最新的 loader 來提升計算性能

2、DNS優化

  • dns-prefetch，提前處理dns
  • 收斂圖片和資源域名

3、接口優化

  • 使用dlp進行mtop接口緩存

4、埋點優化

  • 令箭埋點延遲到 pageload 後發佈
  • 採用非覆蓋式的 aplus 腳本，利用瀏覽器緩存
  • 採用 post 發送，觸發 sendBeacon

5、加載優化

  • 增加 service worker 層緩存 js、css 和圖片
  • 使用 zcache 緩存頁面模板、 js 和 css

6、體感優化

  • 使用 ranger 自動添加 URL 隱藏導航欄參數，防止頁面加載後手動隱藏出現的抖動

4、Weex 時代

17 年雙十一之後，搜索一成不變的模式也受到挑戰，在這個擁有大量流量和強用戶心智的場景下，搜索能夠玩出更多場景化、行業化、品牌化的玩法。因此產品上需要更多快速試錯的機會，也導致對於動態化的訴求則日益增加。而由於一些組織架構調整，天貓搜索的客戶端縮減到 2 人—— 1 個 ios 和 1 個安卓。客戶端需求開發因為需要發版而迭代緩慢。因此催生了 Native 內嵌 Weex 坑位的技術方案和 Weex 模板下發的 Oreo 平臺。

▐ 技術方案

模塊化

其實在我進入到搜索之前，Weex 坑位方案和 Oreo 就已經在小範圍使用了，有一部分客戶端同學寫的 Weex 1.0 的 Vue 模塊，另外還有一些 Weapp 模塊。這些模塊由於人員調整和離職已經很難找到倉庫源碼了。另外這些模塊也是僅面向 Weex 的，對於 H5 來說是完全無法使用的。因此在我進入搜索之後，提出了Prax的解決方案，本質上就是使用 Rax 0.x（當時還是0.4） + Preact 通過編譯轉換供Weex和 Web 同時使用。之所以要用Preact，主要是考慮到我們有端外 H5 場景，但是 Rax 當時 H5 性能是真的堪憂，而且當時的Weex團隊方向對H5性能優化投入並不多。另外從上面 MV* 時代可以看出我們在 Preact + MUI5 的技術方案上也有較多的沉澱。最終我們確定了採用編寫 Preact 代碼，通過自動化工具轉換成 Rax 代碼的方式來實現一份源碼同時複用在 Weex 和 Web 的技術方案。前端的開發流程就變成了如下：

  • 編寫包含基礎可複用純 UI 代碼的 Prax 模塊
  • 對於 Native 內嵌 Weex 坑位，編寫一個 Rax 模塊， Rax 模塊引用 Prax 模塊的 Rax 部分，並增加諸如埋點、容器聯動等功能，並通過專門的模板發佈系統發佈上線以供坑位使用
  • 對於 Prax 頁面，編寫一個斑馬模塊或頁面，引用 Prax 模塊。通過斑馬平臺發佈上線，在端內使用 Prax 頁面的 Weex 版本，在端外引用 Prax 頁面的 H5 版本

頁面渲染

Prax 頁面採用斑馬來處理頁面渲染。由於源碼頁面都是採用打包的方式，因此腳手架中分別有 Web 和 Rax 兩個打包 webpack 配置，構建時候分別運行生成 Weex Bundle 和 Web Bundle 。由於 Web 端本質上就是之前 Preact 的方式，因此渲染沿用了過去 Preact 的頁面框架。而Rax則是開發了一個全新的頁面框架。斑馬渲染 Weex 頁面本質上是通過xtpl控制 Weex Bundle 代碼生成，所以源碼頁面編譯出來的 Weex bundle 只需要在外通過 xtpl 增加一層 Weex 的頭尾就可以了。

對於 Native 內嵌坑位，則是採用模板下發的方式。前端會編譯生成一個自身可運行的 Weex Bundle ，包含頭尾等完整信息，然後通過發佈推送到搜索應用的機器上。當客戶端的搜索請求到達服務端，服務端基於業務邏輯確定需要使用哪些模塊，將模塊相關的信息告訴客戶端，這些信息包括模塊名稱、模塊位置、模塊數據。客戶端渲染 Native 頁面時根據模塊位置創建對應容器，並根據模板名稱請求 Weex 代碼，最後將代碼渲染到 Weex 容器中，並傳入模塊數據，實現模塊最終渲染。而在 Rax 代碼中，我們可以通過 window.__weex_data__ 拿到客戶端塞給容器的數據。當然這些適配都可以在構建層面處理，一般情況下開發者只需要關注純粹的 UI 模塊編寫即可。

模塊管理

Prax 頁面的模塊管理和 Preact 開發時期基本一樣。但是對於 Native 內嵌 Weex 坑位，模塊管理方案就完全不同了。看起來 Native 內嵌 Weex 坑位只展示一個Weex模塊，但實際上是一個只有單一模塊渲染的完整 Weex 頁面。因此 Weex 模塊之間是無法通信的，它們甚至都不在一個 Weex 容器內。正因為如此 Native 內嵌 Weex 大多做一些純展示的需求，對於複雜聯動需求依然是交給客戶端完成。另外客戶端提供了一些通用能力，封裝在 Weex 模塊中，如：

  • 獲取搜索當前篩選條件
  • 合併篩選條件並重新請求
  • 清空篩選條件並重新請求
  • Query 詞替換
  • 一次篩選內 mtop 緩存機制
  • ...

▐ 小結

經過了 Weex 時代，我們的技術又得到了提升：

  • 實現了一套代碼同時複用在 Weex 和 H5
  • 通過引入 Rax ，我們解決了端內的性能問題
  • 對於端外場景，我們也保證了 H5 性能以及優化的抓手
  • 靈活的 Weex 發佈機制突破了發版限制，滿足了產品的動態化訴求
  • 對於搜索坑位，我們也沉澱出了一套行業前端可以參與共建的機制

但這裡還是有幾個問題沒有解決：

  • 源碼頁面開發的問題依舊，不夠靈活，也無法共建，也不好複製多個
  • 搜索坑位內模塊展現邏輯是服務端硬代碼控制，對於行業規模化和動態數據源接入都需要重新編碼
  • 當多個模塊同時作用於同一場景時，模塊的展示邏輯就會交叉，引入各種各樣的問題

5、搭建時代

至此我們可以發現，現在搜索前端核心解決的問題逐漸變成了共建和規模化了，不再是之前單純前端技術問題了。既然要解決共建和規模化，我們可以看看規模化成功的一個典型例子：會場。過去會場也是一個個頁面前端人肉開發，到後來有了斑馬就可以由運營模塊化搭建了，前端只需要開發一些基礎模塊就可以滿足所有行業的訴求，多麼的美好！那麼搜索是不是也能搭建呢？

答案是：“是的”，但畢竟場景不同，我們不能和會場一樣樓層 duangduang 地堆砌，為什麼呢？

首先在場景的層面有比較大的不同。在用戶進入這個會場時，已經決定了當前的場景了，一個會場頁面就是一個場景，因此在這個會場頁面下，只需要圈出這個場景的模塊就可以了。而對於搜索，所有的場景都需要在搜索這一個頁面展示，而在搜索如何定義一個場景呢？

  • Query 詞：用戶搜了什麼詞，是品類詞、還是品牌詞、還是寬泛詞、還是一個具體的型號詞
  • 類目：用戶搜的詞所屬的類目是什麼？哪個一級類目？哪個葉子類目？或者說是橫跨很多類目？
  • 人群：用戶自己是哪個人群，潮流酷女？還是居家好男人？
  • 時間：用戶在什麼時間來的？聖誕節？新年？生日？
  • LBS：用戶在什麼地方來的？附近是不是有提供服務的門店？
  • ...

這麼多維度，一個或多個交織在一起才定義了一個場景，而在這個場景下會展示一個或多個模塊。

以 Query 舉例：

  • 用戶搜索品牌詞（如華為）：展示品牌搭建的 Minisite
  • 用戶搜索特定類目詞（如香水）：展示行業模塊
  • 用戶搜索禮物（寬泛詞）：展示禮物場景導購卡片
  • ....

當然，會場模塊裡面的貨品完全可以多樣化，另外現在會場基於袋鼠也有千人千面模塊的能力了，這裡是拿之前的會場做例子

基於這樣的思考，搜索啟動了一個長頸鹿項目，以之前搜索域就有的一個產品——品牌 Minisite 作為切入點，為品牌商提供了一個全新的品牌運營陣地

如上圖，從左往右分別是三個版本的 minisite 。這裡可以看到， Minisite 作為品牌在搜索的一個搭建陣地，一直採用的是提供模塊讓品牌商填入數據的玩法。品牌商自定義程度很低、維護慾望很低，因此導購效率一直不高。而在長頸鹿項目中，搜索在商品列表層之下又新開了一個層，在這個層裡展示一個品牌商家搭建的頁面，在這個頁面中品牌商家可以充分的自定義。而對於一個跨品類經營的大品牌，我們也開放了品牌詞+主營類目詞的組合，方便其對每個主營類目去搭建自己的頁面。

而在品牌詞下取得成功後，我們將能力擴展到了行業側，在搜索中臺——美高上建設了心智造產品。讓行業也可以基於同樣的方式去經營品類詞，搭建行業自己的頁面。

心智造和 Minisite 分別是針對品類詞和品牌詞，相互不會交叉。而對於一些橫跨多品類的平臺級需求，比如一些頻道相關的需求和拉新相關的需求，是非常容易與其他場景交叉的。為了解決這個問題，我們在美高上沉澱了一套模塊定投方案——無界，通過模塊級別的投放，讓場景之間的交叉變成可能，只需要簡單定義模塊展示的規則，就能讓不同場景的模塊展示在同一個頁面上。

▐ 技術方案

模塊化

首先是心智造和 Minisite ，它們的核心技術方案是相似的——都是採用千葉（一個給商家用的頁面搭建平臺）作為模塊搭建平臺，並在千葉的搭建能力之上，包裝了一層頁面投放策略的管理。其實到這裡，模塊化方案和會場並無二致，只是模塊的實現採用了 Prax 來兼容 Weex 和 H5 ：

1、編碼階段

  • 每個模塊都會定義自己投放的數據結構
  • 模塊內不包含頁面通用依賴（如 Rax 基礎組件、 mtop 等）

2、渲染階段

  • 有一個 Solution 控制頁面模板的渲染，對於Weex則是負責包裹 Weex 頭尾
  • Solution 將模塊相關數據規整，Web模塊代碼可以異步獲取， Weex 一般同步輸出到 Bundle 中，而模塊所用的數據都會放在頁面上提供一個簡易方法獲取
  • Soluiton 引入 PI（頁面初始化腳本）， PI 將相關模塊數據取出根據模塊信息獲取模塊代碼進行渲染，並從頁面上獲取數據交給模塊

而到了無界，這些就完全不 work 了。畢竟無界是走的模塊級定投，容器只有一個，因此投放場景很可能和心智造、 Minisite 場景交叉：

  • 沒有模塊也沒有心智造和 Minisite 頁面，那麼我們就需要創建一個只展示無界模塊的頁面
  • 已經有心智造或 Minisite 頁面了，我們需要將當前模塊插入到頁面中

而我們採用瞭如下技術方案將無界的定投模塊插入到頁面中：

  • 用戶發起搜索，搜索請求到達服務端
  • 服務端請求返回時，若命中心智造和 minisite 場景，則返回對應場景投放的千葉頁面 URL
  • 服務端請求返回時，若未命中心智造和 minisite 場景，且命中有無界定投的模塊，則返回一個空頁面 URL
  • 服務端請求返回時，若命中有無界定投的模塊，則返回無界定投模塊信息，包括模塊名稱、模塊版本、模塊數據、展示規則
  • 客戶端創建長頸鹿 Weex 容器，渲染心智造、 minisite 頁面，或者空頁面，然後將無界定投模塊數據傳入 Weex 容器
  • 頁面渲染觸發 PI 執行，PI 獲取無界定投模塊，拉取定投模塊的代碼，創建實例後基於展示規則插入到頁面中

通過這樣的方式，最終渲染出當前場景下所有的模塊內容。另外在無界後臺我們有一套場景干預規則，解決場景交叉時進行一些手動干預。與此同時我們還提供了一套動態數據源機制，方便三方 HSF、TPP 數據能直接綁定到模塊上，實現模塊和數據解耦。

頁面渲染

頁面渲染如上述所示，依舊是通過斑馬。但與之前不同的是：

由於是模塊化搭建，頁面通過 PI 統一管控，頁面渲染依賴 PI 行為控制

對於長頸鹿容器，同樣是 Native 創建 Weex 容器渲染，但與之前渲染一個僅包含一個模塊的 Weex 頁面顯然是不同的，長頸鹿容器內渲染的也是一個真正的模塊化搭建的頁面

對於無界定投的模塊，提供了插入機制，解決了場景交叉時的展示問題

模塊管理

對於頁面內的通信，由於頁面通過 PI 統一管理，也就意味著模塊的各種能力都需要依賴 PI 來進行實現。由於這樣的樓層搭建一般通信結構都是扁平的，所以基本上通過模塊冒泡給 PI ， PI 下發給對應模塊即可。不過一般不建議模塊之間進行通信。

對於頁面和容器通信，我們是在 PI 中提供了一些封裝的能力，另外也可以自己調用容器提供的方法，與 Native 內嵌坑位的 weex-module 方案別無二致。

▐ 小結

由於長頸鹿內的技術方案已經脫離傳統意義上的前端，更多設計客戶端和服務端，是整個產品架構的設計，這裡很難講清楚，放一張圖可能清晰一點：

這裡前端技術又有了提升：

  • 實現了可視化搭建，為商家&行業提供了運營陣地
  • 為未來品牌&行業規模化提供了基礎
  • 提供了更好的行業共建機制
  • 動態數據源+無界模塊定投能力更加靈活多變

但這裡我們還是有幾個問題尚未解決：

  • 我們和會場越來越像了，但是我們和會場模塊並不互通（會場終於也用 Rax 了，但是他們用的是 Rax 1.0，我們還是 Rax 0.6）
  • 搭建的粒度太粗，基於樓層的搭建不夠靈活無法滿足行業差異化訴求（想象一下搜索列表作為一個模塊時，只是商品樣式改下就要重新開發一個搜索列表）

6、深度搭建時代

搜索最擅長的顯然就是搜索結果頁那種商品瀑布流，將用戶最想要、最有可能購買的商品推薦給他們。而這個能力也可以複用在諸如首頁、頻道、店鋪、專輯等等地方。而當我們要在很多不同的地方去搭建這樣一個列表頁時，在上面所說的樓層搭建中，我們一般會怎麼做呢？

  • 首先，新建一個 XXX 搜索列表模塊
  • 找一個無盡列表組件
  • 開發這個業務需要展示的商品坑位
  • 將無盡列表組件裡的 loadmore 中去請求 mtop 獲取下一頁數據
  • 將模塊發佈供搭建使用
  • 當有新的需求，就新增一個標記來判斷，基於標記換一個數據和商品坑位展示

如果這麼處理，若行業大面積規模化，會出現如下幾個問題：

  • 不同行業對於商品展示的要求不盡相同， 比如家電行業都會展示一些比較關鍵的產品屬性，而家裝行業則要更突出一些上門安裝的服務，這也就導致商品坑位的展示不同，同時也意味著大量的開發成本
  • 由於在一個倉庫內開發，共建會更加困難，所有需求都會更容易壓到自己頭上
  • 由於對接不同數據源，當業務邏輯出現交叉，前端代碼裡的各種判斷也會指數級增加

這些顯然都是我們不願意看到的，我們將組件的不同場景完全剝離，可以發現一些規律：

  • 無盡列表既然能抽出一個組件，那麼它肯定是可以複用的
  • 商品的展示和數據源的接入其實非常適合共建，讓行業的前端對接行業自己的數據=專業的人做專業的事

進一步提煉：

  • 交互可複用
  • UI 根據場景自定義
  • 數據根據場景自定義

所以我們有沒有可能在這個理念上更深一步進行搭建呢？

模塊化

答案顯然是有的，深度搭建的概念其實就是用來解決這種問題。將頁面上的模塊基於功能進行了重新定義：

  • 數據模塊：自身沒有任何 DOM 相關代碼，只負責處理數據。並且監聽一些事件，這些事件觸發時會加載新的數據並將其交給接收其數據的下游模塊，觸發刷新
  • 容器模塊：自身其實是一個容器，負責接收數據，並將數據下發給展示模塊去展示。但它自身不處理數據，而是通過觸發數據模塊定義的事件接口，告訴數據模塊需要做什麼。
  • 展示模塊：UI 展示模塊不處理任何邏輯，唯一做的就是拿到交互容器模塊給他的數據做展示。它內部甚至不處理任何事件，全都是以冒泡的方式到交互容器模塊進行處理。

通過這樣子的劃分，容器模塊控制了交互、展示模塊控制了 UI 、數據模塊控制了業務邏輯，比如一個包含篩選、商品流的列表更具體用 React hooks 來說就是：

  • 數據模塊本質上就是一個 reducer，對外暴露一個 [state, dispatch]， state 包含篩選信息、商品數據，交給容器模塊渲染；而 dispatch 則定義了 loadmore 、 changenav 等方法，供容器模塊調用
  • 容器模塊將導航和商品坑位定義為純展示區域，將篩選信息、商品數據傳給關聯的展示模塊進行渲染，並傳遞給他們一些如點擊事件的回調方法，當篩選發生點擊時通過 dispatch 觸發數據模塊的 changenav ，而頁面下拉到底時通過 dispatch 觸發數據模塊的 loadmore
  • 展示模塊就是純粹的無狀態 UI 組件，不同的組件用來展示不同的區域

合理的場景劃分至關重要，不同業務場景應該是數據模塊+容器模塊+展示模塊的組合，大多數情況下容器模塊和展示模塊可以複用，但數據模塊都是需要重新開發的。儘可能讓數據模塊來適配容器模塊和展示模塊，不改變交互就不改變容器模塊，不改變UI就不改變展示模塊。

在這裡模塊的編寫切換到了 Rax 1.0，與會場的開發方式一致以實現複用。Rax 1.0 針對 Web 做了很多優化，打出來的 bundle 也更加輕量，性能也好了很多，已成為天貓前端業務開發主要方式，因此不再需要使用 Prax 了。

頁面渲染

這裡和傳統搭建有個很重要的不同，模塊之間出現了關聯關係。比如如下關聯：

  • 數據模塊需要和容器模塊&展示模塊相關聯，需要將 state 和 dispatch 交給對應模塊
  • 展示模塊與容器模塊管理，容器模塊定義了區域，不同區域使用不同的展示模塊
  • 容器模塊甚至能和容器模塊做關聯，實現多層嵌套展示，這種情況一般用於純展示，不涉及交互（比如搜索列表篩選欄和商品瀑布流中間的空白區域）

那麼我們就需要一個全局方式來處理這些關聯。首先需要在搭建模塊的時候去指明模塊之間的關聯關係，比如我自己是定義了三個字端：

  • key ：模塊的唯一標識，給容器和數據模塊使用
  • data ：如果是容器或展示模塊，在其中填入所使用數據模塊的 key
  • container ：如果模塊在另外一個容器之內，則在其中填入所在容器模塊的 key

data 和 container 都是二段結構的，比如 data 的 a:b 代表使用 key 為 a 的數據模塊的 b 字端內的數據，而 container 的 a:b 則代表使用key為a的容器模塊，並將自己作為其 b 區域的展示模塊使用。

這幾個字端都會在腳手架裡，初始化時生成到 schema.$attr 中。而真正負責識別這些字端並將模塊串起來的自然是頁面初始化腳本—— PI ，PI 幹了如下事情：

  • 掃描所有數據模塊，並將其實例化，存放在一起，並從渲染模塊中將其剔除
  • 將剩餘的渲染模塊基於 container 組織成一個樹形結構，並對每個節點基於 data 關聯數據模塊，並將最上層進行渲染
  • 將子模塊相關數據傳遞給容器模塊，並提供一個默認渲染方法。容器模塊可以自己決定如何處理這些子模塊相關數據，可以修改、合併、覆蓋，修改完後使用渲染方法渲染即可。若其中還有還有下一級子模塊，重複進行如下操作

模塊管理

這裡模塊的通信有兩種，一種是基於公用數據模塊的通信，一種是基於 PI 的通信。

一個頁面上可以有多個數據模塊，而每個數據模塊都可以對接到多個容器/展示模塊（這也是為什麼 data 是二段結構的原因）。因此當多個模塊需要通信的時候，可以通過dispatch觸發數據模塊行為，數據模塊刷新數據影響另外一個模塊。

基於PI的通信一般都是針對一些特殊場景，比如一個很經典的例子——篩選列表篩選欄變形吸頂：當頁面有很多模塊，篩選列表並不是全屏大小而是跟著整個頁面一起滾動，當篩選欄滾出視野外，它需要變成一個相對簡單的結構吸頂。在這種場景下，由於吸頂層本身是一個絕對定位的層，而頁面文檔流也是一個層，兩個層之間不基於 DOM 操作的話，最合理的方式就是基於一個數據模塊進行狀態共享。如吸頂組件定義了吸頂時的樣式和正常的樣式，並且暴露一個創建數據模塊的方法， PI 在渲染吸頂組件時會單獨為其創建一個數據模塊來共享篩選欄狀態。

▐ 小結

為什麼要搞的這麼複雜？其實就是希望能將 UI 和交互做的更加純粹，當 UI 和交互都是針對單一場景、足夠簡單的時候，我們就能引入一些產品來進行提效，比如：

  • 使用描述式的方式來零開發定義交互，生成容器模塊，如 domagic
  • 使用描述式的方式來零開發定義 UI ，生成展示模塊，如 imgcook /魔切

通過這這樣的方式，最終頁面上大多數內容都變得可以複用，規模化的成本也就大大降低了。

7、智能時代

由於搜索還在深度搭建時代探索，以下是我基於團隊方向 YY 的，但我相信這個時代很快就會到來

但上述的各種方式終究還是人肉的，需要人力投入的。既然工具能讓我們以配置的方式解決交互和 UI ，那麼有沒有可能連配置都不需要，機器自動幫我們生成一個數據模塊或者交互呢？

這就是我們團隊正在做的，目前團隊的同學正在嘗試基於機器學習訓練模型實現智能化UI。只需要給出一些關鍵信息（比如要展示商品的哪些字端），就會自動化的生成對應的展示模塊或交互模塊。這些模塊使用到深度搭建之中，前端就只需要通過數據模塊定義展示的內容來源就能完成整個頁面的搭建。實現UI零成本。

而我們進一步擴展去思考，把大象裝冰箱分為三步，那麼把搜索智能化應該也只需要三步：

  • 第一步，我們告訴機器要展示什麼內容，它用最高效的方式展示給用戶
  • 第二部，我們告訴機器要展示哪一類的內容，它找出這一類內容最高效的方式展示給用戶
  • 第三部，我們告訴機器我們的場景，它找出符合該場景的內容展現給用戶

具體一點可能是：

  • 第一步，我們有了商品，讓模型去訓練出最好的商品展示方式
  • 第二步，我們定義了說這裡要展示一個權益模塊，那麼模型會給出最合適的展示內容，比如說展示一張店鋪券，是展示店鋪券的同時展示幾個爆款商品誘導用戶去下單，還是站在一個商品的維度將店鋪券結合品類券、津貼一起給出一個驚人的折扣價誘導用戶下單，抑或是找出當前用戶期望購買的商品總價最接近的店鋪券展示並誘導其湊單等等
  • 第三步，我們定義了一個場景，比如告訴機器這是一個 XX 頻道，那麼機器自動能分析母嬰頻道的主力消費人群類型，然後針對這個人群類型去推薦最合適的內容。比如假設我們要搭建一個母嬰頻道，機器會自動基於母嬰頻道主力消費人群是寶媽，而寶媽養娃最痛苦的是需求商品種類繁多、同時質量擔心得不到保證，因此給她推薦內容導購為主，同時穿插著一些品質上的背書比如用戶評價、品牌保證、平臺服務等等。

因此，我們可能需要一個三層的模型：

1、第一層基於不同的內容，找出最合理的UI界面來進行展示，也就是智能UI模型。

2、第二層則是需要產出這些內容，這裡分了三個維度：行業&平臺+品牌&店鋪+貨品。

  • 針對行業差異化的訴求，行業運營決定這裡需要展示的行業內容的類型和形式，比如剛才說的權益帶商品的模式，那麼就要通過行業表達模型找到最合適的權益以及結合這個權益之下最合適的貨品；
  • 針對純粹的商品展示，則是將人貨場的匹配做到極致，找到這個用戶目前最想要的商品；
  • 針對品牌&店鋪，品牌需要塑造品牌在消費者側的心智，對應的品牌會有自己的表達，比如品牌想要將一個新品打造成爆款，那麼對於這個爆款的打造就需要一個品牌表達模型來輔助他們運營；

3、第三層則更為複雜

  • 行業內容模型其實就是行業運營經驗的沉澱，將運營目標具像成不同的場景，再將該場景下的運營經驗具像化成內容，這裡就需要行業側各種類型的內容作為輸入，而模型則需要基於以往的各種組合後的運營經驗，給出最合適當前場景的內容組合；
  • 品牌內容模型也是類似的，整合吸收品牌在整個淘系體系下的各種資源投放，為品牌&店鋪的運營提供一些行之有效的運營建議（推薦模塊），提升品牌運營的效率，降低運營成本；
  • 對於人貨匹配則是再拆分了三個模型，分別去處理貨品、場景和用戶；

貨品模型：從貨品中提煉出貨品中的最吸引人的一些特質，比如賣點、好評、銷量，以及將貨品結構化，方便貨品在不同的地方展示；

場景模型：基於用戶的實時行為來定義當前用戶的購買場景，他可能想買一臺電視所以瀏覽了很多電視，抑或是在一家賣電視的專賣店裡面找同型號線上價格，抑或是在過年準備給父母購置一臺電視作為年貨等等；

用戶模型：基於用戶的歷史行為，將用戶劃歸某個類型的人群，進行人群精細化運營。比如一個用戶經常購買年輕人的潮流酷玩，但每到父親母親節或者特定時間就會購買一些長輩禮物的品類，說明是個有孝心的boy/girl，那麼針對這類孝順寶寶就可以在不同的節日或其他重要日子推薦一些給長輩送禮的內容

那麼我們都有哪些數據呢？

  • 行業&平臺：行業有自己的沉澱，平臺有各種營銷玩法
  • 品牌&店鋪：品牌同樣有主打新品、限時優惠、專題活動、線下門店等等
  • 貨品：各種商品沉澱下來的屬性、評價、賣點等等
  • 用戶離線：用戶的各種行為歷史，以及手動收集的一些用戶檔案
  • 用戶實時：比如它最近的一些行為，從哪進入頁面，在什麼地理位置，什麼時間點

One More Thing

由於我們不是活在未來，因此無法斷定前端的未來到底是怎樣的，但前端不會停止對於未來的探索。而這個未來不只是技術的未來，也是業務的未來。技術和業務是我們前進的兩條腿，業務的訴求推動技術邁向新的高度，而更先進的技術也為業務帶來更多的想象空間。
