通過組合漏洞進行權限提升

2020-10-29
| 科技

介紹

這篇文章主要講述了由NCC Group研究人員所發現的兩個通過COM本地服務進行非法提權的漏洞。第一個漏洞CVE-2019-1405是COM服務中的一個邏輯錯誤，可讓本地普通用戶以LOCAL SERVICE身份執行任意命令。第二個漏洞CVE-2019-1322是一個簡單的服務配置錯誤，可讓本地SERVICE組中的任何用戶重新配置以SYSTEM權限運行的服務（此漏洞也被其他研究人員發現）。當以上兩個漏洞結合在一起時，就允許本地普通用戶以SYSTEM權限執行任意命令。

COM背景

為了幫助大家更好的理解漏洞，我們將從一些基礎知識開始講起，如果你熟悉Windows上COM的基本概念，可以跳過本節。

組件對象模型（COM）是微軟於1993年推出的一套軟件組件接口標準。一般來說，COM是一種讓開發人員以與開發語言無關的方式使用某些軟件組件的體系結構。COM的眾所周知的用法包括ActiveX和OLE（例如，在Word文檔中嵌入Excel電子表格）。此外COM在整個Windows生態系統中被微軟官方和眾多第三方應用所使用，它是真的無處不在！

COM對象是通過存儲在Windows註冊表中的稱為CLSID的全局惟一標識符(GUID)來標識的（許多COM對象也在註冊表中被命名，但是名字只是鏈接到相應的CLSID）。一個CLSID條目包含COM子系統在創建對象實例時使用的信息。

COM會對外公開一個或多個接口，隱藏對象的實現細節。本質上，一個接口定義了對象所支持的一系列方法，而沒有規定實現方式。接口在調用方法的代碼和實現方法的代碼之間存在清晰的界線。每個COM對象都支持IUnknown接口，這個接口用於通過AddRef、Release和QueryInterface方法向其他受支持的接口提供引用計數和類型轉換。接口通常通過Microsoft接口定義語言（MIDL）進行定義，並通過被稱為IID的GUID進行標識。同樣，這些都存儲在Windows註冊表中。但是請注意，一個COM對象所支持的特定接口並沒有在註冊表中列出。

一般來說，COM對象可以分為兩類——一類是在調用進程內部創建的，另一類是在進程外部創建的。出於權限提升的目的，以本地服務身份所執行的進程外部COM對象就是我們的攻擊點，Windows的默認安裝中包括很多這樣的對象。

有許多工具可用來檢查在Windows實例上註冊的COM對象和接口。也許其中最好的是由谷歌Project Zero的James Forshaw編寫的OleViewDotNet。在這篇博客文章中，我們不會過多介紹這個工具的特性，如果你有興趣瞭解更多關於COM的知識，可以從James編寫的相關文章開始。

UPnP Device Host服務（CVE-2019-1405）

UPnP Device Host服務在Windows操作系統（從Windows XP到Windows 10）上默認啟用，並以NT AUTHORITY\LOCAL SERVICE權限運行（該服務在眾多Windows版本上默認安裝，但在某些版本上默認不啟動）。本服務中的許多COM對象都以本地服務權限執行，如下圖由OleViewDotNet所展示的：

Windows中的任何安全對象基本都能設置COM服務的訪問控制信息，而OleViewDotNet能夠顯示這些信息。下圖顯示了UPnP Device Host服務的權限設置：

可以看出，DACL阻止網絡用戶在此服務中啟動COM對象，但是所有本地用戶都可以這樣做，這顯然是為了防禦非法提權。

這個服務託管的UPnPContainerManager和UPnPContainerManager64COM對象都實現了IUPnPContainerManager接口。雖然這個接口缺少文檔說明，但是OleViewDotNet能夠找到關於它方法的某些信息。此外，Microsoft還為核心操作系統組件提供調試符，從而產生了以下關於接口的初始定義（作為MIDL的一個片段）：

[ object, uuid(6d8ff8d4-730d-11d4-bf42-00b0d0118b56), pointer_default(unique)]interface IUPnPContainerManager : IUnknown { HRESULT ReferenceContainer([in] wchar_t* string1); HRESULT UnReferenceContainer([in] wchar_t* string1); HRESULT CreateInstance( [in] wchar_t* string1, [in] GUID* guid1, [in] GUID* guid2,  IUnknown** pobject); HRESULT CreateInstanceWithProgID( [in] wchar_t* string1,  [in] wchar_t* guid1,  [in] GUID* guid2,  [out] IUnknown** pobject); HRESULT Shutdown();}

初步看，IUPnPContainerManager所公開的CreateInstance方法似乎特別有趣。對於有一些COM編程經驗的人來說，這個方法的名稱和API都應該與用來創建任何COM對象的標準CoCreateInstance方法有關（儘管第一個參數的目的不是很明確）。但不幸的是，使用一個眾所周知的CLSID和IID以及將一個虛擬字符串作為第一個參數來調用CreateInstance會產生一個未知錯誤代碼。

此時，我們別無選擇，只能打開反彙編軟件，查看實現該方法的代碼。幸運的是，這個過程很簡單（OleViewDotNet標出了實現這個類的模塊和接口方法的偏移量），我們發現大部分運作是在upnphost.dll中的CContainerManagerbase::CreateInstance方法中進行的。

傳遞給CreateInstance的字符串參數（在這段代碼中被標記為a2）被複制到v7，然後通過調用HrAssign來初始化Block對象。然後將Block傳遞給HrLookup，實際工作將由第33行中的虛擬函數調用執行。通過調試這段代碼我們可以很快發現，當將一個偽字符串作為第一個參數傳遞給CreateInstance時，我們無法檢查HrLookup的結果。顯然，這個參數的內容很重要。

下一步是拆分HrLookup函數，但是這並不是易事，我們不得不慎重考慮。由於在實踐中，我們通過IUPnPContainerManager接口與這段代碼進行交互，因此我們決定查看其他公開的方法，看是否能得到靈感。特別是對CContainerManagerbase::ReferenceContainer方法的反編譯，該方法實現了ReferenceContainer：

這個方法的開頭看起來與CreateInstance非常相似，但是在這種情況下，如果調用HrLookup失敗，就會轉而調用HrInsertTransfer方法，將被創建的Block對象傳遞給ReferenceContainer。因此，我們假設在調用CreateInstance之前，我們可能需要調用ReferenceContainer，經過測試表明這確實如此。基於以上，IUPnPContainerManager接口的更進一步定義如下所示：

[ object, uuid(6d8ff8d4-730d-11d4-bf42-00b0d0118b56), pointer_default(unique)]interface IUPnPContainerManager : IUnknown { HRESULT ReferenceContainer([in] wchar_t* containerName); HRESULT UnReferenceContainer([in] wchar_t* containerName); HRESULT CreateInstance( [in] wchar_t* containerName, [in] GUID* clsid, [in] GUID* iid,  IUnknown** pobject); HRESULT CreateInstanceWithProgID( [in] wchar_t* containerName,  [in] wchar_t* progID,  [in] GUID* iid,  [out] IUnknown** pobject); HRESULT Shutdown();}

調用ReferenceContainer方法代入一個任意字符串，然後後將同一個字符串代入到CreateInstance方法，就將在本地服務進程中創建由clsid參數標識的COM對象的實例，並將此對象中由iid標識的接口的引用返回給pobject對象中的調用。

實際上，這就讓低權限的本地用戶可以使用任何已註冊的進程內COM對象，就好像是用戶NT AUTHORITY\LOCAL SERVICE執行進程外本地服務一樣。特別是，它還可以創建一個Windows script Host Shell實例，獲得該對象的IWshShell接口的引用。最後通過從這個接口調用Run方法，利用UPnP Device Host程序執行任意命令。

在Windows 10上，UPnP Device Host被設置為在沒有模擬權限的情況下，以用戶NT AUTHORITY\LOCAL SERVICE（其中ServiceSidType設置為SERVICE_SID_TYPE_UNRESTRICTED）執行。以下截圖顯示了啟用了UPnP Device Host服務的svchost實例的進程屬性，可以確認SeImpersonatePrivilege並沒有被啟用：


不幸的是，這攔截了一些眾所周知的提權方法。但是，服務用戶相比普通用戶擁有更多的權限，比如NT AUTHORITY\SERVICE組的成員權限更大（這和第二個漏洞關聯很大）。

在Windows XP中，進行這種級別的配置是不可能的，因此提權到NT AUTHORITY\SYSTEM非常簡單。

Update Orchestrator（CVE-2019-1322）

在發現了上述漏洞後，我們自然想知道獲得的權限能否允許我們配合另一個漏洞來完全控制主機。對可能使用了默認訪問控制設置的對象進行了全面掃描，很快便找出了Update Orchestrator服務。

Update Orchestrator服務在Windows 10和Windows Server 2019上以NT AUTHORITY\SYSTEM權限默認啟用。而對SysInternals工具accesschk.exe的調用顯示了在Windows 10 1803至1903版本的Update Orchestrator服務所實施的訪問控制。

UsoSvc Medium Mandatory Level (Default) [No-Write-Up] R NT AUTHORITY\Authenticated Users SERVICE_QUERY_STATUS SERVICE_QUERY_CONFIG SERVICE_INTERROGATE SERVICE_ENUMERATE_DEPENDENTS SERVICE_START SERVICE_USER_DEFINED_CONTROL R BUILTIN\Administrators SERVICE_QUERY_STATUS SERVICE_QUERY_CONFIG SERVICE_INTERROGATE SERVICE_ENUMERATE_DEPENDENTS SERVICE_START SERVICE_STOP SERVICE_USER_DEFINED_CONTROL RW NT AUTHORITY\SYSTEM SERVICE_ALL_ACCESS RW NT AUTHORITY\SERVICE SERVICE_ALL_ACCESS

以上輸出的最後兩行顯示，NT AUTHORITY\SERVICE用戶組中的用戶可以完全訪問控制該服務。特別是這個組中的用戶能夠停止、重新配置和啟動服務，這就讓用戶以NT AUTHORITY\SYSTEM權限執行任意命令。

例如，以下命令（當NT AUTHORITY\SERVICE組中的用戶執行時）將向目標機器添加名為_tmpAdmUser的管理用戶(密碼為H.jqt41Kz!a!)，並將該服務恢復到默認狀態：

sc stop UsoSvc sc config UsoSvc binpath= "cmd.exe /c net user /add _tmpAdmUser H.jqt41Kz!a! &" sc start UsoSvc sc stop UsoSvc sc config UsoSvc binpath= "cmd.exe /c net localgroup administrators /add _tmpAdmUser &" sc start UsoSvc sc stop UsoSvc sc config UsoSvc binpath= "C:\WINDOWS\system32\svchost.exe -k netsvcs -p" sc start UsoSvc

我們檢查了一些Windows服務，結果表明，所有LOCAL SERVICE或NETWORK SERVICE身份的用戶都能夠進行這種攻擊。再綜合UPnP Device Host服務的缺陷，通過CVE-2019-1405和CVE-2019-1322，可讓任意本地用戶擁有系統權限。
