5G 網元如何瞬時處理海量數據報文（2）

2020-11-14
| 科技

之前的文章《5G 網絡設備中的軟件如何做到瞬時處理海量數據報文（1）》提到，要想在最短的時間內處理掉儘可能多的數據報文，必須藉助多線程技術。於是，說到了線程（Thread）、隊列（Queue）和任務（Task）的部署關係。這裡我們繼續這一話題，並初步探究下5G 網元的工作原理。

並行多任務下的線程、隊列部署關係

在 5G 網元中，如果只有一個任務，那麼這個任務必然非常龐大，且不好管理。於是，我們需要將一個總的任務劃分成很多小的子任務，於是就出現了上圖的部署方式。

順序多任務下的線程、隊列部署關係

在 5G 網元中，一個總任務的執行往往需要分為多個階段（Phase）。於是，我們在階段1（Phase 1）的時候執行 Task 1，在階段二（Phase 2）的時候執行 Task 2，那麼 Task 1 和 Task 2 就是順序的關係。也就是說，報文經過 Task 1處理完，需要扔到 PacketInQueue 2 裡，然後 Task 2 從隊列 PacketInQueue 2 裡取報文，執行 Task 2，最後再從出口發送出去。

報文處理狀態機

在 5G 網元中，對報文的處理一般都採用狀態機模式，每個狀態對應一個處理動作，狀態之間的跳轉需要滿足特定的條件。上圖中，狀態 1 可以跳轉到狀態 2 ，觸發條件是C12. 狀態 1 也可以跳轉到狀態 3，觸發條件是 C13.

while (1) {

switch (條件)

{

case 條件 1： 動作 1

case 條件 2： 動作 2

┇

case 條件 n： 動作 n

default: 動作 n+1

}

}

比如，

報文1進入 5G 網元后的處理路線為：初始狀態 0 -> 狀態 1 -> 狀態 2 -> 狀態 3；

報文 2 進入 5G 網元后的處理路線為：初始狀態 0 -> 狀態 2 -> 狀態 3.

這怎麼理解呢？舉個例子，如果你訪問某個黃色小網站，5G 網元識別出來之後，肯定將你的報文阻塞掉。如果你訪問騰訊系 APP，而且你還買了騰訊大王卡，那麼你的報文將暢通無阻，電信運營商也不會對你的流量計費。

雖然說 5G 網元內部基本處理邏輯是狀態機模式，但是，可以想象的是：5G 網元系統內部很複雜，涉及的狀態非常多，而且涉及的條件也非常複雜，如果只是採用簡單的狀態機模式，顯然無法管理，很容易失控。那麼怎麼管理這些狀態和跳轉條件呢？藉助 Go 語言中事務（Transaction）和 事件（Event）的概念 會是一個不錯的解決方案。詳細內容且聽下回分解。
