網格邊緣試探——服務網格的探索與實踐

2020-10-29
| 科技

導讀

近些年來，以服務網格為代表的雲原生技術成為技術開發人員的熱門話題。從某種意義來說，雲原生運動改變了微服務體系架構中應用程序設計與開發的方方面面。服務網格作為雲原生中最典型的技術代表，它解決了以前微服務體系中搭建技術門檻高、代碼侵入性強等許多棘手問題。如今，應用以 Istio 為代表的服務網格框架可讓開發人員輕鬆打造一個微服務監控和管理系統。但是由於服務網格還面臨著許多待解決的技術問題，將服務從原來的微服務體系遷移到服務網格中作為生產環境需要付出巨大的成本，且存在風險。基於以上情況，愛奇藝號技術團隊在服務網格方面不斷探索，目前有部分業務運行在服務網格中，本文主要分享愛奇藝在探索中的實踐經驗。

背景

應用在很久以前都是單體結構的，但隨業務發展越來越多，會導致工程出現臃腫複雜的問題，進而促使了微服務設計思想的推出。微服務設計思想的核心是將應用進行拆分，按模塊劃分成一個個微服務，每一個微服務單獨開發和部署。當應用功能簡單時，劃分的服務較少，此舉確實提高了開發速率和維護效率。但當功能越做越多，服務劃分也隨之越來越多時，對大批微服務進行管理成為令開發人員頭疼的問題。

當鏈路追蹤、熔斷、監控、服務發現等問題都需要解決時，服務網格出現了，如果你是第一次接觸服務網格，難免會疑惑服務和網格兩個詞彙是如何關聯的，那服務網格的含義到底是什麼呢？這套設計能帶給開發人員什麼樣的驚喜？下文將帶大家尋找答案。

"A service mesh is adedicated infrastructure layer for handling service-to-service communication.It's responsible for the reliable delivery of requests through the complextopology of services that comprise a modern, cloud native application."

這是服務網格的提出者William Morgan給出的定義——服務網格是管理和監控各個微服務之間流量的'基礎設施層'，用淺顯的方式來說，以前的微服務體系一般是將各個服務單獨部署在自己的虛擬機內，向服務發現中心上報自己的位置和獲取其他服務的位置，再通過Http 或者 Rpc 的方式調用其他服務。這一方式使得業務代碼插入些無關業務的代碼邏輯，引入各種混亂的jar包。而服務網格推翻了這些，當應用部署在'基礎設施層'時，它對應用做了一個對其透明的輕量級網絡代理，即服務發出的請求和接收的請求都被代理攔截，並且服務對這個代理沒有感知。

所有服務間的通信都必須通過其代理完成，這樣相當於在'基礎設施層'中加入了代理層，服務代理組成一個大網，它們將服務編織在一起，就像一個個網格一樣。這樣做有什麼好處呢？即代理層代替服務進行網絡通信，那麼服務發現，負載均衡、路由等微服務需要關注的點就可以全部遷移到'基礎設施層'來做文章了，服務本身只關注它的業務代碼就可以了。這樣的設計模式是不是十分巧妙？



應用實踐

基於對服務網格的興趣，我們考慮到愛奇藝號有不少業務場景如果部署在服務網格中，其服務解決方案會十分的簡單，於是我們搭建了測試和線上兩套服務網格集群，環境之間互相隔離；並且搭建了相應的監控體系及Dashboard部署系統；目前愛奇藝號部分後臺服務已經部署在集群中進行小規模運行。

在實踐過程中，我們對業內關注度比較高的服務網格方案進行了調研，最終我們選擇在Kubernetes 集群上搭建服務網格Istio。Kubernetes 是近幾年十分火熱的容器編排應用，它的底層設計與服務網格設計思想匹配度高，而 Istio 是Google,IBM 和Lyft三家公司協同開發的一款完全遵從服務網格設計思想的框架。作為明星公司的開源框架，Istio推出後迭代十分之快，如今已逐漸趨於成熟。在推出的兩年多裡，Istio吸引了大量的開發者，關於它的技術資料也較為豐富。基於以上因素考慮，我們搭建了這樣的一套測試+線上集群的私有云環境，並將部分業務服務遷移到上面運行。下面主要介紹我們在實踐中用到的服務網格特性和經驗，主要包括服務的流量控制，自研的Dashboard 和網格的監控等。


1、流量管理



上面的圖展示了網格內流量管理中幾個關鍵概念的關係。

- Gateway: 在網格邊緣接受外部訪問，將外部流量導入網格內部。

- VirtualService：Istio 流量治理的核心，大部分流量控制規則由其配置，如重試、金絲雀，路由等。

- DestinationRule：定義了服務的版本子集,Istio支持服務多個版本同時運行，VirtualService通過路由匹配將流量轉發給相應的服務版本

- Service：Kubernetes 提供的資源組件，最終由它將請求均衡轉發到應用的Pod中。

- Pod: Kubernetes中能夠創建和部署的最小單元，應用的容器運行在其裡面。

在以前的微服務體系中，一般網關使用 Nginx等負載均衡技術進行路由管理，將流量轉發到各個服務中，如果需要做一些其他的流量控制功能，如重試、灰度等，單靠Nginx 是無法實現的，這時候一般會由自己的中間團隊在此基礎上進行擴展，來實現這一部分功能。但對於一些小公司而言，這些工作並沒有很多人力來做，使用Istio 只需通過簡單的配置上面幾個組件，就可以輕輕鬆鬆實現這些功能，這對小規模公司來說是十分友好的。

目前愛奇藝某些服務的流量控制是基於Mashape/Kong 開發的API 網關，並且已經在API 網關做了安全、鑑權、限流、路由策略等一系列流量控制功能。在本次實踐中，為了保留現有API網關的鑑權、限流等功能，我們通過Consul Agent方式，將服務網格入口機註冊到API網關，同時通過服務網格的流量管理特性，增加故障注入、重試、熔斷、多版本灰度等功能。

那服務網格是如何實現流量管理的呢？用一個詞概括，就是"代理"。

服務網格的核心是代理，Kubernetes 在將應用的容器運行在Pod 中時，Istio 會以sidecar 的方式在其 Pod 注入一個伴生代理服務--envoy，

在Kubernetes 中，由於運行在同一個Pod 內的容器共享網絡命名空間及ip地址，所以envoy 可以全權代理應用服務對外的網絡通信，並且使業務服務對此毫無感知。Envoy 作為一款高性能的網絡代理，已經在生產環境中進行了長期驗證，Istio 的流量管理功能默認是基於Envoy實現的。

在上面的幾個組件中，VirtualServce 是整個流量控制鏈路的核心，每個Pod的Envoy會監聽並拉取VirtualService的流控配置，並在代理流量的時候根據配置進行轉發。在實踐中，我們控制網格內的流量，也主要通過配置 VirtualService 的方式來實現。下面列舉一些流量控制功能。

1） 流量路由

流量路由是網格提供的最基本的轉發功能。愛奇藝號內部業務微服務均綁定到同一域名的Gateway，並在VirtualService中配置路徑及服務的對應關係，並根據配置進行轉發，服務網格會將對應的請求轉發到相應的服務中。

2）服務多版本

服務上線時，考慮到風險問題，直接將老版本直接升級是非常危險的。所以一般做法是新老版本同時在線，新版本切入一部分到老版本中，待確定無誤後再逐步擴大新版本流量，最終線上全部更新為最新版本。一般服務發佈系統提供的灰度功能相對較單一，其僅支持灰度一個版本，且灰度版本的流量權重是根據容器比例來決定的，流量權重粒度不夠細膩。相反，服務網格提供的灰度發佈功能十分強大，它不僅支持灰度一個版本，而且可以同時讓一個服務的多個版本同時運行，並配置不同的流量權重，這在灰度上線/金絲雀上線/藍綠部署/AB測試等業務場景中十分有用。

3）流量鏡像

流量鏡像這個功能用的比較少，但是在某些特殊的場景下十分有用。之前為某個業務服務新增一個鑑權接口時，由於QPS（QueriesPer Second）相對較高，為了減輕基礎服務調用的壓力，我們給接口加了分佈式緩存。在鑑權接口上線之前，我們需要對緩存進行預熱。在服務網格中，我們只需部署一個服務的臨時版本，之後臨時版本根據線上的流量不斷更新用戶權限信息到緩存中，使緩存預先預熱，完成後再進行上線。除了緩存預熱外，流量鏡像還可以幫助我們進行下列操作：通過預生產環境測試來觀察新系統對生產環境流量的處理能力；複製請求日誌以進行安全分析；複製請求用於數據科學研究。

4）其他

除了以上列舉的特性外，服務網格Istio 還支持重試、故障注入、服務熔斷、HTTP重定向、HTTP 重寫等多種流量管理功能，且只需簡單的幾行 yaml 配置即可，由於我們用到的比較少，這裡不一一介紹了，感興趣的同學可以去Istio 查詢相關資料。


2、Dashboard

在上一節可以看到，在 Kubernetes 中應用都是以 yaml 文件進行描述和部署，Kubernetes 通過解析 yaml 文件的應用信息來分配集群資源運行應用容器。對不熟悉 Kubernetes 的開發人員來說，這種部署方式不太友好。所以，Kubernetes 提供了開源的Dashboard，它以可視化頁面的方式來部署應用。而當我們在Kubernetes 基礎上加了一層服務網格時，Kubernetes 的 Dashboard 就完全不夠用了。為了對網格應用更加靈活精細的控制，我們結合愛奇藝內部資源開發了一套團隊內部使用的定製版 Istio Dashboard，方便團隊內部不熟悉的開發者將服務部署在服務網格中。

- App級操作

我們提出了 App 這一個概念，App對應一個應用，其中包括一套 Deployment、Service、VirtualService、DestinitionRule、Gateway，在Dashboard 中的所有操作都是針對App 來執行的，用戶無需關心 Gateway、VirtualService 等這些底層資源的概念。

- 一鍵部署

一個接口便可將應用部署在集群中，並且通過域名直接訪問。

- 功能齊全

灰度、重試、流量配置等功能通過簡單的操作便可實現。

為了實現App 這一個概念，將一個應用的涉及的資源類型進行了封裝，並定義了規範來保證各個應用間資源的隔離。

1）一個團隊在 Istio 中擁有自己的命名空間，在同一個命名空間部署的應用採用同一個 ingressgateway 網關。

2）每一個應用擁有自己獨自的 VirtualService ,一個空間內的 VirtualService 共用一個網關，各個應用在自己的 VirtualService 做流量控制。

3）每一個應用擁有自己獨自的 DestinatioRule，但是可以擁有多個版本的 Deployment（Deployment 在Kubernetes 中類似於App 這個概念，是應用整個資源操作的封裝體，但不包括服務網格的資源類型），一個應用的多個Deployment 通過標籤的方式進行關聯。比如查這個應用的所有版本，就是根據Deployment 的標籤來聚合查詢的。

4）在Dashboard 部署應用時，後端自動創建其所有的資源類型，並根據App 名字關聯在一起。當刪除時，會自動刪掉其關聯的所有資源。

我們的Dashboard 後端接口基於開源的 Istio client進行開發, 將各種資源的操作通過 Http/Rpc 方式推送給Kubernetes集群，從而無需再編寫 yaml 文件部署應用。


3、監控

除了讓服務在網格內運行，對其進行監控也是必不可少的。我們來看看在服務網格中它的監控和平常微服務體系監控有什麼不同。



前面提到 Envoy 代理了應用的網絡，這時候腦洞大開的Istio 開發者在此基礎上設計了一個 Metric 數據收集和策略控制組件Mixer。那它有什麼用處呢？Envoy 代替應用發出請求時，先訪問 Mixer 來獲取目標應用的網絡情況，然後再與目標應用的Envoy 通信，通信完之後再將請求相關信息上報給 Mixer。這個Mixer 就可以收集到網格內所有應用通信的情況。Mixer 收集到各種指標數據有什麼用呢？聰明的 Istio 提供了可擴展的 Adpater，將這些指標數據適配給APM 後端。網格使用方完全可以結合自己的業務場景來自定義自己的 Adpater，並在自己的 APM 後端對這些Metric 做一些自己的事情，如實現特定業務場景的報警和容災策略。

Prometheus 作為業內開源的十分流行的監控應用，在Istio 的安裝包默認包含了 Prometheus 相關Adapter

和對應的APM 後端，只要啟用 Istio,網格內運行的服務相關數據就可以被Prometheus 收集到了，真正做到了開箱即用。



僅僅監控服務間的通信遠遠是不夠的，集群的各種資源數據指標也要採集進行監控。我們在網格內部部署了各種第三方 Exporter ，如 node_exporter，收集集群的CPU、內存等 Metric 數據。kube-state-metrics，收集Pod,Container，Deployment等資源對象的狀態指標。這些第三方 Exporter 將收集到的 Metric 數據暴露給 Prometheus 進行讀取存儲。

Promethus 採集和存儲各種需要監控的指標後，我們就可以在 Grafana 進行可視化查詢和設置報警。當指標超過閾值時，其再將報警信息以 Webhook 方式推送給公司內部的統一報警平臺，報警平臺通過郵箱、熱聊（公司內部聊天工具）、短信等方式將報警信息推送給訂閱者。



這是集群在 Grafana 監控的Cpu、內存等指標。

另外，為了監控服務的運行錯誤，如報空錯誤等，我們結合公司的 Venus日誌採集系統，將服務中的日誌採集到其中，此舉一方面通過 Venus 將日誌保存在 ElasticSearch 中來提供日誌查詢，另一方面我們單獨開發了一套'關鍵節點報警服務'。該服務基於 Flink 進行開發， 服務持續消費Venus 中的日誌信息，並解析出日誌中的錯誤信息來對訂閱者進行報警。運行在網格中的服務一旦在日誌中打印了報錯信息，訂閱者在5分鐘內就可收到報警信息，並可以通過Rover/Kibana進行查看修復等操作。


4、其他特性

Istio 的三大特性之一--安全，它可以通過插拔的方式將安全證書配置在 Envoy中，來加強服務間通信的安全性；

其次，Istio 的可擴展性強，它的 Mixer 組件所收集的幾十上百個Metric 數據可以通過可擴展的 Adapter 的方式提供給自己的 APM 後端使用，其中存在較大的實踐空間；

另外，'混沌工程'在服務網格上也可以方便的實現，這只需要修改相應的VirtualService和DestinationRule配置，就可以模擬調用延遲及調用失敗等情況；

最後，得益於Kubernetes的CRD擴展機制，我們可以很簡單的實現各種擴展功能，例如：安全數據、聚合網關等功能。

總結與規劃

服務網格是未來微服務的發展趨勢，但是該領域現在仍面臨著很多挑戰，比如將服務運行在服務網格中的時候，難以界定開發人員和運維人員的界限；由於服務網格的高度中心化管理，可觀察性、流量控制和安全的三大特性儘管體現了對開發者友好的特徵，卻也給運維人員帶來了很大的挑戰。服務網格技術目標遠大，旨在解決複雜的微服務架構導致的維護問題，而這樣的高目標也對開發人員有著更高的技術要求。儘管我們面對很多挑戰，但通過探索，我們真切感受到服務網格的功能強大和架構設計的巧妙之處，它解決了我們之前面對的流量控制弱、監控和業務代碼耦合等問題，這也是我們堅持繼續探索服務網格的原因。

從最近各大公司的服務網格的產品動態可知，服務網格上雲+託管成為一大趨勢，開發者使用託管版本的 k8s 和 Service Mesh，將底層系統的複雜度交給雲廠商，這樣客戶只需要在雲的基礎上便可享受 Service Mesh 技術帶來的使用便利和強大功能。面對如此的行業發展形勢，我們相信將來會有越來越多的開發者將自己服務部署在服務網格上運行。

目前，愛奇藝號在服務網格的初步探索已經完成，實現了服務在網格內的正常部署和常規操作，並搭建了監控和報警附屬設施來保證服務的正常運行。後續我們將在此基礎上進行深入探索，在可靠性、可控性、可用性方面進一步優化，如跨集群路由，制定業務場景網絡策略，擴展 Dashboard 功能等。
