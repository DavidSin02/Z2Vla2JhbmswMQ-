IEEE802.15.4協議—幀類型和數據傳輸模型

2020-10-29
| 科技

1. 數據傳輸模型

存在三種類型的數據傳輸方式。第一個，設備發送數據給協調器，第二個，協調器發送數據給設備，第三個，對等設備之間的數據傳輸（比如ZigBee網絡中的路由之間的數據傳輸）。

1.1 設備發送數據給協調器

當設備把數據發送給支持信標的協調器時，它首先需要監聽網絡信標幀，當找到信標幀時設備與超幀結構同步，在適當的時間設備發送數據給協調器。協調器接收到數據後向設備回覆一個ACK表示已經成功收到，具體流程如上圖所示。

當設備把數據發送給不支持信標的協調器時，只需要把數據發送給協調器，協調器收到數據時給設備回覆一個ACK表示成功接收，具體流程如上圖所示。

1.2 協調器發送數據給設備

在啟用信標的網絡中當協調器希望把數據發送給設備時，首先，協調器在網絡信標中指示數據消息已經掛起。其次，設備定期監聽網絡信標，如果有消息等待，就發送MAC請求數據幀。其次，當協調器收到設備發送的MAC請求數據幀時，回覆ACK幀給設備確認數據請求的成功接收。然後，協調器把準備好的數據發送給設備，設備收到後回覆一幀ACK。具體流程如上圖。

在不啟用信標的網絡中當協調器希望將數據傳輸到設備時，首先，協調器把準備發送給設備的數據暫時存儲以便設備發送數據請求幀來請求數據。其次，當協調器接收到設備發送的數據請求幀時，協調器先回復一個ACK給設備表示收到數據請求。然後，協調器把數據發送給設備。最後，設備接收到數據時給協調器回覆一個ACK表示接收到數據。具體流程如上圖所示。

1.3 對等設備之間的數據傳輸

在對等網絡中，每一個設備都可以和它無線電範圍內其他設備進行通信，為了實現這一點需要網絡中的設備保持同步，比如在ZigBee網絡中路由設備會週期性的廣播路由信息來同步。


2. 幀結構介紹

設計幀結構主要考慮兩方面，其一是使幀結構儘可能簡單，其二是保證在具有噪聲的信道中傳輸時保持健壯性。每一層的協議都會在這個結構上添加本層相關的頭部和尾部。IEEE802.15.4標準定義了4種幀結構：

— A beacon frame（信標幀）, 用於協調器發送信標幀

— A data frame（數據幀）, 用於發送數據

— An acknowledgment frame（ACK幀）,用於確認接收成功

— A MAC command frame（命令幀）, 用於處理所有MAC對等實體控制傳輸

2.1 Beacon frame（信標幀）

上圖顯示了信標幀的結構，它起源於MAC子層內部。協調器可以在啟用了信標的PAN中傳輸網絡信標。幀頭（MHR）包含幀控制域、幀序列號、地址域、附加安全頭部。MAC有效負載包含超幀規範、GTS字段、掛起地址字段、信標有效負載。幀尾（MFR）包含一個16位幀校驗序列(FCS)。MHR,MAC Payload和MFR一起形成MAC信標幀(MPDU)。然後將MAC信標幀作為PHY服務數據單元(PSDU)傳遞給PHY layer（物理層）的PHY負載。

2.2 Date frame（數據幀）

數據有效載荷被傳遞到MAC子層，被稱為MAC服務數據單元(MSDU)。MAC有效負載以MHR為前綴並附加MFR。MHR包含幀控制域，幀序列號，地址域，附加安全頭部。MFR包含16位的FCS效驗。MHR、MACPayload和MFR一起構成了MAC數據幀(MPDU)。MPDU作為PSDU被傳遞到PHY成為PHY的有效載荷。

2.3 Acknowledgment frame（ACK幀）

上圖顯示了ACK確認幀的結構，它起源於MAC子層內部。ACK確認幀由幀頭（MHR）和幀尾（MFR）組成,它沒有MAC有效負載。MHR包含幀控制域、幀序列號。MFR包含16位的FCS效驗。MHR和MFR一起構成MAC ACK確認幀（MPDU）。MPDU作為PSDU被傳遞到PHY，它成為PHY有效載荷。


2.4 MAC command frame（命令幀）

上圖顯示了MAC命令幀的結構，它起源於MAC子層內部。MAC Payload包含命令類型和命令有效載荷兩部分。MAC有效負載以MHR為前綴並附加MFR。MHR包含幀控制域、幀序列號、地址域、附加安全頭。MFR包含一個16位的FCS效驗。MHR、MACPayload和MFR一起構成了MAC命令幀(MPDU)。MPDU作為PSDU被傳遞到PHY成為PHY的有效載荷。
