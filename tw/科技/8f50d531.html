Java 併發基礎--進程&線程

2020-11-14
| 科技

1 進程

資源分配和管理的最小單元。

2 線程

CPU調度的最小單元。

線程允許在同一個進程中同時存在多個程序控制流。

2.1 優勢

2.1.1 發揮多處理器的功能

多線程程序可以同時在多個處理器上並行執行。

2.1.2 提高處理器吞吐

多線程還有助於在單處理器系統上獲得更高的吞吐。

當線程發生IO阻塞時，系統將CPU資源切換到其他可運行的線程。

2.1.3 方便建模

使用線程，可以將複雜並且異步的工作流進一步分解為一組簡單的並且同步的工作流，每一個工作流在一個單獨的線程中執行。

2.1.4 簡化異步事件的處理

異步IO複雜性較高，很難編寫安全程序。

如果每個請求都有自己的處理線程，那麼在處理某個請求時發生阻塞將不會影響其他請求的處理。

2.1.5 更靈敏的圖形界面

將運行時間較長的任務放在一個獨立的線程中執行，那麼事件線程就能及時的處理用戶界面的事件。

2.2 風險

2.2.1 安全問題（數據安全問題）

由於多個線程共享相同的內存地址空間，並且是並行運行，因此他們可能訪問或修改其他線程正在使用的變量。

要想多線程程序的行為可以預測，必須對共享變量的訪問操作進行協同，這樣才不會在線程間發生彼此干擾。

2.2.2 活躍性問題

當某個操作無法繼續執行下去時，便發生了活躍性問題。

單線程情況下，活躍性問題就是無意中造成了死循環，導致之後代碼無法運行。

多線程情況下，由於包含數據各種手段的存在容易造成死鎖、飢餓、活鎖等活躍性問題。

2.2.3 性能問題

多線程的引入會導致一些開銷：

  1. 上下文切換開銷
  2. 同步機制導致各類優化手段的失效（編譯器優化、內存緩存優化、CPU指令優化等）同時增加共享內存總線的數據同步開銷

2.3 線程無處不在

每個java進程都在使用多線程。

框架通過框架線程調用應用程序代碼將併發性引入到程序中。

2.3.1 Timer

TimerTask將在Timer管理的線程中執行，而不是由應用程序管理。

2.3.2 Servlet&JSP

當一個servlet訪問在多個servlet或者請求中共享的對象時，必須正確的協同對這些對象的訪問，因為多個請求可能在不同的線程中同時訪問這些對象。

2.3.3 RMI

正確協同在過個對象中共享的狀態，以及對遠程對象本身狀態的訪問。

2.3.4 Swing&AWT

他們創建一個獨立的事件線程來處理用戶出發的事件，並對圖形界面進行更新。

2.4 線程狀態

2.4.1 new

new Thread之後，線程未執行前狀態

2.4.2 runnable

可運行狀態：

  1. new狀態的線程調用start方法
  2. 調用Thread.yield讓出CPU資源
  3. running狀態線程耗盡所分配CPU時間片
  4. join等待線程已完成
  5. sleep()超時
  6. wait資源調用notify或notifyAll或超時
  7. block線程獲取鎖

2.4.3 running

runnable狀態的線程獲取CPU資源後真正執行

2.4.4 waiting

等待狀態，調用wait、join等方法

2.4.5 timed_waiting

超時等待，調用sleep(timeout)、wait(timeout)、join(timeout)

2.4.6 block

進入同步方法同步塊前，等待獲取鎖對象

2.4.7 dead

線程死亡

  1. 線程成功執行完成
  2. 拋出未捕獲異常
