Jenkins擴展共享庫進階

2020-10-29
| 科技


前面我們介紹了Jenkins多分支流水線、Jenkins流水線即代碼之擴展共享庫初體驗，其實都是“流水線即代碼”的體現。我們將Jenkinsfile納入項目版本庫中統一管理，實現了“誰構建、誰運行”的理念。

但是在實際項目中，CI/CD其實是由運維來管理的，這樣就會導致運維、開發都要通過版本庫去修改Jenkinsfile、項目代碼。

試想下運維在調試流水線頻繁提交版本，導致遠程分支不斷更新，如果有鉤子觸發自動發版，勢必會引起開發的煩感。

為了避免這個情況的發生，我們引入了Jenkins擴展共享庫，即將流水線操作拆分為兩塊：

  1. Jenkinsfile定義流水線步驟、環境變量、參數等與項目相關的一切變量；
  2. 擴展共享庫定義流水線調用的方法、函數、類庫等與構建相關的具體操作；

由於一旦流水線步驟及變量確定一般就不會改動了，而擴展共享庫的方法等具體操作實現我們可以以代碼的方式放入遠程版本中，修改提交後Jenkinsfile構建自動加載共享庫，獲取最新的構建修改。

另，通過擴展共享庫我們可以提高構建操作的費用，有效減少構建代碼量；Jenkinsfile、擴展庫還可以作為備份託管在版本庫中，可謂是兩全其美啊。

下面我們對多分支流水線、擴展共享庫結合實現Vue項目的發版、回滾來具體講解下擴展共享庫的使用。

注：多分支流水線可以有效將多個分支放到一個項目下統一管理，避免因分支導致的項目分散。

Vue場景

Jenkins+遠程web服務器

功能實現：

  • 參數化構建：deploy-發版，rollback-回滾。
  • 發版：判斷git版本是否更新，若更新則在Jenkins上打包，並將dist包分發到遠程web服務器上；若未更新，則停止構建。
  • 回滾：回滾archiveArtifacts的版本包，分發到遠程web服務器上。

注意：我們使用archiveArtifacts來歸檔版本包，回滾時可從歸檔路徑中獲取。

擴展共享庫

一、添加擴展共享庫

Manage Jenkins--Configure System--Global Pipeline Libraries中添加

二、代碼庫結構

pipeline-shared-library/├── resources  #空├── src  #空└── vars    └── vueUpdate.groovy

vueUpdate.groovy 是所有vue項目的構建具體操作。

注意：由於所有的vue項目構建由共享庫中的統一的方法實現，因此不同分支對應的環境要高度一致，這樣才能最大限度的實現代碼複用。

三、具體實現

我們將代碼分為三部分，即deploy、rollback、update。

1.deploy-發版

（1）判斷版本是否更新

我們通過將本次git的版本id存入文件，以便下次構建時將其與GIT_COMMIT進行比較，實現版本是否更新。

注意：由於第一次構建時，流水線報錯“

No such property: GIT_PREVIOUS_SUCCESSFUL_COMMIT for class: groovy.lang.Binding

”。此時是無法通過GIT_PREVIOUS_SUCCESSFUL_COMMIT變量來獲取上一次版本的，因此只能將其寫入文件存放。

（2）打包

通過npm 打包vue項目生成dist。

注意：我們將dist壓縮並改名為dist_temp.zip 作為我們分發到項目的版本包。其目的是作為中間臨時文件，用於和項目的實際dist目錄進行替換，更新後銷燬即可。

另最終歸檔的版本包也為dist_temp.zip。

2.回滾-rollback

回滾的版本存在於archiveArtifacts歸檔後的構建目錄中，在此目錄中

${JENKINS_HOME}/jobs/`echo ${JOB_NAME}|awk -F'/' '{print \$1}'`/branches/${BRANCH_NAME}/builds/${version}/archive/

多分支流水線的目錄以分支名區分子目錄。

3.分發更新

Jenkins通過sshpublisher將版本包dist_temp.zip 分發到遠程web服務器上，通過rsync對項目目錄dist進行更新，最後銷燬dist_temp.zip。

注意：歸檔dist_temp.zip 及 郵件通知由Jenkinsfile定義，不放在共享庫中。

具體代碼如下：

所有的變量由跟隨項目的Jenkinsfile提供。

def deploy() {    sh """        #新建commitid文件，初次寫入0        if [ ! -f commitid ];then             echo 0 > commitid         fi        #從commitid獲取上次版本id        previous_id=`cat commitid`        if [ \$previous_id != $GIT_COMMIT ];then            #寫入本次git commit，用於下次判斷            echo $GIT_COMMIT > commitid            echo "start npm install&build"            /usr/local/nodejs/bin/npm install            /usr/local/nodejs/bin/npm run build            [ -f ${ZIP_NAME}.zip ] && rm -rfv ${ZIP_NAME}.zip            [ -d ${ZIP_NAME} ] && rm -rfv ${ZIP_NAME}            echo -e "\\033[34m將dist改名為${ZIP_NAME}：\\033[0m"            mv -v dist ${ZIP_NAME} && echo -e "\\033[32mdist改名為${ZIP_NAME}成功。\\033[0m" || {                echo -e "\\033[32mdist改名為${ZIP_NAME}失敗。\\033[0m"                exit 1            }            zip -r ${ZIP_NAME}.zip ${ZIP_NAME} && echo -e "\\033[32m${ZIP_NAME}壓縮成功。\\033[0m" || {                echo -e "\\033[32m${ZIP_NAME}壓縮失敗。\\033[0m"                exit 1            }        else            echo -e "\\033[31mRepositories not update, stop deploy\\033[0m"            exit 1        fi    """}def rollback() {    sh """    [ ${version} -eq 0 ] && {        echo -e "\\033[31m版本號非0。\\033[0m"        exit 1    } || echo -e "\\033[34mAction：${deploy_env}\\033[0m"    echo -e "\\033[34mRollback version：${version}\\033[0m"    rm -rfv ${ZIP_NAME}.zip    cp -Rv ${JENKINS_HOME}/jobs/`echo ${JOB_NAME}|awk -F'/' '{print \$1}'`/branches/${BRANCH_NAME}/builds/${version}/archive/${ZIP_NAME}.zip .    [ \$? -eq 0 ] && echo -e "\\033[32m指定版本號${version}的${ZIP_NAME}.zip複製到當前部署目錄成功。\\033[0m" || {        echo -e "\\033[31m指定版本號${version}的${ZIP_NAME}.zip複製到當前部署目錄失敗。\\033[0m"        exit 1    }    """}def update(host) {    for(i in host){        sshPublisher(            publishers: [sshPublisherDesc(configName: "$i", transfers: [sshTransfer(cleanRemote: false, excludes: '', execCommand: """            IN_Face=`/sbin/route -n |awk '{if(\$4~/UG/){print \$8}}'|head -n 1`            Local_IP=`/sbin/ip addr show "\${IN_Face}" | grep -w 'inet' | awk '{print \$2}'`            cd /App/htdocs/${APP_NAME}                            echo -e "\\033[34m\${Local_IP}節點解壓新版壓縮包${ZIP_NAME}.zip：\\033[0m"            unzip -o ${ZIP_NAME}.zip && echo -e "\\033[32m\${Local_IP}節點${ZIP_NAME}.zip解壓成功。\\033[0m" || {            echo -e "\\033[31m\${Local_IP}節點${ZIP_NAME}.zip解壓失敗。\\033[0m"            exit 1            }            public_dir=`echo ${ZIP_NAME} | awk -F'_' '{print \$1}'`            echo -e "\\033[34m\${Local_IP}節點將${ZIP_NAME}目錄下數據同步到生產目錄\${public_dir}內：\\033[0m"            [ -d ${ZIP_NAME} ] && rsync -vrl --delete ${ZIP_NAME}/ \${public_dir:-/tmp/aaa}/            [ \$? -eq 0 ] && echo -e "\\033[32m\${Local_IP}節點數據同步成功。\\033[0m" || {            echo -e "\\033[31m\${Local_IP}節點數據同步失敗。\\033[0m"            exit 1            }            echo -e "\\033[34m\${Local_IP}節點將${ZIP_NAME}.zip刪除：\\033[0m"            rm -rfv ${ZIP_NAME}.zip ${ZIP_NAME}            echo -e "\\033[32m${JOB_NAME}發佈成功。\\033[0m"            """, execTimeout: 120000, flatten: false, makeEmptyDirs: false, noDefaultExcludes: false, patternSeparator: '[, ]+', remoteDirectory: "htdocs/${APP_NAME}", remoteDirectorySDF: false, removePrefix: '', sourceFiles: "${ZIP_NAME}.zip")], usePromotionTimestamp: false, useWorkspaceInPromotion: false, verbose: false)]        )    }}

Jenkinsfile

1.參數化構建

通過parameters 定義構建的參數：deploy、rollback

回滾通過BUILD_NUMBER傳輸歷史版本的歸檔。

2.環境變量

我們定義兩個環境變量：

APP_NAME 項目所在目錄，如/App/${APP_NAME}/dist

ZIP_NAME 版本包

3.流水線步驟

發版：調用共享庫deploy方法；

回滾：調用共享庫rollback方法；

測試部署：調用共享庫update方法，傳入測試環境服務器列表；

生產部署：調用共享庫update方法，傳入生產環境服務器列表；

歸檔：不管構建狀態，總是歸檔版本包dist_temp.zip；

郵件通知：構建不穩定、成功、失敗發送郵件通知；


注意：流水線中我們使用when來匹配參數化構建，這樣可以避避免在sh中使用case或if 判斷，減少代碼量。

具體代碼如下：

//引入jenkins擴展共享庫@Library('shared-library') _pipeline {    agent any    options {        ansiColor('xterm')        timestamps()    }    parameters {        choice choices: ['deploy', 'rollback'], description: '''deploy：發佈            rollback：回滾''', name: 'deploy_env'        string defaultValue: '0', description: '''回滾版本號，發佈時忽略該參數            版本號為jenkins job BUILD_NUMBER''', name: 'version', trim: false    }    environment {        APP_NAME="www.test.cn"        ZIP_NAME="dist_temp"    }    stages {        stage('發版') {            //匹配發版參數            when {                expression { params.deploy_env ==~ 'deploy' }            }      steps {                script {                    vueUpdate.deploy()                }            }        }        stage('回滾') {            //匹配回滾參數            when {                expression { params.deploy_env ==~ 'rollback' }            }            steps {                script {                    vueUpdate.rollback()                }            }        }        stage('測試部署') {            //匹配develop分支            when {                branch 'develop'            }            steps {                script {                    def host = ["test-10.13", "test-10.14"]                    vueUpdate.update(host)            }        }        stage('生產部署') {            //匹配master分支            when {                branch 'master'            }            steps {                script {                    def host = ["prod-10.11", "prod-10.12"]                    vueUpdate.update(host)                }            }        }  }    post {        always {            archiveArtifacts "${ZIP_NAME}.zip"        }        unstable {            emailext (                body: """項目名稱：${JOB_NAME}\n構建編號：${BUILD_NUMBER}\n構建日誌：${BUILD_URL}console""",                subject: '【Jenkins構建通知】:$JOB_NAME - Build # $BUILD_NUMBER - Unstable!',                to: 'xx@test.cn',                from: 'test@test.cn'            )        }        success {            emailext (                body: """項目名稱：${JOB_NAME}\n構建編號：${BUILD_NUMBER}\n構建日誌：${BUILD_URL}console""",                subject: '【Jenkins構建通知】:$JOB_NAME - Build # $BUILD_NUMBER - Successful!',                to: 'xx@test.cn',                from: 'test@test.cn'            )        }        failure {            emailext (                body: """項目名稱：${JOB_NAME}\n構建編號：${BUILD_NUMBER}\n構建日誌：${BUILD_URL}console""",                subject: '【Jenkins構建通知】:$JOB_NAME - Build # $BUILD_NUMBER - Failure!',                to: 'xx@test.cn',                from: 'test@test.cn'            )        }    }}

總結

Jenkins擴展共享庫+多分支流水線一方面可以簡化CI/CD過程中的項目管理，一方面可以驅動我們各個環境的標準化，為實現自動化做好鋪墊。

反過來環境標準化是我們靈活應用Jenkins擴展共享庫的前提，沒有足夠的標準化，那麼我們就需要增加代碼量去適配各個環境。

總之，在運維的過程中，你會發現標準化和規範化越來越重要。
