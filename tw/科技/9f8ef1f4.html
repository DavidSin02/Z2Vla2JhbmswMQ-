導入-因特網的三個重要組件

1、IP協議

IP是主機在網絡中的唯一標識符, 且在全網上唯一不可重複, 就像人的身份證一樣(更恰當的說是身份證號)。

2、路由選擇

路由選擇決定了數據報從源到目的地所流經的路徑，好似我們規劃週末與女朋友的散步路線。

3、報告數據報中的差錯和對某些網絡層信息進行響應的設施

互聯網控制報文協議(ICMP)是TCP/IP協議簇的一個子協議，被用來實現在互聯網上傳遞錯誤以及控制信息。

2、關鍵字段

（1）版本號

  • 長度為4bite；

  • 規定了數據報的IP協議版本。

（2）首部長度

  • 因為一個IPV4數據報可包含一些可變數量的選項,所以需要首部長度這個字段來確定數據報中數據部分從何開始；

  • 大多數IP數據報不包含選項,所以一般IP數據報長度為20字節。

（3）服務類型

  • 用於區分不同類型的數據報；

  • 例如實時數據報和非實時數據報。

（4）數據報長度

IP數據報的總長度。

（5）標識, 標誌, 總偏移

下面會詳細講解。

（6）壽命

  • 數據報每被路由器處理一次則壽命字段減一,減至0時則必須丟棄；

  • 這個字段用於確保數據報不會永遠在網絡中循環。

（7）協議

  • 僅在數據報到達最終目的地時才會被使用；

  • 指定了該數據報該被交給哪個特定的運輸層協議；

  • 例如,值為6表示要交給TCP,值為17表示要交給UDP。

（8）首部檢驗和

檢驗數據報傳輸過程中是否發生比特錯誤。

（9）源和目的IP地址

（10）選項

可以擴展IP首部。

（11）數據

這是數據報存在的根本意義。

3、IP數據報分片

（1）最大傳送單元( Max:imum TrallSnnssion lJnil , MTU)

  • 因為並不是所有鏈路層協議都能承擔相同長度的網絡層分組,有的協議能承載大數據報,有的協議只能承載小分組；

  • 最大傳送單元( Max:imum TrallSnnssion lJnil , MTU)指的是一個鏈路層所能承載的最大數據報；

  • 每一個鏈路的MTU都可能不同。

（2）當IP數據報的長度超過當前鏈路的最大傳送單元MTU時, 就必須將數據報進行分片, 分成幾個較小的IP數據報並在到達目的地後重新組裝。

  • IPV4的設計將重新組裝的動作放在端系統中而非網絡路由器；

  • 較小的數據報就稱為片(fragment)。

（3）目的主機與片

  • 當目的主機從網絡中收到一系列數據報時就需要判斷這些數據報是不是一些原來較大的數據報的片。

  • 如果是片,那那一片是最後一片,並且如何進行拼裝；

  • 因此IPv4 的設計者將標識、標誌和片偏移字段放在 IP 數據報首部中用於解決這一問題。

（4）標識

  • 當生成一個數據報時，發送主機在為該數據報設置源和目的地址的同時再貼上標識號。

  • 發送主機通常將為它發送的每個數據報的標識號加 1 。

  • 當某路由器需要對一個數據報分片時，形成的每個數據報(即片)具有初始數據報的源地址、目的地址與標識號。

  • 當目的地從同一發送主機收到一系列數據報時，它能夠檢查數據報的標識號以確定哪些數據報實際上是同一較大數據報的片。

（5）標誌

  • 由於 lP 是一種不可靠的服務，一個或多個片可能永遠到達不了目的地。

  • 因為這種原因，為了讓目的主機絕對地相信它已收到了初始數據報的最後一個片，最後一個片的標誌比特被設為 0 ，而所有其他片的標誌比特被設為 l 。

（6）偏移字段

為讓目的主機確定是再丟失了一個片(且能按正確的順序重新組裝片) ，使用偏移字段指定該片應放在初始 IP 數據報的哪個位置。

（7）當然分片也會產生開銷並且可能被利用來攻擊。

lPv4 編址

1、IP與接口的關係

我們將主機與物理鏈路之間的邊界叫做接口(interface) 。

因為路由器的任務是從鏈路上接收數據報並從某些其他鏈路轉發出去，路由器必須擁有兩條或更多條鏈路與它連接。所以路由器與它的任意一條鏈路之間的邊界也叫做接口。

因此一臺路由器有多個接口，每個接口有其鏈路。

因為每臺主機與路由器都能發送和接收IP數據報， IP 協議要求每臺主機和路由器接口擁有自己的IP地址。

因此，一個IP地址技術上是與一個接口相關聯的，而不是與包括該接口的主機或路由器相關聯的。

2、IP地址

每個IP 地址長度為 32 比特(等價為 4 字節) ，因此總共有 2^32個可能的 IP 地址。

由於210近似於103, 因此可以看出IP地址大約有40億個。

IP地址採用分十進制記法書寫, 即每個字節以其的十進制形式書寫, 並以句號隔開，例如127.0.0.1(這其中127,0,0,1實際上在電腦中都是一個8位二進制數)。

在因特網上的每個接口必須有唯一的IP地址與之對應(暫不考慮NAT,這多用於子網)。

這些地址不能隨意地自由選擇。一個接口的IP地址的一部分需要由其連接的子網來決定。

IP 廣播地址 255. 255. 255.255。

當一臺主機發出一個目的地址為 255.255.255.255 的數據報時，這段報文會交付給同一個網絡中的所有主機。路由器也會有選擇地向鄰近的子網轉發該報文(雖然它們通常不這樣做) 。

3、子網

多個主機接口和多個路由器接口的網絡形成一個子網；

形式為α. b. c. d/x 的地址的 x 最高比特構成了 IP 地址的網絡部分，並且經常被稱為該地址的前綴(prefix) (或網絡前綴)。

其中/x有時被稱為子網掩碼, 因為也因此在同一個子網中的前X位都是一致的。

為了確定子網，分開主機和路由器的每個接口，產生幾個隔離的網絡島，使用接口穿接這些隔離的網絡的端點。這些隔離的網絡中的每一個都叫做一個子網( suhnet) 。

一個地址的剩餘 32 - x 比特可認為是用於區分該組織內部設備的, 只有在組織內部的路由器轉發分組時才會考慮這些分組；

CIDR採用以前, IP的網絡部分只能固定的8.16. 24位, 但是後來由於中小規模的迅速增加, 取消了這一限制。

4、獲取一個IP地址

網絡管理員可以向其ISP索取一個或一些地址, 單這不是唯一的方式。

5、如何獲取主機地址

動態主機配置協議((Dynamic Host Configuration , DHCP)允許主機自動獲取(被分配)一個IP地址；

管理可以配置使一臺機器每次獲得的是同一個地址還是一個臨時地址。

除了主機 IP 地址分配外， DHCP 還允許一臺主機得知其他信息，例如它的子網掩碼、它的第一條路由器地址(常稱為默認網關)與它的本地 DNS 服務器的地址。

即插即用協議-DHCP能夠將一臺主機連入網絡。

從DHCP服務器獲取IP地址的過程：

在最簡單場合下，每個子網將具有一臺 DHCP 服務器或一個DHCP中繼器。

對於一臺新到達的主機而言, DHCP 協議是一個 4 個步驟的過程：

第一步：DHCP服務器發現

  • 新來的主機需要首先發現一個在當前網絡中的DCHP服務器；

  • 所以新來的主機向網絡的廣播地址255.255.2555.255(廣播地址) 發送一個DHCP發現報文,中間包含自己的源地址0.0.0.0。

  • 這一發現報文將被廣播給所有與該子網連接的主機。

第二步：DHCP服務器提供

  • DHCP服務器收到一個發現報文則回覆一個DHCP提供報文

  • 提供報文中包含有收到的發現報文的事務、向客戶推薦的 IP地址、網絡掩碼以及 IP 地址租用期 (address lease time) ，即 IP 地址有效的時間量。服務器租用期通常設置為幾小時或幾天

  • 這一消息依舊是通過全局廣播進行發送。

  • 注意：一個子網可能有多個DHCP服務器,所以該客戶也許能進行選擇。

第三步：DHCP請求

新客戶從一個或多個提供報文中選擇一個,並向選定的DHCP服務器提供一個DHCP請求報文進行響應,同時配置一些參數

第四步：DHCP ACK

  • 服務器用 DHCP ACK 報文 (DHCP ACK message) 對 DHCP 請求報文進行響應，證實所要求的參數。

  • 一旦客戶收到這個ACK報文則交互就完成了,客戶能夠在租用期間內使用DHCP分配的IP地址。

DHCP的不足

當移動節點在子網之間移動時就不能保證其IP是一致的,就不能維持與遠程應用之間的TCP連接。

6、網絡地址轉換

網絡地址轉換 (Network Address Translation ）簡稱NAT。

NAT 使能路由器對於外部世界來說甚至不像一臺路由器 。NAT 路由器對外界的行為反過來就如同一個具有單一 IP 地址的單一設備。

所有離開家庭路由器流向更大因特網的報文都擁有一個源IP地址 138. 76. 29. 7 ，且所有進入家庭的報文都擁有同一個目的地址 138. 76. 29. 7 。(從本質上講， NAT 使能路由器對外界隱藏了家庭網絡的細節。)

家庭網絡中分配的都是專用地址, 即其地址只在當前子網內有意義, 在因特網上找不到這個地址。

路由器從ISP的DHCP獲取IP, 而家庭網內的主機從子網內的DHCP獲取IP。

在NAT路由器上維護一張NAT轉換表, 表中有端口與子網內某臺主機IP以及其端口的對應關係。

但是這種使用端口對應主機及其端口的形式受人詬病。

  • 有人認為端口號是用於進程編址的，而不是用於主機編址的。而這種違規用法對於運行在家庭網絡中的服務器來說確實會引起問題。

  • 他們認為路由器通常僅應當處理高達第三層的分組。

  • 他們認為 NAT 協議違反了所謂端到端原則，即主機彼此應相互直接對話，結點不應介入修改 IP 地址與端口號。

  • 他們認為應使用IPv6 來解決 IP 地址短缺問題，而不是不計後果地用一種如 NAT 之類的權宜之計來修補存在的問題。

NAT訪問與P2P應用：

在一個P2P 應用程序中，任何參與對等方 A 應當能夠對任何其他參與對等方 B 發起一條 TCP 連接。該問題的實質在於如果對等方B 在一個 NAT 後面，它不能充當服務器並接收 TCP 連接。

如果對等方 A 不在一個 NAT 的後面，則該 NAT 問題能夠繞過去。在這種情況下，對等方 A 能夠首先通過一箇中間對等方C 與對等方 B 聯繫

如果雙方都在NAT之後則能夠使用一些NAT穿越（比如下面的UPnP）的技術克服。

7、UPnP

NAT 穿越正越來越多地由通用即插即用(UPnP) 提供。

UPnP是一種允許主機發現並配置鄰近 NAT 的協議 。UPnP要求主機和 NAT 都是 UPnP兼容的。

使用 UPnP ，在主機上運行的應用程序能夠為某些請求的公共端口號請求一個 NAT 映射，該映射位於其(專用 IP地址，專用端口號)和(公共 IP 地址，公共端口號)之間。

映射的公共部分對於所有主機都是可見的, 外部的主機可對次發起連接, 內部主機可以通過此部分向外部通告它的存在。

總而言之， UPnP 允許外部主機使用 TCP 或 UDP 向 NAT 化的主機發起通信會話。長期以來 NAT 一直對P2P 應用程序十分不利;UPnP由於提供了有效和健壯的 NAT 穿越解決方案，可能成為了P2P 應用程序的救世主。

因特網控制報文協議

ICMP被主機和路由器用來彼此溝通網絡層的信息 。

ICMP最典型的用途是差錯報告, 例如目的網絡不可達之類的錯誤。

ICMP與IP：

  • ICMP 通常被認為是 IP的一部分，但從體系結構上講它是位於 IP 之上的，因為其承載在IP分組中就像TCP和UDP

  • 當一臺主機收到一個指明上層協議為 ICMP 的 IP數據報時，它分解出該數據報的內容給 ICMP ，就像分解出一個數據報的內容給 TCP 或 UDP 一樣。

ICMP報文

ICMP 報文有一個類型字段和一個編碼字段，並且包含引起該 ICMP 報文首次生成的E 數據報的首部和前 8 字節內容(以便發送方能確定引發該差錯的數據報)。

IPv6

我們知道IPV4定義的IP地址是4個8位的地址總共只有四十多億個，在如今的時代已日漸不夠用。

1、IPV6數據報格式

新的IPV6定義的IP是4個32位的地址。

圖例：

（1）格式的變化

擴大的地址容量：

  • IPv6 將 IP 地址長度從 32 比特增加到 128 比特 。

  • 並多了一個任播地址。

簡化高效的 40 字節首部。

增加了流標籤與優先級字段。

  • 版本

  • 流量類型

  • 有效載荷長皮

  • 下一 個首部

  • 跳限制

轉發數據報的每臺路由器將對該字段的內容減 1 。如果限制計數到達0時，則該數據報將被丟棄。

  • 源地址和目的地址

  • 數據

有效載荷部分

2、丟棄的IPV4字段

（1）分片/重新組裝

  • IPV6而不允許在中間路由器上進行分片與重新組裝 。這種操作只能在源與目的地上執行；

  • 如果過大直接丟棄並向發送方發送一個分組過大的ICMP差錯報文即可。

（2）首部檢驗和

因為因特網層中的運輸層(如 TCP 與 UDP) 和數據鏈路層(如以太網)協議執行了檢驗操作， IP 設計者大概覺得在網絡層具有該項功能實屬多餘，可以將其去除。

（3）選項

選項字段不再是標準 IP 首部的一部分了。但它並沒有消失，而是可能出現在 IPv6 首部中由"下一個首部"指出的位置上。

3、從IPV4到IPV6的遷移

我們必須考慮一個非常實際的問題: 基於IPV4的公共因特網如何遷移到IPV6(我們可以在設計IPV6使能系統時設計成向後兼容IPV4, 但已有的IPV4卻不能兼容IPV6, 處理其發送的數據報等)。

可能的方法：

（1）標誌日

  • 在某一天同時關機進行升級；

  • 不可想象。

（2）雙線

使用該方法的 IPv6 結點還具有完整的 IPv4 實現。這樣的結點被稱為 IPv6/IPv4 結點，它有發送和接收 IPv4 與 IPv6 兩種數據報的能力。

（3）建隧道

  • 假定兩個 IPv6 結點( 要使用 IPv6 數據報進行交互，但它們是經由中間IPV4路由器互聯的。

  • 將IPV6數據報整體包裹在一個IPV4數據報中然後進行發送；

  • 達到目的地後確認該IPV4數據報中有IPV6數據報則將其取出讀取。

涉足 IP 安全性

一個提供各種安全性服務的網絡層協議-IPsec, 在VPN中得到廣泛部署。

IPsec被設計成IPV4和IPV6向後兼容。

（1）lPsec 的運輸模式：

使用這種模式，兩臺主機首先在它們之間創建一個 IPsec 會話。(因此 IPsec 是面向連接的 ! )使用適當的會話，在這兩臺主機之間發送的所有 TCP 和 UDP 報文段都享受 IPsec 提供的安全性服務。在發送端，運輸層向IPsee 傳遞一個報文段。I Psec 然後加密該報文段，在報文段上添加附加的安全性字段，並且在一個普通的 IP 數據報經過封裝得到的有效載荷。

（2）lPsec提供的安全服務

  • 密碼技術約定

  • lP 數據報有效載荷的加密

  • 數據完整性

IPsec 允許接收主機驗證數據報的首部字段，保證被加密的有效載荷在其數據報從路到目的地的路由器中傳輸時沒有被修改過。

  • 初始鑑別

當一 臺主機從某受信任的源接收到一個 IPsec 數據報時，該主機確信在數據報中的源IP地址是該數據報的實際源。
