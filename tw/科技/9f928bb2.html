CrashScope：一種實用的安卓應用自動化測試工具

2020-11-14
| 科技

摘要

由於移動應用程序具有普遍的事件驅動特性和複雜的上下文功能（例如傳感器，通知）的原因，在測試移動應用程序時會遇到特別的挑戰。由於所需的工具或平臺依賴性，當前用於安卓應用程序的自動輸入生成方法通常不適合開發人員使用，並且通常無法有效地行使上下文功能。為了在開發人員的移動測試任務中更好地支持開發人員，在本項目中，我們介紹了一種新穎的自動化工具 CRASHSCOPE。該工具根據靜態和動態分析提供的幾種策略，使用系統化的輸入生成功能來探索給定的安卓應用，其內在目標是觸發崩潰。當檢測到崩潰時，CRASHSCOPE 會生成一個增強的崩潰報告，其中包含屏幕截圖，詳細的崩潰復現步驟，捕獲的異常堆棧跟蹤信息以及可完全重播的腳本，該腳本可在目標設備上自動復現崩潰。初步研究的結果表明，CRASHSCOPE 能夠發現一樣多的其他最新技術工具發現的崩潰，同時為開發人員提供詳細的有用崩潰報告和測試腳本。

I 簡介

用戶越來越喜歡在智能手機或平板電腦的便捷環境中執行計算任務，這樣的用戶群推動了移動硬件和應用程序市場的快速發展。但是，移動應用程序的手勢驅動特性已經引發了程序員在開發和維護期間，特別是在測試和調試方面的新挑戰。錯誤報告的創建和解決是最困難、最重要的維護任務之一。關於應用程序崩潰的報告對於開發人員而言尤其重要，因為崩潰表示嚴重的錯誤，用戶將直面崩潰錯誤且該錯誤會立即影響應用程序的可用性。如果某個應用因崩潰，功能缺失或其他錯誤而無法正常運行，則近一半的用戶可能會在當下市場競爭的環境下在 Google Play 等市場中放棄該應用。因此，為了確保其應用程序的正常運行，開發人員需要自動化的支持措施來發現，復現並修復崩潰。

儘管在移動應用測試和自動生成輸入方面已經取得了顯著進展，可用的工具卻通常存在一些值得注意的侷限性：（i）在自動輸入生成過程中，先前的方法缺乏為發現的故障提供詳細，易於理解的測試結果的能力，開發人員需要整理和理解堆棧跟蹤信息、日誌文件以及非表達性事件序列；（ii）大多數自動生成輸入的方法對於開發人員都不實用，通常是由於儀器或安裝程序困難。這也說明了為什麼開發人員通常更喜歡手動而不是自動測試方法。（iii）沒有一種方法可以在單一整體方法中結合使用不同的 GUI 開發策略，上下文功能的目標測試以及用於用戶文本輸入的多種策略。這些缺點導致自動測試方法的採用率低，同時讓基於手動和腳本的測試技術得到偏愛。

通過了解開發人員在工作流程中採用自動化測試工具時面臨的上述問題，為了解決這些難題，我們設計並實現了 CRASHSCOPE。CRASHSCOPE 是一種實用的系統，它可以自動發現，報告和復現安卓應用程序的崩潰過程。 CRASHSCOPE 根據一組輸入生成策略來探索給定的應用程序，並生成具有明確步驟的表達性崩潰報告，並以易於閱讀的自然語言格式進行報告生成。這種方法僅需要一個.apk 文件（以及可選的源代碼）和一個 Android 仿真器或設備即可運行，而不依賴主題應用程序或 Android OS 的檢測。 CRASHSCOPE 的整個工作流程都是自動化的，除了閱讀生成的報告外，無需開發人員干預。我們工具支持的不同探索策略旨在引起 Android 應用程序崩潰，幷包括基於文本輸入字段允許字符的上下文自動生成文本的功能，以及針對上下文特徵進行測試（例如設備的方向，網絡的接口和傳感器）。我們專門定製了這些功能，以測試先前研究，所確定的應用程序崩潰的常見原因。在執行期間，CRASHSCOPE 捕獲有關主題應用程序的詳細信息，例如輸入，屏幕截圖，GUI 信息，異常和崩潰。然後，此信息將轉換為詳細的崩潰報告和可重播的腳本，從而大大減輕了開發人員修復故障的負擔。

II CRASHSCOPE 自動化測試工具

CRASHSCOPE 通過幾種方式解決了現有工具的一般侷限性，這些方式說明了該方法的新穎性：（i）CRASHSCOPE 能夠自動生成表達性的錯誤報告（和可重播的腳本），用於發現異常或崩潰；（ii）這種方法很實用，只需要一個.apk 文件即可運行；（iii）探索策略能夠對上下文特徵和自適應文本輸入生成進行有針對性的測試；以及（iv）該工具具有抗崩潰的應用程序功能；它可以檢測到崩潰，並在處理崩潰後繼續測試 GUI 中未訪問的組件和狀態。 CRASHSCOPE 的目標用戶群是希望將實用的自動化測試工具納入其工作流程的移動應用程序開發人員和測試人員。為了使該工具易於使用並有效地集成到典型的開發人員的工作流程中，我們開發了一種基於雲的 Java Web 應用程序，其中封裝了 CRASHSCOPE 工具。該 Web 應用程序包括一個執行引擎後端，該後端會執行測試要求的靜態和動態分析，以及易於提交應用程序進行測試的用戶友好前端。這種體系結構可確保 CRASHSCOPE 作為基於 Web 的工具的靈活性，可伸縮性和可擴展性。

A． CrashScope Design

CRASHSCOPE 的總體體系結構如圖 1 所示。該工具工作流程的第一步是使用 Web 應用程序的 New Task 表單上傳應用程序.apk 文件（以及可選的 src 代碼），提交用於測試的應用程序。並選擇 CRASHSCOPE 用於探索該應用程序的策略配置。一旦開發人員啟動了任務，就可以在 Testing-Dashboard 中查看進度（圖 2）。在測試儀表板頁面上，開發人員可以查看提交的任務和當前正在運行的任務的執行進度（例如，圖 2 的底部）。另外，可以查看統計信息，包括運行時間，應用信息和發現的崩潰數量，以瞭解已完成的任務（例如，圖 2 的頂部）。啟動任務後，Web 應用程序將通過 json 文檔將包括.apk 文件在內的任務信息發送到 CouchDB 的實例中（充當消息隊列）。後端執行引擎會定期輪詢隊列中的新任務，並在收到消息時啟動測試過程。執行引擎在測試過程中執行的第一步是對提交的應用程序進行靜態分析，以識別上下文功能（圖 1-1）。為此，它要麼利用用戶上傳的應用程序源代碼（如果已提供），要麼嘗試使用多種工具來反編譯該應用程序。然後，它將檢測與上下文功能相關的活動（例如屏幕），以針對此類功能進行測試。換句話說，CRASHSCOPE 僅在識別應用程序代碼中的功能實例時才會測試上下文應用程序功能（例如，網絡開/關）（第 II-B 節）。

圖 2 CRASHSCOPE web 應用實驗儀表板

接下來，GUI 翻錄引擎（圖 1-2）使用用戶指定的組合策略（第 II-E 節）系統地執行應用程序，包括在由靜態分析識別的相關屏幕上啟用和禁用上下文功能（如果在模擬器上運行）。 如果在執行過程中引發了未捕獲的異常，或者應用程序崩潰，則在系統的探索過程中系統將動態執行信息保存到 CRASHSCOPE 的數據庫（圖 1-3），其中包括有關每個事件的詳細信息（例如，觸摸事件，文本輸入，傳感器值更改）。可以在大量併發運行的物理設備或使用 android-x86 映像和 VirtualBox 實例化的 Android 虛擬設備（AVD）上執行此過程。

將執行數據保存到 CRASHSCOPE 數據庫之後，自然語言報告生成器（圖 1-4，第 II-F 節）分析數據庫並處理每個事件的信息，這些信息是以崩潰結束的執行序列。然後自然語言報告生成器將該信息以 json 形式發送給 CouchDB。 然後，Java Web 應用程序將為用戶生成基於 HTML 的自然語言崩潰報告，並帶有表達性再現步驟（圖 1-6）。 此外，崩潰腳本生成器（圖 1-5，第 II-G 節）分析數據庫並提取崩潰執行中每個步驟的相關信息，以創建包含 adb 輸入命令和上下文狀態更改標記的可重播腳本。腳本重放器（圖 1-7，第 II-G 節）是一個小型 Java 程序，能夠重放這些腳本。

B． 提取活動和應用級別的上下文特徵

CRASHSCOPE 使用基於抽象語法樹（AST）的分析來提取與上下文功能的調用有關的 API 調用鏈。尤其是，它可以檢測與網絡連接和傳感器（例如，加速度計，磁力計，溫度傳感器和 GPS）相關的 Android API 調用。由於 API 調用可能不會直接由與實現 Activity 的類相關聯的代碼直接執行（在 Android 中包括屏幕），因此 CRASHSCOPE 執行調用圖分析以提取路徑，該路徑以調用上下文 API 的方法作結。由於某些 API 調用可能無法通過向後傳播的調用鏈進行跟蹤（例如，作為服務實現的傳感器或網絡連接），因此 CRASHSCOPE 使用兩種粒度來測試上下文功能：活動（屏幕）級別和應用程序級別。如果 API 分析表明某個應用程序實現了特定的上下文功能，但是無法通過調用鏈分析將該功能鏈接回特定屏幕，則將在“應用程序級別”進行測試或在每個屏幕上測試該功能。通過分析應用程序的 AndroidManifest.xml 文件可以找到可旋轉的屏幕。

C． 應用 & 崩潰檢測的探索

為了探索應用程序，CRASHSCOPE 動態提取在探索過程中訪問的每個應用程序屏幕的 GUI 層次結構（使用 uiautomator），並標識要執行的可單擊和可長時間單擊的組件，以及用於文本輸入的可用組件（例如 EditText 框）。在每個屏幕上觀察時，可（長時間）單擊的組件被添加到動態組件堆棧中，並根據堆棧的順序執行。當前，翻錄引擎支持輕敲，長按和鍵入事件。如果在特定的應用程序屏幕中可以使用文本輸入字段，則在執行屏幕上的每個可（長時間）單擊組件之前，都要填寫每個文本框。 此外，從一個獨特活動到另一個獨特活動的每個轉換以及導致轉換的事件都記錄在轉換圖中。 組件和轉換圖使得 CRASHSCOPE 可以在執行期間構建應用程序的準確實時模型。

來自用戶的文本輸入是許多安卓應用程序功能的主要部分，因此，CRASHSCOPE 的 GUI 翻錄引擎採用了獨特的文本輸入生成機制。CRASHSCOPE 通過使用 adb shell dumpsys input_method 命令查詢與文本字段相關聯的鍵盤類型，來檢測文本字段期望的文本類型（例如，電話號碼，電子郵件地址）。 一旦檢測到預期輸入的類型，CRASHSCOPE 將使用兩種方法生成文本輸入：預期輸入和意外輸入。預期的啟發式會在鍵盤參數內生成一個字符串，而沒有任何標點符號或特殊字符；意外的啟發式會生成具有給定鍵盤類型的所有允許的特殊字符的隨機字符串。

除了文本輸入生成策略之外，CRASHSCOPE 還可以根據幾種 GUI 探索策略之一來生成觸摸輸入。當前，該工具從層次結構的底部向上或從層次結構的頂部向下（通過將訂單組件放置在組件堆棧中進行控制）以類似於深度的方式遍歷 GUI 層次結構。具有兩種這樣策略的基本原理是模仿用戶的操作（即在沒有預定順序的情況下執行 GUI 事件）。但是，CRASHSCOPE 可擴展到其他類型的探索方法，包括基於隨機的方法或有偏隨機的方法。為了檢測和捕獲異常，CRASHSCOPE 過濾了 logcat 以獲取僅與被測試的應用程序相關的未捕獲異常。為了檢測崩潰，CRASHSCOPE 檢查標準安卓崩潰對話框的外觀。如果遇到崩潰，則將執行信息記錄到數據庫中。但是，由於 GUI 翻錄引擎維護轉換圖和未訪問組件堆棧的原因，執行可以繼續操作剩餘的程序路徑。

D． 在不同的上下文環境中測試移動應用

當 GUI 翻錄開始時，CRASHSCOPE 首先檢查應根據探索策略進行測試的應用程序級別的上下文功能。每次訪問活動時，GUI 翻錄引擎都會檢查是否應啟用/禁用上下文功能，並根據當前策略設置功能值。上下文功能的測試僅在使用 telnet 命令的仿真器或 AVD 上有效。雖然 telnet 命令確實支持為仿真器打開/關閉網絡，但它們不支持啟用/禁用傳感器（加速度計，磁力計，GPS，溫度傳感器），但是可以模擬這些傳感器的值。 因此，為了在不利條件下測試與傳感器相關的功能，請禁用網絡連接，併為其他傳感器（如 GPS 和陀螺儀）設置意外的（例如，高度不可行的）值。

E． 多種執行策略

CRASHSCOPE 最強大的功能之一就是能夠通過結合各種可支持的測試功能，根據幾種不同的策略來探索應用程序。這些策略來自三個主要的特徵啟發法：1）GUI 遍歷策略（例如 dfs-top-down，dfs-bottomup 或基於隨機的），2）生成用於用戶文本輸入字段的輸入的方法（無文本，期望的文本，意外的文本），最後，3）啟用或禁用對不利的上下文狀態的測試（例如，如果發現某個活動已經利用了網絡，則應將其打開還是關閉？）。這些策略的不同組合有可能發現不同類型的應用崩潰。在不同策略中通過這三個功能啟發式的所有組合來運行應用程序，CRASHSCOPE 可以有效地測試常見誘導崩潰的不同類型。還可以通過在一組或一組仿真器實例雲上同時運行一個應用程序的幾種組合來並行化這些策略（例如使用執行引擎），從而進一步減少測試開銷。

F． 生成表達性的自然語言崩潰報告

CRASHSCOPE 生成一個崩潰報告（圖 1-6），其中包含四種主要信息：1）常規信息，包括應用程序名稱和版本，Android OS 的版本，圖標——指示應用在復現步驟中的當前上下文狀態，設備以及崩潰發生時的屏幕方向和分辨率； 2）自然語言句子，使用有關每個步驟的 GUI 事件和上下文狀態的詳細信息來描述復現崩潰的步驟（圖 3）； 3）應用程序的屏幕窗口，其中突出顯示了針對特定崩潰情況在執行場景中與每個屏幕交互的組件； （4）刪除堆棧跟蹤，僅包含執行期間發生的應用程序異常。一旦執行引擎報告已完成提交任務的所有策略，它就會通過 json 向 CouchDB 發送一條消息。該消息由 Java Web App 檢索，並且在崩潰報告頁面中可以查看當前任務的結果和崩潰報告。

G． 生成&重放複製腳本

崩潰腳本生成器（圖 1-5）從 CRASHSCOPE 數據庫中解析保存執行信息，並生成可重播的腳本，其中包含用於觸摸和文本輸入的 adb 輸入命令以及用於更改上下文狀態的標記。 通過解析數據庫以獲取與特定執行中每個步驟相關的所有 GUI 事件的腳本。然後，系統會分析該應用的系統探索過程中記錄的每個組件的座標，並根據每個組件的大小推斷中心座標。這些座標用於生成 adb 輸入命令以重現 GUI 事件。這種方法依賴於我們之前在安卓應用中重播測試序列事件的工作。腳本可通過腳本重播器（圖 1-7）進行重播，腳本重播器是一個小型 Java 程序，它將記錄的腳本作為輸入，並重播 adb 命令和狀態更改標記（例如“ 關閉 WiFi”）。

III 評估

為了評估 CrashScope，我們進行了兩項實證研究，旨在調查該工具的崩潰發現能力以及所生成報告的可重複性和可讀性。首先，在“崩潰發現研究”中，我們利用 61 種開源主題應用程序中的一部分主題應用程序和工具，將 CRASHSCOPE 與 4 種最先進的 Android 自動輸入生成工具（以及標準的 Monkey 工具）進行了比較，該工具可用於 61 個開源主題應用程序 Androtest 測試套件。其次，在“崩潰報告研究”中，我們評估了 CRASHSCOPE 生成的自然語言報告與在線問題跟蹤器中發現的人工書面報告相比的可重複性和可讀性。在這項研究中，我們從與 F-Droid 開源市場上的應用程序鏈接的在線問題跟蹤器中提取了 8 個真實的崩潰報告，CRASHSCOPE 能夠為這些崩潰報告生成新報告。然後，有 16 名研究生參與者試圖重現 8 個錯誤報告（其中 4 個來自 CRASHSCOPE，4 個來自在線問題跟蹤器），隨後回答了有關兩種匿名錯誤報告的調查問題。這些研究的結果表明，（i）CRASHSCOPE 在發現崩潰方面的有效性與其他先進的 Android 自動輸入生成工具相同，同時減少了由於使用其他工具而需要的工具導致的誤報的數量，以及（ii）生成了報告 CRASHSCOPE 的報告與人類書面報告一樣具有可重複性，並且從開發人員的角度來看更具可讀性和實用性。

IV 項目備註與未來工作

在這個項目中，我們介紹了 CRASHSCOPE，這是針對 Android 應用程序的實用的自動化測試方法的新穎實現。我們的工具通過（i）利用靜態分析以針對性的方式定位和測試上下文特徵來克服以前方法的一些缺點；（ii）使用幾種不同的策略組合來探索應用程序；（iii）生成非常有用的崩潰報告和可重播的測試方案腳本。將來，我們的目標是研究通過在基於模型的 GUI 測試和靜態分析中採用有前途的新興方法，將錯誤報告整理為僅有必要步驟的模式。同時我們將改善我們的系統探索策略技術。此外，我們旨在將 CRASHSCOPE 分發到更廣泛的安卓開發社區，並評估其實用性。
