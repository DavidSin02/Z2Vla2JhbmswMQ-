01 回顧

之前得出過的結論：事務執行：通過TransactionInterceptor增強器對

目標方法進行環繞增強。

  • 調用目標方法前，創建事務。
  • 執行目標方法
  • 調用目標方法後，提交事務。

TransactionInterceptor功能增強功能的是實現在父類TransactionAspectSupport的invokeWithinTransaction方法

關鍵代碼：

02 事務存在的三個問題

假如有這麼一個類

@Servicepublic class TransactionalService { @Autowired UserDao userDao;	public String noTransactionalSave(){ System.out.println("非事務方法"); return save1(); } @Transactional public String save1(){ User user = new User("主動方事務方法",0); save2(); userDao.save(user); return "save1"; } @Transactional(propagation = Propagation.REQUIRES_NEW) public String save2(){ User user = new User("被事務方法",1); userDao.save(user); return "save2"; } @Transactional public synchronized String syncSave(){ System.out.println("鎖方法"); User user = new User("鎖方法",0); userDao.save(user); return "鎖"; }}

記得在事務代理的創建那一節說過，不管是方法上，還是類上使用 @Transactional。都會對此類進行代理的創建。 創建代理後，調用目標方法首先調用的是代理的方法。

以CGLB為例，執行鏈

調用方法-->動態代理類.方法-->MethodInterceptor.intercept方法-->MethodInvocation.proceed執行增強器鏈-->Adivce.invoke方法-->目標方法

我們再看看MethodInterceptor.intercept()方法

2.1 同類中非事務方法調事務方法

首先因為有事務註解，所以會生成代理對象。

因為首先調用的是非事務方法，所以chain.isEmpty()條件成立，走反射執行目標方法。

執行鏈： 代理類noTransactionalSave()-->MethodInterceptor.intercept方法-->methodProxy.invoke方法--->目標類.noTransactionalSave()-->目標.save1()方法。

我們看出事務方法save1()雖然有事務註解，但是因為其被非事務方法調用。在本次調用鏈中，沒有走事務增強分支，所以導致了事務的失效。

結論：同類中非事務方法調用事務方法，因為執行鏈不會走事務增強，導致此次調用中事務方法的事務失效。

解決辦法：

  • 出現此中業務場景時，可將非事務方法與事務方法抽象到不同的類中。事務方法所在類會創建事務代理，調用事務方法，走事務增強。
  • 看MethodInterceptor.intercept()中開頭有一個if(this.advised.exposeProxy)的判斷，如果為真，就會把事務代理對象放到事務上下文中，我們可以在非事務方法中，使用AopContext.currentProxy()獲取代理對象,強制save1方法走代理執行。

項目中，我們遇到最多的應該就是事務失效問題，以前總是對事務失效問題模模糊糊，最近看了AOP源碼，才對此有了深刻理解。

2.2 同類中事務方法調用事務方法

同類中事務方法調用事務方法，雖然不會出現事務失效問題。但是卻會出現事務傳播失效問題。

執行鏈： 代理方法.save1()--->MethodInterceptor.intercept方法--->methodProxy.invoke方法--->MethodInvocation.proceed執行增強器鏈-->Adivce.invoke增強方法--->目標方法.save1--->目標方法.save2

可以看出save1是事務方法，會走事務增強，但是save2的調用，是在save1方法增強調用鏈中，也就是說save2是依靠save1的事務。save2的事務配置，因為沒有走Adivce(TransactionInterceptor).invoke增強方法,他的事務傳播屬性不會被解析。

結論：同類中事務方法調用事務方法，傳播屬性失效。

解決辦法：同1中解決

2.3 調用同步事務方法

最近同事就遇到了這個問題呢。他在事務方法上加了synchronized關鍵詞，讓對此方法的調用強制串行化。

  • 第一個線程查詢數據庫，沒有對應的實例數據，則創建一個數據
  • 第二個線程進入方法查詢數據庫，有了對應的值，則不用創建。 以此來達到冪等性問題。

想法是好的，但是結果確是有時還是會創建兩條。

問題出在哪裡呢？

首先： 在動態代理一節講過，所謂的代理是在內存中生成了新的字節碼文件。 在CGLB情況下，TransactionalService類生成的代理類TransactionalService$$EnhancerBySpringCGLIB$$cdcbf2e8中的syncSave()並沒有synchronized修飾的。

 public final String syncSave() { try { MethodInterceptor var10000 = this.CGLIB$CALLBACK_0; if (var10000 == null) { CGLIB$BIND_CALLBACKS(this); var10000 = this.CGLIB$CALLBACK_0; } return var10000 != null ? 其次： 再看看TransactionInterceptor對事務的增強那一段關鍵代碼。事務的增強邏輯是在TransactionInterceptor中的。

			//創建事務			TransactionInfo txInfo = createTransactionIfNecessary(tm, txAttr, joinpointIdentification);			Object retVal = null;			try {				//執行目標方法				retVal = invocation.proceedWithInvocation();			}			catch (Throwable ex) {				//異常回滾				completeTransactionAfterThrowing(txInfo, ex);				throw ex;			}			finally {				cleanupTransactionInfo(txInfo);			}			//提交事務			commitTransactionAfterReturning(txInfo);			return retVal;

也就是說：兩個線程都阻塞到目標方法上，當第一個線程離開目標方法時，他的調用回到了TransactionInterceptor中，做提交事務動作。但是此時第二個線程進入目標方法去執行數據庫查詢操作，但第一個線程的提交事務動作可能為完成，此時數據庫中並沒有，所以第二個線程也會創建一個實例數據。這樣就可能出現創建兩條的情況。

可以看出加鎖的範圍沒有包括到整個事務，把synchronized加到方法上出發點時好的。但是真實的調用確實從代理類開始的，生成的代理類中並沒有加synchronized，結合事務的增強。導致了synchronized同步的失效。

結論：@Transcational註解和synchronized鎖一起使用會導致同步失效。加鎖的範圍沒有包括到整個事務

解決辦法：既然兩個在一起，不能使用鎖範圍覆蓋整個事務。那就創造一個覆蓋整個事務的調用。鎖方法與事務方法分離到不同的類中，在鎖方法中調用事務方法，不就完成了鎖範圍覆蓋整個事務了嗎

透過表象看本質。
