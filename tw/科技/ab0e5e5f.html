java concurrent包下，同步器你瞭解多少

2020-11-14
| 科技

同步器，這個概念陌生嗎？可能很多人發沒反應過來這是什麼，這個玩意到底有什麼作用呢，今天主要聊一聊它。

當我們遇到有相互作用的線程集滿足了以下將要提到的模式時，那麼我們可以直接使用合適的 庫類，而不需要也不建議我們手工去實現一些鎖與判別條件。

1、信號量（Semaphore）

之所以第一個提它，可能是因為這個在我們日常中相對最常用的一個了吧。通俗說，信號量就是管理許多的“通行證”，如果要通過信號量，則線程需要調用acquire請求才可得到許可。信號量僅僅只是維護一個計數，而“通行證”的數量是固定的，由此來限制通過的線程數量。而其他線程想要通過，則需要調用release釋放許可（實際中，任何線程都可以釋放任意數目的許可，帶來的後果是可能會增加通行證的數目以至於超過原始設置的閾值，需要謹慎使用）。

總結，它能做什麼：允許線程集等待直到被允許繼續允許為止，達到限制訪問資源的線程總數的目的。假設允許數目設為1，那麼會阻塞線程直到另一個線程去釋放許可為止。

2、倒計時門栓（CountDownLatch）

這個也比較常用，它讓一個線程集等待直到計數變成0為止。值得一提的是，它是一次性的，一旦計數為0，就不能再重用了。

舉個例子，假設有兩個線程A、B、C，A和B要完成最終的任務，但是需要一些前置數據，而C能完成A和B所需要的數據，這時，A和B就會在啟動後一直等待（就像是被栓在門外了），當C的數據準備完成時，調用了countDown，此時，A和B就可以拿著C的數據繼續工作了。

總結，它能做什麼：允許線程集等待直到計數器減至0。

3、障柵（CyclicBarrier）

這個相對前兩個相對陌生一些了吧，解釋一下，障柵就是CyclicBarrier類實現了一個集結點。假設這樣一個場景，有大量線程運行在一次計算的不同部分時，當所有部分都準備好了，需要把結果組合在一起。在這過程中，一個線程完成了他的那部分任務後，我們讓它運行到“障柵”處，一旦所有線程都到達了這個障柵，障柵就撤銷了，線程就可以繼續運行了。

用偽碼簡單描述下：

//先構造一個障柵CyclicBarrier cb = new CyclicBarrier(threads);//每個線程可以做一些自己的事，完成後再障柵上調用awaitpublic void run(){ doSomething(); //await方法可以設置超時參數，如cb.await(1000,TimeUnit.MILLISECOND); cb.await();}

如果任何一個在障柵上等待的線程離開了障柵，則障柵就被破壞了。比如線程可能執行時中斷，或者在調用await方法超時了。此時那些已經在等待的線程將會立即終止await的調用。當我們在構造障柵時，還可以提供一個障柵動作，當所有線程到達障柵時就會執行這個動作。不如：

//動作action可以收集那些thread的運行結果Runnable action = ....;CyclicBarrier cb = new CyclicBarrier(threads,action);

值得注意的CyclicBarrier是循環的，即在所有等待線程被釋放後它是可以被重用的，這一點與CountDownLatch不同，大家可以對比掌握。

總結，它能做什麼：允許線程集等待直至其中預定數目的線程到達一個公共障柵，然後可以選擇執行一個處理障柵的動作。

4、交換器（Exchanger）

這個簡單說明下，當兩個線程在同一個數據緩衝區的兩個實例上工作時，就可以使用交換器了。比如：一個線程A向緩衝區填充數據，另一個線程B消耗這些數據，當A、B都完成後，互相交換緩衝區。

總結，它能做什麼：允許兩個線程在要交換的對象準備好時交換對象。

5、同步隊列（SynchronousQueue）

同步隊列就是一種將生產者與消費者線程配對的機制。當一個線程調用SynchronousQueue的put方法時，它會阻塞直到另一個線程調用take方法為止，反之亦然。

注意SynchronousQueue類實現了BlockingQueue接口，它沒有包含任何元素，因此調用它的size方法時始終會返回0。

總結，它能做什麼：允許一個線程把對象交給另一個線程，與Exchanger不同，數據僅沿著一個方向傳遞，不像Exchanger是雙向的。
