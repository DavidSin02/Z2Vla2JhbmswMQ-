如何打造可以無限擴展的分佈式消息隊列？

2020-10-29
| 科技

幾十年前，消息隊列開始興起，它用於連接大型機和服務器應用程序，並逐漸在企業的服務總線與事件總線設計模式、應用間的路由和數據遷移中發揮至關重要的作用。自此，應用程序架構和數據角色經歷了重大變化：例如，面向服務的架構、流處理、微服務、容器化、雲服務和邊緣計算，這些只是諸多變化中的冰山一角。這些變化創造了大量的新需求，這些新需求遠遠超出了原有消息隊列的技術能力。

為了滿足這些需求，處理消息隊列的全新方法應運而生。現代應用對消息解決方案的要求不僅僅是主動連接、移動數據，而是要在持續增長的服務和應用中智能處理、分析和傳輸數據，並且在規模持續擴大的情況下不增加運營負擔。

為了滿足上述要求，新一代的消息傳遞和數據處理解決方案 Apache Pulsar 應運而生。Apache Pulsar 起初作為消息整合平臺在 Yahoo 內部開發、部署，為 Yahoo Finance、Yahoo Mail 和 Flickr 等雅虎內部關鍵應用連接數據。2016 年 Yahoo 把 Pulsar 開源並捐給 Apache 軟件基金會（ASF），2018 年 9 月 Pulsar 畢業成為 ASF 的頂級項目，逐漸從單一的消息系統演化成集消息、存儲和函數式輕量化計算的流數據平臺。

Pulsar 的設計是為了方便和現有的 Kafka 部署集成，同時也方便開發人員將其連接到應用程序。Pulsar 最初就是為連接 Kafka 構建的。Pulsar 提供和 Kafka 兼容的 API，無需更改代碼，只要使用 Pulsar 客戶端庫重新編譯，現有應用程序即可連接到 Kafka。 Pulsar 還提供內置的 Kafka 連接器，可以消費 Kafka topic 的數據或將數據發佈到 Kafka topic。

系統架構是軟件最底層的設計決策，一旦實施，就很難改變。架構決定了軟件特性和根本不同。Apache Pulsar 在功能上有很多優勢，例如統一的消費模型，多租戶，高可用性等等，但最本質、最重要的區別還是 Apache Pulsar 的系統架構。Apache Pulsar 的設計架構與其他消息傳遞解決方案（包括 Apache Kafka）的架構有著本質不同，Pulsar 從設計時就採用了分層分片式的架構，以提供更好的性能、可擴展性和靈活性。

現實生活中，存在的消息系統有很多，Yahoo 為什麼研發自己的消息系統呢？因為已有的消息系統無法解決 Yahoo 遇到的問題和規模，Yahoo 需要多租戶，能夠支撐上百萬的 topics，同時滿足低延遲、持久化和跨地域複製要求。而現有的消息系統，存在如下諸多問題：

  • 分區模型緊耦合存儲和計算，不是雲原生（Cloud Native）的設計。
  • 存儲模型過於簡單，對文件系統依賴太強。
  • IO 不隔離，消費者在清除 Backlog 時會影響其他生產者和消費者。
  • 運維複雜，替換機器、服務擴容需重新均衡數據。

於是，我們決定開始研發 Pulsar 來解決消息隊列的擴展性問題。解決擴展性問題的核心思路是數據分片，Pulsar 從設計時就採用了分層分片式的架構，以提供更好的性能、可擴展性和靈活性。

下面我們從技術角度來詳細解析 Apache Pulsar 的架構。

Pulsar 的分層架構

從數據庫到消息系統，大多數分佈式系統採用了數據處理和數據存儲共存於同一節點的方法。這種設計減少了網絡上的數據傳輸，可以提供更簡單的基礎架構和性能優勢，但其在系統可擴展性和高可用性上會大打折扣。

Pulsar 架構中數據服務和數據存儲是單獨的兩層：數據服務層由無狀態的 “Broker” 節點組成，而數據存儲層則由 “Bookie” 節點組成。

圖 1. 傳統單體架構 vs. Pulsar 存儲計算分層架構

這種存儲和計算分離的架構給 Pulsar 帶來了很多優勢。首先，在 Pulsar 這種分層架構中，服務層和存儲層都能夠獨立擴展，可以提供靈活的彈性擴容。特別是在彈性環境（例如雲和容器）中能夠自動擴容縮容，並動態適應流量的峰值。並且， Pulsar 這種分層架構顯著降低了集群擴展和升級的複雜性，提高了系統可用性和可管理性。此外，這種設計對容器是非常友好的，這使 得 Pulsar 也成為了流原生平臺的理想選擇。

Pulsar 系統架構的優勢也包括 Pulsar 分片存儲數據的方式。Pulsar 將主題分區按照更小的分片粒度來存儲，然後將這些分片均勻打散分佈在存儲層的 “bookie” 節點上。這種以分片為中心的數據存儲方式，將主題分區作為一個邏輯概念，分為多個較小的分片，並均勻分佈和存儲在存儲層中。這種架構設計為 Pulsar 帶來了更好的性能，更靈活的擴展性和更高的可用性。

Pulsar 架構中的每層都可以單獨設置大小，進行擴展和配置。根據其在不同服務中的作用不同，可靈活配置集群。對於需要長時間保留的用戶數據，無需重新配置 broker，只要調整存儲層的大小。如果要增加處理資源，不用重新強制配置存儲層，只需擴展處理層。此外，可根據每層的需求優化硬件或容器配置選擇，根據存儲優化存儲節點，根據內存優化服務節點，根據計算資源優化處理節點。

圖 2. Apache Pulsar 分層分片的架構

而大多數消息隊列技術（包括 Apache Kafka）都採用單體架構，其消息處理和消息持久化（如果提供了的話）都在集群內的同一個節點上。這種體系結構在大多數傳統的數據庫平臺以及 Hadoop 等大數據系統中也較為常見，與昂貴的外部存儲陣列的常見替代方案相比，其設計目的在於將數據的計算與存儲放到同一臺機器上來處理，以減少網絡流量和訪問延遲，同時降低存儲成本。這種方法在小型環境中很容易部署，但在性能、可伸縮性和靈活性方面存在明顯問題。隨著固態磁盤的廣泛使用，網絡帶寬的迅速提升以及存儲延遲的顯著降低，已經沒有必要採用單體架構進行這種權衡處理了。

接下來，我們結合數據處理中各種不同的 IO 訪問模式來深入瞭解 Pulsar 系統架構的優勢。

IO 訪問模式的優勢

流系統中通常有三種 IO 訪問模式：

1.寫（Writes）：將新數據寫入系統中；

2.追尾讀（Tailing Reads）：讀取最近寫入的數據；

3.追趕讀（Catch-up Reads）：讀取歷史的數據。例如當一個新消費者想要從較早的時間點開始訪問數據，或者當舊消費者長時間離線後又恢復時。

和大多數其他消息系統不同，Pulsar 中這些 IO 訪問模式中的每一種都與其他模式隔離。在同樣 IO 訪問模式下，我們來對比下 Pulsar 和其他傳統消息系統（存儲和服務綁定在單個節點上，如 Apache Kafka）的不同。

傳統消息系統（圖 3 左側圖）中，每個 Broker 只能利用本地磁盤提供的存儲容量，這會給系統帶來一些限制：

1.Broker 可以存儲和服務的數據量受限於單個節點的存儲容量。因此，一旦 Broker 節點的存儲容量耗盡，它就不能再提供寫請求，除非在寫入前先清除現有的部分數據。

2. 對於單個分區，如果需要在多個節點中存儲多個備份，容量最小的節點將決定分區的最終大小。

相比之下，在 Apache Pulsar（圖 3 右側圖）中，數據服務和數據存儲是分離的，Pulsar 服務層的任意 Broker 都可以訪問存儲層的所有存儲節點，並利用所有節點的整體存儲容量。在服務層，從系統可用性的角度來看，這也有著深遠的影響，只要任一個 Pulsar 的 Broker 還在運行，用戶就可以通過這個 Broker 讀取先前存儲在集群中的任何數據，並且還能夠繼續寫入數據。

下面我們來詳細看一下在每種 IO 訪問模式下的架構優勢。

寫

在傳統消息系統架構中，一個分區的所有權會分配給 Leader Broker。對於寫請求，該 Leader Broker 接受寫入並將數據複製到其他 Broker。如圖 3 左側所示，數據首先寫入 Leader Broker 並複製給其他 followers。 數據的一次持久化寫入的過程需要兩次網絡往返。

在 Pulsar 系統架構中，數據服務由無狀態 Broker 完成，而數據存儲在持久存儲中。數據會發送給服務該分區的 Broker，該 Broker 並行寫入數據到存儲層的多個節點中。一旦存儲層成功寫入數據並確認寫入，Broker 會將數據緩存在本地內存中以提供追尾讀（Tailing Reads）。

圖 4. Writes 訪問模式對比


如圖 4 所示，和傳統的系統架構相比，Pulsar 的系統架構並不會在寫入的 IO 路徑上引入額外的網絡往返或帶寬開銷。而存儲和服務的分離則會顯著提高系統的靈活性和可用性。

追尾讀

對於讀取最近寫入的數據場景，在傳統消息系統架構中，消費者從 Leader Broker 的本地存儲中讀取數據；在 Pulsar 的分層架構中，消費者從 Broker 就可以讀取數據，由於 Broker 已經將數據緩存在內存中，並不需要去訪問存儲層。

圖 5. Tailing Read 訪問模式對比


這兩種架構只需要一次網絡往返就可以讀取到數據。由於 Pulsar 在系統中自己管理緩存中的數據，沒有依賴文件系統緩存，這樣 Tailing Reads 很容易在緩存中命中，而無需從磁盤讀取。傳統的系統架構一般依賴於文件系統的緩存，讀寫操作不僅會相互競爭資源（包括內存），還會與代理上發生的其他處理任務競爭。因此，在傳統的單片架構中實現緩存並擴展非常困難。

追趕讀

追趕讀（Catch-up Reads）非常有趣。傳統的系統架構對 Tailing reads 和 Catch-up reads 兩種訪問模式進行了同樣的處理。即使一份數據存在多個 Broker 中，所有的 Catch-up reads 仍然只能發送給 Leader Broker。

Pulsar 的分層架構中歷史（舊）數據存儲在存儲層中。Catch-up 讀可以通過存儲層並行讀取數據，而不會與 Write 和 Tailing Reads 兩種 IO 模式競爭或干擾。

三種 IO 模式放在一起看

最有趣的是當你把這些不同的模式放在一起時，也就是實際發生的情況。這也正是單體架構的侷限性最令人痛苦的地方。傳統的消息系統架構中，所有不同的工作負載都被髮送到一箇中心（Leader Broker）位置，幾乎不可能在工作負載之間提供任何隔離。

然而，Pulsar 的分層架構可以很容易地隔離這些 IO 模式：服務層的內存緩存為 Tailing Reads 這種消費者提供最新的數據；而存儲層則為歷史處理和數據分析型的消費者提供數據讀取服務。

圖 6. 三種 IO 模式對比

這種 IO 隔離是 Pulsar 和傳統消息系統的根本差異之一，也是 Pulsar 可用於替換多個孤立系統的關鍵原因之一。Apache Pulsar 的存儲架構讀、寫分離，能保證性能的一致性，不會引起數據發佈和數據消費間的資源競爭。已發佈數據的寫入傳遞到存儲層進行處理，而當前數據直接從 broker 內存緩存中讀取，舊數據直接從存儲層讀取。

超越傳統消息系統

上面討論了 Pulsar 的分層架構如何為不同類型的工作負載提供高性能和可擴展性。Pulsar 分層架構帶來的好處遠遠不止這些。我舉幾個例子。

無限的流存儲

並行訪問流式計算中的最新數據和批量計算中的歷史數據，是業界一個普遍的需求。

由於 Pulsar 基於分片的架構，Pulsar 的一個主題在理論上可以達到無限大小。當容量不足時，用戶只需要添加容器或存儲節點即可輕鬆擴展存儲層，而無需重新平衡數據；新添加的存儲節點會被立即用於新的分片或者分片副本的存儲。

Pulsar 將無界的數據看作是分片的流，分片分散存儲在分層存儲（tiered storage）、BookKeeper 集群和 Broker 節點上，而對外提供一個統一的、無界數據的視圖。其次，不需要用戶顯式遷移數據，減少存儲成本並保持近似無限的存儲。因此，Pulsar 不僅可以存儲當前數據，還可以存儲完整的歷史數據。

圖 7. 無限的流存儲

數據查詢和數據分析

Pulsar 有能力存儲數據流的完整歷史記錄，因此用戶可以在其數據上使用各種數據工具。Pulsar 使用 Pulsar SQL 查詢歷史消息，使用 Presto 引擎高效查詢 BookKeeper 中的數據。Presto 是用於大數據解決方案的高性能分佈式 SQL 查詢引擎，可以在單個查詢中查詢多個數據源的數據。Pulsar SQL 允許 Presto SQL 引擎直接訪問存儲層中的數據，從而實現交互式 SQL 查詢數據，而不會干擾 Pulsar 的其他工作負載。Pulsar 與 Presto 的集成就是一個很好的例子，如下是使用 Pulsar SQL 查詢的示例。

Apache Pulsar 的周邊生態

批處理是對有界的數據進行處理，通常數據以文件的形式存儲在 HDFS 等分佈式文件系統中。流處理將數據看作是源源不斷的流，流處理系統以發佈 / 訂閱方式消費流數據。當前的大數據處理框架，例如 Spark、Flink 在 API 層和執行層正在逐步融合批、流作業的提交與執行，而 Pulsar 由於可以存儲無限的流數據，是極佳的統一數據存儲平臺。Pulsar 還可以與其他數據處理引擎（例如 Apache Spark 或 Apache Flink）進行類似集成，作為批流一體的數據存儲平臺，這進一步擴展了 Pulsar 消息系統之外的角色。下圖展示了 Pulsar 的周邊生態。

總結

Apache Pulsar 是雲原生的分佈式消息流系統，採用了計算和存儲分層的架構和以 Segment 為中心的分片存儲，因此 Apache Pulsar 具有更好的性能、可擴展性和靈活性，是一款可以無限擴展的分佈式消息隊列。

Apache Pulsar 是一個年輕的開源項目，擁有非常多吸引人的特性。Pulsar 社區的發展迅猛，在不同的應用場景下不斷有新的案例落地。期待大家能和 Apache Pulsar 社區深入合作，一起進一步完善、優化 Pulsar 的特性和功能。
