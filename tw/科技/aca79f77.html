面向接口編程

有一段時間，我是面向接口編程的堅決執行者；而且在大多數場合，我也完全認可並實踐面向接口編程。

但凡事總有例外，我也在反思面向接口編程

套娃式開發

java 比較經典的一種套娃場景就是數據庫裡一些 CURD 的 service 對 mapper(或 dao) 的封裝，很多時候，service 和 mapper 的接口定義是一樣的，service 的實現無非就是簡單的調用一次 mapper 的實現，在長期的開發實踐裡，我甚至做了一個代碼生成器，自動幫我完成 service 到 mapper 的套娃

我甚至還在一些項目裡見過 3 層套娃：service 裡注入 dao，dao 裡注入 mapper，mapper 的最終實現由 mybatis 框架注入。為了看到某些業務邏輯的 sql 語句，要跳轉多次，我不由得發出靈魂質問：有必要嗎？

java 還不夠繁瑣嗎？用 python 寫幾十行代碼可以搞定的需求，java 可能連配置文件都沒有寫完——就不能簡潔一些嗎？

助手類

所謂助手類是一些以 Helper 結尾的類，大多數時候它和 utils 工具類很相似；不同的是它沒法提供靜態方法，因為它需要一個實例化的過程，這個實例化過程一般是因為它需要外部注入一些依賴

比如一個加密的助手類，採用的算法是 AES，但是密鑰是從配置中心注入的，並且還支持配置中心修改後熱加載，這樣就不能用一個靜態的工具類了，那麼我們就可以把它實現為一個助手類，如下示例

public class EncryptHelper {        @Value("aes_key")    private String aesKey;            public String encrypt(String plainText) {        return AesUtil.encryptHex(plainText, aesKey);    }}

又比如，我們的廣告競價業務採用的二價，計算一批候選廣告的價格是可以用一個靜態工具類來完成的，但隨著業務的發展，我們的二價支持壓榨因子，而這個壓榨因子是可以由運營同學隨時修改的，這樣計算價格的靜態工具類就不得不重構成支持注入壓榨因子的助手類

這種助手類本質上就是個工具類，如果一定要抽取出接口來就會非常彆扭

我的實踐

拒絕套娃

  • 如果 service 就是 mapper 的簡單套娃，那麼就不需要 service 了，直接用 mapper
  • 一些簡單業務，需要多個 mapper 一起完成，如果重用性不高，也不會專門聲明一個 service 來實現

助手類

需要謹慎區分當前場景是適用 service 還是 helper，如果確定是 helper，不需要抽象出接口
