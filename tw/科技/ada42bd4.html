1-MyBatis

在移動互聯網時代， MyBatis 成為了目前互聯網 Java 持久框架的首選，與 Hibernate 消除 SQL 不同， MyBatis 不屏蔽 SQL。不屏蔽 SQL 的優勢在於，程序員可以自己制定 SQL規則，無須 Hibernate 自動生成規則，這樣能夠更加精確地定義 SQL，從而優化性能。它更符合移動互聯網高併發、大數據、高性能 、 高響應的要求。與 Hibernate 一樣 ， MyBatis 也需要一個映射文件把 POJO 和數據庫的表對應起來。MyBatis 映射文件如代碼清單 1-8 所示。

這裡的 resultMap 元素用於定義映射規則 ，而實際上 MyBatis 在滿足一定的規則下，完成自動映射，而增、刪、查、改對應著 insert 、 delete 、 select、 update 四個元素，十分明瞭 。
注意， mapper 元素中的 namespace 屬性，它要和一個接口的全限定名保持一致，而裡面的 SQL 的 id 也需要和接口定義的方法完全保持一致，定義 MyBatis 映射文件，如代碼清單 1-9 所示。
package com . learn . chapterl . mapper;

import com . learn.chapterl.pojo.Role ;

public interface RoleMapper {
public Role getRole(Integer id) ;

public int deleteRole(Integer id);

public int insertRole(Role role) ;

public int updateRole(Role role) ;
}

顯然 MyBatis 在業務邏輯上和 Hibernate 是大同小異的。其區別在於， MyBatis 需要提供接口和 SQL，這意味著它的工作量會 比 Hibernate 大，但是由於自定義 SQL 、 映射關係，所以其靈活性、可優化性就超過了 Hibernate。互聯網可優化性、靈活性是十分重要 的， 因為一條 SQL 的性能可能相差十幾倍到幾十倍，這對於互聯網系統是十分重要的 。
2-Hibernate 和 MyBatis 的區別

Hibernate 和 MyBatis 的增、刪、查、改，對於業務邏輯層來說大同小異，對於映射層而言 Hibernate 的配置不需要接口和 SQL，相反 MyBatis 是需要的。對於 Hibernate 而言 ，不需要編寫大量的 SQL，就可以完全映射，同時提供了日誌、緩存、級聯（級聯比 MyBatis強大）等特性， 此外還提供 HQL (Hibernate Query Language ）對 POJO 進行操作，使用十分方便，但是它也有致命的缺陷。由於無須 SQL，當多表關聯超過 3 個的時候，通過 Hibernate 的級聯會造成太多性能的丟失，又或者我現在訪問一個財務的表，然後它會關聯財產信息表，財產又分為機械、原料等，顯然機械和原料的字段是不一樣的，這樣關聯宇段只能根據特定的條件變化而變化，而 Hibernate 無法支持這樣的變化 。 遇到存儲過程， Hibernate 只能作罷。更為關鍵的是性能，在管理系統的時代，對於性能的要求不是那麼苛刻，但是在互聯網時代性能就是系統的根本，響應過慢就會喪失客戶，試想一下誰會去用一個經常需要等待超過 10 秒以上的應用呢？以上的問題 MyBatis 都可以解決， MyBatis 可以自由書寫 SQL、支持動態 SQL、處理列表、動態生成表名、支持存儲過程。這樣就可以靈活地定義查詢語句 ，滿足各類需求和性能優化的需要，這些在互聯網系統中是十分重要 的。但 MyBatis 也有缺陷 。首先，它要編寫 SQL 和映射規則，其工作量稍微大於 Hibernate 。
其次，它支持的工具也很有限，不能像 Hibernate 那樣有許多的插件可以幫助生成映射代碼和關聯關係，而即使使用生成工具，往往也需要開發者進一步簡化， MyBatis 通過手工編碼，工作量相對大些。所以對於性能要求不太苛刻的系統，比如管理系統、 ERP 等推薦使用 Hibernate；而對於性能要求高、響應快、靈活的系統則推薦使用 MyBatis 。
