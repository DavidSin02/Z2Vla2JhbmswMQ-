接口測試斷言詳解(JMeter)

2020-11-14
| 科技

接口測試是目前最主流的自動化測試手段，它向服務器發送請求，接收和解析響應結果，通過驗證響應報文是否滿足需求規約來驗證系統邏輯正確性。接口的響應類型通過Content-Type指定，常見的響應類型有：

• text/html ： HTML格式

• text/plain ：純文本格式

• text/xml ： XML格式

• application/xml ： XML數據格式

• application/json ： JSON數據格式

JMeter通過響應斷言組件，Json斷言組件實現對響應報文的解析和驗證。

• 響應斷言可以驗證任意格式的響應報文

• Json斷言適用於Json格式的響應報文

響應斷言

響應斷言可以選擇斷言驗證的範圍(URL，響應頭，響應碼，響應體，響應附件)，匹配的模式(純文本還是正則表達式匹配)，具體選項如下：

Apply to (響應斷言的應用範圍)

• Sub-samples適用於發送一個請求同時觸發多個子請求的情況，一般情況下推薦使用Main Sample only，僅校驗發起的請求響應。對跟隨重定向的請求，重定向後的請求是主請求。

• Jmeter Variable可對Sampler中生成的Jmeter變量進行校驗，此處寫明變量名。

要測試的響應字段(可通過取樣器結果查看)

• 響應文本：最常用的選項，服務器的響應文本(不包含響應頭信息)

• Document(text)：除了文本響應還支持 PDF/ Office/ Audio/ Video ，Apache Tika 解析服務器響應內容，很耗內存而且很容易解析失敗，非特殊需求不建議使用此選項。

• Response Headers：響應頭

• URL樣本: 請求url，如果有重定向包含重定向url

• 響應代碼:Http返回的響應碼

• 響應信息： 響應代碼對應的響應信息eg:OK(200)，可在結果樹中查看代碼和信息

• Ignore Status: 當http 響應代碼為400/500時，jmeter默認請求失敗，要驗返回碼為500需勾選ignore status。Sampler下多個斷言是疊加作用的，只要有一個斷言勾選了Ignore Status就可以。

響應斷言：模式匹配

• 包括：支持純文本和正則，驗證返回包括指定的內容

• 匹配：支持純文本和正則，正則需全匹配(正則必須匹配全部返回，而非部分返回)

• Equals：字符串相等，純文本匹配，驗證返回結果和指定結果完全一致

• SubString：字符串包含，純文本匹配，驗證返回結果包含指定結果

• 否：結合上述條件取反，若上述斷言結果為false，取否後，最終斷言結果為true

Json斷言

Json斷言是針對Json報文的斷言方式，通Json Path提取出Json響應報文中的字段，再採用純文本或者正則去驗證Json Path的提取結果，Json結合了Json Path和正則表達式，有如下選項：

• Additionally assert value：文本驗證，此處是完全匹配，勾選上此選項後再勾選Match as regular expression，可以觸發正則匹配。

• Match as regular expression：支持正則表達式匹配

• Expect null：判定返回為null

• Invert assertion：倒置斷言結果

Json Path是用於確定 JSON 文檔中某部分元素位置的語言。它將 JSON 數據轉換為 DOM 樹狀結構，並提供在數據結構中尋找節點的能力。Jmeter提供組件來驗證Json Path的語法正確性，以及元素匹配情況。可以在察看結果樹中將響應報文的格式從Text切換為Json Path Tester，並在Json Path Expression欄中輸入相應的Json Path ，點擊Test，察看JsonPath的定位結果。

目前接口測試最主流的交換文檔就是Json報文，無論是響應斷言還是Json斷言，核心都是對正則表達式，JsonPath 語法的理解。Jmeter 內置了JsonPath的校驗工具，針對正則表達式，可通過在線正則表達式工具進行測試：http://tool.oschina.net/regex/

響應斷言和Json斷言可以涵蓋大部分的接口校驗需求，針對更加複雜的接口校驗需求，比如數據庫校驗，比如複雜計算邏輯的校驗 ，可通過Beanshell斷言組件編寫腳本來實現斷言。
