「 原創 」理解 JavaScript 中抽象的執行上下文

2020-10-29
| 科技

乍一看，執行上下文這個概念確實抽象，特別是初學 JavaScript 時，若是未深入去了解的情況下，被問及什麼是執行上下文，或者說說你對執行上下文的理解的時候，就更顯得茫然不知所措了。

先從字面意思入手理解，上下文是由 context 翻譯而來，其也有環境、來龍去脈、背景等意思，理解成執行環境或執行過程的來龍去脈，是否會更好理解？也就說，執行上下文是 JavaScript 代碼執行的來龍去脈，理解這個過程，將更有利於我們的代碼設計，能設計出讓觀者拍手稱其精妙的代碼。

再從 JavaScript 中，萬物皆對象的層面去理解，將“執行上下文”理解成就是一個個層層嵌套或者相互關聯的對象所組成的一個對象，這個對象我們稱之為環境，也即所謂的上下文。從這個層面去切入的話，也有利於接下來詳細分析的整個過程。

眾所周知 JavaScript 是單線程的，但這並不是說 JavaScript 代碼是一行一行執行的，JavaScript 是一塊一塊的執行代碼的，而這一塊一塊的代碼，從物理層面上來理解的話，就是這個執行上下文（一個對象）。

棧

在拆解整個過程前，先簡單瞭解下棧數據結構。棧數據結構的特點是：先進後出，後進先出。類似漢諾塔，最先進入的在底部，稱之為棧底，反之在最上面的稱之為棧頂。

執行上下文棧


類型

全局執行上下文

是基礎或默認的執行上下文，一個程序只有一個，任何不在函數內部的代碼都在全局上下文中。在瀏覽器中，只有當瀏覽器關閉或者程序退出的時候，該執行上下文才會被銷燬，也就是說在 JavaScript 程序啟動後，首先創建了一個全局的執行上下文，且在銷燬前一直都處於執行棧的最底層。

// ECS (Execution Context Stack) 執行棧ECS = {    // 全局執行上下文    globalExectionContext}

函數執行上下文

JavaScript 執行流在遇到函數的時候，會為其創建一個函數執行上下文。不論是否重複調用，每遇到一個函數就會創建一個函數執行上下文，然後將其壓入執行棧，等待調用執行。

function func1() {    // ...    console.log('函數執行上下文');}// 執行棧ECS = {    // 函數執行上下文    func1ExectionContext,    // 全局執行上下文    globalExectionContext}

Eval 函數執行上下文

eval 函數的作用是將傳入的字符串當作 JavaScript 腳本來執行。官方都建議我們不要使用這個危險函數，改用更加安全的 window.Function，這裡就不討論這一塊內容了，而其所創建的執行上下文與函數執行上下文是一樣的，為何單獨形成一個類別，主要是 eval 函數作用域的不同。它是根據調用時所處的環境來確定作用域的。

this 值的綁定

this 值的指向，要根據執行上下文來確定，這裡說的都是在瀏覽器中的情況：

如果是在全局執行上下文中，this 指向的就是 window 對象。

如果是在函數執行上文中，this 的指向主要取決於函數是如何被調用的，也就是說如果調用該函數的是一個引用對象，那 this 指向的就是這個引用對象，如果不是的話，this 就可能指向 window 對象或者在嚴格模式下的 undefined 了。

let mine = {    name: function() {        console.log('my name is makeit.vip')    }}// 這個 this 指向的是 mine// 因為 name 是被 mine 這個引用對象多調用mine.name()const name = mine.name// 這裡 this 指向的就是 window 對象了// 因為它並沒有被其它引用對象調用name()


詞法 / 變量環境

詞法 / 變量環境，這兩個帶“環境”字眼的詞彙，聽起來似乎挺高級的。

先說說詞法環境（Lexical Environments），按照官方定義中的說法，意思就是用 ECMAScript 規定好的詞法（比如 FunctionDeclaration），採用嵌套結構的方式來定義標識符與變量和函數之間的關聯。

還是從對象層面來理解吧，有一個對象，它叫“詞法環境”，規範中，為這個對象定義了兩個 key 值，分別為 Environment Record（環境記錄），outer Lexical Environment（外部詞法環境），在這兩個 key 值下又規定了其它更具體的 key 值，也就是屬性，以便在執行解析時找到相應的屬性，並快速知曉其作用，且能根據此來調用 / 執行某項操作。

需要注意 OuterLexicalEnvironment 外部詞法環境引用，與上下文類別有關，全局執行上下文沒有外部詞法環境引用，即為 null，而函數執行上下文要根據函數本身分情況而論，可能外部詞法環境引用為全局環境，也可能是包含了此函數的外部函數環境，但不會為 null。

FEC = {    LE: {        ER: {...},        OLE: {Global / Outer Function Environment}    }}

環境記錄（EnvironmentRecord）又細分了聲明式環境記錄（Declarative Environment Record）和對象環境變量（Object Environment Record）。DER 主要記錄的是在作用域內聲明定義的各種常量 / let / class / function 等，OER 記錄的就是對象。

FEC = {    LE: {        ER: {            DER: {...},            OER: {...}        },        OLE: {G / OFE}    }}

再往下就更具體的詞法規範了，像聲明式環境記錄 DER 又可繼續細分為 函數環境記錄 / 模塊環境記錄 等等，這裡就先不細說更具體的了。回過頭繼續看變量環境（Variable Environment），變量環境也是詞法環境，唯一不同的是，詞法環境存儲的是函數聲明和變量（let 和 const）綁定，變量環境只是用來存儲變量（var）綁定。舉個簡單的例子：

const hello = 'hello'let world = 'world'var helloWorldfunction sayHello() {    return `${hello} ${world}`}helloWorld = sayHello()// 執行上下文大概長這模樣ECS = {    FEC: {        THIS: <Global Object>,        LE: {            ER: {                DER: {                    type: 'Declarative'                }            }        },        OLE: <GLE>    },    GEC: {        THIS: <Global Object>        LE: {            ER: {                OER: {                    type: 'object',                    hello: <Uninitialized>,                    world: <Uninitialized>,                    sayHello: <Function>                }            }，            OLE: <Null>        },        VE: {            ER: {                OER: {                    type: 'Object',                    helloWorld: <Undefined>                }            },            OLE: <Null>        }    }}

注意 hello / world 兩個變量的定義，在 LE 中，默認是為 Uninitialized，而 helloWorld 這個變量，在 VE 中是 Undefined，也就是說前兩個變量是未初始化的，不能訪問，不然會導致出錯，而後一個 helloWorld 變量是初始化了，且默認值為 undefined，這就是為何在用 let / const 聲明的變量，在未定義前是不能訪問的，而可以在用 var 聲明的變量前調用該變量。


執行 / 銷燬

所有的準備工作都已經完成，接下來就是執行，執行完彈出上下文執行棧，銷燬，過程就這麼結束啦。


小結

執行上下文可以理解為執行環境，分為全局、函數、Eval 函數執行上下文，全局執行上下文肯定處於執行棧的棧底，每個函數都會對應創建一個函數執行上下文，不論該函數是否重複調用。創建執行上下文中，主要乾了三件事，分別為 this 的綁定、創建詞法環境、創建變量環境。this 值的綁定與執行上下文類別有關，全局執行上下文 this 指向的就是 window 對象，而函數執行上下文的 this 指向，要根據函數被調用的情況來區分，如果函數被引用對象調用，則 this 值會被設置成該引用對象，否則 this 會被設置成 window 對象或在嚴格模式下被設置成 undefined。詞法環境和變量環境可以理解成是對變量/函數聲明的規範約束以及存儲，兩者唯一的區別就是詞法環境用於存儲 let / const 變量和函數聲明，而變量環境僅用於存儲 var 變量。另外有個變量聲明提升問題，也就是說 var 變量在準備階段就已完成初始化操作並賦予 undefined 值，而 let / const 變量聲明在執行上下文創建階段還是處於未初始化狀態的。
