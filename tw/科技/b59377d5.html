3分鐘簡述熔斷器使用方法

2020-10-29
| 科技

熔斷器是一種設計模式，廣泛用於分佈式系統中，以防止級聯故障。在這篇文章中，我們將通過級聯故障的問題，來介紹熔斷器模式的使用方法。

動機：級聯故障的問題

在進入熔斷器模式之前，讓我們來看看它到底要解決什麼問題。

當服務 A 嘗試與服務 B 通信時，它會分配一個線程來進行該調用。在進行調用的時候，有 2 種故障可能發生，以 user 調用 friends 服務為例。

''' user service ''' def get_user_info(user_id: str):  try:  friends_service.get_friends(user_id)  except Exception as e:  raise InternalServerError

即時故障：在即時故障中，會立即引發一個異常（如：連接拒絕），並釋放服務 A 線程。

超時故障：如果 service_b 需要很長的時間來響應，當收到服務 A 的新請求時，會有越來越多的線程在等待 service_b。如果在等待超時的過程中發出了多個請求，這就會耗盡服務 A 的線程池，會使服務 A 癱瘓。

“您的代碼不能永遠等待一個不會返回的響應，遲早需要放棄等待，希望這不是一個設計方法。”-Michael T. Nygard，Release It！

下面通過一個社交媒體應用的例子來更好的解釋這個問題。我們有一個聚合器服務，這是客戶端交互的內容，它聚合了一堆服務的結果，包括 User 服務。User 服務調用 Photos 服務和 Friends 服務，而 Friends 服務又依賴 friends_db。

在這裡，friends 服務試圖向 friends_db 發出請求，但是，friends_db 並沒有立即響應失敗，而是讓 friends 服務的線程一直在等待。Friends 服務嘗試重試，從而使用更多線程。隨著它收到新請求，更多線程正在等待 friends_db 返回結果。

現在可以看到，Friends 服務現在如何成為 User 服務超時的來源。User 用盡了它的線程池來等待來自 Friends 服務的請求，就像 Friends 服務是如何等待 friends_db 一樣。現在，我們可以看到 friends_db 中的故障是如何導致間接依賴於它的服務級聯失敗的。

最終，聚合器服務也會由於相同的原因而崩潰。客戶端調用了聚合服務，因此我們的系統實際也變得不可用。

我們看到架構的一個組件中的一個錯誤是如何導致級聯故障，而導致所有其他服務崩潰。

熔斷器模式

熔斷器通常以攔截器模式/責任鏈/過濾器的形式實現，它包含3個狀態：

  • 關閉：允許攔截器將所有請求傳遞給上游服務，並將上游服務的響應傳遞給調用方。

  • 打開：不允許請求傳遞給上游，默認返回通常是錯誤響應。

  • 半開放式：某些請求被允許傳遞到上游，其他請求被終止並返回默認響應。

下圖顯示了處於三種狀態的熔斷器攔截器

讓我們看一下熔斷器的 Python 示例，可以使用以下方法創建自己的熔斷器：

from circuitbreaker import CircuitBreakerclass MyCircuitBreaker(CircuitBreaker): FAILURE_THRESHOLD = 20 RECOVERY_TIMEOUT = 60 EXPECTED_EXCEPTION = RequestException@MyCircuitBreakerdef get_user_info(user_id): try: friends_service.get_friends(user_id) except Exception as e: raise InternalServerError

我們也可以利用 sidecar 模式。通過這種方法，不必通過熔斷器來修改服務，而是使用諸如 Envoy 之類的輔助工具來交付我們的應用序。該服務的所有出站流量都是通過 envoy 進行代理。Envoy 支持開箱即用的斷路功能。以下是 Envoy 的斷路配置示例：

circuit_breakers: thresholds: - priority: DEFAULT max_connections: 1000 max_requests: 1000 - priority: HIGH max_connections: 2000 max_requests: 2000
