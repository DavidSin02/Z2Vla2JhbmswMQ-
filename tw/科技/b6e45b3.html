瞭解兩階段提交分佈式事務的工作原理

2020-10-29
| 科技

當客戶在網店購買商品時，您可能希望訂單被快速處理和交付。作為銀行客戶，您希望確保自己的資金在轉賬期間沒有神祕消失。在企業應用程序中，傳統事務可以確保事務質量，比如一致性，以及與其他事務隔離。分佈式事務可跨多種資源提供這些保證，比如跨商店數據庫和企業資源規劃 (ERP) 系統。事務質量通常由中間件基礎架構提供，而且可簡化普通程序員的工作。

但是，在雲中，中間件通常無法確保這些質量；分佈式事務通常不可用，與此同時，基礎架構可能更加脆弱。甚至細微的錯誤也有可能在應用程序中迅速擴大，並導致相當大的支持成本，因此，在可擴展的雲應用程序中避免或適當處理可能的事務管理錯誤至關重要。

這個系列分兩部分，在第 1 部分中，我將介紹事務處理的一些背景知識，事務提供的質量保證，還將介紹雲設置中的不同之處。在第 2 部分中，我將介紹調用非事務服務的傳統事務處理的替代方案。

“事務的一個主要屬性是原子性。在真實的分佈式事務中，可能無法完全實現原子性，因此一致性可能也會遭到破壞。”

事務的一般屬性

在 IT 領域，事務通常被視為對信息的單獨操作，該操作必須作為一個完整的單元，要麼完全成功，要麼完全失敗。事務絕不能部分完成。這是一個重要屬性。例如，財務操作依靠事務來確保總金額不變：如果從帳戶中取錢，您需要確保資金同時被放入另一個帳戶。

更正式地講，事務通常與“ACID” 4 個屬性有關：

  • 原子性 (Atomicity)— 事務只能完全成功或完全失敗。如果事務成功了，則執行事務中的所有操作；如果事務失敗了，則不執行任何操作。
  • 一致性 (Consistency)— 事務必須將系統從一種一致狀態過渡到另一種一致狀態。例如，寫入一個數據庫的任何數據（在提交時）必須遵守該數據庫中定義的所有約束條件。
  • 隔離 (Isolation)— 多個併發事務相互隔離，它們的行為就好像它們是按順序執行一樣。
  • 耐久性 (Durability)— 事務一旦提交，事務結果就會一直存在，甚至在斷電或出現其他錯誤時也是如此。

儘管這些是事務在理論上的原則，但實際上，出於性能考慮，一些操作和優化措施可能偏離了這些原則。

例如，儘管原子性、一致性和耐久性是事務處理系統中的重要目標，但由於嚴格隔離對性能的潛在影響，對隔離的要求通常會更寬鬆。可以定義不同事務隔離級別來確定系統中允許出現哪些隔離失敗。這些隔離級別包含在單個數據庫中，所以這裡不會重點介紹該主題。

事務通常由一個事務客戶端 調用資源管理器 來發起，該資源管理器通常是一個數據庫或消息隊列。資源管理器負責管理資源，比如數據庫中的數據庫表或行。客戶端打開一個事務，然後可通過多次調用在系統中的一個或多個資源上執行一些操作，然後嘗試提交事務（參見圖 1）。如果事務中只有一個資源管理器，它可以自行確定事務是否具有正確的原子性和一致性，並在提交時確保耐久性。

長期與短期事務

上述事務屬性適合一般意義上的事務。例如，甚至對於通過信用卡購買商品這樣的業務事務，您也希望具有這些屬性。但是，一些事務可能花費很長的時間。例如，在登記入住某家酒店時，酒店通常以電子方式詢問您的帳戶是否有足夠的資金（也就是說，對卡上的資金有限制）。但是，可能在幾天後您退房時才會從該帳戶扣費。這些事務被稱為長期 事務 — 這些業務事務通常分為多個技術事務，以避免在事務和資源管理器中阻塞（技術）資源。

請注意，資源管理器通常無法跨多個技術事務來鎖定資源。長期業務事務要能鎖定資源，通常需要調整功能數據模型來存儲資源的這一額外狀態。

例如，當客戶登記入住酒店時，一個短期技術性事務會在帳戶中預扣一定的金額。預扣狀態可存儲在數據庫中的單獨一行中，而且在預扣時，可以確認總的預扣金額不會超出客戶的信用限額。在退房時，另一個短期技術事務通過實際轉賬來交換該預扣金額。在退房時，另一個短期技術事務通過實際轉賬來交換該保留金額。每個技術事務可能（在上面提到的隔離限制內）具有 ACID 屬性，但業務事務沒有。保留資金沒有與其他事務隔離，因為其他業務事務不能保留它。

可運用特殊技術來確保長期業務事務中的原子性和一致性。例如，信用卡事務的保留資金基本上採用了一種悲觀鎖定形式。本文後面將討論如何在雲設置中使用這些技術。

（有關此主題的更多討論，請參閱 SYS-CON Media 網站上的 “Web 服務事務”。）

本地與分佈式事務

本地事務僅涉及一個資源管理器。但是在一次事務需要兩個或更多資源管理器時，會發生什麼？在需要管理兩個數據庫中的數據，或者需要根據數據庫中的某個狀態來發送消息隊列中的消息時，可能出現這種情況。在這些情況下，僅僅是先提交第一個資源管理器而後提交第二個資源管理器將不起作用。如果第二個資源管理器由於違背約束條件而回滾，而且第一個資源管理器包含在已提交的事務中，您該如何回滾它？

常用且已得到非常成功地使用的方法是，使用一個事務管理器 來協調事務中涉及的資源管理器。然後，事務管理器使用一個分佈式事務協議（通常採用兩階段提交）來提交事務。

在兩階段提交中，事務管理器首先要求所有參與的資源管理器確保事務在提交後成功完成。這是投票階段。如果所有資源管理器都同意，然後在提交階段 將事務提交到所有資源管理器。哪怕一個資源管理器拒絕提交，所有資源管理器都會被回滾，以確保一致性。圖 3 給出了一個兩階段提交事務。

圖 3. 一個使用兩階段提交協議的分佈式事務

此模型已成功用在基於主機的事務處理、Enterprise Java (JEE) 應用服務器、Microsoft Transaction Server 和其他系統中。（請參閱 “Enterprise JavaBeans (EJB) 與 Microsoft Transaction Server (MTS) 模型的詳細比較”。）

請注意，分佈式事務並不僅限於應用服務器，還可以將它用於 Web 服務。有一項針對 Web 服務的 WS-Transaction 標準，但我尚未看到它被實際使用。

現實生活中的分佈式事務

事務的一個主要屬性是原子性。在真實的分佈式事務中，可能無法完全實現原子性，因此一致性可能也會遭到破壞。考慮一個消息隊列與一個數據庫之間的兩階段提交事務。將一個狀態寫入數據庫中，並將一條相關消息發送到隊列。考慮到原子性和一致性，我們可能期望一旦從隊列收到消息，數據庫狀態將會與它保持一致。這是強一致性 模型，可在傳統事務中得到保證。

但是，由於從事務管理器向兩個資源管理器的提交是先後發送的，所以第一個資源中的更改可能先於第二個資源中的更改被看到。例如，一個作為第二個資源的數據庫可能花更長時間來執行實際提交，而且提交到第一個資源管理器的消息可能已被收到。處理會失敗，因為尚未看到數據庫狀態。

這種情況不會經常發生，但偶爾會發生，我也親自體驗過。（請參閱 “WebLogic 上的兩階段提交爭用條件” 和 “消息/數據庫爭用條件”。）因此，甚至在傳統分佈式事務中，也有一個不一致性窗口，事務僅在最終 達到一致。參見圖 4。

圖 4. 甚至在採用兩階段提交時也使用了不一致性窗口

此外，兩階段提交協議要求執行完兩個階段，才會釋放所有被阻塞的資源，比如數據庫記錄。為了確保事務的完整性 — 例如，如果事務管理器停止或重新啟動 — 事務管理器自身會編寫自己的事務日誌，這些日誌在它停止時也會保留下來。事務管理器啟動時會讀取此日誌，並完成未完成的事務。因此，要支持事務管理器中的兩階段提交事務，需要一個持久性事務日誌（通常被寫入文件系統中）。

兩階段提交事務需要對提交操作執行多次遠程調用。由於認識到 有分佈式事務開銷，通常會避免執行這些調用，甚至在傳統 JEE 設置中也是如此。但是，在需要確保跨資源管理器的一致性時，可以使用它們。

事務異步（消息）協議

大力投資分佈式事務的一種具體情況是，在將面向消息的中間件與一個數據庫事務耦合時。確保僅在數據庫事務成功時從隊列中獲取消息，這是一項非常好的功能，可減少應用程序中的錯誤處理工作。確保僅在數據庫事務可成功提交時發送消息，這也有助於在數據庫與消息隊列之間實現一致性。

在這種設置中，分佈式事務僅限於 “本地” 數據庫和消息服務器，不包含其他資源管理器。消息系統然後在發送和接收應用服務器之間保證事務屬性的一致性。此方法是最終一致的，因為接收資源僅在發送資源中的更改提交後才會更改。

圖 5. 消息基礎架構可提供事務最終一致性保證。

使用面向消息的中間件，可以創建一個能確保事務質量的事件驅動架構，而且可以將大量錯誤處理工作委託給事務管理器。

結束語

在第 1 部分中，我們探討了事務的基礎知識和分佈式事務。由於事務的屬性，它們可以簡化錯誤處理，不僅包括單一資源中的錯誤，還包括資源之間的錯誤。我展示了兩階段提交分佈式事務的工作原理，只有時間間隔非常小時，它們才具有 “最終一致性”。您還瞭解到，分佈式事務要求事務管理器中存在持久性事務日誌。

在許多雲部署中（例如 CloudFoundry），沒有持久性文件系統。因此，JEE 應用服務器運行時無法寫入它自己的事務管理器事務日誌，因為沒有兩階段提交事務。另外，在一些新的運行時環境中，比如 JavaScript 服務器，運行時甚至完全沒有事務管理器。（請參閱 “面向 Java 的微服務最佳實踐”。）

第 2 部分將介紹分佈式事務的替代方案，它們仍提供了一定的事務保證。使用這些替代方案，即使在雲應用程序中也可以保證事務質量，確保網店中的訂單得到正確處理。
