使用Spring Boot的事件和事務，容易被忽視的點，你有踩過嗎？

2020-10-29
| 科技

如果您要使用spring transaction和event publishers編寫代碼，則需要記住一些規則：

  • 事務綁定到一個線程
  • 默認情況下，當您跳出標記為@Transactional的方法時，將提交事務
  • 默認情況下，事務內部調用的所有方法都使用它
  • 默認情況下，事件偵聽器在調用程序線程中執行，這意味著它們使用與調用者相同的事務
  • 為了確保在提交調用者的事務之後調用您的方法，您需要使用@TransactionalEventListener
  • 如果您使用@TransactionalEventListener 然後 記住您需要啟動一個新事務來提交監聽器裡面發生的任何寫操作。

現在後端應用程序中存在一種趨勢，即將服務調用流更改為基於事件的調用。我認為這是由於領域驅動設計DDD概念的影響越來越大。挑選這種方法有時是合理的，有時它只是“炒作”。我想集中討論在數據庫事務中可能遇到的基於spring的應用程序中引入事件的後果。

我們來看一下你可以在這裡找到的一個簡單的spring boot應用程序。它包含一個註冊用戶的其餘API端點。註冊包含兩個操作：

  • 使用UserRepository接口將用戶保存到數據庫中
  • 在應用程序中發佈事件UserCreated

此操作發生在UserService中的一個事務方法中，該方法從UserResource（spring controller）調用。到目前為止，沒什麼特別的。

@Transactional@Servicepublic class UserService { public User createUser(User user) { User savedUser = userRepository.save(user); eventPublisher.publishEvent(new UserCreated(savedUser.id)); return savedUser; }}

讓我們擴展我們的應用程序來處理UserCreated事件。

我寫了一個監聽器（UserQueueNotifier），其職責非常簡單：

  • 訂閱活動
  • 根據事件中的數據（userId）從數據庫加載用戶
  • 將電子郵件和生成的註冊令牌放入jms隊列，以模擬與負責發送註冊確認電子郵件的外部系統的通信

注意點：有些人一眼就認識到我的事件中只有部分數據（userId）違反DDD規則。在純DDD世界中，我們應該將用戶包含在UserCreated事件中。我故意這樣做有兩個原因 - 一個是它簡化了示例;第二個，也許更重要的是，我在一個不遵循所有DDD規則的實際項目中遇到了這樣的解決方案。

@Componentpublic class UserQueueNotifier { @EventListener(UserCreated.class) public void onUserCreate(UserCreated userCreated) { User user = userRepository.find(userCreated.id); user.generateToken(); userRepository.save(user); jmsTemplate.convertAndSend("emails", new EmailWithToken(user.email, user.token()).asJmsMessage()); }}

要訂閱spring事件，您只需要編寫一個void方法，該方法將事件作為參數並將其標記為@EventListener。為了證明一切都在一起工作，你可以從類UserResourceHbTest運行測試 （測試應該是不言自明的）。其中一個檢查是否在註冊後，隊列中存在正確的消息。

@Testpublic void registerPutsEmailAndTokenOnQueue() throws Exception { JSONObject createUserJson = new JSONObject() .put("login", "dev-user2") .put("email", "dev-user2@pragmatists.pl"); api.post("users", createUserJson.toString()); String message = (String) jmsTemplate.receiveAndConvert("emails"); List<String> emailAndToken = newArrayList(Splitter.on(";").split(message)); assertThat(emailAndToken.get(0)).isEqualTo("dev-user2@pragmatists.pl"); assertThat(emailAndToken.get(1)).isNotEmpty();}

Listener（UserQueueNotifier）附帶了UserRepository 接口的特殊實現，它在下面使用了spring數據框架。我調用了這個實現HbUserRepository。

public interface UserRepository { User find(UserId userId); User save(User user);}public class HbUserRepository implements UserRepository { private final JpaUserRepository jpaUserRepository; public HbUserRepository(JpaUserRepository jpaUserRepository) { this.jpaUserRepository = jpaUserRepository; } @Override public User find(UserId userId) { return jpaUserRepository.findById(userId).orElse(null); } @Override public User save(User user) { return jpaUserRepository.save(user); }}

讓我們假設由於某種原因，您決定切換UserRepository接口的實現並使用本機sql。我稱之為JdbcTemplateUserRepository - 它使用JdbcTemplate進行查詢。

public class JdbcTemplateUserRepository implements UserRepository {  @Override public User find(UserId userId) { return jdbcTemplate.queryForObject("SELECT * FROM USER WHERE ID = ?", new Object[]{userId.asString()}, new UserRowMapper()); } }

在這個“基礎設施”改變之後，我重新運行了測試（UserResourceJdbcTemplateTest）：

測試結果是紅色的！測試無法通過！

我希望你也會感到驚訝。一切都歸功於Spring的事務傳播。

事務邊界是一個線索

我們需要從線索的頂部開始解釋並查看事務開始的位置。UserService被標記為@Transactional，因此方法createUser中的所有內容（如果在同一個線程中運行）都在同一個事務中。通過存儲庫保存用戶後，我們發佈了一個事件。默認情況下，發佈事件是同步的，因此您可以想象將標記為@EventListener的所有方法體內聯到createUser方法中。在publishEvent方法下面，spring只是循環遍歷所有偵聽器的方法並簡單地調用它們。沒有魔法，沒有隱藏的異步調用。

如果事件偵聽器的代碼與它的調用者在同一個線程中運行，則默認情況下它會傳播相同的事務。這意味著當我們在UserQueueNotifier偵聽器中調用UserRepository上的find方法時，我們仍處於未提交的事務中。在跳出標記為@Transactional的方法後，提交將在離開監聽器之後才會發生。那麼，我們如何從我們的數據庫中讀取用戶呢？

這是棘手的部分 - 如果UserRepository使用spring數據提供的底層實現，那麼可以找到用戶，因為它存儲在hibernate會話中。我們根本不觸及數據庫，我們只是從會話中檢索用戶。

如果我們通過JdbcTemplate將實現更改為直接對數據庫本機sql進行查詢，那麼當然不會在數據庫中找到用戶，因為事務尚未提交！這就是我們進行紅色測試的原因。

進入事件偵聽器時提交事務

我們能用Spring機制處理這種常見情況嗎？答案當然是肯定的。

需要在方法標記上增加@TransactionalEventListener註釋。

默認情況下，它已將屬性階段設置為TransactionPhase。AFTER_COMMIT表示將在提交調用事務時執行偵聽器代碼。

@Componentpublic class UserQueueNotifier { @EventListener(UserCreated.class) @TransactionalEventListener(phase = TransactionPhase.AFTER_COMMIT) public void onUserCreate(UserCreated userCreated) { User user = userRepository.find(userCreated.id); user.generateToken(); userRepository.save(user); jmsTemplate.convertAndSend("emails", new EmailWithToken(user.email, user.token()).asJmsMessage()); }}

提交後是否傳播了事務？

問題是：當我們的偵聽器代碼在提交後執行或我們是否需要打開一個新代碼時，我們仍處於事務中嗎？如果我們在偵聽器中編寫一些寫操作會發生什麼？

關於Spring文檔事務同步,我們需要閱讀的文檔。文檔說“在事務提交後調用。可以在主要事務成功提交後立即執行進一步的操作。“但最重要的部分來自於此。“事務將已經提交，但事務資源可能仍然是活動的和可訪問的。因此，此時觸發的任何數據訪問代碼仍將“參與”原始事務，允許它執行一些清理（比如不再執行任何提交！），除非它明確聲明它需要在單獨的事務中運行”。

所以我們的情況是在一個交易中，但提交將不會完成！？這很奇怪，但文檔附帶了一個解決方案。我們需要聲明這部分代碼將在新事務中運行。它可以通過兩種方式實現：

  • 我們可以使用@Transactional（propagation = Propagation.REQUIRES_NEW）啟動新事務
  • 我們可以使用標準的@Transactional，但由於事務是在同一個線程中傳播的，我們需要運行一個新的。我們可以在方法上使用@Async註釋

在我的簡單示例中，我決定使用相同的線程並在那裡開始新的事務。選擇第二種方法需要重寫測試，必須能夠等待這個新線程完成。

要在偵聽器中提交事務，請在UserResourceTest中的registeredUserHasTokenGenerated方法中刪除@Ignore。除非在偵聽器方法上添加@Transactional（propagation = Propagation.REQUIRES_NEW），否則測試應該是失敗的，因為未提交寫入數據庫（userRepository.save）。
