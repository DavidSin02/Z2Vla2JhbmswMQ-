1 適配器模式概述

現在有這樣一個場景：你們系統調用了訂單中心的服務OrderService，但是訂單中心改造新增了OrderNewService，訂單中心要求你們都改用OrderNewService接口，現在由你主導接口切換工作。

切換接口尤其是核心接口風險還是比較大的，我們希望改動儘量少，減少對原有業務代碼侵入性。這種情況我建議使用適配器模式。我們來分析這個實例。

2 原始調用方式

2.1 老訂單服務

/** * 外部訂單服務 * * @author 今日頭條號「IT徐胖子」 * */public interface OrderService {    /**     * 創建訂單     *     * @param goodsId 商品ID     * @param price   商品價格     * @return 訂單編號     */    public String createOrder(String goodsId, Long price);}/** * 外部訂單服務實現 * * @author 今日頭條號「IT徐胖子」 * */public class OrderServiceImpl implements OrderService {    @Override    public String createOrder(String goodsId, Long price) {        String orderId = "orderId_111";        System.out.println("創建訂單成功[goodsId=" + goodsId + ",price=" + price + "]");        return orderId;    }}

2.2 調用老訂單服務

/** * 模擬本系統調用 * * @author 今日頭條號「IT徐胖子」 * */public class TestDemo {    public static void main(String[] args) {        OrderService service = new OrderServiceImpl();        service.createOrder("goodsId_123", 100 L);    }}

3 使用適配器模式

3.1 新訂單模型

/** * 新訂單模型 * * @author 今日頭條號「IT徐胖子」 * */public class OrderNewModel {    private String goodsId;    private Long price;    public String getGoodsId() {        return goodsId;    }    public void setGoodsId(String goodsId) {        this.goodsId = goodsId;    }    public Long getPrice() {        return price;    }    public void setPrice(Long price) {        this.price = price;    }}

3.2 新訂單服務

/** * 外部訂單新服務 * * @author 今日頭條號「IT徐胖子」 * */public interface OrderNewService {    /**     * 創建訂單     *     * @param orderModel 訂單模型     * @return 訂單編號     */    public String createOrder(OrderNewModel orderModel);}/** * 外部訂單新服務實現 * * @author 今日頭條號「IT徐胖子」 * */public class OrderNewServiceImpl implements OrderNewService {    @Override    public String createOrder(OrderNewModel orderModel) {        String orderId = "orderId_222";        System.out.println("新接口創建訂單成功[goodsId=" + orderModel.getGoodsId() + ",price=" + orderModel.getPrice() + "]");        return orderId;    }}

3.3 訂單服務適配器

/** * 訂單服務適配器 * * @author 今日頭條號「IT徐胖子」 * */public class OrderServiceAdaptor implements OrderService {    private OrderNewService orderNewService;    public OrderServiceAdaptor(OrderNewService orderNewService) {        this.orderNewService = orderNewService;    }    @Override    public String createOrder(String goodsId, Long price) {        OrderNewModel orderModel = new OrderNewModel();        orderModel.setGoodsId(goodsId);        orderModel.setPrice(price);        return orderNewService.createOrder(orderModel);    }}

3.4 替換為新服務

/** * 模擬本系統調用 * * @author 今日頭條號「IT徐胖子」 * */public class TestDemo {    public static void main(String[] args) {        OrderService service = new OrderServiceAdaptor(new OrderNewServiceImpl());        serviceNew.createOrder("goodsId_222", 222 L);    }}

4 思考總結

我們看到將舊服務替換為新服務，在業務代碼層面（TestDemo）並沒有做大量修改，而是利用適配器模式，將新服務組合進適配器，對業務侵入性很低。

適配器模式顧名思義就是用來做適配的，適配器就像一箇中轉站，可以把原來不符合要求的數據、服務、接口轉換為符合要求，從而可以一起工作。

工作中常用的場景是在新老服務切換時，適配器可以減少對業務代碼的侵入性，符合對擴展開放，對修改關閉設計原則。
