背景知識——互聯網分層模型

  1. 應用層（DNS，FTP，HTTP，RTSP，SSH，TLS/SSL， XMPP， DHCP...）
  2. 傳輸層（TCP，UDP...）
  3. 網絡層（IP，...）
  4. 鏈路層（ARP，...）
  5. 實體層

另外，也有將互聯網分為7層的說法，他們分別是：應用層、
表示層（提供數據格式或轉換服務。比如：加密與解迷，編碼與解碼，壓縮與解壓）、
會話層（為客戶端的應用程序提供了打開、關閉和管理會話的機制）、
傳輸層、網絡層、數據鏈路層、物理層。

TCP 是什麼？

TCP（Transmission Control Protocol 傳輸控制協議）是一種面向連接的、可靠的、基於字節流（byte stream）的傳輸層通信協議；
TCP 基於不可靠的 IP 協議，提供穩定、可靠的數據傳輸服務；
對於 TCP 需要知道的是，TCP 所處的第2層傳輸的數據叫 Segment，IP 所處的第3層傳輸的叫 Datagram， 第4層傳輸的數據則叫做 Frame；
數據從應用層往下傳輸，每經過一層都會在數據頭部被加上一個 header ，層層封裝；
反之，則需要層層解封，最後應用層才能取得數據。

TCP 特性

  1. 點對點協議
  2. 可靠
  3. 管道傳輸（同時送多個segments）
  4. sender & receiver 都有 buffer
  5. byte stream
  6. 雙向傳輸
  7. 面向連接（oriented connection）
  8. 重傳機制

TCP 三次握手（如何建立連接？）

  1. Client 傳送一個 SYN segment 給 Server（init sequence number）
  2. Server 收到 SYN，回一個 SYNACK segment 給 Client（alloc buffer; init sequence number）
  3. Client 收到 SYNACK，再回一個ACK，從此連接才算是建立完成（同時，可以帶上 data 做數據傳輸）

TCP 為什麼需要三次握手？

TCP 四次分手（如何關閉連接？）

  1. client 傳送一個 FIN segment 給 server，請求關閉連接
  2. server 回一個 ack 給 client，表示 client 可以關閉連接了
  3. server 再發一個 FIN segment 給 client，請求關閉連接
  4. client 回一個 ack 給 server，表示 server 可以關閉連接，server 收到 ack ,關閉連接。此時，client 等待2MSL後依然沒有收到回覆，則證明 server 端已正常關閉，那好，client 也可以關閉連接了。

tcp_close.png


TCP 為什麼需要四次分手？

簡單來說，因為 TCP 連接是雙向的，所以 sender 需要發送 FIN 徵求 server 的同意；隨後，server 收到 client 的請求，給 client 回覆 ack，告訴它，此時可以關閉連接；而 server 同時也需要通知 client ，告訴它自己不會再傳輸數據了，所以，server 給 client 發送 FIN 信號；最後，client 收到 server 的請求， 回覆 ack 表示同意。

TCP 如何保證數據傳輸是可靠的？

  1. 給 byte stream 編號
  2. receiver 校驗數據的正確性
  3. receiver 回覆 ack
  4. timer. 發生 timeout 時，重傳

TCP 重傳機制

為什麼會重傳？
1. 丟失 ack
Host A 給 Host B 傳輸 數據， 隨後 Host B 給 A 回覆 ack， 但由於網絡原因，這個 ack 在傳輸過程中 loss 了， 那麼經過一個 timeout 時間後，Host A 給 Host B 重新傳送這份數據。
2. timeout
Host A 給 Host B 傳輸數據， 隨後 Host B 給 A 回覆 ack ，但由於網絡延遲，這個 ack 在一個 timeout 時間內沒有被及時送給 Host A，經過一個 timeout 時間後，重送數據，並且在收到延時的 ack 後， 按實際收到的數據回覆 ack
3. 收到3次重複的 ack
由於 receiver 回覆給 sender 的 ack 序列號表示期望下次 sender 送過來的數據的序列號，所以當 sender 收到 3次重複的 ack（也就是 ack 序列號都一樣），這個時候就會重新發送丟包或者延遲的 packet

快速重傳：在發生 timeout 之前重傳數據。

TCP Nagle Algorithm & Delay Ack

以上兩種策略都是針對TCP 頻繁傳送小包的情況，

  1. Nagle Algorithm: 發送一個小包後，如果這個小包還沒有被 ack ，則後面等待發送的少量數據，會進入 buffer 緩衝區，
    這部分數據在等待時間到了後才會被統一發送出去。
  2. Delay Ack: receiver 收到的數據如果不是按順序送進來的，則會先不發 ack ，等一定順序的數據都到了後，統一發 ack， 儘量減少 ack 的次數。
    根據他們的處理機制可以知道，這兩種策略都是犧牲時效性來提升性能。遇到時效性高的應用場景，可以自行關閉。
