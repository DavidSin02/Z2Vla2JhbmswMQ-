SQL 事務機制-transaction

2020-11-14
| 科技

TCount基表

什麼是事務（transaction）？

事務是指不可分割的邏輯工作單元，完全執行，或者完全不執行。該邏輯工作單元可能由多條SQL處理語句組合而成。

事務的特點：

  1. 同生共死。
  2. 有可能對錶數據進行添加或刪除操作，但是對查詢無用。
  3. 實際上是一個處理機制，用以維護數據庫的完整性。

事務的4大屬性（ACID特性）：

1.原子性（Atomic）（Atomicity)：

事務必須是一個自動工作的單元，要麼全部執行，要麼全部不執行。原子性消除了系統處理操作子集的可能性。

2.一致性（Consistent）(Consistency)：

事務結束的時候，所有的內部數據都是正確的。

3.隔離性（Insulation）(Isolation)：

併發多個事務時，各個事務不干涉內部數據，處理的都是另外一個事務處理之前或之後的數據。由併發事務所作的修改必須與任何其它併發事務所作的修改隔離。

由於高度隔離會限制可並行執行的事務數，所以一些應用程序降低隔離級別以換取更大的吞吐量。

4.持久性（Duration）(Durability）：

事務提交之後，數據是永久性的，不可再回滾。該修改即使出現致命的系統故障也將一直保持。

事務的語法：

1.事務起點-begin transaction

begin tran[saction] [<transaction name>|<@transaction variable>][with mark[<'description'>]]

注：

表示一個事務單元的開始，也是回滾的起點。如果沒有提交，那麼回滾就從這裡開始。

2.事務提交- commit transaction

commit tran[saction][<transaction name>|<@transaction variable>]

注：

表示事務的終點，事務單元語句順利執行，提交表示這個事務正常進行。

3.事務回滾-rollback transaction

rollback tran[saction][<transaction name>|<save point name>|<@transaction variable>|<@savepoint variable>]

注：

表示事務回滾，不想或者不能提交，回滾從begin開始的所有語句，類似撤銷。

4.事務保存-save transaction

save tran[saction][<save point name>|<@savepoint variable>]

注：

事務保存即創建一個標記或書籤（bookMark）。在回滾中引用這個標記，即可回滾到代碼中的特定點上。類似電腦系統恢復時，設置的定點恢復位置。

事務的3大處理模型：

1.隱式事務

--***************模式1*******************set implicit_transactions oncommit transaction--****************模式2******************set implicit_transactions onrollback transaction

隱式事務是指每一條數據操作語句都自動地成為一個事務，事務的開始是隱式的，事務的結束有明確的標記。

2.顯式事務

顯式事務是指有顯式的開始和結束標記的事務，每個事務都有顯式的開始和結束標記。

--***********顯式模式1*****************begin transaction commit transaction--**********顯示模式2*******************begin transaction rollback transaction

3.自動事務

自動事務是系統自動默認的，開始和結束不用標記。即每條Sql語句都被看成一個事務進行處理。

顯式事務3大模式

1.事務開始-事務提交

--begin transaction --.... --commit transaction

2.事務開始-事務回滾

--begin transaction --rollback transaction

3.事務開始--事務標記--定點回滾

--begin transaction --save transaction trac--rollback transaction trac

併發控制簡述

1． 數據庫系統一個明顯的特點是多個用戶共享數據庫資源，尤其是多個用戶可以同時存取相同數據。

串行控制：如果事務是順序執行的，即一個事務完成之後，再開始另一個事務

並行控制：如果DBMS可以同時接受多個事務，並且這些事務在時間上可以重疊執行。

2.事務是併發控制的基本單位，保證事務ACID的特性是事務處理的重要任務，而併發操作有可能會破壞其ACID特性。

DBMS併發控制機制的責任：

  1. 對併發操作進行正確調度，保證事務的隔離性更一般，確保數據庫的一致性。
  2. 如果沒有鎖定且多個用戶同時訪問一個數據庫，則當他們的事務同時使用相同的數據時可能會發生問題。

由於併發操作帶來的數據不一致性包括：

  1. 丟失數據修改
  2. 讀”髒”數據（髒讀）
  3. 不可重複讀
  4. 產生幽靈數據

注：

一個邏輯工作單元要成為事務，必須滿足所謂的ACID（原子性、一致性、隔離性和持久性）屬性。事務是數據庫運行中的邏輯工作單位，由DBMS中的事務管理子系統負責事務的處理。
