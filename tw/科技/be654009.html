GitHub CLI 入門指南

2020-11-14
| 科技


GitHub最近發佈了GitHub CLI，這是一個第一方命令行工具，可以通過命令行與GitHub對接。

通過GitHub CLI，開發者可以查看GitHub的問題(issues)和拉取請求(pull request)的狀態，搜索特定的問題或PR，創建/分叉一個Repo，或者直接在命令行創建新的問題和拉取請求。

在這篇文章中，我們將介紹一些常見的日常開發者工作流程。我們將重點討論如何使用 GitHub CLI 管理問題和拉取請求。

什麼是GitHub CLI？

GitHub CLI官方網站：https://cli.github.com/

所以，我們先來簡單介紹一下GitHub CLI。如果你已經很熟悉了，那就提前跳到下一節。

  • Issues

  • Repositories

如何安裝GitHub CLI？

要將GitHub CLI安裝在開發機器上，請查看GitHub上的安裝指南。這裡以Mac為例，有兩種安裝方式

  • brew安裝：brew install github/gh/gh
  • 直接下載安裝包

GitHub CLI可在MacOS，Windows和各種Linux發行版中使用，因此在計算機上啟動並運行該工具應該沒有問題。

使用GitHub CLI管理GitHub Issues

如果你在開發流程中經常使用GitHub Issues，那麼GitHub CLI可能會為你提供一些有用的命令。讓我們看一下典型工作日的樣子，並重點介紹一些使用GitHub CLI處理GitHub上的Issues的示例。

/ 列出GitHub CLI的Issues /

假設我們今天只是坐在電腦前，想把我們項目的GitHub Issues列出來。

gh issue list

如果我們想列出所有問題，可以使用“state”標誌

gh issue list --state "all"gh issue list -s "all"gh issue list --state "all"gh issue list -s "all"

現在，也許我們已經意識到有太多問題需要解決，所以我們決定只列出你當前分配的問題。

gh issue list --assignee "n8ebel"

或者，你可以使用簡寫形式

gh issue list -a "n8ebel"

/ 使用GitHub CLI檢查Issue狀態 /

接下來，我們要檢查昨天創建的幾個Issue的狀態。也許我們不記得它們的確切數字，但是由於創建了它們，我們可以使用status命令在終端上列出它們：

gh issue status

這將向我們提供分配給我們的Issue列表，提到我們或由我們打開的Issue。

在檢查了這些問題之後，我們仍然找不到所需的問題，因此我們可能要檢查它是否已關閉。

gh issue list --state "closed"gh issue list -s "closed"

這下好了！ 我們的問題已經結束了，所以現在我們可以開始尋找下一個任務了。 我們可以開始尋找新的任務，把不同的標籤篩選出來，列出不同的標籤來尋找新的任務。

要列出所有未解決的bug，我們可以按GitHub 倉庫中定義的“bug”標籤進行過濾

gh issue list --label "bug"gh issue list -l "bug"

如果你不確定有哪些標籤，你可以先把所有的未完成的問題都列舉出來，再進行檢查。

gh issue list

現在，我們可以看到還有一個我們可以搜索的 enhancement 標籤

gh issue list -l "enhancement"

/ 查看GitHub CLI的 Issues /

找到要解決的Issue後，我們可能希望將該Issue分配給自己。目前，我們無法直接從命令行執行此操作，但可以使用“view”命令從命令行快速打開問題。

gh issue view "15"

這將在瀏覽器中打開Issue，然後你可以將其分配給自己。

在將任何問題分配給自己之後，你可以從命令行中列出分配給你的打開的Bug，從而重複檢查你的任務。

gh issue list -a "n8bel" -l "bug"

/ 使用GitHub CLI創建Issue /

現在，我們只需探索最後一個命令。

在我們處理分配的Issue時，想象一下我們發現了另一個需要報告的bug。

我們可以使用 gh issue create 命令直接從命令行創建一個新的GitHub Issue。

gh issue create

使用此命令將啟動一個交互式終端工作流，以選擇任何可用的GitHub Issue模板，然後填寫標題和Issue描述。最後，你可以選擇提交Issue、在瀏覽器中打開Issue或取消Issue。

如果你想簡化一些事情，可以使用其他標誌通過命令指定問題

gh issue create -t "Sample Issue Title" -b "Sample issue description"

如果你仍然喜歡從網絡上創建Issue，你可以使用以下命令打開瀏覽器窗口，直接跳轉到你的repo中的Issue創建工作流程中去

gh issue create --web

/ 有用的GitHub CLI別名 /

通過這些不同的命令和標誌變化，我們有相當多的選項來創建、列出和查看我們的GitHub Issue。

通過為它們創建一些命令行別名，我們可以使這些命令更易於使用。

例如，列出所有bug時，不要使用此完整命令

gh issue list --label "bug"

我們可以這樣創建一個別名

alias listbugs='gh issue list --label "bug"'

這樣，當我們要列出所有Bug時，我們只需鍵入：

listbugs

另一個示例是使用以下別名列出分配給你的所有Bug。

alias listmybugs='gh issue list -a "<your username>" -l "bug"'

使用此別名後，我們可以像這樣搜索分配給你的所有Bug：

listmybugs

使用GitHub CLI管理 Pull Request

現在，讓我們來探討一下使用GitHub CLI來管理Pull Request。再次，讓我們通過一個想象中的工作日，來看看這個工具能在哪些方面幫助我們。

/ 列出Pull Request /

再次，我們坐在電腦旁度過一天，我們想要列出項目的未完成Pull Request。

gh pr list

如果我們要列出所有打開和關閉的Pull Request，則可以使用“state”標誌

gh pr list --state "all"gh pr list -s "all"

現在，也許我們已經意識到要解決的PR太多，所以我們決定只想列出你當前分配的問題。

gh pr list --assignee "n8ebel"

或者，你可以使用簡寫形式

gh pr list -a "n8ebel"

/ 檢查Pull Request的狀態 /

接下來，我們要檢查昨天創建的幾個PR的狀態。也許我們不記得它們的確切數字，但是由於創建了它們，我們可以使用status命令在終端上列出它們

gh pr status

這將為我們提供分配給我們、提及我們或由我們打開的PR的列表。

在檢查完這些PR之後，我們仍然找不到所需的Pull Request，因此我們可能要檢查它是否已關閉。

gh pr list --state "closed"gh pr list -s "closed"

那很棒！我們的PR已關閉，因此現在我們可以找到下一個任務。我們可以通過列出由不同標籤過濾的Pull Request來開始尋找新任務。

要列出所有打開的漏洞修復PR，我們可以按GitHub 倉庫中定義的“bug”標籤進行過濾

gh pr list --label "bug"gh pr list -l "bug"

如果你不確定哪些標籤可用於未解決的問題，可以先通過再次列出所有未解決的PR進行檢查

gh pr list

現在，我們看到還有一個可以搜索的 enhancement 標籤

gh pr list -l "enhancement"

/ 查看Pull Request /

找到要審查的PR後，我們可能希望將該PR分配給我們自己。目前，我們無法直接從命令行執行此操作，但是我們可以使用view命令從命令行快速打開PR。

gh pr view "14"

這將在瀏覽器中打開Pull Request，然後你可以在其中將其分配給自己，進行查看等。

你可以通過列出分配給你的PR，從命令行中重複檢查你分配的Pull Request。

gh pr list -a "n8bel" -l "bug"

/ 創建Pull Request /

在處理分配的問題時，有時會希望根據本地更改創建一個新的Pull Request。我們可以使用 gh pr create 命令直接從命令行創建一個新的Pull Request。

gh pr create

使用此命令將啟動一個交互式終端工作流，填寫標題和問題描述。最後，你最終可以選擇提交PR，在瀏覽器中打開PR或取消它。

如果你想簡化一下內容，可以使用帶有其他標誌的命令來具體說明PR信息

gh pr create -t "Sample Issue Title" -b "Sample issue description"

如果你仍然希望從網絡上創建PR，則可以使用以下命令打開瀏覽器窗口，然後直接跳轉進入PR創建工作流程

gh pr create --web

在某個時候，我們將要在本地測試PR。這一般需要查找PR，檢查它所在的分支，然後在本地檢查它。

使用GitHub CLI，我們可以使用pr checkout命令簡化此過程

gh pr checkout "14"

這將檢出與PR 14相關的分支，因此你準備開始測試。因此，有了這些各種命令和標誌的變化，我們有很多創建，列出和查看GitHub pull請求的選項。

通過為它們創建一些命令行別名，我們可以使這些命令更易於使用。

例如，列出所有PR時，不要使用此完整命令

gh pr list --label "bug"

我們可以這樣創建一個別名

alias listprs='gh pr list --label "bug"'

這樣，當我們要列出所有與bug相關的PR時，我們只需鍵入

listprs

另一個示例是使用以下別名列出分配給你的所有PR

alias listmyprs='gh pr list -a "<your username>"'

使用此別名後，我們可以像這樣搜索分配給你的所有PR

listmyprs

提示：

當你運行第一個GitHub CLI命令時，工具會提示你在瀏覽器中使用GitHub進行身份驗證。一旦你輸入了你的憑證，一個新的訪問令牌就會被添加到你的倉庫中，並用於從GitHub CLI驗證。
