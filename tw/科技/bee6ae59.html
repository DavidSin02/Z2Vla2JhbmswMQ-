前言

在JDK1.6中，HashMap採用數組+鏈表實現，即使用鏈表處理衝突，同一hash值的節點都存儲在同一鏈表。但是當某一個鏈表的元素較多時，通過key單鏈查找效率很低。

而在JDK1.8中，HashMap採用數組+鏈表+紅黑樹來實現，當鏈表長度超過閾值（TREEIFY_THRESHOLD=8）時，將鏈表轉換為紅黑樹，這樣節省查找時間。

紅黑樹鏈表閾值

本文采用JDK1.8說明。

概念

散列#Hash

散列就是把任意長度的輸入，通過散列算法變成固定長度的輸出.

Hash衝突

由於散列是不定長到定長，是一種壓縮映射，輸出值域小於輸入值域，造成不同的輸入值可能會有相同的輸出值，這就是Hash衝突。

原理分析

hash存儲位置

當程序調用put方法時，首先計算key的hash值，然後根據位運算算出是新Node節點還是紅黑樹（TreeNode）節點。

if ((p = tab[i = (n - 1) & hash]) == null)tab[i] = newNode(hash, key, value, null);

如果是紅黑樹，還需要判斷key的equals方法，配合key的hash值，雙管齊下，才能確定該節點的紅黑樹的位置

if (p.hash == hash && ((k = p.key) == key || (key != null && key.equals(k)))) e = p;else if (p instanceof TreeNode) e = ((TreeNode<K,V>)p).putTreeVal(this, tab, hash, key, value);

為什麼紅黑樹要通過key的hash和equals方法來確定節點存儲位置呢？

這是由於hash算法本身特性造成的。從Hash衝突的概念我們知道，通過Key算hash的存儲位置，可能存在存儲位置相同！

那如何解決key的hash衝突，保證唯一性呢？

Hash衝突

我們先了解下Java對equals方法和hashCode方法的兩個約定：

當obj1.equals(obj2)為true時，obj1.hashCode() == obj2.hashCode()必須為true當obj1.hashCode() == obj2.hashCode()為false時，obj1.equals(obj2)必須為false

也就是說，如果hashCode不同，equals一定為false，如果hashCode相同，equals不一定為true。

理論上，hashCode可能存在衝突的情況，當衝突發生時，計算出的桶索引（bucketIndex）也是相同的，這時會取到桶索引位置已存儲的元素，最終通過equals來比較，equals方法就是哈希碼碰撞時才會執行的方法。

基於以上兩個約定，HashMap通過hashCode和equals判斷Key是否已存在，如果hashCode和equals相等，則使用新V值替換舊V值；如果hashCode相等當equals為false，則在JDK1.8中會使用紅黑樹法來解決衝突並返回舊V值，如果不存在 ，則存放新的鍵值對<K, V>到桶索引位置。

一般情況下hash衝突在多線程高併發情況下比較明顯。

我們知道JDK1.6是用equals比較來保證key的唯一性，但是某個Node節點（單鏈表結構）隨著內部元素的增多，put和get的效率將越來越低，這裡的時間複雜度是O(n)，假如有1000個元素，put時需要比較1000次，效率很低。

通常情況下HashMap很少會用到equals方法，因為HashMap通過一個哈希表管理所有元素，當我們調用put存值時，HashMap首先會調用key的hashCode方法，獲取哈希碼，通過哈希碼快速找到某個存放位置。

總結

在多線程情況下，一般不會用HashMap，因為線程不安全，容易才產生髒數據。可以考慮用繼承+鎖或者ConcurrentHashMap代替。
