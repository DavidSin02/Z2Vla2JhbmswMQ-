AOP面向切面編程

先上概念

不管你是否能看懂，瞭解一下概念還是有必要的。

AOP是OOP的延續，是Aspect Oriented Programming的縮寫，直譯意思：面向方面編程、面向切面編程。

AOP通過預編譯方式和運行期動態代理實現在不修改源代碼的情況下給程序動態統一添加功能的一種技術。可以在代碼階段實現分離關注，實現鬆耦合，然後在運行時將相關動作特徵植入運行，從而改變程序的運行順序，達到攔截或者增加功能的目的。

AOP的好處

解耦：AOP將程序中的共性問題進行了剝離，毫無疑問地降低了各個業務模塊和共性問題之間的耦合。

重用性：共性問題散落於業務邏輯的各處，十分難維護，使用AOP進行提取後，能夠將相似功能的共性問題收斂，減少重複代碼，提高了代碼的重用性。

拓展性：對於一個程序而言，迭代的重心一定在於業務和功能上。AOP使得每當發生變更時，可以只關注業務邏輯相關的代碼，而減少共性問題上帶來的變化，大大降低了程序未來拓展的成本。

生活中的AOP

第一次接觸的人是會有一點蒙圈，要理解AOP（面向切面編程），首先我們要明白什麼叫做切面，下面舉個生活中的小例子。

廚師做菜的步驟是：拿菜、洗菜、切菜、做菜，這個是主幹流程。

有一天公司老闆說需要給所有廚師統計工作量，一天做了幾個菜都記錄好，有兩種方式：

方法1：廚師做菜完畢後拿起一張紙，每做好一個菜就紙上做一個標記。

方法2：廚師做菜完畢通知另外一個負責登記的同事，讓同事來做登記，這個就是Aop。

那麼Aop的好處是什麼呢？

可以讓廚師專注於做菜這個主幹流程，而不需要去做其他不相干的事情。

程序中的Aop

我們最常用場景是：日誌功能，下面定義一個日誌類。

 public class Log { public void WriteLog() { //寫日誌 } }

需求是：用戶登錄、註冊等所有操作都需要記錄日誌，傳統寫法如下：

 /// <summary> /// 註冊類 /// </summary> public class Reg { public void MemberReg() { //註冊的邏輯省略.... Log log = new Log(); log.WriteLog(); } } /// <summary> /// 登錄類 /// </summary> public class Login { public void MemberLogin() { //登錄的邏輯省略.... Log log = new Log(); log.WriteLog(); } } //其他100個類省略....

這樣寫看起來是沒有問題，當某一天，客戶說日誌需要記錄用戶操作時候用的瀏覽器型號，負責日誌的同事把日誌類改為：

 public class Log { public void WriteLog(string borwerType) { //寫日誌 } }

那之前的代碼都要改為：

 /// <summary> /// 註冊類 /// </summary> public class Reg { public void MemberReg() { //註冊的邏輯省略.... Log log = new Log(); log.WriteLog("IE瀏覽器"); } } /// <summary> /// 登錄類 /// </summary> public class Login { public void MemberLogin() { //登錄的邏輯省略.... Log log = new Log(); log.WriteLog("谷歌瀏覽器"); } } //其他100個類省略....

如果幾個還好說，如果涉及的項目比較大，就是幾百個文件都要修改，我估計你要崩潰。

那麼如何使用Aop的方式改造呢？

使用基類繼承來改造

先定義個基類，把寫日誌的方法定義在基類中

 public class BaseClass { public void WriteLog() { string borwerType = "獲取瀏覽器的省略...."; Log log = new Log(); log.WriteLog(borwerType); } ...其他方法 }

其他需要日誌功能的類繼承這個基類，如下：

 /// <summary> /// 註冊類 /// </summary> public class Reg: BaseClass { public void MemberReg() { //註冊的邏輯省略.... base.WriteLog(); } } /// <summary> /// 登錄類 /// </summary> public class Login : BaseClass { public void MemberLogin() { //登錄的邏輯省略.... base.WriteLog(); } }

以後如果有日誌需要變化，只需要更改BaseClass中的方法即可，我個人不推薦這種做法，編程有一個重要原則是：多用組合，少用繼承。

c#和Java都不支持多基類繼承，這就會導致如果後期有很多類似日誌這種輔助性功能多的時候，基類就會無限膨脹，而且導致繼承了很多不需要的方法。

使用特性來改造

我強烈推薦這種用法，使用過asp.net mvc的小夥伴都知道mvc有一個攔截器，就是通過特性的原理來實現，下面把日誌類改造為特性，如下：

 [AttributeUsage(AttributeTargets.Class | AttributeTargets.Method, Inherited = true)] public class LogAttribute : ActionFilterAttribute { public override void OnActionExecuted(ActionExecutedContext filterContext) { //寫日誌 } }

在需要日誌功能的類或方法上加上特性標記，改造後的代碼如下：

 /// <summary> /// 註冊類 /// </summary> public class Reg { [Log] public void MemberReg() { //註冊的邏輯省略.... } } [Log] public class Login { public void MemberLogin() { //登錄的邏輯省略.... } }

特性可以用在類上，也可以用到方法上。

這樣改造後是不是覺得代碼瞬間簡潔很多，以後日誌需求更改，只需要更改日誌類就可以了。
