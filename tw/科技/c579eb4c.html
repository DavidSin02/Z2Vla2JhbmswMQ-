HashMap容量為什麼要是2的冪次方？看完你就明白了！

2020-10-29
| 科技

相信看過我前面幾篇文章的小夥伴們一定發現了，那就是HashMap的容量都為2的冪次方，每次擴容也是擴大一倍，那麼為啥要保持容量為2的冪次方呢？又是如何保證的呢？讓我們來一起揭祕吧！

首先，我們來看下第一個問題，為什麼容量總是為2的n次冪？這到底是作者深思熟慮之後的考量，還是作者隨手的設定呢？

看過源碼的小夥伴們一定知道HashMap計算節點存儲位置下標是通過 (n-1)&hash 來計算的，這裡的n就是HashMap的容量。因為n始終為2的冪次方，那麼n-1始終能保證後幾位都是1(例如：n=16,那麼n-1:0000 ... 1111)。那麼當 (n-1)&hash 做運算時，就會保留hash值後幾位中的1。廢話不多說，讓我們動手來實踐下吧。

我們先假設 n = 16, hash值從0開始遞增，(n-1)&hash結果分佈很均衡。

那我們再來看看 假設 n = 15, (n-1)&hash的情況又是怎樣呢？

不對比不知道，一對比嚇一跳，如果容量不為2的冪次方的時候，極大的提升了hash碰撞的可能，並且浪費了奇數位的存儲位置。

那麼我們通過上述的對比就可以得知

HashMap 容量始終設定為2的冪次方是為了 (n-1)&hash 索引位置均勻分佈，減少hash碰撞。

那麼我想大家接下來就會很疑惑：

如果我通過構造函數傳入一個初始容量不為2的冪次方的數據，那豈不是就會出現上面所看到容量不為2的冪次方所出現的問題？

答案是不會的

首先，我們來看下HashMap的有參構造器。

public HashMap(int initialCapacity, float loadFactor) {		if (initialCapacity < 0)				throw new IllegalArgumentException("Illegal initial capacity: " +					initialCapacity);		if (initialCapacity > MAXIMUM_CAPACITY)				initialCapacity = MAXIMUM_CAPACITY;		if (loadFactor <= 0 || Float.isNaN(loadFactor))				throw new IllegalArgumentException("Illegal load factor: " +					loadFactor);		this.loadFactor = loadFactor;		this.threshold = tableSizeFor(initialCapacity);}

我們可以看到構造器裡面調用了一個tableSizeFor(initialCapacity) 方法，那這個方法裡面有隱藏著什麼呢？我們繼續往下看：

/*** Returns a power of two size for the given target capacity.* 對於給定的目標容量，返回兩倍大小的冪。*/static final int tableSizeFor(int cap) {	int n = cap - 1;	n |= n >>> 1;	n |= n >>> 2;	n |= n >>> 4;	n |= n >>> 8;	n |= n >>> 16;	return (n < 0) ? 1 : (n >= MAXIMUM_CAPACITY) ? MAXIMUM_CAPACITY : n + 1;}

看到這段代碼，是不是有點雲裡霧裡呢？

我們來試著分析下，我們先假設我們輸入的初始容量 initialCapacity = 10

int n = cap -1; //運算完之後 n = 9;n |= n >>> 1; //將9進行無符號右移一位並9做或運算並將結果賦予n// 0000...1001 | 0000 ... 0100 = 0000 ... 1101 結果 11n |= n >>> 2; //再次將 n 和 無符號右移兩位的n做或運算// 0000 ... 1101 | 0000 ... 0011 = 0000 ... 1111 結果 15n |= n >>> 4;// 0000 ... 1111 | 0000 ... 0000 = 0000 ... 1111 結果 15n |= n >>> 8;// 0000 ... 1111 | 0000 ... 0000 = 0000 ... 1111 結果 15n |= n >>> 16;// 0000 ... 1111 | 0000 ... 0000 = 0000 ... 1111 結果 15n = n+1; //結果 16

由此我們可以看到，當在實例化HashMap實例時，如果給定了initialCapacity，由於HashMap的capacity要保證容量為2的冪次方，因此這個方法就是用來找到大於等於initialCapacity的最小的2的冪。

那麼我想我們還會有一個疑問，那就是為要執行 int n = cap -1 ？來，舉個例子，相信我們很快就能明白，我們繼續假設我們輸入的初始容量 initialCapacity = 8 ，我們不去做 int n = cap -1 操作，那麼到底會發生什麼呢？

n |= n >>> 1; //將8進行無符號右移一位並8做或運算並將結果賦予n// 0000...1000 | 0000 ... 0100 = 0000 ... 1100 結果 10n |= n >>> 2; //再次將 n 和 無符號右移兩位的n做或運算// 0000 ... 1100 | 0000 ... 0011 = 0000 ... 1111 結果 15n |= n >>> 4;// 0000 ... 1111 | 0000 ... 0000 = 0000 ... 1111 結果 15n |= n >>> 8;// 0000 ... 1111 | 0000 ... 0000 = 0000 ... 1111 結果 15n |= n >>> 16;// 0000 ... 1111 | 0000 ... 0000 = 0000 ... 1111 結果 15n = n+1; //結果 16

發現問題沒有？我們給定的初始容量 initialCapacity = 8 ，就是 2的冪次方，但是上面最終返回給我們的是16 ？我們繼續來看看，我們執行 int n = cap -1 會有什麼樣的結果呢？

int n = cap - 1; //n = 7n |= n >>> 1; //將7進行無符號右移一位並7做或運算並將結果賦予n// 0000...0111 | 0000 ... 0011 = 0000 ... 0111 結果 7n |= n >>> 2; //再次將 n 和 無符號右移兩位的n做或運算// 0000 ... 0111 | 0000 ... 0001 = 0000 ... 0111 結果 7n |= n >>> 4;// 0000 ... 0111 | 0000 ... 0000 = 0000 ... 0111 結果 7n |= n >>> 8;// 0000 ... 1111 | 0000 ... 0000 = 0000 ... 1111 結果 7n |= n >>> 16;// 0000 ... 1111 | 0000 ... 0000 = 0000 ... 1111 結果 7n = n+1; //結果 8

如果不減去1 得到的結果為16,不符合我們的預想，輸入8的時候返回不小於輸入結果的最小2的冪次方的結果應該是8，所以這裡進行 int n = cap -1 就是為了避免這種情況。

最後不得不說，寫這段源碼的作者的這個算法實在是太犇了。
