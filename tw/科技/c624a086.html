計算機網絡自學筆記：IP數據報格式與編址

2020-11-14
| 科技

本篇文章最重要的知識點是子網的概念，同一個子網內的主機可以直接通信，不同子網的主機通信需要通過路由器轉發。因此即使是接在同一個交換機上的兩臺主機，如果屬於不同子網(IP 地址表示子網部分不同)，它們也將無法直接通信。因為發送主機會根據子網地址判斷接收主機和它不在同一個子網，會默認交付到它自己的網關(默認設置，網絡就這麼設計的)，而不是將數據通過交換機直接轉發給目標主機。

目前有兩個版本的 IP 在使用。IPv4，IPv6。

因特網的網絡層有三個主要的部分。第一個組件是 IP 部分。第二個主要組件是選路部分，它決定數據報從源到目的地所流經的路徑。選路協議計算在網絡中用於轉發分組的轉發表。最後一個部分是 ICMP，報告數據報中的差錯和對某些網絡層信息請求進行響應。

一： 數據報格式

1.1數據報

網絡層分組被稱為數據報。IPv4 數據報中的關鍵字段如下:

·版本號：這 4 比特規定了數據報的 IP 協議版本。通過查看版本號，路由器可確定如何解釋數據報的剩餘部分。

·首部長度：因為一個 IPv4 數據報可包含一些可選項，故需要用這 4 比特來確定 IP 數據報中的數據部分實際從哪裡開始。

·服務類型：服務類型比特用來區分不同類型的數據報。例如，將實時數據報與非實時數據報之間的區分。

·數據報長度：IP 數據報的總長度(首部加上數據)，以字節計。因為該字段長為 16 比 特，所以 IP 數據報的理論最大長度為 65 535 字節。然而，數據報長度一般都是 1500 字節(以太網幀可承載不超過 1500 字節的數據)。

·標識 ID、標誌 Fragflag、片偏移 Offset：該三個字段與 IP 分片有關。

·壽命：每當數據報經過一臺路由器時，壽命(Time-To-Live, TTL)字段的值減 1。若 TTL字段減為 0，則該數據報必須丟棄。

·協議：該字段在一個 IP 數據報到達其最終目的地時，字段值指明瞭 IP 數據報的數據部分應交給哪個傳輸層協議。例如，值為 6 表明數據部分要交給 TCP，而值為 17 表明數據要交給 UDP。 注意，數據報中的協議號所起的作用類似於傳輸層報文段中端口號字段所起的作用。

·首部校驗和：首部校驗和用於幫助路由器檢測收到的 IP 數據報中的比特錯誤。

注意，在 IP 層只對 IP 首部進行了檢驗，而 TCP 校驗和是對整個報文段進行的。

·源和目的 IP 地址。

·選項：選項字段允許 IP 首部被擴展。

·數據(有效載荷)。

大多數 IP 數據報不包含可選項，所以一般的 IP 數據報都有 20 字節的首部。

一個無選項字段、長度為 1500 的 IP 數據報若承載一個 TCP 報文段，則每個(無分片的)數據報共承載 40 字節首部(20 字節 IP 首部加上 20 字節 TCP 首部)以及長度為 1460 的應用層消息。

1.2：數據報分片

井不是所有的鏈路層協議都能承載相同長度的網絡層分組。有的協議能承載大分組，而有的協議只能承載小分組。例如，以太網幀可承載不超過 1500 字節的數據，而某些廣域網鏈路的幀可承載不超過 576 字節的數據。

一個鏈路層幀能承載的最大數據量叫做最大傳輸單元 MTU。

假設有一個互連幾條鏈路的路由器，且每條鏈路運行具有不同 MTU 的鏈路層協議。假定你從某條鏈路收到一個 IP 數據報，通過檢查轉發表決定出鏈路，但該出鏈路的 MTU 比該 IP數據報的長度小。

如何將這個過大的 IP 數據報封裝成鏈路層幀呢?

解決該問題的方法是將 IP 數據報中的數據分片成兩個或更多個較小的數據報，用單獨的鏈路層幀封裝這些較小的 IP 數據報，然後向輸出鏈路上發送這些幀。這些較小的數據報叫做分片。

在其到達目的地傳輸層以前需要被重新組裝。數據報的重新組裝工作在端系統中，而不是在途徑的路由器中。

因此，一個具有長度較大的數據報流經具有較小 MTU 的鏈路時需要分片，分片的重組中在目的端系統中進行。

例子，一個 4000 字節的數據報(20 字節 IP 首部加上 3980 字節 IP 有效載荷)到達一臺路由器，被轉發到一條 MTU 為 1500 字節的鏈路上。這就意味著初始數據報中的 3980 字節數據必須被分成 3 個獨立的片(每個片也是一個 IP 數據報)。假定初始數據報附加上的標識號為 777。三個片的特點如表所示。

除了最後一片外所有初始有效載荷數據的數量應當是 8 字節的倍數，並且偏移值應當被規定以 8 字節塊為單位(片偏移的表示範圍比數據報長度字段小了 8 倍，所以以 8 字節為單位)。

分片是有開銷的。首先，它使路由器和端系統更為複雜，即必須將數據報分成適當大小的片並且需要重新組裝。其次，分片可能引發致命的 DoS 攻擊，例如 Jolt 攻擊就是一個典型的例子，其中攻擊者向目標主機發送小片的流，這些片的偏移標誌都不是 0 ，當目標主機試圖根據這些不正確分組重建數據報時，就可能會崩潰。

二: IP 編址

主機與物理鏈路之間的邊界叫做接口。路由器與它的任意一條鏈路之間的邊界也叫做接口。

因為路由器的任務是在不同的網絡間轉發數據報，所以一臺路由器有多個接口，每個接口有一條鏈路。

IP 要求每臺主機和路由器接口都擁有自己的 IP 地址。因此，一個 IP 地址在技術上是與一個接口相關聯的，而不是與擁有該接口的主機或路由器相關聯的。

每個 IP 地址長度為 32 比特(等於 4 字節)，因此總共有 2^32個可能的 IP 地址。這些地址一般按點分十進制記法的方式書寫，即地址中的每個字節用十進制形式書寫，各字節間以句號(點)隔開。

在因特網中，每臺主機和路由器上的每個接口都必須有一個全球唯一的 IP 地址。

一個接口的 IP 地址由其連接的子網來決定。

下圖給出了一個 IP 編址與接口的例子。在該圖中，一臺路由器(有 3 個接口)用於互連7臺主機。

圖中左上側部分的 3 臺主機以及它們連接的路由器接口都有一個形如 223.1.1.x/24 的 IP 地址。這就是說，在它們的 IP 地址中，最左側的 24 比特是相同的。用 IP 的術語來說，互連這 3 臺主機的接口與路由器的一個接口的網絡形成一個子網。

IP 編址為這個子網分配一個地址，其中的/24 記法有時稱為子網掩碼，它表明 32 比特中的最左側 24 位特定義了子網地址。因此子網 223.1.1.1/24 是由 3 臺主機接口和 1 個路由器接口(223.1.1.4)組成的。

一個子網的 IP 定義並不侷限干連接多臺主機到一臺路由器接口的以太網段。路由器端口之間的點對點鏈路也可以構成子網。

對於一般由路由器和主機組成的互連繫統，可以使用下列方法定義系統中的子網：

物理上相連，包括通過交換機相連或者點對點鏈路相連；

子網地址相同，也就是 IP 地址中表示子網的地址相同。

因特網的地址分配策略被稱為無類別域間選路 CIDR。

對於子網尋址 a.b.c.d/x，32 比特的 IP 地址被劃分為兩部分，其中 x 指示了在地址的第一部分中的比特數。地址的高位 x 比特構成了 IP 地址的網絡部分，並且經常被稱為該地址的前綴 (或網絡前綴)。

一個單位通常被分配一塊連續的地址，即具有相同前綴的一段地址。當單位外部的路由器轉發一個數據報，且該數據報的目的地址在單位內部時，僅需要考慮該地址前面的 x 比特。 這大大減少了這些路由器中的轉發表的長度，因為形式為 a.b.c.d/x 單一項足以將數據報轉發到單位內的任何目的地。

一個地址的剩餘 32-x 比特可認為是用於區分單位(子網)內部主機的，這個子網所有設備具有相同的網絡前綴。

在採用 CIDR 之前，IP 地址的網絡部分被限制長度為 8、16 或 24 比特，因為具有 8、16 或 24 比特子網地址的網絡分別被稱為 A、B 和 C 類網絡，所以這種編址方案稱為分類編址。

一個 C 類(/24 )子網僅能容納多達 256-2=254 臺主機(其中的主機部分全 0 地址用於表示子網，全 1 地址用於廣播)，這對於許多單位來說太小了。然而，一個 B 類(/16)子網可支持多達 65 534 臺主機，又太大了。在分類編址方法下，導致了 B 類地址空間的迅速損耗以及所分配地址空間的低利用率。

另一種類型的地址，即廣播地址 255.255.255.255。當一臺主機發出一個目的地址為 255， 255 .255 .255 的數據報時，該報文會被交付給同一個子網中的所有主機。

提出問題：主機或子網最初是如何得到其 IP 地址的？

•獲取一塊地址

為了獲取一塊 IP 地址用於一個單位的子網，網絡管理員首先與 ISP 聯繫，ISP 會從他擁有的地址塊中取出一些地址給客戶。ICANN 負責全球範圍內的 IP 地址分配。

•獲取主機地址:動態主機配置協議

一個單位一旦獲得了一塊地址，它就可為該單位內的主機與路由器接口分配獨立的 IP 地址。 對於路由器接口地址，通常是由系統管理員手工配置的 IP。

對於主機，常使用動態主機配協議來完成 IP 地址的分配。

DHCP 使主機可以自動獲取 IP 地址。DHCP 還允許一臺主機獲取其他信息，如它的子網掩碼、第一跳路由器地址(常稱為默認網關)與本地 DNS 服務器的地址。

DHCP 是一個客戶機/服務器協議。

DHCP 協議是一個 4 個步驟的過程：

DHCP 服務器發現：

客戶機在 UDP 分組中向端口 67 發送 DHCP 發現消息。使用廣播目的地址並且使用“本 主機”地址

DHCP 服務器提供：

DHCP 服務器收到一個 DHCP 發現消息時，用一個 DHCP 提供消息對客戶機做出響應， 仍然使用 IP 廣播地址 255.255.255.255。子網中可能有多個 DHCP 服務器，所以該客戶機可以選擇其中的一個 DHCP 服務器。每個服務器提供消息中含有客戶機請求的事務 ID、向客戶機推薦的 IP 地址、網絡掩碼以及 IP 地址租用期

DHCP 請求：新到達的客戶機從一個或多個服務器中選擇一個，並用一個 DHCP 請求消息對選中的服務器進行響應，回顯配置參數。

DHCP 確認：服務器用這個消息對 DHCP 請求消息進行響應，確認所要求的參數。 一旦客戶機收到 DHCP ACK 後，交互便完成了，該客戶機就能夠在租用期內使用 DHCP 分配給它的IP 地址。

•網絡地址轉換 NAT

在 IP 地址的 A-B-C 三種主要類型裡各保留了三個區域作為私有地址，雖然現在使用 CIDR 技術，但這些地址的用途沒有改變，它們分別是：

10.0.0.0/24 172.16.0.0/16～172.31.0.0/16 192.168.1.0/24～192.168.255.0/24。

私有地址永遠不會被當作公有地址來分配，也就是說這幾塊地址裡面的 IP 地址不會出現在公共的外部網絡中。同時外部網絡中的路由器也無法處理以私有地址為目的地的數據報。

家庭網絡中的編址通常就使用私有的網絡地址 192.168.1.0/24。地址 192.168.1.0/24 是保留的 3 部分 IP 地址空間之一。

家用的無線路由在連接 ISP 的網絡後，ISP 會分配一個公有的 IP 地址。使用私有 IP 地址的設備如果要上網，需要把私有 IP 轉換為公有 IP。

家用的無線路由具有 NAT 功能。從本質上講，NAT 使能路由器對外界隱藏了家庭網絡的細節。

來自廣域網的所有數據報到達 NAT 路由器時都有相同的目的 IP 地址(ISP 會分配一個公有的 IP 地址)，那麼路由器是怎樣將一個給定數據報轉發給哪個內部主機的呢? NAT 路由器使用了一張 NAT 轉換表，並且在表項中包含了端口號和 IP 地址，將私有地址轉換為公有地址，這個過程中利用端口號作為轉換的索引。

假設一個用戶坐在家庭網絡主機 192.168.1.10 旁，請求 IP 地址為 128.119.40.186 的 Web 服務器(端口 80 )上的一個 Web 頁面。主機 192.168.1.10 為瀏覽器指派了(任意)源端口號 3345 並將該數據報發送到 LAN 中。

NAT 路由器收到該數據報，為該數據報生成一個新的源端口號 5001，將源 IP 地址改為其廣域網一側接口的 IP 地址 138.76.29.7，且將源端口 3345 更換為新端口號 5001。當生成一個新的源端口號時，NAT 路由器可選擇任意一個當前未在 NAT 轉換表中使用的源端口號。路由器也在其 NAT 轉換表中增加一表項，表現記錄了端口 5001 和源地址 192.168.1.10 的對應關係。

Web 服務器並不知道剛到達的包含 HTTP 請求的數據報已被 NAT 路由器進行了改裝，它會發回一個響應數據報，其目的地址是 NAT 路由器的 IP 地址，目的端口是 5001。 當該數據報到達 NAT 路由器時，路由器使用目的 IP 地址與目的端口號從 NAT 轉換表中檢索出家庭網絡瀏覽器使用的正確 IP 地址 192.168.1.10 和目的端口號 3345。然後，路由器改寫該數據報的目的 IP 地址與目的端口號，並向家庭網絡轉發該數據報。

NAT 穿越正越來越多地使用即插即用 UPnP 協議，UPnP 是一種允許主機發現並配置鄰近 NAT 的協議。大部分家用路由都支持 UPnP 協議。
