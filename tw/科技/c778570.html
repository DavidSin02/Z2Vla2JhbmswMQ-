Elasticsearch集群配置文件elasticsearch.yml詳解

2020-10-29
| 科技

下面分享Elasticsearch v7.x 集群搭建過程中，節點配置參數詳細說明。在實際工作中，要根據不同的業務場景和現實條件，做出切合實際的方案。

Elasticsearch的配置文件有elasticsearch.yml、log4j2.properties兩個配置文件，其中elasticsearch.yml是用來配置Elasticsearch服務的，log4j2.properties是用來配置日誌文件的。

下面是elasticsearch.yml配置文件的詳細說明。

Elasticsearch 7.x 訂單中心技術專欄，從真實項目實踐、項目源碼和生產故障分析等各個方面介紹了Elasticsearch相關的知識。


一、基本配置參數

# 集群名稱

cluster.name: "es_ordercluster"

# 節點名稱，這兒我直接取名為order-node1

node.name:order-node1

# 是否可以成為master節點

node.master: true

# 是否允許該節點存儲數據，默認開啟

node.data: true

# 網絡綁定，這裡我綁定 10.19.156.194

network.host: 10.19.156.194



二、網絡端口號

Elasticsearch 啟動監聽兩個端口，9300和9200（默認）。

9300端口是使用tcp客戶端連接使用的端口；

9200端口是通過http協議連接Elasticsearch使用的端口；


可以根據需要修改默認的端口號，比如和其他應用使用的端口號存在衝突：

#

# Set a custom port for HTTP:

#

transport.port: 8300

http.port: 8200


Elasticsearch支持開箱即用，可以沒有任何網絡配置。Elasticsearch將綁定到可用的環回地址，並將掃描本地端口9300到9305以嘗試連接到在同一服務器上運行的其他節點。 這提供了自動集群體驗，無需進行任何配置。


三、主節點的節點地址配置

discovery.seed_hosts 配置

此設置通常應包含集群中所有符合主節點的節點地址（ master-eligible nodes ）。 此設置包含主機數組或逗號分隔的字符串。

每個值應採用host:port或host的形式（其中port默認為設置transport.profiles.default.port，如果未設置則返回transport.port）。 請注意，必須將IPv6主機置於括號內。 此設置的默認值為127.0.0.1，[:: 1]。

該設置以前稱為discovery.zen.ping.unicast.hosts。它的舊名稱已經不贊成使用了，但為了保持向後兼容性，它將繼續工作。在將來的版本中，對舊名稱的支持將被刪除。

因此，在配置文件中，建議使用discovery.seed_hosts。

discovery.seed_hosts: ["10.131.113.95:8300", "10.131.213.95:8300", "10.131.213.97:8300"]


四、主節點選舉列表

cluster.initial_master_nodes 配置

在第一次啟動全新的Elasticsearch集群時，會出現一個集群引導步驟，該步驟確定在第一次選舉中計票的主合格節點集（master-eligible）。

在開發模式下，如果未配置，則此步驟由節點本身自動執行。

由於此自動引導本質上是不安全的，因此當我們在生產模式下啟動全新集群時，必須明確列出符合條件的節點的名稱或IP地址，這些節點的投票應在第一次選舉中計數。

使用cluster.initial_master_nodes設置此列表。

cluster.initial_master_nodes: ["order-node1", "order-node2","order-node3"]


五、設置最少數量主節點

discovery.zen.minimum_master_nodes 參數

為了防止數據丟失， discovery.zen.minimum_master_nodes 配置至關重要， 以便每個候選主節點知道為了形成集群而必須可見的最少數量的候選主節點。這個配置就是告訴 Elasticsearch 當沒有足夠 master 候選節點的時候，就不要進行 master 節點選舉，等 master 候選節點足夠了才進行選舉。

沒有這種設置，遇到網絡故障的群集有可能將群集分成兩個獨立的群集（腦裂），這將導致數據丟失。

為了避免腦裂，候選主節點的數量應該設置為：

(master_eligible_nodes / 2) + 1

換句話說，如果現在有3個節點，最小候選主節點數應該是（3/2)+1=2。

discovery.zen.minimum_master_nodes: 2

如果你有兩個節點，你遇到難題了。法定數當然是 2 ，但是這意味著如果有一個節點掛掉，你整個集群就不可用了。 設置成 1 可以保證集群的功能，但是就無法保證集群腦裂了，像這樣的情況，你最好至少保證有 3 個節點。

discovery.zen.minimum_master_nodes: 2


注意：在Elasticsearch 7.0中，已經完全重新考慮並重建了集群協調層（cluster coordination layer）。新的實現提供了安全的、秒級的master節點選舉時間，其中可能需要幾秒鐘選擇一個新的master，針對關鍵任務的部署而言，這個時間是值得的。這樣，discovery.zen.minimum_master_nodes 參數就取消了，集群的增長和收縮將變得更加安全和方便，從而也大大減少了錯誤配置系統參數的可能性。

在Elasticsearch 7.x節點上，可以配置discovery.zen.minimum_master_nodes 參數，但是會被忽略。

參考官方文檔鏈接：https://www.elastic.co/guide/en/elasticsearch/reference/current/breaking-changes-7.0.html


六、設置存儲路徑

# ----------------------------------- Paths ------------------------------------

# 設置配置文件的存儲路徑，默認是es根目錄下的config文件夾

# path.conf: /path/to/conf

#

# 設置索引數據的存儲路徑，默認是es根目錄下的data文件夾，可以設置多個存儲路徑，用逗號隔開，例：

# path.data: /path/to/data1,/path/to/data2

# path.data: /path/to/data

#

# 設置日誌文件的存儲路徑，默認是es根目錄下的logs文件夾

# path.logs: /path/to/logs

一般情況下，可以保持默認值，不需要調整。


七、鎖定物理內存地址

bootstrap.memory_lock: true

這個配置的意義：鎖定物理內存地址，防止Elasticsearch內存被交換出去，也就是避免Elasticsearch使用swap交換分區，頻繁的交換，會導致IOPS變高。

# ----------------------------------- Memory -----------------------------------

#

# Lock the memory on startup:

#

bootstrap.memory_lock: true

#

# Make sure that the heap size is set to about half the memory available

# on the system and that the owner of the process is allowed to use this

# limit.

#

# Elasticsearch performs poorly when the system is swapping the memory.

#

設置好後，啟動Elasticsearch會報錯：

[1] bootstrap checks failed

[1]: memory locking requested for elasticsearch process but memory is not locked


這裡需要額外設置下系統配置文件 需要切換到root用戶：

（1）修改/etc/security/limits.conf

最後添加：

* soft nofile 65536

* hard nofile 65536

* soft nproc 131072

* hard nproc 131072

* soft memlock unlimited

* hard memlock unlimited

（2）修改/etc/systemd/system.conf

修改下面幾項：

DefaultLimitNOFILE=65536

DefaultLimitNPROC=32000

DefaultLimitMEMLOCK=infinity


（3）最後執行重新載入

systemctl daemon-reload

有關如何正確設置內存鎖定的更多詳細信息，請參閱啟用bootstrap.memory_lock。

https://www.elastic.co/guide/en/elasticsearch/reference/current/setup-configuration-memory.html#bootstrap-memory_lock
