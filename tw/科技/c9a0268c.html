一、時序數據與時序數據庫
什麼是時序數據庫？
按照維基百科解釋，時間序列數據庫（TSDB）是一個為了用於處理時間序列數據而優化的軟件系統，其按時間數值或時間範圍進行索引。

時序數據庫增長趨勢
時序數據庫從2014年開始就被DB-Engine列入了單獨的數據庫類別進行統計。下圖所示的是在過去的24個月內所有數據庫類別在DB-Engine上的增長趨勢，可以看出時序數據庫的增長趨勢排在第一位。

時序數據舉例
列舉一個時序數據的例子，比如在張北有一些風力發電機組，這些風力發電機都具有能夠代表自身唯一性的標籤(Tag)，比如生產廠商、風場、型號以及自身ID等，而每個風機都會產生一些指標(Metrics)，比如當前的運轉功率、當前風速等。隨著時間，這些風機會產生新的數據，這些數據就是時序數據。

時序數據庫的應用場景
如下圖所示的是一些時序數據庫的應用場景，第一個就是證券交易場景，大家所熟知的K線圖就是一個比較標準的時間序列。又比如像廣告數據展示，網站PV/UV，這些對運營人員很重要的時序數據。智能手環、智能手錶等可穿戴設備可以採集個人健康數據，比如睡眠信息、運動信息等，並按照時間順序將其發送到雲端，經過一系列計算，最終為用戶提供一些建議和提示。除上述比較常見的時序數據之外，還有氣溫變化情況、工業傳感器數據、服務器監控數據、網絡設備監控數據等。最後，還有車聯網數據場景，如今新能源汽車越來越多，大部分汽車都支持通過APP直接管理，它們會時時刻刻地產生車輛的GPS、續航、安全事件等數據，雲端可以利用這些數據進行分析處理，用戶也可以用來查看車輛情況。

時序數據庫的特性
時序特性：對於時序數據的時序特性而言，每條數據都會有一個時間戳，而按照要求的不同，時間戳在精度上有不同。通常而言，大多數時序數據以秒為單位，而在一些工業上會有以毫秒或者納秒為單位的情況。數據的採樣頻率主要分為兩種，一種是以可感知的週期頻率來採集的，另外一種則是以不可感知的網站PV/UV進行離散型採樣，這是因為雖然用戶訪問可以被提前預測，但是精確度不會很高。

數據特性：對於時序數據的數據特性而言，數據都是順序可追加的，也是多維可關聯的，它們具有不同的指標。近期的時序數據往往是大家所關注的，訪問頻率會比較高，而時間較遠的數據所帶來的價值則會隨著時間的推移而逐漸降低，因此時序數據也存在冷熱歸檔的特性。此外，時序數據也會包含一些具體的數值、狀態和事件。

CRUD特性：而對於時序數據庫而言，也會有一些CRUD的特性。時序數據庫的讀寫有點類似LSM數據庫的特點，寫操作的頻率遠遠大於讀，並且存在一定的時間窗口。此外，時序數據庫通常極少更新數據，存在一定的覆蓋寫，並且支持批量刪除操作。而隨著業務對於技術的要求越來越高，因此時序數據庫也需要支持高可用、高可靠以及可伸縮等特性。通常情況下，時序數據庫不具備事務能力。

二、時序數據庫的演變


如下圖所示的是隨著時間而誕生的一些比較具有代表性的時序數據庫。最早誕生的時序數據庫就是RRDTool，RRD是Round-Robin Database的意思，它屬於基於平板文件的簡單存儲。後來，伴隨著大數據和Hadoop生態的發展，OpenTSDB，KairosDB這些基於Hadoop之上通用存儲而專門構建的時間序列數據庫開始出現，它可以按時間間隔高效地存儲和處理數據。 而由於OpenTSDB基於HBase這樣一個通用的NoSQL存儲，雖然可以解決時序相關的一些場景問題，但無法滿足對於成本和性能要求更高的場景。而隨著Docker、Kubernetes、微服務等技術的發展，以及對於IoT的發展預期越來越強烈。在數據隨著時間而增長的過程中，時間序列數據成為增長最快的數據類型之一。高性能、低成本的垂直型時序數據庫開始誕生，以InfluxDB為代表的具有時序特徵的數據存儲引擎逐步引領市場。而在後來的2015年，FaceBook開源了Gorilla，在時序領域應用比較廣泛。在2017年，微軟發佈了Azure Series Insights。2018年，Amazon發佈了Timestream預覽版。由此可以看出，時序數據庫的需求與應用愈發廣泛。


可以簡單概括時序數據庫發展演變歷程：

  • 第一代時序數據庫：基於平板文件的簡單存儲。代表：RRDTools，Graphite。RRD（Round Robin Database）。
  • 第二代時序數據庫：依賴與Hadoop生態的通用分佈式存儲系統。代表：OpenTSDB，KairosDB
  • 第三代時序數據庫：對高性能，低成本有強需求，需要針對時序領域特別專門設計。代表：InfluxDB
  • 特例：TimescaleDB。基於PostgreSQL關係數據庫構建。

時序數據庫發展現狀
如下圖所示的是時序數據庫在不同領域中比較具有代表性的產品，主要分為了開源、公有云、學術以及商業&工業四大領域。

三、時序數據庫的對比

OpenTSDB
OpenTSDB以HBase作為底層存儲，對應的會有一些TSD，也就是OpenTSDB的主要進程。TSD互相是對等結構，不存在Master-Slave主從關係，在向OpenTSDB寫數據之前需要做好負載均衡。

下圖所示的是OpenTSDB存儲數據的邏輯關係，以MySQL的指標數據為例。最前面是指標Metric，接著是Timestamp，需要注意的是這裡的Timestamp和HBase以及HFile上的Timestamp是沒有任何關係的。HFlie裡面的Timestamp是用來MVCC或TTL等來使用的，而時序領域所使用的Timestamp主要指的是數據產生的時間或落盤的時間。在Timestamp之後是指標的Value以及Tag，也就是DataSource。以上是邏輯上的關係，實際情況中會因存儲的不同有所變化。

對於每條數據而言，都會有指標名稱和對應的DataSource，如果直接把這些數據存儲到HBase中則會造成極大的存儲浪費，這是因為字符串存儲佔用的空間比數值類型多很多。因此，OpenTSDB對應的解決辦法是將每一個指標都編碼成如下圖所示的數值類型，進而減少存儲佔用，而將對應的Mapping映射存儲一張全局映射表中。這樣做的優勢是相對節省存儲，而缺點則是由於全局映射關係，使得對於硬件的要求有所提升。

下圖所示的是HFile結構。HFile以Block作為最小存儲單位，主要的存儲類型有Index Block以及Data Block等。對於數據部分而言，每個Data Block可以拆分成一個個Key-Value對象，再進一步細化則會有偏移量KeyLength和ValueLength以及字符串對象KeyBytes和ValueBytes。而對於KeyBytes再進一步細分，可以分為RowLength、RowBytes、ColumnFamilyLength、ColumnFamily、TimeStamp以及KeyType等對象。很多偏移量字段對於HBase而言有用而對於時序領域而言這些偏移量字段沒有任何意義，但是因為HFlie的設計原因，所以不得不造成存儲甚至性能上的浪費。

OpenTSDB做了一些優化。首先，OpenTSDB的一行數據的Timestamp會以小時為粒度，然後將小時中的每一秒設置到列上，用邏輯上的一行來存儲一小時的數據，用偏移量替代原始數值類型，以此來減少存儲的佔用。其次，對數據源(Tag)和指標(Metric)使用全局編碼。但是經過全局編碼後依然存在數據冗餘。再次，還是由於通用存儲的原因，HFile主要解決的還是Key-Value等場景，並沒有針對時序場景特定壓縮，因此也會造成存儲和性能浪費。最後，OpenTSDB是時序數據庫中少數幾個沒有實現多維查詢的數據庫。

Druid
Druid是一個不折不扣的列式數據庫，所以每一列都會獨立存儲。Druid和HBase的處理方式一樣，都是採用編碼字典對標籤值進行編碼，將String類型的標籤值編碼成int值。但和HBase不一樣的是，Druid編碼是局部編碼，Druid和HBase都採用LSM結構，數據先寫入內存再flush到數據文件，Druid編碼是文件級別的，局部編碼可以有效減小對內存的巨大壓力。

Druid的特性
Druid的這種列式存儲模式還有如下好處：

  • 有數據類型概念，數據壓縮率高。每列獨立存儲，可以針對每列進行壓縮，而且可以為每列設置對應的壓縮策略，比如時間列、int、fload、double、string都可以分別進行壓縮，壓縮效果更好。
  • 支持多維查找。Druid為Datasource的每個列分別設置了Bitmap索引，利用Bitmap索引實現多維查找。

Druid存儲模型也有一些問題：

  • 數據依然存在冗餘。和OpenTSDB一樣，Tags存在大量的冗餘。這是因為NoSQL的列存儲也需要一些指標來記錄列的信息。
  • 指定數據源的範圍查找並沒有OpenTSDB高效。這是因為Druid會將數據拆成多列，每列在查詢時走Bitmap索引，再最後使用與操作找到滿足條件的行號，這個過程需要一定的開銷。而OpenTSDB中直接可以根據數據源拼成rowkey，在索引block按B+樹方式進行查找，效率更高。

InfluxDB
如下圖所示的是InfluxDB邏輯架構圖。InfluxDB其內部會有一些Rentention Policy，負責配置數據Duration，ShardGroup Duration和副本數量。對於Shard Group而言，可以簡單地按字面意思理解，Influx Cluster會在Shard Group內創建一系列的Shard。Shard Group是一個邏輯概念。對於Shard，開源版本Shard數量默認為1，商業版會按照數據節點數以及所需的數據副本數量創建。N個數據節點X個副本，則創建N/X個Shard。Shard就是數據分區，具體形式就是TSM Engine，其主要負責數據編碼，數據讀寫。TSM是LSM的變形，因此相似於LSM的數據庫，也具有Cache，WAL，Compaction，Flush等等這些組件與操作，只不過TSM在LSM的基礎之上進行了一些優化。

如下圖所示的是InfluxDB的時序數據在內存中的組織形式，時序數據寫入InfluxDB內存後會按照SeriesKey進行組織。可以認為SeriesKey就是一個數據源，而Field就是採集的指標。比如SeriesKey是一部iPhone，Field包含經緯度，高度，行走步數等。具體的數值會藉助偏移量的方式進行存儲。

如下圖所示的是TSM文件的最粗粒度的結構。在InfluxDB中，數據以Block為最小單位存儲在文件中，如圖所示文件結構主要包含Data Block、Index Block以及最後的Footer。

將上圖進一步細化的結果如下圖所示。在TSM中，索引做到了文件級別，索引的實現類似於B+樹，而且數據塊和索引結構存儲在同一個文件中。InfluxDB中有數據類型的概念，同一個Block中只會存儲同一種Key的數據。

SM文件的詳解解釋如下：

  1. Type：表示該SeriesKey對應的時間序列的數據類型，數值數據類型通常為int、long、float以及double等。不同的數據類型對應不同的編碼方式。
  2. Length：len(Timestamps)，用於讀取Timestamps區域數據，解析Block。時序數據的時間值以及指標值在一個Block內部是按照列式存儲的：所有的時間值存儲在一起，所有的指標值存儲在一起。使用列式存儲可以極大提高系統的壓縮效率。
  3. Timestamps：時間值存儲在一起形成的數據集，通常來說，時間序列中時間值的間隔都是比較固定的，比如每隔一秒鐘採集一次的時間值間隔都是1s，這種具有固定間隔值的時間序列壓縮非常高效，TSM採用了Facebook開源的Geringei系統中對時序時間的壓縮算法：delta-delta編碼。
  4. Values：指標值存儲在一起形成的數據集，同一種Key對應的指標值數據類型都是相同的，由Type字段表徵，相同類型的數據值可以很好的壓縮，而且時序數據的特點決定了這些相鄰時間序列的數據值基本都相差不大，因此也可以非常高效的壓縮。需要注意的是，不同數據類型對應不同的編碼算法。

對於InfluxDB而言，在1.3版本之前，所有的索引都做到了內存裡。內存索引的好處就是查詢速度優於其他持久化存儲介質，但是缺點是索引的容量會受到限制。如果具有海量的標籤，那麼索引量非常大，可能內存中無法存放；其次，如果發生宕機，那麼在Startup階段，就需要重新掃描所有的TSM文件，並在內存中重新排列這些索引。正是因為上述問題，使得索引對於資源的消耗過高，所以在1.3版本之後，InfluxDB實現了基於磁盤的索引。雖然這樣的磁盤索引能夠解決大部分場景，但是因為存儲佔用的關係，所以也只實現了Tag到SeriesKey的映射，沒有實現Tag到SeriesKey + FieldKey + Timestamp的映射。

Prometheus
與其他時序數據庫不同，Prometheus不僅包含存儲，更是一個生態系統，提供了幾乎整個運維領域的全部組件，提供了服務發現、數據採集服務、存儲相關服務、報警相關組件等，並且提供了Web UI界面。Prometheus的優點是其提供了一整套監控解決方案，不需要運維人員去逐個拼裝開源組件。此外，Prometheus還提供了一套標準，研發人員可以根據協議來替換組件，提供了更好的靈活性。

Prometheus提供瞭如下的關鍵功能

  • 多維數據模型：Metric，Labels
  • 靈活的查詢語言：PromQL， 在同一個查詢語句，可以對多個 Metrics 進行乘法、加法、連接、取分數位等操作。
  • 可獨立部署，拆箱即用，不依賴分佈式存儲
  • 通過Http pull的採集方式
  • 通過push gateway來做push方式的兼容
  • 通過靜態配置或服務發現獲取監控項
  • 支持圖表和dashboard等多種方式

TimeScaleDB
TimeScaleDB是一個比較特殊的例子，它是基於關係型數據庫實現的時序數據庫。關係數據庫通常默認的引擎所採用的都是B+樹，而對於B+樹而言，隨著數據量不斷寫入，其內部節點的排序會造成很大壓力。從用戶的視角，TimescaleDB對外暴露的是一個叫做hypertables的表，其是一個抽象概念。隨著數據的不斷寫入，其將會形成多個物理上的子表，使得每個子表上的數據不至於過高，解決了單表數據量的問題。

TimeScaleDB的特性

  • 基於標準的PG，未對PG存儲層做任何修改
  • 時序數據表的透明自動分區特性
  • 提供了若干面向時序數據應用場景的特殊SQL接口(使用時空UDF實現)
  • 針對時序數據的寫入和查詢對PostgreSQL的Planner進行擴展和優化
  • 面向時序數據表的定製化並行查詢

除了上述特性之外，TimeScaleDB還具有時序數據表的透明自動分區功能。該功能所要實現的目標如下：

  • 隨著數據寫入的不斷增加，將時序數據表的數據分區存放，保證每一個分區的索引維持在一個較小規模，從而維持寫入的性能。
  • 基於時序數據的查詢場景，自動分區時以時序數據的時間戳為分區鍵，從而確保查詢時可以快速定位到所需的數據分區，保證查詢性能。
  • 分區過程對用戶透明，從而達到Auto-Scalability的效果。

藉助透明化自動分區的特性，根據官方的測試結果，在同樣的數據量級下，TimescaleDB的寫入性能與PG的傳統單表寫入場景相比，即使隨著數量級的不斷增大，性能也能維持在一個比較穩定的狀態。

TimeScaleDB的特性
對於TimeScaleDB的優勢而言，首先，其100%繼承自PostgreSQL的生態，且由於完整支持SQL，對於未接觸過時序數據庫的用戶有很高吸引力；其次，因為TimeScaleDB基於PG，所以也具有靈活的JSON格式支持，無需在JSON解析上浪費計算資源；並且TimeScaleDB還可以實現快速的複雜查詢，也繼承了PostgreSQL的高品質與穩定性優勢，還具有強ACID支持。

對於TimescaleDB的缺陷而言，因為其是PostgreSQL的一個延伸，因此不能像從內核和存儲層面針對時序數據庫的場景進行極致優化。雖然官網上宣傳能夠支持集群模式，但是仍舊處於開發中，目前從產品的架構來看仍然是一個單機庫，不能發揮分佈式技術的優勢。而且數據雖然自動分區，但是由於時間戳決定分區，比較容易形成I/O熱點。而在功能層面，面向時序數據庫場景的特性有限，更像是一個傳統OLTP+部分時序特性。在NoSQL具有優勢的簡單快速查詢場景，非結構化數據，存儲成本和離線分析場景方面，TimescaleDB不具有優勢。

阿里雲時序時空數據庫TSDB
阿里雲時序時空數據庫TSDB的發展演變經歷了三個階段，最開始1.0階段主要基於OpenTSDB，底層實現了雙引擎——HBase和HiStore。而隨著阿里業務場景對於時序數據庫不斷提出新的挑戰，在2.0階段中，阿里雲就將OpenTSDB引擎換成了自研的TSDB引擎，實現了OpenTSDB所沒有支持的倒排索引、基於時序場景的特殊編碼、分佈式流計算聚合數據等功能和特性。而隨著業務場景以及整個公司生態的完善，阿里雲時序時空數據庫TSDB實現了邊緣計算和雲的一體化，實現了TSQL，兼容了Prometheus生態，並且除了能夠支持時序數據引擎之外，還能夠支持時空數據引擎。

阿里雲時序時空數據庫TSDB的特性

  • 支持兩種存儲模型。時間線的ID + TS和繼承自OpenTSDB的Metric + TS + Tag。
  • 雙層Sharding結構。先基於時間分區索引，並在存儲層做Salting，避免時間分區熱點。
  • 基於Coprocessor實現列合併壓縮算法，在通用ZSTD+DIFF編碼的基礎上進一步將存儲空間減小至壓縮前的1/3。
  • 支持分佈式流聚合計算。
  • 工作負載優化：讀寫線程池分離，大小查詢資源隔離，超時大查詢Skip，限流及狀態管理。

四、總結


對於時序數據庫而言，主要存在三種流派。第一種是以Timescale為代表的標準SQL，其能夠極大地降低入門門檻，也能解決時序問題，但是體量上不佔優勢；第二種是以OpenTSDB為代表的基於通用NoSQL的時序存儲引擎；最後一種則是為了應對時序場景的業務挑戰而單獨設計的以InfluxDB為代表NoSQL數據庫。

這三種時序數據庫的主要區別如下所示：

  • SQL vs NoSQL
  • 固定Schema vs 動態Schema
  • 通用存儲 vs 時序專用存儲
  • 單機 vs 分佈式
  • 結構化數據 vs 非結構化數據
  • 低學習成本 vs 相對高的學習成本
