什麼是內存攻擊

內存攻擊指的是攻擊者利用軟件安全漏洞，構造惡意輸入導致軟件在處理輸入數據時出現非預期錯誤，將輸入數據寫入內存中的某些特定敏感位置，從而劫持軟件控制流，轉而執行外部輸入的指令代碼，造成目標系統被獲取遠程控制或拒絕服務。

內存攻擊的表面原因是軟件編寫錯誤，諸如過濾輸入的條件設置缺陷、變量類型轉換錯誤、邏輯判斷錯誤、指針引用錯誤等；但究其根本原因，是現代電子計算機在實現圖靈機模型時，沒有在內存中嚴格區分數據和指令，這就存在外部輸入數據成為指令代碼從而被執行的可能。任何操作系統級別的防護措施都不可能完全根除現代計算機體系結構上的這個弊端，只能試圖去阻止攻擊者利用 (Exploit)。

緩衝區溢出漏洞機理

緩衝區溢出 (Buffer Overflow 或 Buffer Overrun) 漏洞是程序由於缺乏對緩衝區邊界條件檢查而引起的一種異常行為。通常是程序向緩衝區中寫數據，但內容超過了程序員設定的緩衝區邊界，從而覆蓋了相鄰的內存區域，造成覆蓋程序中的其他變量甚至影響控制流的敏感數據，造成程序的非預期行為。而 C 和 C++ 語言由於缺乏內在安全的內存分配與管理機制，因此很容易導致緩衝區溢出的相關問題。

如下圖：內存中保存了相鄰的兩個變量。A 是 char[] 字符串類型，作為緩衝區用於存儲外部輸入的字符串，長度為 8 字節；而變量 B 是短整數型。

在程序執行時，某指令向 A 中寫入了長度大於 8 的字符串，越過了 A 的邊界覆蓋了 B 中的內容，造成變量 B 的值被修改。如：寫入的字符串是 “abcdefghi”，長度為 9，加上結束符 “\0” 之後將修改 B 的值，從原先的 65535 修改為 0x0069 (十六進制)，即105。

一般根據緩衝區溢出的內存位置不同，將緩衝區溢出又分為 棧溢出 (Stack Overflow) 與 堆溢出 (Heap Overflow)。

棧溢出利用原理

程序執行過程中的棧，是由操作系統創建和維護的，同時也支持了程序內的函數調用功能。在進行函數調用時，程序會將返回地址壓入棧中，而執行完被調用的函數代碼後，則會通過 ret 指令從棧中彈出返回地址，裝載到 EIP 指令寄存器，從而繼續程序運行。

然而這種將控制程序流程的敏感數據與程序變量同時保存在同一段內存空間中的馮諾依曼體系，必然會給緩衝區溢出攻擊帶來本質上的可行性。

棧溢出發生在程序向位於棧中的內存地址寫數據時，當寫入的數據長度超過棧分配給緩衝區的空間時，就會造成棧溢出。從棧溢出的原理出發，攻擊者可以找到如下三種方式來利用這種類型的漏洞：

  • 覆蓋緩衝區附近的程序變量：改變程序的執行流程和結果
  • 覆蓋棧中保存的函數返回地址：修改為攻擊者指定的地址，當程序返回時，程序流程將跳轉到攻擊者指定地址，理想情況下可以執行任意代碼
  • 覆蓋某個函數指針或程序異常處理結構：只要溢出之後目標函數或異常處理例程被執行，同樣可以讓程序流程跳轉到任意地址

而其中最常見的利用方式就是覆蓋棧中的函數返回地址。

1. 覆蓋函數返回地址利用方式

函數調用是程序中最常見的命令，程序調用函數時，程序流程將暫時轉到被調用的函數，函數執行完之後再跳轉回原來的位置，所以在執行調用函數前需要保存下一跳指令的地址，讓程序在執行完函數調用後能夠從這個指令地址處繼續執行。若程序將該函數返回地址和函數的調用參數、局部變量異同保存在棧中，這就給了攻擊者溢出棧緩衝區從而達到修改函數返回地址的機會。

棧溢出代碼示例：

#inclide <string.h>void foo(char *bar){	char c[8];	strcpy(c, bar);	//沒有進行邊界檢查，從而存在棧溢出漏洞}int main(){	char array[] = "ABCDABCDABCD18FF1800"	foo(array);	//調用函數	return 0;}

主程序執行了一次對自定義函數 foo 的調用，在子函數中執行了一次字符串複製操作 (strcpy)，執行結果是字符串 array 中的內容被複制到局部變量字符串 c 中。

編譯執行這一段代碼，當調用 strcpy 函數時，進程棧佈局如下圖左側所示，從地地址到高地址分別是未分配的棧內存空間、局部變量字符串 c 分配的空間、進入子函數時自動保存的 EBP 寄存器值、返回地址、調用自定義函數 foo 的參數、父函數棧空間。執行 strcpy 函數之後的棧佈局如下圖右側所示，上述代碼中，源字符串 array 中的16 個字符將複製到 c 中，其中最後 4 個字符覆蓋了棧中返回地址，該為字符串 c 的起始地址 0x0018FF18。所以當函數 foo 返回時，程序就會跳轉到該地址處，將字符串中的數據作為指令執行。因此，如果攻擊者可以控制源字符串，那麼他就可以將其替換為 Shellcode 並複製到棧中，然後利用該漏洞執行 Shellcode。

由於程序每次運行時，棧中變量的地址都會發生變化，即上述字符串 c 在棧中位置往往不固定，所以一般會通過一些跳轉寄存器的指令作為跳板，使得程序能夠執行到棧中的 Shellcode。最常見的是以 JMP ESP 的地址來覆蓋返回地址，從而使得程序執行該指令之後重新跳轉回棧中，來執行緩衝區溢出之後的數據。

2. 覆蓋異常處理結構利用方式

程序在運行過程中可能會發生一些異常，比如除 0 計算、訪問無效內存地址等，此時就需要正常指令序列之外的代碼處理這些異常。Windows 提供結構化異常處理機制 SEH，可以利用程序自定義的異常處理函數或者操作系統默認的處理函數處理異常。異常處理結構以鏈表形式春促在棧中，寄存器 FS 指向當前活動線程的 TEB (線程環境塊) 結構。結構體有兩個 DWORD (雙字變量，4字節) 類型的變量：

  • 異常處理函數 SEH 例程的地址

程序在執行生產指令序列出現異常時，會由異常處理過程接管，操作系統從鏈表頭到尾尋找能處理此異常的函數，由找到的第一個函數進行處理。如果沒有任何合適的處理函數，則由最後一個函數即系統默認的處理函數來負責處理，通常彈出錯誤對話框，強制關閉程序。

棧溢出之後覆蓋異常處理結構的利用方式，就是用特定地址覆蓋棧中異常處理結構體中的異常處理函數指針，並處罰異常，導致去加載篡改之後的處理函數指針。

覆蓋異常處理結構的棧溢出利用方式，與覆蓋棧中函數返回地址利用方式並沒有本質區別。一般來說，異常處理結構接近棧底，所以從緩衝區頭部到異常處理結構之間的內存空間很大，利用起來可能更方便。最關鍵的是，有時緩衝區溢出之後到程序執行到函數返回之前就不可避免地處罰異常，這種情況下，就必須使用覆蓋異常處理結構的利用方式，於此同時，這種利用方式也可以繞過操作系統的棧保護機制。

堆溢出利用原理

不同於棧，堆是程序運行時動態分配的內存，用戶通過 malloc、new 等函數申請內存，通過返回的起始地址指針對分配的內存進行操作，使用完後要通過 free、delete 等函數釋放這部分內存，否則會造成內存洩露。對的操作分為分配、釋放、合併三種。因為堆在內存中位置不固定，大小比較自由，多次申請、釋放後可能會更加凌亂，系統從性能、空間利用率還有越來越受到重視的安全角度出發，來管理堆，具體實現比較複雜。只簡介最常見的空閒塊堆操作引起的緩衝區溢出。

系統根據大小不同維護一系列的堆塊，如下圖所示：堆塊分為塊首和數據區，其中空閒堆塊數據區的前兩個雙字 (DWORD) 分別是雙向鏈表的兩個指針。通常同樣大小的空閒堆塊通過雙向鏈表連接在一起，分配與釋放堆，分別對應插入與刪除雙向鏈表節點的操作，而合併則會同時進行著兩種操作。空閒堆塊中兩個指針 “Previous block” 和 “Next block”，分別指向雙向鏈表中此堆塊的前後兩個堆塊的數據部分。分配一個堆塊時，將分配堆塊從空閒堆塊雙鏈表中刪除，會有如下所示的操作：

同一個堆中的堆塊在內存中通常是連續的，由此可能發生的狀況是：在向一個已分配堆塊中寫入數據時，由於數據長度超出了該堆塊的大小，導致數據溢出覆蓋堆塊後方 (高地址處) 的相鄰空閒堆塊，而包含的兩個堆塊指針 (即 Previous block 和 Next block) 會被覆蓋。

假設有空閒堆塊 *p，則 p->previous 是指向雙向鏈表中的 p 的前一堆塊的前向指針，p->next 是後向指針。若 p 的兩個堆塊指針被覆蓋，即 p->previous = X，p->next = Y。如果這個空閒堆塊被分配出去，需要將這個節點從空閒堆塊鏈表中刪除，那麼分配過程中的 DeleteBlock 函數 (上述函數) 就會將 p 下一個空閒的前向指針 (p->next->previous)指向 p 之前的空閒塊前向指針 (p->previous)。需要注意的是：每個堆塊指針指向的就是堆塊的 Previous block。所以 p->next->previous 相當於對 Y 進行解引用，即 *Y，因此執行的效果就是 *Y=X。從而可以利用超長數據覆蓋空閒堆塊的這兩個指針，達到向 Y 指向的任意地址處寫入 X 包含的任意內容的目的。

涉及內存鏈表操作的堆內存分配、釋放、合併操作都可能實現這一效果，即向攻擊者任意指定地址寫入 4 字節的任意內容，業內人士稱之為 “arbitrary DWORD reset” 或者 “DWORD shoot” 攻擊。在得到一個將指定內存地址改寫為任意值的機會後，攻擊者可以寫出利用的程序，用於覆蓋內存堆中的一些函數指針地址、C++ 類對象虛函數表、GOT 全局偏移表入口地址或者 DTORS 地址等，而改寫的值就是指向內存中的 Shellcode 的地址。
