【安全】三分鐘瞭解JWT

2020-11-14
| 科技

令牌身份驗證，OAuth或JSON Web令牌的新手？ 這是一個很好的起點！

首先，什麼是JSON Web令牌，或JWT（發音為“jot”）？ 簡而言之，JWT是用於令牌認證的安全且值得信賴的標準。 JWT允許您使用簽名對信息（稱為聲明）進行數字簽名，並且可以在以後使用祕密簽名密鑰進行驗證。

什麼是令牌認證？

應用程序確認用戶身份的過程稱為身份驗證。傳統上，應用程序通過會話cookie保持身份，這些cookie依賴於服務器端存儲的會話ID。在此結構中，開發人員被迫創建獨特且特定於服務器的會話存儲，或實現為完全獨立的會話存儲層。

令牌認證是一種更現代的方法，設計解決了服務器端會話ID無法解決的問題。使用令牌代替會話ID可以降低服務器負載，簡化權限管理，並提供更好的工具來支持分佈式或基於雲的基礎架構。在此方法中，為用戶提供可驗證憑據後會生成令牌。初始身份驗證可以是用戶名/密碼憑據，API密鑰，甚至來自其他服務的令牌。 （Stormpath的API密鑰身份驗證功能就是一個例子。）

有興趣瞭解更多？查看此博客文章，瞭解如何使用令牌擴展用戶管理或完整的產品文檔。

JWE，JWS和JWT

根據JWT規範，“JWT將一組聲明表示為以JWS和/或JWE結構編碼的JSON對象。”術語“JWT”在技術上僅描述了無符號標記;我們稱之為JWT的通常是JWS或JWS + JWE。

JWS - JSON Web簽名

在JWS方案中，服務器對JWT進行簽名並使用簽名將其發送到客戶端。簽名保證了JWT要求沒有被偽造或篡改。但是，JWT未加密（內容基本上是純文本）。

JWE - JSON Web加密

另一方面，JWE方案在不簽名的情況下加密內容。這為您的JWT帶來了機密性，但不是JWE簽名和封裝JWE的安全性。

什麼是OAuth？

OAuth 2.0是與可以委派身份驗證或提供授權的服務進行交互的框架。它被廣泛用於許多移動和Web應用程序。 OAuth 2.0沒有指定令牌格式，但JWT正在迅速成為業界的事實標準。

在OAuth範例中，有兩種令牌類型：訪問和刷新令牌。首次進行身份驗證時，通常會為您的應用程序（以及您的用戶）提供兩個令牌，但訪問令牌設置為在短時間後過期（此持續時間可在應用程序中配置）。初始訪問令牌到期後，刷新令牌將允許您的應用程序獲取新的訪問令牌。刷新令牌具有設置的到期時間，允許無限制地使用，直到達到該到期點。 Access和Refresh Tokens都具有內置安全性（簽名時）以防止篡改，並且僅在特定持續時間內有效。

Stormpath使用OAuth，因為它是一個行業標準，任何兼容的庫都可以利用它。 Stormpath目前支持三種OAuth的授權類型：

  • 密碼授予類型：提供基於用戶名和密碼獲取訪問令牌的功能
  • 刷新授權類型：提供基於特殊刷新令牌生成另一個訪問令牌的功能
  • 客戶端憑據授權類型：提供為訪問令牌交換API密鑰對的功能。這通過API密鑰管理功能得到支持

用Java創建和驗證JWT

所以，你在代幣上出售，現在，你如何在你的應用程序中使用它們？

好吧，如果你是Java開發人員，你應該從JJWT開始。 JJWT是一個Java庫，提供由我們自己的Les Hazlewood開發並由開發人員社區維護的端到端JSON Web令牌創建和驗證。永遠免費和開源（Apache許可證，版本2.0），它設計了一個以構建者為中心的界面，隱藏了其大部分複雜性。

最終的JWT將是一個由三部分組成的Base64編碼字符串，使用提供的密鑰使用指定的簽名算法進行簽名。在此之後，令牌已準備好與另一方共享。

這是使用JJWT庫從上面創建JWT的示例：

String jwt = Jwts.builder() .setSubject("users/TzMUocMF4p") .setExpiration(new Date(1300819380)) .claim("name", "Robert Token Man") .claim("scope", "self groups/admins") .signWith( SignatureAlgorithm.HS256, "secret".getBytes("UTF-8") ) .compact();

例外

JJWT在與JWT合作時進行了各種驗證。所有與JJWT相關的異常都是RuntimeExceptions，以JwtException作為基類。

這些錯誤會導致拋出特定異常：

  • ClaimJwtException：在驗證JWT聲明失敗後拋出
  • ExpiredJwtException：表示JWT在過期後被接受，必須被拒絕
  • MalformedJwtException：當JWT未正確構造並且應該被拒絕時拋出
  • PrematureJwtException：表示JWT在被允許訪問之前被接受，必須被拒絕
  • SignatureException：表示計算簽名或驗證JWT的現有簽名失敗
  • UnsupportedJwtException：在接收到與應用程序預期格式不匹配的特定格式/配置的JWT時拋出。例如，如果在應用程序需要加密簽名的聲明JWS時解析無符號明文JWT，則會拋出此異常
  • JJWT使用了許多其他Exception類。它們都可以在JJWT源代碼中的io.jsonwebtoken包中找到。

令牌安全嗎？

這裡真正的問題是，你安全地使用它們嗎？在Stormpath，我們遵循這些最佳實踐，並鼓勵我們的客戶也這樣做：

  • 將您的JWT存儲在安全的HttpOnly cookie中。這可以防止跨站點腳本（XSS）攻擊。
  • 如果您使用cookie來傳輸JWT，CSRF保護非常重要！未經用戶同意，向您的網站提出請求的其他域名可能會惡意使用您的Cookie。如果您的服務器盲目地對用戶進行身份驗證，只是因為他們有cookie，那麼您遇到的問題比硬盤驅動器大。您還允許進行CSRF攻擊，其他網站會在未經用戶同意的情況下觸發您服務器上的狀態更改操作。這是可能的，因為瀏覽器將始終自動發送用戶的cookie，無論請求是如何被觸發的。使用眾多CSRF預防措施之一來降低此風險。
  • 使用僅可用於身份驗證服務的強密鑰對您的令牌進行簽名。每次使用令牌對用戶進行身份驗證時，您的服務器必須驗證令牌是否已使用您的密鑰簽名。
  • 不要將任何敏感數據存儲在JWT中。這些令牌通常被簽名以防止操縱（未加密），因此可以容易地解碼和讀取權利要求中的數據。如果您必須在其中放入敏感的，不透明的信息，請加密您的令牌。祕密簽名密鑰只能由發行方和消費者訪問;它不應該在這兩方之外進行。
  • 如果您擔心重播攻擊，請在聲明中包含nonce（jti聲明），到期時間（exp聲明）和創建時間（ifat聲明）。這些在JWT規範中有明確定義。

JJWT，JSONWebToken.io和JWT Inspector

Stormpath支持開發幾個與JWT相關的開源開發人員工具，包括：

JJWT

JJWT是一個易於使用的工具，供開發人員用Java創建和驗證JWT。在Stormpath支持的許多庫中，JJWT是完全免費和開源的（Apache License，Version 2.0），因此每個人都可以看到它的作用以及它是如何做到的。不要猶豫，報告任何問題，建議改進，甚至提交一些代碼！

JSONWebToken.io

JSONwebtoken.io是我們創建的一個開發工具，可以輕鬆解碼JWT。將現有JWT簡單粘貼到適當的字段中以解碼其標頭，有效負載和簽名。 JSONWebToken.io由nJWT提供支持，nJWT是Node.js開發人員最乾淨的免費和開源（Apache License，Version 2.0）JWT庫。

JWT檢查器

JWT Inspector是一個開源的Chrome擴展程序，允許開發人員直接在瀏覽器中檢查和調試JWT。 JWT Inspector將在您的站點上發現JWT（在cookie，本地/會話存儲和標題中），並通過導航欄和DevTools面板輕鬆訪問它們。
