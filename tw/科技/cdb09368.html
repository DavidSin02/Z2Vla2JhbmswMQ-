使用C++ Build Insights對模板代碼進行性能分析

2020-11-14
| 科技

C++中的模板編程

在C++程序中使用模板有時會導致很長的編譯時間。C++ Build Insights可以提供一些工具，用來分析模板使用模式(template usage patterns)及其對編譯時間的相關影響。

在本文中，我們將演示如何使用vcperf分析工具和C++ Build Insights SDK來理解和修復有問題的模板模式。

我們通過一個案例研究證明了這些工具的實際使用，在該案例中，Sprout開源元編程庫的編譯時間減少了25%。我們希望本文中介紹的這些方法可以幫助你放心地進行C++模板代碼開發。

如何獲取和使用vcperf

在本文中的示例程序中，我們使用了vcperf這個工具。它可以採集編譯過程中產生的各種有用的信息，並可以在Windows Performance Analyzer(WPA)中進行可視化地查看。

最新版本的vcperf包含在Visual Studio 2019中。

1. 獲取和配置vcperf和WPA的步驟如下：

1) 下載並安裝最新版本的Visual Studio 2019。

2) 下載並安裝最新版本的Windows ADK，WPA工具包含在這個ADK中。

3) 從Visual Studio的MSVC安裝目錄下拷貝perf_msvcbuildinsights.dll到新安裝的WPA目錄下。這個文件是WPA的C++ Build Insights擴展，主要用來在WPA中正確地顯示C++ Build Insights事件信息。

a. MSVC安裝目錄：C:\Program Files (x86)\Microsoft Visual Studio\2019\{Edition}\VC\Tools\MSVC\{Version}\bin\Hostx6464

b. WPA安裝目錄：C:\Program Files (x86)\Windows Kits\10\Windows Performance Toolkit4) 在WPA目錄下打開perfcore.ini文件並添加一個對應perf_msvcbuildinsights.dll的條目。這樣WPA就會在其啟動的時候加載C++ Build Insights擴展。

2. 採集工程編譯信息的步驟如下：

1) 以管理員權限打開x64 Native Tools Command Prompt for VS 2019。

2) 採集工程編譯信息

1. 執行命令：vcperf /start /level3 MySessionName。[/level3]選項表示啟用模板事件採集。

2. 編譯你的C++工程，可以直接在Visual Studio中編譯（因為vcperf將在整個系統範圍僅限信息採集）。

3. 執行命令：vcperf /stop /templates MySessionName outputFile.etl。這條命令將會停止信息採集，同時會分析所有事件信息，包括模板事件，然後保存所有的分析結果到outputFile.etl這個文件中。

在WPA中查看模板信息

在工程編譯過程中，與模板有關的最耗時的活動是模板的實例化。C++BuildInsights提供了一個名為[模板實例化]的WPA視圖，通過這個視圖，我們可以查看程序中耗時最長的模板實例化時間。

在WPA中打開跟蹤後，可以通過將其從[圖形資源管理器]窗格拖到[分析]窗口中來打開該視圖，如下圖所示：

關於vcperf跟蹤和WPA中模板事件的說明

如果在[圖形資源管理器]窗格中看不到[模板實例化]視圖，請確保已正確完成[如何獲取和使用vcperf]部分中的WPA配置步驟，並且在啟動和停止時已將正確的參數傳遞給vcperf。

出於擴展性方面的考慮，vcperf只會在分析結果文件中記錄最耗時的模板實例化活動。如果模板實例化不是編譯時間的重要因素，則vcperf將忽略模板信息，並且不會出現[模板實例化]視圖。

案例研究：加快模板元編程庫Sprout的編譯時間

在本案例研究中，我們以一個真實開源項目為例，展示如何使用vcperf和WPA在模板元編程代碼中診斷和處理較長的編譯時間。具體來說，我們演示瞭如何使用這些工具將Sprout庫的編譯時間減少了約25%。

以下是具體的操作步驟：

1. 克隆Sprout工程到本機。

2. Checkout代碼版本：6b5addba9face0a。

3. 獲取一份Sprout完整編譯的信息：

1) 以管理員權限打開x64 Native Tools Command Prompt for VS 2019。

2）執行命令：vcperf /start /level3 Sprout。

3) 執行編譯命令：cl /std:c++latest /D_HAS_DEPRECATED_IS_LITERAL_TYPE=1 /D_SILENCE_CXX17_IS_LITERAL_TYPE_DEPRECATION_WARNING /EHsc /I. /constexpr:steps100000000 .\testspr\sprout.cpp4) 執行命令：vcperf /stop /templates Sprout sprout.etl。這條命令將會將採集到的編譯信息保存到信息跟蹤文件。

4. 在WPA中打開信息跟蹤文件。

我們打開[構建資源管理器]和[模板實例化]視圖。[構建資源管理器]視圖指示編譯持續了大約13.5秒，這可以通過查看視圖底部的時間軸(標記為A)來看出來。[模板實例化]視圖顯示了在時間8到10.5(標記為B)之間某個位置的模板實例化活動。

代碼分析

通過對代碼的分析，我們發現在源文件sprout\random\shuffle_order.hpp中，對以下類型的operator()調用觸發了對sprout::tpp::all_of的實例化。typedef sprout::random::shuffle_order_engine<sprout::random::minstd_rand0, 256> knuth_b;

這個類型在內部包含一個256個元素的數組，最終將其傳遞到sprout\container\container_construct_traits.hpp頭文件中的default_remake_container函數。

該函數具有以下三個模板定義。為了簡單起見，模板定義代碼已替換為簡單的註釋。

這些定義全部使用std::enable_if標準類型traits類來實現在特定條件啟用或禁用。你可以看出來在第二個定義的std :: enable_if條件中發現對sprout::tpp::all_of的調用嗎? 下面我們來複制它:

!(sizeof…(Args) == 2 &&sprout::tpp::all_of<sprout::is_input_iterator<typename std::remove_reference::type>…>::value)

從總體上看，如果調用default_remake_container時使用的參數數量不是2，則無需對sprout::tpp::all_of進行求值。在我們的例子中，我們有256個參數，並且知道無論sprout::tpp::all_of返回什麼條件，該條件都將為false。

在編譯器看來，這無關緊要。它在解析對default_remake_container的調用時，還是會嘗試在我們的256個參數上對sprout::tpp::all_of進行求值，從而導致非常耗時的遞歸模板實例化。

使用C++ Build Insights SDK定位模板實例化問題

在使用模板元編程的代碼庫中，遞歸，耗時的模板實例化並不是一個普遍的問題，因此我們希望將來能夠更快地識別這些問題，而不必經歷啟動WPA和手動檢查跟蹤的麻煩。幸運的是，大多數使用vcperf和WPA手動執行的分析任務也可以使用C++ Build Insights SDK以編程方式執行。

為了說明這一點，我們準備了RecursiveTemplateInspector SDK示例。它打印出構建中最耗時的模板實例化層次結構，以及有關它們的統計信息，例如遞歸樹深度，實例化總數和模板特化名稱。

讓我們重複上一節中的Sprout案例研究，但是這次使用RecursiveTemplateInspector這個示例工程。具體操作步驟如下：

1. 克隆C++ Build Insights SDK示例工程。

2. 打開Sample.Sln並選擇對應的目標平臺(x86/x64)和配置(Debug/Release)進行編譯。編譯出來的二進制文件將位於：out/{architecture}/{configuration}/RecursiveTemplateInspector

3. 使用上面案例研究中的步驟進行編譯信息收集。這一次，使用vcperf /stopnoanalyze Sprout sprout-raw.etl命令來停止收集，而不是使用/stop選項。

4. 將sprout-raw.etl作為第一個參數傳遞給RecursiveTemplateInspector二進制文件。

如下圖所示，RecursiveTemplateInspector正確地識別出導致我們出現問題的sprout::tpp::all_of模板實例化，該模板實例遞歸地觸發其他實例化，總共4043個實例化。

在經過修改後的代碼上重新運行RecursiveTemplateInspector會顯示所有有問題的模板實例均已消失，剩下的持續時間很短，可以忽略不計。

理解示例代碼

首先，通過要求C++BuildInsightsSDK將我們需要的內容轉發給OnTemplateRecursionTreeBranch和OnSymbolName函數，來過濾所有停止活動和簡單事件。

函數的名稱對C++BuildInsightsSDK如何過濾事件沒有影響。只有它們的參數很重要。具體代碼如下：

我們使用OnTemplateRecursionTreeBranch函數來一一捕獲模板實例化遞歸樹的每個分支。因為C++BuildInsights事件以堆棧表示，所以捕獲事件之間的遞歸關係是一件很容易的事情。TemplateInstantiationGroup捕獲類自動展開事件堆棧，並將解析線程中發生的所有模板實例化顯示為從根到葉有序的類似矢量的C++容器。

因為我們將OnTemplateRecursionTreeBranch函數綁定到了stop活動事件，所以我們將始終在解析線程從最深層返回的點處接收給定遞歸樹中的分支。

我們利用這一事實來計算遞歸樹的所有分支時的最大深度。一旦根實例化本身達到其停止事件，我們就通過存儲樹的總實例化時間以及發生它的轉換單元來包裝實例化樹。

模板實例化事件不包含實例化的符號的名稱。派生名稱是一項昂貴的操作，而在測量實例化時這樣做會扭曲時間測量。而是發出了一個數字鍵，稍後我們可以通過偵聽SymboName事件將其與適當的名稱匹配。OnSymbolName函數就是這樣做的，並存儲每個根模板實例化的名稱。

在分析的最後，我們遍歷所有根模板實例，按最長持續時間對其進行排序，然後顯示它們。
