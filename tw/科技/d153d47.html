詳解oracle嵌套循環及實例說明

2020-10-29
| 科技

概述

嵌套循環連接處理的兩個數據集被稱為外部循環（outer loop，也就是驅動數據源，driving row soulce ）和內部循環〔 inner loop ）。外部循環為左子節點，內部循環為右子節點。當外部循環執行一次的時候，內部循環需要針對外部循環返回的每條記錄執行一次。

處理過程

外部循環稱為外表或者驅動表，而內部循環稱為內表或者被驅動表。

嵌套循環連接處理過程

嵌套循環連接有以下幾點特徵：

口左子節點（外部循環）只會執行一次，而右子節點（內部循環）一般會執行很多次。

口在所有數據處理完之前，就可以返回結果集的第一條記錄。

口可以有效利用索引來處理限制條件與連接條件。

口支持所有類型的連接。

兩表連接

下面是個簡單的兩表嵌套循環連接的執行計劃。同時也描述瞭如何使用提示1eading 與use_nl來強制使用嵌套循環連接。Leading提示要求先訪問表tl ，也就是，它指定了哪張表作為外部循環表使用。use_nl提示指定了具體使用哪種連接方法來將內部循環返回的數據（表t2 ）與表t1 連接起來。有必要指出的是：usenl 提示並沒有引用表t1 。

SELECT /* + leading (t1) use_nl(t2) full(t1) full(t2）*/ * FROM t1 ,t2 WHERE tl.id =t2.id AND tl.n=19

NESTED LOOPS操作是一種相關聯合型操作。它意味著是第二個子節點（內部循環）的執行是由第一個子節點（外部循環）控制的。在這個例子中，執行計劃的執行過程可以總結如下。

口 通過全表掃描訪問表t1 的所有記錄，訪問中應用限制條件n = 19 過濾數據口

口 前面一個步驟返回多少條記錄，就在表t2 上執行多少次全表掃描。

無疑，當操作2 返回的記錄超過1 條時，這種執行計劃就不是很有效了，因此，查詢優化器幾乎從來不會選擇它。正是因為這個原因，才必須指定兩個訪問提示（full）來強制查詢優化器使用這個執行計劃。另一方面，如果外部循環只返回一條記錄，並且內部循環的選擇性也很高，對錶t2 進行全表掃描可能也不錯。為了展示這一點，下面我們在表t1 的n 字段卜創建一個唯一索引：

CREATE UNIQUE INDEX t1_n ON t1（n）

有了這個索引以後，前面的語句就以下面的執行計一劃執行了。由於操作3 （INDEX UNIQUE SCAN)的緣故，可以確保內部循環只會被執行一次。

SELECT /* + leading (t1) use_nl(t2) index(t1) full(t2）*/ * FROM t1 ,t2 WHERE tl.id =t2.id AND tl.n=19

如果內部循環的選擇性很低，對內部循環適應索引掃描就是一個較好的選擇。由於嵌套循環連接是相關聯合型操作，對內部循環來講，這個地方也可能利用到連接條件。例如，在下面的執行計劃中，操作5就是利用操作3的返回值t1.id做了索引查找。

SELECT /* + leading (t1) use_nl(t2) index(t1) index(t2）*/ * FROM t1 ,t2 WHERE tl.id =t2.id AND tl.n=19

通過建立2個索引，使得性能得到最好的優化。嵌套循環適合2個選擇性低的效率數據量的選擇，到數據量大的時候優化器會默認選擇hash join 連接，除非我們手動選擇提示，並且通過索引來達到很好的效率。

總的來講，如果內部循環會執行多次，只有具有好的選擇性的訪問路徑以及導致比較少的邏輯讀的路徑才有意義

四表連接

下面的執行計劃是個典型的用嵌套循環連接實現的左深樹。請注意各個表是如何通過索引進行訪問的ordered 提示要求這些表按它們在FROM 子句中的順序進行訪問。use_nl提示要求後面的表與第一張表（或者前面一個操作的結果集）進行連接的時候使用嵌套循環連接。

SELECT /* + ordered use_nl(t2 t3 t4)*/ t1.*,t2.*,t3.*,t4.* FROM t1,t2,t3,t4 WHERE t1.id = t2.t1_id AND t2.id =t3.t2_idAND t3.id =t4.t3_id AND t1.n = 19

這種執行計劃的執行流程可以總結如下(不考慮使用行預取的情況):

( 1）當讀取第一條記錄的時候（也就是說，不是當這條語句解析或者執行的時候），在應用了表t1 上的限制條件tl.n = 19 並取得第一條記錄的時候，處理過程就開始了。( 2）表t2 基於表t1 找到的結果進行查找。數據庫引擎會利用連接條件tl.id = t2.t1_id 來訪問表t2。事實上．表t2 上沒有任何限制條件。只有第一條滿足連接條件的記錄才會返回給上級操作。( 3）表t3 基於表t2 找到的結果進行查找數據庫引擎也是利用連接條件t2.id =t3.t2_id 來訪問表t3 。只有第一條滿足連接條件的記錄才會返回給上一級操作。( 4）表t4 基於表t3 找到的結果進行查找。這兒也一樣，數據庫引擎也是利用連接條件t3.id =t4.t3_id 來訪問表t4 。滿足條件的第一條記錄會立即返回給客戶端。( 5）後續的操作也是按照第一條記錄樣的行為來進行的。很明顯，執行過程是在下一次匹配（也可能是表t4 裡面匹配的第二條記錄，如果有的話）的位置開始的。需要特別強調的是，滿足條件的記錄會在第一時間被返回給客戶端。換句話講，就是完全沒有必要在返回第條記錄之前完成整個執行過程。

塊預取

在一般情況下，當緩存沒有命中的時候，基於單塊處理（例如，rowid 訪問、索引範圍掃描）的訪問路徑會導致一個單塊的物理讀。對於嵌套循環連接來講，特別是當有很多行數據需要處理的時候，效率會比較差。實際上很多時候，嵌套循環連接也會使用多個單塊物理讀來訪問多個相鄰的塊。數據庫引擎可以利用塊預取功能來提高嵌套循環連接的效率。這種優化技巧的目的是，對與多個相鄰的塊使用一次多塊物理讀取，來代替多次單塊物理讀取。塊預取對於表和索引都是適用的。無法通過查看執行計劃來看出數據庫引擎是否使用了塊預取功能。唯一能夠看到的方式是，查看服務器進程執行的物理讀，特別是與物理讀相關的等待事件。

口db file sequential read 是個與單塊物理讀取相關的事件。因此，如果這個事件出現，就意味著塊預取要麼沒有被使用，要麼是無法被使用（例如，由於請求的塊己經在高速緩存中）。

口db file scattered read 是一個與多塊物理讀取相關的事件。因此．如果在～id 訪問或者索引範圍掃描中看到這個等待事件，就表明在使用塊預取功能。

需要特別提示的是，我們無法控制塊預取功能的使用。如何以及是否使用塊預取功能是由數據庫引擎決定的。

其他可選的執行計劃可以用下面的執行計劃來執行嵌套循環連接。

實際上，在Oracle 的最近幾個版本中，只有當內部循環或者外部循環是基於唯一索引掃描（index unique scan ）的時候才會使用這種類型的執行計劃。下面我們來看看，如果在列n 上的索引t1_n 如下面這樣定義（為非唯一）時會如何：

CREATE INDEX t1_n ON t1(n)

當換成這個索引時，下面的執行計劃將被採用。請注意，表t2 上的rowid 訪問是處在不同的位置的．在前一個執行計劃，它是在操作4 ，然而在後一個執行計劃中，它是在操作1 。比較特別的是，row 記訪問（操作1）的子操作是一個嵌套循環連接（操作2 ）。從我們的角度看，這兩個執行計劃做了同一件事。下面的這個執行計劃可能是為了利用一些內部優化（比如塊預取）。

在oracle 11g 中，可能會使用下面的執行計劃，而不是之前的那個。注意，雖然查詢始終是一個兩表連接，執行計劃卻含有兩個嵌套循環連接！

oracle多結果集嵌套循環處理優化

--性能差begin for a in (select id,name,sex,idcard from people) loop for b in (select id,name,sex,idcard from english) loop if a.idcard = b.idcard then --do something end if; end loop; end loop;end;

優化後如下：

--性能優：將多表合併成一個結果集，避免嵌套循環begin for a in (select p.id,p.name,p.sex,p.idcard from people p,english e where p.idcard = e.idcard) loop --do something end loop;end;
