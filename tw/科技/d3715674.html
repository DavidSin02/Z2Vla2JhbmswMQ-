活用時序數據庫，玩轉自動化監控

2020-11-14
| 科技

InfluxDB是一款用Go語言編寫的開源分佈式時序、事件和指標數據庫，無需外部依賴。該數據庫現在主要用於存儲涉及大量的時間戳數據，如DevOps監控數據，APP metrics, loT傳感器數據和實時分析數據。

InfluxDB特徵

✔ 無結構(無模式)：可以是任意數量的列(tags)。

✔ 可以設置metric的保存時間。

✔ 支持與時間有關的相關函數(如min、max、sum、count、mean、median等)，方便統計。

✔ 支持存儲策略：可以用於數據的刪改(influxDB沒有提供數據的刪除與修改方法)。

✔ 支持連續查詢：是數據庫中自動定時啟動的一組語句，和存儲策略搭配可以降低InfluxDB的系統佔用量。

✔ 支持類似SQL語法。

✔ 支持設置數據在集群中的副本數。

✔ 支持定期採樣數據，寫入另外的measurement，方便分粒度存儲數據。

✔ 自帶web管理界面，方便使用(登入方式：http://< InfluxDB-IP >:8083)。

✔ 最關鍵的一點，也是我採用的原因，支持Grafana畫圖展示。

有了InfluxDB+Grafana後，你就可以用python寫一些簡單的程序了，可以只負責寫後端邏輯部分，數據都可以存入InfluxDB，然後通過Grafana展示出來，然後做一系列的監控工作，後面會進行詳細介紹

在具體的講解influxdb的相關操作之前先說說influxdb的一些專有名詞，這些名詞代表什麼。先看下面一段Influxdb中的表信息。
> select * from disk;
name: disk
time count device free hostname used used_percent
---- ----- ------ ---- -------- ---- ------------
1502089306183159978 1 /data 80 server01 100 60.33
1502089342879496791 1 /data 90 server02 110 68.33
InfluxDB是時序數據庫，所以怎麼都繞不開時間，第一縱列time存儲著時間戳，而時間戳是與數據進行關聯，這樣才能將時間和數據進行展示。
influxDB中的名詞

database 數據庫名，在InfluxDB中可以創建多個數據庫，不同數據庫中的數據文件是隔離存放的，存放在磁盤上的不同目錄measurement 測量指標名,相當於傳統數據庫表名tag 標籤，在InfluxDB中，tag是一個非常重要的部分，表名+tag一起作為數據庫的索引，是”key-value”的形式tag key 各種索引的keytag value 各種索引的valuetag set 由一系列的tag組成。tags 在 InfluxDB 中會按照字典序排序，不管是 tagk 還是 tagv，只要不一致就分別屬於兩個 key，例如 host=server01,region=us-west 和 host=server02,region=us-west 就是兩個不同的 tag setfield 數據，field主要是用來存放數據的部分，也是”key-value”的形式field key 各種記錄keyfield value field對應的value就是fieldName，InfluxDB中支持一條數據中插入多個fieldName，這其實是一個語法上的優化，在實際的底層存儲中，是當作多條數據來存儲field set 由一系列的key=value組成key settimestamp 時間戳，作為時序型數據庫，時間戳是InfluxDB中最重要的部分，在插入數據時可以自己指定也可留空讓系統指定,在插入新數據時，tag、field和timestamp之間用空格分隔。在 TSM 存儲引擎中會特殊對待，為了優化後續的查詢操作series 序列，所有在數據庫中的數據，都需要通過圖表來展示，而這個series表示這個表裡面的數據，可以在圖表上畫成幾條線，通過tags排列組合算出來Retention policy 數據保留策略，可以定義數據保留的時長，每個數據庫可以有多個數據保留策略，但只能有一個默認策略，InfluxDB 會定期清除過期的數據
