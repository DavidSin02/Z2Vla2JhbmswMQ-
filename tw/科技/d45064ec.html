遇事不決先求github，內功不行的算法工程師有哪些表現？

2020-11-14
| 科技

在知乎上看到討論“最差算法工程師”的話題，個人感覺作者LinT的總結很到位，大致有七類：


混跡於各類創新競賽、課程答辯，張口閉口就是「人工智能」「神經網絡」的百度百科型選手/PPT選手;


數學公式不想看、論文不想讀，輾轉各個博客網站希望找到好的解讀的博客型選手;


拿到需求，Github一搜，全克隆了再說；代碼跑通了(沒有error)，任務完成就功德無量的Github選手;


把DL當萬精油用，帶一個PPT/Github選手，參加各類創新競賽/項目，經費拿到手軟的AI+型選手(教師);


行走的AI術語詞典，討論問題時就喜歡堆砌名詞，但從不解釋，故弄玄虛，讓外行不明覺厲的名詞流選手;


走在AI潮流前列，「不加attention不行星人」，吐槽「2020年了，還用概率圖模型？！」，對新算法的狂熱程度堪比娛樂圈流量小生的潮流型選手;


半路出家，轉行做AI的，但奈何基礎弱，經常拋下「非cs科班可以學xxx嗎？」「本人xx，應不應該xxx？」問題的何去何從型選手.



在異步君看來，最差勁的算法工程師，不是算法基礎薄弱或者業務能力不強的人，而是明知自己的缺陷還瞧不起這些方面的人。


優秀的算法工程師什麼樣？


如果你有顆精進的心，那變得優秀只是時間問題。而一個優秀算法工程師的成長之路，大致要經歷三個階段。


第一階段"Operating"：會使用工具


這個層次的工程師，對常用的模型比較熟悉，來了數據以後，好歹能挑個合適的跑一下。


達到這個層次，其實門檻不高。早些年，您只要掌握了什麼叫LDA、哪叫SVM，再玩過幾次libnear、mahout等開源工具，就可以拿到數據後跑個結果出來。到了深度學習時代，這件事兒似乎就更簡單了：管它什麼問題，不都是拿神經網絡往上堆嘛！


這裡要給大家狠狠澆上一盆冷水：掌握了一堆模型並且會跑，其實並沒有什麼卵用。而算法工程師的真正價值，就是洞察問題的數據先驗特點，把他們表達在模型中，而這個，就需要下一個層次的能力了。


第二階段"Optimization"：能改造模型


這個層次的工程師，能夠根據具體問題的數據特點對模型進行改造，並採用相應合適的最優化算法，以追求最好的效果。


改造模型的能力，就不是用幾個開源工具那麼簡單了，這需要有兩方面的素養：


一、深入瞭解機器學習的原理和組件。

當你看到前輩高人把材料烹調在一起，變成LDA、CNN這些成品菜餚端上來的時候，也要想想如果自己下廚，是否瞭解食材，會不會選擇和搭配。僅僅會吃幾個菜，說出什麼味道，離好廚師差的還遠著呢。


二、熟練掌握最優化方法。

最優化，是機器學習最、最、最重要的基礎。你要知道，在目標函數及其導數的各種情形下，應該如何選擇優化方法；各種方法的時間空間複雜度、收斂性如何；還要知道怎樣構造目標函數，才便於用凸優化或其他框架來求解。而這些方面的訓練，要比機器學習的模型還要紮實才行。


在這個階段，記得主動拓寬自身的知識廣度和深度，畢竟寬度決定適應性，深度決定不可替代性。


第三階段"Objective"：擅定義問題


這個層次的工程師（似乎叫工程師不太合適了），扔給他一個新的實際問題，可以給出量化的目標函數。


在算法領域，最難的是有明確的量化目標函數，這也是區別於玄學方法、神學方法的重要標誌。


定義目標函數，初聽起來並沒有那麼困難，不就是制定個KPI麼？其實不然，要做好這件事，在意識和技術上都有很高的門檻。既要有“萬般皆下品、唯有目標高”的意識，又要讓構建的目標函數準確（信）、可解（達）、優雅（雅）。


一個團隊的定海神針，就是能把問題轉化成目標函數的那個人——哪怕他連開源工具都不會用。當你掌握了工具、會改造模型，進而可以駕馭新問題的建模，就能成長為最優秀的人才。


如果一個企業能用百萬年薪簽下這樣的人才，屬於物超所值。那繼續沿著這條路走下去，又會是怎樣的一番景象呢？


《編程珠璣》Jon Bentley就是典型


怎樣成為算法大師？有資格寫這個話題的人，要不已經名垂青史，要不就躺在你的書櫃裡。不過，異步君倒是可以介紹一位大師給你認識——Jon Bentley。



為什麼說他是大師？1983年，ACM通訊雜誌為Jon Bentley專門開闢了Programming pearls專欄，他以短文系列的形式，總結和提煉了自己在長期的計算機程序設計實踐中積累下來的寶貴經驗。受歡迎的程度頗似早年金庸先生在《明報》上連載其武俠小說的盛況。


之後，Jon Bentley 將“珍珠”系列仔細修訂和補充整理，對各篇文章的先後次序做了精心編排。在1986年，涵蓋從算法理論到軟件工程各種主題的《編程珠璣》正式誕生。當然，這些都是後話。


早年間Jon Bentley在卡耐基梅隆大學任教時，給專業程序員上課時佈置過“二分搜索”的問題。令他驚詫的是，在幾個課堂裡對一百多名程序員的檢查結果大同小異：90%的程序員都在他們的程序中發現了錯誤。


但這也在情理之中，畢竟第一篇二分搜索論文在1946年就發表了，但是第一個沒有錯誤的二分搜索程序卻直到1962年才出現。


他在1983年的論文《 Writing correct programs》，專門以二分搜索為例討論了算法的理解和其程序實現。這是個不小的挑戰，我們來看看作為算法大師，Bentley是怎麼做的。


首先介紹下二分搜索，它通過持續跟蹤數組中包含目標元素的範圍(如果存在於數組的話)來解決問題。一開始，這個範圍是整個數組；然後通過將目標元素與數組的中間項進行比較並拋棄一半的範圍來縮小範圍。該過程持續進行，直到在數組中找到或確定包含目標元素的範圍為空時為止。


在有n個元素的表中, 二分搜索大約需要執行log2n次比較操作。


問題的關鍵在於，是如果目標元素在數組中, 那麼它就一定存在於某個特定範圍之內。Bentley使用了mustbe(range)來表示，將上面描述的二分搜索轉換成下面的程序框架：

initialize range to 0..n-1loop    { invariant: mustbe(range) }    if range is empty,        break and report that t is not in the array    compute m，the middle of the range    use m as a probe to shrink the range        if t is found during the shrinking process ,        break and report its position


該程序的最重要部分是大括號內的循環不變式( loop invariant)。之所以把這種關於程序狀態的斷言( assertion )稱為不變式( invariant)，是因為在每次循環迭代之前和之後，該斷言都為真。這個名稱將前面已有的直觀概念形式化了。


斷言：輸入、程序變量和輸出之間的關係勾勒出了程序的“狀態”，斷言使得程序員可以準確闡述這些關係。


經過不斷優化，最終的函數只有9行代碼和一個不變式斷言：

l = 0;u=n-1loop         { mustbe(l, u) }        if l> u            p = -1; break        m=(l+u)/2    case        x[m] < t:l=m+1        x[m] == t:P = m; break        x[m] > t:u =m-1

感興趣的讀者可以自己嘗試寫一個二分搜索程序，看看你是不是那10%（可在評論區晒出來）


來自一顆珍珠的啟發


算法大師做的事，是提供一種思考方式，它不僅適用於編程，還適用於其它領域。


《編程珠璣》被很多牛人推崇，不僅是Jon Bentley的名號，比如Java之父James Gosling的老師。儘管這是本主要講算法和程序設計的書，但更重要的是其思考方式。比如基礎部分，用大部分人能聽得懂的話來複述，就是如下這樣：

1. 搞清楚需求

2. 設計算法

3. 根據數據選用合適的數據結構

4. 寫代碼

5. 調試測試


對於以上每一個環節，可以怎麼做，能做成什麼樣子，這本書闡述的可謂是淋漓盡致。稍微有些許年份開發經驗的人，回過來總結，無非就是以上那些。玩出花來，也逃不過那五個部分。細細品味，這本書真是一本『授人以魚不如授人以漁』的書籍。


至於說這本書提供的思考方式不僅適用編程，也適用於其它領域，其實這也是顯而易見的，例如：搞清楚問題，就單說這一點，在任何領域做事，不都是這樣嗎？


不用編程術語來複述以上五個步驟，應該就是如下這個樣子了：

1. 搞清楚問題

2. 思考解決問題的方法和思路

3. 選用合適的工具

4. 執行

5. 優化迭代


如果想看提高編程技能，想研究算法和數據結構這些內功，還是建議加大看看《編程珠璣》，有些人說它簡單，我倒不覺得，我覺得它還是需要一些基礎的。比如第一章涉及到的知識點有：歸併排序（可用作一種外部排序方式）、快速排序、位圖、位向量、時間複雜度分析、空間複雜度分析、計算機存儲數據大小估算等等。


Bentley想表達的是，僅僅熟練地運用這些可靠的工程原理，不見得一定能夠如期完成軟件並順利運行。在可靠的工程之外，通過洞察力和創造力結晶而出的編程珠璣，這也是計算機編程有魅力的地方。
