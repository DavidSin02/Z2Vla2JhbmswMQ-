網絡故障的檢修思路

2020-10-29
| 科技

網絡問題往往是很獨特的，且有時是難以解決的。故障檢修要對付一些不希望出現的事情，通常需要的只是掌握概念性的知識， 而不是為配置網絡所需的細節的知識。 要想正確、順利地解決問題，就需要清楚的瞭解整個局域網，瞭解網絡的佈線、電氣環境、 TCP/IP 如何在網絡上、在單個主機、在協議棧的各層之間為數據選擇路由等。對於檢修網絡，這些是非常重要的。


在檢修網絡故障的工作中，下面的一些提示是很有用的：

1、不要忽略顯而易見的東西。網絡電纜鬆動是很常見的問題，應檢查插頭、連接器、 電纜、集線器和開關等，小事情可能引起大問題。

2、大多數問題是由人為因素（錯誤）造成的，通過提供網絡配置和作用信息或提供這方面的培訓，可以杜絕其中的大部分錯誤。

3、要注意解決問題的方式方法，應利用每次測試時收集到的信息去指導你的測試，如果不能確保你所選擇的原始測試環境，就千萬不要根據主觀臆斷轉移到另一個測試環境中。

4. 應廣開思路、靈活變通，不要認為問題的原因太多，不要認為在應用程序級發現的問題就不是下一級引起的。 有些人總認為網絡有故障， 而另一些人總認為遠程端有問題， 某些人如此肯定他們知道問題的原因， 以致不管測試的結果。 千萬不要重蹈這些覆轍， 應測試每一種可能的情況，根據測試結果決定你的行動。

5、採用幾種簡單的故障檢修工具。 對於大多數的 TCP/IP 軟件問題， 用幾種簡單的工具就足以解決問題， 花些時間學習如何使用新的檢修工具是值得的。 由於很多網絡問題的原因都很簡單， 因而對問題有一個清晰的瞭解往往就可能找到答案。 遺憾的是情況並不總這這樣！ 下面介紹幾種簡單的工具，可以幫助你去攻克最難解決的問題。


Ping ：這個命令工具在 Linux/Unix 、 Dos、 Windows 9x 、 Windows NT 等系統下都可以找到。

這一工具可以測試你的系統是否能到達一臺遠程的主機， 這一簡單的功能對於測試網絡的連接是非常有用的， 而與在其中檢測到問題的應用程序無關。 Ping 允許你下一步是測試網絡連接層 （較低層）還是應用程序層（較高層）。如果 ping 顯示分組報文可以到遠程系統並返回，用戶的問題就可能在較高層中；如果分組報文不能返回傳送，那麼故障可能在較低的協議層或物理層中。

利用用戶的主機名或 IP 地址， 可先對遠程主機使用 ping 命令； 如果執行成功， 就由用戶對該主機使用 ping 命令；如果也執行成功，那麼就應集中精力去分析用戶遇到問題的那個應用程序。

如果你的 ping 命令執行成功， 而用戶的 ping 命令卻失敗， 就可以集中測試該用戶的系統配置文件，以及將你和用戶到該遠程主機的路徑進行比較，找到它們的不同之處。

如果你和用戶的 ping 命令都失敗了， ping 命令顯示的出錯信息是很有幫助的，可以指導你進行下一步的測試計劃。以下是幾種基本的出錯類型： unknow host 該遠程主機的名字不能被DNS（域名服務器）轉換成 IP 地址， DNS可能出故障、該名字可能是不正確的、你的系統和遠程服務器之間的網絡可能出毛病。 如果你知道該遠程主機的 IP 地址， 可以再試一試 ping 命令。 如果利用它的 IP 地址能達到該主機，問題就可能出在 DNS上。

Network unreachable 本地系統沒有到達該遠程系統的路由。如果在 ping 命令中使用 IP地址， 則利用主機名重新輸入 ping 命令， 這就消除了輸入不正確 IP 地址的可能性。 如果使用路由選擇協議，一定要確保它正在運行，並使用 nestat 去檢查其路由表。

no answer 遠程系統沒有響應。大多網絡實用程序都有不同形式的類似信息，某些系統的ping 命令可打印成 100% packet loss, telnet 打印成 connection timeout 。所有的這些錯誤信息都說明同一個問題： 本地系統有一條到達遠程系統的路由， 但它卻接收不到它發給該遠程系統的任何分組報文響應。 這種問題的原因有很多， 遠程主機可能沒有工作、 本地或遠程主機可能配置不當、 本地和遠程主機之間的線路不正常等等。 只有用其它的方法才可以隔離該問題的原因。


廢了一大堆口舌， 終於“ UNPING”的原因道出個頭緒來了， 更加高深的我也不懂了， 請多多包涵。下面我們繼續看看 ping 上了後它所顯示的內容說明了什麼。

ping 命令的基本格式是： ping host [packetsize][count] （不同的系統，格式略有不同）host 是被測試遠程主機的主機名或 IP 地址。 Packetsize 是可選項，定義測試分組報文的長度（Byte ），只有在使用 count 字段是才使用。 卻省的分組長度是 56Byte ，count 是測試時發送的分組數。 Count 一般設置為較低值，一般設為 4 或 5 就可以了。

測試從 Link 上能否到達 ly ，用以下命令： C:>ping ly Pinging ly [222.222.222.15] with 32 bytes of data:Reply from 222.222.222.15:bytes=32 time=1ms TTL=128

Minimun = 1ms, Maximum = 1ms, Average = 1ms 該測試表明 link 到 ly 的連接非常正常，沒有丟失分組，響應很快。在 link 到 ly 之間的往返平均只需 1 毫秒。對於局域網的連接，分組丟失越少和往返時間越小則越正常。 TTL（ Time To Live ） =128。該 ping 命令顯示的統計信息可以指出下一級的網絡問題。 其關鍵的統計信息是： * 一個分組往返傳送需要多長時間， 它顯示在 time= 之後。

* 分組丟失的百分比。它顯示在 ping 輸出結束處的總統計行中。

* 分組到達的次序。如每個分組的 ICMP 序號（ icmp_seq ）。

如果分組丟失率高、 響應時間非常慢， 或者各分組不按次序到達， 那麼就有可能是硬件有毛病；當然，如果這些情況發生廣域網上就不必擔心太多。

在本地網絡中， 其往返時間約等於 1 毫秒，幾乎或完全沒有分組丟失， 各分組應按次序到達。

如果不是這樣， 那麼網絡硬件就有問題。 在以太網中可能是由於： 不合適的電纜端接 （終端電阻）、不良電纜、不良的有源硬件（ Hub等）。首先要檢查電纜的端接， It's so easy ，看看有沒有終端電阻。 這是一個最常見最簡單的檢測電纜的方法是用一個萬用表測量某一端口的電阻值。 如果測得端口電阻為零，說明電纜短路，為無窮大則斷路， 50 歐左右說明有一個終端電阻脫落，正常應在 25 歐左右。

一個簡單的 ping 測試的結果，即使該測試順利通過，也能指導你做進一步的測試，幫助你找到最可能發生問題的地方。 但是要深入檢查問題， 並找到潛在的原因， 還需要其它的診斷工具。

最後再簡單介紹幾個有用的工具： netstat 可提供各種各樣的信息， 通常用來顯示每個網絡接口、網絡插口、網絡路由表等的詳細統計資料。

arp 提供有關以太網 IP 地址轉換的信息， 它可以用來檢測本地網絡中， 用錯誤的 IP 地址配置的系統。

Ifconfig 提供接口的基本配置信息。 它對於檢測不正確的 IP 地址、 子網掩碼和廣播地址是很有用的。
