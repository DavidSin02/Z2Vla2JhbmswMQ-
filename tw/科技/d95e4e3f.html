MySQL進階之InnoDB事務原子性實現原理

2020-11-14
| 科技

概述

事務（Transaction）是訪問和更新數據庫的程序執行單元；事務中可能包含一個或多個語句，這些語句要麼都執行，要麼都不執行，事務是保證數據一致性的重要手段。事務具有4個屬性，就是原子性（Atomicity）、一致性（Consistency）、隔離性（Isolation）、持久性（Durability），簡稱ACID屬性。MySQL的事務是由存儲引擎實現的，支持事務的數據庫引擎包括InnoDB、NDB Cluster等，其中InnoDB的使用最為廣泛。下面以InnoDB為基礎進行描述，剖析InnoDB實現事務特性的原理。

事務提交與回滾

首先，回顧以下在MySQL中顯式使用事務的方式

-- 開啟事務start transaction;...-- 一條或多條DML或DDL語句select | insert | update | delete...-- 提交或回滾commit | rollback;

MySQL的InnoDB存儲引擎默認是自動提交事務，可以通過下面的設置和查看

-- 關閉自動提交set autocommit = 0;-- 開啟自動提交set autocommit = 1;-- 查看自動提交變量狀態show variables like 'autocommit';

在自動提交模式下，如果沒有start transaction顯式地開始一個事務，那麼每個sql語句都會被當做一個事務執行提交操作。

在MySQL中，存在一些特殊的命令，如果在事務中執行了這些命令，會馬上強制執行commit提交事務務，比如如DDL語句create table、drop table、alter table、lock tables語句等。對於常用的select、insert、update和delete命令，則不會強制提交事務。

原子性實現原理

原子性是指一個事務是一個不可分割的工作單位，其中的操作要麼都做，要麼都不做；如果事務中一個語句執行失敗，則已執行的語句也必須回滾，數據庫退回到事務前的狀態。InnoDB主要基於回滾日誌實現事務的原子性。

  • 隱藏字段

在InnoDB中，每一行數據除了包括我們設計的字段之外，還包含了一些內部字段，也叫隱藏字段。比如在InnoDB的聚簇索引記錄中會包含3個隱藏列row_id、trx_id、roll_pointer等

row_id

數據行id，用於標識一行數據。row_id並不是必要的，如果創建的表中有主鍵或者非NULL唯一鍵時都不會包含row_id列

trx_id

事務id，每次對某數據記錄進行修改時，都會把對應的事務id賦值給trx_id列。每次事務操作都會分配一個事務id，它是一個自增id。

roll_pointer

當前數據記錄的上一個版本的指針。每次對某條數據記錄進行改動時，都會把舊版本數據記錄按照一定格式寫入到回滾日誌（undo log） 中，而roll_pointer列則保存了該舊版本數據記錄在回滾日誌中的位置，相當於一個指針。

  • 回滾日誌（undo log）與版本鏈

    • MySQL架構設計當中，其有一個完整的日誌體系，包括二進制日誌、通用查詢日誌、慢查詢日誌等，為排查問題、性能優化、實現事務控制等提供了強大的支撐。在這些日誌當中，和事務直接相關的日誌有兩個，一個是重做日誌（redo log），一個是回滾日誌（undo log）。其中的回滾日誌，是事務原子性和隔離性實現的基礎。

    回滾日誌分為兩類，一類是插入回滾日誌（insert undo log），一類是更新回滾日誌（update undo log）。插入回滾日誌由insert操作產生的，在事務提交後可以被立即丟棄；更新回滾日誌則是由update和delete操作產生的，更新回滾日誌不僅在事務回滾時使用，在InnoDB實現事務隔離性時也需要用到，所以不能隨便刪除，只有在快速讀或事務回滾不涉及該日誌時，對應的日誌才會被purge線程統一清除。

    比如以下三條語句

    insert into user (id,name) values (1,'蕃豆咖啡');start transaction;update user set name = 'coffee' where id = 1;update user set name = '最新的名字' where id = 1;commit;

    回滾日誌和版本鏈

    每次對數據記錄進行修改時（事務操作），都會產生一條回滾日誌 ，每條回滾日誌也都有一個roll_pointer屬性（insert操作對應的回滾日誌沒有該屬性，因為該記錄並沒有更早的版本），隨著更新次數的增多，所有的版本都會被roll_pointer屬性連接成一個鏈表，這個鏈表稱之為版本鏈。版本鏈的頭節點就是當前記錄最新的值。注意，每個版本中還包含生成該版本時對應的事務id，這個信息很重要，是InnoDB實現事務隔離性的關鍵部分。

    • 實現原理

    實現原子性的關鍵，是當事務回滾時能夠撤銷所有已經成功執行的sql語句。當開啟一個事務對數據進行增刪改時，InnoDB會生成一條對應的回滾日誌（修改前的數據信息）到回滾日誌緩衝中。如果事務執行失敗或調用了rollback命令，導致事務需要回滾時，InnoDB就會根據回滾日誌中的信息，執行與當前事務相反的工作，將數據回滾到增刪改之前的狀態。

    InnoDB的回滾實現原理，簡單理解，就是在需要回滾事務時，InnoDB根據回滾日誌的信息，對於每個insert語句，執行一條對應的delete語句；對於每個update語句，會執行一條更新為原值的update語句；對於每條delete語句，會執行一條對應的insert語句。

    這樣，通過回滾日誌，InnoDB可以在任何需要回滾的地方將相關數據回滾到事務前的狀態。

    • 回滾日誌示例

    比如創建一條新的用戶記錄

    insert into user (id,name) values (1,'蕃豆咖啡');

    當需要回滾時，InnoDB將根據回滾日誌會生成一條刪除語句並執行

    delete from user where id = 1;

    再比如更新該用戶名字

    update user set name = 'coffee' where id = 1;

    當需要回滾時，InnoDB將根據回滾日誌會生成一條更新語句並執行

    -- InnoDB根據update語句所更新的主鍵id、更新列name、列更新前內容（蕃豆咖啡）-- 生成對應的回滾日誌update user set name = '蕃豆咖啡' where id = 1;

    如要刪除該用戶記錄時

    delete from user where id = 1;

    當需要回滾時，InnoDB將根據回滾日誌會生成一條插入語句並執行

    insert into user (id,name) values (1,'蕃豆咖啡');

    總結

    InnoDB實現回滾的關鍵是回滾日誌。回滾的過程比上面描述的會複雜一些，比如在併發情況下，多個事務之間，回滾時需要區分當前事務可見的回滾日誌，然後再根據回滾日誌進行回滾，這實際上已經屬於事務隔離性的範疇了，這裡暫不深入探討。
