算法連載之散列表

2020-11-14
| 科技

散列表

散列表（hash table）是實現字典操作的有效數據結構。在最壞情況下，查找一個元素的時間複雜度是O(n)；而在合理假設情況下，查找一個元素的時間複雜度是O(1)。因此，散列表查找元素的性能是極好的。

常用於作為在常量時間範圍內輔助查找一個元素。

對於普通數組，可以直接尋址，即可以直接通過數組下標訪問一個元素。如果空間允許的話，可以將元素的key作為數組下標映射。但是，如果實際存儲的元素要比實際可存儲的元素小很多時，仍使用直接尋址法，會帶來很大的空間浪費。主要缺點：

1、空間浪費；

2、一次申請龐大的空間，內存容量限制。

因此，散列表是作為對普通數組的擴展。在散列表中，不是通過關鍵字作為數組下標，而是根據關鍵字來計算出數組下標。如果多個關鍵字計算出同一個下標，就會出現“衝突”。所以，在理想情況下，如果完全可以避免衝突的話，散列表的元素查找時間就是O(1)。

在直接尋址方式下，元素是放在關鍵字為K的槽中；而散列方式下，元素是放在f(K)的槽中，其中，函數f()稱為散列函數，通過散列函數來計算出槽的位置。

散列函數

  • 除法散列法

f(k) = k % m

其中，k是關鍵字，m是所映射的槽數。假設：m = 10，k = 99，則f(99) = 99 % 10 = 9

m不要取2的冪。為什麼？

假設k是0~1000範圍內數值，m為8，即2的3次冪。

96 % 8 = 0110 0000 % 2³ = (1*2³*2³ + 1*2²*2³) % 2³ = 0

97 % 8 = 0110 0001 % 2³ = (1*2³*2³ + 1*2²*2³ + 1*2º) % 2³ = 1*2º= 1

98 % 8 = 0110 0010 % 2³ = (1*2³*2³ + 1*2²*2³ + 1*2¹) % 2³ = 2¹= 2

99 % 8 = 0110 0011 % 2³ = (1*2³*2³ + 1*2²*2³ + 1*2¹ + 1*2º) % 2³ = 2¹+ 2º = 3

100 % 8 = 0110 0100 % 2³ = (1*2³*2³ + 1*2²*2³ + 1*2²) % 2³ = 2² = 4

101 % 8 = 0110 0101 % 2³ = (1*2³*2³ + 1*2²*2³ + 1*2² + 1*2º) % 2³ = 2² + 2º = 5

102 % 8 = 0110 0110 % 2³ = (1*2³*2³ + 1*2²*2³ + 1*2² + 1*2¹) % 2³ = 2² + 2¹ = 6

103 % 8 = 0110 0111 % 2³ = (1*2³*2³ + 1*2²*2³ + 1*2² + 1*2¹+ 1*2º) % 2³ = 2² + 2¹ + 2º = 7

104 % 8 = 0110 1000 % 2³ = (1*2³*2³ + 1*2²*2³ + 1*2³) % 2³ = 0

105 % 8 = 0110 1001 % 2³ = (1*2³*2³ + 1*2²*2³ + 1*2³ + 1*2º) % 2³ = 2º = 1

以上分析可知，餘數在0~7範圍內，不會超過8，也就是k的後三位就可以完全表示餘數。如果關鍵字均勻分佈在後三位，此方法可行。但如果遇到：

0000 0000（0）

0000 1000（8）

0001 0000（16）

0001 1000（24）

如上後三位均為0的情況，這些關鍵字都處於同一個槽中，無法很好的避免衝突。所以，好的散列函數為了儘可能的避免衝突的發生，應該儘量考慮所有位。

一個不接近2的冪次的素數是一個好的選擇。

假設一個存放2000個字符的hash表，可以接受的衝突是3次，則2000/3 ≈ 667，m的選擇是接近667，但不接近2的冪次的素數，因此m可以選擇為673，或677，或683，或691，或701。

  • 乘法散列法

f(k) = ⎣m(kA % 1)⎦

其中：

1、0<A<1

2、kA % 1得到的是kA結果的小數部分

3、m是槽數，一般取2的p次

假設k=123_456，p=14，m=2^14=16_384，w=32（每個關鍵字佔32位）

∵ 0<A<1，每個關鍵字佔32位，2^32=4.294967296E9

∴ A=s / 2^32，其中 0 < s < 2^32，當A ≈(√5 - 1)/2時，是個比較理想的值

s = A * 2^32

s=2.6544357694972305E9

∵ k和s都是32位

∴ ks = r1 * 2^32 + r2 <1>

kA = ks / 2^32，相當於將ks右移32位 <2>

∵ kA % 1 取小數部分

∴ 其中r2即為小數部分 <3>

∵ m(kA % 1)，m=2^14

∴ 即將r2左移14位 <4>

r2的高14位即是f(k)的結果，f(123_456) = 67

解決衝突

  • 鏈接法

把散列在同一個槽中的所有元素都放在一個鏈表中。

1、插入最壞情況時間複雜度是O(1)，因為在發生衝突時，始終插入到鏈表頭位置；

2、查找平均情況是O(1)，最壞情況是O(n)，即所有元素都散列到同一個槽中。

假設有n個元素，放到m個槽位中，則α=n/m，α稱為裝載因子。在簡單均勻散列假設下：

1）一次不成功查找平均時間為O(1+α)；

2）一次成功查找平均時間為O(1+α)；

3、刪除，對於雙向鏈表最壞情況O(1)，而對於單向鏈表刪除和查找漸進相同。

全部字典操作都可以在平均情況O(1)時間內完成。

  • 開放尋址法

所有元素都放在散列表中，當插入元素髮生衝突時，通過線性探查（二次探查/雙重散列）將元素放入到槽位為NULL的位置。查找和插入都可以在線性時間O(1)內完成。

開放尋址法不需要額外的空間存儲衝突元素，而是通過計算出存放的槽位提高檢索速度，從而提高了空間的利用率。

α絕對不會超過1，因為當散列表被填滿時，不能再插入任何元素，此時n和槽數m相等。

假設有n個元素，放到m個槽位中，則α=n/m<1。在簡單均勻散列假設下：

1）一次不成功查找平均時間為1/(1-α)；

2）一次成功查找平均時間為(1/α)㏑(1/(1-α))；

1、線性探查

h(k, i) = (h'(k) + i) % m

1）i = 0, 1, 2, ... m-1

2）共有m中不同的探查序列

3）會造成一次群集（重度），連續槽被佔用（查找是順序自增），相應的查找時間也會相應增加。

2、二次探查

h(k, i) = (h'(k) + c1i+c2i²) % m

1）i = 0, 1, 2, ... m-1

2）共有m中不同的探查序列

3）會造成二次群集（輕度），當初始位置相同時，探查序列也就相同了。

3、雙重散列

h(k, i) = (h1(k) + ih2(k)) % m

1）開放尋址的最好方法之一，所產生的探查序列具有隨機性。

2）為了可以探查整個散列表，要求h2(k)同m必須互素（公約數為1）

a、m是2的冪次，h2(k)總返回基數

b、m是素數，h2(k)總返回比m小的正整數

  • 完全散列（perfect hashing）

能夠在O(1)最壞情況時間內完成關鍵字的查找。要求關鍵字集合是靜態的，各個關鍵字一旦進入散列表，就不會再變化了。如：cd-rom的文件名，程序設計語言的關鍵字。

Java集合框架中的HashSet和HashMap

HashSet

1、基於Hash表，實際上是基於HashMap實例。每一個元素對應HashMap的key。

2、無序（基於hash表），元素不可重複。

3、對於基本操作，如add、remove、contains、size提供了時間複雜度是O(1)的性能。對於iterate，同Hash表的容量相關，因此為保證性能，初始容量不要太大或負載因子不要太小。

HashMap

1、與Hashtable大致一致，除了它是不同步、key和value接受null值。

2、無序，特別是不能保證一直保持不變。

3、對於基本操作，如get、put提供了時間複雜度是O(1)的性能。對於iterate，同Hash表的容量相關，因此為保證性能，初始容量不要太大或負載因子不要太小。

4、有兩個參數影響其性能

1）初始容量，Hash表的槽數。

2）負載因子，一個度量散列表在自動增加其容量之前被允許達到的容量佔比的程度。當哈希表的長度超過了負載因子和當前的容量的乘積，哈希表會重建（重新散列），最後的容量是原來容量的兩倍。

5、默認情況容量是16，負載因子是0.75。

6、HashMap使用鏈接法解決衝突（相同hash值），當鏈表長度大於等於TREEIFY_THRESHOLD（默認為8）時，將鏈表轉換為紅黑樹，當小於等於UNTREEIFY_THRESHOLD（默認為6）時，又會將紅黑樹轉換為鏈表以達到最佳性能。
