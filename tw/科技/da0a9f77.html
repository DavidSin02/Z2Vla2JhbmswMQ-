Linux學習第23節，什麼是中斷？從C源碼分析內核如何設計和實現它

2020-10-29
| 科技

前面幾節，我們主要分析和討論了 Linux 內核中常用的幾種數據結構（鏈表、隊列、映射、二叉樹、紅黑樹）以及相關C語言代碼實現，這是研究 Linux 內核其他內容的基礎，本節將介紹一下內核中的“中斷”機制。

中斷的基本概念

現在來設想一個簡單，但是又很常見的場景：當用戶按下鍵盤時，Linux 內核是如何響應的呢？如果採取始終監測鍵盤的方法，效率就太低了，因為用戶可能幾分鐘，甚至幾小時都不會按下鍵盤。就算用戶正在使用鍵盤寫文章，手指的速度相對於 cpu 的速度也差了好幾個數量級。

cpu 每秒運算次數以億計，但是用戶能一秒按鍵盤 100 次嗎？

那麼內核該如何響應鍵盤的輸入，而又同時不會降低機器的整體性能呢？如果鍵盤在被敲擊的時候，能夠主動向 Linux 內核發送信號，就好辦了。這樣 Linux 內核就無需關心鍵盤，安心做自己的事就可以了，反正若是有人按下鍵盤，鍵盤會主動通知內核的。這其實就是中斷機制。

中斷本質上是一種電信號，由硬件設備（如鍵盤鼠標）產生，並直接送到中斷控制器的輸入口，經複用處理後，中斷控制器會向 cpu 發送一個電信號。cpu 接收到該信號後，會中斷自己當前的工作，並通知 Linux 內核，這時內核就能做出適當的處理了（例如處理鍵盤輸入的字符）。

現代計算機一般都較為複雜，硬件外設也非常多，不僅僅有鍵盤，還有聲卡，顯卡，硬盤等。這麼多硬件設備都產生中斷的話，Linux 內核怎麼知道某個中斷是誰產生的呢？所以，內核為每個中斷都設定了一個唯一的數字標誌（中斷號）。這麼一來，有中斷產生時，只需對比中斷號就能知道是哪個硬件設備產生的了。

這裡再提一下“異常”的概念。中斷是由硬件主動產生的，它才不會管 cpu，隨時可能發生。而異常則是由 cpu 主動產生的，一般是遇到無法處理，必須求助內核的情況時才會產生，例如某個數被 0 除。

Linux 內核中的中斷處理程序

上面我們提到，Linux 內核為了分辨中斷是哪個硬件設備產生的，為每個中斷都定義了獨一無二的中斷號，其實，內核還為每個中斷都定義了中斷處理程序。中斷處理程序運行在中斷上下文中，而不是進程上下文中，這就要求中斷處理程序應儘可能簡潔，以便能快速完成工作，儘快恢復中段代碼的執行。

但有時需求是矛盾的，即：想中斷處理程序運行的快，又想中斷處理程序幹得活多。例如網絡設備的中斷處理程序，它需要對硬件做出應答，把網絡數據從硬件設備拷貝到內存，還要把數據送往合適的協議棧分析，這樣的工作量顯然不小，但是又要求系統快速做出響應的同時，不影響其他設備。

針對這種情況，Linux 內核將中斷處理程序分為兩部分（上半部，下半部），上半部只做必須快速完成的工作，例如對硬件設備做出應答，下半部則負責完成餘下的較為耗時的工作。上半部的工作一般都是“現在立刻馬上做”，而下半部的工作則可以暫緩到 cpu 沒那麼忙的時候再做。

現在知道了 Linux 內核關於中斷的總體設計了，來看看具體的 C語言代碼是怎樣實現的吧。

註冊中斷處理程序

每一箇中斷都對應一箇中斷處理程序，那麼這個“對應”關係 Linux 內核是如何實現的呢？上面我們提到，每個中斷都有獨一無二的中斷號，所以將中斷處理程序與中斷號對應起來可以了。因此，Linux 內核定義了 request_irq() 函數用於註冊中斷處理程序，它的C語言代碼如下，請看：

 536 int request_irq(unsigned int irq, irq_handler_t handler, 537 unsigned long irqflags, const char *devname, void *dev_id)- 538 {| 539 struct irqaction *action;| 540 int retval;  ...|| 586 local_irq_save(flags);|| 587 handler(irq, dev_id);|| 588 local_irq_restore(flags);|| 589 }| 590 #endif| 591 | 592 retval = setup_irq(irq, action);| 593 if (retval)| 594 kfree(action);| 595 | 596 return retval;| 597 }

其中 irq 是分配給中斷的中斷號，這個值有的設備是預先設定好的，有些則可以通過C語言編程動態獲取。handler 就是指向中斷處理函數的指針了，它的原型如下：

typedef irqreturn_t (*irq_handler_t)(int, void *);

irqflags 是標誌位，常用的主要有以下幾個：

  • IRQF_DISABLED，使用該標誌位註冊的中斷處理程序運行時，會禁止其他中斷。
  • IRQF_SAMPLE_RANDOM，Linux 內核維護一個熵池，用於實現隨機功能，該標誌位表明中斷會對熵池有貢獻。
  • IRQF_SHARED，每個中斷線都可以有不止一箇中斷處理程序，此標誌位則決定是否使用多箇中斷程序複用一條中斷線。

request_irq() 函數的 devname 參數則是用於描述中斷名字的，dev_id 在使用了 IRQF_SHARED 標誌位時非常有用，它是區分共享一箇中斷線的中斷處理程序們的重要參數，這一點從 request_irq() 函數的C語言源代碼可以看出：

| 554 if ((irqflags & IRQF_SHARED) && !dev_id)| 555 return -EINVAL;| 556 if (irq >= NR_IRQS)| 557 return -EINVAL;| 558 if (irq_desc[irq].status & IRQ_NOREQUEST)| 559 return -EINVAL;| 560 if (!handler)| 561 return -EINVAL;

如果沒有使用 IRQF_SHARED 位，dev_id 參數可以賦值為 NULL。

request_irq() 函數的其他部分就比較簡單了，它的主要工作就是保存傳遞給它的參數，請看如下C語言代碼：

Linux 內核是如何執行中斷處理程序的

內核中大多中斷處理程序都是在 do_IRQ() 中完成執行的，它的 C語言源代碼如下：

 69 unsigned int do_IRQ(struct pt_regs *regs)- 70 { | 71 struct pt_regs *old_regs;| 72 /* high bit used in ret_from_ code */| 73 int irq = ~regs->orig_ax;| 74 struct irq_desc *desc = irq_desc + irq;...| 143 irq_exit();| 144 set_irq_regs(old_regs);| 145 return 1;| 146 }

能夠看出，do_IRQ() 函數是從全局變量 irq_desc 獲取中斷信息的，irq_desc 的內容則來自 request_irq() 函數。這就非常清楚了，Linux 內核通過 request_irq() 函數註冊中斷處理程序，並將相關信息保存在 irq_desc，do_IRQ() 函數則負責從 irq_desc 中獲取中斷處理程序，在合適的時候執行。

從上圖中的C語言代碼可以看出，do_IRQ() 函數首先把當前的寄存器信息保存下來，然後檢查棧空間是否夠用。然後確保進入了中斷棧，否則就手動新建一個棧幀供中斷處理程序使用。Linux 內核執行中斷處理程序是通過在註冊中斷處理程序時傳遞給 irq_desc 結構體成員 handle_irq 的函數指針實現的。這一過程可以從下圖C語言代碼看出。

Linux 內核是如何釋放中斷處理程序的

既然有註冊，就應該有釋放。當 Linux 卸載驅動程序時，需要註銷相應的中斷處理程序，並釋放中斷線，這一過程由 free_irq() 函數完成，它的C語言代碼原型如下：

void free_irq(unsigned int irq, void *dev_id);

free_irq() 函數會根據 dev_id 遍歷整個 irq_desc，查找到要釋放的中斷處理程序後，就將其釋放，並且設置 IRQ_DISABLED 標誌位。

不過對於以 IRQF_SHARED 標誌位註冊的中斷處理程序，還要特殊處理一下：

if (action->flags & IRQF_SHARED) { local_irq_save(flags); action->handler(irq, dev_id); local_irq_restore(flags);}

至此，Linux 內核是如何註冊中斷處理程序，又是如何執行如何釋放的，相信大家已經比較清楚了。
