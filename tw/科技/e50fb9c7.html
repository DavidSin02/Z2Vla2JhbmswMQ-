1.1 什麼是Fragment Shader(片段著色器)？

我們把 shaders 和古騰堡印刷術相提並論。為什麼這樣類比呢？更重要的是，什麼是 shader？

如果你曾經有用計算機繪圖的經驗，你就知道在這個過程中你需要畫一個圓，然後一個長方形，一條線，一些三角形……直到畫出你想要的圖像。這個過程很像用手寫一封信或一本書 —— 都是一系列的指令，需要你一件一件完成。 Shaders 也是一系列的指令，但是這些指令會對屏幕上的每個像素同時下達。也就是說，你的代碼必須根據像素在屏幕上的不同位置執行不同的操作。就像活字印刷，你的程序就像一個 function（函數），輸入位置信息，輸出顏色信息，當它編譯完之後會以相當快的速度運行。

1.2 為什麼 shaders 運行特別快？

為了回答這個問題，不得不給大家介紹並行處理（parallel processing）的神奇之處。 想象你的 CPU 是一個大的工業管道，然後每一個任務都是通過這個管道的某些東西 —— 就像一個生產流水線那樣。有些任務要比別的大，也就是說要花費更多時間和精力去處理。我們就稱它要求更強的處理能力。由於計算機自身的架構，這些任務需要串行；即一次一個地依序完成。現代計算機通常有一組四個處理器，就像這個管道一樣運行，一個接一個地處理這些任務，從而使計算機流暢運行。每個管道通常被稱為線程。

CPU視頻遊戲和其他圖形應用比起別的程序來說，需要高得多的處理能力。因為它們的圖形內容需要操作無數像素。想想看，屏幕上的每一個像素都需要計算，而在 3D 遊戲中幾何和透視也都需要計算。 讓我們回到開始那個關於管道和任務的比喻。屏幕上的每個像素都代表一個最簡單的任務。單獨來看完成任何一個像素的任務對 CPU 來說都很容易，那麼問題來了，屏幕上的每一個像素都需要解決這樣的小任務！也就是說，哪怕是對於一個老式的屏幕（分辨率 800x600）來說，都需要每幀處理480000個像素，即每秒進行14400000次計算！是的，這對於微處理器就是大問題了！而對於一個現代的 2800x1800 視網膜屏，每秒運行60幀，就需要每秒進行311040000次計算。圖形工程師是如何解決這個問題的？

這個時候，並行處理就是最好的解決方案。比起用三五個強大的微處理器（或者說“管道”）來處理這些信息，用一大堆小的微處理器來並行計算，就要好得多。這就是圖形處理器（GPU : Graphic Processor Unit）的來由。

把GPU設想成一堆小型微處理器排成一個平面的畫面，假設每個像素的數據是乒乓球。14400000個乒乓球可以在一秒內阻塞幾乎任何管道。但是一面800x600的管道牆，每秒接收30波480000個像素的信息就可以流暢完成。這在更高的分辨率下也是成立的 —— 並行的處理器越多，可以處理的數據流就越大。 另一個 GPU 的魔法是特殊數學函數可通過硬件加速。非常複雜的數學操作可以直接被微芯片解決，而無須通過軟件。這就表示可以有更快的三角和矩陣運算 —— 和電流一樣快。

1.3 Hello world!

“Hello world!”通常都是學習一個新語言的第一個例子。這是一個非常簡單，只有一行的程序。它既是一個熱情的歡迎，也傳達了編程所能帶來的可能性。

然而在 GPU 的世界裡，第一步就渲染一行文字太難了，所以我們改為選擇一個鮮豔的歡迎色，來吧躁起來！

儘管這幾行簡單的代碼看起來不像有很多內容，我們還是可以據此推測出一些知識點：

1.shader 語言 有一個 main 函5

2.數，會在最後返回顏色值。這點和 C 語言很像。

2.最終的像素顏色取決於預設的全局變量 gl_FragColor。

3.這個類 C 語言有內建的變量（像gl_FragColor），函數和數據類型。在本例中我們剛剛介紹了vec4（四分量浮點向量）。之後我們會見到更多的類型，像 vec3 （三分量浮點向量）和 vec2 （二分量浮點向量），還有非常著名的：float（單精度浮點型）， int（整型） 和 bool（布爾型）。

4.如果我們仔細觀察 vec4 類型，可以推測這四個變元分別響應紅，綠，藍和透明度通道。同時我們也可以看到這些變量是規範化的，意思是它們的值是從0到1的。之後我們會學習如何規範化變量，使得在變量間map（映射）數值更加容易。

5.另一個可以從本例看出來的很重要的類 C 語言特徵是，預處理程序的宏指令。宏指令是預編譯的一部分。有了宏才可以 #define （定義）全局變量和進行一些基礎的條件運算（通過使用 #ifdef 和 #endif）。所有的宏都以 # 開頭。預編譯會在編譯前一刻發生，把所有的命令複製到 #defines 裡，檢查#ifdef 條件句是否已被定義， #ifndef 條件句是否沒有被定義。在我們剛剛的“hello world!”的例子中，我們在第2行檢查了 GL_ES 是否被定義，這個通常用在移動端或瀏覽器的編譯中。

6.float類型在 shaders 中非常重要，所以精度非常重要。更低的精度會有更快的渲染速度，但是會以質量為代價。你可以選擇每一個浮點值的精度。在第一行（precision mediump float;）我們就是設定了所有的浮點值都是中等精度。但我們也可以選擇把這個值設為“低”（precision lowp float;）或者“高”（precision highp float;）。

7.最後可能也是最重要的細節是，GLSL 語言規範並不保證變量會被自動轉換類別。這句話是什麼意思呢？顯卡的硬件製造商各有不同的顯卡加速方式，但是卻被要求有最精簡的語言規範。因而，自動強制類型轉換並沒有包括在其中。在我們的“hello world!”例子中，vec4 精確到單精度浮點，所以應被賦予 float 格式。但是如果你想要代碼前後一致，不要之後花費大量時間 debug 的話，最好養成在 float 型數值里加一個 . 的好習慣。如下這種代碼就可能不能正常運行：
