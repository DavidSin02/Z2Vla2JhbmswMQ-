「ReentrantLock」可重入鎖，使用場景

2020-11-14
| 科技

  • 場景1：如果發現該操作已經在執行中則不再執行（有狀態執行）
  • 場景2：如果發現該操作已經在執行，等待一個一個執行（同步執行，類似已同步）
  • 場景3：如果發現該操作已經在執行，則嘗試等待一段時間，等待超時則不執行（嘗試等待執行）
  • 場景4：如果發現該操作已經在執行，等待執行。這時可中斷中斷的操作立即釋放鎖繼續下一操作。
  • 可重入概念

    • 場景1：如果發現該操作已經在執行中則不再執行（有狀態執行）

    • a，用在定時任務時，如果任務執行時間可能超過下次計劃執行時間，確保該有狀態任務只有一個正在執行，忽略重複觸發。
    • b，用在界面相互作用時單擊執行插入時間請求操作時，防止多次單擊導致後臺重複執行（忽略重複觸發）。

    以上兩種情況多用於進行非重要任務防止重複執行，（如：清除無用臨時文件，檢查某些資源的可用，數據備份操作等）

    private ReentrantLock lock = new ReentrantLock();...//如果已經被lock，則立即返回false不會等待，達到忽略操作的效果 if (lock.tryLock()) { try { //操作 } finally { lock.unlock(); }}

    場景2：如果發現該操作已經在執行，等待一個一個執行（同步執行，類似已同步）

    這種比較常見大家也都在用，主要是防止資源使用衝突，保證同一時間內只有一個操作可以使用該資源。

    但與synchronized的明顯區別是性能優勢（伴隨jvm的優化這個差異在細分）。

    同時鎖有更靈活的鎖定方式，公平鎖與不公平鎖，而同步永遠是公平的。

    這種情況主要用於對資源的爭搶（如：文件操作，同步消息發送，有狀態的操作等）

    可重入鎖定最小情況下為不公平鎖

    //參數默認false，不公平鎖private ReentrantLock lock = new ReentrantLock();//公平鎖private ReentrantLock lock = new ReentrantLock(true);try { // 如果被其它資源鎖定，會在此等待鎖釋放，達到暫停的效果 lock.lock(); //操作} finally { lock.unlock();}

    不公平鎖與公平鎖的區別：

    • 公平情況下，操作會排一個隊按順序執行，來保證執行順序。（會消耗更多的時間來排隊）
    • 不公平情況下，是無序狀態允許插隊，jvm會自動計算如何處理更快速來調度插隊。（如果不關心順序，這個速度會改變）

    場景3：如果發現該操作已經在執行，則嘗試等待一段時間，等待超時則不執行（嘗試等待執行）

    這種其實屬於場景2的改進，等待獲得鎖的操作有一個時間的限制，如果超時則放棄執行。

    採取了防止由於資源處理不當而導致連續堆積導致死鎖的情況（大家都在等待資源，導致線程堆積）。

    try { // 如果已經被lock，嘗試等待5s，看是否可以獲得鎖，如果5s後仍然無法獲得鎖則返回false繼續執行 if (lock.tryLock(5, TimeUnit.SECONDS)) { try { //操作 } finally { lock.unlock(); } }} catch (InterruptedException e) { // 當前線程被中斷時(interrupt)，會拋InterruptedException e.printStackTrace();}

    場景4：如果發現該操作已經在執行，等待執行。這時可中斷中斷的操作立即釋放鎖繼續下一操作。

    同步與鎖定在某種情況下是不會響應中斷（interrupt）操作，會繼續執行完。

    （場景2的另一種改進，沒有超時，只能等待中斷或執行完畢）

    這種情況主要用於取消某些操作對資源的佔用。如：（取消正在同步運行的操作，來防止不正常操作消耗佔用造成的一部分）

    try { lock.lockInterruptibly(); //操作} catch (InterruptedException e) { e.printStackTrace();} finally { lock.unlock();}

    可重入概念

    可重入（reentrant或re-entent）：一個程序或子程序可以“安全的被並行執行（並行計算）”。

    即當該子程序正在運行時，可以再次進入並執行它（並行執行時，個別的執行結果，都符合設計時的預期）。

    可重入概念是在單線程操作系統的時代提出的。
