美女程序員揭祕《阿里巴巴Android開發手冊》｜提供更新下載

2020-10-29
| 科技

此前，阿里巴巴對外發布了面向移動開發者的《阿里巴巴 Android 開發手冊》，在Android開發者當中產生了不小的影響。近期， InfoQ 對本手冊作者之一芸墨進行了採訪報道，並揭祕《阿里巴巴 Android 開發手冊》編寫過程中的相關故事。更新版的手冊下載，可關注“碼先生”，並私信回覆“碼先生”，即可下載。

前 言

不久之前谷歌的 Android P 預覽版問世，幾大手機廠商也聯合推出了快應用，我們看到了移動開發迎來了新的活力，移動應用已經滲透到人們生活的方方面面，那麼阿里繼去年推出 Java 手冊之後，在春節期間推出的《阿里巴巴 Android 開發手冊》的目的是什麼呢？為此，InfoQ 第一時間採訪了阿里巴巴高級無線開發工程師、手冊的主要作者之一芸墨，為大家揭開其背後創作故事，並在此與大家分享採訪結果。

Android 手冊有哪些內容？

《阿里巴巴 Android 開發手冊》全篇分為 9 個章節，分別是 Java 語言規範，Android 資源文件命名與使用，Android 基本組件，UI 與佈局，進程、線程與消息通信，文件與數據庫，Bitmap、Drawable 與動畫，安全及其他內容，即主要涵蓋了 Android 開發的性能、安全性和穩定性問題。作者認為這些章節不分主次，都是開發者在進行業務開發時容易忽視的問題，大家可根據實際情況參考。

Android 手冊是怎樣誕生的？

阿里做這件事的初衷源於雙十一，由於每年在雙十一準備期間都要成立專家組去集中掃蕩線上 crash，把 crash 率降到 0.1% 以下，修復 bug 以後，可能在不久的將來又會重現，那如何把專家的經驗沉澱下來得以複用呢？

在這些過程中，他們把一些常見 crash 的分析沉澱了下來，把一些結合 codestyle 寫到了阿里 Android 開發手冊裡，還有一些寫成了自定義規則放到了持續集成平臺和 IDEA 插件裡，比如常見的 CursorWindowAllocationException、SuperNotCalledException 和 ConcurrentModificationException 這一類的問題。後來他們想把這些好的經驗開放出來跟大家一起分享，所以《阿里巴巴 Android 開發手冊》就面世了。

然而手冊發佈之後，在引起高關注度的同時，也引發了不少讀者的質疑，很多人認為阿里在技術分享上動作頻頻，連發兩個手冊，是否是內部 KPI 推動的結果。對此，芸墨表示這並不是 KPI 任務，各研發團隊的主管並沒有給大家硬性指標，手冊的編者都是集團各大 APP 的一線業務開發者，來自淘寶、天貓、閒魚和支付寶等團隊，他們都是自發地組成一個虛擬項目組，本著前人栽樹後人乘涼，前人踩坑後人平路的心態，總結日常中解決的諸如性能、穩定性、安全、UI 適配等問題的經驗，並在日常的業務工作之餘完成了手冊的內容。

三個等級劃分的標準是什麼？

不管是之前 Java 手冊還是現在的 Android 手冊，阿里都按照約束力強弱將規則劃分了強制、推薦、參考三個等級，對於這幾個等級的制定，芸墨是這麼解釋的：

（1）把會造成線上穩定性問題、嚴重的性能問題和展示問題，常見的安全問題的寫法定義為【強制】級別的。

（2）把一些阿里的最佳實踐和一些經驗沉澱定義為【推薦】級別的，比如很多開發者最近都在問的“文本大小使用單位 dp 這一條是不是有問題，難道不是應該用 sp 嗎？”

實際上，在 App 在開發時，視覺都是設計好了整體 UI 佈局和控件的尺寸、位置，根據操作系統字體大小設置動態調整 app 字體會造成開發成本高到不可承受。在 Android 上，很多 App 對文本採用了 sp 作為單位，事實上就造成了它們的 UI 在字體大小變化的情況下完全無力適配，內容顯示異常。所以，約定的時候就考慮到了現實情況。如果一個產品在設計的時候就已經考慮到對字體設置的需求，那是可以無視這一條；只是大部分的 app 在設計的時候視覺交互其實只有一套，對他們來說的話如果忽略了這個問題用 sp 就會造成 UI 顯示的缺陷。

（3）把一些阿里內部各個團隊自己的約定定義為【參考】，即大家可以自行決定是否使用。

若規範與需求發生衝突，如何應對？

一般說來，產品經理需要保證自己需求的還原性和整體體驗，開發者也希望自己寫出的代碼美觀整潔 bug 少，大家都是同一戰線的夥伴，但是手冊中明令禁止的行為可能會對產品的可用性造成很嚴重的影響，會影響產品的線上使用，所以芸墨並不贊成開發者違背規範中強制的內容，而對於推薦和參考的內容，只要不影響產品的用戶體驗，開發者擇優選擇即可。

Android 手冊面向哪些開發者？

雖然手冊面向的主要是 Android 開發者，非 Native 的開發者如果指純 H5 的開發者那可能跟規約並沒有直接聯繫，如果指的是 RN 這種新興的動態化框架的話，那麼規約在性能優化、安全等方面依舊是可以為他們提供開發上的參考。

再者，芸墨認為，無論是 H5、RN、微信小程序的開發，還是 IOS/Android 原生開發，在 UI framework 這個層面原理是相通的，並不是新技術新框架出現了，已有的框架就被完全推倒重來。其次，RN、微信小程序越來越受關注是因為他們體現出來的組件化、動態化、快速部署的能力對原生開發者產生了巨大的吸引力，但是在這個過程中，JavaScript 扮演的更多的是一種 DSL（domain specific language）的角色，而不是說一看到 JavaScript 就等同於前端開發，他們本質上還是運行在原生操作系統上的，所以 Android 手冊裡的內容對那些學習新框架的 Android 原生開發者仍是不可或缺的基礎。

關於未來

對於 Android 手冊，阿里會時刻關注和及時回覆線上的用戶反饋，前期保持一個月一版本的迭代速度，中後期會根據 Android 的系統更新情況半年一次迭代，並表示 iOS 開發手冊內部版本正在籌劃對外開放中，敬請期待~

Android 手冊的內容都是阿里集團業務開發者的經驗總結，也結合了 Google 和業界的一些最佳實踐，去年已經在集團內部公示過很久，也得到了廣泛的認可。一個行業標準手冊出之不易，阿里表示他們會秉持著開放的態度，歡迎業內同行一起參與意見反饋和完善工作，這是他們願意見到的行業幸事。

受訪嘉賓簡介

龔能（花名：芸墨）阿里巴巴高級無線開發工程師，2014 年入職阿里巴巴，主要負責集團無線靜態代碼掃描體系建設和持續集成平臺開發，《阿里巴巴 Android 規約》的主要作者之一。
