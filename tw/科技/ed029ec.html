異步(async和await)

2020-10-29
| 科技

一.為什麼要使用異步編程？

現在的大多數的應用都廣泛使用文件和網絡I/O，而這些操作大多都是阻塞API的，為了充分利用CPU資源，提高效率。

注意：異步(async和await)可以提高響應能力也就是吞吐量，但是不能提高響應速度。也就是說處理一個請求的時間是保持不變的，但是可以釋放當前線程，增加處理能力

二.基於任務的異步編程模型TAP

計算密集型任務

  • .net frameowrk 4 可以使用TaskFactory.StartNew或者Task.Factory.StartNew的方式創建異步任務
  • .net 4.5以上以及.net core使用Task.Run作為TaskFactory.StartNew的快捷方式

其中Task.ContinueWith、TaskFacotry.ContinueWhenAll、TaskFactory.ContinueWhenAny方法可以實現創建一下再另一個任務完成後需要執行的新任務.

如果在任務執行前，可以通過取消標記Cancellation對象，可以停止任務

  class Program    {        //聲明CancellationTokenSource對象        static CancellationTokenSource cancelTokenSource = new CancellationTokenSource();        //程序入口        static void Main(string[] args)        {            Task.Factory.StartNew(Test, cancelTokenSource.Token);            Console.ReadLine();            cancelTokenSource.Cancel();            Console.WriteLine("已停止");            Console.ReadLine();        }        //測試方法        static void Test()        {            //判斷是否取消任務            while (!cancelTokenSource.IsCancellationRequested)            {                Console.WriteLine(DateTime.Now);                Thread.Sleep(1000);            }        }    }

I/O密集型任務

I/O密集型任務是以Windows事件為基礎的，因此不需要新建一個線程或使用線程池裡面的線程來執行具體工作。不要使用Task.Run,反而浪費資源。 IO場景中，文件IO由win32用戶模式的API到windows內核模式，在內核模式中操作磁盤驅動程序。可能會調入系統 API 調用（例如 Linux 上套接字的 write()）。可能會使用 TaskCompletionSource 在本機/託管邊界創建一個任務對象。

三.async 和 await

什麼是async和await

這是是一種語法糖，是對異步編程模型(TPL基於任務的異步編程模型)的封裝

async和await的本質
  • async會在編譯層生成狀態機，每一個await就是一個狀態，如果存在await嵌套，那就是一個狀態一個狀態的轉換。如果只有async而沒有await，此時狀態機將不會生成getawater，此時將不會等待處理結果。相當於多線程編程
  • async和await本身並不會在生成的代碼中新建一個線程，新開的線程都是在await後面的方法中自己創建的。

四.異步和同步流程比較

當主線程讀取一個應用程序之外的資源，比如處理文件資源或者外部服務的時候。主線程會從用戶模式進入到內核模式。在內核模式中，windows會將請求數據移交給對應的驅動程序去處理。

  • 如果是同步的話，那麼當前主線程會進入休眠狀態。當驅動程序和外部資源進行交互後，網絡驅動程序拿到結果，然後移交給當初睡眠的線程，windows會喚醒線程，繼續執行後面的C#代碼

  • 異步的話，此時主線程不會休眠，而是直接返回線程池。當網絡驅動程序拿到結果後，會將數據交給CLR線程池中的IO線程，然後由這個線程觸發回調函數。

 class Program    {        static void Main(string[] args)        {            CallerWithAsync("jack");            Console.ReadKey();        }                //重點調用方法        async static void CallerWithAsync(string name)        {            Console.WriteLine($"異步調用頭部執行，當前線程Id為：{Thread.CurrentThread.ManagedThreadId}");            string result =  SayHiAsync(name).Result;            Console.WriteLine($"異步調用尾部執行，當前線程Id為：{Thread.CurrentThread.ManagedThreadId}");            Console.WriteLine(result);        }                 async static void CallerWithAsyn1c(string name)        {            Console.WriteLine($"異步調用頭部執行，當前線程Id為：{Thread.CurrentThread.ManagedThreadId}");            string result =await  SayHiAsync(name);            Console.WriteLine($"異步調用尾部執行，當前線程Id為：{Thread.CurrentThread.ManagedThreadId}");            Console.WriteLine(result);        }        static Task<string> SayHiAsync(string name)        {            return Task.Run<string>(() => { return SayHi(name); });        }        static string SayHi(string name)        {            Task.Delay(2000).Wait();//異步等待2s            Console.WriteLine($"SayHi執行，當前線程Id為：{Thread.CurrentThread.ManagedThreadId}");            return $"Hello,{name}";        }             }
  • CallerWithAsync採用的是Result的方式獲取方法的值，這種方式是阻塞當前線程的。也就是當前線程是在等待。此時會造成當前線程資源的浪費。
  • CallerWithAsync1採用的是await的方式獲取返回值。這是非阻塞的方式獲取，此時會將當前線程直接釋放回線程池。當Task.Run中的代碼執行完畢後，會重新獲取一個線程，然後執行主線程後面的代碼。所以重新獲取的這個線程可能和原來的主線程一樣。但是基本上是另外一個線程。
