為你揭祕神祕的spring監聽機制

2020-11-14
| 科技

從觀察者模式說起

觀察者模式是軟件設計的常用模式之一，是一種一對多的組合關係，存在觀察者和被觀察者兩種角色。在觀察者模式中，觀察者與被觀察者沒有直接的類調用關係，有著清晰的模塊劃分界限，從而提高了軟件的可維護性。



被觀察者

被觀察者就是我們感興趣的對象，比如用戶、商品、文件、一個變量，我們希望瞭解被觀察者的狀態變化，並且被動接受變化消息。當被觀察者狀態變化時，發送消息通知觀察者。

觀察者

觀察者可以有多個，互相之間一般沒有依賴，也不需要確保觀察者接收消息的先後順序。觀察者其實是現實世界在程序中的一個縮影，比如我們不用一直盯著手機，只需要聽到鈴聲就知道有電話進來。

Java 中的觀察者模式

在 Java 中，內置了 Observable 類和 Observer 接口來實現觀察者模式。使用時，Observable 作為被觀察對象，通過調用 addObserver 方法來增加觀察者，當狀態變化時，被觀察者調用 setChanged 方法來標識狀態變化，然後調用 notifyObservers 方法來通知觀察者。Observer 接口用於實現觀察者，只有一個方法 update，當狀態變化時，觀察者的 update 方法被調用，從而獲知狀態變化。

public class MyObservable extends Observable {    public static void main(String[] args){        MyObservable myObservable = new MyObservable();        myObservable.addObserver(new Observer() {            @Override            public void update(Observable o, Object arg) {                System.out.println(arg);            }        });        myObservable.setChanged();        myObservable.notifyObservers("hello");    }}

Spring 中的事件

Spring 中的事件通知機制就是觀察者模式的一種實現。觀察者是 ApplicationListener，可以實現接口定義觀察者，也可以使用註解定義觀察者。觀察者感興趣的是某種狀態的變化，這種狀態變化使用 ApplicationEvent 來傳達，也就是事件對象。我們說的 Spring 中的事件，就是 ApplicationEvent。在事件中，被觀察者可以認為是發出事件的一方，只有在狀態變化時才發佈事件。

當有狀態發生變化時，發佈者調用 ApplicationEventPublisher 的 publishEvent 方法發佈一個事件，Spring 容器廣播事件給所有觀察者，調用觀察者的 onApplicationEvent 方法把事件對象傳遞給觀察者。調用 publishEvent 方法有兩種途徑，一種是實現接口由容器注入 ApplicationEventPublisher 對象然後調用其方法，另一種是直接調用容器的方法，兩種方法發佈事件沒有太大區別。

相關類型總結如下：

  • ApplicationListener：事件監聽者，觀察者；
  • ApplicationEvent：Spring 事件，記錄事件源、時間和數據；
  • ApplicationEventPublisher：發佈事件；

優勢

使用事件可以解耦代碼，觀察者與被觀察者可以分開開發，中間只有事件作為聯繫，不用關心另一方如何實現。觀察者可以有多個，所以對於同一個事件可以有多種不同的處理方式，不過要確保不依賴處理的順序。使用事件後，觀察者可以單獨開發，對主流程沒有任何影響，可以簡化主流程的開發。

事件可以用於各種場景的消息通知，比如系統收到停機指令後，可以發出停機事件，這樣相關的子系統就可以進行停機前的各種處理。那麼 Spring 中的事件是如何實現的呢？

Spring 框架提供了四種容器事件，我們可以直接觀察，包括：

  • ContextStartedEvent：ApplicationContext 啟動事件；
  • ContextRefreshedEvent：ApplicationContext 更新事件；
  • ContextStoppedEvent：ApplicationContext 停止事件；
  • ContextClosedEvent：ApplicationContext 關閉事件。


Spring 4.2 之前的版本，事件必須繼承 ApplicationEvent，從 Spring 4.2 版開始，框架提供了 PayloadApplicationEvent 用於包裝任意類型，不強制事件對象繼承 ApplicationEvent。當我們發送一個任意類型的事件對象時，框架內部自動包裝為 PayloadApplicationEvent 事件對象。

事件監聽者

事件監聽者 ApplicationListener 繼承自 JDK 的 EventListener ，JDK 要求所有監聽者繼承它。監聽者只有一個方法 onApplicationEvent，用來處理事件 ApplicationEvent。

@FunctionalInterfacepublic interface ApplicationListener<E extends ApplicationEvent>     extends EventListener {    void onApplicationEvent(E event);}

在容器啟動的時候檢測應用中的監聽者並把用戶實現的監聽者註冊到 SimpleApplicationEventMulticaster 集合中。

Spring 也支持直接註解的形式進行事件監聽，使用註解 @EventListener 即可。使用註解時，方法可以返回任意類型，如果返回值不為 null 則當做一個新事件再次發佈。

@Servicepublic class AnnoEventListener {    @EventListener    public void listen(MyEvent myEvent){        System.out.println("receive " + myEvent.getSource());    }}

註解形式的監聽者是通過 EventListenerMethodProcessor 註冊到容器中的。該類定義了一個 Bean，在初始化完成後，調用它的後置回調方法 afterSingletonsInstantiated，在方法中遍歷容器中所有的 bean，提取出 EventListener 註解修飾的方法並根據註解的參數創建 ApplicationListener 對象加入到 ApplicationContext 監聽列表中。

發佈事件

ApplicationEventPublisher 接口定義了事件發佈方法，代碼如下：

@FunctionalInterfacepublic interface ApplicationEventPublisher {    default void publishEvent(ApplicationEvent event) {        publishEvent((Object) event);    }    void publishEvent(Object event);}

ApplicationContext 接口繼承了 ApplicationEventPublisher ，並在 AbstractApplicationContext 實現了具體代碼，實際執行是委託給 ApplicationEventMulticaster。


ApplicationEventMulticaster 接口定義了對監聽者的操作，如增加監聽者、移除監聽者，以及發佈事件的方法。框架提供了 ApplicationEventMulticaster 接口的默認實現 SimpleApplicationEventMulticaster，如果本地容器中沒有 ApplicationEventMulticaster 的實現就會使用這個默認實現。

public interface ApplicationEventMulticaster {    void addApplicationListener(ApplicationListener<?> listener);    void addApplicationListenerBean(String listenerBeanName);    void removeApplicationListener(ApplicationListener<?> listener);    void removeApplicationListenerBean(String listenerBeanName);    void removeAllListeners();    void multicastEvent(ApplicationEvent event);    void multicastEvent(ApplicationEvent event, @Nullable ResolvableType eventType);}

在 SimpleApplicationEventMulticaster 中，可以看到 multicastEvent 方法中遍歷了所有的 Listener，並依次調用 Listener 的 onApplicationEvent 方法發佈事件。Spring 還提供了異步發佈事件的能力，taskExecutor 不為 null 時即異步執行。

public class SimpleApplicationEventMulticaster extends AbstractApplicationEventMulticaster {    @Nullable    private Executor taskExecutor;    @Override    public void multicastEvent(final ApplicationEvent event,                                @Nullable ResolvableType eventType) {        ResolvableType type =         (eventType != null ? eventType : resolveDefaultEventType(event));        for (final ApplicationListener<?> listener : getApplicationListeners(event, type)) {            Executor executor = getTaskExecutor();            if (executor != null) {                executor.execute(() -> invokeListener(listener, event));            }            else {                invokeListener(listener, event);            }        }    }}

可以看到，Spring 處理事件是同步的，監聽者的執行時間最好不要太長，以免影響主流程。如果事件的處理一定要耗費比較長的時間，請使用異步方式進行處理。

高級監聽者

SmartApplicationListener

SmartApplicationListener 接口是 ApplicationListener 的子接口，還繼承了 Ordered 接口。SmartApplicationListener 定義了兩個 support 方法用於判斷事件類型、來源類型是否和當前監聽者匹配，這樣監聽者可以篩選自己感興趣的事件和來源。繼承 Ordered 接口後，該監聽者具備了排序的功能，可以按照 order 從小到大的順序給監聽者確定一個優先級，從而確保執行順序。

public interface Ordered {    int HIGHEST_PRECEDENCE = Integer.MIN_VALUE;    int LOWEST_PRECEDENCE = Integer.MAX_VALUE;    int getOrder();}public interface SmartApplicationListener     extends ApplicationListener<ApplicationEvent>, Ordered {    boolean supportsEventType(Class<? extends ApplicationEvent> eventType);    boolean supportsSourceType(Class<?> sourceType);}

GenericApplicationListener

GenericApplicationListener 接口是 ApplicationListener 的子接口，也繼承了 Ordered 接口，同 SmartApplicationListener 一樣具有事件篩選能力和排序能力。但篩選事件使用的是 ResolvableType 類型，而不是 ApplicationEvent 類型。

總結

本文探討了 Spring 中事件的實現原理，基於事件可以很好地解耦代碼，瞭解了背後的實現原理可以更好地使用事件。
