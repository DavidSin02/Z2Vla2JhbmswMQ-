「Web應用架構」WebSocket用例，性能和性能檢查列表

2020-11-14
| 科技

WebSocket API為客戶機和服務器之間的文本和二進制數據的雙向、面向消息流提供了一個簡單的接口:向構造函數傳遞一個WebSocket URL，設置幾個JavaScript回調函數，然後我們就啟動並運行了——其餘的由瀏覽器處理。再加上WebSocket協議，它提供了二進制幀、可擴展性和子協議協商，WebSocket成為在瀏覽器中交付定製應用協議的完美工具。

然而，就像任何關於性能的討論一樣，儘管WebSocket協議的實現複雜性對應用程序是隱藏的，但它仍然對如何以及何時使用WebSocket有重要的性能暗示。WebSocket不是XHR或SSE的替代品，為了獲得最佳性能，我們必須充分利用每種傳輸的優勢!

要查看每個傳輸的性能特徵，請參考XHR用例和性能以及SSE用例和性能。

請求和響應流

WebSocket是唯一允許在同一個TCP連接上進行雙向通信的傳輸(圖17-2):客戶端和服務器可以隨意交換消息。因此，WebSocket在兩個方向上都提供了文本和二進制應用程序數據的低延遲交付。


圖17-2。XHR, SSE, WebSocket的通信流程

  • XHR為“事務性”請求-響應通信進行了優化:客戶機向服務器發送完整的、格式良好的HTTP請求，服務器以完整的響應響應。不支持請求流，在Streams API可用之前，沒有可靠的跨瀏覽器響應流API。
  • SSE實現了基於文本數據的高效、低延遲的服務器到客戶端的流化:客戶端啟動SSE連接，服務器使用事件源協議將更新流化到客戶端。在初始握手之後，客戶端不能向服務器發送任何數據。

傳播和排隊延遲

交換從XHR到SSE或WebSocket的傳輸不會減少客戶端和服務器之間的往返!無論傳輸方式如何，數據包的傳播延遲都是相同的。然而，除了傳播延遲之外，還有排隊延遲:消息在被路由到另一方之前必須在客戶機或服務器上等待的時間。

在XHR輪詢的情況下，排隊延遲是客戶端輪詢間隔的函數:消息可能在服務器上可用，但直到下一個客戶端XHR請求才能發送;請參閱XHR輪詢的建模性能。相比之下，SSE和WebSocket都使用一個持久連接，這允許服務器在消息可用的時候就發送消息(以及客戶端，在WebSocket的情況下)。

因此，對於SSE和WebSocket來說，“低延遲交付”是指消除消息排隊延遲。我們還沒有想出如何讓WebSocket數據包的傳輸速度超過光速!

消息開銷

一旦WebSocket連接建立，客戶端和服務器通過WebSocket協議交換數據:應用消息被分成一個或多個幀，每個幀增加2到14字節的開銷。而且，由於框架是通過自定義二進制格式完成的，因此UTF-8和二進制應用程序數據都可以通過相同的機制進行有效編碼。與XHR和SSE相比如何?

  1. SSE只增加了5字節的每條消息，但僅限於UTF-8內容;參見事件流協議。
  2. HTTP / 1。x請求(XHR或其他)將攜帶額外的500-800字節HTTP元數據，加上cookie;參見測量和控制協議開銷。
  3. HTTP/2壓縮HTTP元數據，這大大降低了開銷;看到標題壓縮。事實上，如果頭在請求之間不改變，開銷可以低至8字節!

請記住，這些開銷數字不包括IP、TCP和TLS幀的開銷，這些開銷會增加每條消息60-100字節的綜合開銷，而與應用程序協議無關;參見優化TLS記錄大小。

數據效率和壓縮

每個XHR請求可以通過定期的HTTP協商來協商最佳的傳輸編碼格式(例如，基於文本的數據的gzip)。類似地，由於SSE僅限於utf -8傳輸，所以可以通過在整個會話中應用gzip有效地壓縮事件流數據。

使用WebSocket，情況會更加複雜:WebSocket可以傳輸文本和二進制數據，因此壓縮整個會話沒有意義。二進制有效負載可能已經被壓縮了!因此，WebSocket必須實現自己的壓縮機制，並有選擇地將其應用於每個消息。

好消息是HyBi工作組正在為WebSocket協議開發每消息壓縮擴展。然而，它還沒有在任何瀏覽器中可用。因此，除非應用程序通過仔細優化其二進制有效負載(參見用JavaScript解碼二進制數據)來實現自己的壓縮邏輯，併為基於文本的消息實現自己的壓縮邏輯，否則它可能會對傳輸的數據產生較高的字節開銷!

Chrome和一些基於webkit的瀏覽器支持WebSocket協議壓縮擴展的舊版本(每幀壓縮);參見WebSocket多路複用和壓縮。

自定義應用程序協議

瀏覽器針對HTTP數據傳輸進行了優化:它理解該協議，並提供廣泛的服務，如身份驗證、緩存、壓縮等等。因此，XHR請求免費繼承所有這些功能。

相比之下,流允許我們提供定製的客戶端和服務器之間的協議,但代價的繞過許多瀏覽器提供的服務:最初的HTTP握手可以執行一些協商參數的連接,但是一旦建立了會話,所有進一步的客戶端和服務器之間的數據流是不透明到瀏覽器。因此，交付自定義協議的靈活性也有其缺點，應用程序可能必須實現自己的邏輯來填補缺失的空白:緩存、狀態管理、消息元數據的交付，等等!

初始的HTTP升級握手允許服務器利用現有的HTTP cookie機制來驗證用戶。如果驗證失敗，服務器可以拒絕WebSocket升級。

利用瀏覽器和中間緩存

使用常規HTTP具有顯著的優勢。問您自己一個簡單的問題:客戶機會從緩存接收到的數據中獲益嗎?或者，如果中介可以緩存資產，它是否可以優化資產的交付?

例如，WebSocket支持二進制傳輸，這允許應用程序流任意的圖像格式而沒有開銷——好漂亮!然而，圖像是在自定義協議中交付的這一事實意味著它不會被瀏覽器緩存或任何中介(例如CDN)緩存。因此，您可能需要向客戶機進行不必要的傳輸，並向源服務器傳輸更多的流量。同樣的邏輯也適用於所有其他數據格式:視頻、文本等等。

因此，確保你為工作選擇了合適的交通工具!解決這些問題的一個簡單而有效的策略是使用WebSocket會話來傳遞非緩存數據，比如實時更新和應用“控制”消息，這可以觸發XHR請求來通過HTTP協議獲取其他資產。

部署WebSocket基礎設施

HTTP為短時間和突發傳輸進行了優化。因此，許多服務器、代理和其他中介經常被配置為主動超時空閒HTTP連接，當然，這正是我們不希望看到的長時間WebSocket會話。為了解決這個問題，有三方面需要考慮:

  • 路由器，負載平衡器，和代理在自己的網絡
  • 外部網絡中的透明和顯式代理(如ISP和運營商代理)
  • 客戶網絡中的路由器、防火牆和代理

我們無法控制客戶端網絡的策略。事實上，一些網絡可能會完全阻止WebSocket流量，這就是為什麼您可能需要一個備用策略。類似地，我們無法控制外部網絡上的代理。然而，這正是TLS可以幫助的地方!通過在安全的端到端連接上使用隧道，WebSocket通信可以繞過所有的中間代理。

使用TLS並不會阻止中間層對空閒TCP連接計時。然而，在實踐中，它極大地提高了協商WebSocket會話的成功率，並且通常還有助於擴展連接超時間隔。

最後，還有我們自己部署和管理的基礎設施，這通常也需要關注和調優。儘管責怪客戶端或外部網絡很容易，但問題往往是在家裡。服務路徑中的每個負載均衡器、路由器、代理和web服務器都必須進行調優，以允許長時間連接。

例如，Nginx 1.3.13+可以代理WebSocket流量，但默認為60秒超時!為了增加限制，我們必須明確地定義較長的超時:

location /websocket {    proxy_pass http://backend;    proxy_http_version 1.1;    proxy_set_header Upgrade $http_upgrade;    proxy_set_header Connection "upgrade";    proxy_read_timeout 3600;     proxy_send_timeout 3600; }
  1. 設置讀取間隔60分鐘的超時時間
  2. 在寫之間設置60分鐘的超時時間

類似地，在一個或多個Nginx服務器前安裝一個負載均衡器也很常見，比如HAProxy。毫不奇怪，我們在這裡也需要應用類似的顯式配置。HAProxy,因為:

defaults http  timeout connect 30s  timeout client  30s  timeout server  30s  timeout tunnel  1h
  • 隧道的60分鐘不活動超時

前面示例的問題是額外的“隧道”超時。在HAProxy中，連接、客戶端和服務器的超時只適用於初始的HTTP升級握手，但是一旦升級完成，超時由隧道值控制。

Nginx和HAProxy只是在我們的數據中心運行的數百個不同的服務器、代理和負載平衡器中的兩個。我們不能列舉這些頁面中所有的配置可能性。前面的示例只是說明了大多數基礎設施需要自定義配置來處理長期存在的會話。因此，在實現應用程序保持生命值之前，首先要仔細檢查基礎設施。

長期會話和空閒會話佔用所有中間服務器上的內存和套接字資源。因此，短超時通常被認為是安全、資源和操作方面的預防措施。部署WebSocket、SSE和HTTP/2(它們都依賴於長壽命會話)帶來了各自的新操作挑戰。

性能檢查表

部署高性能WebSocket服務需要在客戶端和服務器端進行仔細的調優和考慮。列入議程的標準的簡短清單:

  • 使用安全的WebSocket (WSS over TLS)進行可靠的部署。
  • 密切關注polyfill 的性能(如有必要)。
  • 利用子協議協商來確定應用程序協議。
  • 優化二進制有效載荷以最小化傳輸大小。
  • 考慮壓縮UTF-8內容以最小化傳輸大小。
  • 為接收到的二進制有效負載設置正確的二進制類型。
  • 監視客戶機上緩衝的數據量。
  • 分割大型應用程序消息以避免線頭阻塞。
  • 在適用的情況下利用其他傳輸。

最後，但絕對不是最不重要的，優化移動!在手機上，實時推送可能是一個代價高昂的性能反模式，因為電池壽命總是非常寶貴。這並不是說WebSocket不應該在移動設備上使用。相反，它可以是一種高效的交通工具，但一定要考慮到它的要求:

  • 保護電池
  • 消除週期性和低效的數據傳輸
  • 高效的服務器推送
  • 消除不必要的應用程序保持活動
