對java併發鎖的思考：有鎖，無鎖，不可重入，可重入的舉例說明

2020-11-14
| 科技

前言：發個原創，以饗讀者。感謝大家對本頭條號的支持。如果有異議、有問題請到mybatis.cn找我詳聊。本文屬於原創作品。

最近看了一下關於鎖的內容。感覺很多文章寫的不是空洞就是細節繁瑣。空洞，說明寫的人水平有限；寫的繁瑣也不見的是好事，也可能說明作者沒有吃透，雲深不知處。有的作者從java講到jvm的底層c++實現，翻代碼雖不是人人都會，但是會的人也不少，列出源碼未必就說明作者能做到雲深也知處。我感覺有的作者也許就是在譁眾取寵，大段代碼貼上去，顯示文章的高大上而已，我們讀者千萬別被坑了。

什麼樂觀鎖，什麼悲觀鎖，那都是好事者的意淫，很多學者要出論文，總得造點新東西出來吧。悲觀鎖，樂觀鎖，概念一出，豈不是一片立意新穎的論文啊。

我感覺在併發鎖的發展史上，在那些造“鎖”的技術人眼裡，壓根就沒有什麼樂觀，什麼悲觀，人家就是為了提高性能而已，只是後人扣了很多的帽子而已，也許這些造“鎖”人看到“併發”和“樂觀”都一臉懵逼，這是啥玩意呢？

說一下個人的見解吧，時間有限，並未能詳細查閱各種資料，如有疏忽請指正，不喜亂噴，只願坦誠交流。在jdk5之前，鎖只有一種，就是synchronized，用某個共享資源的時候先上鎖，用完就釋放，在jdk5之後，java出現了併發包，這些包裡面的類都是拿到共享資源，直接就用，它壓根就沒有鎖的概念和意識了，用：就是先read再write。

說到read和write，這裡面也有說道，你看，拼寫出read敲擊了四次鍵盤，拼寫出write敲擊了五次鍵盤，在cpu裡面也是一樣，它都是分為好幾步完成read或者write。後來，cpu發展了，read或者write可以一步完成，就跟你複製粘貼一樣，一次就完成了。

在java的併發包裡面，採用的都是read或者write一步完成的方式，整體的思路就是：

for(;;)

{

old = read();

result = write(old,new);

if (result)

{

break;

}

}

後來，好事者說這是樂觀鎖，至於是不是，我也不想論證，我在這裡只是說了一下有鎖和無鎖。

無鎖，就沒有什麼可說了，但是有鎖，就有問題出現了。鎖，剛開始是不可重入的，後來變成了可重入的。不可重入的鎖，也很好模擬，我可以用redis模擬一下，此時的鎖就是redis裡面的key，鎖可以是多種東西，這裡採用redis key為了方便說明：

SomeBody{

doSomeThing()

{

redis.setnx(key);

doSomeThing();

redis.del(key);

}

}//SomeBody End

SomeBody sb = new SomeBody();

Thread run ()

{

sb.doSomeThing();//卡住了，因為不可重入

}

```

下面說一下可重入鎖：

```java

SomeBody

{

@lock[this][threadId]//this表明lock的是此對象；threadId表明是被那個線程lock住了。

doSomeThing()

{

doSomeThing();

}

}//SomeBody End

SomeBody sb = new SomeBody();

Thread run ()

{

sb.doSomeThing();

}

此代碼的解釋如下：

sb.doSomeThing();和@lock[this][threadId]兩行代碼的理解是：

（1）表明以sd為鎖

（2）是Thread上的鎖

這樣的話，就知道是誰上的鎖，那麼誰就可以重入了。在這裡大家要注意一下，對threadId的理解，其實線程thread在cpu裡面就是一個數據結構，threadId就是來唯一標示這個數據結構的東西。
