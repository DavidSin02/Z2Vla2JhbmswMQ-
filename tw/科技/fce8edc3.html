gzthermal使用方式如下：

gzthermal: displays compression efficiency heatmap of gzipped files.Version 0.4b (23 May 2014) by Frederic KayserUsage: gzthermal [-e|-s|-m|-l] [-n|-w] [-b] [-z] [-g] file.gzOutput file is a PNG image called "gzthermal-result.png"Options: e, s, m, l  extra small, small, medium or large text output         n, w        narrow or wide text lines (default is in between)         b           binary/hex mode (small text size equivalent only)         z           bicolor (distinguish LZ matches from literals)         g           grayscale mode (may help colorblind people)

除了以上，還包括未記錄的數字選項（壓縮級別），只需添加數字即可將壓縮級別設置為0到9，例如，“ gzthermal -ew9”應生成比“ gzthermal -ew”小的輸出PNG文件（默認壓縮級別為4）。

壓縮原理

Gzip / Deflate壓縮基本上使用LZSS + Huffman算法，由Huffman算法提供的可變長度編碼（VLE，Variable Length Encoding ）應用於三種不同類型的元素：文字（不屬於LZ匹配的獨立符號），LZ長度和LZ匹配對的LZ距離。
用於表示符號的位數是從其出現頻率得出的，基本上，較多出現的符號通常以較短的代碼結尾（例如3、4、5位，這也取決於源字母表的大小），而較少的經常以較大的代碼（例如8、9、10，...位）結尾。這種VLE（可變長度編碼）實際上早於Huffman編碼，Morse（摩斯碼）以這種方式工作，Lineotype排字機鍵盤佈局也是根據字符頻率設計。
在文件末尾（顯示更藍，更深的藍色）或至少在處理了一定數量的數據（幾千個字節/符號）之後，壓縮會會更好地工作。預熱期之後的主要原因是，在數據流的早期，字典（或更確切地說是此處的滑動窗口）幾乎是空的，並且第一次出現了許多符號/詞，之後出現了一組符號/詞。一個單詞/一組單詞再次出現，它將由對先前出現的引用的替換（這是LZ匹配），這樣可以節省大量空間，顯示為藍色區域。相比之下，首次出現似乎顯得很龐大，這也是首次出現的代價。
這也是支持CSS和JavaScript文件結合的一個很好的論據。。
