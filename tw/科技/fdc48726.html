Redis設計與實現

Redis設計與實現 (豆瓣): https://book.douban.com/subject/25900156/

通過前面這本書我們已經知道redis怎麼用比較好了，現在我們來看看 Redis 的實現原理。這本書是作者自己看著源碼寫出來的，不得不佩服作者的智慧與毅力。這本書基於redis3.0，此刻redis最新版是4.0.9，我們看書的時候可以自己去看看源碼，看看redis有啥變化沒，源碼在此。

亮點：

  • redis並沒有用C語言的字符串作為其基準字符串表示，而是使用了SDS（簡單動態字符串）這一抽象類型，被用於鍵、字符串的表示，還有AOF中的緩衝區、客戶端輸出緩衝區等buffer的表示。 SDS的優勢在於：記錄了長度信息不需要遍歷（空間換時間）、有長度所以會先檢查空間是否足夠而不會溢出、記錄了free（未使用空間）長度從而避免頻繁的內存重新分配（實現了空間預分配和惰性釋放）、二進制安全等等
  • redis還定義了鏈表Linkedlist（用於實現List結構、發佈訂閱、慢查詢、監視器等功能）、字典Dict（基於兩個hash表，一個平時用、一個rehash用，主要用於實現數據庫、Hash結構，需要注意漸進式rehash而不是java中Hashmap的一次性rehash，漸進式主要是避免鍵數量太大導致服務器暫時停止服務）、跳躍表SkipList（用於實現SortedSet結構、集群節點）、整數集合（用於實現Set結構）、壓縮列表（用於實現List和Hash結構，主要是面向小整數和短字符串，可以節省空間）、對象redisObject（redis的五種數據結構就是五種對象，key value都是對象，是對前面所有數據結構的一種封裝，實現類型判斷、編碼、引用計數與對象共享和內存回收、過期刪除等功能）等數據結構來實現特定功能
  • redis服務器是一個redisServer數據結構，包含一個redisDb的鏈表，每一個數據庫都是一個數據結構redisDb，中間有一個dict的數據結構保存了這個數據庫所有的鍵值對，叫做鍵空間（key space）。鍵空間會維護命中次數、最後使用時間、過期清除、鍵監視、鍵空間（關注鍵本身）/鍵事件（關注操作本身如del、sadd、set等）通知等事件。此外還有一個clients字段保存了所有客戶端
  • redisDb有一個dict類型的鏈表expire，保存了所有鍵的過期時間。過期鍵有三種刪除策略：定時刪除，設置timer，過期立即刪除，對內存友好，但是浪費CPU時間片，而且大量的timer也不現實；惰性刪除，放任不管，獲取的時候檢查是否過期，過期再刪除，會浪費一定的內存，但是節省CPU時間；定期刪除，每隔一段時間就檢查過期鍵並刪除，需要好好設計執行時長與頻率。redis實際使用的是後兩者
  • 持久化分為AOF和RDB。RDB可以手動執行也可定期執行（BGSAVE）用內存數據生成壓縮過的二進制RDB文件，SAVE命令會阻塞服務器，而BGSAVE會fork一個子進程來創建RDB文件，所以期間新的數據變化是不能提現到RDB文件中的，此外，RDB文件的載入是服務器啟動自動完成的；AOF不直接記錄數據，而是記錄每一個redis命令，分為append、文件寫入（可能在系統緩衝中）、文件同步sync（強制刷到磁盤）三個步驟，文件同步有三種選項always（寫入文件就刷盤）、everysec（默認）、no（由操作系統決定何時刷盤）
  • redis是一個事件驅動的程序，這種程序一般會有一個類似於while(true)的無限循環，叫做事件循環，每個循環裡服務器要處理兩種事件：文件事件，完成網絡通信（基於Reactor模式）；時間事件，完成定時、週期操作之類。
  • Sentinel（哨兵）是redis的高可用解決方案，由>=1個Sentinel實例監視任意多的redis主服務器以及所屬的從服務器（從服務器複製主服務器的數據），完成主備切換等功能。實現原理是Sentinel每隔10秒就向所有主服務器發送info命令，獲取各個主服務器及其對應的從服務器的信息，然後也會以相同形式給從服務器發送info命令。另外會以兩秒一次的頻率向所有主從服務器發送一個publish的命令，然後會subscribe這個頻道，這樣就能獲得每個服務器的相關信息了。Sentinel之間只有命令連接，沒有頻道連接。
  • 在線檢測通過每秒的ping來實現，當超過一個Sentinel的時限down-after-milliseconds還沒獲得一個服務器的回覆這個Sentinel就認為此服務器主觀下線，然後會詢問其他Sentinel，如果超過quorum個Sentinel都認為此服務器主觀下線那麼這些Sentinel就認為這個服務器就客觀下線了。當一個主服務器下線時，Sentinel們會選舉一個領頭Sentinel（Raft leader選舉）來對這個服務器群進行主備切換，具體算法就不說了
  • redis集群通過槽指派實現，所有數據分配到16284個槽，所有槽都有節點處理時集群才能對外服務，每個節點可以處理多個槽（一個節點可以是主從服務器群，提高該節點可用性），當一個節點收到命令發現該槽不在本節點時會向客戶端返回重定向指令，讓其請求對應的節點。重新分槽可以在線分，由redis-trib實現。
  • redis 支持慢查詢，超過showlog-log-slower-than選項執行時間的命令會被記錄到慢查詢日誌，日誌採用保存最近的，數量由slowlog-max-len決定。

本書配合著官方文檔來學習是最佳的，文檔在此，中文文檔。

MySQL技術內幕

MySQL技術內幕 (豆瓣): https://book.douban.com/subject/24708143/

通過前面這本書我們已經知道 mysql 怎麼用比較好了，現在我們來看看 mysql 的實現原理。

亮點：

  • mysql採用單進程多線程模式。邏輯上，採用了分層可插拔架構，上層是連接池管理模塊，中層是語句解析、緩存、優化模塊，下層是可插拔的存儲引擎如myisam、Innodb（存儲引擎基於表而不是數據庫），下層依賴於操作系統層面的文件系統。
  • InnoDB支持事務、行鎖、外鍵，主要面向在線事務處理應用（OLTP），通過MVVC提高併發性，默認可重複讀級別使用 next-key locking避免幻讀，此外還提供插入緩衝、二次寫、自適應hash索引、預讀等高性能與高可用功能；MyISAM不支持事務與行鎖，但是支持全文索引，主要面向OLAP應用；NDB是一個集群引擎，數據全部在內存所以速度很快，但是連接操作不是在引擎層面解決的，效率較低；Memory也是內存存儲的引擎，適合臨時表存儲中間結果使用，默認使用hash索引，只支持表鎖；Archive只支持insert和select，支持壓縮適合存儲歸檔數據；還有太多不常用的，就不一一列舉了
  • innodb採用多線程模型，Master Thread負責將緩衝池中的數據異步刷新到磁盤，保證數據一致性，包括髒頁刷新，合併插入緩衝，undo頁回收；IO thread負責寫操作AIO的回調處理；Purge Thread 負責undo log的回收，減輕Master Thread的負擔；Page cleanner Thread 負責髒頁刷新，減輕master thread 負擔
  • mysql通過寫緩衝與WAL將隨機寫改成了順序寫提高了吞吐量，同時保證數據持久性，如果宕機就會用redo與undo兩階段的日誌來將數據恢復到數據庫文件中。但是mysql基本頁面一般是16KB，而操作系統基本頁面一般是4KB，如果在寫入到數據庫文件時發生宕機，就可能引起partial write問題，亦即一個mysql頁面只寫入了一部分，這時就要依靠double write：同步緩衝區數據時，不是直接寫入文件，而是寫入內存中的double write buffer再調用fsync寫到共享表空間文件，然後馬上調用fsync同步至數據庫文件，這樣如果發生了partial write，就可以去共享表空間找到該頁副本並複製到數據庫文件中，然後再進行redo,undo。當然，如果文件系統本身有防範機制就不必啟用double write了
  • InnodB中頁是管理數據庫的最小磁盤單位，分為多個部分，其中Fileheader、Pageheader、FileTrailer（檢查完整性）是固定的，記錄一些該頁的信息如checksum、頁所在B+樹層數。userrecords、freespace、pagedirectory（頁相對位置，B+索引只記錄一條數據所在頁，真正查找還需把這一頁加載如內存，靠二分搜索這一條記錄）是實際的行記錄存儲空間，因此大小是動態的。
  • Cardinality能幫助我們決策是否需要建立索引，可以通過show index來觀察，一般比如性別這個字段Cardinality就很低，因為可能一個類型female就佔據了所有行的一半，根本沒必要建立索引，而名字這個字段Cardinality就很高，適合建立索引（當然也得是應用常常按名字檢索，這樣才有必要建立索引）
  • Innodb1.2.x 開始支持全文檢索，並支持myisam引擎所有功能。全文檢索一般通過倒排索引實現，亦即存儲每一個詞彙及其對應文檔，這樣關鍵詞檢索時就能迅速返回相應文檔
  • 事務分為幾種：扁平事務，所有操作位於同一層次，要麼都提交要麼都回滾（也就是說不支持部分回滾），是應用程序成為原子操作的基本模塊；帶有保存點的扁平事務，在扁平事務基礎上支持部分回滾，用保存點（volatile，易失的）實現；鏈事務，事務的鏈接；嵌套事務，頂層事務管理多個子事務；分佈式事務，通常是在分佈式場景下運行的扁平事務
  • 事務的實現主要依靠：
  • redo實現持久性，亦即commit操作必須將專門的redo日誌文件fsync刷新到磁盤才能返回成功，合併多條語句插入優於一句一句插入也是因為減少了日誌刷盤頻率，由於redolog塊與磁盤扇區塊大小一致，所以無需doublewrite；
  • undo實現回滾操作和MVCC，是存在於數據庫內部共享表空間的一個特殊段（undo段）中的邏輯日誌，記錄了事務執行的反效果便於回滾，MVCC的實現是通過若某行記錄已被其他事物佔用，當前事務可以通過undo日誌得到改行之前版本的信息，需注意undolog也會產生redolog，亦即undolog也需要持久性維護；
  • purge最終完成delete和update操作，因為Innodb支持MVCC，所以記錄不能在提交時立即處理，purge操作判斷合何時真正清理數據並回收undo page，具體來說，如果一行數據不被真正引用那就可清理delete和update數據了；
  • group commit將多個事務的數據一次性調用fsync刷新到磁盤減少刷盤次數；
  • 需要注意的是binlog，用來實現Point-In-Time的恢復以及主從複製的實現，非常類似於redolog，但是兩者本質有很大不同：redolog產生於下層只針對Innodb引擎產生，是物理格式日誌，記錄針對每一頁的修改，事務過程中不斷被寫入；binlog產生於中層，針對任何引擎都會產生，是邏輯日誌，記錄針對每條SQL語句，事務提交後一次寫入
  • Mysql通過XA事務來保證分佈式的一致性，同時內部也有XA的使用，如現在數據庫為了複製一般都是打開binlog的，事務提交時既要寫binlog也要寫redolog，這就會涉及原子性問題，亦即兩個日誌必須同時寫入，通過先做prepare操作，將事務的xid寫入，然後寫入binlog，然後提交存儲引擎並寫入redolog

深入Linux內核架構

深入Linux內核架構 (豆瓣): https://book.douban.com/subject/4843567/

這是一本能把linux內核全貌展現給我們的大部頭，涵蓋了包括進程管理，內存管理、鎖與通信、設備驅動、文件系統等等。要採取觀其大略，用時細讀的策略，去宏觀的把握linux的設計哲學。

亮點：

  • 操作系統架構主要涉及兩種類型：微內核，最基本的功能由中央內核實現，其他功能委託給一些進程實現，通過通信接口與中心內核通信，這種模型研究活躍但是實用性進展甚微；宏（單）內核，內核的全部代碼打包到一個文件中，內核中每個函數都可以訪問其他部分，目前性能強於微內核，linux也是使用這種模式，但是linux模塊化做得很好（可以動態添加減少功能，內核文件也不會膨脹），也消除了單內核的一個重要弱點
  • linux虛擬地址空間是每個進程管理自身資源並與其他進程隔離的工具，與物理內存無直接關係（由CPU訪問總線數決定），分為內核空間（task-size~最高位）和用戶空間（0~task-size），這樣就為所有進程提供了一個統一的系統視圖，無需計較其他進程；Intel處理器有4中特權級別，但是linux只使用兩種不同的狀態：核心態和用戶態，用戶態進程禁止直接訪問內核空間（這是內核的專用領域）的數據和執行其代碼，用戶態進程只能藉助於系統調用或者中斷來陷入內核態，由內核代為執行操作。物理內存基本單位是頁框（頁幀），虛擬內存基本單位是頁，兩者大小一樣，對應關係由頁表（linux採用4級頁表）決定
  • 創建進程有兩種方式，fork（父子進程只有PID不同，使用COW使得fork更高效，亦即子進程只複製頁表，只有父子某一進程要向內存寫入數據時才真正複製物理內存，若只讀則可以共享而不必複製，這樣就延遲甚至消除了了大量的複製，節約了內存和CPU時間）和exec（將一個新程序加載到當前內存中執行，舊的程序內存頁將刷出）。linux使用clone創建線程（或者說輕量級進程），類似於fork，但是能精確檢查哪些資源與父進程共享哪些為線程獨立創建
  • Linux的進程調度分為兩大部分：調度策略和上下文切換。非實時進程的調度策略一般考慮公平性，linux所有可運行進程都在一個紅黑樹中排序，vruntime（一次調度間隔的虛擬運行時間，實際運行時間*(NICE_0_LOAD/權重nice值)）最小的進程位於左下方會被調度器優先考慮，休眠進程被放入等待隊列中，喚醒後會重新被加入紅黑樹。實時進程是指該進程必須在指定的時間內完成比如飛機飛行控制指令，因而一般都有相對非實時進程較高的優先級，不同於非實時進程調度
  • 物理內存管理分為兩個層次：夥伴系統負責物理頁幀的管理，圍繞多頁組成的連續內存塊的拆分與合併，新進的內核引入了反碎片技術（可移動頁與可移動內存區）防止物理內存碎片發生；slab負責處理小塊內存的分配，並提供了用戶層malloc函數族的內核等價物。
  • Linux使用了SystemV（Unix的一個分支）引入的機制來支持用戶進程間通信（IPC）與同步。主要通過信號量、消息隊列、共享內存三種方式實現。此外，linux還有其他IPC機制如信號（kill -s）、管道（以虛擬文件系統實現，適用於父子進程單向通信）及命名管道（適用於無關係的進程間通信）、socket（更通用，可多主機進程通信）
  • Linux內核為了統一文件訪問接口，在用戶進程（或者C標準庫）和文件系統（Ext2/3,XFS,Reiserfs）之間引入抽象層：虛擬文件系統（VFS）。文件系統一般分為3種：基於磁盤的FS（如Ext2/3），無持久FS（如/proc），網絡文件系統。處理文件時，用戶程序關注一個整型的文件描述符fd（由內核分配，只在一個進程內有效，兩個進程即使有相同的fd也是指向不同的文件），而內核關注的是inode，包含了一個文件/目錄的元數據但不含文件名（文件名只是inode的別稱）。軟連接採用文件名指向的方式，獨立inode；硬鏈接採用引用計數，共享inode
  • 無持久FS有：proc標準掛載點是/proc，其信息不能從塊設備讀取，只有在讀取文件內容時才動態生成相關信息，也可以在不重新編譯內核源碼的情況下修改內核行為，與內核通信，包含網絡信息、內存管理、進程特徵數據、文件系統、設備驅動、電源、終端、系統控制參數；sysfs的標準掛載點是/sys，是一個向用戶空間導出內核對象的文件系統，提供了查看和修改內核數據結構的能力；用於專門目的的小文件系統
  • 應用程序看到的內核是一個執行多種系統功能（內存管理、系統資源請求、訪問外設、進程通信、讀取文件等）的大的例程集合，標準庫是一箇中間層，用於在不同的體系結構和系統間，標準化並簡化內核例程的管理

同學們，堪稱整個現代互聯網軟件基石的Linux內核源碼就在這裡，有沒有勇氣去瞅瞅？

Kafka權威指南

Kafka權威指南 (豆瓣): https://book.douban.com/subject/27665114/

Kafka是一個高吞吐量的分佈式（支持分區partition、多副本replica、使用zookeeper協調）消息系統，目前廣泛用來進行實時大數據處理如：hadoop批處理、storm/Spark流式處理引擎，web日誌、消息隊列等等，所以作為一個後端開發者，很有必要了解一下。我想的是把kafka作為消息隊列的代表進行學習，將來如果要用ActiveMQ、RabbitMQ、ZeroMQ、RocketMQ等或者要自己要開發一個MQ都可以舉一反三或者進行借鑑。

亮點：

  • Kafka是一個流平臺，允許發佈和訂閱數據流，可以看做傳統的MQ，但是有很多不同點：支持多生產者，多消費者，另外多個消費者還可以組成群組，共享一個消息流並且特定消息整個群組只消費一次；以集群方式運行，可伸縮，還有MirrorMaker支持不統集群之間的複製；數據可以持久化，類似於數據庫，保證消息一定送達；流式處理使得你可以用很少的代碼就能動態的處理派生流和數據集。也可以看做實時版的Hadoop（大數據中，Hadoop主要應用於批處理與數據分析，Kafka由於低延遲更適合業務應用，能夠在業務事件發生時及時作出響應，為業務運營提供支撐，比如商品瀏覽數據轉換為商品推薦）。也可以看做ETL，但是並非簡單移動數據而是面向實時數據流。這三個方面是Kafka的主要特性
  • 生產者產生一個ProducerRecord對象，包含Topic和value，還可能有key或者partition，對象序列化後發送到分區器上，如果有partition則不操作，若無則根據key來指定partition（partition的設計目的是取消消息大小限制和提高並行度）。選好partition後發往對應的topic和partition的記錄批次，這一批記錄等待一次性的發往對應的broker（所以批的大小影響延遲和吞吐量），整個過程有失敗自動重試功能，超過一定次數返回客戶端異常。生產者通過設置ack=n，可以保證一個ProducerRecord有n個副本（一個Partition的所有副本中有一個Leader負責與客戶端交互，其餘副本作為Follower從Leader複製數據，提供容錯性）被寫入成功才會收到成功的ack消息。整個topic的順序無法保證，1個分區的順序可以通過設置retries>0（重試次數），和max.in.flight.requests.per.connection=1（1次只發1條消息避免後到的消息先成功）來保證，不過會嚴重影響生產者性能，若非嚴格順序要求不建議使用
  • 一個消費者group總是能獲得一個topic的全部消息不管還有沒有其他group訂閱該topic，group內通過添加消費者來伸縮讀取能力和處理能力，每個消費者只處理一部分數據，而且消費者數量不要超過partition數量，不然多餘的會閒置浪費。group內所有消費者向群組協調器發送心跳保持活躍，添加和刪除消費者時的再均衡（分配分區與消費者對應關係）也是由群組協調器完成
  • Kafka使用ZooKeeper維護集群成員信息，所有broker都對應/brokers/ids的臨時節點並訂閱其變更事件，在broker加入或退出時得到消息；集群中有一個控制器對應/controller臨時節點（所有broker都可以嘗試創建節點成功則成為Controller，失敗則知道Controller已經存在，所有broker都會監聽這個節點的變更，通過遞增的controller epoch來忽略舊控制器消息避免腦裂），除了完成broker功能之外還負責partition副本的leader選舉；持續得到來自leader最新消息的副本叫做同步的副本，當leader失效時，只有同步的副本才可能被選為新的leader；不同partition的副本和leader都均勻的分佈在所有broker之間避免熱點問題，副本儘量放在不同機架提高容錯性
  • kafka處理常見幾種請求：元數據請求，可以發送給任意broker，包含客戶端感興趣的主題列表，返回主題包含的分區、副本及其leader；生產請求，ack可以為0,1,all三個值；讀取請求，零拷貝返回客戶端指定的消息（省掉了用戶緩衝區，直接從磁盤到內核），要注意的是，leader只有把數據安全複製到副本中才能給客戶端返回這個數據；其他請求，如偏移量（不再保存在ZooKeeper中）
  • 因為大文件的查找和刪除都很費時易錯，所以每個partition都分成了許多segment，一個segment包含1GB或者1周的數據（較小為準），正在寫入數據的叫做活躍片段，永遠不會被刪除。每個分區還有一個索引提升查詢效率
  • 可靠性和速度（簡便性）通常是負相關的，Kafka在這兩者之間是可以高度配置的，適應從跟蹤用戶點擊到支付操作的不同可靠性級別要求。kafka的可靠性機制包括：分區消息順序性、多副本確認實現容錯性、只要還有一個副本是活躍的已提交消息就不會丟失、消費者只能消費已提交數據。這些基本機制加上一些配置參數（unclean.leader.election.enable、min.insync.replicas、生產者ack）的權衡就能構建不同級別的可靠系統，可靠級別可以是針對broker或者topic。總而言之、可靠性不僅是kafka單方面的事情，需要整個系統配合如系統架構、生產者消費者的使用方式與配置、topic和broker配置等等
  • 跨集群的複製在kafka中叫做mirror功能，應用場景：將不同區域的集群數據複製到中心集群便於分析數據、冗餘實現災備、雲遷移。跨數據中心通信有高延遲、有限帶寬、高成本等特點，常見架構有：Hub-Spoke架構（一中央多本地集群）、雙活架構（兩個集群都對外服務，就近原則）、主備架構（主集群服務、備集群只在主不可用時對外服務）
  • 事件流是無邊界的（持續增長）、有序的、可重播的、不可變的記錄。流式處理就是實時的處理一個或多個事件流，是一種編程範式就像請求響應式和批處理式，但是延遲和吞吐量都位於兩者之間。

看完這本書基本就能把kafka的使用方法和基本原理搞清楚了，但是翻譯有時候有點點問題，我覺得普通的段落可以翻譯，但是專用名詞如segment、partition等還是直接保留比較好，免得有歧義，當然整體上來來說配合官網文檔讀起來還是沒問題的。

NoSQL精粹

NoSQL精粹 (豆瓣): https://book.douban.com/subject/25662138/

本書先分析了傳統關係數據庫的不足然後引入NoSQL，講到了鍵值、文檔、列族、圖等多種數據庫並分析了其優勢劣勢，可以讓我們對NoSQL有一個全面的瞭解，為我們進一步NoSQL的探索之路開一個好頭。

亮點：

  • NoSQL泛指最近誕生非關係型數據庫如Cassandra、MongoDB、Neo4j等，他們主張無模式（schemaless）的數據。傳統關係數據庫有可持久化、模型標準便於共享和集成、事務支持等優點，NoSQL有無模式便於程序對象與數據庫之間的映射、適應集群等優點。所以NoSQL和傳統關係數據庫各有所長，發展過程中也在互相借鑑優點，誰也不能取代誰，將來都會在各自的領域中發光發熱
  • 關係模型把數據分割成簡單的行，通常一個對象可能需要多行數據通過外鍵連接在一起來表示（如果按照嚴格的關係數據庫範式）；聚合是領域驅動設計的術語，把一組相關聯的對象視為一個整體單元來操作，這個單元就叫做聚合（aggregate）。聚合的邊界劃分沒有標準答案，取決於你之後準備怎樣操作數據。因為以聚合為單位複製和分片比較自然所以集群中操作數據庫還是聚合更簡單。若數據操作大多在同一聚合內執行則應使用面向聚合的數據庫（鍵值、文檔、列族數據庫），若交互需要多種不同的格式則最好選聚合無知式（aggregate-ignorant）數據庫（關係數據庫、圖數據庫），若待處理數據中有大量的關係最好就選關係數據庫，但是如果關係複雜、並行交錯最好選圖數據庫（插入費時、查詢比複雜的join快得多）
  • 寬泛的說數據分佈有兩條正交的路徑：複製，一份數據拷貝到多個節點，有主從式（主節點瓶頸問題）和對等式（一致性問題）兩種形式；分片，將不同的數據存放到不同的節點。這裡注意不要和另一個概念混淆：數據庫水平（一個表的不同行數據置於不同節點）、垂直（一個表的不同列置於不同節點或者拆為多個表）拆分。實際分佈式系統中一般都是複製和分片結合的（如上面提到的kafka）
  • 版本戳可以檢測衝突用來實現CAS，可以用計數器、GUID、Hash、時間戳等方式實現也可以組合實現，分佈式系統中，可以採用“由版本戳構成的數組”來檢測不同節點是否發生了“相互衝突的更新操作”
  • 鍵值數據庫最簡單，只能按鍵檢索，適用於會話信息、用戶配置信息、購物車等；文檔數據庫呈樹狀結構可以看做支持值檢索的鍵值數據庫，適用於CMS及博客、電子商務等；列族數據庫將數據存在列族中，列族裡的行則把許多列數據與本行的行鍵聯繫起來，列族通常把需要一併訪問的相關數據分成組，提高訪問效率，適用於事件記錄、博客、計數器等；圖數據庫把具有屬性的實體（節點）按關係（邊）聯繫起來，適用於互聯網數據如社交網絡、安排運輸線、推薦引擎等
  • Neal Ford 2006年造出了多語言編程（polyglot programming）：應該以多種語言混合編寫同一應用程序，以各自語言的優勢來解決其中不同的問題，同理數據庫也可以遵循類似的原則混合持久化（polyglot persistence），比如購物車可以用鍵值數據庫存儲，但是訂單要用關係型數據庫。所以除非是編程語言狂熱愛好者，我們大部分人都不應該去爭論什麼語言好，什麼數據庫好，不要想著一把刷子走遍世界，而應該廣泛瞭解，解決問題的時候選擇合適的工具，必要的時候也要自己改工具甚至造工具

本書的一個問題就是有點舊了，2013年初版，這5年來NoSQL發展很快，已經被廣泛用於企業級的系統之中了，尤其是Redis和MongoDB，所以有需求就大膽的用吧。

大型網站技術架構

大型網站技術架構 (豆瓣): https://book.douban.com/subject/25723064/

把事情做大是我們許多人的追求，這本書就可以讓我們瞭解一下一個大型網站的架構應該要注意些什麼，無論是摸得著的tech還是比較摸不著的leadership，都值得借鑑。當然，書很薄，大多隻能泛泛而談，要想真正掌握，每一部分我們都得自己單獨深挖，所以本書可以看做一個大目錄，為接下來的學習指引方向。

亮點：

  • 講述了一個單機網站發展到大型分佈式網站的完整歷程，處於業務快速發展期間的公司可以借鑑，提前做好規劃
  • 架構設計時要注意：不要盲目追隨大公司而是要考慮自己業務；要結合網站實際使用新技術，不要盲目追求新奇酷炫；不要想著用技術解決一切問題，業務模式等也需要考慮；
  • 模式：每一個模式描述了一個在我們周圍不斷重複的問題以及該問題解決方案的核心，這樣你就能1次又1次的使用該方案而不必重複勞動。模式的關鍵在於可重複性，包括問題和方案的可重複性。
  • 大型網站有一些常見的架構模式：分層，如應用層、服務層、數據層；分割，如購物、論壇、搜索、廣告；分佈式，包括服務、靜態資源、數據庫、計算、鎖、文件、配置信息等；集群，相同業務通過集群負載均衡提高服務能力；緩存，如CDN、反向代理、本地緩存、分佈式緩存；異步，事務關係越少就越能獨立發展，異步也是解耦的一種協作方式；冗餘，冷備、熱備、災備等；自動化，包括髮布、測試、部署、報警、失效轉移、恢復、降級、分配資源等，自動化能提高效率並減小人為出錯概率；安全，手機驗證碼、圖像驗證碼、轉義、風控等。所有模式都是在性能（響應時間、qps等）、可用性（冗餘）、伸縮性（機器的增減）、擴展性（業務的增減）、安全性（惡意訪問、竊取、攻擊）這幾個維度之間進行平衡
  • 性能針對不同群體需要不同的定義（針對用戶的響應時間，針對系統的吞吐量、併發量等）來衡量，也需要測試（壓測等），同時要從架構的多個層面（前端、應用層、存儲層）來進行優化
  • 可用性一般用幾個9來形容（4個9就是99.99%，一年小於53分鐘不可用），實現高可用架構的主要手段是數據和服務的冗餘備份和失效轉移。應用層（文庫、貼吧、知道）和服務層（賬戶服務、session服務）因為一般無狀態所以很容易擴展（負載均衡），新節點上下線都很容易，但是服務層需要注意好分級管理、超時設置、異步調用、服務降級、冪等設計等。數據層（數據庫、文件、緩存）為了保證數據不丟失需要複製並均勻分佈在集群中
  • 集群處理能力如果能和服務器數量正相關，那麼網站就具有伸縮性架構，如果能成線性關係那更是高伸縮性，主要分為不同服務物理拆分和單一服務集群伸縮（類比於數據庫垂直拆分和水平拆分）。應用集群伸縮性一般通過負載均衡實現如http重定向、DNS輪詢、反向代理、IP負載均衡、數據鏈路層負載均衡等；緩存集群伸縮性僅靠負載均衡是不夠的，還需要一致性hash、虛擬槽分區等方法使得緩存在節點發生變化時不會集體失效；存儲集群伸縮性對持久性和可用性要求更高，但是這裡作者沒怎麼講具體做法
  • 擴展性是在對現有系統影響最小的情況下持續提升或增加服務的能力，表現在系統基礎設施穩定，應用之間耦合較少等，是系統架構層面的開閉原則體現。主要有：分佈式消息隊列解耦生產者與消費者、分佈式服務打造可複用的業務平臺、利用開放平臺打造生態圈實現開放共贏等方式
  • 安全性一般主要注意XSS、注入、CSRF等攻擊，主要手段有：用戶輸入轉義、表單token、驗證碼、加密、過濾、風控等
  • 領導的真諦就是尋找一個值得共同奮鬥的目標，營造一個讓大家都能最大限度發揮自我價值的氛圍；發掘人的優秀比發掘優秀的人更有意義

深入理解 Java 內存模型

這本書是infoq上的系列博客形成的一本很薄的書，對於瞭解JMM還是很有用的。

深入理解Java內存模型（二）——重排序: http://www.infoq.com/cn/articles/java-memory-model-2/

亮點：

  • 重排序的目的是提高性能，分為三種：編譯器、處理器指令級並行技術、內存系統的重排序。JMM屬於語言級別的內存模型，通過禁止特定類型的編譯器重排序（直接可控）和處理器重排序（通過內存屏障實現）來在不同的平臺對為程序員提供一致的內存模型
  • 內存屏障有四種：Store和Load的排列 ，其中StoreLoad Barriers是一個全能型的屏障，同時具有其他三個屏障的效果，它確保store數據對其他處理器的load可見
  • 處理器內存是硬件級別的內存模型，JMM是一個語言級別的內存模型，順序一致性內存是一個理論參考模型，性能依次降低，可編程性依次上升。所以JMM主要關注易編程性和性能，在兩者之間求得平衡

深入分佈式緩存：從原理到實踐

深入分佈式緩存：從原理到實踐 (豆瓣): https://book.douban.com/subject/27602483/

緩存在如今的互聯網中已經幾乎是標配了，分佈式緩存更是大型網站不可或缺的，本書就讓我們從原理到開源實現到業務實踐一起打包瞭解了。

亮點：

  • 計算機世界中，無論是硬件還是軟件，無論是單機還是分佈式，緩存都被用於解決響應緩慢的問題，是提升性能的關鍵技術。其核心就是空間換時間，解耦高速部件與低速部件。緩存由於不同硬件的訪問速度和價格不同而進行分級，每一級存儲對應熱度級別的數據，分層逐級減少流量的衝擊，保護用戶體驗與系統可用性。分佈式環境下要關注其命中、更新、失效、一致性等
  • 緩存是無所不在的，從架構的角度看可以分為三塊：客戶端緩存 在客戶端減少網絡請求，如頁面緩存（文件化）、瀏覽器通過HTTP協議實現的緩存（etag，304 not-modified）、APP緩存（sqlite）等；網絡緩存 位於客戶端與服務端之間，代理部分客戶端請求給服務端減負，如web代理緩存（nginx）、邊緣緩存（CDN）；服務端緩存 是緩存的重點和難點，如數據庫緩存（query_cache）、平臺級緩存（大而全的Ehcache、極簡的Guava ）、應用級緩存（redis）
  • 書中澄清了許多分佈式方面的概念和理論，如集群是一組互相獨立的計算機通過網絡連接構成一個組實現同一個功能，對客戶端表現為一個獨立服務器，可以增加可用性與伸縮性，而分佈式系統就是許多不同功能的集群構成的一個實現特定業務的系統；冪等性是指調用1次和N次結果一樣，這個在分佈式系統中很重要是因為網絡抖動等因素常常導致重發；分佈式系統的本質是一堆廉價的硬件攢在一起獲得更好的吞吐量和性能以及可用性，有幾個問題普遍關心：活性檢測（週期心跳、累計失效檢測）、高可用（主備、雙主、集群）、容錯處理（冷熱切換、冗餘、備份）、重試機制（失敗重試、事務補償）、負載均衡（多機分擔請求），也有幾個實踐很重要：全局ID（數據庫自增id如flicker、UUID、snowflake、預生成置於緩存中）、哈希取模分配、一致性哈希（單調性、分散性）、路由表與數據拆分
  • 書中在JSR規範下帶領我們實現了一個緩存框架CSCache，架構由上到下是客戶端、緩存提供層、緩存管理層、緩存存儲層（基本存儲，LRU存儲，Weak存儲），設計思路值得借鑑
  • redis有瓶頸：單機數據量限制、單線程吞吐量限制（尤其是大塊數據讀取延遲明顯）、單點故障等，目前已有的解決方案有：水平拆分如Proxy和Twemproxy、自帶主從複製（含斷點續傳）、sentinel監控自動故障轉移（使用raft類似算法選主）、redis3.0後出來的去中心化（採用Gossip協議在節點之間達成一致）的cluster提供了完整的sharding、replication、failover等解決方案
  • 本書覆蓋了廣告、社交、電商等多種業務的緩存模式，能開開眼界，將來可能還真用得上，尤其是社交和電商這兩塊講得很細緻，幾乎可以作為一精簡而完整的解決方案了
