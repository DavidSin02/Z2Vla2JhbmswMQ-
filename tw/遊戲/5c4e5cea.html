前言

前面幾篇文章我們瞭解了實時操作系統的主要概念以及內核結構，接下來我們一起看下操作系統的精髓《任務管理》，本章所講的內容包括如何在用戶的應用程序中建立任務、刪除任務、改變任務的優先級、掛起和恢復任務，以及獲得有關任務的信息。

一、任務的創建

µC/OS-Ⅱ可以管理多達 64 個任務，並從中保留了四個最高優先級和四個最低優先級的任務供自己使用，所以用戶可以使用的只有 56 個任務。任務的優先級越高，反映優先級的值則越低。在最新的µC/OS-Ⅱ版本中，任務的優先級數也可作為任務的標識符使用。

μC/OS-I是通過任務控制塊來管理任務的。因此,創建任務的工作實質上是創建一個任務控制塊,並通過任務控制塊把任務代碼和任務堆棧關聯起來形成個完整的任務。當然,還要使剛創建的任務進入就緒狀態,並接著引發一次任務調度。c/os-i有兩個用來創建任務的函數: OSTaskCreate和 OSTaskCreateExt(其中OSTaskCreateExt()是 OSTaskCreate()的擴展,並提供了一些附加功能。用戶可根據需要使用這兩個函數之一來完成任務的創建工作。

任務的創建有以下三個要點

  1. 任務可以在多任務調度開始 (即調用OSStart())之前創建，也可以在其它任務的執行過程中被創建。
  2. 在OSStart()被調用之前，用戶必須創建至少一個任務；
  3. 不能在中斷服務程序(ISR)中創建新任務

下面是OSTaskCreate的源碼


INT8U OSTaskCreate (void (*task)(void *pd), void *pdata, OS_STK *ptos, INT8U

prio)

{

void *psp;

INT8U err;

if (prio > OS_LOWEST_PRIO) {

return (OS_PRIO_INVALID);

}

OS_ENTER_CRITICAL();

if (OSTCBPrioTbl[prio] == (OS_TCB *)0) {

OSTCBPrioTbl[prio] = (OS_TCB *)1;

OS_EXIT_CRITICAL();

psp = (void *)OSTaskStkInit(task, pdata, ptos, 0);

err = OSTCBInit(prio, psp, (void *)0, 0, 0, (void *)0, 0);

if (err == OS_NO_ERR) {

OS_ENTER_CRITICAL();

OSTaskCtr++;

OSTaskCreateHook(OSTCBPrioTbl[prio]);

OS_EXIT_CRITICAL();

if (OSRunning) {

OSSched();

}

} else {

OS_ENTER_CRITICAL();

OSTCBPrioTbl[prio] = (OS_TCB *)0;

OS_EXIT_CRITICAL();

}

return (err);

} else {

OS_EXIT_CRITICAL();

return (OS_PRIO_EXIST);

}

}

從上面的源碼中，我們可以看到創建任務的過程如下：

  • 任務優先級檢查 ：該優先級是否在0到OS_LOWSEST_PRIO之間？該優先級是否空閒？
  • 調用OSTaskStkInit()，創建任務的棧幀；
  • 調用OSTCBInit()，從空閒的OS_TCB池（即OSTCBFreeList鏈表）中獲得一個TCB並初始化其內容，然後把它加入到 OSTCBList鏈表的開頭，並把它設定為就緒狀態；
  • 任務個數OSTaskCtr加1；
  • 調用用戶自定義的函數OSTaskCreateHook()；
  • 判斷是否需要調度（調用者是正在執行的任務）

返回值

  • OS_NO_ERR：函數調用成功；
  • OS_PRIO_EXIT：任務優先級已經存在；
  • OS_PRIO_INVALID：任務優先級無效。

二、任務的棧空間

每個任務都有自己的棧空間（Stack），棧必須聲明為OS_STK類型，並且由連續的內存空間組成；

  • 棧空間的分配方法靜態分配：在編譯的時候分配
  • static OS_STK MyTaskStack[stack_size]; OS_STK MyTaskStack[stack_size];
  • 動態分配：在任務運行的時候使用malloc()函數來動態申請內存空間；
  • OS_STK *pstk;
  • pstk = (OS_STK *)malloc(stack_size);
  • /* 確認malloc()能得到足夠的內存空間 */
  • if (pstk != (OS_STK *)0)
  • {
  • Create the task;
  • }

在動態分配中，用戶要時刻注意內存碎片問題。特別是當用戶反覆地建立和刪除任務時，內存堆中可能會出現大量的內存碎片，導致沒有足夠大的一塊連續內存區域可用作任務堆棧，這時 malloc()便無法成功地為任務分配堆棧空間。

上圖表示了一塊能被 malloc()動態分配的 3K 字節的內存堆 。為了討論問題方便，假定用戶要建立三個任務(任務 A,B 和C)，每個任務需要 1K 字節的空間。設第一個 1K 字節給任務 A, 第二個 1K 字節給任務 B, 第三個 1K 字節給任務 C。然後，用戶的應用程序刪除任務 A 和任務 C，用 free()函數釋放內存到內存堆中。現在，用戶的內存堆雖有 2K 字節的自由內存空間，但它是不連續的，所以用戶不能建立另一個需要 2K 字節內存的任務(即任務 D)。如果用戶並不會去刪除任務，使用 malloc()是非常可行的

棧的增長方向的設置

從低地址到高地址：在OS_CPU.H中，將常量OS_STK_GROWTH設定為 0；

從高地址到低地址：在OS_CPU.H中，將常量OS_STK_GROWTH設定為 1；

OS_STK TaskStack[TASK_STACK_SIZE]; OSTaskCreate(task, pdata,&TaskStack[TASK_STACK_SIZE-1],prio);

三.任務的掛起和恢復

1.掛起任務

所謂掛起一個任務,就是停止這個任務的運行。在uC/OS-Ⅱ中,用戶任務可通過調用系統提供 OSTaskSuspend()函數來掛起自身或者除空閒任務之外的其他任務。用函數OSTaskSuspend)掛起的任務,只能在其他任務中通過調用恢複函數 OSTaskResume()使其恢復為就緒狀態。任務在運行狀態、就緒狀態和等待狀態之間的轉移關係下圖所示。

掛起任務函數 OSTaskSuspend()的原型為INT8U OSTaskSuspend (INT8U prio);函數的參數prio為待掛起任務的優先級別。如果調用函數 OSTaskSuspend()的任務要掛起自身,則參數必須為常數s_pIO_SELF(該常數在文件uCOS_II.H中被定義為OxFF)

當函數調用成功時,返回信息 OSNOERR;否則根據出錯的具體情況返回OS_TASKSUSPEND_IDLE、OS_PRIO_INVALID和OS_TASK_SUSPEND_PRIO

該函數在一系列的判斷中主要是判斷待要掛起的任務是否調用這個函數的任務本身。如果是任務本身,則必須刪除任務在任

務就緒表中的就緒標誌,並在任務控制塊成員 OSTCBStat中做了掛起記錄之後,引發一次任務調度,以使CPU去運行就緒的其他任務。如果待掛起的任務不是調用函數的任務本身而是其他任務,那麼只要刪除任務就緒表中被掛起任務的就緒標誌,並在任務控制塊成員OSTCBStat中做了掛起記錄即可。 2.恢復任務

OSTaskResume()：

  • 恢復一個任務恢復被OSTaskSuspend()掛起的任務；
  • 清除TCB中OSTCBStat字段OS_STAT_SUSPEND位

函數的參數為待恢復任務的優先級別。若函數調用成功,則返回信息OS_NO_ERR;否則,據出錯的具體情況返回OS_PRIO_INVALID、OS_TSK_ RESUME_PRIO和OS_TASK_NOT_SUSPEND等。OSTaskResume()函數在判斷任務確實是一個已存在的掛起任務,同時它又不是一個等待任務(任務控制塊成員 OSTCBDly=0)時,就清除任務制塊成員 OSTCBStat中的掛起記錄並使任務就緒,最後調用調度器 OSSched()進行任務調度,並返回函數調用成功的信息OS_NO_ERR

四.刪除任務

所謂刪除一個任務,就是把該任務置於睡眠狀態。具體做法是,把被刪除任務的任務控制塊從任務控制塊鏈表中刪除,並歸還給空任務控制塊鏈表,然後在任務就緒表中把該任務的就緒狀態位置0,於是該任務就不能再被調度器所調用了。簡單地說,就是把它的身份證給吊銷了。在任務中,可以通過調用函數 OSTaskDel來刪除任務自身或者除了空閒任務之外的其他任務。

如果一個任務調用這個函數是為了刪除任務自己,則應在調用函數時令函數的參數pri為OS_PRIO_SELF有時,任務會佔用一些動態分配的內存或信號量之類的資源。這時,如果有其他任務把這個任務刪除了,那麼被刪除任務所佔用的一些資源就會因為沒有被釋放而丟失,這是任何系統都無法接受的。

因此,在刪除一個佔用資源的任務時,一定要謹慎。具體的辦法是,提出刪任務請求的任務只負責提出刪除任務請求,而刪除工作則由被刪除任務自己來完成。這樣,被刪除任務就可以根據自身的具體情況來決定何時刪除自身,同時也有機會在刪除自身之前佔用的資源釋放掉。

顯然,如果想使提出刪除任務請求的任務和被刪除任務之間,能夠像上述方式來執行刪工作,則它們雙方必須有某種通信方法μC/OS-II利用被刪除任務的任務控制塊成員OSTCBDelReq作為請求刪除方的被刪除方的聯絡信號,同時提供了一個雙方都能調用的函數請求刪除任務函數 OSTaskDelReq()。這樣,提出刪除任務請求的任務和被刪除任務的雙方就都使用這個函數來訪問 OSTCBDelReq這個信號,從而可以根據這個信號的狀態來決定各自的行為。

刪除任務請求方要用被刪除任務的優先級別prio作為參數來調用這個函數。從圖中可以看出,刪除任務請求方調用這個函數的目的就是要查看被刪除的任務控制塊是否還在如果還在,則令被刪除任務的任務控制塊成員 OSTCBDelReq的值為 OSTaskDelReq且通知該任務:“已經有任務要求在合適的時候要刪除自己”;如果不在,認為被刪除任務經被刪除了。
